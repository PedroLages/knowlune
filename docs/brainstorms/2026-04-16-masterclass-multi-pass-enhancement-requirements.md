# Masterclass: Multi-Pass Enhancement with Self-Critique

## Problem Frame

The `/masterclass` skill currently executes all 12 pedagogical enhancement layers in a single Claude pass. Asking the model to hold structure, anchoring, scaffolding, and engagement concerns simultaneously degrades quality across all of them — particularly on longer or denser content. A multi-pass architecture narrows the focus of each pass, making layer compliance verifiable and quality failures diagnosable.

## Enhancement Flow

```
Raw transcript
      │
      ▼
┌─────────────────────────────────────┐
│ Pass 1 — Structure                  │
│ Layers 1-2: paragraph breaks,       │
│ section signposts                   │
└────────────────┬────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────┐
│ Pass 2 — Pedagogy                   │
│ Layers 3-5: examples, micro-recaps, │
│ step labels                         │
└────────────────┬────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────┐
│ Pass 3 — Engagement                 │
│ Layers 8-9-10-11-12: mnemonic,      │
│ reflection, practice, cross-refs,   │
│ emphasis                            │
└────────────────┬────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────┐
│ Pass 4 — Bookends (final position)  │
│ Layers 6-7: opening preview,        │
│ closing recap (depend on full text) │
└────────────────┬────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────┐
│ Pass 5 — Critique                   │
│ Faithfulness check + layer audit    │
│ Flags issues, does NOT re-enhance   │
└────────────────┬────────────────────┘
                 │
                 ▼
            Enhanced text
         (+ critique report)
```

## Requirements

**Pass Architecture**
- R1. The 12 enhancement layers are split across four sequential passes. The order is intentionally out of numerical sequence — Layers 6-7 (Bookends) are deferred to Pass 4: Structure (L1-2) → Pedagogy (L3-5) → Engagement (L8, L9-12) → Bookends (L6-7). Each pass receives the output of the previous pass as its input.
- R2. Layers 6-7 (Opening Preview and Closing Recap) are applied in a dedicated final pass (Pass 4), after all other content layers, because they must reference the complete enhanced text to be accurate.
- R3. Each pass operates on the output of the previous pass as its full input. No pass re-reads the raw original transcript — except the critique pass (R6). Pass 4 (Bookends) writes its opening and closing based on the enhanced text alone, not the original.
- R4. In `light` mode, only Pass 1 (Structure) and Pass 4 (Bookends, L6-7 only) run. Passes 2, 3, and 5 are skipped.
- R5. In `simple` mode, no enhancement passes run (unchanged from current behavior).

**Critique Pass**
- R6. After Pass 4, a dedicated critique pass re-reads the original transcript and the fully enhanced text and checks: (a) are all expected layers present for the mode? (b) does the enhanced text faithfully represent the source — no hallucinated claims, no distorted arguments? (c) were any layers skipped or thin?
- R7. The critique pass produces a short report (printed to terminal, not written to file) listing any gaps or faithfulness issues found. It does NOT attempt to fix them.
- R8. If the critique finds no issues, it outputs a single line: "Critique: OK".
- R9. The critique pass does not block TTS generation — it is advisory only.

**Continuity**
- R10. Each pass must preserve all content from the previous pass and only add to it. No pass may delete or restructure content generated by a prior pass. If the word count of a pass output is more than 5% shorter than its input, the skill reports a warning and halts.
- R11. After each pass completes, the skill logs to terminal: "Pass [N] complete — [word count] words." Logged after completion, before the next pass starts.

**Mode Table**

| Mode | Passes run | Layers covered |
|------|-----------|----------------|
| `full` | 1, 2, 3, 4, 5 (critique) | All 12 |
| `light` | 1, 4, 5 (critique) | L1-2, L6-7 |
| `simple` | None | None |

## Success Criteria

- Each pass's output is visibly better in its specific dimension than the single-pass output for the same content.
- The critique pass catches at least one real issue on a content sample where a layer was missed or the source was distorted.
- Enhancement time is accepted to increase compared to current single-pass (multiple LLM calls over a growing document); no hard ceiling is set, but passes 1-3 should each take under 60s on typical content.
- The skill's quality checklist (currently manual) is partially automated by the critique report.

## Scope Boundaries

- The critique pass flags issues only — it does not auto-fix or trigger a re-run. Iterative re-enhancement is out of scope for this change.
- No changes to Stage 1 (extraction), Stage 2 (transcription), or Stage 4 (TTS) — this change is Stage 3 only.
- No new user-facing flags or mode keywords introduced by this change.
- Content-adaptive layer selection (choosing which layers to apply based on content type) is a separate idea and out of scope here.

## Key Decisions

- **Bookends in their own pass (not pass 3):** Opening Preview and Closing Recap must reference the full enhanced content to be accurate. Embedding them in Pass 3 would require a second read of content still being written.
- **Critique is advisory, not blocking:** Blocking TTS on critique findings would require user intervention on every run. The value is visibility, not gatekeeping — critique output informs the user whether to re-run or accept.
- **Passes build sequentially, not in parallel:** Parallelizing Pass 1 and 2 (both on raw transcript) is possible but sacrifices the quality benefit of structured input for pedagogy layers. Structure first makes examples and recaps more coherent.

## Dependencies / Assumptions

- MimikaStudio TTS is invoked after all passes complete — no change to Stage 4 integration.
- The skill runs in Claude Code where multi-step execution within a single skill invocation is supported.
