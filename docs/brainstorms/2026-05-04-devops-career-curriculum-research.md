## Certification Path

### Recommended Certification Sequence

```
1. CKA (Certified Kubernetes Administrator)    — Month 3–5    (~$395)
2. AWS Solutions Architect Associate            — Month 5–7    (~$150)
3. HashiCorp Terraform Associate (003)          — Month 7–8    (~$70)
4. CKAD (Certified Kubernetes App Developer)    — Month 8–9    (~$395, optional)
```

**Total certification cost: $615–$1,010**

### Why This Order?

- **CKA first**: Most respected K8s cert. Practical, hands-on, terminal-only exam. The studying
  itself builds real skills. Includes 2 free Killer.sh simulator sessions.
- **AWS SAA second**: Most valuable cloud cert. Teaches how AWS services fit together. Complements
  CKA (many K8s clusters run on EKS).
- **Terraform Associate third**: Ubiquitous IaC tool. Cheapest cert ($70). Can be done in 4–6 weeks.
- **CKAD fourth (optional)**: Developer perspective on K8s. Significant overlap with CKA — do it
  while CKA knowledge is fresh if you want both admin + developer credentials.

### Certs You Can Skip (for junior level)

- **KCNA** — too basic if you're committed to CKA. Extra $250 for low signal.
- **CKS** — requires CKA first, security focus. Do 1–2 years into your career.
- **LFCS** — Linux cert. CKA already tests Linux extensively. Redundant.
- **Any Azure cert** — unless targeting Azure shops specifically. AWS has larger Dutch market share.

| Exam | Primary Prep | Simulator | Study Time |

### 12-Month Curriculum (Recommended — Sustainable Part-Time)

#### Phase 1: Foundations (Months 1–2)

**Goal:** Become comfortable on the Linux command line. Understand how computers and networks work.
Start scripting.

**Week 1–4: Linux & Command Line**
- [ ] Install Ubuntu (or any Linux) as your daily driver (dual-boot or dedicated machine)
- [ ] Complete the free KubeCraft Linux course on Skool (8 hours, 4,300 members) or Linux Journey (linuxjourney.com)
- [ ] Learn: file system navigation, permissions, users/groups, processes, package management
- [ ] Learn: grep, sed, awk, find, xargs, pipes, redirection
- [ ] Practice: navigate and fix things without a GUI. Break things intentionally.

**Week 5–6: Networking Fundamentals**
- [ ] OSI model (understand layers 2–4 deeply, rest conceptually)
- [ ] TCP vs UDP, DNS, HTTP/HTTPS
- [ ] Subnetting, CIDR notation, routing basics
- [ ] Practical: Use `curl`, `dig`, `ping`, `traceroute`, `nc`, `tcpdump`
- [ ] Understand: What happens when you type a URL into a browser — end to end

**Week 7–8: Bash Scripting + Git**
- [ ] Write scripts to automate common tasks (log rotation, system health checks, backups)
- [ ] Git fundamentals: clone, commit, branch, merge, rebase, pull requests
- [ ] GitHub: create repos, understand Issues/PRs/Actions conceptually
- [ ] **Project:** Server performance stats script that outputs CPU, memory, disk, network (Bash)

**Milestone:** You can SSH into a server, navigate, troubleshoot, write a script, commit to Git.

---

#### Phase 2: Programming + Containers (Months 3–4)

**Goal:** Write Python for automation. Dockerize applications. Understand CI/CD concepts.

**Week 9–12: Python for Automation**
- [ ] Python basics: data types, loops, functions, file I/O, error handling
- [ ] Working with JSON, YAML, CSV
- [ ] HTTP requests (requests library)
- [ ] AWS SDK (boto3) basics
- [ ] **Project:** Log analysis tool — parse a directory of JSON log files, extract error counts by service, output a summary report
- [ ] **Project:** CLI tool that interacts with a REST API (e.g., GitHub API to list repos/issues)

**Week 13–16: Docker**
- [ ] Docker fundamentals: images, containers, Dockerfiles, layers, registries
- [ ] Multi-stage builds, optimizing image size
- [ ] Docker Compose: multi-container apps, networking, volumes
- [ ] **Project:** Dockerize a simple web app (frontend + backend + database) with Docker Compose
- [ ] **Project:** Write a Dockerfile with multi-stage builds, push to Docker Hub / GitHub Container Registry

**Milestone:** You can write Python scripts for automation and Dockerize any application.

---

#### Phase 3: CI/CD + Cloud Basics (Months 5–6)

**Goal:** Build automated pipelines. Understand cloud infrastructure conceptually and practically.

**Week 17–20: CI/CD**
- [ ] GitHub Actions: workflows, jobs, steps, secrets, environments
- [ ] Build a pipeline: lint → test → build Docker image → push to registry → deploy
- [ ] GitLab CI concepts (understand the YAML structure, know it exists)
- [ ] CI/CD best practices: trunk-based development, fast feedback, immutable artifacts
- [ ] **Project:** CI/CD pipeline that builds, tests, and deploys a Dockerized app to a VPS on every push
- [ ] **Project:** Add automated testing, linting, and security scanning (Trivy) to the pipeline

**Week 21–24: AWS Cloud Basics**
- [ ] AWS core services: EC2, S3, IAM, VPC, RDS, Lambda, CloudWatch
- [ ] AWS Free Tier: launch EC2 instances, create S3 buckets, configure IAM roles
- [ ] Understand: regions, availability zones, security groups, networking
- [ ] Start AWS SAA study (Adrian Cantrill course)
- [ ] **Project:** Deploy your Dockerized app to AWS (EC2 + RDS or ECS Fargate)

**Milestone:** You have automated CI/CD pipelines and can deploy to AWS from code.

---

#### Phase 4: Kubernetes Core (Months 7–8)

**Goal:** The main event. Deploy and manage applications on Kubernetes. Prepare for CKA.

**Week 25–28: Kubernetes Fundamentals**
- [ ] Kubernetes architecture: control plane, kubelet, etcd, API server, scheduler
- [ ] Pods, Deployments, Services, ConfigMaps, Secrets, Namespaces
- [ ] kubectl fluency — do everything from the command line
- [ ] KodeKloud CKA course (or Mumshad's Udemy course)
- [ ] Set up k3s on your homelab or VPS
- [ ] **Project:** Deploy a multi-service app to your k3s cluster with proper networking, config, and storage

**Week 29–32: Advanced Kubernetes + CKA Prep**
- [ ] Ingress controllers (nginx-ingress), TLS certs (cert-manager)
- [ ] Helm: packages, values, templating. Create your own Helm chart.
- [ ] RBAC: ServiceAccounts, Roles, RoleBindings
- [ ] Pod security, resource limits, probes (liveness/readiness)
- [ ] Killer.sh simulator sessions — aim for 80%+ before scheduling exam
- [ ] **CKA Exam** — schedule and pass

**Milestone:** CKA certified. You can deploy and manage production-like workloads on Kubernetes.

## Homelab Build

### The "Why" (The KubeCraft Insight Worth Keeping)

Your homelab is not just a learning tool — it's your **portfolio, proof of competence, and
interview differentiator.** When an interviewer asks "have you worked with Kubernetes in production?"
you can say "I run a cluster at home that mirrors production patterns — here's my architecture
diagram, here's my GitOps repo, here's my monitoring dashboard." This is infinitely more convincing
than "I took a course."

### Budget-Optimal Build (Netherlands — EUR 200–350)

**Hardware:**
- 1× used mini PC from Marktplaats (Lenovo M920q, Dell Optiplex Micro, HP EliteDesk Mini)
- i5-8500T or similar, 32GB RAM, 256GB+ NVMe SSD
- Cost: EUR 150–250

**Software:**
- **Proxmox VE** (free) — hypervisor, runs 3–4 VMs
- **3× Ubuntu Server VMs** — Kubernetes nodes
- **k3s** (months 1–3) — easy to start, huge community
- **Talos Linux** (months 4+) — API-driven, immutable OS. Teaches clean K8s/OS separation

**Why Proxmox + VMs vs bare-metal K8s?**
- You can snapshot before risky changes and roll back instantly
- You learn virtualization (VMware/Proxmox concepts are enterprise fundamentals)
- You can simulate multi-node clusters on one machine
- LXC containers for lightweight services (Pi-hole, Nginx, databases)

**Why eventually Talos Linux?**
- KubeCraft uses it for good reason: it strips away the Linux OS layer and forces you to think
  about Kubernetes as the platform, not Linux + K8s duct-taped together
- No SSH, no shell, no package manager — everything via `talosctl` gRPC API
- Immutable filesystem means you cannot "fix things the wrong way"
- Excellent interview signal: "I run Talos on my homelab"

### Beyond the First Machine (Optional)
- Add a second mini PC for true HA
- Dedicated NAS for persistent storage (Longhorn or Rook/Ceph)
- Raspberry Pi for edge/ARM experimentation

### Cloud-Only Alternative (Free Tier)

If you cannot buy hardware:
- **Oracle Cloud Free Tier**: 4 ARM cores, 24GB RAM — enough for a 3-node k3s cluster (run VMs inside for multi-node simulation)
- **GCP**: $300 credit for 90 days
- **Azure**: $200 credit
- **AWS Free Tier**: t2.micro for 12 months (underpowered for multi-node K8s)

**Downside:** You miss the physical hardware experience (cables, boot issues, hardware failure
simulation) — but you can still learn K8s.

## Portfolio Projects

### What Hiring Managers Actually Look At

They do **not** want another todo app. They want **systems thinking and automation.**

A standout DevOps portfolio shows:
1. You can automate a problem end-to-end (code commit → production + monitoring)
2. You understand infrastructure as code (no UI-clicked resources)
3. You think about failure modes (monitoring, alerting, rollback)
4. You document architectural decisions (README with diagrams, tradeoff explanations)

### Recommended Projects (Do 5–6)

**Beginner (do all 3):**
1. **Server performance stats script** (Bash/Python) — outputs CPU, memory, disk, network, top processes
2. **Log archive tool** (CLI) — compresses and rotates logs, sends summary email
3. **CI/CD pipeline for a static site** — GitHub Actions → build → test → deploy to VPS

**Intermediate (do 2–3):**
4. **Docker Compose microservice app** — frontend + API + database with proper networking
5. **Terraform AWS environment** — VPC, subnets, EC2, load balancer, RDS, DNS, all from `terraform apply`
6. **K3s multi-service deployment** — Helm chart, Ingress, TLS, persistent storage, Prometheus monitoring

**Capstone (do 1):**
7. **Full platform automation** — Terraform provisions infra → Ansible configures → GitHub Actions CI/CD → ArgoCD deploys to K8s → Prometheus+Grafana monitors → Loki aggregates logs. Fully documented with architecture diagram, deployment GIF, and decisions log.

### What Makes a Portfolio Repo Stand Out


## Interview Preparation

### Primary Resource

**[devops-exercises](https://github.com/bregman-arie/devops-exercises)** — 82.2k GitHub stars,
2,624 questions. The definitive free resource. Organized by topic: Linux, networking, Docker,
Kubernetes, CI/CD, cloud, monitoring, Python, Go, and system design scenarios.

### What Junior DevOps Interviews Test

**Not LeetCode.** Nobody asks you to invert a binary tree. They test:

1. **Linux command line proficiency** — "A user says 'my system is slow.' Walk me through your
   diagnostic process from `top` to `strace` to flame graphs."

2. **Live scripting exercise** — "Here's a directory of JSON log files. Write a script to extract
   error counts by service and output a summary." (Python or Bash, your choice.)

3. **Kubernetes troubleshooting** — "A pod is stuck in CrashLoopBackOff. Walk me through your
   diagnosis: `kubectl describe pod`, `kubectl logs`, check resource limits, check liveness probe."

4. **System design scenarios** — "Design a CI/CD pipeline for a microservice app" or "Design
   monitoring for a 5-service application."

5. **Practical exercise** — SSH into a broken VM and fix it in 30 minutes (misconfigured services,
   disk full, permissions wrong, process not starting).

### Key Conceptual Questions (Know These Cold)

**Linux:**
- What happens from power-on to login prompt? (Boot process)
- What happens when you run `ls -l`? (fork, exec, readdir — trace through system calls)
- Explain file permissions. What do 777, 644, 750 mean? What is setuid? Sticky bit?
- A process is writing to an unknown file. How do you find and kill it? (lsof + kill)
- "Disk is full" but `df` shows 20% used. What's happening? (Deleted files held open, inode exhaustion)

**Networking:**
- What happens when you type a URL into a browser? (DNS, TCP handshake, TLS, HTTP)
- Explain the OSI model. What does each layer do?
- What is a subnet mask? What is CIDR?

**Docker:**
- Container vs VM — what's the actual difference?
- Dockerfile layers — why does order matter for caching?
- CMD vs ENTRYPOINT — when do you use which?

**Kubernetes:**
- Explain what happens from `kubectl apply -f deployment.yaml` to the pod actually running
- Pod in Pending state — what do you check?
- Users cannot reach the application — walk through the diagnostic flow

**CI/CD:**
- CI vs CD vs Continuous Deployment — explain the difference
- Design a pipeline from PR to production. Include rollback strategy.

### System Design Practice

Practice designing these on a whiteboard/document and explaining aloud:
1. CI/CD pipeline for a microservice app (source → lint → test → build → scan → deploy → monitor)
2. Monitoring stack for a 5-service app (Prometheus + Grafana + Loki + AlertManager)
3. Infrastructure for a web app on AWS (VPC, subnets, load balancer, auto-scaling, RDS, S3)
4. Kubernetes cluster architecture (control plane, worker nodes, networking, storage)

## Remote Work Reality Check

### The Hard Truth

**Junior DevOps remote roles essentially do not exist.** Across 4 major job boards (WeWorkRemotely,
RemoteOK, LinkedIn Remote, Indeed Remote), zero junior remote DevOps roles were found. Every remote
DevOps role requires 3–5+ years of experience.

**Why?** Junior DevOps engineers need mentorship, pair programming, hands-on infrastructure access,
and the safety net of senior engineers nearby when they accidentally break production. Remote
companies cannot provide this safely for juniors.

### Market-by-Market Reality

| Market | Junior Viability | Why |
|--------|-----------------|-----|
| **Netherlands (on-site/hybrid)** | ★★★★★ Realistic | Active traineeships, talent shortage, omscholing programs |
| **Netherlands (remote)** | ★★ Very rare | Most Dutch companies do hybrid, not fully remote for juniors |
| **EU Remote** | ★ Almost non-existent | Zero junior remote DevOps roles found |
| **UK Remote** | ★ Not viable | Post-Brexit visa sponsorship needed; juniors don't meet salary threshold (GBP 38,700) |
| **US Remote** | ☆ Not viable | Timezone (6–9 hrs behind), no work visa path for remote, contractor route requires senior level |
| **Global Remote** | ☆ Not viable | Remote-first companies (GitLab, Hotjar, Zapier) only hire mid-senior DevOps |

### The Realistic Path to Remote

1. Get 2–3 years of on-site/hybrid DevOps experience in the Netherlands first
2. Build a strong portfolio and reputation in the community
3. Then pursue EU-remote or hybrid roles with remote flexibility

The remote DevOps market is for experienced engineers who can work autonomously on production
infrastructure. It opens up at mid-level (3+ years), not entry-level.

## Coding Requirements by Entry Route

### Python

**For junior DevOps:** Script-level Python is sufficient. 50–200 line automation scripts, CLI
tools, API clients. You need: file I/O, JSON/YAML parsing, HTTP requests (requests library), boto3
(AWS SDK), basic error handling, list/dict comprehensions. You do NOT need: web frameworks
(Flask/Django), async programming, complex OOP patterns, type hint mastery.

**For Linux SysAdmin → DevOps:** Script-level Python + Bash. Same level.

**For Cloud Support:** Script-level Python (automation, not application development).

### Bash

**Expected: basic-to-intermediate.** Considered table stakes, not a differentiator. File operations,
piping, redirection, loops, conditionals, grep/sed/awk/find/xargs. Writing CI/CD pipeline scripts.
You do NOT need 500-line Bash scripts (that's when you rewrite in Python).

### Go

**Not required for junior roles in the Netherlands.** Zero junior Dutch DevOps listings mentioned
Go. Go becomes relevant when writing Kubernetes operators, custom controllers, or Terraform
providers — all mid/senior work. Learn it after you're on the job if your team uses it.

### What Employers Actually Test

- Linux command line proficiency (always)
- Live scripting exercise ("parse this log file") in Python or Bash
- Basic coding concepts (not LeetCode): loops, conditionals, data structures for logs/configs
- Tool familiarity ("how would you set up a CI/CD pipeline?")
- Practical debugging ("SSH into this broken server and fix it")

**They test practical automation skills, not CS fundamentals.**
