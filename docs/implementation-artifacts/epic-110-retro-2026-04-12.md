## Patterns and Themes

**Theme 1 — Recurring (x7 epics): Action items requiring inter-session prep don't get done**

Seven consecutive epics, sub-20% follow-through on action items that require work between epic close and the first story open. Action items executed inline during story work get done. Action items requiring a deliberate gap do not. This is no longer a habit problem — it's a structural problem. The `finish-story/SKILL.md` does not enforce an inter-epic prep phase. Until it does, this pattern will continue.

**Theme 2 — New: Design review HIGH findings merged unfixed (process failure)**

S03's design review flagged two HIGH issues: 24×24px touch targets on drag handle and remove button (minimum 44×44px), and missing `type="button"` on both buttons. Both were documented in the review report. Neither was fixed. The story was marked reviewed and the PR was merged. The review process generated output that was not acted upon. This is not a review quality failure — the finding was correct. It is a merge discipline failure: HIGH findings should block merge until resolved or explicitly deferred to a known issue.

**Theme 3 — New: Series progress corrupts under active filters (design defect shipped)**

`getBooksBySeries()` scopes series membership to currently-filtered books rather than the full book set. This means series progress counts — "completed/total" — are meaningless when any filter is active. A series with 5 books shows "1 of 1" under a status filter matching only 1 book. The user sees false completion data. This shipped and is in production.

**Theme 4 — New: `filteredBookIds` prop defeats useMemo (performance regression shipped)**

`filteredBooks.map(b => b.id)` creates a new array reference on every Library.tsx render. `LocalSeriesView` depends on this prop for its `useMemo`. The memo recomputes on every render. The optimization has zero effect. `Library.tsx` re-renders frequently (keyboard shortcuts, dialog state, ABS sync). This is a compounding performance issue that shipped.

**Theme 5 — Positive: Schema versioning discipline held**

Three stories, three sequential Dexie migrations (v44, v45, v46), each in a single commit with the schema checkpoint test updated simultaneously. Zero migration conflicts. Zero mystery regressions. The pattern established in E109 applied cleanly across all three stories without rediscovery.

**Theme 6 — Positive: DnD infrastructure reuse paid off**

`@dnd-kit/*` was pre-existing. S03 applied it to a new domain without any dependency evaluation, bundle cost, or API learning curve. S03 passed in one review round — the only story this epic to do so. Reusing established infrastructure is the fastest path to a clean first-pass review.

## Lessons Learned

### 1. Design review HIGH findings must block merge — not just document

A design review HIGH finding that is documented but not fixed before merge provides zero quality improvement. The merge gate is the point of control. Until HIGH findings are either fixed or explicitly filed as known issues with `schedule_for`, they should prevent the story from being marked reviewed. The mechanism to enforce this is a pre-review checklist item: "All HIGH design review findings addressed or filed as KI with schedule." (E110-S03, touch targets and missing `type="button"`)

### 2. `getBooksBySeries()` should operate on the unfiltered book set

Series membership and series progress reflect the user's full library — not a filtered subset. Applying active UI filters to the series grouping selector produces misleading completion data: a 5-book series shows "1 of 1" when only 1 book matches the filter. The fix: `getBooksBySeries()` reads from `get().books`, not `getFilteredBooks()`. If per-filter series scoping is wanted, it should be an explicit option, not the default. (E110-S02)

### 3. `Array.map()` in JSX props defeats downstream `useMemo`

Passing `filteredBooks.map(b => b.id)` as a prop creates a new array reference on every render. Any `useMemo` in the child that depends on this prop recomputes every render — the memo is functionally inert. The fix: stabilize the array with `useMemo` at the callsite, or restructure to pass a stable reference. This is a silent performance regression — the code appears optimized (the useMemo is there) but achieves nothing. Pattern: when memoizing based on a prop, verify the prop reference is stable at its origin. (E110-S02, `LocalSeriesView`)

### 4. DnD infrastructure pre-existing = zero-friction reuse

S03 used `@dnd-kit/core`, `@dnd-kit/sortable`, and `@dnd-kit/utilities` — all pre-existing. The story passed in one review round. The contrast with S02 (4 rounds) is largely explained by infrastructure novelty: S02 introduced a new component pattern (LocalSeriesCard with accordion behavior, useId-based ARIA pairs, custom grouping selector) while S03 applied known patterns. When planning epics, stories that reuse pre-established infrastructure are faster and cleaner than stories that introduce it. (E110-S03)

### 5. Pre-epic action items that require stopping before S01 will not happen without a structural gate

This has now appeared in seven consecutive epics. Items executed inline during story work: completed. Items requiring deliberate pre-story time: not completed. The `finish-story` skill does not enforce an inter-epic gap. Checklists in the retrospective document are insufficient. The only fix is a structural one: either `finish-story` pops a "pre-S01 prep" checklist before the retro closes, or the action items are converted to chore commits before the retro is filed. (Recurring pattern, E104–E110)
