## What Went Well

**Bob (Scrum Master):** "The story itself — the work inside E105 that was actually scoped — went well. Pedro, what are you genuinely proud of from this epic?"

**Pedro (Project Lead):** "The CSP discovery is the thing I'm most proud of. When `courses.spec.ts` started failing, my first instinct was that the Courses component had changed. I spent time looking at component diffs. But the test failure was a blank page — no React at all. That's not a component bug, that's a boot failure. Thinking through what could prevent the entire React app from mounting led me to the Vite dev server headers, then the CSP meta tag, then specifically to `upgrade-insecure-requests` in the context of WebKit's interpretation of that directive. It's a subtle, non-obvious root cause. And the fix — a Vite plugin that strips those directives only in test mode — is clean and won't silently recur."

**Charlie (Senior Dev):** "That's infrastructure-level thinking. Most developers would have patched the test to work around the symptom. The CSP plugin fix means any future test that navigates on WebKit will just work, without every developer needing to understand why."

**Alice (Product Owner):** "The WelcomeWizard centralization in `local-storage-fixture.ts` is the same category of fix — instead of patching each failing test individually, you identified the root cause was that tests bypassing `navigateAndWait` didn't get the dismissal. The centralization means the pattern holds going forward."

**Quinn (QA Engineer):** "E105-S02 also caught a real data drift bug in `dashboard-reordering.spec.ts` — the `DEFAULT_SECTION_ORDER` constant had 7 items but the source had grown to 10. ## Lessons Learned

### 1. Retro action items must be in the developer's line of sight when a story starts, not in a separate document

Two epics in a row with zero action item follow-through is a workflow failure, not a memory failure. The action item list needs to be surfaced at story creation time — either by the `/start-story` skill or by embedding the top carry-forwards into the new story's checklist. Writing action items in a retrospective document and hoping they get read is not a process.

### 2. When a root cause fix is at the infrastructure layer, it's worth the extra effort

`testModeCspPlugin` prevents an entire class of WebKit test failures from reoccurring. `local-storage-fixture.ts` centralization prevents WelcomeWizard from blocking any test that uses the fixture. These fixes cost more upfront than patching individual tests but pay off on every future test added. When an E2E fix involves touching more than two test files for the same underlying issue, ask: can this be fixed in the fixture or the dev server instead?

### 3. Discovered scope reduction is still a deliverable — document it clearly

When an investigation shows that 4 of 5 planned fixes are already done, that's a valid outcome. But the story file, story title, and AC list must be updated to reflect what actually happened. A story that says "fix KI-016 through KI-020" but only fixed KI-017 is a permanent misleading artifact. Discovery that reduces scope should be documented as prominently as discovery that increases it.

### 4. Coverage threshold reductions must be paired with a tracked follow-up issue

Lowering a coverage threshold is a form of accepting technical debt. Accepted technical debt without a corresponding tracking entry in `known-issues.yaml` is invisible debt. The pragmatic call to lower 70% → 55% was correct. Not filing a KI for the gap was the error. These two actions must always be coupled.

### 5. Scope conservatism in test code is the wrong trade-off

Choosing to hard-code `DEFAULT_SECTION_ORDER` (rather than importing it) to avoid "touching more files" preserves fragility. Test code is not less important than production code — it's the mechanism by which production code is trusted. The refactor to import from source was estimated as risky but is actually a 3-line change with zero production impact. When the right fix is clear, the conservative path in test code should be the correct fix, not the minimal patch.
