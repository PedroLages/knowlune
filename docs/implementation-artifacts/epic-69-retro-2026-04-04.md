## What Went Well

### 1. Service Isolation Was the Right Call

Bob (Scrum Master): "Every story built cleanly on top of `storageEstimate.ts`. S02 and S03 modified it but never had to touch component rendering concerns. That boundary held."

Charlie (Senior Dev): "It paid off immediately in S02. Extracting `PerCourseStorageTable` was clean because the data layer was already stable from S01. No entanglement."

Pedro (Project Lead): "That's the pattern I want to see repeated. Compute in the service, render in the component. When they're mixed, everything becomes harder to test."

**Pattern to carry forward:** Define the estimation/data service interface before any component work. Components consume; services compute.

---

### 2. `Promise.allSettled` for Estimation Resilience

Charlie (Senior Dev): "One of my favorite decisions in this epic. `Promise.allSettled` means a corrupted Dexie table doesn't tank the entire storage overview. We get partial data instead of a blank card."

Dana (QA Engineer): "I specifically tried to trigger cascade failures in QA. Couldn't. Each category degrades to zero bytes independently. That's solid."

Elena (Junior Dev): "I want to use this pattern everywhere. `Promise.all` feels risky now."

Bob (Scrum Master): "Good instinct. For any parallel async work where individual failures are non-fatal, `Promise.allSettled` should be the default, not the exception."

**Pattern to carry forward:** `Promise.allSettled` for any parallel estimation, aggregation, or non-critical fetch work. `Promise.all` only when all results are required.

---

### 3. Component Extraction in S02

Alice (Product Owner): "S02 delivering `PerCourseStorageTable` as a standalone extracted component rather than inline in `StorageManagement.tsx` -- that was a judgment call, right?"

Charlie (Senior Dev): "Yes, and the right one. The table has its own sort state, pagination state, and action dialogs. Keeping it inline would have made StorageManagement unreadable and the table untestable in isolation."

Pedro (Project Lead): "14 unit tests on PerCourseStorageTable alone. You can't get that coverage on a buried inline component."

**Pattern to carry forward:** When a component manages its own non-trivial state (sort, pagination, dialog open/close), extract it. The rule of thumb: if it needs more than 3 unit tests, it deserves its own file.

---

### 4. Dexie Transactional Deletes in S03

Charlie (Senior Dev): "S03's atomic delete implementation was clean. Dexie `db.transaction('rw', [...tables], callback)` wraps every cleanup operation. If one table's delete fails, the entire operation rolls back."

Dana (QA Engineer): "I ran the rollback scenario. It works. No partial deletes, no data integrity issues. The unit tests for transaction rollback are thorough."

Bob (Scrum Master): "Orphan detection is also worth calling out -- finding embeddings with no matching course and deleting only those is exactly the right scalpel vs sledgehammer approach."

**Pattern to carry forward:** For any multi-table write operation, use `db.transaction('rw', [...allTablesAffected], ...)`. Never write to multiple tables outside a transaction.

---

### 5. Multi-Round Review Discipline for Accessibility

Bob (Scrum Master): "S01 took three review rounds. That's more than we'd like, but each round had a clear, specific finding that was fixed."

Dana (QA Engineer): "Round 1 caught the missing `bg-warning-soft` token and the empty-state short-circuit that blocked banners entirely. Round 2 caught the contrast failure -- 4.19:1 on the critical banner. Round 3 verified the fix."

Pedro (Project Lead): "The lesson I take from this: the first implementation of anything with custom warning/error UI should get an early contrast check, not wait for the full design review. Those token combinations are easy to get wrong."

## What Could Be Improved

### 1. Silent Error on Manual Refresh (BUG-002)

Bob (Scrum Master): "This one bothers me. The `toast.error` promised in the AC for refresh failure never fires. The error is swallowed in `storageQuotaMonitor.ts` and the component silently falls back to 'API unavailable'."

Pedro (Project Lead): "From a user perspective, I clicked Refresh to verify my cleanup worked. Nothing happened. No spinner stopping, no error message. I thought the button was broken. That's a bad experience when you're stressed about storage."

Charlie (Senior Dev): "The root cause is a layered catch. `storageQuotaMonitor.ts` catches the `navigator.storage.estimate()` rejection and returns null. `getStorageOverview` accepts null and proceeds. The component never sees an error -- it just sees 'no API available'."

Dana (QA Engineer): "The fix is clear: either let the error propagate through the service, or move the catch to the component and let `handleRefresh` decide how to surface it."

**Action item:** Fix BUG-002 in next sprint. Propagate estimation errors from `storageQuotaMonitor.ts` so `handleRefresh` can fire `toast.error`. Add `vi.mock('sonner')` assertion in the test.

---

### 2. Category Mapping is a Maintenance Trap

Charlie (Senior Dev): "The 6-category mapping in `storageEstimate.ts` covers the primary tables, but approximately 15 tables are entirely invisible to users. Study sessions, quiz attempts, AI usage events -- those could be the actual storage hogs for a power user."

Pedro (Project Lead): "A user sees 'Courses: 450 KB, Notes: 120 KB' -- that adds up to 600 KB. But the browser says total usage is 4 MB. Where's the other 3.4 MB? We don't tell them."

Alice (Product Owner): "That's a trust problem. The storage card is supposed to help them understand their data. If most of it is labeled 'Other' without explanation, we've failed that goal."

Elena (Junior Dev): "The dev notes even called this out -- 'this mapping needs updating when new tables are added.' We shipped a known maintenance trap."

Bob (Scrum Master): "The short-term fix is an 'Other data' row in the legend that explicitly accounts for uncategorized bytes. The long-term fix is a more comprehensive mapping."

**Action item:** Add 'Other data' catch-all category to `storageEstimate.ts` covering the remaining Dexie tables. Ensure the chart total matches the browser-reported total.

### 4. ArrayBuffer Estimation Gap

Dana (QA Engineer): "The `instanceof Blob` check handles blob fields correctly. But `ArrayBuffer` and typed arrays get `JSON.stringify`'d as `{0:0,1:0,...}` -- potentially 5-10x inflated estimates for any binary table."

Charlie (Senior Dev): "We don't know which tables, if any, currently store ArrayBuffers. But the `youtubeVideoCache` and `screenshots` tables are candidates. This is a latent bug that could embarrass us when someone has video cache enabled."

**Action item:** Add `instanceof ArrayBuffer` and `ArrayBuffer.isView()` checks in `estimateTableSize` alongside the existing `instanceof Blob` guard.

## Pedro's Reflection

Pedro (Project Lead): "Looking back on E69, I'm genuinely proud of the storage management UX we shipped. The overview card is clean, the category breakdown is useful, and the cleanup section in S03 gives users real control over their data for the first time.

What went especially well: the component extraction pattern in S02. `PerCourseStorageTable` as a standalone component made S02 much easier to test and reason about than if we'd kept it inline. That discipline -- extract when state is non-trivial -- is something I want us to apply automatically in future epics.

What I'd do differently: I'd catch the silent-error regression earlier. BUG-002 was visible in the exploratory QA report and should have been fixed before ship, not deferred. When an AC explicitly promises `toast.error` on failure and the test suite doesn't verify it, that's a signal we missed a test. The lesson is: for every user-facing error path in an AC, there should be a corresponding test that verifies the visible output (toast, message, state change) -- not just that the operation doesn't throw.

The category mapping incompleteness is a longer-term concern. We built a storage dashboard that understates actual storage by potentially 70-80% for power users. That undermines the trust the feature is supposed to build. Adding the 'Other data' category is non-negotiable before we call this epic truly done."

## Key Patterns Identified for Engineering Patterns Doc

Bob (Scrum Master): "Charlie, let's make sure these make it into `engineering-patterns.md`."

Charlie (Senior Dev): "On it. Three clear additions from this epic:"

1. **Storage Estimation Service Pattern**: Isolate all Dexie estimation logic in a dedicated service (`storageEstimate.ts`). Components consume typed output only. Use `Promise.allSettled` for parallel table reads. Handle `ArrayBuffer`, `Blob`, and typed array fields distinctly in size estimation.

2. **Component Extraction Threshold**: Extract a component when it manages its own non-trivial state (sort, pagination, dialogs) AND needs more than 3 unit tests. Inline components with complex state are untestable.

3. **Dexie Transaction Discipline**: All multi-table write operations (cleanup, migration, bulk delete) must use `db.transaction('rw', [...tables], callback)`. Never write to multiple tables outside a transaction.
