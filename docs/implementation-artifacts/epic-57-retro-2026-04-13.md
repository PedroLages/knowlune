## Successes and Strengths

### 1. llmErrorMapper.ts — Proactive Deduplication Before Spread

`llmErrorMapper.ts` was extracted before three stories (S01, S02, S05) independently implemented LLM error classification. This is the same pattern established in E56 (knowledgeTierUtils.ts). The habit is forming: when a utility will be consumed by 2+ stories in the same epic, extract it in S01 or as a pre-story shared util task.

### 2. Two BLOCKERs, Zero Rounds Beyond R2

S05 had two BLOCKERs in R1 — schema.test.ts not updated for v50, and an unused TypeScript import. Both are mechanical issues, not architectural ones. Both were fixed and closed in R2. No story in E57 needed a third review round.

### 3. Pre-Analysis in Story Specs Prevented Mid-Implementation Surprises

The EC-HIGH edge cases documented in S02, S03, S04, and S05 specs before implementation — AbortController memory leaks, blob corruption, frustration false positives on short valid answers (`"ok"`, `"42"`, `"TCP"`), Embedding type `noteId`-as-primary-key limitation — were all anticipated and mitigated before a line of code was written. This is the target standard for complex AI stories.

### 4. ARIA Conflict Found and Fixed

The radiogroup+radio+aria-checked conflict is a subtle accessibility bug that accumulates as silent tech debt without targeted ARIA auditing. It was caught and fixed during E57 rather than carried forward.

### 5. Complete Phase 1-2 Tutoring Pipeline Shipped

Streaming chat, conversation persistence, Socratic pedagogy with a 5-level hint ladder, and RAG-grounded answers from both notes and video transcripts — all shipped as a single epic. The architecture spec provided enough pre-decision context that no story required an architecture pivot mid-implementation.

## Challenges and Growth Areas

| Epic | Finding | Class |

The issue is not knowledge — developers know to update the schema. The mental model of "update schema" does not automatically include "update schema test file." The fix is to name the file explicitly in the checklist: `src/db/schema.test.ts`.

### 2. useTutor @/db Mock Added in R2, Not R1

The useTutor unit tests needed a `vi.mock('@/db')` registration that was not in the initial test setup. It was caught in R2 rather than during development. Hooks that depend on Dexie need the `@/db` mock as boilerplate in the hook test template.

### 3. Unused TypeScript Import Slipped Pre-Commit

An unused TS import reached review. ESLint `no-unused-vars` may not be configured to catch TypeScript imports specifically. Light gap — verify and patch.

## Lessons Learned

1. **Pre-analysis in story specs prevents mid-implementation surprises.** EC-HIGH edge cases documented before implementation started (S02 AbortController, S03 blob corruption, S04 frustration false positives, S05 Embedding type) were all mitigated cleanly. This is the target standard for all AI and data-layer stories.

2. **Dedup before spread, not after.** Extract shared utilities when 2+ stories in the same epic will need the same logic. llmErrorMapper.ts (E57) and knowledgeTierUtils.ts (E56) are the pattern. Codify: if a util is needed by ≥2 stories, extract it in S01 or as a named pre-story task.

3. **Dexie migrations require explicit `schema.test.ts` update as part of DoD.** Two consecutive epics, two BLOCKERs, same file. The checklist item must name the file: "After any Dexie version bump, update `src/db/schema.test.ts` to include new tables and run `npm test -- schema`."

4. **Custom ARIA patterns need a named axe scan step.** radiogroup+radio+aria-checked conflict was caught this epic. Any story with custom selection UI, radio groups, focus traps, or ARIA role composition should include `axe` scan as a named pre-review step.

5. **Hook test templates need a Dexie mock registry.** Any hook that reads from Dexie needs `vi.mock('@/db')` from the start. Add it to the hook test template boilerplate.

6. **0% first-pass rate, 0 stories needing 3 rounds.** These two stats together mean the review system is working correctly — issues are found and resolved efficiently. The pre-review checklist is the layer that needs tightening, not the review agents.
