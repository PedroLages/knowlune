## Successes and Strengths

### Bob (Scrum Master)

**Bob:** Six stories in one day, all passing review in round 1. That is the fastest epic delivery we have seen. Pedro, what enabled that throughput?

### Pedro (Project Lead)

**Pedro:** Three architectural decisions made this epic almost frictionless, and one of them was made in a previous epic.

**1. The scan/persist separation in S01 was the single most impactful decision of the entire epic.** Every subsequent story — the wizard (S02-S03), AI suggestions (S04), editing (S05), video reordering (S06) — benefited from the clean boundary between "reading the filesystem" and "writing to IndexedDB." The wizard calls `scanCourseFolder()`, gets back a `ScannedCourse` object, displays it for editing, and then calls `persistScannedCourse()` with whatever overrides the user made. No story after S01 needed to touch the data layer for import. This is textbook "invest in the right abstraction early" — S01 took the most careful thought per line of code, but it paid dividends five times over.

The overrides pattern specifically deserves credit. Instead of requiring callers to reconstruct the entire course object, `persistScannedCourse(scanned, overrides?)` accepts a partial overlay. This meant S02 could pass `{ name: editedName }`, S03 could add `{ tags, coverImageHandle }`, and S04 could add `{ description }` — all without changing the persist function signature. That kind of extensibility is rare in a first attempt.

**2. Reusing established patterns from previous epics eliminated ramp-up time.** S04 built on the Ollama courseTagger infrastructure from E22-S04 — the structured output pattern, the response parsing fallback chain, the fire-and-forget architecture were all in place. S06 reused @dnd-kit from E21-S06 with the exact same sensor configuration (PointerSensor + KeyboardSensor). S05's optimistic UI + rollback pattern is the same one used in the quiz store and progress store. None of these stories required inventing new patterns, just applying proven ones to new contexts.

**3. The unit test coverage was genuinely comprehensive without being wasteful.** 160+ unit tests across the epic, zero E2E tests. That sounds like a testing gap, but it is not — the File System Access API cannot be automated in Playwright, and every dialog interaction, state transition, and edge case (duplicate tags, empty names, AI unavailable, persist failure rollback) is covered by unit tests. The decision to skip E2E was correct and well-documented in every story file. The 91% trace pass rate confirms the coverage is real.

### Charlie (Senior Dev)

**Charlie:** The wizard's step-by-step architecture in S02-S03 is clean because it is state-machine-like without actually using a state machine library. Each step is just a number, the back button decrements, the next button increments, and `resetWizard()` sets everything to initial values. The explicit `resetWizard` function that covers all state variables prevented the "stale state on re-open" bug that plagues most wizard implementations. Simple tools, well applied.

### Dana (QA Engineer)

**Dana:** All 6 stories passing review in round 1 is a first for any epic. The fix commits show that issues were found but they were minor — formatting, mock completeness, defensive patterns. No architecture-level rework was needed in any story. The S01 foundation was solid enough that subsequent stories did not introduce structural problems.

## Challenges and Growth Areas

### Bob (Scrum Master)

**Bob:** What friction did you encounter, and what concerns remain?

### Pedro (Project Lead)

**Pedro:** Three concerns, one of which I consider significant enough to flag for the next epic.

**1. Zero E2E tests for the entire epic is a real coverage gap, not just a documented waiver.** I signed off on it because the File System Access API is genuinely untestable in Playwright — there is no way to programmatically trigger `showDirectoryPicker()` or mock its return value in a browser context. But the consequence is that the entire wizard flow — folder selection, step navigation, scanning, AI suggestions, cover image selection, and final import — has never been tested as an integrated user journey. The unit tests cover each piece in isolation, but integration gaps between steps are not exercised. If we ever add a testable import path (e.g., drag-and-drop a folder, or URL-based import), we should backfill E2E coverage for the wizard flow.

**2. The 21 pre-existing test failures (12 in Courses.test.tsx, 9 in autoAnalysis.test.ts) were carried through all 6 stories without action.** These failures are related to IndexedDB mock issues and module mock incompleteness — they are not caused by E24 code. But they create noise in every test run, make it harder to spot genuine regressions, and erode confidence in the test suite. The autoAnalysis failures specifically are in code that E24-S04 extended (the AI tagging pipeline), which means we are building new features on top of a module whose existing tests are broken. This is technical debt that compounds.

**3. The SSRF bypass in courseTagger was not addressed despite E22's critical action item.** E24-S04 added `generateCourseTags` and `generateCourseDescription` to courseTagger, which makes direct HTTP requests to Ollama without going through the proxy with SSRF validation. The E22 retrospective flagged this as critical action item #1. I chose to proceed with S04 because the wizard is a local-first feature (the user themselves enters the Ollama URL), but this is still a gap in the security model. It should be fixed before any feature that makes Ollama URLs configurable by external input.

### Charlie (Senior Dev)

**Charlie:** The Object URL lifecycle management in S03 was more complex than it should have been. `URL.createObjectURL()` for cover image previews requires explicit `revokeObjectURL()` cleanup, but the interaction between React's re-render cycle and the async scanning means URLs can be created and revoked in unexpected orders. The inline comment explaining the intentional dependency array exclusion was the only thing that prevented a code review escalation. We should consider a `useObjectUrl` hook that encapsulates the create/revoke lifecycle.

### Dana (QA Engineer)

**Dana:** The S03 BLOCKER (broken mocks when adding `isImageFile` to `fileSystem.ts`) is a pattern we have seen before. Any time a new export is added to a module that is mocked in existing tests, those mocks break. The lesson was documented in S03, but we do not have an automated way to catch this. An ESLint rule or a test convention (always mock with `...jest.requireActual()`) would prevent this class of failure.

## Key Insights and Learnings

1. **Scan/persist separation is the canonical pattern for wizard-based data entry.** The investment in E24-S01 to split `importCourseFromFolder` into `scanCourseFolder()` (read-only) and `persistScannedCourse(scanned, overrides?)` (write) paid dividends across all 5 subsequent stories. The `overrides` parameter pattern allowed each story to extend the import data without modifying the persist function. This pattern should be reused for any future multi-step data entry flow (e.g., author creation wizard, learning path builder).

2. **All-round-1 reviews are achievable when stories build on proven patterns.** E24 reused Ollama structured output (E22), @dnd-kit drag-and-drop (E21), optimistic UI + rollback (E12/E18), and Zustand persistence (E21). When the architectural patterns are established and well-documented, implementation stories become assembly rather than invention, and code review finds only minor issues.

3. **File System Access API creates a testing blind spot that should be explicitly tracked.** The entire import wizard flow has zero E2E coverage because `showDirectoryPicker()` cannot be mocked in Playwright. This is the largest untested user journey in the application. Any future refactoring of the wizard should include a plan for testable alternatives (e.g., a test-only file input fallback).

4. **Pre-existing test failures that persist across epics become invisible and compound.** The 21 failures in autoAnalysis.test.ts and Courses.test.tsx were present before E24, carried through all 6 stories, and never fixed. Each epic that ignores them normalizes the noise. A dedicated "test health" story should fix or remove these tests.

5. **Module mock brittleness is a recurring problem when adding exports to shared modules.** S03's BLOCKER was caused by adding `isImageFile` to `fileSystem.ts` while existing tests mocked the module without the new export. This is the third time this pattern has caused test failures (E22-S04 had a similar issue). The fix is either a convention (`vi.mock` with `...vi.importActual()`) or an ESLint rule that detects incomplete module mocks.

### Team Agreements (Effective Immediately)

1. **Scan/persist separation for wizard flows** — Any multi-step data entry should split into a read-only scan phase (returns a preview object) and a write phase (accepts overrides). Do not combine filesystem/network reads with persistence in the same function.
2. **`vi.importActual()` in module mocks** — When mocking a module, always spread `vi.importActual()` as the base to prevent breakage when new exports are added. Only override the specific functions under test.
3. **Pre-existing test failures are not acceptable carry-forward** — If unit test failures exist at epic start, fix them as the first story or as a pre-epic chore. Do not normalize red test suites.
4. **Document untested journeys explicitly** — When a user journey cannot be E2E tested due to API limitations (File System Access, Web Audio, etc.), log it in known-issues.yaml with conditions under which backfill becomes possible.
