## Cross-Cutting Patterns

**Pattern 1 — Zustand selector anti-patterns are a recurring BLOCKER risk.**
Calling `state.computedFn()` inside a selector where `computedFn()` returns a new array/object reference triggers infinite re-renders. The fix is always to pre-compute as reactive state. This has appeared before and is in `project-context.md` — but spec authors are not pattern-checking against it before recommending integration approaches.

**Pattern 2 — E2E tests written ≠ E2E tests validated.**
S03 had 4 E2E tests written but not run before review submission. "Written" and "passing" are different states. The pre-review gate needs to enforce passing, not just presence.

**Pattern 3 — Spec errors propagate as review rounds.**
S03's 4-round review traces directly to a spec-level anti-pattern recommendation. Every review round beyond the spec error was downstream amplification of that single authoring mistake.

**Pattern 4 — GLM false positive rate elevated for Tailwind v4 responsive patterns.**
The `lg:flex` / `sm:grid` responsive override cascade is not understood by the GLM agent. Generates false positives on every story using responsive Tailwind breakpoint overrides.

---

## Lessons Learned

**Lesson 1 — Zustand computed selectors are a first-class architectural rule, not a dev note.**
Calling any function that creates a new object/array reference inside `useStore(state => state.fn())` is a BLOCKER. Document this as an explicit rule in `.claude/rules/styling.md` or a dedicated Zustand patterns rule file. Link to the `computeScores()` pre-computation pattern from E71-S03 as the canonical fix.

**Lesson 2 — The pre-review E2E gate must require locally passing tests, not just written tests.**
Story E2E tests must pass locally before submitting for review. "I wrote the tests" is not the same as "the tests pass." Add this as an explicit pre-review checklist item in the story workflow.

**Lesson 3 — Spec authoring requires Zustand integration pattern review.**
When a story spec recommends a Zustand integration pattern involving computed getters, the spec author must verify it against the selector rules in `project-context.md` before finalizing. A spec pointing toward an anti-pattern costs multiple review rounds downstream.
