## What Went Well

### 1. CSS-Class-Driven Mode Switching — Excellent Architecture

Charlie (Senior Dev): "The decision to use `.reading-mode` as a CSS class on `<html>` — same as `useContentDensity` and theater mode — was the right call. We got layout changes for free without any React rendering overhead. No component tree re-renders when you hide the sidebar."

Bob (Scrum Master): "And it made the implementation predictable. The pattern was already established, so S01 went faster than expected."

Elena (Junior Dev): "It also made testing straightforward — you just check for the class on `document.documentElement`. No complicated DOM assertions."

**Pattern to extract:** CSS class on `<html>` for mode switching. Zero JavaScript-driven layout changes, no re-render cascades. Combine with `[data-theater-hide]` attribute for element-specific hiding.

---

### 2. The `useAutoHide` Hook — Clean Encapsulation

Dana (QA Engineer): "The auto-hide pattern for the toolbar was one I was worried about — those timing behaviors usually end up fragile in tests. But because it was encapsulated in its own hook with a `disabled` flag, testing was clear."

Charlie (Senior Dev): "The key insight was disabling auto-hide entirely when `prefers-reduced-motion` is active. No conditional timeout — just a disabled state. That made the hook's contract explicit."

**Pattern to extract:** Encapsulate timing-based UI behaviors in a dedicated hook with a `disabled` prop. Accessibility preferences (reduced motion) disable the behavior, not modify it.

---

### 3. Focus Trap via `inert` Attribute — Modern, Standards-Compliant

Charlie (Senior Dev): "Using `inert` for focus trapping instead of the traditional `tabindex="-1"` loop was the right call. It's semantically correct, blocks both keyboard and pointer events, and screen readers handle it properly."

Bob (Scrum Master): "The implementation in `useFocusMode` — traversing `document.body.children` and applying inert to everything except the portal and target — is clean and reversible."

Alice (Product Owner): "Worth noting this required Epic 30 (Accessibility) as a foundation. The `inert` attribute is well-supported now but wasn't 2 years ago."

**Pattern to extract:** Use `inert` attribute for focus containment instead of custom focus trap loops. Pair with `data-focus-target` attribute on the elevated component.

---

### 4. Notification Piercing Architecture — Well-Designed Seam

Charlie (Senior Dev): "The `notificationPiercing.ts` module was the most elegant part of S04. The split between `CRITICAL_TYPES` (pierce overlay) and `NON_CRITICAL_TYPES` (suppress + queue) is explicit and maintainable. No ad-hoc `if` chains scattered through notification handlers."

Dana (QA Engineer): "The `MAX_INDIVIDUAL_TOASTS = 5` cap on flushing was a nice defensive touch. We won't spam users with 20 queued notifications after a long focus session."

Bob (Scrum Master): "Pedro, this pattern of separating critical from non-critical notification handling — did it feel natural to implement?"

Pedro (Project Lead): "It did once we named the types explicitly. The hardest part was deciding where the boundary is — `review-due` as critical, `streak-milestone` as non-critical. But once those decisions were in the constants, the rest followed."

**Pattern to extract:** Define explicit sets of critical vs. non-critical event types as `ReadonlySet<T>` constants. Notification routing logic flows from those constants, not from inline conditions.

---

### 5. Settings Integration Discipline

Alice (Product Owner): "S05 caught something important — the auto-activation settings were implemented in the store but had no Settings UI until S05. That's a pattern failure we've had before."

Charlie (Senior Dev): "Every configurable behavior must have a Settings UI before the story is closed. Otherwise users can't discover it, and the 'You can disable this in Settings' tooltip becomes a lie."

**Action item:** Add checklist item to story template: 'If this story adds a user-configurable behavior, verify Settings UI exists before marking done.'

## What Was Hard

### 1. Hook Decoupling — Custom Events Over Props

Charlie (Senior Dev): "S03's big lesson was about hook coupling. Originally `useFocusMode` was going to receive `exitReadingMode` as a prop, but that creates a tight dependency between two hooks that are both consumed in `Layout.tsx`."

Elena (Junior Dev): "The `window.dispatchEvent(new CustomEvent('exit-reading-mode'))` solution was surprising at first — it felt like reaching outside React. But it's actually the right tool when two hooks need to communicate without sharing a parent component's state."

Bob (Scrum Master): "This is a recurring pattern in this codebase. The `settingsUpdated` custom event does the same thing. We should document this pattern explicitly."

**Lesson captured:** When two React hooks need to communicate but neither is the parent of the other, use `window.dispatchEvent(new CustomEvent(...))`. Each hook listens independently. This is the project's established inter-hook communication pattern.

---

### 2. Overlay Exit Animation Needs Separate Mount State

Charlie (Senior Dev): "The overlay animation bug was subtle. If you immediately return `null` when `isFocusMode` is false, the exit animation never plays — the element is removed from the DOM before the animation starts."

Elena (Junior Dev): "The fix — `isMounted` state that stays `true` for 200ms after `isFocusMode` goes false — is a pattern that comes up any time you have exit animations on conditionally-rendered components."

Dana (QA Engineer): "And the `shouldReduceMotion` guard that skips the 200ms delay entirely is the right accessibility handling."

**Pattern to extract:** For exit animations on conditional elements, use a separate `isMounted` state: `isFocusMode=false` → start animation → wait 200ms → `isMounted=false` → unmount. Always guard with `shouldReduceMotion` to bypass delay.

---

### 3. Missing E2E Spec for S03

Dana (QA Engineer): "S03 has no E2E spec. That's the biggest quality gap in the epic. Focus mode's most critical behaviors — Escape key exit, overlay click, focus trap, Escape during quiz — are all untested at the E2E level."

Bob (Scrum Master): "The traceability report flagged this as our main CONCERNS item. We need `tests/e2e/story-e65-s03.spec.ts` before E65 can be considered fully validated."

**Action item:** Create `tests/e2e/story-e65-s03.spec.ts` covering Escape exit (AC2), overlay click deactivation (AC3), and `inert` attribute focus trap assertion (AC5). Schedule as chore commit on `main`.

---

### 4. Singleton State Desync on Navigation

Charlie (Senior Dev): "The `focusModeState.ts` module-level flag `_isFocusModeActive` is the one reliability risk we shipped. If a user navigates away while focus mode is active via the browser back button, the React state resets but the singleton stays `true`. Non-critical notifications get suppressed until page reload."

Bob (Scrum Master): "The NFR report flagged this as HIGH severity. R-03."

Pedro (Project Lead): "Fix: add `window.addEventListener('popstate', ...)` in `useFocusMode` cleanup, or hook into React Router's `useEffect` on pathname change to call `setFocusModeInactive()`."

**Action item:** Fix R-03 — add route change handler in `useFocusMode` to call `setFocusModeInactive()` on navigation away from focus-mode-capable pages.

---

### 5. `useCallback` Ordering — Define Before Use

Elena (Junior Dev): "S05 taught me that `useCallback` declarations must come before the `useEffect` that references them. Otherwise the dependency array is wrong and you need eslint-disable comments."

Charlie (Senior Dev): "This is a general React hooks rule: define all callbacks and memoized values first, then the effects that depend on them. Top-to-bottom dependency order."

**Pattern to extract:** In React components/hooks: `useState` → `useCallback`/`useMemo` → `useEffect`. Never reference a callback in an effect defined above it.

## Lessons Learned (Extractable Patterns)

| # | Pattern | Source | Applies To |
|---|---------|--------|-----------|
| L-01 | CSS class on `<html>` for mode switching | S01 | Any full-page layout mode |
| L-02 | `useAutoHide(delayMs, disabled)` hook for timing UI | S02 | Toolbars, tooltips, banners |
| L-03 | `inert` attribute for focus containment | S03 | Modals, overlays, focus traps |
| L-04 | Separate `isMounted` state for exit animations | S03 | Any conditional element with exit animation |
| L-05 | `CustomEvent` on `window` for inter-hook communication | S03 | Hooks that need to communicate without shared parent |
| L-06 | `ReadonlySet<T>` for explicit type-based routing | S04 | Notification routing, event filtering |
| L-07 | `focus-request` / `focus-release` pair pattern | S04 | Any component that auto-activates a mode |

| L-09 | `useState(lazyInit)` for localStorage values | S02 | Any setting loaded from localStorage at mount |
| L-10 | `useCallback` before `useEffect` ordering rule | S05 | All React hooks |
