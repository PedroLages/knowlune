Figuring out that `vi.fn(() => obj)` doesn't work as a constructor mock but `vi.fn(function(this) { Object.assign(this, ...) })` does — that's a non-obvious pattern that's now captured in the S02 challenges and will apply to every future test that needs to mock `new AudioContext()` or `new WebSocket()`.

Third: the Canvas API limitation insight. Rather than fighting jsdom, documenting 'avatarUpload.ts is capped at 35% — canvas APIs require real browser DOM' is more valuable than fake coverage from mocked canvas. The lesson is clear: some code genuinely can't be unit-tested in jsdom, and that's fine."

**Charlie (Senior Dev):** "The review loop discipline stood out to me. Every story had two review rounds. S01 went 5 issues → fixed. S02 went 10 issues → fixed. S03 went 2 issues → fixed. That's a zero-tolerance loop — nothing was deferred to 'fix later.' Every finding was addressed before merge. Seventeen issues caught and resolved before any of this hit main. **Quinn (QA Engineer):** "The pre-existing test bug fix in S02 is worth calling out. `AudiobookshelfService.test.ts` had a `fetchCollections` test asserting `result.data.length` but the API returns `{ results, total }`. **Alice (Product Owner):** "KI-036 resolved. The threshold is at 60% with a CI gate. That means future code additions that drop coverage below 60% will fail the build. That's a permanent structural improvement — not just a number improvement but a floor that will hold going forward."

**Bob (Scrum Master):** "Pedro, what do you think was the most technically interesting challenge you solved during this epic?"

**Pedro (Project Lead):** "The `vi.mock()` hoisting issue in S03. Multiple test files initially failed because I declared `const mockFn = vi.fn()` at module scope and tried to reference it inside the `vi.mock()` factory. That pattern doesn't work — `vi.mock()` factories are hoisted above all imports and declarations. The fix is `vi.hoisted()`, which explicitly runs before the hoisting. It's counterintuitive, but once you know it you can't forget it.

The second technical challenge was the fake timers + async hooks combination. `useQuizGeneration` uses `setInterval` for health checks. I tried `vi.useFakeTimers()` to control the interval, but combining fake timers with `await waitFor()` caused infinite waits — the async resolution never triggered because the async machinery itself depends on real-time microtask scheduling. The solution was to use real timers and let the hook settle naturally. That's a nuanced interaction between Vitest's fake timer implementation and React Testing Library's async utilities."

## Lessons Learned

### 1. Coverage percentage targets without composition analysis produce unachievable goals

Setting a coverage target (70%) without first categorizing the types of uncovered code (stores/libs/hooks vs. React components) leads to stories that cannot deliver their primary AC. The realistic ceiling for non-component unit tests in Knowlune is ~62-63%. Any target beyond that requires React Testing Library component tests — a different test type requiring different infrastructure, patterns, and scope. Future coverage improvement epics must specify both the target percentage AND the test tier required to achieve it.

### 2. Story specs that name specific files must verify those files exist before story approval

Six non-existent source files were referenced across two story specs in E106 (three lib files in S02, three hook files in S03). Every file-targeting test story should validate target file existence against the filesystem at story creation time using `ls src/[module]/` or extracting exact paths from `coverage-final.json`. Human-guessed module names based on product brief terminology are unreliable. Use the actual coverage report file paths.

### 3. The vi.mock() hoisting boundary requires vi.hoisted() for shared mock state

Mock functions declared at module scope with `const mockFn = vi.fn()` are not available inside `vi.mock()` factory functions — the factories are hoisted above all module-level declarations. The correct pattern is `const mockFn = vi.hoisted(() => vi.fn())`, which runs before the hoisting boundary. This is a silent failure mode: the test file runs but the mock isn't set up correctly, causing downstream test failures that appear to be unrelated to mocking.

### 4. Fake timers and React Testing Library's waitFor() are fundamentally incompatible in certain patterns

When a hook uses `setInterval` or other timer-based async behavior, combining `vi.useFakeTimers()` with `await waitFor()` creates an unresolvable deadlock — `waitFor()` needs real-time microtask scheduling that fake timers suppress. The solution for hooks with timer-based async effects is to use real timers and let `waitFor()` retry naturally. Only use fake timers for hooks where the timer behavior itself is the thing being asserted (e.g., auto-hide delay), not for hooks where timers are implementation details of async work.

### 5. Retro action items that survive three consecutive epics need a different format, not more emphasis

The pattern of carrying action items across E104 → E105 → E106 with zero completion on items 4-10 indicates the retro action table format is broken for these items. They survive because they're consistently underscoped (labeled 'chore commits' but actually requiring component understanding and test writing), lack urgency in the retro format, and compete with concrete epic deliverables for attention. Items that cannot be done 'at the start of the next epic' three times in a row should be filed as tracked stories with acceptance criteria, not retro action items.
