## Successes and Strengths

### 1. Single-Gateway Pattern Made Migration Surgical

The decision to confine all ts-fsrs imports to `spacedRepetition.ts` (single-gateway pattern) meant every downstream consumer — stores, UI components, metrics — only needed field name updates, not algorithm logic changes. This architectural boundary turned a potentially risky algorithm swap into a series of mechanical field renames. The stores and UI never directly touched ts-fsrs, which kept each story focused and reviewable.

### 2. Story Sequencing Was Excellent

The 8-story sequence (types -> algorithm -> migration -> stores -> consumers -> UI -> unit tests -> E2E tests) created a clean dependency chain where each story resolved its predecessor's expected type errors. TypeScript's compiler served as a built-in progress tracker — the error count dropped predictably with each merge. This made it safe to work on stories that intentionally left type errors for downstream stories to resolve.

### 3. Deterministic Test Architecture from Day One

Establishing `fsrsTest` (FSRS instance with `enable_fuzz: false`) in E59-S02 and requiring `FIXED_DATE` constants across all test files prevented any flakiness. The ESLint `test-patterns/deterministic-time` rule caught attempts to use `Date.now()` at save time. Zero test flakiness across the entire epic.

### 4. Migration Safety with Schema Checkpoint Testing

The `schema-checkpoint.test.ts` pattern — comparing migration-built vs checkpoint-built database schemas — caught schema drift before it could reach production. This pattern, established in earlier epics, paid dividends again in E59-S03.

### 5. Low Review Friction

50% of stories (4/8) passed code review on the first round. The remaining stories had only MEDIUM or LOW findings — zero BLOCKERs across the entire epic. The automation pipeline (ESLint design-token enforcement, format checks, type checks) caught the easy issues before review, letting reviewers focus on architecture and correctness.

## Challenges and Growth Areas

### 1. Migration Version Number Mismatch

The epic planning documents referred to "Dexie v29 schema migration" (including the sprint-status slug `59-3-dexie-v29-schema-migration-with-data-transformation`), but the actual migration was v31 because v29 and v30 were already consumed by E43-S06 (notifications) and E89-S01 (courses-drop) respectively. This caused momentary confusion during implementation and left a misleading slug in sprint-status.

**Root cause:** Epic planning used the schema version known at planning time without accounting for concurrent epic work that would consume intermediate versions.

**Lesson:** Schema version numbers should be assigned at implementation time (just-in-time allocation), not during epic planning. The story description should reference "next available Dexie version" rather than a specific number.

### 2. FSRS Short-Term Scheduling Surprised Test Assertions

FSRS's `enable_short_term: true` configuration means new cards rated Hard/Good stay in the Learning phase with `scheduled_days: 0` and minute-based intervals. Only Easy jumps directly to Review state. This broke initial test assertions that expected `scheduled_days` to increase monotonically with rating quality (as SM-2 did). Tests had to be rewritten to compare `due` dates instead of `scheduled_days`.

**Root cause:** Insufficient study of FSRS short-term scheduling behavior before writing test expectations.

**Lesson:** When migrating to a new algorithm, write exploratory tests first to understand actual output values before writing assertion-heavy test suites.

### 3. Non-Deterministic Date.now() in Migration Callback

Round 1 code review on E59-S03 caught `Date.now()` usage inside the Dexie upgrade callback. Since the migration iterates over all records, different records could get different timestamps. The fix was capturing a single `migrationNow` timestamp at callback entry.

**Root cause:** Standard ESLint `deterministic-time` rule only covers test files, not migration code.

**Lesson:** Time-sensitive operations in migration callbacks deserve the same determinism rigor as test code.

### 4. FSRS Retention Calibration Differs from SM-2

Test data calibrated for SM-2 retention thresholds (e.g., "low retention" at interval=3 with 14 days elapsed) produced different retention percentages under FSRS power-law decay. Stability=3 with 14 days gives ~69% retention (above the "at-risk" threshold), while SM-2's linear decay would have produced much lower values. This required recalibrating all retention test fixtures.

**Root cause:** SM-2 and FSRS use fundamentally different forgetting curves (exponential vs power-law). Test fixtures assumed the old curve's outputs.

**Lesson:** When changing a mathematical model, identify all downstream consumers that embed assumptions about the model's output values — not just its field names.

### 5. Missing E59-S06 Story File

E59-S06 (Review UI — Again Button and Keyboard Shortcuts) has code review reports in `docs/reviews/` but no story file in `docs/implementation-artifacts/stories/`. This is likely because it was implemented and reviewed without creating the formal story file first, or the file was created in the wrong location.

## Key Insights and Patterns

### Pattern 1: Type-Driven Migration Sequencing

Using TypeScript's compiler as the migration progress tracker was highly effective. By intentionally introducing type errors in E59-S01 (changing interfaces) and resolving them story-by-story, each story had a clear, mechanically-verifiable definition of done: resolve all type errors in your scope, leave the rest for downstream stories. This pattern is reusable for any cross-cutting interface change.

### Pattern 2: Single-Gateway for Third-Party Libraries

Confining `ts-fsrs` imports to a single file (`spacedRepetition.ts`) meant the migration's blast radius was fully contained. If ts-fsrs ever needs to be replaced or upgraded, only one file changes. This validates the pattern established in the architecture doc and should be enforced for all third-party algorithm libraries.

### Pattern 3: Factory Pattern for Test Migration

The `makeCardState()`, `makeReview()`, and `createFlashcard()` factory functions made it easy to construct FSRS-compatible test data with sensible defaults and per-test overrides. When the underlying data model changes, only the factory needs updating — individual test assertions stay stable. This pattern matured significantly during this epic.

### Pattern 4: Schema Checkpoint Validation

The `schema-checkpoint.test.ts` approach — comparing a fresh database created with the latest schema definition against one created through all migrations — catches schema drift. This pattern prevented a potential data corruption bug in E59-S03.

## Key Takeaways

1. **Type-driven migration sequencing works.** Using TypeScript's compiler as a progress tracker — introducing interface changes early and resolving errors story-by-story — made an 8-story cross-cutting migration predictable and reviewable.

2. **Single-gateway patterns pay for themselves during migrations.** Confining ts-fsrs to one file meant 35+ consumer files only needed field renames, not algorithm understanding.

3. **Algorithm migrations require recalibrating all downstream assumptions.** Changing the forgetting curve formula meant every retention threshold, test fixture, and due-date assertion needed recalibration — not just the field names. "Change the model, audit the consumers" is the takeaway.
