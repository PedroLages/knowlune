## Challenges and Growth Areas

1. **Accessibility gaps recurred across stories** — `aria-describedby` missing (S02, S03), `aria-live` missing on character counter (S03), redundant `role="group"` (S02). Each new component reintroduces the same ARIA mistakes despite the pattern being documented. **Root cause:** No component-level accessibility checklist applied during implementation. Design review catches these, but they could be prevented earlier.

2. **PCM scoring dual-path caused test confusion (E14-S02)** — `isCorrectAnswer()` uses exact set-match (boolean), while `calculatePointsForQuestion()` uses PCM formula (fractional points). Initial unit tests assumed all-or-nothing for multiple-select. **Root cause:** The dual-path design is non-obvious; tests were written from intuition rather than ACs.

3. **Unsafe type casts in scoring (E14-S02)** — `as string[]` cast without `Array.isArray` runtime guard. Code review caught it, but it's a runtime safety gap with malformed quiz data. **Root cause:** TypeScript casts bypass runtime checks; the developer trusted compile-time types for runtime data.

4. **Double-fire bug in FillInBlank debounce (E14-S03)** — `handleBlur` and debounce timer both calling `onChange`, plus debounce firing on prop-driven value sync during navigation. **Root cause:** Debounce pattern using `useEffect` + `setTimeout` + `isInitialMount` ref is 3 mechanisms for one concern. A shared `useDebouncedCallback` hook would encapsulate this complexity.

5. **"Tests that don't test what you think" pattern (E14-S04)** — Contrast E2E test initially checked if background color was non-empty, not the actual WCAG ratio. Would pass for any non-transparent background regardless of contrast. **Root cause:** E2E assertion precision — testing presence vs testing correctness.

## Key Insights and Learnings

1. **Pattern compounding has a compounding return curve** — Epic 14's single-day completion (4 stories) vs Epic 12's multi-day effort (6 stories) shows that investment in reusable patterns (quiz components, factories, design tokens, review gates) pays exponential returns. Each new epic is faster than the last.

2. **Accessibility is the one quality dimension that resists pattern compounding** — Unlike styling (design tokens enforce), testing (factories provide), and architecture (component patterns guide), accessibility requires per-component attention to ARIA relationships. A component-level checklist is the right intervention.

3. **"Write tests from ACs, not intuition"** — When scoring has dual paths (boolean correctness vs fractional points), intuition misleads. The AC formula is the contract; test expectations must reference it explicitly. This applies to any domain where the "obvious" behavior differs from the specified one.

4. **Existing code audit is now a proven practice** — Two consecutive epics where it saved real effort (E13: 50% of stories found pre-existing code; E14: 50% of stories). The step should be permanent in story planning.

5. **Programmatic quality verification > manual review** — E14-S04's WCAG contrast test using canvas pixel readback is more reliable than visual inspection and catches issues automated review can't. This pattern should extend to other measurable quality attributes.

6. **Retro follow-through is proportional to commitment specificity** — Epic 13 committed to 8 items, achieved 1/8. That retro then committed to "automatable only." Epic 14 shows 5/8 follow-through. The meta-insight: commit to fewer, more specific items.
