## Successes and Strengths

### Bob (Scrum Master)

**Bob:** Five stories in three days with 96 tests — that is a significant output. Pedro, what specifically worked in this epic?

### Pedro (Project Lead)

**Pedro:** Three things stand out, and they are connected.

**1. Zero new dependencies was a deliberate architectural choice that paid off enormously.** The initial plan assumed we would need `ollama-ai-provider` or similar community packages. Research showed they were fragmented and stale. Instead, we used `createOpenAI` from `@ai-sdk/openai` (already in the project) with a custom `baseURL` pointing at Ollama's OpenAI-compatible `/v1/` endpoint for the proxy path. For the client-side tagging, we used native `fetch` against Ollama's `/api/chat`. Zero new packages means zero supply chain risk, zero version conflicts, and zero bundle growth beyond our own code. The lesson: when integrating with an API that has an OpenAI-compatible layer, start with the adapter you already have. Only add a new dependency if the existing adapter genuinely cannot do the job.

**2. Schema-enforced JSON output (`format` parameter) was the single best technical decision in the epic.** The courseTagger uses Ollama's grammar-constrained generation to guarantee structurally valid JSON. This turned the traditional LLM integration problem (parsing flaky text output) into a solved problem — 99%+ structural validity. The 4-level fallback chain in `parseTagResponse` is defensive code that almost never fires. Compared to the prompt-and-pray approach, schema enforcement eliminated an entire class of runtime errors. This pattern should be the default for any local LLM integration that needs structured output.

**3. The fire-and-forget pattern for AI tagging kept the import flow fast and resilient.** Imports complete immediately. Tagging runs in the background. If Ollama is down, the import succeeds without tags — no error, no blocking, no degraded experience for the core workflow. The user sees "AI tagging..." on the card for 0.5-2 seconds, then tags appear. This is the right UX for any AI-augmented feature: the AI is additive, never a gate. The race condition between cloud and Ollama tag writers (both reading stale snapshots) was caught in review and fixed by reading fresh tags from IDB before merging — a real concurrency bug that would have caused data loss.

### Charlie (Senior Dev)

**Charlie:** The SSRF protection story is worth highlighting. When the proxy was first implemented, it forwarded to any URL the user provided. The code review in S01 round 1 flagged this, and by round 2 we had `isAllowedOllamaUrl()` blocking loopback, link-local, and `0.0.0.0` addresses with 14 dedicated security tests. The server-side test infrastructure (`server/__tests__/` with a custom Vitest config) was created specifically for this — it did not exist before Epic 22. That infrastructure is now available for all future server-side code. Security done right: caught in review, fixed with tests, established as a pattern.

### Dana (QA Engineer)

**Dana:** The testarch trace at 88% coverage with 96 tests across 7 files is strong for an epic that deliberately skipped E2E on 4 of 5 stories. The decision to skip E2E for external-dependency features was well-documented and compensated with deep unit tests. ## Challenges and Growth Areas

### Bob (Scrum Master)

**Bob:** What friction points emerged?

### Pedro (Project Lead)

**Pedro:** Four issues, and the first one is the most concerning because it was identified by the adversarial review and is a genuine architectural inconsistency.

**1. The courseTagger bypasses the proxy SSRF protection entirely, and this was not caught until the post-epic adversarial review.** The proxy has `isAllowedOllamaUrl()` with 14 security tests. The courseTagger uses native `fetch()` directly to the user-configured Ollama URL from the browser — it never touches the proxy. This means the most data-intensive AI endpoint (tagging, which fires on every course import) has zero server-side SSRF validation. The per-story code reviews did not catch this because each story was reviewed in isolation: S01 review validated the proxy security, S04 review validated the tagging logic, but nobody checked whether the tagging used the proxy. The adversarial review, which looks at cross-story interactions, caught it immediately. The lesson: cross-story architectural consistency is invisible to per-story reviews. We need either an architecture review gate at the epic level or a pattern where reviewers explicitly check "does this new endpoint use the established security pattern?"

**2. No concurrency control on batch tagging was the second critical adversarial finding.** If a user imports 20 courses, 20 concurrent `fetch()` calls fire simultaneously. Ollama processes sequentially (single model in VRAM), so 19 of those requests queue up and likely timeout at 10 seconds. The story doc even acknowledges "10-40s for batch of 20 (sequential)" but the implementation is concurrent. The fix is straightforward — `p-limit` with concurrency of 1 — but the gap between the documented plan (sequential) and the implementation (concurrent) went undetected through 1 review round on S04. This is a failure to implement what was designed, similar to the AC drift pattern we identified in E27 but applied to technical design rather than acceptance criteria.

**3. The stale closure in `handleModelSelectCallback` is a React pattern bug that is hard to catch in review.** The `useCallback` wraps a function that closes over state, but the dependency array does not include the function itself. In specific interaction sequences (change model, then change URL, then change model again), the callback uses stale state. This was the third critical adversarial finding. The pattern is well-documented in React literature but manifests subtly — the `useCallback` appears correct at a glance, and the bug only surfaces under specific timing. We should add a linting rule or at minimum a team convention: never wrap a naked function in `useCallback` — either define the logic inline in the callback or use `useRef` to keep the latest version.

**4. E22-S01 required 3 review rounds — the most in the epic — and each round surfaced progressively deeper issues.** Round 1: SSRF, orphaned JSDoc, missing `// silent-catch-ok`. Round 2: `0.0.0.0` bypass, missing server tests. Round 3: verification pass. This iterative deepening is the review process working correctly, but it also means the first submission was significantly short of quality. S01 was the foundational story — it introduced the proxy, the client, the factory, the configuration UI. Foundational stories carry more risk and should get extra pre-review scrutiny. A self-review checklist specifically for "stories that introduce new infrastructure" would have caught at least the SSRF and missing server tests before the first review round.

## Key Insights and Learnings

1. **Schema-enforced JSON output eliminates the LLM parsing problem.** Ollama's `format` parameter with a JSON schema guarantees structural validity (99%+). The fallback parsing chain is defensive-only. This pattern should be the default for any structured output from local models — it converts a flaky, retry-heavy integration into a reliable one.

2. **Per-story reviews miss cross-story architectural inconsistencies.** The courseTagger bypassing the proxy SSRF protection was invisible to per-story reviews because the proxy (S01) and the tagger (S04) were reviewed independently. Only the epic-level adversarial review caught the bypass. For security-sensitive patterns (SSRF validation, authentication, authorization), we need an explicit "does this new code use the established pattern?" check during per-story reviews.

3. **Fire-and-forget is the right pattern for AI-augmented features, but it needs concurrency control.** The pattern of "complete the primary action immediately, run AI in the background, show results when ready" is excellent UX. But when that background AI work can be triggered N times concurrently (batch imports), a simple queue (`p-limit(1)`) is required. The pattern is: fire-and-forget for single items, queue-and-forget for batches.

4. **Foundational stories need higher pre-review scrutiny than feature stories.** E22-S01 introduced the proxy, the client class, the factory integration, and the configuration UI — all new infrastructure. It took 3 review rounds to reach quality. Stories that introduce new infrastructure should have a heavier pre-review checklist: security model documented, server-side tests included, integration points verified.

5. **Zero new dependencies is achievable for LLM integration.** Ollama's OpenAI-compatible endpoints mean `createOpenAI` (already in the project) works as a proxy adapter. Native `fetch` handles direct API calls. The entire epic was 5 stories, 96 tests, and zero new npm packages. This should be the default approach: exhaust existing adapters before introducing new dependencies.

### Team Agreements (Effective Immediately)

1. **Schema-enforced output for local LLM integration** — When requesting structured data from Ollama (or compatible servers), always use the `format` parameter with a JSON schema. Never rely on prompt instructions alone for structured output.

3. **Fire-and-forget with concurrency control** — AI-augmented features should use fire-and-forget for single items and a serialized queue (p-limit) for batch operations. Never fire N concurrent requests to a single-model inference server.
4. **Close the test path alias** — The `@test/` path alias has been carried for 4 epics with zero action. Formally close as wont-fix. If it becomes painful in the future, reopen with a concrete story.

**Overall Assessment:** Epic 22 is functionally complete (5/5 stories, 100%) and delivers a fully working Ollama integration with zero new dependencies. The technical patterns established — schema-enforced JSON output, fire-and-forget AI augmentation, `createOpenAI` adapter reuse, server-side SSRF validation — are genuinely strong and reusable. The 96 tests across 7 files provide solid regression protection for the business logic layer.

However, the post-epic validation exposed real architectural gaps that per-story reviews missed. The courseTagger bypassing the proxy SSRF protection is the most significant: it means the security investment in S01 (14 tests, loopback/link-local blocking) does not cover the primary AI workload. The batch concurrency issue and stale closure are correctness bugs that will manifest under specific (but realistic) usage patterns.

The defining tension of Epic 22 is between **per-story quality** (which was high — 31 issues caught and fixed across 8 review rounds) and **cross-story architectural consistency** (which was not validated until the adversarial review). This is a structural gap in the review process, not a failure of individual reviews. The fix is a lightweight cross-story pattern check added to the code review template.

The central learning from Epic 22: **Local LLM integration with zero new dependencies is achievable and preferable. Schema-enforced output and fire-and-forget patterns make AI features reliable and non-blocking. But security patterns must be validated across story boundaries — per-story reviews are necessary but not sufficient for architectural consistency.**
