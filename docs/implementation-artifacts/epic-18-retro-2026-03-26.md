## Successes and Strengths

### Bob (Scrum Master)

**Bob:** Eleven stories in roughly four days. That is the highest throughput we have achieved in any epic. Pedro, what made that possible?

### Pedro (Project Lead)

**Pedro:** Three things worked well in this epic, and they are worth preserving.

**1. The accessibility stories (S01-S04) benefited enormously from the pre-existing epics 14-18 batch review.** The batch review from 2026-03-14 had already identified the structural issues -- nested `<main>` landmarks, missing `aria-valuetext`, hardcoded colors, wrong breakpoints. When we finally implemented E18-S04 (contrast/touch targets), the developer had a concrete checklist of what to fix. That batch review was done months before implementation and it still saved time because the findings were specific and actionable. The lesson: adversarial pre-reviews of story plans pay off at implementation time even when there is a long gap between review and implementation.

**2. The `useAriaLiveAnnouncer` hook in E18-S02 is the best reusable accessibility primitive we have built.** Zero-width-space deduplication, auto-clear after 5 seconds, timer cleanup on unmount, `isFirstRender` guard -- this is a pattern I want to see adopted in every component that needs screen reader announcements. The hook is 30 lines, testable in isolation, and the API is trivial (`announce(message)`). The fact that Round 1 found a correctness bug (triple-message dedup) and Round 2 verified it was fixed cleanly is exactly how the two-round review process should work for reusable primitives.

**3. The integration stories (S05-S11) were architecturally independent, which let us ship them in any order.** QuizPerformanceCard, QuizAnalyticsDashboard, QuizBadge, QuizPreferencesForm, and QuizExportCard all operate on different pages and different data models. No integration story depended on another integration story at runtime. This is the same pattern that worked in Epic 20, and it is now a proven approach: when scoping integration work, pick platform touchpoints that are architecturally isolated from each other.

### Charlie (Senior Dev)

**Charlie:** The axe-core integration in E18-S04 is a significant quality improvement. Running automated WCAG scans in E2E tests with `['wcag2a', 'wcag2aa', 'wcag21a', 'wcag21aa']` tag coverage catches violations that manual review cannot. The transition-disable pattern for dark mode testing (`* { transition: none !important; }`) eliminates the timing uncertainty we had in earlier dark mode tests. This should be the standard pattern for any page-level accessibility test.

### Dana (QA Engineer)

**Dana:** E18-S05's fire-and-forget pattern for streak logging is the right architecture. The isolated try/catch around `logStudyAction` in `useQuizStore.ts` correctly protects quiz submission from streak failures. The unit tests cover the happy path, failure isolation, and the study log's date deduplication. The only gap is the missing E2E test for AC4 (failure isolation), which is covered at the unit level. For a store-level concern with no user-visible UI, unit coverage is sufficient.

## Challenges and Growth Areas

### Bob (Scrum Master)

**Bob:** What caused friction or wasted time?

### Pedro (Project Lead)

**Pedro:** Three friction points, ranked by severity.

**1. The adversarial review found that 4 BLOCKER/HIGH code review findings shipped unfixed across E18-S04, E18-S09, and E18-S10.** This is the most concerning finding of the epic. The per-story code reviews did their job -- they identified the CSV formula injection, the focus indicator contrast failures, and the `saveQuizPreferences` silent data loss. But the stories were marked `done` anyway. The review process identifies bugs; the shipping process ignores them. We need a hard gate: if a code review identifies a BLOCKER, the story cannot move to `done` until the BLOCKER is either fixed or explicitly deferred with a known-issues entry. Right now, "pass with issues" is treated the same as "pass clean," and that defeats the purpose of the review.

**2. Pre-existing test failures were investigated 11 times instead of once.** Every code review in this 11-story epic documented the same three pre-existing issues: MyClass test failures, onboarding modal blocking E2E tests, and 197 ESLint warnings. At 5-10 minutes per investigation per story, that is 55-110 minutes of pure waste across the epic. The fix has been proposed since E27 (register in `known-issues.yaml`) and has never been implemented. This is now the third consecutive retrospective flagging this waste. At some point, the cost of not fixing it exceeds the cost of fixing it, and we passed that point two epics ago.

**3. E18-S03's nested `<main>` landmark violation should have been caught before review.** In an accessibility epic, rendering multiple `<main>` elements is a fundamental HTML spec violation. The developer converted `<div>` wrappers to `<main>` in four components without checking whether the parent `Layout.tsx` already provides a `<main>`. This is not an obscure edge case -- it is the first thing any accessibility audit checks. The review caught it and the fix was trivial (change to `<section>`), but the fact that it made it to review at all suggests that the developer was applying the semantic HTML changes mechanically rather than understanding the landmark model.

### Charlie (Senior Dev)

**Charlie:** The E18-S06 code review found nested interactive elements (`<button>` wrapping `<Link>`) in `QuizPerformanceCard`. This is the third time we have seen this pattern (E20-S01 had a similar issue with `aria-disabled` on plain `<div>`). The pattern keeps recurring because there is no ESLint rule for it. `jsx-a11y/no-nested-interactive` would catch this at save-time, and we should add it.

### Dana (QA Engineer)

**Dana:** E18-S08 duplicated the IndexedDB seeding logic instead of using the shared `seedQuizzes()` and `seedQuizAttempts()` helpers from `tests/support/helpers/indexeddb-seed.ts`. The ESLint rule `test-patterns/use-seeding-helpers` exists precisely to prevent this. The rule either did not fire (the test file may not match the rule's pattern) or was ignored. We need to verify the rule is active for all test files, not just files matching `tests/**/*.spec.ts`.

## Key Insights and Learnings

1. **Code review findings that ship unfixed are worse than no review at all.** They create a false sense of quality -- the review was done, findings were documented, but the bugs remain. The adversarial review found 4 BLOCKER/HIGH findings that were explicitly identified in per-story reviews and shipped anyway. The review process needs a hard gate: BLOCKERs block shipping, period.

2. **Large epics (11 stories) amplify pre-existing issue investigation costs linearly.** With N stories, every pre-existing test failure is investigated N times. Epic 18's 11-story scope turned 3 pre-existing issues into 33 investigation events. The solution is not smaller epics (though that helps) -- it is registering known issues once so they are never re-investigated.

3. **Accessibility remediation requires understanding the accessibility model, not just mechanical HTML changes.** E18-S03's nested `<main>` and E18-S04's unfixed `ring-ring` contrast failures both resulted from applying changes without understanding the underlying WCAG success criteria. A `<main>` landmark conversion is not "replace div with main" -- it is "ensure exactly one main landmark per page." A contrast audit is not "run axe-core on some pages" -- it is "verify every focus indicator on every page meets 3:1."

4. **The `useAriaLiveAnnouncer` hook is the best reusable pattern to emerge from this epic.** Zero-width-space deduplication for screen reader announcements, auto-clear timers, mount-guard refs -- this is a solved pattern that should be adopted wherever dynamic content needs screen reader narration. The two-round review process validated it thoroughly.

5. **Architecturally independent integration stories maximize throughput.** E18-S05 through E18-S11 touch different pages (Overview, Reports, Courses, Settings, export) with no runtime dependencies between them. This is the same lesson from Epic 20, now confirmed in a second epic. Theme-grouped but architecture-isolated stories merge cleanly and can be prioritized independently.

### Team Agreements (Effective Immediately)

1. **BLOCKER findings are ship-blockers.** A story cannot move to `done` if its code review has any unresolved BLOCKER findings. This is non-negotiable. "Pass with BLOCKER issues" is not a pass.
2. **Register pre-existing test failures before starting multi-story epics.** If an epic has 5+ stories, spend 15 minutes registering all known test failures in `known-issues.yaml` before starting Story 1. The amortized time savings across N stories outweighs the upfront cost for any N > 3.
3. **Accessibility stories require landmark model understanding, not mechanical HTML changes.** Before submitting an accessibility story for review, verify: (a) exactly one `<main>` per page, (b) all focus indicators meet 3:1 contrast, (c) no nested interactive elements. These three checks take 5 minutes and would have prevented 3 of E18's 4 BLOCKERs.
