## Challenges and Growth Areas

1. **IDB seeding pattern still catches people (E11-S01)** — Despite being documented since E08-S01, the initial E2E approach for E11-S01 seeded data on `/` then navigated to `/review` via `page.goto()`. This caused 7 tests to fail because Dexie re-initialized without the seeded data. The fix (seed on TARGET page, then `page.reload()`) is proven but keeps being rediscovered. **Root cause:** The pattern isn't enforced — it's only documented.

2. **Stale `now` reference flagged in 3 of 5 stories** — Code reviews for E11-S02, E11-S03, and E11-S05 all flagged that `now` is captured once on mount and never refreshes. For E11-S02 (dashboard) and E11-S05 (interleaved review), this means retention percentages become stale during long sessions. The team consistently chose to document the trade-off rather than fix it. **Pattern:** This is a conscious UX decision — a page reload refreshes `now` — but it should be standardized.

3. **Design review contrast issues (E11-S02)** — Three BLOCKER-level findings: "Fading" badge text contrast 3.63:1, "Weak" badge text contrast 4.05:1, and `text-muted-foreground` on white cards at 3.88:1. All below the 4.5:1 WCAG AA minimum. **Root cause:** The semantic tokens `text-warning` and `text-muted-foreground` don't guarantee sufficient contrast on all backgrounds. Need a contrast verification step for custom badge compositions.

4. **E11-S04 had the most code review findings (9 items, 4 HIGH)** — Issues included split-brain localStorage restoration outside Dexie transactions, non-deterministic `new Date().toISOString()` in exports, silent schema migration skips, and `w-4 h-4` instead of `size-4` (19 instances). This was the most complex story (8 ACs, multi-format export, round-trip import) and shows that data export features accumulate edge cases faster than UI features.

5. **ATDD tests didn't account for empty-data guards (E11-S04)** — Tests written RED before implementation assumed export would always proceed. Implementation added early-return guards for empty data, causing tests to time out. **Lesson:** ATDD tests should account for guard clauses from the start, not just the happy path.

6. **Error mock targeted wrong API (E11-S04)** — AC8 error test mocked `showSaveFilePicker` (File System Access API) but implementation used blob + anchor pattern (`URL.createObjectURL`). Mock had no effect. **Lesson:** When writing ATDD tests before implementation, mock at the lowest common denominator (blob/URL APIs), not the most modern API.

7. **E11-S05 code review found data integrity risk** — `rateInterleavedNote` advances past failed ratings, meaning a failed IDB write silently loses the rating but progresses to the next card. Unlike E11-S01's optimistic-with-rollback pattern, E11-S05 didn't implement rollback. **Inconsistency:** The error handling strategy differs between stories doing the same operation (rating notes).

## Key Insights and Learnings

1. **Pure functions with `now: Date` parameter is the Epic 11 signature pattern** — Used in all 5 stories for testability and determinism. This should be formalized as a project convention: any time-dependent logic MUST accept `now` as a parameter, never call `Date.now()` or `new Date()` internally (except at the call site).

2. **Pattern compounding is real and measurable** — Epic 11's 5-story-in-2-days velocity is directly attributable to reusable patterns established in Epics 1-10: IDB seeding, design tokens, Zustand store patterns, motion/react animations, shadcn/ui components, pure function architecture. Each epic adds patterns; later epics assemble them.

3. **Data export features are complexity magnets** — E11-S04 (Data Export) had 8 ACs, the most code review findings (9), and the most diverse testing challenges (download mocking, empty data guards, schema migration). Multi-format serialization + round-trip import creates a combinatorial explosion of edge cases. Future export features should budget extra review time.

4. **Zustand direct mutation is a valid optimization pattern** — E11-S03 proved that bypassing `set()` for high-frequency counters (interaction tracking) avoids ~100 re-renders per session without sacrificing correctness. Key constraints: values must only be consumed at session end, never rendered live. This needs documentation as an approved exception to the "always use set()" convention.

5. **ATDD test assumptions break when implementation adds guard clauses** — E11-S04's RED tests assumed exports would always proceed, but implementation correctly added empty-data guards. ATDD tests should always include at minimum: (a) happy path, (b) empty/no-data state, (c) error state.

6. **Semantic token contrast is not guaranteed across all compositions** — `text-warning` on `bg-card` at 3.63:1 is below WCAG AA. Tokens guarantee contrast against their "natural" background (e.g., page background) but not arbitrary card surfaces. Need a contrast verification checklist for custom badge/card compositions.

7. **CustomEvent wiring is clean for decoupling store from UI** — E11-S03's `session-quality-calculated` event pattern keeps the store agnostic about which UI component handles the result. This is superior to passing callback props through component trees. Worth standardizing for all store→UI notification patterns.

8. **Jaccard similarity on tags is sufficient for note interleaving** — E11-S05 chose Jaccard similarity on `tags[]` over vector embeddings for topic weighting. Simpler, no AI dependency, and sufficient for the interleaving use case. Validates the "start simple" principle from Epic 9's postmortem.

## Significant Discovery: Rating Error Handling Inconsistency

During story analysis, the team uncovered an inconsistency that affects code quality but not Epic 12:

**The Issue:** E11-S01 (`rateNote`) implements optimistic updates with rollback on IDB failure — the rating is applied to state, persisted to IDB, and rolled back if persistence fails. E11-S05 (`rateInterleavedNote`) advances to the next card regardless of IDB write success, silently losing failed ratings.

**Impact:** Not a blocker for Epic 12, but creates inconsistent user experience within the review system. A learner could rate a note in interleaved mode, see it advance, but the rating is lost if IDB fails.

**Recommended Action:** Harmonize error handling in E11-S05 to match E11-S01's rollback pattern before Epic 12. This is a focused fix (1 function in `useReviewStore.ts`).

**Epic Update Required:** NO — Epic 12 is independent of the review system.
