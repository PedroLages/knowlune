## What Went Well

1. **BookContentService abstraction** — Routes by `source.type`, returns uniform `ArrayBuffer`. BookReader component has zero awareness of local vs. remote source. Textbook abstraction that actually shipped as designed.

2. **OPDS XML parsing** — Native `DOMParser`, no xml2js dependency, discriminated union return types (`FetchEntriesResult`). All 24 unit tests passing from Round 1. No bundle bloat.

3. **Cache API integration** — LRU eviction via `X-Cached-At` header is pragmatic and correct for binary data. AbortController with 30s timeout, `useEffect` cleanup preventing stale state. S03 needed only 2 review rounds despite being technically complex.

4. **Lazy-loaded music-metadata** — Dynamic import for M4B parsing. Zero bundle impact for non-M4B users. Clean integration decision.

5. **Test coverage recovery** — S04 delivered 28 unit tests for M4bParserService. Contrast with E87 where all 6 stories deferred Task 7. Partial recovery of the E87 test debt.

6. **Defensive programming in S03** — UTF-8 base64 encoding (TextEncoder), URL validation before fetch, CORS mention in error messages, structured `RemoteEpubError` with `code` property. Error handling done right.

## What Could Be Improved

1. **Stale closures — recurring pattern (S02 + S04)** — Two separate stories required ref-based fixes for stale closure problems. The S02 fix itself introduced a TDZ (Temporal Dead Zone) crash because `fetchFeed` was referenced in a `useEffect` before its declaration. Round 2 caught a BLOCKER page crash that was a mechanical consequence of the Round 1 fix. Root cause: no shared pattern for callback stability in complex hook graphs.

2. **S02 needed 3 review rounds** — Above our 2-round target. The TDZ crash in Round 2 was avoidable if declaration order was verified after dep array changes. 3 rounds extends the review cycle even in an automated pipeline.

3. **E87 action items missed** — 3 of 4 high-priority E87 carry-forwards were not completed before E88:
   - File API mock utility — not created
   - Audiobook NFR validation suite — not created
   - `useAudiobookRenderer()` extraction hook — not extracted (AudiobookRenderer grew past 400 lines in S04)

4. **Per-book credential storage** — OPDS catalog auth credentials are duplicated per-book in IndexedDB. If a user rotates catalog credentials, previously-added books retain stale auth. Documented as intentional local-first trade-off, but it's real technical debt.

## Lessons Learned

1. **Stale closure fixes require declaration-order verification** — When adding an identifier to a `useEffect` dep array, explicitly verify the identifier is declared before that `useEffect` in the file. JavaScript `const` TDZ makes ordering non-optional.

2. **Shared ref patterns should be extracted** — The `singleFileRef`/`bookRef`/`loadChapterInternalRef` pattern in S04 and the `fetchFeed` stability problem in S02 are both variations of the same problem. A `useStableCallback` hook prevents reinventing the solution each story.

3. **BookContentService abstraction pattern is worth reusing** — Route by discriminated union at the service boundary, return a uniform type, keep the consumer ignorant of the source variant. Applicable pattern beyond just content loading.

4. **Action item follow-through requires explicit pre-conditions** — Carry-forwards get lost when there's no structural check at epic start. The tracking file's "Post-Epic Validation" section exists but a "Prior-Epic Carry-Forwards" section does not.

5. **music-metadata lazy loading is the right model** — Heavy parsing libraries should always be dynamic imports. The pattern from S04 (dynamic import at call site, zero bundle impact for other users) should be the default for any future specialty parsers.

6. **3 review rounds signals a systemic gap** — When a story requires 3 rounds, a Round 1 fix introduced a Round 2 blocker. This indicates the fix was applied without understanding the full local context. Slowing down to trace implications before committing a fix would save a round.
