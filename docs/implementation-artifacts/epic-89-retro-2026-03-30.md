## What Went Well

### 1. Adapter/Factory/Capabilities Pattern
The `CourseAdapter` interface with `LocalCourseAdapter` and `YouTubeCourseAdapter` implementations, created in S02, became the single source of truth for all subsequent stories. The `createCourseAdapter()` factory and `getCapabilities()` method enabled truly source-agnostic components. Every story from S03 onward consumed this pattern without needing to rethink data access.

### 2. 100% First-Pass Review Rate
All 11 stories passed code review in a single round. While 48 issues were found and fixed total, none were severe enough to require a second review round. This reflects good story scoping and clear acceptance criteria.

### 3. Dead Code Removal (~3,448 Lines)
S01 removed ~1,754 lines of dead regular course infrastructure. S11 removed ~1,694 lines of old page components. Combined, this significantly reduced cognitive load and eliminated false-positive grep results.

### 4. Well-Ordered Story Dependencies
The epic's dependency chain (S01 -> S02 -> S03 -> S04-S10 -> S11) flowed logically. Each story had exactly the right foundation from its predecessors. S09 (quiz wiring) was nearly free because S01/S03 had already done the heavy lifting.

### 5. useLiveQuery for Adapter Reactivity
Using `useLiveQuery` in the `useCourseAdapter` hook gave automatic reactivity when Dexie data changed (video reorder, metadata refresh) without manual subscription management.

## What Didn't Go Well

### 1. PlayerSidePanel.tsx Grew to 656 Lines (Spec Target: 200)
The component handles four tabs (Notes, Transcript, AI Summary, Bookmarks), mobile Sheet rendering, and state management for tab switching. While the functionality is correct, the 3.3x overshoot of the spec target indicates extraction discipline broke down during implementation.

### 2. Recurring Review Findings
Three patterns appeared across multiple code reviews:
- **Adapter bypass**: Components checking `course.source` directly instead of using `adapter.getCapabilities()`
- **Missing `.catch()`**: Fire-and-forget async calls without error handling
- **Blob URL leaks**: `URL.createObjectURL()` without corresponding `revokeObjectURL()` in cleanup

### 3. Sprint-Status Diverged from Reality
Three unplanned sub-stories (S12a, S12b, S12c) were created during implementation but never added to sprint-status.yaml. Tracking diverged from actual execution.

### 4. Pre-Existing Unit Test Failures Carried Forward
21 unit test failures across 7 files existed before E89 started and were carried through the entire epic without resolution. This normalizes test failure.

### 5. ~57% Trace Coverage
Many acceptance criteria were validated by build success or manual testing rather than dedicated E2E assertions. This creates regression risk for the adapter layer behavior.

---

## Key Insights

1. **Adapter patterns have compounding returns**: The upfront investment in S02's adapter layer paid dividends in every subsequent story. S09 was nearly free. Future unification work should follow this pattern.

2. **Component extraction needs a hard checkpoint**: When a component crosses 300 lines during active development, stop and extract before continuing. Post-hoc refactoring is riskier and less likely to happen.

3. **Dead code removal has compounding returns**: Beyond the obvious ~3,448 lines removed, the elimination of false-positive grep results, reduced import confusion, and simplified route configuration improve developer velocity on every future story.

4. **Sprint tracking must reflect reality**: Unplanned work should be added to sprint-status immediately when it's created, not retroactively.
