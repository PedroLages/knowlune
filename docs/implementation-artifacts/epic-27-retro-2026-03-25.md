## Successes and Strengths

### Bob (Scrum Master)

**Bob:** Small epic, fast delivery. Pedro, what made this one efficient?

### Pedro (Project Lead)

**Pedro:** Three specific things made Epic 27 remarkably clean.

**1. Story execution order was deliberately out-of-sequence, and it paid off.** E27-S03 (sidebar links) was implemented first even though it logically depends on E27-S01 (URL-aware tabs). This worked because the sidebar changes are structurally independent — they just produce links with `?tab=` query params. By shipping S03 first, when S01 landed, the full circuit (sidebar -> URL -> tab activation) worked immediately without integration surprises. The key insight: if two stories connect via URL contracts rather than code imports, the "dependent" story can often ship first.

**2. E27-S02's scope reduction was a genuine process win, not a shortcut.** The original plan for S02 included Tasks 1 and 3 (URL-aware tabs + unit tests) which duplicated what S01 was doing. After S01 merged to main and S02 rebased, those tasks were already complete. Rather than padding the story or inventing work, we acknowledged the overlap, reduced scope to just the redirect routes and E2E tests, and shipped a tight, focused diff. The code review returned zero story-related findings — the cleanest review in the entire epic. This is what good story decomposition looks like in hindsight: when overlap resolves itself cleanly instead of creating merge hell.

**3. Following an established routing pattern eliminated design decisions.** The `/library` -> `/notes?tab=bookmarks` redirect pattern already existed in `routes.tsx`. E27-S02 copied it verbatim for three new paths. No architectural decisions, no abstraction debates, no custom logic. Pattern reuse at its best — the diff was 6 lines of production code plus tests. This reinforces the value of establishing patterns early and documenting them.

## Challenges and Growth Areas

### Bob (Scrum Master)

**Bob:** What didn't go smoothly?

### Pedro (Project Lead)

**Pedro:** Three friction points, two of which are recurring and one is new.

**1. The welcome wizard blocking E2E tests was a time sink that should have been caught by a shared fixture.** E27-S01 tests failed because the `knowlune-welcome-wizard-v1` localStorage key was not seeded in `addInitScript`. The wizard overlay blocked all tab and button interactions, but `getByText` could find text through the overlay while `getByRole('tab')` could not — making the failure mode subtle. The fix was trivial (one line in `beforeEach`), but diagnosing the root cause took disproportionate time. Every new test file that navigates to a page with the welcome wizard will hit this same issue. We need a shared `seedCommonLocalStorage()` helper that all E2E tests call, or the welcome wizard seed needs to be in the global Playwright setup.

**2. Pre-existing test failures created noise during review.** The code review for E27-S02 identified 4 pre-existing test failures (Reports.test.tsx, navigation.spec.ts, Courses.test.tsx, story-e11-s01.spec.ts) that exist on main and are not caused by Epic 27. These get flagged in every review, require investigation to confirm they are pre-existing, and clutter the findings. The known-issues register (`docs/known-issues.yaml`) was created precisely for this purpose, but it is currently empty — none of these pre-existing failures have been registered. We are not using the tool we built.

**3. The two BLOCKERs in E27-S01 were both avoidable with closer AC reading.** The `replace: true` on `setSearchParams` directly contradicted AC 5 ("browser back button returns to the previous tab"). The missing best/worst performing quizzes contradicted AC 7. Both were caught in code review, which is the system working, but they represent implementation-time AC drift — the developer understood the acceptance criteria but made implementation choices that conflicted with them. A quick re-read of the ACs before requesting review would have caught both. This is a discipline issue, not a tooling issue.

### Dana (QA Engineer)

**Dana:** The `MemoryRouter` wrapper requirement for unit-testing components with `useSearchParams` is a known React Router pattern, but it tripped up the existing Reports unit tests when E27-S01 converted from `defaultValue` to controlled tabs. This is documented in the story's lessons learned, but it is not in `engineering-patterns.md`. Every component that adds `useSearchParams` will break its existing unit tests the same way.

## Key Insights and Learnings

1. **URL contracts decouple story dependencies better than code imports.** E27-S03 shipped before E27-S01 because the dependency was a URL convention (`?tab=study`), not a code import. When two stories connect through URLs, query params, or data contracts rather than shared functions, they can be developed and shipped in any order. This is a generalizable principle for story decomposition.

2. **Scope reduction after rebase is a feature, not a failure.** E27-S02 was planned with 5 tasks but shipped with 2 because the rebase onto main (with E27-S01 merged) resolved 3 tasks. This is the correct outcome — the planning assumed worst-case sequencing, and the actual sequencing was better. The lesson: plan for independence, benefit from overlap.

3. **The welcome wizard localStorage seed is a cross-cutting test concern that needs centralization.** Two stories in Epic 27 (S01 and S03) independently discovered that E2E tests fail without `knowlune-welcome-wizard-v1` in localStorage. This is the kind of test infrastructure gap that creates repeated debugging time across stories and epics. A shared `seedCommonLocalStorage()` function in `tests/support/` would eliminate this class of failure permanently.

4. **Pre-existing test failures must be registered in known-issues.yaml to avoid repeated investigation.** The known-issues register was built during this epic (commit `f7600b0d`) but is currently empty. The 4 pre-existing failures discovered during E27-S02 review should have been registered immediately. The register is only valuable if it is actually populated.

5. **Re-reading acceptance criteria before requesting review prevents BLOCKER findings.** Both E27-S01 BLOCKERs were AC compliance issues, not architectural or logic bugs. A 2-minute re-read of the ACs against the implementation would have caught `replace: true` conflicting with the back-button AC and the missing best/worst quiz display. Adding "AC compliance check" to the pre-review checklist would formalize this.
