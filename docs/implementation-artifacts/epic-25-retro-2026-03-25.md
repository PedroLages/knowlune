## Successes and Strengths

### Bob (Scrum Master)

**Bob:** Nine stories, all passing review in round 1. That is the best review performance we have ever seen across any epic, including E24 which held the previous record at 6/6. Pedro, what made this possible?

### Pedro (Project Lead)

**Pedro:** Three things converged to make E25 the smoothest epic we have shipped.

**1. The scan/persist pattern from E24-S01 paid dividends again in E25-S04 and E25-S05.** Author auto-detection during import (S04) hooks into the existing `scanCourseFolder()` → `persistScannedCourse()` pipeline. The author detection runs between scan and persist — it reads the scanned folder name, matches or creates an author, and passes the `authorId` as an override to persist. This is exactly the extensibility that the overrides pattern was designed for. Similarly, photo detection (S05) scores image candidates from the scanned file list and attaches the best handle to the author record. Neither story needed to modify the import pipeline's core flow. The architectural investment in E24 is compounding.

**2. The epic had a natural dependency chain that forced correct ordering.** S01 (data model) → S02 (CRUD) → S03 (page) → S04 (detection) → S05 (photos) → S06 (linking) → S07-S09 (UX polish). Each story built on a stable foundation. There were no "build two things simultaneously and hope they integrate" situations. The dependency chain meant each PR merged cleanly into a progressively richer feature set. The one friction point — merge conflicts — was caused by sprint-status.yaml and shared Layout.tsx changes, not by actual feature conflicts.

**3. The new user experience stories (S07-S09) were deliberately scoped as UI polish, not infrastructure.** The onboarding overlay already existed from E10 — S07 just rendered it in Layout.tsx and updated the messaging. Progressive disclosure (S08) is a pure localStorage + store-query pattern. Empty states (S09) extracted a shared component and wired CTAs. None of these stories required new data models, API integrations, or complex state management. They were refinement stories, and refinement stories pass review in round 1 because the risk surface is small.

### Charlie (Senior Dev)

**Charlie:** The `detectAuthorFromFolderName()` implementation in S04 deserves recognition for being a genuinely well-designed pure function. It handles 6 separator patterns (` - `, ` — `, ` – `, `: `, ` by `, `/`), validates extracted names against a `PERSON_NAME_PATTERN` regex, and returns `null` for ambiguous results. The unit tests cover 20+ edge cases including Unicode names, single-word names, numeric prefixes, and empty strings. This is the kind of function that will never cause a regression because it has no side effects and comprehensive test coverage.

## Challenges and Growth Areas

### Bob (Scrum Master)

**Bob:** With 9 stories passing review in round 1, it sounds like there were no problems. What friction did you actually encounter?

### Pedro (Project Lead)

**Pedro:** Four concerns, two of which I consider process gaps worth fixing.

**1. Merge conflicts in 5 out of 9 stories were the primary friction.** Every story that touched `sprint-status.yaml`, `Layout.tsx`, or `navigation.ts` conflicted with the previous story's merge. The conflicts were always trivial to resolve — additive changes in YAML, new imports in Layout — but they required manual intervention and rebasing. For a 9-story epic where stories merged sequentially, this is structural. The sprint-status.yaml conflicts specifically are a file-format problem: YAML does not merge well when multiple stories append entries to the same section.

**2. Two stories (S07 and S08) shipped without running `/review-story`.** The adversarial review flagged this as CRITICAL. Both stories had their PRs merged and all tests passing, but they bypassed the structured review process (code review agent, design review agent, test coverage agent). The implementations are likely fine — S07 had 5 E2E tests, S08 had 8 — but the principle matters. Every story should go through the same quality gate, especially when the gate is automated and costs nothing but time.

**3. E25-S09 shipped with zero of its planned E2E tests.** Task 7 in the story file is unchecked. The story refactored 4 pages to use a shared `EmptyState` component, but the CTA navigation and content replacement behaviors (AC5, AC6) are completely untested. The story relies on pre-existing E10-S02 regression tests, but those test the old ad-hoc empty states, not the refactored implementations. This is a real gap.

**4. The 78% trace coverage, while passing, masks that S06 and S08 are both at 67%.** S06's gaps are in bidirectional link maintenance assertions. S08's gaps are in testing only 2 of 6 disclosure keys. Both stories have the logic implemented correctly (verified by manual testing and unit tests), but the test coverage does not match the acceptance criteria granularity.

### Charlie (Senior Dev)

**Charlie:** The `matchOrCreateAuthor()` function in S04 has a read-then-write race condition. It calls `db.authors.toArray()`, does a linear scan for a case-insensitive match, and then calls `db.authors.add()` if no match is found. These operations are not wrapped in a Dexie transaction. For the current single-import UX, this is fine. But if batch import is ever added, two concurrent imports with the same author name could create duplicate author records. The adversarial review flagged this as HIGH. I accepted it for now, but it should be wrapped in `db.transaction('rw', db.authors, ...)` before any batch feature ships.

### Dana (QA Engineer)

**Dana:** The adversarial review also flagged that progressive disclosure uses `CustomEvent` + `localStorage` for cross-component communication. This is fragile — CustomEvents are fire-and-forget with no delivery guarantee, and localStorage changes in one tab can affect another. For a personal app this is acceptable, but it is a pattern we should not replicate in more critical features. Zustand with persistence would be the conventional approach.

## Key Insights and Learnings

1. **Reusing the scan/persist pattern from E24 across epic boundaries confirms it as a project-wide architectural convention.** E25-S04 and S05 extended the import pipeline without modifying its core. The `overrides` parameter in `persistScannedCourse()` accepted `authorId` and photo handles seamlessly. When a pattern works across two epics and six+ stories, it has graduated from "good idea" to "project standard." Document it as such.

2. **All-round-1 reviews across 9 stories proves that review velocity is a function of architectural stability, not story count.** E24 achieved this with 6 stories, E25 with 9. The common factor is that both epics built on established patterns (Dexie stores, Radix dialogs, Zustand state, Sonner toasts). When the team is assembling proven components rather than inventing new ones, code review finds only minor issues. The review process is not the bottleneck — novelty is.

3. **Merge conflicts in sequential story delivery are a file-format problem, not a process problem.** Five stories conflicted on sprint-status.yaml. YAML does not support concurrent appends well. Options: (a) use a JSON format that merges better, (b) batch sprint-status updates at epic completion instead of per-story, or (c) accept the friction as the cost of sequential delivery. Option (b) is the most practical.

4. **Skipping `/review-story` creates a process debt that is hard to backfill.** S07 and S08 work correctly, but they were never examined for edge cases, accessibility gaps, or architectural concerns by the review agents. Running `/review-story` retroactively is possible but loses the "review before merge" benefit. The lesson: the review step should be mandatory in the PR template checklist, not optional.

5. **Empty state stories are deceptively simple and under-tested.** S09 refactored 4 pages but added zero E2E tests. The shared `EmptyState` component's CTA navigation — the most user-facing behavior in the story — has no automated verification. Refinement stories need at least smoke-level E2E coverage for their primary interaction paths.
