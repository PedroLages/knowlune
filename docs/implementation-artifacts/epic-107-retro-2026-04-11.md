## What Was Challenging

**Bob (Scrum Master):** "S06 required three review rounds — the most of any story this epic. Pedro, walk us through what happened."

**Pedro (Project Lead):** "Round one found the stale closure: `handlePlayPause` in `useCallback` captured `isPlaying` from render scope. Rapid clicks would read the stale render-time value instead of the current store state. The fix — using `useAudioPlayerStore.getState().isPlaying` imperatively inside the callback while keeping a separate render subscription — is the correct Zustand dual-subscription pattern. That's documented in Zustand's own best practices. I knew the pattern but didn't apply it on the first pass.

Round two was the test gap. The initial implementation had no E2E spec. The review flagged it correctly — a fix to interactive elements (play/pause toggling, close button) with no E2E coverage is inadequate for a component that was previously broken. I wrote the spec, but that meant a third round to verify the tests passed and no new issues appeared.

Round three also surfaced the `coverError` reset issue: the `useState(coverError)` flag wasn't being reset when the resolved cover URL changed. If a user navigated from a book with a broken cover to one with a valid cover without unmounting the component, the error state would persist and the valid cover would be hidden. Fixed with a `useEffect` that resets `coverError` when `resolvedCoverUrl` changes."

**Charlie (Senior Dev):** "The missing tests in round one is the pattern to focus on. S06 had interactive behavior changes — play/pause toggling, close button — and no spec. The story should have included E2E tests in the initial implementation, not as a round-two addition. The review loop caught it, but the loop shouldn't be where test gaps are discovered for functional interactive elements."

**Quinn (QA Engineer):** "The `coverError` reset is an interesting case. It's a stateful bug that only manifests when the component is reused without unmounting — a pattern that's common in list/detail navigation but easy to miss in isolation. State reset on prop change is a class of bug that's worth watching for in component refactors. Whenever you initialize `useState` from a prop, you need a `useEffect` to sync subsequent prop changes."

## Lessons Learned

### 1. Stale closure bugs in Zustand callbacks require the dual-subscription pattern

When a `useCallback` depends on store state for its logic (not just display), using `useAudioPlayerStore.getState().fieldName` imperatively inside the callback — rather than reading from the render-time subscription — is the correct pattern. The render subscription (`const isPlaying = useAudioPlayerStore(s => s.isPlaying)`) handles display (icon, aria-label). The imperative `getState()` call handles action logic where stale state causes wrong behavior. If `eslint-plugin-react-hooks`/`exhaustive-deps` had been active, missing `isPlaying` from the `useCallback` deps array would have been flagged at save-time, potentially preventing this review round.

### 2. E2E tests for interactive UI changes belong in the implementation phase, not the review loop

S06 required a third review round specifically to add E2E tests for play/pause toggling — behavior that was in the acceptance criteria from the start. Any story whose AC describes user-observable interaction (button toggles playback, dialog dismisses mini-player, icon updates on state change) should have an E2E spec as part of the initial implementation, not as a round-two finding. The review loop exists to catch mistakes, not to define what needs testing.

### 3. Test-mode infrastructure in main.tsx must use both `import.meta.env.DEV` AND dynamic imports

Two conditions must both hold for E2E test bridges to stay out of production builds: (1) the enabling code must be inside `if (import.meta.env.DEV)` blocks (Vite tree-shakes these in production), AND (2) any imports used only inside DEV blocks must be dynamic (`await import()`) rather than static. Static imports at the module level are always bundled by Vite regardless of whether the code that uses them is DEV-gated. A static `import { MockEpubLoader } from '__tests__/...'` leaks into production even when the function that calls it is DEV-only.

### 4. useState initialized from a prop must have a useEffect to sync subsequent prop changes

When a component uses `useState(prop)` to initialize state from a prop (e.g., `useState(resolvedCoverUrl)`), the initial value is captured once at mount. If the prop changes without the component unmounting, the state is stale. The fix is a `useEffect(() => { setState(newPropValue); }, [newPropValue])`. This pattern applies anywhere state is derived from props that change within a reused component — cover URLs, user IDs, book selections in list/detail navigation.

### 5. Performance regressions must be either fixed before merge or filed as known issues in the same review session

The AboutBookDialog FCP regression (+77% on dev server) shipped without a code-split fix and without a filed known issue. Shipping with a documented CONCERNS status in the NFR report is not the same as having a scheduled fix. Any performance regression flagged by the benchmark agent should result in one of: (a) fixed before merge, or (b) a known issue entry filed during the review session with a target epic for resolution. 'Sub-budget therefore acceptable' is not a valid disposition unless there's a tracked item to revisit.

### 6. Retro action items without a sprint-status story entry will not be done

Four consecutive epics of sub-20% action item follow-through establishes that retro action tables are not execution plans — they're wish lists. Any retro action item that requires more than a one-line code change must be filed as a story stub in sprint-status.yaml before the retrospective session closes, or explicitly closed as wont-fix. The retro document is an artifact for reflection, not a backlog.
