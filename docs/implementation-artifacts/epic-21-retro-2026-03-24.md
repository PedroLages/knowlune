### Pedro (Project Lead)

**Pedro:** Three things specifically enabled the velocity this session.

**1. Zustand for reactive preferences was the right abstraction.** E21-S05 introduced `useEngagementPrefsStore` with Zustand + localStorage persistence, and it turned out to be the backbone for E21-S06 (dashboard config) and E21-S07 (age defaults). The pattern — Zustand store, `persist()` middleware, selector hooks for minimal re-renders — is now used in at least 8 places across the codebase. It is genuinely zero-friction to add a new preference: define the slice, expose a selector, persist to localStorage. The `useEngagementVisible` hook in S05 is a textbook example of how selectors prevent unnecessary re-renders.

**2. CSS-only approaches for E21-S04 (vibrant mode) and E21-S07 (font scaling) kept complexity low.** The vibrant color scheme is a single `.vibrant` class on `<html>` that overrides CSS custom properties — zero JavaScript runtime cost. Font scaling changes one CSS variable (`--font-size`) and the entire app scales because everything uses `rem` units. Both features could have been over-engineered with JavaScript-driven theming systems, but the CSS cascade approach is simpler, faster, and inherently correct. This is a pattern worth naming: "CSS cascade for theme variants."

**3. The review system caught real integration issues, not just style nits.** E21-S05 Round 1 found that `celebrateCompletion()` (a standalone function) was not gated by the animations preference — it would fire confetti even when animations were toggled off. E21-S06 found that IntersectionObserver throws in jsdom, which would crash unit tests. E21-S07 found that the welcome wizard needed to integrate with App.tsx's route setup, not just Settings. Each of these was a genuine integration bug that would have shipped otherwise.

**Pedro:** Four things created friction, and one of them is a recurring problem I want to address definitively.

**1. sprint-status.yaml merge conflicts on every single story merge.** This is the most persistent friction point in the entire project. Every story updates the epic/story status lines in sprint-status.yaml, and when stories are merged sequentially, every merge creates a conflict on overlapping YAML lines. In this session alone, all five PRs (#37-#41) had sprint-status.yaml conflicts. The fix is always trivial (accept both changes), but it interrupts the merge flow every time. We need to either (a) stop updating sprint-status.yaml in feature branches and do it on main after merge, or (b) split the file so each epic has its own status file.

**2. E21-S06 had 11 issues — nearly double the average — because smart dashboard reordering touches multiple concerns.** The story introduced IntersectionObserver (visibility tracking), @dnd-kit (drag-and-drop), relevance scoring (algorithm), and localStorage persistence (state) — four distinct technical concerns in one story. The IntersectionObserver jsdom guard alone took a non-trivial amount of investigation to resolve. The lesson is that stories with 4+ technical concerns should be estimated higher and potentially split.

**3. Canvas pixel readback for WCAG contrast validation in E21-S04 was a surprising complexity spike.** The vibrant mode needed to verify that boosted OKLCH colors still pass WCAG AA contrast ratios in both light and dark mode. The initial approach of computing contrast from CSS custom property values hit a wall because OKLCH-to-sRGB conversion is non-trivial. The solution — rendering to a canvas element and reading back pixel values — works but feels like a workaround. The root issue is that we lack a programmatic OKLCH contrast checker.

**4. E21-S07 touching App.tsx, Settings.tsx, and the theme system made it the riskiest integration.** The welcome wizard needed to render before the main layout on first visit, which meant modifying the root `App.tsx` to conditionally show the wizard. Font scaling required updating `theme.css` with the `--font-size` variable and ensuring all rem-based values scale correctly. Age defaults needed to pre-populate the Zustand stores from S05 and S06. This was a 3-way integration that could only be tested end-to-end, not in isolation.

## Key Insights and Learnings

1. **"CSS cascade for theme variants" is a reusable pattern.** Both E21-S04 (vibrant color scheme via `.vibrant` class) and E21-S07 (font scaling via `--font-size` variable) achieve complex personalization features with zero JavaScript runtime cost. The pattern: define CSS custom properties in `:root`, override them in a class selector on `<html>`, let the cascade handle everything. This works because Tailwind v4 uses CSS variables and `rem` units throughout.

2. **IntersectionObserver requires jsdom guards in any component that might be unit-tested.** E21-S06's `useDashboardOrder` hook crashed in jsdom because `IntersectionObserver` is undefined. The fix — `if (typeof IntersectionObserver === 'undefined') return` — is simple but easy to forget. Any hook or component using IntersectionObserver, ResizeObserver, or MutationObserver needs the same guard. This should be an ESLint rule.

3. **Zustand + localStorage is the project standard for UI preferences.** After E21, the codebase has 8+ Zustand stores with `persist()` middleware for UI state: onboarding, suggestions, engagement prefs, dashboard order, quiz preferences, color scheme, font scale, and age defaults. The pattern is proven, well-understood, and should be documented as the canonical approach in `engineering-patterns.md`.

4. **Stories with 4+ distinct technical concerns generate 2x the review issues.** E21-S06 (11 issues) combined IntersectionObserver, @dnd-kit, relevance scoring, and localStorage persistence. The average across the other four stories was 5.5 issues. When a story touches 4+ independent APIs or patterns, the integration surface area increases non-linearly. Consider splitting or accepting the higher review cost upfront.

5. **sprint-status.yaml is a structural bottleneck for sequential story merges.** Every feature branch modifies it, guaranteeing conflicts. This is an organizational problem, not a technical one — the file format (single YAML with all stories) doesn't match the workflow (parallel/sequential feature branches). A per-epic status file or post-merge-only update policy would eliminate this entirely.

### Team Agreements (Effective Immediately)

1. **CSS cascade for theme variants** — Use CSS custom property overrides on `<html>` class selectors for theme personalization features. Zero JavaScript runtime cost. Do not build JS-driven theming unless CSS cascade is insufficient.
2. **jsdom guard for browser-only APIs** — Always check `typeof IntersectionObserver !== 'undefined'` (and ResizeObserver, MutationObserver) before use in hooks or components that may be unit-tested.
3. **Zustand + persist() for UI preferences** — Canonical pattern for all user preference state. Do not use raw localStorage or context for new preference features.
4. **Split or estimate higher for 4+ concern stories** — If a story touches 4+ independent APIs/patterns (e.g., S06: IntersectionObserver + @dnd-kit + scoring + persistence), either split into 2 stories or budget 2x review time.
