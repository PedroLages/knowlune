## What Went Well

### 1. Pure Functions ‚Äî Zero Drift, Zero Bugs in the Algorithm Core

**Charlie (Senior Dev):** "I'm actually proud of `chapterMatcher.ts`. It's 225 lines. No imports except the type. No async. No side effects. The Jaro-Winkler implementation is correct ‚Äî I checked it against the canonical description. And we shipped it with zero runtime issues."

**Bob (Scrum Master):** "The unit test suite for it is solid too ‚Äî 12 tests covering exact matches, normalization, greedy assignment, fallback pass, empty inputs, sort order."

**Pedro (Project Lead):** "This is the pattern I want to see everywhere. When you isolate computation into pure functions, you can test it in isolation, reason about it clearly, and the review agents have something concrete to analyze. `chapterSwitchResolver.ts` followed the same discipline ‚Äî pure, 12 unit tests, 100% path coverage."

**Alice (Product Owner):** "From a product perspective, the 'no npm dependencies' constraint forced the right outcome. If we'd pulled in `natural` or `similarity` or any other string matching library, we'd have a black box. Now we own the algorithm and can tune the threshold per book pair in the future."

### 2. `db.transaction()` Atomicity Introduced for Linked Book Writes

**Charlie (Senior Dev):** "The code review on S03 flagged that `linkBooks()` was doing two sequential `db.books.update()` calls. If one succeeded and the other failed, books would be permanently in a half-linked state ‚Äî A links to B but B doesn't link back. We fixed it with `db.transaction()`."

**Pedro (Project Lead):** "That's the kind of issue that only surfaces under adversarial review. The happy path never exercises the failure case. Good catch."

**Quinn (QA Engineer):** "I want to note: E88's retrospective first introduced the `db.transaction()` pattern for Dexie atomicity. We've now applied it twice. It should probably be in engineering-patterns.md explicitly."

### 3. Position Save Before Navigation ‚Äî Correctly Ordered

**Charlie (Senior Dev):** "R1 of E103-S02 caught a race condition I'm embarrassed about. The original `onSwitchToReading` handler called `navigate()` before the `savePosition()` callback. React Router's `navigate()` is synchronous in the sense that the component unmounts before any remaining callstack executes. Position would have been silently lost."

**Pedro (Project Lead):** "The lesson is now in the story file: 'Position must be saved before navigation, not after.' And the implementation uses a clear pattern ‚Äî `savePosition?.()` then `navigate()` ‚Äî with a comment explaining why the order matters."

**Alice (Product Owner):** "This is the kind of bug that's invisible in testing but devastating in production. User reads to chapter 8, switches to audio ‚Äî next time they open the book it's back at chapter 1. Very bad experience."

### 4. CFI Ordering Bug Caught Early

**Bob (Scrum Master):** "The story notes mention a CFI ordering bug. Can someone explain that for the record?"

**Charlie (Senior Dev):** "CFI strings ‚Äî the EPUB position format ‚Äî look like paths but they're not lexicographically ordered. The original `findCurrentEpubChapterHref` function tried to sort chapters by CFI string comparison to find which chapter the user was in. That doesn't work because CFI sort order isn't the same as reading order. The fix was to sort by `chapter.order` ‚Äî the integer we store explicitly ‚Äî and use that for all ordering decisions."

**Pedro (Project Lead):** "This is a general lesson: epub.js internals are weird. Don't assume CFI strings are comparable by string operators. Always use the metadata we've explicitly set (`order`, `position`) rather than the CFI itself for ordering."

### 2. Normalization Documentation vs. Implementation Mismatch

**Charlie (Senior Dev):** "The Dev Notes in S01 say `normalizeChapterTitle('Chapter 1: The Journey Begins')` produces `'the journey begins'`. It doesn't. It produces `'chapter 1: the journey begins'` because the regex only strips leading *numeric* characters, not leading alphabetic words like 'Chapter'. The unit test asserts the wrong behavior and passes ‚Äî because the test was written to match the implementation, not the documentation."

**Bob (Scrum Master):** "So the algorithm is documented wrong, the test asserts the wrong thing, and the adversarial review caught it."

**Charlie (Senior Dev):** "Correct. The practical impact: EPUB chapters titled 'Chapter 3: Into the Dark' and audio chapters titled 'Into the Dark' won't match reliably ‚Äî they differ after normalization. The Jaro-Winkler score for those two normalized strings is around 0.6, below the 0.7 threshold."

**Quinn (QA Engineer):** "That's a silent failure. The user links their books, the matching engine runs, and 40% of chapters don't match because of this normalization gap ‚Äî with no error message."

**Pedro (Project Lead):** "I want this fixed in a chore commit before E104. The regex needs to also strip leading 'Chapter N', 'Part N', 'Book N' patterns. And the test needs to assert the correct output per the documented spec ‚Äî not the current behavior."

**Pedro (Project Lead):** "The pre-review checklist has 'No optimistic UI updates before persistence ‚Äî position saved before navigation.' It was there. It wasn't checked. The checklist is only useful if you actually go through it line by line before submitting for review."

**Charlie (Senior Dev):** "I'll admit ‚Äî I skimmed the checklist. The position-save issue would have been caught if I'd actually run through it carefully."

## Lessons Learned

| # | Lesson | Category | Applies To |
|---|--------|----------|-----------|
| L1 | Isolate domain algorithms into pure synchronous functions ‚Äî testable in isolation, no hidden state, no side effects | Architecture | Any story with string processing, matching, or scoring logic |
| L2 | Save position before navigation, never after ‚Äî React Router `navigate()` unmounts the source component synchronously | React patterns | Any format switch, navigation-on-action, or cross-route handoff |
| L3 | CFI strings are not lexicographically ordered ‚Äî never compare or sort them as strings; use explicit `order` integer metadata | epub.js | Any story touching EPUB position or chapter ordering |
| L4 | Use `db.transaction()` for any Dexie write that touches multiple records that must stay in sync (e.g., `linkBooks`) | Dexie patterns | Any story creating bidirectional references in the database |
| L5 | Documentation examples must be tested ‚Äî if the Dev Notes say "normalizes to X", a test must assert "normalizes to X" | Quality | All stories with algorithm documentation |
| L6 | Features deferred during a story must be tracked as explicit backlog items ‚Äî "no routing integration needed yet" without a follow-up story is a dead end | Process | All stories that defer entry points or triggers |
| L7 | Retro action items must be committed as code/docs before the next epic's first story starts ‚Äî markdown lists without commits are aspirations | Process | All epics |
| L8 | Burn-in validation is mandatory for any story with navigation + live queries + ref-based guards, not optional based on developer confidence | Testing | Any story with `useLiveQuery` + `navigate()` combination |
