## What Was Challenging

**Bob (Scrum Master):** "Seven issues in S01's two rounds. What drove that number?"

**Pedro (Project Lead):** "Three clusters. First, the ETA formula — the initial implementation computed `avgPagesPerDay` as `totalPages / 30` rather than `(totalPages / sessionCount) * 7`. Off by a factor of 4.3x for a user reading twice weekly. The code looked correct in isolation; the bug only showed up when I verified the formula against a manual example with realistic session cadence data. The fix was straightforward once the formula was derived correctly, but it required stopping to do the derivation, which is the kind of thing that gets rushed in implementation."

**Charlie (Senior Dev):** "Second cluster: speed inflation from old books. If a user finished a book five years ago with 12 hours of session data, that book dominates the speed average even though their current reading pace is different. The fix — filter finished books to only those with recent sessions (last 90 days) — is a meaningful change to the semantics of 'average reading speed.' The original implementation counted all finished books regardless of recency."

**Pedro (Project Lead):** "Third cluster: AC2 was implemented in the service (`computeETA()`) and tested in unit tests, but never wired to the UI. Code review caught it. The acceptance criterion says 'each in-progress book shows an ETA.' I had the data. The component didn't render it. That's a scope gap — the service work felt like completion, and the UI wiring was skipped."

**Quinn (QA Engineer):** "In S02, the `avgPagesPerSession` metric had a numerator/denominator mismatch. Pages were summed from finished books only, but session count included all sessions — including sessions on in-progress books. The result was artificially low averages for users with many in-progress sessions. The fix was to scope sessions to finished-book session IDs before counting, so both numerator and denominator refer to the same scope."

**Alice (Product Owner):** "The 5% threshold and 8-genre cap interaction is the kind of thing that's obvious in retrospect and non-obvious at design time. A genre above 5% can still get merged into 'Other' if 8 larger genres fill the named slots. The cap takes priority. That interaction should be commented in the code — and it is now — but it's also a case where the spec should have made the priority order explicit."

## Patterns and Themes

**Theme 1 — Recurring (x8 epics): Inter-session action items don't get done**

Eight consecutive epics with sub-20% follow-through on action items requiring work between epic close and first story open. The pattern is stable. Calling it out in retrospectives has not changed it. The only effective mitigation is converting action items to commits before the retro is filed, or embedding them in the first story as explicit subtasks. Aspirational action items in a YAML table do not ship.

**Theme 2 — Recurring (x2 epics): Service implementation ahead of UI wiring**

AC2 (ETA) was implemented in the service, covered by unit tests, and not wired to the UI until code review caught it. This is the same class of gap that appeared in E110-S02 (the `filteredBookIds` memoization bug was implementation-correct but semantically wrong). The pattern: implementation-level satisfaction (tests pass, service correct) does not guarantee AC satisfaction (feature visible to user). The pre-review checklist item "Verify each AC appears in the UI, not just in the service layer" is a recurring fix for this.

**Theme 3 — Positive: Service-first architecture + unit test isolation**

Both stories extended `ReadingStatsService.ts` with pure functions before building components. The service functions are independently testable, have no React dependencies, and handle all business logic. The UI components are thin: they call the service, receive data, render. This split produced two-round review cycles where review feedback was about presentation quality, not correctness. When the service layer is well-tested, review rounds shrink.

**Theme 4 — Positive: Formula-first derivation prevents unit conversion bugs**

The ETA formula bug was caught before review by manually deriving the expected output with a concrete example (user reads twice weekly, 200 pages remaining, 30 pages/hour speed → ~3.3 hours → ~1.7 days at 2 hours/day). Deriving the expected output before coding the formula, then validating the code against that derivation, is a cheap gate that catches unit conversion errors before they reach tests. Tests validate implementation; derivations validate the formula itself.

**Theme 5 — Positive: Scope matching for numerator/denominator**

The `avgPagesPerSession` bug (S02) and the speed inflation bug (S01) share the same root cause: mismatch between the scope of the numerator and the scope of the denominator. Both were caught in review. The pattern to prevent them: when computing a ratio, write out "numerator = X over scope Y; denominator = count of Z over scope Y" — if the scopes differ, the ratio is suspect. This applies to any aggregate metric in the analytics service.

## Lessons Learned

### 1. Service implementation without UI wiring is an AC gap, not an implementation success

A service method that computes the correct value and is covered by passing unit tests still fails the acceptance criterion if the result is never rendered. The gap is invisible in tests — tests validate the computation, not the user's experience. The pre-review checklist must include: "For each AC, trace the path from data source to visible UI element. If the path is broken at any point, the AC is not met." (E112-S01, AC2 ETA display)

### 2. ETA and similar chained formulas require derivation validation, not just unit tests

Unit tests validate that the code produces the correct output for the inputs given. If the formula is wrong, the tests are wrong in the same direction. For chained unit-conversion formulas — `pages/hour → hours/day → days/remaining` — derive the expected output independently using a realistic example before coding. Compare the derived value against the coded output. If they match, the formula is sound. If not, one of the conversion steps is wrong. (E112-S01, ETA formula bug: avgPagesPerDay = totalPages/30 instead of totalPages/(sessionCount/7))

### 3. Aggregate ratios require explicit scope matching

`avgPagesPerSession = totalFinishedPages / totalSessionCount` is incorrect when `totalFinishedPages` is scoped to finished books and `totalSessionCount` includes all sessions. The ratio is meaningful only when numerator and denominator operate on the same scope. When writing any aggregate metric, declare the scope for both components and verify they match. (E112-S02, avgPagesPerSession; E112-S01, speed inflation from unrecency-filtered finished books)

### 4. Genre thresholding rules with multiple constraints need explicit priority documentation

The genre distribution chart applies two constraints: a 5% threshold (small genres → "Other") and an 8-genre cap (max named genres). When both apply, the cap takes priority — a genre above 5% can still be demoted to "Other" if 8 larger genres fill the named slots. This interaction is non-obvious and produced a subtle test case failure during development. Whenever business logic has two constraints that interact, document the priority order with an `// Intentional: cap takes priority over threshold` comment at the point of interaction. (E112-S02)

### 5. Recent-session filtering for speed computation is semantically correct and must be explicit

Reading speed computed from all-time finished books inflates the average for users whose reading habits have changed. A book finished five years ago with slow, irregular sessions should not dominate the computed speed used for today's ETA. The correct scope is sessions within the last 90 days from finished books. This scope decision is a product choice, not an implementation detail — it belongs in the acceptance criterion and the service function signature comment, not discovered during review. (E112-S01)
