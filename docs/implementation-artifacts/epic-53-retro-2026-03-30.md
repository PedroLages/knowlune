## Successes and Strengths

### 1. Composable Export Architecture Paid Off

The existing `exportService.ts` and `noteExport.ts` established clear patterns (YAML frontmatter, `sanitizeFilename()`, `yieldToUI()`, progress callbacks) that S01 and S02 followed almost mechanically. S03's `pkmExport.ts` orchestrator is only 122 lines because it simply composes the three existing exporters. This composability made a 3-story export epic completable in one day with zero architectural decisions to debate.

### 2. 100% First-Pass Review Rate — Best in Recent History

All 3 stories passed code review on the first round. This is the best first-pass rate across all epics in the recent batch (E59, E54, E53). Contributing factors:
- Stories were pure library code (S01, S02) or thin UI wiring (S03) — no complex state management
- Edge case review was done during planning (`docs/reviews/adversarial/edge-case-review-2026-03-28-pkm-export.md`), and all EC-HIGH items were addressed proactively in implementation
- The established export pattern from `noteExport.ts` gave reviewers a known-good reference to compare against

### 3. Proactive Edge Case Handling (EC-HIGH Items)

Four HIGH-severity edge cases were identified during pre-implementation adversarial review and all were addressed in S02:
- sql.js WASM binary issue: solved by using ASM.js build instead of WASM
- HTML content in Anki cards: `stripHtml()` applied before adding to deck
- Unmaintained `anki-apkg-export`: bypassed entirely — hand-built .apkg with sql.js + JSZip
- Dynamic import failure: wrapped in try-catch with descriptive error

This pre-implementation edge case review saved at least one review round per story.

### 4. Design Token Compliance: Zero Violations (Continuing Trend)

S03's UI cards used `bg-brand-soft`/`bg-success-soft` design tokens correctly. The ESLint automation continues to enforce this at save-time.

## Challenges and Growth Areas

### 1. E53-S03 Has No E2E Test Coverage (MEDIUM Gap)

The test coverage review flagged that S03 has 0/8 acceptance criteria covered by E2E tests. The story spec described E2E scenarios (lines 137-144) but none were implemented. The `tests/e2e/story-e53-s03.spec.ts` file exists (76 lines) but the test coverage agent found all ACs uncovered in its matrix.

Unit tests for `pkmExport.ts` were added (214 lines), but the Settings UI integration — card visibility, button clicks triggering downloads, disabled state during export, empty state toasts — is only manually verified.

**Root cause:** The story's E2E tests were treated as lower priority since the export logic was unit-tested. But the UI integration (button wiring, toast messages, disabled states) is exactly what E2E tests should validate.

**Lesson:** Export UIs need at least a smoke-level E2E test verifying the buttons render and respond to clicks, even if the download interception is complex.

### 2. `yieldToUI()` Duplicated Across 3 Files

Code review flagged this as LOW in both S01 and S02. The identical `yieldToUI()` helper (`new Promise(resolve => setTimeout(resolve, 0))`) now exists in `flashcardExport.ts`, `bookmarkExport.ts`, and `ankiExport.ts`. The existing `exportService.ts` has the same concept inline.

**Root cause:** Each story was implemented sequentially, and extracting a shared utility felt like scope creep within each story.

**Lesson:** When a utility is duplicated 3+ times, extract it. A shared `src/lib/exportUtils.ts` would be a 5-minute task that prevents 4 copies of the same function.

### 3. `anki-apkg-export` NPM Package Was Unusable

S02's story spec assumed the `anki-apkg-export` package would work. In practice, it's unmaintained since ~2019 and incompatible with modern bundlers. The implementation pivoted to hand-building the .apkg format using raw sql.js + JSZip, resulting in a 417-line module instead of the estimated ~80 lines.

The edge case review had flagged this risk (EC-HIGH: unmaintained package), which made the pivot smooth — but the story estimate was still optimistic.

**Lesson:** When depending on unmaintained packages for file format generation, prototype the fallback before committing to the package approach. The story spec should have allocated tasks for both paths.

### 4. Non-Deterministic Timestamps in Export Output

Code review flagged `new Date().toISOString()` in `bookmarkExport.ts` and `Date.now()` in `ankiExport.ts`. These make exports non-reproducible and harder to snapshot-test. S01 partially addressed this by injecting a `nowFn` parameter for testability, but S02 did not follow the same pattern.

**Lesson:** Export functions that embed timestamps should accept an optional `now` parameter for deterministic testing. This should be added to the export pattern documentation.

## Key Insights and Patterns

### Pattern 1: Pre-Implementation Edge Case Review Eliminates Review Rounds

The adversarial edge case review done during E53 planning identified 4 HIGH-severity issues that would have been BLOCKERs or HIGH findings in code review. Because they were addressed during implementation, all 3 stories passed first review. This pattern should be repeated for future epics that involve file format generation or third-party dependencies.

### Pattern 2: Thin Orchestrator Over Fat Module

`pkmExport.ts` (122 lines) demonstrates the right abstraction level for a "batch export" feature. It composes 3 existing exporters, adds weighted progress, generates a README, and that's it. No business logic duplication, no custom export logic. Future batch features (e.g., "export everything" or "scheduled export") should follow this orchestrator pattern.

### Pattern 3: ASM.js Over WASM for Optional Features

The sql.js ASM.js build avoids the WASM binary file distribution problem entirely. For features that are optional (Anki export is a niche feature), the small performance cost of ASM.js vs WASM is irrelevant compared to the deployment simplicity. Prefer ASM.js builds for optional/rare features; reserve WASM for performance-critical paths.

## Key Takeaways

1. **Pre-implementation edge case reviews eliminate review rounds.** All 4 EC-HIGH findings from the adversarial review were addressed during implementation, resulting in 100% first-pass review rate. This saved an estimated 3-6 fix commits and 1-2 extra review rounds across the epic. Apply this practice to every epic involving file formats or third-party dependencies.

2. **Composable export architecture scales linearly.** Adding 3 new export types (flashcard Markdown, bookmark Markdown, Anki .apkg) required zero changes to existing modules. The PKM bundle orchestrator is 122 lines because it delegates everything. This validates the export architecture decisions from Epic 11.

3. **Unmaintained packages are not worth the risk for niche features.** The `anki-apkg-export` package was dead-on-arrival. Hand-building the .apkg with sql.js + JSZip produced a more maintainable, better-understood module (417 lines) than wrapping an abandoned dependency would have. For niche features, own the implementation.
