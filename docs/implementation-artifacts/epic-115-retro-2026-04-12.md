## Patterns and Themes

**Theme 1 — Recurring (x11 epics): Inter-session action items don't get done**

Eleven consecutive epics with zero or near-zero follow-through on action items requiring work between epic close and next story open. The items are valid and achievable. The mechanism — write in retro, do before next story opens — has an 11-epic failure rate. The only viable fix: commit the changes before the retro session closes. Not a "before next story opens" commitment — a commitment measured in minutes from the retro.

**Theme 2 — Positive: Schema-free feature extension**

E115 extended the challenge system with two new types without a Dexie schema migration. This is a pattern worth preserving: when new type variants can be expressed as new string literal values in an existing union type and the existing schema columns support the new data shape, no migration is needed. The `ChallengeType` union extension is a clean example. Future feature extensions should check whether schema-free extension is possible before planning a migration.

**Theme 3 — Positive: Filter loop optimization**

Extracting derived comparison values (timestamps, computed scores) from inside `.filter()` callbacks to the outer scope is a consistent finding pattern in Knowlune's Dexie queries. `const createdAtMs = new Date(challenge.createdAt).getTime()` extracted once before the filter loop vs. called on every iteration. The optimization is small on typical data sizes but the code clarity improvement is immediate. This should be added to `docs/engineering-patterns.md` as a Dexie filter pattern.

**Theme 4 — Positive: Accept-and-document over scope-creep**

The pages baseline limitation was a candidate for scope creep: "we could add a baseline snapshot to the schema." Instead, the decision was explicit: accept the limitation, document it in JSDoc, ship the feature. This is the correct trade-off for a v1 implementation. The baseline snapshot improvement is a well-defined future story. Accept-and-document prevents scope creep while preserving the improvement path.

**Theme 5 — Recurring: E2E coverage gap for state-heavy features**

E115's E2E tests were skipped due to complex Dexie state seeding requirements. This is a recurring gap across epics — features that depend on non-trivial IndexedDB state (challenge progress, reading stats, streak continuity) are difficult to E2E test with current infrastructure. The regression specs exist but don't run in CI. This gap will grow as state-dependent features accumulate.

## Lessons Learned

### 1. Dexie filter loop: extract derived comparison values before the loop

When a `.filter()` callback uses a derived value computed from an outer variable (e.g., `new Date(challenge.createdAt).getTime()`), extract it before the loop. Inside the callback it runs once per element; outside it runs once total. The pattern: `const createdAtMs = new Date(challenge.createdAt).getTime(); db.books.filter(b => b.updatedAt >= createdAtMs)`. This applies to any Dexie query that compares against a derived timestamp, score, or computed threshold. (E115-S01, calculatePagesProgress R1 MEDIUM finding)

### 2. Accepted limitations belong in JSDoc, not just story notes

When a design decision consciously accepts a limitation (no baseline snapshot, approximate metric, simplified calculation), document it at the function site in JSDoc — not only in the story file. The story file is archived; the function JSDoc is what the next developer reads. The rule: any `@param`, `@returns`, or implementation comment that describes a scope boundary or known limitation should appear in the function's JSDoc block. If it's important enough to discuss in review, it's important enough to document in code. (E115-S01, calculatePagesProgress JSDoc LOW finding)

### 3. Schema-free feature extension via union type expansion

New feature variants that map cleanly to new string literal values in an existing union type — and whose data fits the existing schema columns — can be shipped without a Dexie migration. The check: does the new variant require new columns, or only new behavior on existing columns? For E115's `'books'` and `'pages'` challenge types, the existing `target`, `createdAt`, and `type` columns were sufficient. No migration needed. Before planning a schema migration for a new variant, verify that union type expansion alone covers the data model. (E115-S01, no migration needed)

### 4. Accept-and-document prevents scope creep while preserving the improvement path

When a v1 implementation has a known limitation that could be addressed by extending scope (e.g., adding a baseline snapshot to the Challenges schema), the correct default is: accept the limitation, document it explicitly, ship the feature. The improvement is then a well-defined follow-on story rather than scope creep on the current story. The documentation must be in the code (JSDoc), not just in the story file. This keeps delivery velocity high while preserving a clear path for improvement. (E115-S01, pages progress baseline trade-off)

### 5. Inter-session action items require committed changes, not retro notes

Eleven consecutive epics with near-zero follow-through on action items documented in retrospectives. The mechanism — write the item, commit to doing it before next story opens — does not work. The only reliable mechanism: commit the change before the retro session closes. If the item cannot be committed in the current session, it should be evaluated for whether it belongs in the next story's tasks rather than an inter-session action item.
