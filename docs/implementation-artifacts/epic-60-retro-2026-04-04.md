## What Went Well

### 1. Event Bus Extension Pattern — Mechanical Consistency
The event bus extension pattern (AppEvent union → `EVENT_TO_NOTIF_TYPE` mapping → `handleEvent()` switch → dedup function → startup check) was battle-tested by S01 and then applied identically in S02 and S03. Each new trigger type took a predictable path. No guesswork, no architectural decisions mid-story.

**Evidence:** S02 and S03 implementation notes both explicitly say "follow the exact same patterns established in E60-S01."

### 2. Dedup Logic — Clear, Typed, and Testable
Each trigger type has its own per-key dedup function (`hasDecayToday`, `hasRecommendationMatchToday`, `hasMilestoneApproachingToday`), filtering on `metadata.topic`, `metadata.courseId`, and `metadata.courseId` respectively. The pattern is: query notifications by type + date, filter by metadata key, return boolean. This made dedup easy to test (12+ dedup-specific unit tests) and easy to reason about.

**Key insight from S05:** The cross-day dedup test pattern — `vi.setSystemTime()` advance by 1 day + reset mock → re-emit → verify notification created — is now a reusable pattern for any time-scoped dedup test.

### 3. Preference Suppression Architecture — Single Point of Control
The `useNotificationPrefsStore` with `TYPE_TO_FIELD` mapping made preference suppression a 2-line addition per new trigger type. The store design from earlier epics scaled perfectly to three new types with no structural changes.

**Evidence:** The preference suppression test suite (`describe('preference suppression')`) covers all three new types plus quiet hours in 4 clean test cases.

### 4. Schema Versioning — Explicit Sequencing Pattern
Documenting "next available Dexie version" in story guardrails (S01 used v32, S02 explicitly called out "must use v33") prevented version collision between stories. This is a pattern worth keeping.

### 5. Unit Test Coverage — Comprehensive Edge Cases
S05 delivered exceptional coverage: empty notes, empty review records, exact-threshold boundary (not `<=`, only `<`), zero-remaining course (course complete, not milestone), courses with no videos and no PDFs, idempotent `initNotificationService`, `destroyNotificationService` teardown. 82 test cases — this is a high-quality test suite.

## What Could Be Improved

### 1. Agentation Dev Toolbar Interfering with E2E Click Events
The toggle persistence test in `settings-notification-prefs.spec.ts` could not use a direct `click()` on the Switch component because the Agentation dev toolbar intercepts pointer events in the dev environment. The workaround — calling `useNotificationPrefsStore.getState().setTypeEnabled()` directly via `page.evaluate()` — works and is arguably better (tests the store contract directly), but it's a deviation from the "click the UI element" pattern.

**Reflection (Pedro):** This is the second time the Agentation toolbar has caused E2E test friction. The workaround pattern (store API via `page.evaluate()`) is legitimate — it tests the persistence contract more directly than UI interaction — but the underlying issue is that the dev toolbar sits in the pointer-events Z-stack. Worth raising with the Agentation plugin maintainer, or adding a `--no-dev-toolbar` flag to the Playwright config's `webServer` command.

**Action item:** Investigate adding `AGENTATION_TOOLBAR=false` (or equivalent) to the Playwright `webServer` env config so E2E tests run without the toolbar overlay.

### 2. Schema Version Tracking — Still Manual
The "check `db.verno` at implementation time" lesson from S01 is good, but we still rely on developers remembering to do it. The plan document had v30 when the actual version was v31. A small helper (e.g., logging `db.verno` in dev console on startup, or a CI check that validates schema version increments sequentially) would make this automatic.

### 3. `ImportedCourse` Type Gap — `modules` Field Absent
S03 discovered that `ImportedCourse` lacks a `modules` field, requiring the startup check to join `importedVideos` and `importedPdfs` tables to count lessons. The real-time check uses the `modules` parameter from `setItemStatus`. This type inconsistency (`modules` in store params but not in the persisted type) is a known asymmetry. It didn't block the story, but it's technical debt worth addressing when the course type is next touched.

### 4. `recommendation:match` Event Has No Emitter Yet
S02 wires up the handler for `recommendation:match` but the emitter lives in E52 (tag-based recommendation engine, not yet implemented). This is intentional (event-consumer-only story), but it means the recommendation trigger is dead code until E52 ships. Low risk, but worth noting in known issues.

## Lessons Learned (Pattern Extraction)

### Pattern 1: Event Bus Extension is Now a Recipe
Adding a new notification trigger to Knowlune is now a documented recipe:
1. Add `| { type: 'event:name'; ...payload }` to `AppEvent` in `eventBus.ts`
2. Add `NotificationType` variant to `types.ts`
3. Add `NotificationPreferences` field + Dexie migration
4. Add `EVENT_TO_NOTIF_TYPE` mapping in `NotificationService.ts`
5. Add `handleEvent()` switch case with dedup
6. Add startup check function (if applicable)
7. Add preference suppression via `TYPE_TO_FIELD`
8. Wire UI toggle in `NotificationPreferencesPanel.tsx`

This recipe should be added to `docs/engineering-patterns.md`.

### Pattern 2: Dedup Test Template
For any time-scoped dedup test:
```typescript
// Day 1: emit → notification created
// Day 2: advance clock + reset mock → emit again → notification created
vi.setSystemTime(new Date(FIXED_NOW.getTime() + 86_400_000))
mockFirst.mockResolvedValue(undefined)
appEventBus.emit({ type: 'event:name', ...payload })
await vi.runAllTimersAsync()
expect(mockAdd).toHaveBeenCalledTimes(2)
```

### Pattern 3: Store API for E2E Toggle Tests
When the Agentation dev toolbar interferes with Switch click events in E2E tests, use the store API directly:
```typescript
await page.evaluate(async () => {
  const mod = await import('/src/stores/useNotificationPrefsStore')
  await mod.useNotificationPrefsStore.getState().setTypeEnabled('type-name', false)
})
```
This tests the persistence contract directly and is more reliable than UI pointer interaction in dev environments.

### Pattern 4: Schema Version Guardrail
Always include a "next available Dexie version" note in story Implementation Notes when the story adds a migration:
```
> Schema version used: v32. Next story needing a migration must start at v33.
```
