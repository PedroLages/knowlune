## Successes and Strengths

### 1. Exceptional Story File Quality
Story S08 (Auth UX Polish) had the most thorough implementation notes in the project -- documenting the exact Supabase issue numbers, subscription-before-getSession ordering requirement, the CSP dual-config trap, anti-patterns to avoid, known limitations, and precedence chains. This level of detail directly translated to correct-first-time implementation for the hardest parts (OAuth timing, referrer policy, avatar protocol guard).

### 2. Event Bus Architecture Pattern
The typed event bus pattern from S07 achieved clean store-to-notification decoupling. Each store adds 2-5 lines to emit a domain event at the right point, and the NotificationService maps events to notifications independently. This prevents circular dependencies and is directly reusable for sync triggers in upcoming epics.

### 3. Proactive Test Health Resolution
Stories S01-S03 (56 test failures) were resolved proactively during Epic 33's test paydown, before E43 execution even began. This forward-thinking approach meant the epic could focus entirely on feature work (S04-S08) rather than test debt.

### 4. Research-to-Execution Pipeline
Brainstorming sessions upstream (test fix strategy, session expiry, notifications architecture) provided well-researched decision records. Key decisions like "local-only notifications" (no userId field), "extend hook don't rebuild" (useAuthLifecycle), and "wire callers don't rebuild function" (completion %) came from these sessions and prevented scope creep during implementation.

### 5. Security-Conscious Implementation
S08 demonstrated security awareness: HTTPS-only protocol guard on avatar URLs prevents XSS via javascript:/data: injection, CSP updated in both config locations, referrerPolicy="no-referrer" prevents Google CDN 403s, and the `ignore` flag prevents state updates after React strict mode unmount.

### 6. Pragmatic "Wiring Fix" Philosophy
S05 (completion %) exemplified the principle of wiring existing infrastructure rather than building new. The `getImportedCourseCompletionPercent()` function already existed -- the story just connected it to callers. This 5-15 line change delivered user-visible value without code bloat.

## Challenges and Growth Areas

### 1. Test-Last Pattern on Integration Stories (S07, S08)
Both S07 and S08 required 2 review rounds. S07's code review explicitly flagged "0/7 ACs have test coverage" -- tests were backfilled during the review-fix pass. S08 missed updating the `getSession` mock after adding `supabase.auth.getSession()`, causing 6 test failures found in review.

**Root cause:** Infrastructure stories (S06) naturally support test-first (clear inputs/outputs), but integration/wiring stories (S07, S08) touch multiple stores and components, making test planning harder. Without explicit test planning in the story file, tests get deferred.

**Lesson:** Require a "Test Plan" section in story files that lists specific test cases BEFORE implementation begins. For wiring stories, write event bus and service tests first, then wiring tests.

### 2. Event Semantic Mismatch (Streak Milestone)
The `streak:milestone` event was emitted on every positive streak, not just milestone thresholds. While NotificationService filtered correctly, the event name is semantically misleading for future subscribers.

**Root cause:** The store and service shared responsibility for filtering, creating ambiguity about the event's contract.

**Lesson:** Event names must match their actual semantics. The emitter is responsible for ensuring the event condition is truly met before emitting.

### 3. Date Deduplication Fragility (Review-Due)
The `hasReviewDueToday()` function uses ISO string comparison for day boundaries, which is fragile across timezones. The engineering-patterns.md explicitly documents `toLocaleDateString('sv-SE')` for this pattern.

**Root cause:** Pattern documented but not followed during implementation. No automated enforcement for date handling patterns.

**Lesson:** Consider adding an ESLint rule for date comparison patterns, similar to the design token rule.

### 4. Informally-Resolved Stories Lack Documentation
S01-S03 were resolved during Epic 33 but their story files remain at status: draft with empty Challenges and Lessons Learned sections. Institutional knowledge about mock pattern alignment, component API rewriting, and E2E cascade debugging was lost.

**Root cause:** No process for backfilling story files when work is completed informally during another epic.

**Lesson:** Add to story workflow: "If a story is resolved during another epic, the developer must backfill the Challenges and Lessons Learned section before marking done."

### 5. CSP Dual-Configuration Trap
S08 documented that both `index.html` (meta tag) and `vite.config.ts` (dev server header) define CSP, and missing either one silently blocks resources. This is the second time this has caused issues.

**Root cause:** Historical tech debt -- two CSP sources diverged over time. No automated validation that they stay in sync.

**Lesson:** Add a CSP checklist to engineering-patterns.md and PR template.

## Key Insights and Patterns

### Pattern 1: Story File Depth Predicts Implementation Quality
Stories with deep implementation notes (S04, S08) had better first-pass code quality than those with minimal notes. The correlation is strong enough to justify investing more time in story file research upfront. Specific elements that helped: exact file paths and line numbers, root cause analysis with issue references, anti-patterns to avoid, and code snippets showing the target pattern.

### Pattern 2: Typed Event Bus for Cross-Cutting Concerns
The `AppEvent` union type + synchronous `appEventBus.emit()` + subscriber pattern is the right architecture for Knowlune's store-to-service communication. It avoids circular dependencies, keeps stores focused, and is easy to test in isolation. This pattern should be extended for sync triggers in E44+.

### Pattern 3: Dexie Null-Index Workaround
Dexie cannot index `null` values, so `where('dismissedAt').equals(null)` fails silently. The workaround (`toArray() + filter()`) is acceptable for small collections (capped at 100) but won't scale. Document this limitation in engineering-patterns.md for future reference.

### Pattern 4: `_userInitiatedSignOut` Flag for Auth Event Disambiguation
Supabase fires `SIGNED_OUT` for both user-initiated and system-initiated (token expiry) cases. Setting a flag on the auth store BEFORE calling `signOut()` and checking it in the event handler is a clean, simple pattern. The flag must be reset after consumption.

### Pattern 5: ULID for Time-Sortable Identifiers
The `ulid` package generates IDs that are lexicographically sortable by creation time, eliminating the need for a separate `createdAt` index for ordering. Useful for any table where chronological ordering is important.

### Pattern 6: `sessionStorage` for Ephemeral Auth State
The return-to-route pattern uses `sessionStorage` (not `localStorage`) because the intent is ephemeral -- it should not persist across browser sessions. This is the correct choice for any auth-related temporary state.
