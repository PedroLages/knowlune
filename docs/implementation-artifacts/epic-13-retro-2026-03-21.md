## Challenges and Growth Areas

1. **Stacked branch rebase pain (E13-S02)** — E13-S02 was branched from E13-S01's feature branch. When E13-S01 was rebased and merged to main, E13-S02 inherited all old E13-S01 commits, causing extensive conflicts during rebase. **Root cause:** Using feature-on-feature branching without stacked PR tooling. **Fix:** Always branch from main.

2. **Silent failure pattern recurred for the 3rd consecutive epic** — E13-S04's `handleRetake` catch block logged to console but never showed feedback to the user. Same pattern flagged in E03-S03, E12-S06, and now E13-S04. Documentation-only enforcement has failed 3 times. **Root cause:** Silent catches are the path of least resistance when writing error handling. **Fix:** ESLint rule that flags catch blocks in event handlers without `toast.error()`.

3. **Dark mode contrast failures in 2 stories** — E13-S01 (current question bubble 2.91:1) and E13-S03 (brand button 2.91:1). Both below WCAG AA 4.5:1 minimum. Resolved by darkening `--brand` to `#6b72c4` in E13-S03. **Root cause:** The original `--brand` value was designed for light mode; dark mode compositions weren't contrast-tested upfront.

4. **Previous retro action items largely not followed through (1/8)** — Continues the pattern: items without automated enforcement drift. This undermines retrospective credibility. **Root cause:** Aspirational commitments without enforcement mechanisms don't survive contact with sprint work.

5. **E2E shuffle assertions are inherently probabilistic (E13-S05)** — `not.toEqual` for randomness has a 1/120 false-failure rate per assertion. Acceptable for E2E, but creates flakiness risk in CI. The unit distribution test is the real correctness gate.

6. **Dual localStorage write paths in E13-S06** — Zustand persist middleware and the per-quiz backup subscriber are independent write paths to localStorage. Both needed separate quota-aware error handling. This architectural complexity was necessary but increases maintenance surface.

## Key Insights and Learnings

1. **Enforcement beats documentation — only automated items achieve follow-through** — After 3 epics of tracking retro action items, the pattern is clear: ESLint rules, review gates, and git hooks achieve near-100% compliance. Documentation-only items (conventions, patterns, best practices) achieve <20% follow-through. Future retros should only commit to automatable items; others are labeled aspirational.

2. **Story scoping overestimates when existing code isn't inventoried first** — 3 of 6 stories (50%) discovered significant pre-existing code during implementation. A quick "existing code audit" step during `/start-story` planning would produce more accurate task breakdowns and set better expectations.

3. **Silent failure is a systemic pattern, not an individual mistake** — Three consecutive epics with the same anti-pattern proves it's not about developer discipline. It's about code structure: catch blocks default to logging because there's no ergonomic error surfacing pattern. An ESLint rule is the right fix.

4. **`quotaResilientStorage`'s 3-phase fallback is a pattern worth standardizing** — Detect error → attempt recovery → graceful degradation → user notification. This applies beyond localStorage to any fallible storage operation (IndexedDB, File System API, etc.).

5. **Pure algorithm extraction + distribution testing is the gold standard** — E13-S05's Fisher-Yates work demonstrates how to elevate inline utility code: extract to `src/lib/`, add comprehensive tests (including statistical validation), export as generic function. This is the "promote to library" pattern.

6. **jsdom shares a single storage backing map** — In Vitest's jsdom, `localStorage` and `sessionStorage` share the same underlying Map. Tests that verify fallback between storages must override `Storage.prototype` methods to simulate isolation. This is a testing environment gotcha worth documenting.

7. **Firefox `NS_ERROR_DOM_QUOTA_REACHED` vs standard `QuotaExceededError`** — Browser error name inconsistency requires explicit handling. The `isQuotaExceeded()` helper checking both names is a defensive pattern for any quota-sensitive code.
