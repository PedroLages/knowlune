# Story E102-S04: Socket.IO Real-Time Sync

## Story

As a Knowlune user who listens on multiple devices simultaneously,
I want live progress sync between Knowlune and Audiobookshelf,
so that switching devices shows my exact position immediately without manual refresh.

## Context & Prerequisites

**IMPORTANT — Optional Story:** Implement only if REST polling from E102-S01 (Bidirectional Progress Sync) proves insufficient on LAN. If REST polling provides acceptable latency (sub-2s position update on device switch), this story adds no user-visible value and should be skipped.

**Prerequisite:** E102-S01 (Bidirectional Progress Sync via REST) must be complete and deployed. This story upgrades the sync mechanism — it does not replace the REST fallback, which must remain operational.

**FRs:** FR43, FR44

**Epic:** E102 — Audiobookshelf Sync & Discovery (Growth)

## Acceptance Criteria

**Given** a connected ABS server with Socket.IO support
**When** Knowlune opens a connection
**Then** it establishes a Socket.IO connection to the ABS server with the Bearer token

**Given** a Socket.IO connection is active
**When** a progress update arrives from ABS (e.g., user listened on ABS mobile app)
**Then** Knowlune updates the local Book progress if the incoming position is ahead (FR43)

**Given** a user is listening in Knowlune
**When** progress updates occur (chapter change, seek, or periodic interval)
**Then** Knowlune pushes the update to ABS via Socket.IO (FR44)

**Given** the Socket.IO connection drops
**When** the connection is lost
**Then** the system falls back to REST polling behavior from Story E102-S01
**And** no error is shown to the user — fallback is silent

## Design Guidance

This story has no new UI. The Socket.IO sync operates entirely in the background. No visible indicators are added (the connection status card in E101-S01 shows "Connected" based on server reachability — Socket.IO is a sub-feature of that connection, not a separately surfaced status).

**No UI changes required.** All implementation is in the service layer and hooks.

## Implementation Notes

### Critical Architecture Decisions

**NFR5 — Zero New npm Dependencies**

The project hard constraint is zero new npm dependencies. Socket.IO client library (`socket.io-client`) is **not allowed** without explicit approval.

Options in priority order:
1. **Native WebSocket + Engine.IO framing**: ABS Socket.IO uses Engine.IO protocol. The protocol is open-source and documented. A minimal WebSocket wrapper (~100 lines) can speak Engine.IO without importing `socket.io-client`. Research this approach first.
2. **ABS bundled client**: Some ABS versions expose `/socket.io/socket.io.js` at the server URL. If available, it can be loaded dynamically without bundling it. Risk: not guaranteed to exist in all ABS versions.
3. **Defer story**: If both approaches are infeasible, document findings and mark story as "not-implementable without npm dependency" for future reconsideration.

**Do not add `socket.io-client` to package.json without Pedro's explicit approval.**

### Service Layer Pattern

Socket.IO connection management goes in `AudiobookshelfService.ts` following the existing stateless function pattern. Do NOT create a class or singleton. The hook (`useAudiobookshelfSocket`) owns the connection lifecycle.

```typescript
// src/services/AudiobookshelfService.ts additions

export interface AbsSocketConnection {
  ws: WebSocket  // or Engine.IO connection
  isConnected: boolean
  disconnect: () => void
}

export interface AbsProgressEvent {
  libraryItemId: string
  currentTime: number
  duration: number
  progress: number  // 0-1
  isFinished: boolean
}

// Returns a connection handle; caller is responsible for disconnect()
export function connectSocket(
  baseUrl: string,
  apiKey: string
): AbsSocketConnection

export function onProgressUpdate(
  connection: AbsSocketConnection,
  handler: (event: AbsProgressEvent) => void
): () => void  // returns unsubscribe function

export function pushProgressUpdate(
  connection: AbsSocketConnection,
  itemId: string,
  progress: { currentTime: number; duration: number; progress: number; isFinished: boolean }
): void
```

### Hook Pattern (follows useOpdsCatalogStore conventions)

```typescript
// src/app/hooks/useAudiobookshelfSocket.ts
// Lifecycle: connect when AudiobookRenderer opens ABS book, disconnect on unmount

export function useAudiobookshelfSocket(
  server: AudiobookshelfServer | null,
  activeItemId: string | null
): { isSocketConnected: boolean }
```

The hook:
- Connects on mount (when `server` and `activeItemId` are set)
- Subscribes to `onProgressUpdate` — applies LWW merge (same logic as E102-S01)
- Pushes on chapter change and seek events from `useAudioPlayer`
- Disconnects on unmount — cleans up WebSocket connection
- On disconnect: sets `isSocketConnected = false` (component re-renders, REST polling from E102-S01 takes over)
- Uses `useEffect` cleanup to prevent connection leak: `return () => connection.disconnect()`

### Conflict Resolution (Latest-Wins — Same as E102-S01)

When an incoming Socket.IO progress event arrives:
```typescript
const incoming = event.currentTime
const local = book.currentPosition?.seconds ?? 0
if (incoming > local) {
  // Adopt the ABS position (they're ahead)
  useBookStore.updateBookProgress(book.id, { currentTime: incoming })
}
// If local is ahead, ignore the incoming event — we'll push our position next interval
```

This mirrors the REST LWW strategy from E102-S01. Do NOT invent a different conflict resolution approach.

### Fallback Strategy

When socket disconnects, REST polling from E102-S01 must resume automatically:
- `useAudiobookshelfSocket` exposes `isSocketConnected`
- `useAudiobookshelfSync` (E102-S01) checks `isSocketConnected`: if `true`, skip periodic REST push; if `false`, resume REST push
- Transition is seamless — no gap in sync coverage

### ABS Socket.IO Events (Research Starting Points)

ABS uses Socket.IO for real-time notifications. Known events from ABS GitHub source:
- Client connects to `http://<abs-url>` with `auth: { token: apiKey }`
- Server emits `user_media_progress_updated` when progress changes
- Payload includes: `{ id, libraryItemId, episodeId, duration, progress, currentTime, ... }`
- Client emits `updateMediaProgress` to push progress to server

**Verify these against ABS v2.x source before implementing.** ABS API is not semantically versioned — event names may have changed.

Reference: https://github.com/advplyr/audiobookshelf (check `server/SocketAuthority.js` and `client/plugins/socket.js`)

### Testing Strategy

Since Socket.IO is network-level, E2E tests need a WebSocket mock:

- Option A: Mock WebSocket in the Playwright page context via `page.addInitScript()` to intercept WebSocket constructor
- Option B: Use MSW's WebSocket support (if available in test MSW version)
- Option C: Create a minimal test WebSocket server in the Playwright global setup

Test scenarios:
1. Socket connects successfully → assert `isSocketConnected === true`
2. Incoming progress event → assert `book.currentPosition.seconds` updated
3. Socket disconnect → assert REST polling resumes (no error toast shown)
4. Push progress on chapter change → assert WebSocket `send()` called with correct payload
