# Story 59.01: FSRS Type Definitions and Dependency Setup

## Story

As a developer,
I want updated TypeScript interfaces for FSRS fields and the `ts-fsrs` dependency installed,
so that all downstream implementation stories have a stable type foundation to build on.

## Acceptance Criteria

**Given** the project has SM-2 type definitions in `src/data/types.ts`
**When** this story is implemented
**Then** `ts-fsrs@^4.7.0` is installed as a production dependency
**And** `ReviewRating` type is updated to `'again' | 'hard' | 'good' | 'easy'`
**And** `Flashcard` interface replaces SM-2 fields with FSRS fields (`stability`, `difficulty`, `reps`, `lapses`, `state`, `elapsed_days`, `scheduled_days`, `due`, `last_review`)
**And** `ReviewRecord` interface is updated with the same FSRS field replacements
**And** `FlashcardSessionSummary.ratings` includes an `again` key
**And** FSRS field names match ts-fsrs library exactly (snake_case for multi-word fields)
**And** `tsc --noEmit` reports type errors only in files that consume these interfaces (expected — they will be fixed in subsequent stories)

**Given** the `ReviewRating` type now includes `'again'`
**When** any consumer references `ReviewRating`
**Then** TypeScript enforces exhaustive handling of the new variant

## Implementation Notes

- `ts-fsrs@4.7.1` installed (latest matching ^4.7.0)
- FSRS field names use snake_case (`elapsed_days`, `scheduled_days`, `last_review`) to match ts-fsrs library exactly
- `CardState` type alias (0|1|2|3) added for readability — maps to New/Learning/Review/Relearning
- `last_review` is optional (undefined = never reviewed) matching ts-fsrs `createEmptyCard()` which has no `last_review` field
- `due` is required (always has a value — set to now for new cards)
- SM-2 fields removed: `easeFactor`, `interval`, `reviewCount`, `nextReviewAt`, `reviewedAt` (from Flashcard); same from ReviewRecord
- `lastRating` kept on Flashcard for UI display purposes

## Testing Notes

This is a type-definition story. TypeScript compiler (`tsc --noEmit`) is the primary verification — it should report errors in consumer files that still reference SM-2 fields. Those errors are expected and will be resolved in Stories E59-S02 through E59-S08.

## Challenges and Lessons Learned

- The ts-fsrs library uses `state` as a numeric enum (0-3) rather than string literals. We created a `CardState` type alias to document the mapping while keeping numeric values for ts-fsrs compatibility.
- FSRS uses snake_case field names (`elapsed_days`, `scheduled_days`, `last_review`) which differs from the project's camelCase convention, but matching the library exactly prevents mapping bugs.
