# Story 107.6: Fix Mini-Player Interactivity

## Story

As a learner,
I want the video mini-player and audio mini-player to respond correctly to play/pause, close, and navigation actions,
so that I can control playback without unexpected behavior like the player disappearing when I pause.

## Acceptance Criteria

- **AC-1**: The video mini-player remains visible when the user pauses the video while scrolled past the main player (does not hide on pause)
- **AC-2**: The video mini-player play/pause button correctly toggles playback and updates its icon state
- **AC-3**: The video mini-player close button dismisses the mini-player; scrolling back to the main video area re-shows the main player
- **AC-4**: The audio mini-player play/pause button toggles audio playback without stale state or UI lag
- **AC-5**: All mini-player interactive elements (buttons) have proper `type="button"` attributes and visible focus styles for keyboard navigation
- **AC-6**: The audio mini-player cover image error fallback uses a CSS class-based approach instead of inline style manipulation

  - [ ] 1.1 Change `isVisible` prop to not require `isVideoPlaying` — show when video is out of view and not dismissed

## Challenges and Lessons Learned

- **Stale closure in useCallback**: The `handlePlayPause` callback captured `isPlaying` from render scope, causing toggling to read stale state after rapid clicks. Fix: read `useAudioPlayerStore.getState().isPlaying` inside the callback while keeping a separate `isPlaying` subscription for the render path (icon/aria-label). This dual-subscription pattern (render subscription + imperative getState) is documented in Zustand best practices.
- **Cover image error handling**: The original `onError` handler used `e.currentTarget.style.display = 'none'` (inline style mutation), which is an anti-pattern in React. Replaced with `useState(coverError)` + conditional rendering. Added a `useEffect` to reset `coverError` when the resolved cover URL changes (book switch without unmount).
- **E2E testing without audio playback**: Headless browsers cannot play audio. Tests seed Zustand store state via `window.__audioPlayerStore__` test handle (exposed in dev mode) and mock `HTMLMediaElement.play()/pause()` to assert component state (aria-labels, visibility) rather than audio output.
