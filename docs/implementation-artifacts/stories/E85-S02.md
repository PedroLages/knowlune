# Story 85.2: Highlight Annotations and Management

Status: ready-for-dev

## Story

As a learner,
I want to add notes to my highlights and manage them (edit, delete),
so that I can annotate important passages and maintain my highlights over time.

## Acceptance Criteria

1. **Given** the HighlightPopover is showing after text selection **When** the user taps the note button **Then** a highlight is saved (yellow default) and a note input field appears below the popover **And** the user can type an annotation and press Enter or tap Save to persist it to the BookHighlight record

2. **Given** a highlighted passage exists in the reader **When** the user hovers over or taps the highlight **Then** a mini-popover appears showing: note preview (if exists), "Edit" button, "Delete" button, and "Create Flashcard" button (or "View Flashcard" if one exists)

3. **Given** the user selects "Edit" on an existing highlight **When** the edit interface appears **Then** the user can change the highlight color, edit the annotation text, and save changes **And** changes are persisted to Dexie and the highlight overlay updates in the reader

4. **Given** the user selects "Delete" on an existing highlight **When** the user confirms deletion **Then** the BookHighlight record is removed from Dexie **And** the highlight overlay is removed from the epub.js rendition **And** if a linked flashcard exists, the `sourceHighlightId` on the flashcard is cleared (flashcard preserved, link severed)

## Dev Notes

### Architecture Constraints

- **epub.js annotation callback**: The callback parameter in `rendition.annotations.highlight(cfiRange, data, callback)` fires when the user taps/clicks the highlight. This is how we detect interaction with existing highlights. The `data` parameter can store the highlight ID for lookup.
- **Annotation removal and re-add**: epub.js does not have an "update annotation color" API. To change color, you must `rendition.annotations.remove(cfiRange)` and then re-add with new styles. This is a known pattern.
- **Flashcard link severing on delete**: When a highlight is deleted, the linked flashcard continues to function in the FSRS queue. Only the back-navigation link (`sourceHighlightId`) is cleared. The flashcard's `sourceBookId` remains for context.
- **Event bus**: Emit `highlight:deleted` on deletion, `highlight:updated` on edit.

### Key Dependencies (from E85-S01)

- HighlightPopover component with note button
- HighlightLayer with `rendition.annotations.highlight()` and callback registration
- useHighlightStore with CRUD actions
- BookHighlight type with `flashcardId` field
