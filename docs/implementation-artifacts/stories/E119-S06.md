# Story E119-S06: Large-Export Async Path + Signed URL

## Story

As a user with a large dataset (> 500 MB),
I want my data export to be generated asynchronously and delivered via a signed URL,
so that I receive a complete export even when it cannot be streamed inline.

## Acceptance Criteria

- **AC-1**: When `export-data` returns `{ status: 'too-large', route: 'async' }`, the UI acknowledges with "We're preparing your export — you'll receive an email with a download link."
- **AC-2**: `export-data` enqueues a job `(user_id, request_id, created_at)` and returns `202 { status: 'queued', eta }`.
- **AC-3**: `export-worker` Edge Function picks up the job, builds the ZIP bundle, uploads to `exports/<user_id>/<request_id>.zip`, creates a signed URL with 7-day expiry, emails the user.
- **AC-4**: If a user requests two exports in quick succession, the second is de-duplicated or superseded; only one URL is emailed.
- **AC-5**: If the worker fails, the job is retried once (idempotency key); on second failure the user receives "we couldn't build your export — please contact support".
- **AC-6**: Signed URL expiry (7-day) is enforced by `retention-tick` (S11) purging `exports/` objects older than 7 days.
- **AC-7**: `docs/deployment/retention-cron-setup.md` updated with `exports/` purge details.
