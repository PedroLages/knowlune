# Story 71.3: Knowledge Map Integration and Tests

Status: ready-for-dev

## Story

As a learner navigating to the Knowledge Map,
I want the Suggested Actions panel to appear alongside the treemap with live data from my topic scores,
so that I can act on declining topics without leaving the Knowledge Map page.

## Acceptance Criteria

1. **Given** the `/knowledge-map` page renders with topic scores from `useKnowledgeMapStore`, **When** at least one topic has a declining knowledge score (tier = 'fading' or 'weak'), **Then** the `SuggestedActionsPanel` appears with action cards sorted by urgency.

2. **Given** the `/knowledge-map` page renders on desktop (>= 1024px), **When** topics have declining scores, **Then** the page layout is `flex gap-6` with the treemap as `flex-1` and the `SuggestedActionsPanel` as a `w-80` sidebar.

3. **Given** the `/knowledge-map` page renders on mobile (< 640px), **When** topics have declining scores, **Then** the `SuggestedActionsPanel` renders above the topic list (below the header), not as a sidebar.

4. **Given** all topics have strong scores (>= 70), **When** the `/knowledge-map` page renders, **Then** the panel shows the empty state message "All topics looking strong!" (panel is still visible, not hidden).

5. **Given** the `useKnowledgeMapStore` has a `getSuggestedActions()` computed getter, **When** it is called, **Then** it reads `topicsWithScores` from the store, calls `generateActionSuggestions()` with the topic data, and returns `ActionSuggestion[]`.

6. **Given** the `SuggestedActionsPanel` is rendered on the Knowledge Map, **When** the learner interacts with the treemap (clicking cells, filtering categories), **Then** the panel does not interfere with treemap interactions (no z-index overlap, no event capture issues).

7. **Given** `actionSuggestions.ts` is imported in unit tests with a declining topic having flashcard data, quiz data, and lesson data, **When** `generateActionSuggestions()` is called, **Then** it produces correct action suggestions with proper routes, labels, and time estimates.

8. **Given** `generateActionSuggestions()` is called with FSRS stability data for some topics and no data for others, **When** unit tests run, **Then** topics with FSRS data use FSRS-derived urgency and topics without use recency decay fallback — both in the same result set.

9. **Given** `generateActionSuggestions()` is called with an empty topics array, **When** unit tests run, **Then** it returns an empty array (no errors).

10. **Given** all unit and E2E tests reference dates, **When** tests run, **Then** they use the `FIXED_DATE` constant per project ESLint `test-patterns/deterministic-time` rule.

11. **Given** the `/knowledge-map` page is loaded in a Playwright E2E test with seeded declining topics, **When** the page renders, **Then** the `SuggestedActionsPanel` is visible with at least one `ActionCard` containing a CTA button.

12. **Given** a Playwright E2E test clicks an ActionCard CTA button, **When** the CTA links to `/flashcards?topic=body-language`, **Then** the browser navigates to that route (URL assertion).

## Dev Notes

### Architecture Compliance

**Store extension pattern:** The `getSuggestedActions()` getter in `useKnowledgeMapStore` follows the project's computed getter pattern. It reads existing store state (`topicsWithScores`) and delegates computation to the pure function `generateActionSuggestions()`. No direct DB calls in the getter.

**Zustand individual selectors (CRITICAL):** Always access via `useKnowledgeMapStore(state => state.getSuggestedActions())`. NEVER destructure the full store.

### Page Layout Integration

The KnowledgeMap page (E56-S04) currently has this approximate structure:
```
KnowledgeMap (page)
├── PageHeader
├── CategoryFilterChips
├── TopicTreemap (desktop) or TopicList (mobile)
└── FocusAreasPanel
```

After integration:
```
KnowledgeMap (page)
├── PageHeader
├── CategoryFilterChips
├── ContentArea (flex gap-6 on desktop)
│   ├── TopicTreemap (flex-1) or TopicList (mobile)
│   └── SuggestedActionsPanel (w-80 sidebar on desktop, above topic list on mobile)
└── FocusAreasPanel
```

**Desktop sidebar styling:**
```
w-80 shrink-0 sticky top-6 self-start max-h-[calc(100vh-8rem)] overflow-y-auto
```

**Panel visibility:** Always render the panel. When no suggestions exist, it shows the empty state. Do NOT conditionally render based on suggestion count — the empty state is an intentional positive reinforcement message.

### E2E Test Strategy

**Seeding:** Use project's IndexedDB seeding helpers (see `tests/helpers/` or `tests/fixtures/`). Seed topics with known scores to produce predictable suggestions.

**Test IDs:** Add `data-testid="suggested-actions-panel"` to the panel container and `data-testid="action-card"` to each ActionCard for reliable E2E selection.

**Viewport testing:** Use Playwright `page.setViewportSize()` to test responsive behavior:
- Desktop: `{ width: 1280, height: 720 }`
- Mobile: `{ width: 375, height: 667 }`

**URL assertions:** After clicking a CTA, use `await expect(page).toHaveURL(/\/flashcards\?topic=/)` pattern.
