# Story 35.04: Client-Side isPremium() Refactor & Dev Proxy

## Story

As a developer,
I want the Vite dev proxy to mirror server-side entitlement enforcement with a DEV_SKIP_ENTITLEMENT escape hatch,
so that dev-prod parity is maintained while local development remains frictionless without Supabase.

## Acceptance Criteria

- AC1: Given the `isPremium()` function, when a developer reads its JSDoc, then the comment explicitly states: "UX hint for fast rendering. Server-side middleware is the security boundary."
- AC2: Given `DEV_SKIP_ENTITLEMENT=true` in `.env.local`, when the Vite dev server starts, then all auth/entitlement checks are bypassed and a console warning is logged.
- AC3: Given `DEV_SKIP_ENTITLEMENT` is not set, when the Vite dev server handles `/api/ai/*` requests, then it proxies to Express on `:3001` (existing behavior) or applies middleware directly.
- AC4: Given `.env.example`, when a developer reads it, then all new env vars (`DEV_SKIP_ENTITLEMENT`) are documented.
- AC5: Given a 403 structured error from the server, when the client receives it, then the PremiumGate re-subscribe CTA is shown.
- AC6: Given `npm run build`, when the build completes, then there are no TypeScript errors.

## Tasks / Subtasks

- [x] Task 3: Update `.env.example` with `DEV_SKIP_ENTITLEMENT` documentation (AC: 4)
- [x] Task 4: Handle 403 structured error in LLM client to surface re-subscribe CTA via PremiumGate (AC: 5)

## Implementation Notes

- JSDoc added to `useIsPremium()` hook explicitly documenting it as a UX hint, not a security boundary. No logic changes to the function itself.
- `DEV_SKIP_ENTITLEMENT=true` support added to `ollamaDevProxy()` in `vite.config.ts`. When set, logs a yellow console warning on startup. The existing Vite `server.proxy` config continues to proxy `/api/ai/*` to Express on :3001, which is where the actual middleware chain runs.
- Added `ENTITLEMENT_ERROR`, `AUTH_REQUIRED`, and `RATE_LIMITED` error code handling to `useChatQA` hook. The 403 -> ENTITLEMENT_ERROR mapping was already implemented in `BaseLLMClient.mapHttpStatusToLLMErrorCode()` (E35-S02), so the client already receives structured errors. The hook now surfaces appropriate messages for each server-side middleware error.
- No new dependencies added.

## Testing Notes

- Existing proxy-client tests already cover the 403 -> ENTITLEMENT_ERROR mapping.

## Challenges and Lessons Learned

- **JSDoc-only changes require discipline**: The requirement was explicitly "no logic changes to isPremium()" -- only JSDoc additions. This is a good pattern for documentation-focused stories: clearly separating documentation from behavior changes prevents scope creep and makes reviews faster.
- **Error handling gap discovered in useChatQA**: While the `BaseLLMClient` already mapped 403 to `ENTITLEMENT_ERROR` (implemented in E35-S02), the consumer hook `useChatQA` did not handle this error code. The fix was minimal (3 additional switch cases) but highlights that end-to-end error handling requires checking both the producer (client) and consumer (hook) sides.
- **DEV_SKIP_ENTITLEMENT is an env var read at process startup**: Since Vite reads `process.env` at plugin initialization time, the flag is evaluated once when the dev server starts. Changing `.env.local` requires a server restart -- this is expected behavior and documented in the architecture doc.
- **Pre-existing Prettier drift**: Two files (`ollama-client.ts`, `proxy-client.ts`) had formatting drift unrelated to this story. Auto-fixing these in a separate commit keeps the story commit clean and reviewable.
