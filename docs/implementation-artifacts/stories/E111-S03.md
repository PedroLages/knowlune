# Story 111.3: Sleep Timer End of Chapter

## Story

As an audiobook listener,
I want the "End of chapter" sleep timer option to fade out and stop at the chapter boundary,
so that playback stops gracefully and my sleep isn't disturbed by an abrupt cut or a chapter that continues past my timer.

## Acceptance Criteria

- **AC-1**: Given the sleep timer is set to "End of chapter", when the current chapter ends, then audio fades out over 5 seconds and playback pauses (identical fade behavior to countdown timers)
- **AC-2**: Given the sleep timer is set to "End of chapter", when EOC is active, then a chapter progress bar is visible inside the sleep timer popover showing how far through the current chapter the listener is, so they can see when the timer will fire
- **AC-3**: Given the sleep timer is set to "End of chapter" for a single-file M4B audiobook, when the chapter boundary is crossed (custom `chapterend` event), then audio fades and pauses — same as AC-1
- **AC-4**: Given the sleep timer is set to a countdown (15/30/45/60/custom minutes), when the countdown reaches zero, then audio fades over 5 seconds, playback pauses, and on next app open a toast shows "Sleep timer ended" (existing behavior — regression test)
- **AC-5**: Given the sleep timer fires (any mode), when the user re-opens the app, then a toast notification appears: "Sleep timer ended" — and it appears only once (existing behavior — regression test)
- **AC-6**: The sleep timer selector, EOC progress indicator, and all controls are accessible via keyboard with proper ARIA labels and screen reader announcements

**EOC fade-out fix — before (`useSleepTimer.ts:89-109`):**

```typescript
if (option === 'end-of-chapter') {
  const handleChapterEnd = () => {
    const audio = audioRef.current
    if (!audio) return
    localStorage.setItem(LS_KEY, '1')
    setActiveOption(null)
    setRemainingSeconds(null)
    onPause()  // BUG: no fade-out, abrupt stop
  }
  const audio = audioRef.current
  if (audio) {
    audio.addEventListener('ended', handleChapterEnd)      // BUG: races with useAudioPlayer auto-advance
    audio.addEventListener('chapterend', handleChapterEnd)
  }

```

**EOC fade-out fix — after:**

```typescript
if (option === 'end-of-chapter') {
  const handleChapterEnd = () => {
    const audio = audioRef.current
    if (!audio) return
    fadeOutAndPause(audio, () => {
      localStorage.setItem(LS_KEY, '1')
      setActiveOption(null)
      setRemainingSeconds(null)
      onPause()
    })
  }
  const audio = audioRef.current
  if (audio) {
    // ONLY listen to 'chapterend' — NOT 'ended'
    // 'ended' races with useAudioPlayer auto-advance (useAudioPlayer.ts:173-189)
    // useAudioPlayer dispatches 'chapterend' AFTER handling the transition
    audio.addEventListener('chapterend', handleChapterEnd)
    eocCleanupRef.current = () => {
      audio.removeEventListener('chapterend', handleChapterEnd)
    }
  }
}
```

**Race condition detail**: `useAudioPlayer.ts:173-189` listens to `ended` and calls `loadChapterInternal(nextIndex, true)` to auto-advance. If `useSleepTimer` also listens to `ended`, both fire in unpredictable order — the next chapter may start playing before the pause takes effect. Fix: `useSleepTimer` EOC must ONLY listen to `chapterend` (dispatched by useAudioPlayer at line 258-260 after the transition is handled). For multi-file audiobooks, `useAudioPlayer` must also dispatch `chapterend` when auto-advancing — check if it already does (it currently dispatches `chapterend` only for single-file M4B at line 258). If not, add a `chapterend` dispatch in the multi-file `handleEnded` handler.

**Chapter progress computation in AudiobookRenderer:**

- `currentChapterIndex` from `useAudioPlayerStore`
- `currentTime` from `useAudioPlayerStore`
- For single-file M4B: `chapterStart = getChapterStartTime(book.chapters[currentChapterIndex])`, `chapterDuration = book.chapters[currentChapterIndex].duration`, `chapterProgress = (currentTime - chapterStart) / chapterDuration * 100`
- For multi-file: `chapterStart = 0`, `chapterDuration = book.chapters[currentChapterIndex].duration`, `chapterProgress = currentTime / chapterDuration * 100`
- Clamp to `[0, 100]`; pass `null` when `activeOption !== 'end-of-chapter'`

**`useSleepTimer` doesn't know about chapter state** — keep it that way. Pass computed `chapterProgressPercent` from AudiobookRenderer as a prop to `SleepTimer` directly; no need to add it to the hook.

**No new Dexie migration needed** — this is a pure hook/component change.

**`getChapterStartTime` utility** exists at `useAudioPlayer.ts:58-62` but is NOT exported (confirmed: internal function, 3 internal usages). Extract to `src/app/utils/audiobook-utils.ts` and update all 3 usages in `useAudioPlayer.ts` (lines 255, 388, 471) plus new usage in `AudiobookRenderer.tsx`.

**Chapter position type**: `book.chapters[i].position` uses `ContentPosition` from `src/data/types.ts:697-700`. For M4B: `{ type: 'time', seconds: number }`. For multi-file: `{ type: 'time', seconds: 0 }` (each file starts at 0). Check `position.type === 'time'` before accessing `.seconds`.

**Playwright visibility gotcha (E111-S02 lesson)**: Never use `opacity-0` for conditional visibility. Playwright's `toBeVisible()` returns `true` for `opacity-0` elements because they occupy layout space. Use `hidden`, Tailwind `invisible`, or conditional rendering instead.

## Challenges and Lessons Learned

### Race condition: `ended` vs `chapterend` in EOC sleep timer

The original EOC implementation listened to both `ended` and `chapterend`. For multi-file audiobooks, `useAudioPlayer` also listens to `ended` to auto-advance chapters — both handlers fire in unpredictable order. Fix: EOC mode now listens ONLY to `chapterend`, which `useAudioPlayer` dispatches *after* handling the transition. This gives a clear sequencing guarantee: advance first, then the sleep timer decides whether to intercept.

The `chapterend` event is cancelable (`cancelable: true`) — the sleep timer calls `event.preventDefault()` to signal that auto-advance should be blocked, then fades out instead.

### `getChapterStartTime` was an unexported internal — extraction required

The utility existed in `useAudioPlayer.ts` as a private function. To reuse it in `AudiobookRenderer.tsx` for computing chapter progress percent, it had to be extracted to `src/lib/audiobook-utils.ts`. Pattern: utility functions shared across two or more files belong in `src/lib/`, not inlined in a hook file.

### Fade-out state cleanup must happen in `onDone` callback, not before

The original EOC `handleChapterEnd` called `setActiveOption(null)` and `onPause()` synchronously before any fade. Correct pattern: call `fadeOutAndPause(audio, onDone)`, then in `onDone` write to localStorage, call `setActiveOption(null)`, and `onPause()`. Moving cleanup into `onDone` ensures state is consistent with the audio being paused — calling state setters before the fade completes causes a 5-second window where the timer shows as inactive but audio is still playing.

### Pre-existing unit test failures: blocked review gate

26 unit test failures across 10 files (unrelated to this story) blocked the pre-checks gate. They were already documented in `docs/known-issues.yaml` from E110-S02. Discovery: always check `docs/known-issues.yaml` before assuming a test failure is a regression introduced by the current story.
