# Story 101.1: AudiobookshelfService & Data Foundation

## Story

As a developer,
I want a typed REST API wrapper and Dexie schema for Audiobookshelf server connections,
so that all subsequent stories have a reliable, tested foundation to build on.

## Acceptance Criteria

**Given** the Dexie schema is at version 39
**When** the app initializes after this story ships
**Then** the schema upgrades to v40 with a new `audiobookshelfServers` table indexed on `id, name, url, status, lastSyncedAt`
**And** the `CHECKPOINT_VERSION` in `src/db/checkpoint.ts` is updated to 40
**And** the `CHECKPOINT_SCHEMA` in `src/db/checkpoint.ts` adds the `audiobookshelfServers` entry
**And** the checkpoint schema test in `src/db/__tests__/schema-checkpoint.test.ts` is updated to expect v40 and include `audiobookshelfServers` in the expected tables list

**Given** a valid Audiobookshelf server URL and API key
**When** `testConnection(url, apiKey)` is called
**Then** it returns `{ ok: true, data: { serverVersion: string } }` within the 10-second timeout
**And** the Authorization header is `Bearer <apiKey>`

**Given** an invalid API key
**When** `testConnection(url, apiKey)` is called and the server returns HTTP 401
**Then** it returns `{ ok: false, error: 'Authentication failed. Check your API key.' }`

**Given** a server URL that is unreachable or CORS-blocked
**When** `testConnection(url, apiKey)` is called and fetch throws TypeError
**Then** it returns `{ ok: false, error: 'Could not connect to server. Check the URL and CORS settings.' }`

**Given** the request takes longer than 10 seconds
**When** the AbortController fires
**Then** it returns `{ ok: false, error: 'Connection timed out. Check the URL and try again.' }`

**Given** a valid server connection
**When** `fetchLibraries(url, apiKey)` is called
**Then** it returns `{ ok: true, data: AbsLibrary[] }` with library id, name, and mediaType

**Given** a valid server and library ID
**When** `fetchLibraryItems(url, apiKey, libraryId, { page: 0, limit: 50 })` is called
**Then** it returns `{ ok: true, data: { results: AbsLibraryItem[], total: number } }`

**Given** any AudiobookshelfService function
**When** the function is called
**Then** it uses native fetch with no external HTTP library (NFR5)
**And** all API calls go through the shared `absApiFetch` helper with timeout and error handling
**And** the service file exports only pure functions (no class, no singleton)

**Given** the `AudiobookshelfServer` type
**When** it is defined in `src/data/types.ts`
**Then** it includes: `id, name, url, apiKey, libraryIds, status ('connected' | 'offline' | 'auth-failed'), lastSyncedAt?, createdAt, updatedAt`

**Given** the `Book` type in `src/data/types.ts`
**When** it is extended for ABS integration
**Then** it includes optional fields: `absServerId` (FK to AudiobookshelfServer) and `absItemId` (ABS item ID for dedup)

**Given** the AudiobookshelfService is implemented
**When** unit tests run with MSW mocking
**Then** all service functions are tested for success, auth failure, timeout, CORS error, and server error cases

### Architecture Decisions (Non-Negotiable)

**AudiobookshelfService pattern (mirrors OpdsService exactly):**
- Export pure functions — NO class, NO singleton, NO stored state
- Every async function signature: `(url: string, apiKey: string, ...params) => Promise<AbsResult<T>>`
- Every `fetch` call wrapped in `absApiFetch` internal helper with `AbortController`
- All catch blocks return `{ ok: false, error: 'user-friendly message' }` — NEVER throw
- Error message mapping (exact strings expected by E2E tests in later stories):
  - 401 → `'Authentication failed. Check your API key.'`
  - 403 → `'Access denied. Your API key may lack permissions.'`
  - `TypeError` (CORS/network) → `'Could not connect to server. Check the URL and CORS settings.'`
  - `AbortError` (timeout) → `'Connection timed out. Check the URL and try again.'`
  - Other non-ok response → `` `Server error (${response.status}). Try again later.` ``

**Naming conventions (critical — later stories depend on exact names):**
- File: `src/services/AudiobookshelfService.ts` (not `AbsService`, not `absService`)
- Type: `AudiobookshelfServer` (not `AbsServer`, not `ABSServer`)
- ABS API response types: prefix with `Abs` — `AbsLibrary`, `AbsLibraryItem`, `AbsItem`, `AbsResult`, `AbsProgress`, `AbsSearchResult`
- Dexie table name: `audiobookshelfServers` (camelCase plural, matches `opdsCatalogs`)

**ABS API endpoints (v2.26.0+ minimum — NFR15):**
- Server ping: `GET /api/ping` → `{ success: true, version: string }`
- Libraries: `GET /api/libraries` → `{ libraries: AbsLibrary[] }`
- Library items: `GET /api/libraries/{id}/items?limit=50&page=0` → `{ results: AbsLibraryItem[], total: number }`
- Single item: `GET /api/items/{id}` → `AbsItem`
- Search: `GET /api/libraries/{id}/search?q={query}` → `{ book: AbsLibraryItem[] }`
- User progress: `GET /api/me/progress/{itemId}` → `AbsProgress`
- Update progress: `PATCH /api/me/progress/{itemId}` → `AbsProgress`
- Stream direct: `GET /api/items/{id}/play?token={apiKey}` (used in getStreamUrl — NOT a service fetch)
- Cover image: `GET /api/items/{id}/cover` (used in getCoverUrl — NOT a service fetch)

**Authentication for HTML5 audio streaming:**
`getStreamUrl` uses `?token=` query parameter, NOT an `Authorization` header. This is required because HTML5 `<audio>` elements cannot send custom headers. The architecture explicitly chose this approach (Decision 5).

### ContentSource Type — Existing Remote Support

`ContentSource` already supports `type: 'remote'`:

```typescript
export type ContentSource =
  | { type: 'local'; opfsPath: string }
  | { type: 'remote'; url: string; auth?: { username: string; password: string } }
  | { type: 'fileHandle'; handle: FileSystemFileHandle }
```

ABS books will use `{ type: 'remote', url: ..., auth: undefined }` (auth is in `absServerId` FK, not inline). The auth object on `ContentSource` is unused for ABS streaming — authentication uses the `?token=` query param in the stream URL (not a header).

### Security: API Key Storage

API key is stored in `audiobookshelfServers.apiKey` in Dexie (IndexedDB). This matches the OPDS credentials pattern from E88. The `apiKey` field:
- Is never logged to console
- Is never included in error messages
- Is stored in IndexedDB only (not localStorage)
- Must be encrypted before any Supabase cloud sync (tracked separately — same note exists in OpdsCatalog)

Add a comment identical to the one in `OpdsCatalog.auth.password` noting plaintext-acceptable-for-local-first.

## Challenges and Lessons Learned

- **Timeout test requires fake timers + abort signal listener**: Using `vi.useFakeTimers()` alone is insufficient. The test must use `vi.advanceTimersByTimeAsync(10_001)` with a `controller.signal.addEventListener('abort', resolve)` in the MSW handler to actually observe the abort. Without the listener the promise never resolves and the test hangs.
- **SyntaxError must be caught separately from DOMException**: When `response.json()` fails on a malformed body, JavaScript throws `SyntaxError`, not a network error. The catch block needs an explicit `instanceof SyntaxError` check before the generic fallback — otherwise malformed responses are reported as the wrong error message.
- **Trailing-slash normalization belongs in `absApiFetch`, not callers**: Three separate fix commits were needed before centralizing the normalization in the shared helper. Apply `baseUrl.replace(/\/+$/, '')` once at the entry point and all derived URL builders inherit the fix.
- **Pure URL builder functions (`getStreamUrl`, `getCoverUrl`) don't go through `absApiFetch`**: These return plain strings for use in HTML5 `<audio src>` and `<img src>`, so they can't use the fetch helper. They need their own trailing-slash normalization since they bypass `absApiFetch`.
- **`Content-Type: application/json` header breaks GET requests**: The header should only be sent when a request body exists (`hasBody` guard). Sending it unconditionally caused linting and some server-side validation issues.
- **GLM-5.1 flagged `clearTimeout` before `response.json()` as a blocker**: This is a pattern inherited from `OpdsService` and is intentional — `AbortController` only aborts `fetch()`, not `.json()` parsing. Documenting this architectural decision in the cross-model review notes saved future re-investigation.
