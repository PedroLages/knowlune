# Story 71.1: Action Suggestion Data Layer

Status: ready-for-dev

## Story

As a learner with declining knowledge scores on some topics,
I want the system to compute ranked action suggestions based on my topic scores and available learning activities,
so that the most urgent remediation actions are identified and ready for display.

## Acceptance Criteria

1. **Given** a topic with a knowledge score of 35 (tier = 'weak') that has flashcard data, **When** `generateActionSuggestions()` is called with this topic, **Then** it returns an `ActionSuggestion` with `actionType: 'flashcard-review'`, `actionLabel: 'Review 5 flashcards on [topic]'`, `actionRoute: '/flashcards?topic={canonicalName}'`, and `estimatedMinutes: 5`.

2. **Given** a topic with a knowledge score of 55 (tier = 'fading') that has quiz data, **When** `generateActionSuggestions()` is called, **Then** it returns an `ActionSuggestion` with `actionType: 'quiz-refresh'`, `actionLabel: 'Take a refresher quiz on [topic]'`, `actionRoute: '/quiz?topic={canonicalName}'`, and `estimatedMinutes: 10`.

3. **Given** a topic with a knowledge score of 42 (tier = 'fading') that has lessons with varying completion, **When** `generateActionSuggestions()` is called, **Then** it returns an `ActionSuggestion` with `actionType: 'lesson-rewatch'` targeting the lesson with the lowest completion, `actionRoute: '/courses/{courseId}/lessons/{lessonId}'`, and `estimatedMinutes` matching lesson duration or 15 as default.

4. **Given** two topics with urgency scores of 80 and 45, **When** `generateActionSuggestions()` returns suggestions, **Then** the suggestion with urgency score 80 appears before the one with 45 (descending urgency order).

5. **Given** a topic that has both flashcard and quiz data (multiple action types possible), **When** `generateActionSuggestions()` processes it with deduplication, **Then** only one suggestion per topic is returned (the highest-priority action type).

6. **Given** FSRS stability data is provided via `options.fsrsStability` map, **When** urgency is calculated for a topic with FSRS stability of 5 (low), **Then** the `decayFactor` in the urgency formula uses the FSRS-derived value, producing a higher urgency score than a topic with stability of 50.

7. **Given** FSRS stability data is NOT provided (pre-E59 or no flashcards), **When** urgency is calculated, **Then** the existing recency decay model is used as fallback (`decayFactor = 100 - recencyScore`).

8. **Given** `generateActionSuggestions()` is called with `maxSuggestions: 3`, **When** 7 topics have declining scores, **Then** only 3 suggestions are returned (the 3 most urgent).

9. **Given** an empty array of topics or all topics with strong scores (>= 70), **When** `generateActionSuggestions()` is called, **Then** it returns an empty array.

10. **Given** a topic that has lessons but no flashcard data and no quiz data, **When** action suggestions are generated, **Then** only a 'lesson-rewatch' suggestion is produced (no flashcard or quiz suggestions for this topic).

## Dev Notes

### Architecture Compliance

**Pure function boundary (CRITICAL):** `src/lib/actionSuggestions.ts` must have ZERO React, ZERO Zustand, and ZERO Dexie imports. It is a pure computation module following the same pattern as `src/lib/qualityScore.ts`. All data is passed in as plain objects.

**Pattern reference:** `src/lib/qualityScore.ts` — uses exported `WEIGHTS` constant, pure calculation functions, and a `QualityScoreResult` return type with `tier` classification. Follow this exact structure.

### Urgency Formula

```
urgencyScore = (100 - score) * 0.6 + decayFactor * 0.4
```

Where `decayFactor`:
- **With FSRS:** `Math.max(0, 100 - fsrsStability * 2)` — stability 0 → decay 100, stability 50+ → decay 0
- **Without FSRS (fallback):** `100 - recencyScore` — lower recency = higher urgency

### Action Type Priority (for deduplication)

1. `flashcard-review` (highest — fastest remediation, 5 min)
2. `quiz-refresh` (medium — diagnostic + practice, 10 min)
3. `lesson-rewatch` (lowest — most time-consuming, 15 min)

### Tier Thresholds

- Strong: score >= 70 (excluded from suggestions)
- Fading: score 40-69
- Weak: score < 40

### Dependencies

- **E56 (Knowledge Map Phase 1):** This story defines the `TopicWithScore` input interface to match E56's `useKnowledgeMapStore` output. E56 is NOT yet implemented — the types defined here must be compatible with E56's eventual shape. The `ScoredTopic` type from E56-S02 includes: `topicName`, `canonicalName`, `score`, `tier`, `confidence`, `lastEngagement`, and score breakdown fields.
- **E59 (FSRS Migration):** Optional. When FSRS data is available, it's passed via `options.fsrsStability` map. Feature works without it.
- **No new npm dependencies required.**
