# Story 85.1: Text Selection and Highlight Creation

Status: done

## Story

As a learner,
I want to select text in the EPUB reader and save it as a colored highlight,
so that I can mark important passages for later review.

## Acceptance Criteria

1. **Given** the user is reading an EPUB in the reader **When** the user selects text (click-drag on desktop, long-press on mobile) **Then** epub.js fires the `selected` event with a CFI range **And** a HighlightPopover appears above the selection (or below if no space, full-width bar on mobile) **And** the popover contains: 4 color circle buttons (yellow #FFEB3B, green #4CAF50, blue #42A5F5, pink #EC407A), a separator, note button (StickyNote icon), flashcard button (Layers icon), and close button (X icon) **And** the popover has `role="toolbar"` with `aria-label="Highlight actions"` and each button has an `aria-label` **And** color buttons are `size-10` on mobile for 44px touch targets

2. **Given** the HighlightPopover is showing **When** the user taps a color button **Then** a BookHighlight record is created in Dexie with: the CFI range as `cfiRange`, a text anchor object `{ prefix, exact, suffix }` with 30 chars context each, the selected text as `text`, the chosen color, and timestamps **And** highlight creation completes in <200ms (NFR2) **And** the highlight is rendered as a colored overlay on the epub.js content via `rendition.annotations.highlight()` with 0.3 opacity and multiply blend mode **And** the popover closes and the selection clears **And** a brief yellow flash animation (200ms) confirms the highlight

3. **Given** the user is on iOS Safari and epub.js text selection is non-functional **When** attempting to highlight text **Then** the reader shows a dismissible informational banner: "Text selection is limited on iOS Safari. For best highlighting experience, use Chrome on desktop or Android." The banner does not block reading functionality.

## Dev Notes

### Architecture Constraints

- **Dual position storage is mandatory** (architecture decision 3): Every highlight MUST store both `cfiRange` (primary, fast) AND `textAnchor: { prefix, exact, suffix }` (fallback, resilient). Never CFI alone.
- **epub.js annotations API**: `rendition.annotations.highlight(cfiRange, data, callback, className, styles)` is the correct API. The callback fires when the annotation is tapped (used in E85-S02 for edit/delete popover).
- **Text anchor extraction**: Get the epub.js chapter content to extract 30-char prefix/suffix around the selection. This context enables recovery if the EPUB is re-imported with a different edition.
- **Highlight colors are NOT design tokens** — these are reader-specific content overlay colors (yellow/green/blue/pink at 30% opacity with multiply blend). They appear inside the epub.js iframe, not in Knowlune UI chrome.
- **Event bus**: Emit `highlight:created` event (kebab-case with namespace prefix) consistent with existing event bus pattern (`src/lib/eventBus.ts`).

### epub.js iOS Text Selection Bug (Upstream)

epub.js GitHub issue [#904](https://github.com/futurepress/epub.js/issues/904) — text selection is broken on iOS iPad Safari for many EPUBs. The selection drag region collapses to one character and cannot be extended. This is an **unresolved upstream bug** since 2019.

**Detection:** Check `navigator.platform === 'iPad' || (navigator.userAgent.includes('Mac') && navigator.maxTouchPoints > 1)` at reader initialization. If true, and text selection fails within 2s of user interaction, show the informational banner.

**Impact:** The highlight-to-flashcard pipeline is non-functional on affected iOS devices. Users can still read, navigate, and use all other features.

### Performance Requirements

- Highlight creation (selection → persisted in Dexie) must complete in <200ms (NFR2)
- Highlight overlay rendering should not cause visible jank
- Restoring existing highlights on book open: batch all annotations, do NOT render one-by-one with delays

### Mobile Considerations (UX-DR19)

- HighlightPopover is larger on mobile: `min-h-[48px]` with bigger touch targets (`size-10` color buttons for 44x44px)
- Positioned to avoid mobile keyboard coverage
- Full-width bottom bar variant on mobile for easier reach

### Accessibility (NFR15, UX-DR22)

- HighlightPopover: `role="toolbar"` with `aria-label="Highlight actions"`
- Color buttons: `aria-label="Highlight yellow"`, `aria-label="Highlight green"`, etc.
- Note/flashcard/close buttons: descriptive `aria-label`
- Focus management: popover receives focus on open, returns focus to content on close
- Keyboard: Tab through toolbar buttons, Enter/Space to activate
