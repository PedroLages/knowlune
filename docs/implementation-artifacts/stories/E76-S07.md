# Story 76.7: Send Test Email Integration

Status: ready-for-dev

## Story

As a user,
I want to send myself a test digest email from Settings,
so that I can preview what the weekly digest looks like before enabling it.

## Acceptance Criteria

1. **Given** the "Send Test Email" button in the Email Digest section
   **When** the user clicks it
   **Then** it calls the `send-weekly-digest` Edge Function with `?test=true&user_id=<current_user_id>`
   **And** the button shows a loading spinner during the request
   **And** the button is disabled during the request to prevent double-sends

2. **Given** the test email request succeeds
   **When** the response is received
   **Then** a success toast appears: "Test email sent! Check your inbox."
   **And** the button returns to its default state

3. **Given** the test email request fails
   **When** the error is caught
   **Then** an error toast appears with the failure reason
   **And** the button returns to its default state

4. **Given** the `send-weekly-digest` Edge Function receives `?test=true`
   **When** processing the request
   **Then** it sends a digest email ONLY to the specified user (ignores batch logic)
   **And** it does NOT update `last_digest_sent_at` (test sends don't affect scheduling)
   **And** it sends even if `has_activity = false` (so user can see the template with sample/current data)

5. **Given** the "Send Test Email" button
   **When** the email digest toggle is OFF
   **Then** the button is not visible (hidden with the rest of the conditional section)

6. **Given** the button component
   **When** rendered
   **Then** it uses `variant="brand-outline"` with a leading `Send` icon from lucide-react
   **And** has `aria-busy="true"` during loading state

## Dev Notes

### Architecture Decisions

- **AD-8**: "Send Test Email" button calls the existing `send-weekly-digest` Edge Function with `?test=true` — no separate Edge Function needed for test sends.
- Test mode bypasses: batch user query (sends only to requesting user), `has_activity` check (sends even with no activity), `last_digest_sent_at` update (doesn't affect scheduling).

### Implementation Pattern

The test email button invokes the Edge Function via the Supabase client:

```typescript
const { data, error } = await supabase.functions.invoke('send-weekly-digest', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ test: true, userId: currentUserId }),
})
```

Alternatively, use query params: `send-weekly-digest?test=true&user_id=<id>`

### Auth Considerations

- **pg_cron invocation**: Uses service_role_key (server-to-server)
- **Test send from UI**: Uses the user's Supabase session JWT (client-to-server)
- The Edge Function must handle BOTH auth methods:
  - If `Authorization: Bearer <service_role_key>` → batch mode
  - If `Authorization: Bearer <user_jwt>` with `?test=true` → test mode for that user only

### Styling

- Button: `variant="brand-outline"` (per styling.md brand button variants)
- Leading icon: `Send` from lucide-react
- Loading state: show spinner in button (use shadcn Button's built-in loading pattern or a `Loader2` icon with `animate-spin`)
- Do NOT use hardcoded colors — follow design token system

### Critical Constraints

- The button is ONLY visible when `emailDigestEnabled = true` (it's inside the conditional reveal section from E76-S06).
- Do NOT create a separate Edge Function for test sends — reuse `send-weekly-digest` with `?test=true`.
- Test sends must NOT update `last_digest_sent_at` — this would break the next scheduled send's idempotency check.
- Loading state must be resilient: always reset on both success and error (use try/finally).

### Dependencies

- E76-S04 (Edge Function) — `send-weekly-digest` must exist with test mode support
- E76-S06 (UI) — button placeholder rendered in NotificationPreferencesPanel
