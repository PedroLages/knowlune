# Story 80.03: API Rate Limiting

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a Knowlune platform operator,
I want API requests to be rate-limited per token,
so that no single token can overwhelm the API.

## Acceptance Criteria

1. **Given** an API token with default rate limits, **When** a consumer makes requests, **Then** the token bucket allows 1000 requests per hour with burst of 50 requests per 10-second window.

2. **Given** rate limit headers, **When** each API response is returned, **Then** it includes `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset` headers.

3. **Given** a token exceeds its rate limit, **When** the next request arrives, **Then** a 429 Too Many Requests response is returned with `Retry-After` header indicating seconds until the next allowed request.

4. **Given** the rate limiter, **When** tracking request counts, **Then** it uses a `rate_limits` table in Supabase (token_hash, window_start, request_count) checked by the Edge Function before forwarding to PostgREST.

## Dev Notes

- **Architecture**: Token bucket algorithm via Supabase Edge Function middleware. Default: 1000 req/hour per token, 50 req/10s burst. [Source: architecture-api-plugin-system.md#Decision 12]
- **Dependency on E80-S02**: Extends the `verify-api-token` Edge Function created in E80-S02. Rate limit check happens after token validation but before proxying to PostgREST.
- **Atomic increment**: Use PostgreSQL's `INSERT ... ON CONFLICT ... DO UPDATE` for atomic counter increments. This prevents race conditions with concurrent requests.
- **Window calculation**: Hourly window uses `date_trunc('hour', now())` as `window_start`. Burst window uses `date_trunc('second', now()) - (extract(second from now())::int % 10) * interval '1 second'` to get 10-second window start.
- **Cleanup strategy**: Old rate limit windows can be cleaned via pg_cron (`DELETE FROM rate_limits WHERE window_start < now() - interval '2 hours'`). This prevents table bloat.
- **Headers on 429**: Even the 429 response must include rate limit headers so the consumer knows when they can retry.
- **NFR5**: API p95 < 200ms. The rate limit check adds a DB query, so ensure the `(token_hash, window_start)` index is used for fast lookups.
