# Story 63.3: Prompt Builder Slot 6 Integration

## Story

As a learner using the AI tutor,
I want the tutor to understand my learning strengths and weaknesses when answering my questions,
So that I receive personalized guidance that addresses my specific struggles rather than generic responses.

## Acceptance Criteria

**Given** the `tutorPromptBuilder` assembles a system prompt for a 4K Ollama model
**When** the LEARNER_PROFILE slot (slot 6) is populated via `buildLearnerProfile`
**Then** the slot content fits within 40 tokens
**And** the total system prompt does not exceed the 4K token budget

**Given** the `tutorPromptBuilder` assembles a system prompt for a 128K model
**When** the LEARNER_PROFILE slot is populated
**Then** the slot content uses up to 100 tokens of richer profile data

**Given** E52 quiz system is not yet implemented and `db.quizAttempts` is empty
**When** `buildLearnerProfile` is called during prompt assembly
**Then** no error is thrown
**And** the quiz section is simply omitted from the profile
**And** other available signals (flashcards, study sessions) still appear

**Given** none of the upstream epics (E52, E56, E59) are implemented and all data sources are empty
**When** `buildLearnerProfile` is called during prompt assembly
**Then** it returns an empty string
**And** the prompt builder omits slot 6 entirely (existing behavior for empty optional slots)

**Given** the tutor prompt builder calls `buildLearnerProfile` asynchronously
**When** the profile is built
**Then** the builder properly awaits the Promise before assembling the final system prompt

**Given** course tags are available from `courseTagger.ts` for the current course
**When** the prompt builder calls `buildLearnerProfile`
**Then** it passes course tags as `lessonTopics` for topic-scoped profile filtering

## Dev Notes

### Architecture: E57 Slot-Based Prompt Builder
The `tutorPromptBuilder.ts` (from E57) uses a 7-slot priority system:
- Slots: 1=base, 2=mode, 3=course (required); 4=rag, 5=transcript, 6=learner, 7=resume (optional)
- Optional slots omitted when token budget exceeded (lower priority = dropped first)
- Slot 6 = 'learner' (optional — auto-omitted under 4K token pressure)
- Current implementation: `buildLearnerSlot()` returns empty string (placeholder)
- `buildTutorSystemPrompt()` is synchronous — learner profile content must be pre-resolved and passed in, or the function signature must be made async

### Integration Points

**Files to modify:**
- `src/ai/tutor/tutorPromptBuilder.ts` — add `learnerProfile` param to `buildTutorSystemPrompt()`, use it in `buildLearnerSlot()` (currently a placeholder returning `''`)
- `src/ai/hooks/useTutor.ts` — in stage 3, call `await buildLearnerProfile()` before `buildTutorSystemPrompt()`, pass result as the new param

**File to import from:** `src/ai/tutor/learnerProfileBuilder.ts` (created in S01/S02)
- `buildLearnerProfile(config: ProfileBuilderConfig): Promise<string>`

**File to import from:** `src/ai/courseTagger.ts`
- Provides course tags that map to lesson topics for topic-scoped filtering

### Fallback Strategy
When `buildLearnerProfile` returns empty string (no data from any source):
- Slot 6 should still include minimal progress data: "Progress: 45%, Lesson 5/12, Streak: 7 days"
- This preserves the existing behavior from before E63
- Only when the enriched profile returns non-empty content does it replace/augment the basic progress line

### Critical: Async Handling
`buildLearnerProfile` returns a Promise (because aggregation functions query Dexie). The prompt builder must `await` this call. If `tutorPromptBuilder.ts` currently generates slot content synchronously, the preferred approach is to resolve the profile in `useTutor.ts` (stage 3, already async) and pass the result string into `buildTutorSystemPrompt()` via a new `learnerProfile` parameter.

- Soft: E52, E56, E59 — data sources (all handled via graceful degradation)
