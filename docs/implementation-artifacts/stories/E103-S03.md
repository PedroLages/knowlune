# Story 103.3: Dual Position Tracking

Status: ready-for-dev

## Story

As a Knowlune user who reads and listens to the same book,
I want my reading and listening positions tracked independently,
So that I can progress at different rates in each format without losing my place.

## Acceptance Criteria

**Given** a user has both EPUB and audiobook versions of a book
**When** they read the EPUB to chapter 8 and listen to the audiobook to chapter 5
**Then** the EPUB Book record stores `currentPosition: { type: 'cfi', value: '...', percent: 65 }`
**And** the audiobook Book record stores `currentPosition: { type: 'time', seconds: 12345 }`
**And** each position is independent — updating one does not affect the other (FR42)

**Given** dual positions are tracked
**When** the user views either book in the Library
**Then** the progress shown reflects that specific format's position
**And** a small badge indicates: "Also available as audiobook" (on the EPUB) or "Also available as EPUB" (on the audiobook)

**Given** a chapter alignment is computed for a book pair (by E103-S01)
**When** the alignment result is stored
**Then** the chapter mapping is cached in Dexie — computed once per book pair and persisted indefinitely
**And** the alignment cache is deterministic: the same ebook + same audiobook always produces the same mapping
**And** the chapter mapping is stored in a `chapterMappings` Dexie table indexed on `epubBookId, audioBookId`

**Given** the user switches format via the format switch button (E103-S02)
**When** they switch from EPUB chapter 8 to audiobook
**Then** the audiobook resumes at the MATCHED chapter (chapter 8 equivalent), not at the audiobook's independent position (chapter 5)
**And** after switching, the independent audiobook position updates to the new location
**And** the EPUB position is preserved at chapter 8 (not cleared)

### New Dexie Table: `chapterMappings`

```typescript
// src/data/types.ts — add this type
export interface ChapterMapping {
  id: string                         // UUID
  epubBookId: string                 // FK to Book.id (EPUB format)
  audioBookId: string                // FK to Book.id (audiobook format)
  mappings: {
    epubChapterHref: string          // EPUB chapter src href
    audioChapterIndex: number        // 0-based index into audiobook Book.chapters
    confidence: number               // Jaro-Winkler score 0-1
  }[]
  createdAt: string                  // ISO 8601
  updatedAt: string                  // ISO 8601
}
```

Schema index: `'id, epubBookId, audioBookId, [epubBookId+audioBookId]'`

The compound index `[epubBookId+audioBookId]` enables efficient lookup by book pair without scanning the whole table.

### `useChapterMappingStore` Pattern

Must follow `useOpdsCatalogStore` pattern exactly:

```typescript
// src/stores/useChapterMappingStore.ts
import { create } from 'zustand'
import { db } from '@/db/db'
import { ChapterMapping } from '@/data/types'
import { toast } from 'sonner'

interface ChapterMappingState {
  mappings: ChapterMapping[]
  isLoaded: boolean
  loadMappings: () => Promise<void>
  saveMapping: (epubBookId: string, audioBookId: string, mappings: ChapterMapping['mappings']) => Promise<void>
  getMapping: (epubBookId: string, audioBookId: string) => ChapterMapping | undefined
  deleteMapping: (id: string) => Promise<void>
}

export const useChapterMappingStore = create<ChapterMappingState>((set, get) => ({
  mappings: [],
  isLoaded: false,
  loadMappings: async () => {
    if (get().isLoaded) return  // isLoaded guard — prevents redundant DB reads
    const mappings = await db.chapterMappings.toArray()
    set({ mappings, isLoaded: true })
  },
  // ... rest follows useOpdsCatalogStore pattern
}))
```

Individual selectors at call sites — NEVER destructure the full store:
```typescript

const getMapping = useChapterMappingStore(state => state.getMapping)
```

### "Also available as" Badge — Design Token Rules

Do NOT use hardcoded colors. Use only design tokens:
- `bg-brand-soft` + `text-brand-soft-foreground` for a brand-colored badge

### Zustand Optimistic Update Pattern

When updating `Book.currentPosition` during position saves:

```typescript
// In useBookStore (or inline in the hook that saves position):
// 1. Optimistic Zustand update (instant UI feedback)
set(state => ({ books: state.books.map(b => b.id === id ? { ...b, currentPosition, progress } : b) }))
// 2. Dexie persist
await db.books.update(id, { currentPosition, progress, lastOpenedAt: new Date().toISOString() })
// 3. Rollback on failure
catch (err) {
  set(/* restore previous state */)
  toast.error('Failed to save reading position')
}
```

### No Merging of EPUB + Audiobook Records

The EPUB Book record and the audiobook Book record are **separate Dexie rows**. They share the same title/author but have different `id`s, different `format` values (`'epub'` vs `'audiobook'`), and entirely separate `currentPosition` fields.

The `linkedBookId` field is the ONLY connection between them. Do not add a combined "dual position" object or any cross-contaminating state.

## Code Review Feedback

**Reviewed 2026-04-06**

### STORY-RELATED Issues

1. **[MEDIUM] TypeScript narrowing workaround in BookCard.tsx** — IIFE used to work around TS inability to narrow discriminated unions via computed array index. Fixed during pre-checks. The pattern works but is unusual; a named helper function would be cleaner.

2. **[MEDIUM] `linkBooks` non-atomic Dexie writes** — `useBookStore.linkBooks()` performs two sequential `db.books.update()` calls. If the first succeeds and the second fails, books are in an inconsistent state (A links to B but B does not link to A). Should use `db.transaction()` for atomicity.

3. **[LOW] `new Date().toISOString()` in store methods** — `updateBookPosition` and `linkBooks` both call `new Date()` directly. This is fine for production but makes unit testing harder. Consider accepting an optional `now` parameter or using a testable time utility.

4. **[LOW] `as Parameters<typeof db.books.update>[1]` type assertions** — Used 4 times in useBookStore.ts to bypass Dexie typing. This suggests the Dexie table typing doesn't include `linkedBookId` yet (schema may need updating). Not a runtime issue but a type safety gap.

## Challenges and Lessons Learned

- TypeScript discriminated union narrowing does not work through computed array indices (`arr[i + 1].position.type`). Workaround: extract to a local variable or IIFE before the type guard check.
- Dexie `db.books.update()` does not natively type-check new fields added to the interface but not yet in the schema index. Using `as Parameters<>` casts is a pragmatic workaround but should be tracked as tech debt.
- The `chapterSwitchResolver.ts` module is a clean pure-function utility with 12 unit tests and 100% coverage of its logic paths. Good pattern for testable domain logic separated from React.
