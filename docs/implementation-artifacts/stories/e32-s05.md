# Story 32.05: Create Dexie migration checkpoint at v27

## Story

As a developer,
I want a migration checkpoint system that snapshots the full v27 schema,
so that fresh installs skip replaying 27 incremental migrations and new users get faster first-load times.

## Acceptance Criteria

- AC1: Given a fresh install (no existing IndexedDB), when the app initializes, then the full v27 schema is created from the checkpoint constant in a single version declaration
- AC2: Given an existing user at a lower schema version, when they upgrade, then incremental migrations run normally through the legacy version chain
- AC3: Given the checkpoint schema constant, when compared to the schema produced by running all 27 migrations sequentially, then both produce identical table names and indexes
- AC4: Given a `CHECKPOINT_VERSION = 27` constant, then it matches the current latest Dexie version
- AC5: Given the checkpoint module, then it exports `CHECKPOINT_SCHEMA` with all 29 tables and their index definitions

## Implementation Notes

- Dexie already optimizes fresh installs by not running `.upgrade()` callbacks — the checkpoint provides an additional optimization by consolidating schema declarations
- The `_declareLegacyMigrations()` function encapsulates all 27 version declarations with their upgrade callbacks
- `createCheckpointDb()` creates a DB with only the checkpoint version — useful for testing and potential async initialization paths
- The `ElearningDatabase` type is exported for external typing of DB instances
- Re-exported constants and functions from `db/index.ts` for convenient imports

## Testing Notes

  4. Round-trip validation: checkpoint schema constant produces expected tables when Dexie creates from it

## Challenges and Lessons Learned

- Dexie's architecture means `.upgrade()` callbacks are already skipped for fresh installs. The real value of the checkpoint is: (1) a single source-of-truth schema constant, (2) the `createCheckpointDb()` factory for testing/benchmarking, and (3) a foundation for future schema version consolidation.
- When extracting the migration chain into a function, all `db.version()` calls needed updating to `database.version()` — a mechanical but error-prone refactor that the existing 56 tests caught any regressions in.
- The schema equivalence test (comparing extracted index strings) is the critical invariant — it will catch any drift between the checkpoint constant and the actual migration chain.
