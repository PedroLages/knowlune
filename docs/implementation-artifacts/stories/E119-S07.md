# Story E119-S07: Consent Schema + Inventory

## Story

As Pedro (controller/operator),
I want a `user_consents` table and a `docs/compliance/consent-inventory.md` documenting every processing purpose with its lawful basis,
so that consent records are durable and the purpose inventory is the single source of truth for the UI (S08).

## Acceptance Criteria

- **AC-1**: `docs/compliance/consent-inventory.md` lists every processing purpose with: purpose key, description, lawful basis (contract / legitimate_interest / consent), data categories, processor(s), default state, withdrawable (y/n).
- **AC-2**: Initial purposes: `ai_tutor` (consent), `ai_embeddings` (consent), `voice_transcription` (consent), `analytics_telemetry` (consent, off by default), `marketing_email` (consent, off by default). Core learning/sync/auth/billing â†’ `contract`, no toggle.
- **AC-3**: Supabase migration creates `user_consents(user_id, purpose, granted_at, withdrawn_at, notice_version, evidence JSONB)` with unique `(user_id, purpose)` and RLS owner-only.
- **AC-4**: `tableRegistry.ts` gains a `user_consents` entry (synced, P0 or P1).
- **AC-5**: `src/lib/compliance/consentService.ts` exposes `isGranted(userId, purpose)` (fail-closed: unknown purpose â†’ false + warning log) and `listForUser(userId)`.
- **AC-6**: A test asserts that every purpose key in `consent-inventory.md` exists in `consentService` purpose enum (and vice versa).
- **AC-7**: `user_consents` syncs bidirectionally to Dexie; LWW resolves conflicting grant/withdraw timestamps correctly.
