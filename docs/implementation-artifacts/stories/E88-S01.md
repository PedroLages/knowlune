# Story 88.1: OPDS Catalog Connection

Status: ready-for-dev

## Story

As a power user with a self-hosted library,
I want to connect Knowlune to my Calibre-Web OPDS catalog by entering a URL,
so that I can access my existing book collection without re-importing files.

## Acceptance Criteria

**Given** the user opens Library settings (accessible from Library page header menu)
**When** the user enters an OPDS catalog URL (e.g., `https://calibre.local/opds`)
**Then** Knowlune fetches the catalog root feed and validates it is a valid OPDS (Atom/XML) document using the browser's native DOMParser
**And** if valid, the catalog is saved with a user-assigned name and optional basic auth credentials
**And** if invalid or unreachable, a clear error message appears ("Could not connect to OPDS catalog. Check the URL and try again.")
**And** the user can manage multiple OPDS connections (add, edit, remove)

## Dev Notes

### OPDS Format

OPDS (Open Publication Distribution System) catalogs are Atom/XML feeds. The root feed looks like:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<feed xmlns="http://www.w3.org/2005/Atom"
      xmlns:opds="http://opds-spec.org/2010/catalog">
  <title>My Library</title>
  <entry>
    <title>Book Title</title>
    <author><name>Author Name</name></author>
    <link rel="http://opds-spec.org/acquisition" href="/download/123.epub" type="application/epub+zip"/>
    <link rel="http://opds-spec.org/image" href="/cover/123.jpg"/>
  </entry>
</feed>
```

**Validation**: Check for `<feed>` root element with Atom namespace. Check for at least one `<entry>` or OPDS navigation link.

### CORS Considerations

- Calibre-Web and most self-hosted OPDS servers do NOT set CORS headers by default
- Users will need to either: configure CORS on their server, or use a reverse proxy with CORS headers
- Surface this clearly in the error message — don't just show "Network error"

### No New Dependencies

- Use browser native `DOMParser` for XML parsing — no XML library needed
- Use `fetch` for HTTP requests — no HTTP library needed
- Authentication: HTTP Basic Auth via `Authorization` header

### Dependencies

- **Hard dependency on E83**: Library page and infrastructure

## Challenges and Lessons Learned

- **Zero-dependency XML parsing works well**: Browser-native `DOMParser` handles OPDS Atom/XML validation without needing any XML library. The `parsererror` element detection pattern is reliable for catching malformed XML. This is a good pattern for future lightweight protocol integrations.

- **CORS is the main deployment friction**: Self-hosted OPDS servers (Calibre-Web, COPS) do not set CORS headers by default. The service distinguishes `TypeError` (network/CORS) from other errors to surface actionable guidance ("Check CORS settings") rather than generic "Network error" messages. Future stories should consider a proxy option.

- **Discriminated union pattern for service results**: Using `{ ok: true; meta } | { ok: false; error }` instead of throwing exceptions simplifies UI error handling — no try/catch needed in components, and TypeScript narrows the type automatically. This pattern should be standard for all service layer functions.

- **Schema checkpoint test pattern**: The `schema-checkpoint.test.ts` validates that the Dexie version and table list match expectations after each schema migration. This catches accidental version conflicts between stories that touch `schema.ts` concurrently.

- **Component size warning at 402 lines**: `OpdsCatalogSettings.tsx` hit the 300-line ESLint warning threshold. The form, list, and dialog are all in one component. A future story should extract the catalog list and form into separate components.
