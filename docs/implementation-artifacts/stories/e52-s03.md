# Story 52.03: Quiz Quality & Feedback

## Story

As a learner,
I want generated quiz questions to be high-quality and relevant to the transcript, with the ability to provide feedback and regenerate,
so that I can trust the quizzes test real comprehension and improve over time.

## Acceptance Criteria

**Given** the LLM generates a set of quiz questions for a chunk
**When** the QC pipeline runs
**Then** duplicate questions are detected via embedding cosine similarity (>0.85 threshold)
**And** duplicate questions are removed from the set

**Given** a generated question
**When** the QC pipeline validates answer uniqueness
**Then** it verifies `correctAnswer` is present in `options`
**And** all `options` are distinct strings

**Given** a generated question
**When** the QC pipeline checks transcript grounding
**Then** key terms from the question text must appear in the source transcript chunk
**And** questions that fail grounding check are flagged for retry

**Given** a chunk fails QC validation
**When** the system retries generation
**Then** it retries up to 2 times with the same chunk
**And** if all retries fail, the chunk is skipped and remaining valid questions still produce a quiz

**Given** a quiz is displayed to the learner
**When** the learner views an individual question
**Then** thumbs up and thumbs down feedback buttons are visible on each question
**And** clicking a feedback button stores the feedback locally in the quiz data

**Given** the learner wants fresh questions for a lesson
**When** they click "Regenerate"
**Then** a new quiz is generated (new `id`) while the previous quiz is preserved
**And** the learner can access both the old and new quiz

**Issues found during review-story:**

- [BLOCKER] Unit test `"generates quiz via LLM when no cache hit"` in `quizGenerationService.test.ts` fails — mock transcript uses `word0..word299` but mock LLM response mentions React/Virtual DOM. QC grounding check rejects all questions. Test needs updating to use matching transcript/question content.
- [HIGH] Optimistic UI update in `QuestionFeedback.tsx` violates the project pre-review checklist rule ("No optimistic UI updates before persistence"). `setOptimistic(value)` is called before `onFeedback()` succeeds — if persist fails, UI shows feedback given but nothing was stored.
- [MEDIUM] `QuizReviewContent.tsx` stores feedback in `questionFeedback` field via `as any` cast — this field is not declared in the `Quiz` type or Dexie schema, creating a type safety gap and potential schema drift.
- [MEDIUM] `GenerateQuizButton`: when `cachedQuiz` exists but `onRegenerate` is not provided (optional prop), click falls through to `onGenerate` — showing "Regenerate Quiz" label/icon but performing a regular (cached) generate, effectively a no-op with misleading UX.
- [LOW] No unit test coverage for grounding threshold boundary (exactly 30% match ratio).

## Challenges and Lessons Learned

- **QC transcript grounding breaks existing tests**: Adding a grounding check to the QC pipeline caused a pre-existing test to fail because the test used generic `word0..wordN` transcript text while the mock LLM response referenced domain terms (React, Virtual DOM). The fix requires using semantically consistent mock data — both transcript text and generated questions must share key terms. Pattern: when adding validation to a generation pipeline, audit all existing mocks.

- **Dexie schema vs runtime fields**: Storing `questionFeedback` as an unindexed, untyped field via `as any` is pragmatic but creates hidden coupling between `QuizReviewContent` and the DB record shape. Better pattern: declare optional fields in the Quiz type definition and add them to the Dexie schema table spec, even if unindexed.

- **Optimistic vs. confirmed state for feedback**: Using `setOptimistic` before await creates a divergence if persist fails. The project engineering checklist explicitly prohibits this. The safer pattern is: await the DB write first, only update UI state on success, and use a toast on failure without reverting (since toast IS the error feedback).

- **Optional prop safety nets**: Making `onRegenerate` optional in `GenerateQuizButton` required careful fallback logic — the current fallback (`onGenerate`) silently does the wrong thing when a cached quiz exists. Better pattern: require the prop when the feature flag is on, or use a no-op fallback that shows a toast ("Regenerate not available").

- **Text-based cosine similarity is effective for deduplication**: Using TF-IDF-like term frequency vectors (no embeddings needed) correctly identifies near-duplicate questions at 0.85 threshold. This avoids expensive LLM calls or vector DB lookups for QC, which is the right tradeoff for a deterministic pipeline.
