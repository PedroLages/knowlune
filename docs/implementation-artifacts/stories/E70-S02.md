# Story 70.2: Multi-Signal Scoring Algorithm and Productivity Heatmap

Status: ready-for-dev

## Story

As a learner,
I want study time suggestions ranked by how well each slot fits my productivity patterns and course urgency,
so that the recommended times maximize my learning effectiveness.

## Acceptance Criteria

1. **Given** the `productivityHeatmap` module exists at `src/lib/scheduling/productivityHeatmap.ts`, **When** the user has 10 or more completed study sessions in the last 90 days, **Then** a `ProductivityHeatmap` is computed by grouping sessions by `dayOfWeek` and `hourOfDay`, computing average `qualityScore` per cell, and normalizing to 0-1 range. **And** the heatmap is cached in the scheduling Zustand store and recomputed daily or on new session completion.

2. **Given** the user has fewer than 10 completed study sessions, **When** the productivity heatmap is requested, **Then** circadian science defaults are used: morning (9-11am) = 0.9, early afternoon (1-3pm) = 0.6, evening (7-9pm) = 0.7, late night (11pm-6am) = 0.3. **And** no error is raised -- this is the expected cold-start behavior.

3. **Given** the `computeSlots` module exists at `src/lib/scheduling/computeSlots.ts` with signal weights in `constants.ts`, **When** a list of `TimeWindow[]` and a `courseId` + `duration` are provided, **Then** each window is scored using 5 weighted signals: `productivityScore` (0.30), `gapUrgency` (0.25), `deadlineProximity` (0.20), `fsrsUrgency` (0.15), `recommendationWeight` (0.10). **And** the composite score is a weighted sum normalized to 0-100. **And** each `ScoredSlot` includes a human-readable `reasoning` string (e.g., "Your best focus time, 3 days since last session").

4. **Given** FSRS data is unavailable (E59 not yet shipped), **When** scoring is computed, **Then** `fsrsUrgency` defaults to 0.5 (neutral) and a warning is logged.

5. **Given** a diversity filter is applied to scored results, **When** results are ranked, **Then** no more than 3 slots per day are included. **And** the top N slots (default: 5) are returned sorted by score descending.

6. **Given** unit tests exist at `src/lib/scheduling/__tests__/computeSlots.test.ts` and `productivityHeatmap.test.ts`, **When** tests are run, **Then** they cover: all 5 signal computations with mock data, ranking order verification, diversity filter, cold-start fallback, FSRS unavailable fallback, reasoning string generation. **And** all tests use `FIXED_DATE` pattern.

## Dev Notes

### Architecture Compliance

**Pure computation boundary (CRITICAL):** Both `productivityHeatmap.ts` and `computeSlots.ts` live in `src/lib/scheduling/` -- ZERO React, ZERO Dexie imports. All data passed as plain objects.

**Depends on E70-S01:** Uses `TimeWindow`, `ScoredSlot` types from `@/data/scheduling-types` and constants from `@/lib/scheduling/constants`.

### computeSlots Function Signature

```typescript
export interface ComputeSlotsParams {
  windows: TimeWindow[]
  courseId: string
  duration: number  // requested session duration in minutes
  heatmap: ProductivityHeatmap
  lastSessionDate: string | null  // ISO 8601, null if never studied
  courseDeadline: string | null    // ISO 8601, null if no deadline
  fsrsDueCount: number | null     // null = FSRS unavailable
  recommendationWeight: number    // 0-1, from E52 tag system
  now?: string  // ISO 8601, for testing
}

export function computeSlots(params: ComputeSlotsParams): ScoredSlot[]
```

### Signal Computation Details

| Signal | Weight | Input | Computation |
|--------|--------|-------|-------------|
| `productivityScore` | 0.30 | heatmap + window start time | Lookup `heatmap[dayOfWeek][hourOfDay].avgQualityScore` (already 0-1) |
| `gapUrgency` | 0.25 | lastSessionDate + now | `min(daysSinceLastSession / 14, 1.0)` -- caps at 14 days |
| `deadlineProximity` | 0.20 | courseDeadline + now | `1 - min(daysUntilDeadline / 30, 1.0)` -- 1.0 = imminent, 0 = no deadline or >30 days |
| `fsrsUrgency` | 0.15 | fsrsDueCount | `min(fsrsDueCount / 20, 1.0)` -- caps at 20 cards. Default 0.5 if null |
| `recommendationWeight` | 0.10 | E52 tag weight | Passthrough (already 0-1) |

**Composite score:** `sum(signal * weight) * 100` (produces 0-100 range)

### Reasoning String Templates

Generate 1-2 line human-readable explanations:
- High productivity: "Your best focus time" / "High-quality study window"
- High gap urgency: "{N} days since last session"
- High deadline: "Deadline in {N} days"
- High FSRS: "{N} flashcards due for review"
- Combine top 2 signals: "Your best focus time, 3 days since last session"

### Productivity Heatmap Input

```typescript
export interface StudySessionInput {
  startTime: string   // ISO 8601
  duration: number    // minutes
  qualityScore: number // 0-100
}
```

This maps directly to existing `studySessions` Dexie table fields. The Zustand store (E70-S03) will query Dexie and pass this data.

### Testing Patterns

- Use `FIXED_DATE` pattern for deterministic time
- Mock sessions should span multiple days/hours to test heatmap grouping
- Test scoring with known inputs and verify exact composite scores
- Test diversity filter with 5+ slots on the same day
