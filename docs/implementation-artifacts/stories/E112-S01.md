# Story 112.01: Reading Speed, ETA & Time-of-Day Patterns

## Story

As a reader,
I want to see my real reading speed, estimated finish date, and when I read most,
so that I can plan my reading time and track my actual pace.

## Acceptance Criteria

**AC1 — Real Reading Speed Display**
- Given a user has finished at least one book (status = 'finished', has finishedAt, importedAt or createdAt, and totalPages > 0)
- When they view the Reports page (Study tab)
- Then the ReadingStatsSection shows their average reading speed in pages/hour (or wpm for audiobooks)
- And the speed is computed as: `totalPages / (totalReadingMinutes / 60)` across all reading sessions for that book
- And the display format is: "X pages/hr" (rounded to nearest integer)

**AC2 — ETA for Books In Progress**
- Given a user has at least one book with status = 'reading' with a known totalPages and current progress
- When they view the ReadingStatsSection in Reports
- Then each in-progress book shows an ETA (estimated finish date) in the stat section
- And the ETA is computed as: `remainingPages / avgReadingSpeed` where avgReadingSpeed comes from recent sessions (last 30 days)
- And if no sessions exist in the last 30 days, show "—" instead of an ETA
- And ETA format is: "≈ N days" (or "≈ X weeks" for > 14 days)

**AC3 — Fix KI-044: Replace 2 min/page Heuristic with Session-Based Speed**
- Given the `usePagesReadToday` hook currently uses a 2 min/page heuristic
- When E112-S01 is implemented
- Then `usePagesReadToday` is updated to use a user-specific reading speed derived from completed books' sessions
- And if no completed books exist (new user), fall back to 2 min/page as before
- And the `KI-044` item in `docs/known-issues.yaml` is marked `status: fixed`, `fixed_by: E112-S01`

**AC4 — Time-of-Day Reading Pattern**
- Given a user has at least 7 reading sessions recorded
- When they view the Reports page
- Then a "Reading Patterns" section shows which hour-buckets they read most (Morning 5–12, Afternoon 12–17, Evening 17–21, Night 21–5)
- And the dominant time is highlighted (e.g., "You read most in the Evening")
- And the section renders nothing (null) when fewer than 7 sessions exist

**AC5 — SpeedControl KI-060 Verification**
- Verify that `SpeedControl.tsx` already uses `VALID_SPEEDS` from `useAudiobookPrefsStore` (11 speeds)
- If the old `SPEED_OPTIONS` constant still exists anywhere, remove it
- Mark `KI-060` in `docs/known-issues.yaml` as `status: fixed`, `fixed_by: E112-S01`

## Implementation Notes

**Reading Speed Formula:**
```ts
// For a finished book:
const totalReadingSeconds = sessions.reduce((sum, s) => sum + (s.duration ?? 0), 0)
const totalPagesRead = book.totalPages ?? 0
const avgSpeed = totalReadingSeconds > 0 ? (totalPagesRead / totalReadingSeconds) * 3600 : null
```

**ETA Formula:**
```ts
const remainingPages = Math.round((1 - book.progress / 100) * (book.totalPages ?? 0))
const etaDays = avgSpeed > 0 ? remainingPages / (avgSpeed * dailyReadingHours) : null
```
Where `dailyReadingHours` = average daily reading time from last 30 days sessions.

**Time of Day Buckets:**
```ts
const BUCKETS = { Morning: [5, 12], Afternoon: [12, 17], Evening: [17, 21], Night: [21, 5] }
```
Night wraps around midnight — hour < 5 || hour >= 21.

## Challenges and Lessons Learned

**ETA formula requires careful unit conversion.** The ETA computation chains multiple unit conversions (pages/hour → minutes/page → pages/day → days). Off-by-one in hour/minute conversion produces wildly incorrect estimates. Validated with `sessionCount / 7` to get a weekly average, then applied to remaining pages.

**Test cap boundary: progress % vs raw page count.** The `estimatePagesFromSessions` helper caps pages at `currentPage` (derived from `progress / 100 * totalPages`). A test that used `progress: 25` (→ currentPage=75) for a "120 pages read" assertion silently produced 75. Correct approach: set progress high enough that the theoretical estimate is below the cap.

**Prettier auto-formats on precheck.** The precheck script runs Prettier and leaves formatted files in the working tree. These must be committed before re-running prechecks, or the commit-gate blocks. Pattern: always check `git status` after a precheck run before re-running.

**Pre-existing unit test failures are a baseline reality.** Tests in `courseAdapter`, `pkmExport`, `Settings`, and `scanAndPersist` fail but are unrelated to E112 (last touched before this epic). These should be filed as known issues rather than blocking story reviews.
