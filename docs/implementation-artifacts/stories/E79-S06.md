# Story 79.06: Plugin Test Utilities Package

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a plugin author,
I want test utilities with Dexie data factories,
so that I can write reliable tests for my plugin without manual database setup.

## Acceptance Criteria

1. **Given** a plugin author adds `@knowlune/test-utils` as a dev dependency, **When** they write tests, **Then** they can use `createTestCourse()`, `createTestSession()`, `createTestFlashcard()` factories to seed test data.

2. **Given** a test uses `renderPluginInSlot(MyPlugin, { slotType: 'dashboard-widget' })`, **When** the test runs, **Then** the plugin renders within a mock PluginSlot with PluginContext, error boundary, and seeded Dexie data.

3. **Given** the test utilities, **When** they are used with `fake-indexeddb`, **Then** tests run in Node.js without a browser.

## Dev Notes

### Architecture Decisions

- **Separate package**: `@knowlune/test-utils` is a standalone npm package in `packages/test-utils/`. Plugin authors add it as a devDependency alongside `@knowlune/plugin-types`. This keeps test utilities out of production bundles.
- **Factory pattern**: Each factory returns a complete object with sensible defaults. Overrides are shallow-merged. This matches the factory pattern already used in Knowlune's test infrastructure (see existing test helpers).
- **fake-indexeddb integration**: `fake-indexeddb` provides an in-memory IndexedDB implementation for Node.js. The test utils automatically set up `globalThis.indexedDB = require('fake-indexeddb')` so Dexie works in Vitest without a browser.
- **Schema mirroring**: The test Dexie instance must mirror Knowlune's actual schema. Import the schema definition from `@knowlune/plugin-types` (which should export the table interfaces) or maintain a compatible copy.
- **@testing-library/react**: The `renderPluginInSlot` helper wraps `render()` from `@testing-library/react`, adding the plugin-specific context providers. This is the standard React testing approach.

### Dependencies

- `@knowlune/plugin-types` (E78-S07) -- TypeScript types for Dexie schema, PluginContext, manifest
- `fake-indexeddb` -- already used in Knowlune's test suite

- `@testing-library/react` -- already used in project

### Existing Patterns to Follow

- Knowlune uses Dexie v4 with schema defined in `src/db/schema.ts` -- factories must produce objects matching these table schemas
- Existing test files use `fake-indexeddb` setup pattern -- match this approach
- EventBus pattern: `src/lib/eventBus.ts` uses typed `on()`/`emit()` -- MockEventBus must mirror this interface
- Factory override pattern: `{ ...defaults, ...overrides }` -- standard in the codebase

### Dexie Schema Reference

Factories must produce objects compatible with Knowlune's Dexie tables. Key tables and their primary fields (reference `src/db/schema.ts` and `src/data/types.ts` for exact types):
- `courses`: id, title, description, progress, status, author, imageUrl
- `studySessions`: id, courseId, startTime, endTime, duration
- `flashcards`: id, courseId, front, back, interval, easeFactor, nextReview
- `quizzes`: id, courseId, title, questions, score, completedAt
- `streaks`: id, currentStreak, longestStreak, lastStudyDate
- `notes`: id, courseId, content, createdAt, updatedAt
