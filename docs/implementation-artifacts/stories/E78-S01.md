# Story 78.01: Plugin Manifest Schema and Validation

Status: ready-for-dev

## Story

As a plugin author,
I want a typed plugin manifest schema with install-time validation,
so that I can declare my plugin's metadata, capabilities, and settings with confidence that invalid manifests are caught early.

## Acceptance Criteria

1. **Given** a plugin author creates a `knowlune-plugin.json` file **When** it conforms to the PluginManifest interface (name, version, displayName, description, author, minKnowluneVersion, capabilities, slots) **Then** ajv validation passes and the manifest is accepted **And** `src/plugins/types/index.ts` exports `PluginManifest`, `PluginCapability`, and `PluginSlotType` types.

2. **Given** a plugin manifest with an invalid field (missing required field, wrong type, unknown capability) **When** ajv validation runs at install time **Then** validation fails with a descriptive error message identifying the invalid field **And** the plugin is not registered in the system.

3. **Given** a plugin manifest declares `minKnowluneVersion: "2.0.0"` but Knowlune is at version `1.5.0` **When** manifest validation runs **Then** validation fails with message: "Plugin requires Knowlune >= 2.0.0".

4. **Given** the manifest schema is defined **When** a developer inspects the validation module **Then** `src/plugins/validation/manifestSchema.ts` contains the JSON Schema and `src/plugins/validation/validateManifest.ts` provides the validation entry point **And** `ajv` is added as a project dependency.

## Dev Notes

### Architecture Decisions

- **Trust-based runtime model** (Obsidian pattern): Capabilities are declared for transparency, not runtime enforcement in Phase 1. The manifest is the user-facing contract of what a plugin accesses.
- **ajv for validation**: JSON Schema 7 validated at install time. ajv is the industry-standard JSON Schema validator for JavaScript.
- **Semver comparison for minKnowluneVersion**: Use a simple semver compare function (no need for `semver` package — a lightweight utility comparing major.minor.patch suffices). Alternatively, use a minimal semver compare utility in `src/plugins/validation/semver.ts`.

### Type Definitions (from Architecture Doc)

```typescript
interface PluginManifest {
  name: string                     // npm package name (must match package.json name)
  version: string                  // semver
  displayName: string              // Human-readable (max 50 chars)
  description: string              // One-line (max 200 chars)
  author: string
  minKnowluneVersion: string      // semver
  capabilities: PluginCapability[]
  slots: PluginSlotType[]
  settingsSchema?: JSONSchema7     // Optional: JSON Schema for auto-generated form
  activationEvents?: string[]      // Optional: Phase 2
  icon?: string                    // Optional: Lucide icon name
  homepage?: string
  repository?: string
}

type PluginCapability =
  | 'read:courses' | 'read:sessions' | 'read:flashcards'
  | 'read:quizzes' | 'read:streaks' | 'read:notes'
  | 'write:notes' | 'write:plugin-data'
  | 'subscribe:events'
  | 'add:routes' | 'add:sidebar'    // Phase 2
  | 'access:network'

type PluginSlotType =
  | 'dashboard-widget' | 'course-detail-panel'
  | 'settings-panel'
  | 'sidebar-item' | 'custom-page'  // Phase 2
```

### Critical Conventions

- **Import alias**: Use `@/` for `src/` imports (e.g., `import { PluginManifest } from '@/plugins/types'`)
- **No hardcoded colors**: All styling uses design tokens from `src/styles/theme.css`
- **Zustand store selectors**: When stores are involved, use individual selectors (not relevant to this story but keep in mind for E78-S02+)
- **Knowlune app version**: Read from `package.json` version field (`0.0.1` currently). The `validateManifest` function needs access to this version string for `minKnowluneVersion` comparison.

### Testing Standards

- **Framework**: Vitest (unit tests)
- **Location**: `src/plugins/__tests__/validateManifest.test.ts`
- **Pattern**: Each AC maps to one or more test cases
- **No `Date.now()` or `new Date()`** — use `FIXED_DATE` pattern if dates are needed
- **ajv validation errors**: ajv returns structured errors with `instancePath` and `message` — map these to user-friendly strings in the `errors` array

### Dependencies

- **Add**: `ajv` (latest stable, currently v8.x) via `npm install ajv`
- **Do NOT add**: `semver` package — implement a lightweight comparison utility instead to minimize bundle size

### Project Structure Notes

- `src/plugins/` is a new directory — this story creates the foundational structure that E78-S02 through E78-S07 will build upon
- The `@/` import alias resolves to `./src` (configured in `vite.config.ts`) — no additional config needed for `@/plugins/`
- Types defined here will later be mirrored in `packages/plugin-types/` (E78-S07)
