# Story 63.4: Learner Profile Builder Unit Tests

## Story

As a developer maintaining the learner profile builder,
I want comprehensive unit tests covering all aggregation functions, the formatter, the orchestrator, and graceful degradation,
So that future changes don't break the profile builder's correctness, token compliance, or fault tolerance.

## Acceptance Criteria

**Given** the `aggregateQuizScores` function
**When** tested with mock quizAttempt data containing a mix of passed and failed attempts
**Then** it correctly computes average percentage, failed count, and weak topic list
**And** returns `null` when quiz data is empty

**Given** the `aggregateFlashcardWeakness` function
**When** tested with flashcards containing FSRS fields (`stability`, `lapses`, `state`)
**Then** it uses FSRS heuristics to detect weak cards
**And** when tested with flashcards containing only SM-2 fields (`easeFactor`, `reviewCount`)
**Then** it falls back to SM-2 heuristics without errors

**Given** the `formatLearnerProfile` function
**When** tested with `maxTokens: 40`
**Then** the output character count is approximately 160 or fewer (40 tokens * 4 chars/token)
**And** when tested with `maxTokens: 200`
**Then** all signal categories are included in the output

**Given** the `buildLearnerProfile` orchestrator
**When** one aggregation function throws an error
**Then** the error is caught via `Promise.allSettled`
**And** the other signals still appear in the formatted output

**Given** the `formatLearnerProfile` function
**When** tested with data where knowledge weaknesses overlap with quiz failures on the same topics
**Then** duplicate topic mentions are deduplicated in the output

**Given** snapshot tests for formatted profile output
**When** the formatter produces output for standard test fixtures
**Then** the snapshots match expected compact formatting
**And** any unintended formatting changes cause test failures

## Dev Notes

### Testing Framework
- Vitest with `vi.mock()` for Dexie and Zustand mocking
- Follow pattern from existing tests: `src/stores/__tests__/useFlashcardStore.test.ts`, `src/lib/__tests__/qualityScore.test.ts` (if exists)
- Use `vi.fn()` for Dexie table methods

### Dexie Mocking Pattern
```typescript
vi.mock('@/db/schema', () => ({
  db: {
    quizAttempts: {
      toArray: vi.fn(),
      where: vi.fn(() => ({
        equals: vi.fn(() => ({
          toArray: vi.fn(),
        })),
      })),
    },
    flashcards: {
      where: vi.fn(() => ({
        equals: vi.fn(() => ({
          toArray: vi.fn(),
        })),
      })),
    },
    studySessions: {
      where: vi.fn(() => ({
        equals: vi.fn(() => ({
          toArray: vi.fn(),
        })),
      })),
    },
    quizzes: {
      where: vi.fn(() => ({
        equals: vi.fn(() => ({
          first: vi.fn(),
        })),
      })),
    },
  },
}))
```

### Dynamic Import Mocking for Knowledge Store
```typescript
// Mock the dynamic import for useKnowledgeMapStore
vi.mock('@/stores/useKnowledgeMapStore', () => ({
  useKnowledgeMapStore: {
    getState: vi.fn(() => ({
      // mock state
    })),
  },
}))
```

### Snapshot Test Approach
Use `expect(output).toMatchInlineSnapshot()` for formatted profile output at each token tier (40, 100, 200). This catches unintended formatting regressions without brittle exact-match assertions.

### Test Data Factories
Create factory functions that return valid mock objects with sensible defaults:
- `makeQuizAttempt({ percentage: 45, passed: false })` — overrides defaults
- `makeFlashcard({ easeFactor: 1.5, reviewCount: 5 })` — SM-2 weak card
- `makeFlashcard({ stability: 3, lapses: 4, state: 3 })` — FSRS weak card
- `makeStudySession({ duration: 3600, qualityScore: 72 })` — 1-hour session
