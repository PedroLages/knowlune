# Story 82.01: In-App Plugin Browser

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a Knowlune user,
I want to browse available community plugins from within the app,
so that I can discover and install useful plugins without searching npm manually.

## Acceptance Criteria

1. **Given** the user clicks "Browse Plugins" in Settings > Plugins, **When** the plugin browse page loads at `/settings/plugins/browse`, **Then** a grid of plugin cards is displayed (3 columns desktop, 2 tablet, 1 mobile), **And** each card shows: icon, name, short description, install count, and Install button.

2. **Given** the user types in the search field, **When** they enter a query, **Then** the plugin grid filters by name and description matching the query.

3. **Given** category tabs (All, Widgets, Pages, Tools, Analytics), **When** the user clicks a tab, **Then** the grid filters to show only plugins in that category.

4. **Given** the user clicks a plugin card, **When** the detail view opens, **Then** it shows: full description, capability badges with plain-language explanations, screenshots (horizontal scroll), npm install command with copy button, and links (npm, GitHub, docs).

## Dev Notes

- **Architecture**: Plugin browse page queries the `plugin_registry` Supabase table. This table is created in E82-S02. If implementing E82-S01 first, use mock/seed data for development.
- **Route placement**: `/settings/plugins/browse` sits under the settings route tree. Ensure it's added to React Router in `src/app/routes.tsx` following existing lazy-loading patterns.
- **UX spec**: Detailed wireframes in ux-design-api-plugin-system.md sections 6 (PluginBrowsePage) and 7 (PluginDetailView). Follow layouts precisely. [Source: ux-design-api-plugin-system.md#Section 6, 7]
- **CapabilityBadge**: Reuse the CapabilityBadge component from E78-S03 if it exists. If not, create `src/app/components/plugins/CapabilityBadge.tsx` following UX spec section 8 -- read capabilities use `bg-brand-soft`, write/network capabilities use `bg-warning/15`. [Source: ux-design-api-plugin-system.md#Section 8]
- **Install action**: Clicking "Install" copies the npm install command to clipboard + shows a toast. There is no in-app npm install -- users must run the command in their terminal. This is by design for the current npm-based plugin distribution model.
- **Search implementation**: Client-side filtering is sufficient for Phase 4 (small registry). No need for server-side search initially.
- **Design tokens**: Card backgrounds use `bg-card`, descriptions use `text-muted-foreground`, install count uses `text-muted-foreground`. Brand variant buttons for Install. Card border radius `rounded-[24px]`.
