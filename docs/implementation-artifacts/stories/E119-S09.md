# Story E119-S09: AI Provider Change Re-consent

## Story

As a user (data subject),
I want to be prompted for fresh consent whenever the AI provider routing changes,
so that my data is never sent to a new processor without my explicit agreement.

## Acceptance Criteria

- **AC-1**: AI route selection code checks `consentService.isGrantedForProvider(userId, purpose, providerId)` before routing; if not granted, blocks the request and surfaces `ProviderReconsentModal`.
- **AC-2**: `ProviderReconsentModal` shows: new provider name, what data flows there, link to updated notice. User can Accept (new consent row) or Decline (feature gracefully disabled).
- **AC-3**: Consent grant captures `provider_id` in the `evidence` JSONB field of `user_consents`.
- **AC-4**: Provider version bump (e.g., GPT-4 → GPT-5) does NOT trigger re-consent — only provider identity change does. This rule is documented in `consent-inventory.md`.
- **AC-5**: If a notice update + provider change happen together, a single combined reconsent flow is shown, not two separate modals.
- **AC-6**: User decline: AI feature shows "AI features require consent for [provider]" inline; no other features affected.
- **AC-7**: E2E: simulate provider change; verify reconsent modal appears, accept path proceeds, decline path blocks cleanly.
