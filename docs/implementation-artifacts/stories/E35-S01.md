# Story 35.01: Server-Side Middleware Foundation

## Story

As a platform operator,
I want server-side middleware modules for JWT authentication, origin checking, entitlement caching, and rate limiting,
so that premium API endpoints are secured before routes are wired up.

## Acceptance Criteria

- AC1: Given a valid Supabase JWT, when the `authenticate` middleware processes the request, then it extracts and validates the token using `jose` (HS256) and attaches the user payload to the request.
- AC2: Given an invalid or expired JWT, when the `authenticate` middleware processes the request, then it returns 401 Unauthorized.
- AC3: Given the SUPABASE_JWKS_URL env var is set (future), the authenticate module is structured to support JWKS key resolution.
- AC4: Given a request with an Origin header matching the allowlist, when `origin-check` middleware processes it, then the request proceeds.
- AC5: Given a request with an unrecognized Origin, when `origin-check` middleware processes it, then it returns 403 Forbidden.
- AC6: Given a user's entitlement lookup, when the `entitlement` middleware processes it, then it first checks the LRU cache (5-min TTL, max 1000 entries) before falling back to Supabase.
- AC7: Given a request from an authenticated user, when the `rate-limiter` middleware processes it, then it enforces per-user token bucket rate limiting using `rate-limiter-flexible`.
- AC8: Given a rate-limited user, when they exceed the limit, then the middleware returns 429 Too Many Requests.
- AC9: Shared TypeScript types (AuthenticatedRequest, EntitlementCacheEntry) are defined in a `types.ts` module.
- AC10: All middleware modules have unit tests.
- AC11: New env vars (SUPABASE_JWT_SECRET, SUPABASE_SERVICE_ROLE_KEY, ALLOWED_ORIGINS) are documented in .env.example.
- AC12: `npm run build` passes without errors.

## Implementation Notes

- Middleware modules are standalone — NOT wired to routes yet (future story)
- JWT config accepts SUPABASE_JWT_SECRET (HS256) or SUPABASE_JWKS_URL (future JWKS)
- LRU cache: 5-minute TTL, max 1000 entries for entitlement lookups
- Rate limiter: per-user token bucket via rate-limiter-flexible (memory store)

## Testing Notes

- Unit tests in server/__tests__/middleware/
- Tests mock Supabase client and jose verification
- Rate limiter tests verify token bucket behavior
