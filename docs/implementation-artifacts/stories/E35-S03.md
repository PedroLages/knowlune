# Story 35.03: BYOK Pass-Through Logic

## Story

As a user who provides their own API keys,
I want to use AI features through the proxy without needing a premium subscription,
so that I can use my own OpenAI/Anthropic/Ollama resources while still being a registered Knowlune user.

## Acceptance Criteria

- AC1: Given a signed-in user (any tier), when they send an AI request with `apiKey` in the request body (cloud providers), then the middleware detects `req.isBYOK = true` and the entitlement check is skipped entirely and the request proceeds to the route handler.
- AC2: Given a signed-in user (any tier), when they send an AI request with `ollamaServerUrl` in the request body, then the middleware detects `req.isBYOK = true` and the entitlement check is skipped and the existing SSRF protection (`isAllowedOllamaUrl`) still applies.
- AC3: Given a BYOK request without a valid JWT, when the middleware chain processes it, then it is rejected with 401 `AUTH_REQUIRED` and the BYOK key is never forwarded to the provider (no open relay).
- AC4: Given a BYOK user (any tier), when they send requests, then they are rate-limited at the BYOK tier (30 burst, 15/min refill, 1 point per request) independent of their subscription tier's rate limit.
- AC5: Given a free-tier user with BYOK keys configured, when they use AI features, then BYOK requests succeed (entitlement skipped) and non-BYOK requests are rejected with 403.
- AC6: Given a request that has BOTH `apiKey` in body AND is from a premium user, then it is treated as BYOK (entitlement check skipped, BYOK rate limit applied) and the entitlement middleware layers remain independent.

## Implementation Notes

- BYOK detection implemented as separate middleware (`createDetectBYOKMiddleware`) rather than inline in entitlement middleware, following the architecture doc's guidance that BYOK detection and entitlement checking are independent layers
- Middleware chain order: origin-check -> authenticate -> detectBYOK -> entitlement (skipped for BYOK) -> rateLimiter
- BYOK rate limiter uses a separate `RateLimiterMemory` instance (30 points / 120s window) to keep BYOK and default rate limits fully independent
- No new dependencies added — reuses existing `rate-limiter-flexible` and `lru-cache`
- Critical design rule enforced: BYOK rejection never depends on hosted-AI entitlement state (the `if (req.isBYOK) { next(); return }` check is the first thing in the entitlement middleware)

## Challenges and Lessons Learned

- The BYOK rate limit tier (30 burst, 15/min refill) maps to `RateLimiterMemory({ points: 30, duration: 120 })` — the duration is the window in which all points must be consumed, not the refill interval directly. This is a subtle distinction in rate-limiter-flexible's API.
- Separating BYOK detection into its own middleware function (rather than embedding it in the entitlement middleware) makes the chain easier to test and follows the architecture doc's emphasis on independent middleware layers.
- The BYOK detection is synchronous (no async needed) since it only reads `req.body` — this keeps the middleware chain fast for the common BYOK path.
