# Story 70.6: Auto-Schedule Greedy Allocation Engine

Status: ready-for-dev

## Story

As a learner,
I want the system to automatically plan my entire week of study sessions,
so that I can review and approve an optimized study plan without manually scheduling each course.

## Acceptance Criteria

1. **Given** the `autoSchedule` module exists at `src/lib/scheduling/autoSchedule.ts`, **When** called with all active courses and available windows for the week, **Then** it ranks courses by urgency: `fsrsUrgency * 0.4 + gapUrgency * 0.3 + deadlineProximity * 0.2 + recommendationWeight * 0.1`. **And** for each course (highest urgency first): determines needed session count based on FSRS due cards and course pacing target, finds the highest-scored available window that fits the needed duration, allocates the window and removes it from the available pool. **And** tags each allocation as `'review'` (FSRS cards due), `'new-content'` (ahead on reviews), or `'mixed'` (both). **And** returns a `WeekPlan` with `allocations`, `unscheduledCourses` (with reason strings), and `totalStudyMinutes`.

2. **Given** a course cannot be scheduled (no windows fit its duration requirement), **When** the auto-schedule completes, **Then** the course appears in `unscheduledCourses` with a reason: "No {N}-min windows available" or "No Perfect Time".

3. **Given** the auto-schedule algorithm, **When** the computation runs, **Then** it completes in <200ms for up to 10 courses and 30 weekly windows (per NFR1).

4. **Given** unit tests exist at `src/lib/scheduling/__tests__/autoSchedule.test.ts`, **When** tests are run, **Then** they cover: basic allocation, priority ordering, unscheduled course reporting, session type tagging, overlapping course needs, empty courses list, fully packed week. **And** all tests use `FIXED_DATE` pattern.

## Dev Notes

### Architecture Compliance

**Pure computation boundary (CRITICAL):** `autoSchedule.ts` in `src/lib/scheduling/` -- ZERO React, ZERO Dexie imports. Receives plain objects, returns plain objects.

**Depends on E70-S01, S02, S03:** Uses `TimeWindow`, `ScoredSlot`, `WeekPlan`, `WeekAllocation` types, `mergeSlots`, `computeSlots`, `buildProductivityHeatmap`, and constants.

### Algorithm (ADR-8)

```
autoSchedule(params):
  1. windows = mergeSlots(studyBlocks, googleEvents, quietHours)
  2. courses = rankByUrgency(activeCourses)  // descending
  3. availablePool = [...windows]
  4. allocations = []
  5. unscheduled = []
  6. for each course in courses:
       neededSessions = computeNeededSessions(course)
       for i = 0; i < neededSessions; i++:
         scoredWindows = computeSlots(availablePool, course, duration)
         if scoredWindows.length > 0:
           bestWindow = scoredWindows[0]
           allocations.push({ course, bestWindow, sessionType })
           availablePool.remove(bestWindow)
         else:
           unscheduled.push({ courseId, reason })
           break
  7. return { allocations, unscheduled, totalStudyMinutes }
```

**Complexity:** O(c * w) where c = courses, w = windows. For 10 courses and 30 windows = 300 iterations -- fast.

### Function Signature

```typescript
export interface AutoScheduleParams {
  activeCourses: Array<{
    courseId: string
    courseName: string
    lastSessionDate: string | null
    deadline: string | null
    fsrsDueCount: number | null
    recommendationWeight: number
    targetSessionDuration: number  // minutes
    targetSessionsPerWeek: number
  }>
  studyBlocks: Array<{ start: string; end: string }>
  googleCalendarEvents?: Array<{ start: string; end: string }>
  quietHours?: { start: string; end: string } | null
  heatmap: ProductivityHeatmap
  now?: string  // ISO 8601, for testing
}

export interface UnscheduledCourse {
  courseId: string
  courseName: string
  reason: string  // "No 45-min windows available" or "No Perfect Time"
}

export function autoSchedule(params: AutoScheduleParams): WeekPlan
```

### Course Urgency Ranking

Different weights than slot scoring (optimized for course prioritization):

```typescript
courseUrgency = fsrsUrgency * 0.4 + gapUrgency * 0.3 + deadlineProximity * 0.2 + recommendationWeight * 0.1
```

### Session Type Determination

```typescript
function determineSessionType(fsrsDueCount: number | null): 'review' | 'new-content' | 'mixed' {
  if (fsrsDueCount === null || fsrsDueCount === 0) return 'new-content'
  if (fsrsDueCount > 10) return 'review'
  return 'mixed'
}
```

### Performance Requirement (NFR1)

The algorithm must complete in <200ms for 10 courses + 30 windows. The greedy approach is inherently fast (no backtracking). Add a `performance.now()` check in tests to verify.
