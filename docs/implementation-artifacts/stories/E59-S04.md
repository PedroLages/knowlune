# Story 59.04: Zustand Store Updates (Flashcard and Review Stores)

## Story

As a learner,
I want my flashcard and review stores to use FSRS scheduling fields,
so that all spaced repetition operations use the new algorithm end-to-end.

## Acceptance Criteria

**Given** `useFlashcardStore` uses SM-2 fields (`interval`, `easeFactor`, `reviewCount`, `nextReviewAt`, `reviewedAt`)
**When** this story is implemented
**Then** `createFlashcard` initializes FSRS defaults (stability=0, difficulty=0, reps=0, lapses=0, state=0, due=now)
**And** `rateFlashcard` delegates to FSRS `calculateNextReview` and writes all FSRS fields
**And** `getDueFlashcards` uses `isDue(card)` with FSRS `due` field
**And** `getStats` and `getSessionSummary` use `due` and `last_review` instead of `nextReviewAt`

**Given** `useReviewStore` uses SM-2 fields
**When** this story is implemented
**Then** `rateNote` constructs `ReviewRecord` with FSRS fields from `calculateNextReview` result
**And** `getDueReviews` and `getNextReviewDate` use FSRS `due` field
**And** `InterleavedSessionSummary.ratings` includes `again` key

**Given** consumer files reference SM-2 field names
**When** this story is implemented
**Then** all consumer files use FSRS field names (`due`, `last_review`, `reps`, `stability`)
**And** `npm run build` passes without errors
**And** all store unit tests pass

## Implementation Notes

- `createFlashcard` now initializes `state: 0` (New), `stability: 0`, `difficulty: 0`, `due: now` — matching FSRS `createEmptyCard()` semantics
- `rateFlashcard` spreads all 9 FSRS scheduling result fields onto the updated card, replacing the old SM-2 3-field update
- For new cards (reps=0), `calculateNextReview` is called with `null` card state, triggering `createEmptyCard()` in the FSRS wrapper
- `FlashcardReviewCard.tsx` uses `reps` instead of `reviewCount` for the review count label
- `NotificationService.checkSrsDueOnStartup` simplified: uses `isDue(card, now)` directly instead of checking `nextReviewAt` existence
- `retentionMetrics.ts` updated to use `last_review` (with fallback for undefined) instead of `reviewedAt`
- Remaining TypeScript errors are in test files for other stories (E59-S05 retention metrics, E59-S07 unit tests, E59-S08 E2E tests)

## Testing Notes

- 38 store unit tests pass (16 flashcard + 22 review)
- FSRS interval ordering test uses `due` date comparison instead of `scheduled_days` — FSRS short-term scheduling assigns `scheduled_days: 0` to learning-phase cards (Hard/Good), so date comparison is more reliable
- Build passes cleanly — no type errors in production code
- Lint passes with 0 errors (25 pre-existing warnings in unrelated files)

## Lessons Learned

- FSRS short-term scheduling means new cards rated Hard/Good stay in learning phase with `scheduled_days: 0` and minute-based intervals. Only Easy jumps directly to Review state. Tests should compare `due` dates rather than `scheduled_days` for ordering assertions on new cards.
- The `InterleavedSessionSummary.ratings` type needed `again` added since `ReviewRating` now includes it — TypeScript's index signature enforcement caught this at compile time.
