# Story 32.03: Add IndexedDB Quota Monitoring with User Warnings

## Story

As a learner with growing course data,
I want to be warned when storage is running low,
so that I can manage my data before hitting browser limits.

## Acceptance Criteria

- AC1: Given the app starts, when IndexedDB usage exceeds 80% of browser quota, then a persistent warning toast appears linking to Settings > Data Management
- AC2: Given a Dexie write fails with QuotaExceededError, when the error is caught, then a user-friendly error toast appears (not a raw browser error)
- AC3: Given a bulk operation completes (course import, data import), when the operation finishes, then a storage quota check runs automatically
- AC4: Given the Storage API is unavailable, when quota check runs, then it fails silently without blocking any functionality
- AC5: Given multiple quota warnings would fire in quick succession, when the throttle is active, then only one warning appears per 5-minute window

## Implementation Notes

- Uses `navigator.storage.estimate()` (widely supported, returns `{usage, quota}`)
- Warning threshold at 80% with persistent toast (requires user dismissal)
- Toast links to Settings > Data Management via programmatic navigation + scroll
- `persistWithRetry.ts` now short-circuits on QuotaExceededError (no retry — pointless on full disk)
- `isIndexedDBQuotaExceeded()` handles both standard DOMException and Dexie's wrapped error format
- All quota checks are fire-and-forget — never block main functionality
- Dynamic imports used to keep storageQuotaMonitor out of the critical path

## Testing Notes

- Unit testable: `getStorageEstimate()`, `isIndexedDBQuotaExceeded()`, throttle behavior
- `navigator.storage.estimate()` can be mocked in tests
- QuotaExceededError detection covers: DOMException, Dexie-wrapped errors, Firefox variant
- Integration: quota check runs after import — verified by mock
