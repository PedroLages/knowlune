# Story 63.2: Token-Aware Profile Formatter + Orchestrator

## Story

As the AI tutor system,
I want to format aggregated learner profile data into a compact text string that respects the token budget and prioritizes the most impactful signals,
So that the system prompt stays within limits while conveying the most relevant learner context.

## Acceptance Criteria

**Given** aggregated data from all 4 signals (quiz, knowledge, flashcard, study)
**When** `formatLearnerProfile(data, maxTokens: 40)` is called
**Then** the output is approximately 40 tokens or fewer (~160 characters)
**And** only the highest-priority signals are included (knowledge weaknesses and/or quiz failures)
**And** lower-priority signals (flashcard struggles, study patterns) are omitted entirely

**Given** aggregated data from all 4 signals
**When** `formatLearnerProfile(data, maxTokens: 200)` is called
**Then** the output includes all signal categories with appropriate detail
**And** the output is approximately 200 tokens or fewer (~800 characters)

**Given** all 4 aggregation functions return `null` (no data available)
**When** `formatLearnerProfile(data, maxTokens: 100)` is called
**Then** it returns an empty string

**Given** `buildLearnerProfile` is called with `lessonTopics: ["React Hooks", "JSX"]`
**When** quiz data includes scores for "React Hooks" (45%), "CSS Grid" (90%), and "JSX" (60%)
**Then** the formatted profile prioritizes "React Hooks" and "JSX" weakness data
**And** "CSS Grid" (strong topic, not matching lesson) is omitted or deprioritized

**Given** `buildLearnerProfile` is called with `courseId` and `maxTokens`
**When** all 4 aggregation functions are executed
**Then** they run in parallel via `Promise.allSettled` (not sequentially)
**And** any individual aggregation failure does not prevent other signals from being included

**Given** the knowledge signal reports weak topics and the quiz signal reports failed topics
**When** both are formatted into the profile
**Then** knowledge weaknesses appear before quiz failures in the output (priority order)

## Dev Notes

### Architecture Pattern
Continues the `qualityScore.ts` pure function pattern from S01. The orchestrator (`buildLearnerProfile`) is the main entry point called by `tutorPromptBuilder.ts` (S03).

### Token Budget Rules
- 1 token ≈ 4 characters (conservative approximation)
- 40 tokens = ~160 chars = ~30 words (4K Ollama models)
- 100 tokens = ~400 chars = ~75 words (128K models)
- 200 tokens = ~800 chars = ~150 words (generous models)
- Always omit entire signal blocks — never truncate mid-sentence
- If budget is extremely tight (< 20 tokens), include only the single highest-priority non-null signal

### Signal Priority Order (highest first)
1. **Knowledge weaknesses** — topics classified as weak/fading by Knowledge Map
2. **Quiz failures** — topics with low quiz scores (< 70%)
3. **Flashcard struggles** — cards with high lapses, low stability, overdue reviews
4. **Study patterns** — session frequency, total hours, quality trends

### Compact Formatting Examples

**40-token output** (~160 chars):
```
Weak: CSS grid, React hooks. Quiz avg: 58%. 5 cards overdue. Studied 3h this week.
```

**100-token output** (~400 chars):
```
Knowledge weak: TypeScript generics, async patterns. Fading: state management. Quiz avg: 62%, failed: CSS grid (42%), React hooks (55%). 12 flashcards in relearning, 5 overdue. 4 sessions this week, 3.2h total, avg quality 72/100.
```

### Topic Filtering Logic
When `lessonTopics` are provided:
- Quiz weakTopics matching lesson topics → boosted to top
- Knowledge weak/fading topics matching → boosted
- Non-matching topics still included but after matching ones
- Use case-insensitive includes for partial matching (e.g., "React Hooks" matches "react hooks basics")

### Deduplication
When knowledge weaknesses and quiz failures mention the same topic, deduplicate in the formatted output. Prefer the knowledge signal phrasing since it's higher priority.
