# Story E119-S04: Deletion Verification + Confirmation Email

## Story

As a user (data subject),
I want to receive email confirmation at both deletion request and final hard-delete,
and I want the system to have an automated CI probe that verifies deletion completeness after any schema change.

## Acceptance Criteria

- **AC-1**: Email sent within 30s of deletion request: "Deletion scheduled — cancel within 7 days" containing cancel link.
- **AC-2**: Email sent when hard-delete completes: "Your data has been deleted" receipt.
- **AC-3**: The email at hard-delete time uses the address recorded at deletion-request time (stored in a `pending_deletions` row to handle pre-scrub timing).
- **AC-4**: Email delivery failure does NOT block the deletion operation; failure is logged to ops alert channel.
- **AC-5**: `scripts/ci/deletion-probe.ts` queries every registry-listed table and 4 Storage buckets for a given user ID; returns zero-rows success or names the offending table.
- **AC-6**: CI runs `deletion-probe.ts` on PRs that touch `tableRegistry.ts`, `delete-account/`, or `retention-tick/`.
- **AC-7**: Email templates in `src/lib/compliance/emailTemplates.ts` with plain-text + HTML variants.

## Tasks / Subtasks

- [ ] Task 1: Inspect existing Edge Functions for a mailer; if absent, add minimal SMTP/provider client (AC: 1, 2)
- [ ] Task 2: Create or extend `supabase/functions/delete-account-email/index.ts` (AC: 1, 2, 4)
- [ ] Task 3: Create `pending_deletions` table / add to delete-account flow to capture email before scrub (AC: 3)
- [ ] Task 4: Create `src/lib/compliance/emailTemplates.ts` with both templates (AC: 7)
- [ ] Task 5: Create `scripts/ci/deletion-probe.ts` (AC: 5)
- [ ] Task 6: Wire CI config to run probe on relevant diffs (AC: 6)
- [ ] Task 7: Write unit tests for email functions (AC: 1, 2, 4)

## Context

- Depends on S03 (cascade delete must exist first).
- S12 (breach runbook) references the deletion confirmation path.
- The 7-day grace cancel link must route to the `cancel-account-deletion` Edge Function created in S03.
- Deletion probe is intentionally simple: a query script, not a full E2E test.
- CI probe must be diff-scoped (only runs on registry/delete-related diffs) to keep CI fast.

## Out of Scope

- Full E2E coverage of the email delivery chain (staging-only test).
- Implementing the retention enforcement job (S11).
