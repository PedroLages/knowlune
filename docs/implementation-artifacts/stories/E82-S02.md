# Story 82.02: Plugin Registry and Community Submission

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a plugin author,
I want to submit my plugin to the Knowlune community registry,
so that other users can discover and install it from the in-app browser.

## Acceptance Criteria

1. **Given** the Supabase `plugin_registry` table exists, **When** a plugin author submits their plugin, **Then** the registry stores: npm package name, display name, description, author, version, capabilities, slot types, icon, homepage URL, repository URL, install count, category, and submission date.

2. **Given** a user clicks "Install" on a plugin in the browser, **When** the install action triggers, **Then** the npm install command is copied to clipboard, a toast confirms: "Install command copied. Run in your terminal.", and instructions explain: "Then restart Knowlune. The plugin will appear in Settings > Plugins."

3. **Given** a plugin author wants to submit, **When** they follow the submission process, **Then** they can submit via a form in the plugin detail view or via a documented API endpoint, **And** submitted plugins are marked `pending_review` until approved.

## Dev Notes

- **Architecture**: The `plugin_registry` is a Supabase table that acts as the community plugin directory. It's separate from the local `plugins` Dexie table (which tracks installed plugins). The registry is server-side, the installed plugins are client-side.
- **Dependency on E82-S01**: The PluginBrowse page and PluginBrowseCard from E82-S01 must exist. This story provides the data source and adds the submission flow.
- **Install count**: Incrementing install_count on "Install" click is approximate (counts clipboard copies, not actual installs). This is acceptable for Phase 4.
- **Review process**: Submitted plugins are `pending_review` by default. For a single-developer project, review can be manual (update status directly in Supabase). No automated review pipeline needed in Phase 4.
- **Seed data**: Include 3-5 example plugins in the migration for immediate UI testing. The Daily Quote Widget from E78-S06 should be one of them (it's the reference implementation).
- **npm validation**: The optional npm registry check (`HEAD https://registry.npmjs.org/{package}`) verifies the package exists. This is a nice-to-have validation, not a blocker if npm is unreachable.
