# Story 32.02: Implement Lazy Zustand Store Loading

## Story

As a learner opening the app,
I want non-critical stores to load on-demand when I navigate to their pages,
so that the initial app load is faster and more responsive.

## Acceptance Criteria

- AC1: Given the app starts, when Layout mounts, then only critical stores (useCourseStore, useAuthStore) are eagerly loaded ‚Äî non-critical stores (useQuizStore, useNoteStore, useAuthorStore, useFlashcardStore, useCareerPathStore, useReviewStore, useChallengeStore) are NOT loaded until their page is visited
- AC2: Given a page that uses a deferred store, when the page mounts, then a loading skeleton is shown while the store initializes from IndexedDB
- AC3: Given the useLazyStore hook, when a store's load function is called multiple times, then it only loads once (idempotent)
- AC4: Given the app builds, when running `npm run build`, then there are no TypeScript or build errors

## Implementation Notes

### Store Classification

**Critical (eager-loaded in main.tsx / Layout.tsx):**
- `useCourseStore` ‚Äî loaded in main.tsx deferInit, backup in Layout.tsx
- `useAuthStore` ‚Äî loaded in Layout.tsx for auth state

**Non-critical (deferred to page-level via useLazyStore):**
- `useAuthorStore` ‚Äî Authors, AuthorProfile, CourseDetail, CourseOverview, ImportedCourseDetail
- `useCourseImportStore` ‚Äî Courses, Authors, AuthorProfile, ImportedCourseDetail, Overview
- `useNoteStore` ‚Äî Notes, RetentionDashboard, ReviewQueue, InterleavedReview
- `useSessionStore` ‚Äî Overview (loadSessionStats)
- `useQuizStore` ‚Äî Quiz (uses persist middleware, self-initializing)
- `useFlashcardStore` ‚Äî Flashcards
- `useCareerPathStore` ‚Äî CareerPaths, CareerPathDetail
- `useReviewStore` ‚Äî RetentionDashboard, ReviewQueue, InterleavedReview
- `useChallengeStore` ‚Äî Challenges

### Pattern: useLazyStore hook
- Uses `useRef` to track if loadFn has been called (idempotent)
- Calls loadFn exactly once on mount via `useEffect`
- Pages that already have custom loading logic (error handling, chained promises) keep their existing `useEffect` pattern ‚Äî useLazyStore is for the simple cases only

### Pages NOT converted (complex loading patterns):
- `Challenges.tsx` ‚Äî chains loadChallenges ‚Üí refreshAllProgress ‚Üí fireMilestoneToasts
- `CareerPaths.tsx` ‚Äî has error handling with retry logic
- `RetentionDashboard.tsx` ‚Äî Promise.all with 3 stores
- `InterleavedReview.tsx` ‚Äî Promise.all with session reset

## Challenges and Lessons Learned

- Many pages already had page-level store loading via useEffect ‚Äî the main value of useLazyStore is providing a consistent, idempotent pattern that prevents double-loading and reduces boilerplate
- Complex loading patterns (error handling, chained promises, Promise.all) are better left as explicit useEffect blocks rather than forced into a one-size-fits-all hook
- The critical/non-critical store classification is important documentation ‚Äî it clarifies which stores are safe to defer
