# Story 35.02: Express Proxy Entitlement Guard

## Story

As a platform operator,
I want the middleware chain wired into Express and the client sending auth headers,
so that all AI proxy requests are authenticated and authorized in production.

## Acceptance Criteria

- AC1: `server/index.ts` applies the middleware chain to all `/api/ai/*` routes in order: 1. Origin check, 2. JWT authentication, 3. Entitlement check, 4. Rate limiter.
- AC2: `ProxyLLMClient` and `OllamaLLMClient` (proxy mode) include `Authorization: Bearer <JWT>` from `useAuthStore.getState().session?.access_token`.
- AC3: Signed-in premium users pass all middleware checks and AI features work identically to before.
- AC4: Free-tier users receive 403 `ENTITLEMENT_ERROR` and client maps to `LLMError`.
- AC5: Expired JWT returns 401; client maps to `AUTH_REQUIRED`.
- AC6: `/api/ai/ollama/health` bypasses the entire middleware chain.
- AC7: `ENTITLEMENT_ERROR` and `RATE_LIMITED` added to `LLMErrorCode` enum.
- AC8: Error mapping in `BaseLLMClient`: 401 -> AUTH_REQUIRED, 403 -> ENTITLEMENT_ERROR, 429 -> RATE_LIMITED.
- AC9: `npm run build` passes without errors.

## Implementation Notes

- Middleware order: Origin check -> JWT auth -> Entitlement check -> Rate limiter
- Health check endpoint registered BEFORE middleware chain to bypass all auth
- Tags endpoint moved AFTER middleware chain (requires auth unlike health check)
- Middleware is conditional on env vars being configured -- graceful degradation for dev without Supabase
- Both server and client change simultaneously to avoid 401 failures
- Error codes map: 401->AUTH_REQUIRED, 403->ENTITLEMENT_ERROR, 429->RATE_LIMITED
- `mapHttpStatusToLLMErrorCode` added to BaseLLMClient as shared method replacing per-client private methods
- OllamaLLMClient.fetchModels also includes auth header in proxy mode

## Challenges and Lessons Learned

1. **Express route registration order matters for middleware bypass**: The health check endpoint must be registered BEFORE `app.use('/api/ai', middleware)` to bypass the middleware chain. Routes registered after `app.use` go through the middleware. The `/api/ai/ollama/tags` endpoint was initially registered before middleware (unintentionally bypassing auth) and had to be moved after the middleware block.

2. **Consolidating error mapping into base class**: ProxyLLMClient and OllamaLLMClient both had private `mapHttpStatusToErrorCode` methods with identical logic. Moving this to `BaseLLMClient.mapHttpStatusToLLMErrorCode` eliminates duplication and ensures consistent error mapping. The direct-provider clients (OpenAI, Anthropic) keep their own private methods since 403 from those providers means auth error, not entitlement.

3. **Conditional middleware activation**: Using `isMiddlewareConfigured` to check all required env vars before applying middleware ensures the server works in dev environments without Supabase configured. This avoids breaking the existing development workflow while enabling enforcement in production.

4. **Test mock coordination**: Adding `useAuthStore` mocks to all three test files (proxy-client, ollama-client, ollama-client-models) was necessary because the import is now at module scope. Using `vi.mock` with a factory function ensures the mock is hoisted above the module import.
