# Story 80.02: REST API Endpoints with Bearer Auth

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an API consumer,
I want to access my learning data via REST endpoints using my API token,
so that I can pull data into scripts, dashboards, and automation tools.

## Acceptance Criteria

1. **Given** an API consumer sends `GET /rest/v1/courses` with `Authorization: Bearer klp_<token>`, **When** the Supabase Edge Function receives the request, **Then** it computes SHA-256 of the received token, looks up the hash in `api_tokens` table, and if found and not revoked/expired: updates `last_used_at`, proxies to PostgREST with user context, and returns JSON response with course data.

2. **Given** an invalid or revoked token is used, **When** the Edge Function looks up the hash, **Then** it returns 401 Unauthorized with `{ "message": "Invalid or revoked token" }`.

3. **Given** an API consumer uses PostgREST query syntax, **When** they send `GET /rest/v1/courses?select=name,progress&order=progress.desc&limit=10`, **Then** the response contains only the selected fields, ordered by progress descending, limited to 10 results.

4. **Given** the API endpoints, **When** all tables are queried, **Then** RLS policies ensure the user can only access their own data (single-user: `auth.uid() = user_id`).

5. **Given** available endpoints, **When** the API consumer checks documentation, **Then** `GET /rest/v1/courses`, `GET /rest/v1/study_sessions`, `GET /rest/v1/flashcards`, `GET /rest/v1/quizzes`, `GET /rest/v1/streaks`, `GET /rest/v1/notes` are all accessible.

## Tasks / Subtasks

- [ ] Task 1: Create Supabase Edge Function for token verification (AC: 1, 2)
  - [ ] 1.1 Create `supabase/functions/verify-api-token/index.ts` -- Deno Edge Function
  - [ ] 1.2 Extract bearer token from `Authorization` header, strip `klp_` prefix for hash computation
  - [ ] 1.3 Compute SHA-256 hash using Web Crypto API (`crypto.subtle.digest`)
  - [ ] 1.4 Query `api_tokens` table: match `token_hash`, check `revoked_at IS NULL`, check `expires_at IS NULL OR expires_at > now()`
  - [ ] 1.5 If valid: update `last_used_at = now()`, extract `user_id` and `scopes` from token record
  - [ ] 1.6 If invalid/revoked/expired: return `401 { "message": "Invalid or revoked token" }`
  - [ ] 1.7 Forward authenticated request to PostgREST with user context (set `request.jwt.claims.sub = user_id`)
- [ ] Task 2: Configure RLS policies for API access (AC: 4)
  - [ ] 2.1 Verify existing RLS policies on `courses`, `study_sessions`, `flashcards`, `quizzes`, `streaks`, `notes` tables enforce `auth.uid() = user_id`
  - [ ] 2.2 If RLS policies missing, create migration to add SELECT policies with `auth.uid() = user_id` for each table
  - [ ] 2.3 Ensure API token auth flow sets the correct `auth.uid()` context so RLS policies apply
- [ ] Task 3: Implement scope enforcement (AC: 1, 5)
  - [ ] 3.1 Map scopes to tables: `read:courses` -> `courses`, `read:sessions` -> `study_sessions`, `read:flashcards` -> `flashcards`, `read:quizzes` -> `quizzes`, `read:streaks` -> `streaks`, `read:notes` -> `notes`

  - [ ] 3.3 If scope not granted: return `403 { "message": "Token does not have access to this resource" }`
- [ ] Task 4: PostgREST query support (AC: 3)
  - [ ] 4.1 Pass through PostgREST query parameters: `select`, `order`, `limit`, `offset`, `eq`, `neq`, `gt`, `lt`, `gte`, `lte`, `like`, `ilike`, `in`, `is`
  - [ ] 4.2 Validate that query parameters are well-formed before forwarding
  - [ ] 4.3 Return PostgREST response headers including `Content-Range` for pagination
- [ ] Task 5: Edge Function deployment and testing (AC: 1, 2, 3, 4, 5)
  - [ ] 5.1 Configure Edge Function in `supabase/config.toml` if needed
  - [ ] 5.2 Write integration tests: valid token returns data, revoked token returns 401, expired token returns 401, scope mismatch returns 403, PostgREST query syntax works
  - [ ] 5.3 Test all 6 endpoints with valid token
  - [ ] 5.4 Document API usage in `docs/api/rest-api.md` with endpoint list, auth header format, query syntax examples, and error responses

## Dev Notes

- **Architecture**: Uses Supabase's auto-generated PostgREST endpoints with URL path versioning (`/rest/v1/<table>`). The Edge Function acts as middleware for token verification before proxying to PostgREST. [Source: architecture-api-plugin-system.md#Decision 11]
- **Supabase Edge Functions**: Written in Deno/TypeScript. Deployed via `supabase functions deploy verify-api-token`. Uses Web Crypto API for SHA-256 hashing (same as browser crypto.subtle).
- **Token flow**: Bearer token -> strip `klp_` prefix -> SHA-256 hash -> lookup in `api_tokens` -> if valid, set user context and proxy to PostgREST -> PostgREST applies RLS policies.
- **Read-only by default**: All endpoints are read-only (GET). No write endpoints in Phase 3.
- **Dependency on E80-S01**: Requires the `api_tokens` table created in E80-S01 to exist.
- **Self-hosted Supabase**: Knowlune uses self-hosted Supabase on Unraid. Edge Functions run in the Supabase runtime. Ensure the Edge Function can access the `api_tokens` table via the service role key internally.
- **PostgREST conventions**: Supabase automatically exposes tables via PostgREST. The Edge Function wraps this with custom auth. Standard PostgREST query syntax (select, order, limit, filters) is passed through.
