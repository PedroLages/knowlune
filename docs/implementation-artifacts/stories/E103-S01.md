# Story 103.1: Chapter Title Matching Engine

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a Knowlune user who owns both the EPUB and audiobook of the same title,
I want the system to match chapters between the two formats,
So that switching formats takes me to the corresponding chapter.

## Acceptance Criteria

1. **Given** a `Book` record with `format: 'epub'` and a `Book` record with `format: 'audiobook'` that share the same title and author,
   **When** the user links the two formats (or the system auto-detects a match by title+author),
   **Then** the system extracts chapter titles from the EPUB TOC (`epub.js` navigation) and ABS audiobook chapter metadata via `AudiobookshelfService.fetchChapters(url, apiKey, itemId)`

2. **Given** EPUB chapter titles and audiobook chapter titles for the same book,
   **When** the chapter matching engine runs (`computeChapterMapping()`),
   **Then** it normalizes both title lists (lowercase, strip leading numbers, trim whitespace, collapse multiple spaces)
   **And** computes Jaro-Winkler similarity for each EPUB×audio chapter pair
   **And** falls back to Levenshtein distance when Jaro-Winkler produces no match above threshold
   **And** stores matches as `ChapterMapping[]`: `{ epubChapterHref: string; audioChapterIndex: number; confidence: number }`
   **And** only matches above the configurable confidence threshold (default `0.7`) are stored

3. **Given** automatic chapter matching returns fewer matches than either chapter list,
   **When** the system detects an incomplete mapping,
   **Then** a manual pairing UI is presented showing EPUB chapters and audio chapters side-by-side
   **And** the user can assign a matching audio chapter to each EPUB chapter via dropdown or drag-drop
   **And** manual pairings are stored with `confidence: 1.0` and override any automatic match for that pair

4. **Given** the chapter matching engine,
   **When** it executes,
   **Then** all computation is pure client-side JavaScript — no server calls, no `async` operations, no npm dependencies

5. **Given** a completed chapter mapping,
   **When** the mapping is stored,
   **Then** it is persisted in Dexie in a new `chapterMappings` table (v41 schema bump) indexed on `[epubBookId+audioBookId]`
   **And** it can be invalidated and recomputed if either book's chapter list changes (manual re-run or on source update)

6. **Given** the Dexie schema,
   **When** the app initializes after this story ships,
   **Then** the schema upgrades to v41 with a new `chapterMappings` table indexed on `[epubBookId+audioBookId], epubBookId, audioBookId`
   **And** `checkpoint.ts` `CHECKPOINT_VERSION` is updated to 41
   **And** the checkpoint schema test is updated to expect v41

7. **Given** the `AudiobookshelfService`,
   **When** `fetchChapters(url, apiKey, itemId)` is called for the audiobook,
   **Then** it calls `GET /api/items/<itemId>/chapters` with `Authorization: Bearer <apiKey>`
   **And** returns `AbsResult<{ chapters: AbsChapter[] }>` following the discriminated-union pattern
   **And** `AbsChapter` includes at minimum: `id`, `start` (seconds), `end` (seconds), `title`

8. **Given** a unit test suite for the matching engine,
   **When** tests run with Vitest,
   **Then** `computeChapterMapping()` is tested for: exact matches, numbered vs. named chapters (e.g., "Chapter 1" vs "Chapter One"), stripped-number normalization, below-threshold fallback, empty input handling, and mixed-confidence output

### Algorithm: Jaro-Winkler (primary) + Levenshtein (fallback)

**Jaro-Winkler** is the primary algorithm (Storyteller's proven choice for chapter title matching):
- Score range 0–1; 1 = identical
- Prefix bonus: boosts strings sharing a common prefix (handles "Chapter 1" vs "Chapter 1: The Hero")
- Default threshold: `0.7` (70%) — configurable per book pair

**Levenshtein fallback**: Applied when Jaro-Winkler scores all pairs below threshold. Normalized by max string length: `1 - editDistance / maxLen`. Handles cases where chapter numbers are very different lengths ("I" vs "1").

**Normalization step** (critical for reducing false negatives):
```
normalizeChapterTitle("Chapter 1: The Journey Begins")
  → "the journey begins"

normalizeChapterTitle("1. The Journey Begins")
  → "the journey begins"

normalizeChapterTitle("PART TWO")
  → "part two"
```
Normalization: `.toLowerCase().replace(/^[\d\s.:\-–]+/, '').replace(/\s+/g, ' ').trim()`

**Hungarian assignment (not needed for MVP)**: For large books where many chapters have similar titles (e.g., 50+ chapters named "Track 01"–"Track 50"), a greedy approach is fine since we only keep high-confidence matches. True optimal assignment via Hungarian algorithm is deferred.

### EPUB Chapter Extraction via epub.js

The project already uses `epubjs` (imported in `EpubMetadataService.ts`). Use `book.navigation.toc` to get the table of contents:

```typescript
import ePub from 'epubjs'

export interface EpubTocItem {
  href: string   // spine href, e.g., "OEBPS/chapter01.xhtml"
  label: string  // displayed title, e.g., "Chapter 1"
}

export async function extractEpubChapters(
  fileArrayBuffer: ArrayBuffer
): Promise<EpubTocItem[]> {
  const book = ePub(fileArrayBuffer)
  await book.ready
  const toc = book.navigation.toc
  return toc.map(item => ({
    href: item.href,
    label: item.label.trim(),
  }))
}
```

Note: `epub.js` is already in `package.json` — do NOT add a new dependency.

### Anti-Patterns to Avoid

- **Do NOT** use `npm install` for any string similarity library (NFR5: zero new deps). Jaro-Winkler in ~15 lines of JS, Levenshtein in ~20 lines.
- **Do NOT** make `computeChapterMapping()` async — it's pure synchronous computation
- **Do NOT** create a class `ChapterMatcher` — export pure functions (mirrors `OpdsService` pattern)
- **Do NOT** use `book.spine` for chapter ordering — use `book.navigation.toc` for titles
