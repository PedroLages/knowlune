# Story 76.5: Unsubscribe Edge Function and pg_cron Schedule

Status: ready-for-dev

## Story

As a user,
I want to unsubscribe from digest emails with a single click from my email client or the email footer,
so that I can stop receiving digests without navigating to the app.

## Acceptance Criteria

1. **Given** the `unsubscribe` Edge Function at `supabase/functions/unsubscribe/index.ts`
   **When** a POST request is received with a valid JWT token (RFC 8058 one-click)
   **Then** the function:
   1. Verifies the JWT signature and expiry via `jose`
   2. Extracts `user_id` from the `sub` claim
   3. Updates `notification_preferences.email_digest_enabled = false` for that user
   4. Returns HTTP 200 with "Unsubscribed" body

2. **Given** the unsubscribe Edge Function
   **When** a GET request is received with a valid JWT token (footer link click)
   **Then** it returns an HTML page with confirmation: "You have been unsubscribed from Knowlune weekly digests."
   **And** includes a link to re-enable digests in Settings

3. **Given** an invalid or expired JWT token
   **When** the unsubscribe function receives it
   **Then** it returns a friendly HTML page: "This link has expired. Visit Settings > Notifications to manage your email preferences."
   **And** does not modify any database records

4. **Given** the pg_cron scheduling requirement
   **When** the cron job migration is applied
   **Then** a job named `weekly-learning-digest` is registered with schedule `0 8 * * 0` (Sunday 8:00 AM UTC)
   **And** the job invokes `send-weekly-digest` Edge Function via `net.http_post` with service_role_key authorization from Supabase Vault
   **And** the job is visible in the Supabase Dashboard under Integrations > Cron

## Dev Notes

### Architecture Decisions

- **AD-4**: Signed JWT unsubscribe tokens with HS256, 90-day expiry. Stateless verification — no database lookup needed to validate the token, only to perform the unsubscribe action.
- **AD-6**: Separate Edge Function from `send-weekly-digest` because unsubscribe is user-facing with HTML responses vs. internal batch job with JSON responses.
- **AD-10**: pg_cron schedule `0 8 * * 0` = Sunday 8:00 AM UTC. Provides "week in review" for Mon-Sun period.

### Unsubscribe Flow (Two Paths)

**RFC 8058 One-Click (POST)**:
- Email client (Gmail, Apple Mail) shows native "Unsubscribe" button
- Client sends POST to the `List-Unsubscribe` URL
- No user interaction needed — automated by email client
- Response: plain text "Unsubscribed" (200 OK)

**Footer Link Click (GET)**:
- User clicks "Unsubscribe" link in email footer
- Browser navigates to the URL
- Response: HTML page confirming unsubscription

### pg_cron SQL

```sql
SELECT cron.schedule(
  'weekly-learning-digest',
  '0 8 * * 0',
  $$
  SELECT net.http_post(
    url := 'https://<project-ref>.supabase.co/functions/v1/send-weekly-digest',
    headers := jsonb_build_object(
      'Authorization', 'Bearer ' || current_setting('app.settings.service_role_key'),
      'Content-Type', 'application/json'
    ),
    body := '{}'::jsonb
  ) AS request_id;
  $$
);
```

### Critical Constraints

- **No dark patterns**: Unsubscribe must work immediately with no "Are you sure?" confirmations.
- **Token expiry (90 days)**: Expired tokens show a friendly error page — never a 500 or raw error.
- **GDPR compliance**: Honor opt-out immediately — `email_digest_enabled = false` is set in the same request.
- **Service role key in Vault**: pg_cron job uses `current_setting('app.settings.service_role_key')` — key must be stored in Supabase Vault.
- **pg_cron monitoring**: Failed jobs visible in `cron.job_run_details` table.
- **CAN-SPAM**: Every email must have an unsubscribe mechanism — this story provides both paths.
