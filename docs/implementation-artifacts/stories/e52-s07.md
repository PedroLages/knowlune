# Story 52.07: LLM Provider Abstraction (Optional)

## Story

As a learner,
I want quiz generation to work seamlessly with both my local Ollama server and OpenAI cloud API,
so that I can choose the provider that best fits my needs (free local vs. faster cloud).

## Acceptance Criteria

**Given** the learner has configured Ollama as their AI provider in settings
**When** quiz generation is triggered
**Then** the request routes through `callOllamaChat` using the existing `/api/ai/ollama/chat` proxy
**And** uses the `format` parameter for constrained JSON decoding

**Given** the learner has configured OpenAI with a BYOK API key
**When** quiz generation is triggered
**Then** the request routes through the existing BYOK proxy middleware
**And** uses `response_format: { type: "json_schema" }` for structured output

**Given** the AI configuration in settings
**When** the learner selects a model (e.g., Llama 3.2, GPT-4o-mini)
**Then** the quiz generation service uses the selected model
**And** stores the `modelId` in the generated quiz metadata

**Given** the learner has not granted AI consent
**When** they attempt to generate a quiz
**Then** the generation is blocked with a toast explaining consent is required
**And** a link to the AI settings page is provided

**Given** the Ollama server is offline and OpenAI is configured as fallback
**When** the learner triggers quiz generation
**Then** the system respects the configured provider (no silent fallback)
**And** shows the appropriate error for the selected provider

## Tasks / Subtasks

- [ ] Task 1: Provider Resolution (AC: 1, 2, 3)
  - [ ] 1.1 Read provider configuration from `aiConfiguration` (existing settings)
  - [ ] 1.2 Resolve active provider: Ollama or OpenAI based on settings
  - [ ] 1.3 Resolve active model from settings
  - [ ] 1.4 Pass provider/model to `callOllamaChat` (existing proxy handles routing)

- [ ] Task 2: Structured Output Adaptation (AC: 1, 2)
  - [ ] 2.1 For Ollama: use `format` parameter with `QUIZ_RESPONSE_SCHEMA`
  - [ ] 2.2 For OpenAI: use `response_format: { type: "json_schema", json_schema: QUIZ_RESPONSE_SCHEMA }`
  - [ ] 2.3 Verify existing proxy middleware handles both formats correctly

- [ ] Task 3: Consent Gate (AC: 4)
  - [ ] 3.1 Check AI consent status before quiz generation
  - [ ] 3.2 If not consented: block with sonner toast "AI consent required for quiz generation"
  - [ ] 3.3 Include link/button to navigate to AI settings page

- [ ] Task 4: Model Metadata Storage (AC: 3)
  - [ ] 4.1 Store selected `modelId` in generated quiz metadata
  - [ ] 4.2 Display model info on quiz detail (e.g., "Generated by Llama 3.2")

- [ ] Task 5: Error Handling Per Provider (AC: 5)
  - [ ] 5.1 Ollama offline: "Quiz generation unavailable -- Ollama server is offline"
  - [ ] 5.2 OpenAI key invalid: "Quiz generation failed -- check your OpenAI API key"
  - [ ] 5.3 No silent provider fallback (explicit error for configured provider)

- [ ] Task 6: Unit Tests (AC: all)
  - [ ] 6.1 Test Ollama provider resolution from settings
  - [ ] 6.2 Test OpenAI provider resolution with BYOK key
  - [ ] 6.3 Test consent gate blocks when not consented
  - [ ] 6.4 Test modelId stored in quiz metadata
  - [ ] 6.5 Test provider-specific error messages

## Implementation Notes

**Existing files to modify:**
- `src/ai/quizGenerationService.ts` — Add provider resolution and consent check
- Quiz metadata display components — Show model info

**Existing files to reuse (not modify):**
- `src/ai/llm/ollama-client.ts` — `callOllamaChat` (handles both providers via proxy)
- `src/lib/aiConfiguration.ts` — Provider selection, consent, BYOK keys
- `server/index.ts` — Express proxy (already handles Ollama + OpenAI routing)

**Key patterns:**
- `callOllamaChat` already handles both Ollama and OpenAI through the proxy
- The proxy middleware chain resolves BYOK keys and routes requests
- No new HTTP clients needed — this story is about wiring up the existing infrastructure
- Zero new npm dependencies

**Dependencies:**
- E52-S01 (Quiz Generation Pipeline)
