# Story 59.07: Unit Test Rewrite for FSRS Algorithm

## Story

As a developer,
I want comprehensive unit tests for the FSRS algorithm wrapper,
so that scheduling behavior is verified and regressions are caught.

## Acceptance Criteria

**Given** `src/lib/__tests__/spacedRepetition.test.ts` tests SM-2 behavior
**When** this story is implemented
**Then** all tests are rewritten to verify FSRS behavior
**And** tests cover: new card + each rating (again/hard/good/easy) -> correct state transitions
**And** tests cover: review card + each rating -> correct stability/difficulty changes
**And** tests cover: `predictRetention()` power-law decay (not exponential)
**And** tests cover: `isDue()` with `due` field
**And** tests cover: edge case — card with `last_review: undefined` (new card)
**And** tests use deterministic scheduling (FSRS instance with `enable_fuzz: false`)
**And** tests use `FIXED_DATE` constant (no `Date.now()` or `new Date()` per ESLint rules)

**Given** the migration formulas in the architecture doc
**When** unit tests verify migration logic
**Then** easeFactor-to-difficulty inverse mapping is tested at boundaries (1.3 -> 10, 2.5 -> ~5, 3.5 -> ~1)
**And** interval-to-stability direct mapping is tested (including 0 -> 0.1 floor)
**And** state determination is tested (reviewCount 0 -> state 0, reviewCount > 0 -> state 2)

## Implementation Notes

- Complete rewrite of `spacedRepetition.test.ts` from SM-2 to FSRS algorithm coverage
- Uses `fsrsTest` instance with `enable_fuzz: false` for deterministic scheduling results
- All tests use `FIXED_DATE` constant (`2026-03-15T12:00:00.000Z`) per ESLint `test-patterns/deterministic-time` rule
- `makeCardState()` helper factory builds `FSRSCardState` objects with sensible defaults and overrides
- Migration logic tests (AC 9-11) already covered by `migration-v31-fsrs.test.ts` — no duplication needed

## Testing Notes

- 30+ unit tests across 6 describe blocks: new card ratings, review card ratings, edge cases, predictRetention, isDue, full review flow integration
- Edge cases: `last_review: undefined`, `stability: 0`, invalid date strings, 10 consecutive Again/Easy rating loops to verify difficulty stays in 0-10 range
- Integration tests verify multi-step review flows (new → Good → Good → Good) and recovery paths (Again → Good)
- Power-law decay verified at `t = stability` where FSRS yields ~90% retention by design

## Challenges and Lessons Learned

- FSRS `enable_fuzz: false` is essential for deterministic test assertions — with fuzz enabled, scheduling intervals vary randomly.
- `makeCardState()` factory pattern keeps test setup concise while allowing per-test overrides.
- Power-law decay at `t = stability` yields ~90% retention by FSRS design — useful as a smoke-test anchor.
