# Story E119-S10: Retention Policy Matrix

## Story

As Pedro (controller/operator),
I want `docs/compliance/retention.md` covering every data store with retention periods and deletion mechanisms,
mirrored to `src/lib/compliance/retentionPolicy.ts` with a parity test,
so that the retention matrix never drifts from the enforcing code.

## Acceptance Criteria

- **AC-1**: `docs/compliance/retention.md` is a table with columns: artefact (table/bucket/log), data categories, lawful basis, retention period, deletion mechanism, owner, notes. Covers: all 26 sync tables, 4 Storage buckets, auth/session logs, `syncQueue` dead-letter, breach register, invoices, exports bucket.
- **AC-2**: Retention defaults per plan §10:
  - UGC (notes/highlights/etc.): account lifetime + 30d.
  - AI chat transcripts: 365d rolling unless pinned.
  - Embeddings/learner model: purge with source or consent withdrawal.
  - Auth/session logs: 90d.
  - `syncQueue` dead-letter: 30d.
  - Breach register: 5y pseudonymised.
  - Invoices: 10y (placeholder — Pedro confirms EU member-state number with accountant during this story).
  - Exports bucket: 7d signed-URL TTL.
- **AC-3**: `src/lib/compliance/retentionPolicy.ts` exports typed entries mirroring the markdown matrix.
- **AC-4**: Parity test asserts every table in `tableRegistry`, every purpose in `consentService`, and every entry in `retentionPolicy.ts` are all present in `retention.md` (and vice versa).
- **AC-5**: Privacy notice `PrivacyPolicy.tsx` retention section links to `retention.md` and reflects actual periods.
- **AC-6**: A test surfaces entries with `period: null` (indefinite retention) as a flag requiring a reviewer sign-off comment.

## Tasks / Subtasks
