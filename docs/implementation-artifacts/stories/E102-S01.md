# Story 102.1: Bidirectional Progress Sync (REST)

Status: ready-for-dev

## Story

As a Knowlune user who also listens on the ABS mobile app,
I want my listening progress to sync between Knowlune and Audiobookshelf,
So that I can seamlessly continue from where I left off in either app.

## Acceptance Criteria

**Given** a user opens an ABS audiobook in Knowlune
**When** the book detail loads
**Then** the system fetches the user's listening progress from ABS via `AudiobookshelfService.fetchProgress()`
**And** if the ABS progress is further ahead than Knowlune's, Knowlune updates to the ABS position (FR33)

**Given** a user finishes a listening session in Knowlune
**When** the session ends (pause, close, or chapter complete)
**Then** the system pushes the current position to ABS via `AudiobookshelfService.updateProgress()` (FR34)
**And** the push includes: currentTime, duration, progress (0-1), isFinished

**Given** the ABS position is at 45:00 and Knowlune is at 30:00
**When** sync occurs on book open
**Then** Knowlune adopts the ABS position (45:00) because it has a later timestamp (FR35)

**Given** the Knowlune position is at 50:00 and ABS is at 45:00
**When** sync occurs on book open
**Then** Knowlune keeps its position (50:00) and pushes it to ABS
**And** conflict resolution uses latest-timestamp-wins strategy

**Given** the ABS server is unreachable during sync
**When** the push or fetch fails
**Then** the system silently queues the sync for next successful connection
**And** no error is shown to the user (sync is best-effort, not blocking)

### ABS REST API Endpoints for Progress Sync

**Fetch progress (FR33):**
```
GET /api/me/progress/{libraryItemId}
Authorization: Bearer <apiKey>
Response: {
  id: string,
  libraryItemId: string,
  currentTime: number,       // seconds
  duration: number,          // total seconds
  progress: number,          // 0-1 ratio
  isFinished: boolean,
  lastUpdate: number         // Unix timestamp ms
}
```
404 = no progress exists yet (new book) — treat as 0 position, not an error.

**Push progress (FR34):**
```
PATCH /api/me/progress/{libraryItemId}
Authorization: Bearer <apiKey>
Content-Type: application/json
Body: {
  currentTime: number,
  duration: number,
  progress: number,          // 0-1 ratio
  isFinished: boolean
}
Response: 200 OK (empty body on success)
```

**Note on ABS API versioning:** These endpoints are present in ABS v2.26.0+ (minimum version per NFR15). No backwards compatibility issues expected.

### Conflict Resolution: Latest-Timestamp-Wins (FR35)

```typescript
function resolveConflict(
  absProgress: AbsProgress,
  localBook: Book
): 'use-abs' | 'use-local' {
  const absTimestamp = absProgress.lastUpdate  // Unix ms from ABS
  const localTimestamp = localBook.lastOpenedAt
    ? new Date(localBook.lastOpenedAt).getTime()
    : 0

  return absTimestamp > localTimestamp ? 'use-abs' : 'use-local'
}
```

- If ABS timestamp is newer → adopt ABS position (update local Book record, seek audio)
- If local timestamp is newer or equal → keep local, push to ABS
- If no local progress (`currentPosition` null or 0) → adopt ABS unconditionally
- If ABS returns 404 (no progress) → keep local, push local to ABS

### Where to Hook the Sync

**Fetch-on-open:** Best entry point is inside `useAudioPlayer`'s `loadChapter` callback for `source.type === 'remote'`, before seeking to position. Or in `AudiobookRenderer` `useEffect` on `book.id` change. Do NOT add a new component — extend existing hooks.

**Push-on-session-end:** `useAudioListeningSession` already has session-end logic. Add the `updateProgress()` call after recording the local session (fire-and-forget with queue fallback).

### Sync is Best-Effort — Never Blocking

This is critical: sync failures must NOT:
- Block book opening
- Show error toasts
- Interrupt playback
- Prevent the session from being recorded locally

Sync is an async side-effect. The UX should be transparent — users just notice their position is right.

### AudiobookshelfService Pattern — Follow Exactly

```typescript
// src/services/AudiobookshelfService.ts — append these functions

export async function fetchProgress(
  url: string,
  apiKey: string,
  itemId: string
): Promise<AbsResult<AbsProgress>> {
  return absApiFetch<AbsProgress>(url, apiKey, `/api/me/progress/${itemId}`)
}

export async function updateProgress(
  url: string,
  apiKey: string,
  itemId: string,
  progress: { currentTime: number; duration: number; progress: number; isFinished: boolean }
): Promise<AbsResult<void>> {
  return absApiFetch<void>(url, apiKey, `/api/me/progress/${itemId}`, {
    method: 'PATCH',
    body: JSON.stringify(progress),
  })
}
```

The `absApiFetch` helper already handles: `AbortController` timeout, 401/403/500 error messages, `TypeError` for CORS/network, `DOMException` for timeout. **Do NOT reimplement error handling inline.**

### Pending Sync Queue — Keep It Simple

Store as in-memory array in `useAudiobookshelfStore` for this story. Queue persists for the session only — if the user refreshes, unsent progress is lost (acceptable for this story; Dexie persistence can be added if needed). The architecture doc deferred full offline queue to E102, so a lightweight in-memory approach is correct.

```typescript
// In useAudiobookshelfStore:
interface SyncQueueItem {
  serverId: string
  itemId: string
  payload: { currentTime: number; duration: number; progress: number; isFinished: boolean }
  enqueuedAt: string  // ISO timestamp for debugging
}

pendingSyncQueue: SyncQueueItem[]
enqueueSyncItem: (item: Omit<SyncQueueItem, 'enqueuedAt'>) => void
flushSyncQueue: () => Promise<void>  // called when server transitions to 'connected'
```
