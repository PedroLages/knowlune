# Story E119-S11: Retention TTL Enforcement Job

## Story

As Pedro (controller/operator),
I want a daily `retention-tick` Edge Function triggered by a Cloudflare cron at 03:00 UTC,
so that expired data is automatically deleted and the result is auditable.

## Acceptance Criteria

- **AC-1**: `retention-tick` Edge Function iterates `retentionPolicy` entries, computes cutoff dates, deletes expired rows/objects, and writes to `retention_audit_log(run_id, entry, rows_affected, started_at, completed_at, error)`.
- **AC-2**: Function is idempotent: re-running the same day finds nothing new to delete.
- **AC-3**: If one table errors mid-iteration, other tables continue processing; the failed table is logged with the error; the function returns non-2xx for alerting.
- **AC-4**: Dead-letter purge: `syncQueue` rows where `status = 'dead-letter'` and `updated_at < now() - 30d` are deleted.
- **AC-5**: Soft-delete grace finaliser: users whose `pending_deletion_at` is > 7 days old are hard-deleted via S03's cascade.
- **AC-6**: `exports/` bucket objects older than 7 days are purged (satisfying S06 TTL requirement).
- **AC-7**: Cloudflare cron trigger configured to call `retention-tick` daily at 03:00 UTC; configuration documented in `docs/deployment/retention-cron-setup.md`.
- **AC-8**: Heartbeat alert: if no audit log row in 48h, alert fires (op channel / structured log).
- **AC-9**: Supabase migration creates `retention_audit_log` table.
- **AC-10**: E2E staging: manual dry-run with seeded expired data verifies correct rows deleted and audit log populated.
