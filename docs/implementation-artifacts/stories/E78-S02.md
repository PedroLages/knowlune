# Story 78.02: Plugin Manager and Lifecycle State Machine

Status: ready-for-dev

## Story

As a Knowlune user,
I want the app to discover, register, and manage the lifecycle of installed plugins,
so that plugins are tracked reliably and I can install/uninstall them cleanly.

## Acceptance Criteria

1. **Given** a plugin npm package matching `knowlune-plugin-*` is present in node_modules **When** PluginManager initializes on app start **Then** the plugin is discovered and its manifest is validated **And** a new `PluginRecord` is created in the Dexie `plugins` table with state `inactive`.

2. **Given** a Dexie schema migration adds `plugins` and `pluginData` tables **When** the app starts for the first time after the migration **Then** both tables exist with correct indexes (`plugins`: `id, state, installedAt`; `pluginData`: `[pluginId+key], pluginId`) **And** the migration uses the existing `upgrade()` callback pattern.

3. **Given** PluginManager tracks a plugin in state `active` **When** the user deactivates the plugin **Then** the plugin transitions to `inactive`, its EventBus subscriptions are cleaned up, and its PluginSlot stops rendering.

4. **Given** a user uninstalls a plugin **When** the uninstall flow completes **Then** all `pluginData` rows matching the plugin's ID are deleted **And** the `plugins` record is removed **And** no artifacts remain in IndexedDB.

5. **Given** PluginManager is a singleton **When** multiple components access it **Then** they all share the same instance and state.

## Dev Notes

### Architecture Decisions

- **Singleton PluginManager**: Module-level exported instance, NOT a class with `getInstance()`. Pattern: `export const pluginManager = new PluginManager()` at module scope.
- **State machine**: Four states: `active`, `inactive`, `error`, `uninstalled` (transient). The `error` state is set by E78-S04 (auto-disable on 3 crashes) — this story implements the state field but does not implement crash detection.
- **Plugin discovery**: In a Vite SPA, we cannot scan `node_modules` at runtime. Discovery must use a **build-time** or **config-based** approach. Two viable options:
  1. **Build-time plugin registry**: A Vite plugin or build script that scans `node_modules` for `knowlune-plugin-*` and generates a `src/plugins/registry.ts` file with imports
  2. **Manual registration**: User adds plugins to a config file (`plugins.config.ts`)
  - Recommended: **Manual registration** for Phase 1 MVP simplicity. E79+ can add auto-discovery.

### PluginRecord Type (from Architecture Doc)

```typescript
interface PluginRecord {
  id: string                // Plugin npm package name
  version: string
  state: 'active' | 'inactive' | 'error'
  settings: Record<string, unknown>
  crashCount: number
  crashWindowStart: string  // ISO timestamp
  installedAt: string       // ISO timestamp
  lastActivatedAt?: string
  lastError?: string
}

interface PluginData {
  pluginId: string
  key: string
  value: unknown
}
```

### Dexie Migration Pattern

Existing pattern in `src/db/schema.ts` — increment the version number and add new table declarations. The current schema uses `EntityTable` typed generics. Follow the same pattern:

```typescript
// In the .stores() chain, add:
plugins: 'id, state, installedAt',
pluginData: '[pluginId+key], pluginId'
```

Also add the typed table properties to the `ElearningDatabase` type interface.

### Critical Conventions

- **Import alias**: `@/` resolves to `src/`
- **Zustand selectors**: Always individual selectors, never destructure the full store
- **Optimistic updates**: Zustand update first (instant UI), then Dexie persist. Rollback on failure. (See `docs/project-context.md`)
- **No hardcoded colors** in any UI this touches

### Testing Standards

- **Framework**: Vitest with `fake-indexeddb` for Dexie testing
- **Dexie mocking**: Use `fake-indexeddb` package already in project for IndexedDB tests
- **EventBus mocking**: Import the real eventBus singleton and spy on `.on()` / `.off()` methods
