# Story 70.3: Scheduling Store and Suggestion API

Status: ready-for-dev

## Story

As a learner,
I want the scheduling engine's results accessible through React hooks,
so that UI components can display study time suggestions without directly calling the computation engine.

## Acceptance Criteria

1. **Given** the scheduling Zustand store exists at `src/stores/useSchedulingStore.ts`, **When** a component calls `useStudySuggestions(courseId, duration)`, **Then** the store reads `studySessions`, `courses`, `studyBlocks`, and `flashcards` from Dexie. **And** passes the data as plain objects to the scheduling engine (`mergeSlots` then `computeSlots`). **And** returns `{ suggestions: ScoredSlot[], isLoading: boolean, refresh: () => void }`. **And** suggestions are cached to avoid recomputation on re-renders.

2. **Given** the store also exposes `useWeekPlan()`, **When** a component calls `useWeekPlan()`, **Then** it returns `{ plan: WeekPlan | null, isLoading: boolean, generate: () => void, accept: (allocation) => void, acceptAll: () => void }`.

3. **Given** the store calls `refresh()`, **When** underlying data has changed (new session completed, calendar synced), **Then** suggestions are recomputed using fresh data from Dexie.

4. **Given** the scheduling constants module exists at `src/lib/scheduling/constants.ts`, **When** imported, **Then** `SIGNAL_WEIGHTS`, `MIN_SESSION_MINUTES` (15), `MAX_SLOTS_PER_DAY` (3), `DEFAULT_SUGGESTION_COUNT` (5), and `SCHEDULING_HORIZON_DAYS` (7) are exported.

5. **Given** the scheduling barrel export exists at `src/lib/scheduling/index.ts`, **When** imported, **Then** all public functions (`computeSlots`, `mergeSlots`, `buildProductivityHeatmap`, `autoSchedule`) and types are re-exported.

## Dev Notes

### Architecture Compliance

**Store boundary (CRITICAL):** `useSchedulingStore.ts` is the ONLY module that bridges Dexie and the scheduling engine. It reads from Dexie, transforms to plain objects, passes to the pure engine, and stores results.

- `src/lib/scheduling/` -- ZERO Dexie imports (pure computation)
- `src/stores/useSchedulingStore.ts` -- Dexie imports allowed, React hooks allowed

**Depends on E70-S01 + E70-S02:** Uses `mergeSlots`, `computeSlots`, `buildProductivityHeatmap`, all types, and constants.

### Dexie Data Transformation

The store reads these Dexie tables and transforms to plain objects for the engine:

| Dexie Table | Engine Input | Transformation |
|-------------|-------------|----------------|
| `studyBlocks` | `mergeSlots({ studyBlocks })` | Extract `{ start, end }` from each block |
| `studySessions` | `buildProductivityHeatmap(sessions)` | Extract `{ startTime, duration, qualityScore }` |
| `courses` | `computeSlots({ courseDeadline, ... })` | Extract deadline, active status |
| `flashcards` | `computeSlots({ fsrsDueCount })` | Count due cards per course (or null if FSRS unavailable) |

### Study Block Persistence

When user accepts an allocation, write to Dexie `studyBlocks` table:

```typescript
await db.studyBlocks.add({
  id: generateId(),
  courseId: allocation.courseId,
  start: allocation.slot.window.start,
  end: allocation.slot.window.end,
  type: allocation.sessionType,
  createdAt: new Date().toISOString(),
  source: 'smart-scheduling'  // distinguish from manual blocks
})
```
