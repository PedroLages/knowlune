# Story 102.3: Collections

Status: ready-for-dev

## Story

As a Knowlune user with collections in Audiobookshelf,
I want to access my ABS collections as browsable groups,
so that I can organize my library by topic or project as I do in ABS.

## Acceptance Criteria

**Given** a connected ABS server with user-created collections
**When** the user selects a "Collections" view on the Library page
**Then** ABS collections are listed by name with item count (FR38)

**Given** a collection is selected
**When** the user taps a collection card
**Then** the collection expands to show all audiobooks in the collection
**And** each book displays the same card format as the main library grid

## Design Guidance

### Layout Approach

The Collections view follows the same structural pattern as the Series view introduced in E102-S02. The Library page already has tab-based source/view switching from E101; "Collections" is an additional tab alongside "All", "Local", "Audiobookshelf", and "Series".

**Two-level navigation:**
1. **Collections list** — grid of `CollectionCard` components, each showing collection name and book count
2. **Collection detail** — when a card is tapped, either:
   - Expand inline (accordion-style, consistent with how Series works in E102-S02), OR
   - Navigate to a detail view (full-page or drawer) showing all books in the collection

Defer to the E102-S02 Series implementation pattern — use the same UX model for consistency. If Series uses inline expansion, Collections uses inline expansion.

### Component Reuse

- `BookCard` — reuse unchanged for displaying books inside a collection. Do NOT create a new card variant.
- `LibrarySourceTabs` — extend the existing tab component, do not create a parallel navigation structure.
- Skeleton loading from Library grid — reuse the same skeleton pattern.

### Design System

- Use design tokens only — never hardcoded Tailwind colors (ESLint `design-tokens/no-hardcoded-colors` blocks violations at save time)
- Collection cards: `bg-card`, `border-border`, `text-foreground`, `text-muted-foreground` for item count
- ABS source badge: consistent with how ABS books are badged in the Library grid (from E101-S03)

## Implementation Notes

### Architecture Decisions (follow strictly)

**AudiobookshelfService — must follow OpdsService pattern exactly:**
- Export pure functions, not a class
- `fetchCollections(url, apiKey)` takes both params each call — no stored state
- Returns `AbsResult<AbsCollection[]>` discriminated union
- Uses `absApiFetch()` helper with `AbortController` timeout (10s)
- Never throw — catch returns `{ ok: false, error: '...' }`
- File: `src/services/AudiobookshelfService.ts` (add to existing file, do NOT create a new service file)

**ABS Collections API endpoint:**
```
GET /api/collections
Authorization: Bearer <apiKey>
Response: { results: AbsCollection[], total: number, page: number, limit: number }
```

Each `AbsCollection` in the response contains:
```typescript
{
  id: string
  libraryId: string
  name: string
  description?: string
  books: Array<{
    id: string          // ABS item ID — matches Book.absItemId
    media: {
      metadata: { title: string; authorName: string; ... }
    }
  }>
}
```

**Resolving collection books to local Book records:**
- Collection books from ABS have `absItemId` values
- Match against `useBookStore` books via `book.absItemId` to get full Book records (cover, progress, etc.)
- Books in a collection that haven't been synced yet (not in local Dexie) may not resolve — show ABS metadata as fallback

**useAudiobookshelfStore — must follow useOpdsCatalogStore pattern:**
- `create<State>((set, get) => ({...}))` shape
- Dexie-first: update Dexie then `set()` state
- Individual Zustand selectors at call sites — never destructure full store
- `toast.error()` on all catch blocks

**Naming conventions (mandatory — anti-patterns blocked by ESLint):**
- Type: `AbsCollection` (not `ABSCollection`, not `Collection`)
- Store field: `collections`, action: `loadCollections()`
- Components: `CollectionCard.tsx`, `CollectionsView.tsx`, `CollectionDetailView.tsx`
- File location: `src/app/components/library/` (not a new folder)

## Challenges and Lessons Learned

- **Collection detail implemented inline rather than as a separate view**: The story spec described a `CollectionDetailView.tsx` component, but following the E102-S02 Series pattern, collection expansion was implemented inline within `CollectionCard.tsx` using accordion-style toggling. This kept navigation simple and consistent with how Series works.
- **Book resolution from ABS item IDs**: Collections return ABS item IDs that must be resolved against the local book store via `absItemId`. Books not yet synced locally fall back to ABS metadata display. This two-tier resolution pattern is reusable for any ABS entity that references books.
- **Tab visibility conditional on server connection**: The Collections tab only appears when at least one ABS server is connected, consistent with the Series tab from E102-S02. This prevents confusing empty states for users who haven't configured ABS.
- **fetchCollections follows the established discriminated union pattern**: Added to the existing `AudiobookshelfService.ts` using `absApiFetch()` with `AbsResult<AbsCollection[]>` return type — no new service file, no class, consistent with OpdsService pattern.
