# Story 76.1: Email Digest Schema Migration

Status: ready-for-dev

## Story

As a developer,
I want the notification_preferences table extended with email digest columns,
so that the system can store per-user digest preferences and track send history.

## Acceptance Criteria

1. **Given** the Supabase migration infrastructure is in place (E44-E49 completed)
   **When** the migration is applied
   **Then** the notification_preferences table has three new columns:
   - `email_digest_enabled BOOLEAN DEFAULT false` (opt-in by default)
   - `digest_frequency TEXT DEFAULT 'weekly' CHECK (digest_frequency IN ('weekly', 'daily', 'off'))`
   - `last_digest_sent_at TIMESTAMPTZ` (nullable, tracks last send time)
   **And** a partial index `idx_notif_prefs_digest_enabled` exists on `email_digest_enabled` WHERE `email_digest_enabled = true`

2. **Given** the NotificationPreferences TypeScript type in `src/data/types.ts`
   **When** the type is updated
   **Then** it includes `emailDigestEnabled: boolean`, `digestFrequency: 'weekly' | 'daily' | 'off'`, and `lastDigestSentAt: string | null`
   **And** the Zustand store `useNotificationPrefsStore.ts` exposes `emailDigestEnabled`, `digestFrequency`, and setter actions

3. **Given** existing notification_preferences rows
   **When** the migration runs
   **Then** existing rows receive default values (`email_digest_enabled = false`, `digest_frequency = 'weekly'`, `last_digest_sent_at = null`) without data loss

## Dev Notes

### Architecture Decisions

- **AD-2**: Extend existing `notification_preferences` table rather than creating a new table. E58 already created this table with per-type toggles; email digest is another notification type.
- **Opt-in by default**: `email_digest_enabled = false` — users must explicitly enable digests (CAN-SPAM/GDPR compliance).
- **Dual storage**: Currently Dexie (IndexedDB) locally; Supabase migration prepares for E44-E49 sync. Both must be updated.

### Existing Code to Modify

| File | Change |
|------|--------|
| `src/data/types.ts` (line ~448) | Add 3 fields to `NotificationPreferences` interface |
| `src/stores/useNotificationPrefsStore.ts` | Add defaults + `setEmailDigest` action |
| `src/db/schema.ts` | Bump Dexie version, add new fields |
| `supabase/migrations/` (new file) | SQL migration for PostgreSQL columns + index |

### Patterns to Follow

- **Store pattern**: Follow `setQuietHours` in `useNotificationPrefsStore.ts` for the new `setEmailDigest` action — same optimistic update + `persistWithRetry` + `toast.error` on failure pattern.
- **Type pattern**: New fields go inside the existing `NotificationPreferences` interface after the quiet hours block.
- **Migration pattern**: Use `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` for idempotency.

### Critical Constraints

- Do NOT create a new table — extend the existing `notification_preferences` table (AD-2).
- Do NOT change the `id: 'singleton'` pattern — single-row config is by design.
- The `digest_frequency` CHECK constraint must include `'off'` as a valid value (distinct from `email_digest_enabled = false` — allows preserving last-selected frequency when toggling off).

### Project Structure Notes

- Supabase migrations go in `supabase/migrations/` with timestamp prefix (check existing migrations for naming convention).
- TypeScript types are centralized in `src/data/types.ts`.
- Zustand stores are in `src/stores/`.
