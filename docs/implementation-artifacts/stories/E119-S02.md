# Story E119-S02: Notice Acknowledgement at Signup + Material-Change Re-ack

## Story

As a user (data subject),
I want to explicitly acknowledge the privacy notice at signup and whenever it materially changes,
so that my consent is properly recorded and Knowlune has evidence of my acknowledgement.

## Acceptance Criteria

- **AC-1**: Supabase migration creates `notice_acknowledgements(user_id, document_id, version, acknowledged_at, ip_hash)` with RLS owner-only.
- **AC-2**: Signup flow includes checkbox "I have read the Privacy Notice (v{version})" linking to `/legal/privacy`; submit is disabled without it.
- **AC-3**: On successful signup + checkbox, an acknowledgement row is written with the correct version.
- **AC-4**: `useNoticeAcknowledgement` hook compares the user's latest ack version against `CURRENT_NOTICE_VERSION`; returns `{ acknowledged, stale }`.
- **AC-5**: When stale, `LegalUpdateBanner` displays a re-ack banner (non-blocking for 30 days).
- **AC-6**: After 30 days without ack, the app enters read-only mode (soft-block) via `useSoftBlock()` flag with "Acknowledge to continue" CTA; read actions still work, write actions surface the gate.
- **AC-7**: The 30-day window is calculated from server-side notice release date, not client clock.
- **AC-8**: E2E test covers: new user signup ack, stale-version re-ack, and soft-block transition.

## Context

- Depends on S01 for `CURRENT_NOTICE_VERSION` constant.
- `LegalUpdateBanner.tsx` shell already exists; extend for soft-block mode.
- Use deterministic time helper from `.claude/rules/testing/test-patterns.md` for E2E clock manipulation.
- RLS must prevent cross-user ack reads.
- Soft-block: read-only mode, write actions are gated. CTA is clear — "Acknowledge to continue".
- Error path: if ack write fails, user sees retry CTA, is NOT silently blocked from signup.
