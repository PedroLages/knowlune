# Story 81.01: Webhook Registration and Management

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a Knowlune user,
I want to register webhook URLs for specific learning events,
so that external tools are notified automatically when I complete courses or reach streak milestones.

## Acceptance Criteria

1. **Given** the user navigates to Settings > Developer, **When** webhooks are supported, **Then** a "Webhooks" subsection appears below the API tokens section showing a list of registered webhooks or empty state.

2. **Given** the user clicks "Add Webhook", **When** the form opens, **Then** they can enter: URL (required, validated as https://), event types (checkboxes: course.completed, streak.milestone, quiz.submitted, session.completed), and optional secret for HMAC signature verification.

3. **Given** a webhook is registered for `course.completed`, **When** a course is completed in Knowlune, **Then** a POST request is sent to the webhook URL with JSON payload containing event type, timestamp, and relevant data (course name, completion date), **And** the payload excludes sensitive data (no API tokens, no password hashes) (NFR9).

4. **Given** the user views their webhook list, **When** webhooks exist, **Then** each row shows: URL (truncated), event types as badges, last delivery status (success/failed/pending), and Delete button.

## Dev Notes

- **Architecture**: Webhooks are stored in Supabase and dispatched via Edge Function. The client-side EventBus triggers the Edge Function when matching events occur. This keeps webhook delivery server-side for reliability.
- **Dependency on E80-S01**: The DeveloperPanel component created in E80-S01 must exist. Webhooks are added as a subsection within it.
- **NFR9**: Webhook payloads MUST exclude sensitive data. Sanitize payloads to include only: event type, timestamp, and non-sensitive entity data (names, dates, progress -- no tokens, no passwords, no internal IDs that could be exploited).
- **HMAC signing**: If a webhook has a `secret`, compute `HMAC-SHA256(secret, JSON.stringify(body))` and include as `X-Knowlune-Signature` header. This allows consumers to verify payload authenticity.
- **HTTPS only**: Webhook URLs must use HTTPS. This is a security requirement -- no plaintext HTTP endpoints.
- **EventBus integration**: The existing EventBus in `src/lib/eventBus.ts` already has typed events. The WebhookTriggerService subscribes to relevant events and triggers dispatch. [Source: architecture-api-plugin-system.md#Decision 9]
- **Design tokens**: Use `bg-surface-sunken/30` for section headers, `text-muted-foreground` for metadata, `Badge variant="secondary"` for event type badges.
- **Retry logic**: NOT in this story. E81-S02 adds retry with exponential backoff. This story records delivery status but does not retry on failure.
