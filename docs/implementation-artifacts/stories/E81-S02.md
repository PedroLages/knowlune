# Story 81.02: Webhook Retry with Exponential Backoff

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a Knowlune user,
I want webhook deliveries to be retried on failure,
so that temporary endpoint outages don't cause me to miss events.

## Acceptance Criteria

1. **Given** a webhook delivery fails (non-2xx response or timeout), **When** the retry logic activates, **Then** the delivery is retried with exponential backoff: 1 minute, 5 minutes, 30 minutes, 2 hours, **And** maximum 4 retry attempts.

2. **Given** all retry attempts fail, **When** the final retry fails, **Then** the webhook delivery is marked as `failed` in the `webhook_deliveries` table, **And** the user can see the failure status in Settings.

3. **Given** a webhook delivery succeeds on retry, **When** the response is 2xx, **Then** the delivery is marked as `delivered`, **And** the retry queue clears for that delivery.

4. **Given** webhook delivery tracking, **When** the user views a webhook in Settings, **Then** they can see recent delivery attempts with status and timestamp.

## Dev Notes

- **Architecture**: Retry logic is server-side via Supabase Edge Functions. The `dispatch-webhook` function handles initial delivery and tracking. The `process-webhook-retries` function handles scheduled retries. This decouples retry logic from the client.
- **Dependency on E81-S01**: Requires the `webhooks` table and `dispatch-webhook` Edge Function from E81-S01.
- **Backoff schedule**: Exponential backoff with fixed intervals: 1 min -> 5 min -> 30 min -> 2 hours. Total retry window: ~2.5 hours from first failure. After 4 retries, the delivery is permanently marked as failed.
- **pg_cron for retry processing**: The retry processor needs to run periodically. Use Supabase's pg_cron extension (`SELECT cron.schedule('process-webhook-retries', '*/1 * * * *', ...)`) or Supabase Queues if available. The 60-second interval balances timeliness with overhead.
- **Response body truncation**: Store only the first 500 characters of the response body to avoid bloating the deliveries table with large error pages.
- **Delivery retention**: Consider adding a cleanup job to delete deliveries older than 30 days to prevent table growth. Not required for this story but worth noting for future maintenance.
- **Design tokens**: Delivery status indicators use `text-success` for delivered, `text-destructive` for failed, `text-muted-foreground` for pending.
