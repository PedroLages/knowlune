# Story 101.4: Streaming Playback

## Story

As a Knowlune user browsing their Audiobookshelf library,
I want to tap an audiobook and have it stream directly from my server,
So that I can listen without downloading or importing files.

## Acceptance Criteria

**Given** a user taps an ABS audiobook in the Library
**When** the AudiobookRenderer opens
**Then** the book opens in the same BookReader layout as local audiobooks
**And** audio streams directly from the ABS server without downloading to OPFS (FR15)

**Given** the useAudioPlayer hook loads a chapter for an ABS book
**When** `book.source.type === 'remote'`
**Then** the hook constructs the streaming URL via `AudiobookshelfService.getStreamUrl()`
**And** sets `audio.src` to the streaming URL with token query parameter
**And** does NOT call `opfsStorageService.readBookFile()` or `URL.createObjectURL()`
**And** playback starts within 2 seconds on LAN (NFR2)

**Given** the useAudioPlayer hook loads a chapter for a local book
**When** `book.source.type === 'local'`
**Then** the existing OPFS path is used with `createObjectURL()` (no regression)

**Given** audio is streaming from ABS
**When** the user interacts with playback controls
**Then** play, pause, seek, skip forward (30s), and skip backward (15s) all work (FR16)
**And** all controls are keyboard-navigable (NFR10)

**Given** audio is streaming
**When** the user adjusts playback speed
**Then** speed changes between 0.5x and 3x with pitch preservation (FR17)
**And** the speed persists for the session

**Given** audio is streaming
**When** the user sets a sleep timer
**Then** the timer options work (15/30/45/60 min, end of chapter) (FR18)
**And** playback pauses when the timer expires

**Given** an audiobook with multiple chapters
**When** the user opens the chapter list
**Then** chapters are displayed with titles from ABS metadata (FR19)
**And** tapping a chapter starts streaming from that chapter

**Given** the user is listening to an ABS audiobook
**When** they close the player and reopen the same book within the session
**Then** playback resumes from the last known position (FR20)
**And** the position is stored in the Book record's `currentPosition` field

**Given** the ABS server becomes unreachable during streaming
**When** the audio element fires an error event
**Then** a toast appears: "Lost connection to server. Check your network."
**And** playback pauses (does not retry automatically)
**And** the play button remains active for manual retry

## Design Guidance

This story has no new UI components — it's a pure hook/service modification. The AudiobookRenderer UI is completely unchanged. Playback controls, speed slider, sleep timer, chapter list, and bookmark FAB are all inherited from E87.

**Key UX concern**: Streaming start time (NFR2: <2s on LAN). The `canplay` event fires when enough data is buffered — this is the correct signal to seek and play. Do NOT use `loadedmetadata` for seeking (fires too early, seek may fail on streams).

**Error UX**: When the ABS server becomes unreachable mid-stream, `audio` fires an `error` event (code `MEDIA_ERR_NETWORK`). Show toast: "Lost connection to server. Check your network." — do not retry automatically. The play button remains enabled for manual retry.

## Implementation Notes

### Critical Architecture Decision: Remote Books as Single-File Streams

ABS audiobooks stream as a single continuous file (the server sends the full M4B/MP3). Chapter navigation is achieved by seeking `audio.currentTime` to the chapter's `position.seconds` value — exactly like local M4B playback. This means:
- `isSingleFileAudiobook()` returns `true` for remote books
- The single-file `singleFileRef` tracking is reused for chapter boundary detection
- The `handleEnded` path already handles book completion correctly for single-file mode

Do NOT attempt per-chapter file loading for remote books (the ABS API doesn't serve chapters as separate files in the simple case).

### Bearer Token via Query Parameter

HTML5 `<audio>` cannot send `Authorization` headers. ABS supports authentication via `?token=<apiKey>` query parameter for streaming endpoints. This is the standard approach used by ABS mobile apps. The API key appears in the URL but this is acceptable for LAN-only usage (API key stays on the local network).

### ContentSource `auth` Extension

The existing `ContentSource.remote.auth` uses `{ username, password }` for OPDS Basic Auth. ABS uses Bearer token. The cleanest approach is to extend the discriminated union within `auth`:

```typescript
type RemoteAuth =
  | { username: string; password: string }  // OPDS Basic Auth (E88)
  | { bearer: string }                       // ABS Bearer token (E101)

export type ContentSource =
  | { type: 'local'; opfsPath: string }
  | { type: 'remote'; url: string; auth?: RemoteAuth }
  | { type: 'fileHandle'; handle: FileSystemFileHandle }
```

Then in `useAudioPlayer` remote branch: `const apiKey = auth && 'bearer' in auth ? auth.bearer : ''`

**Alternative**: Store the API key as `absItemId` lookup → `useAudiobookshelfStore.getServer(book.absServerId).apiKey`. This avoids putting the key in the Book record. Evaluate which E101-S01 chose and be consistent.

### `_loadedBookId` tracking for remote books

The `_loadedBookId` singleton tracks which book's M4B is currently loaded to avoid re-fetching. For remote books, set `_loadedBookId = book.id` when the stream URL is assigned. On chapter seek (same book), the check `_loadedBookId !== book.id` prevents reloading the stream. Seeking `audio.currentTime` is sufficient for chapter nav on an already-loaded stream.

### Anti-Patterns to Avoid

- **DO NOT** create a `useRemoteAudioPlayer` hook — source abstraction belongs inside `loadChapterInternal` only
- **DO NOT** add `book.source.type` checks anywhere outside `loadChapterInternal` and `isSingleFileAudiobook`
- **DO NOT** use `waitForTimeout()` in tests — use `audio.addEventListener('canplay', ...)` or store state assertions
- **DO NOT** log the API key (it appears in streaming URLs — don't log the full URL either)
- **DO NOT** auto-retry on stream error — show toast, keep play button active for manual retry
- **DO NOT** use `Date.now()` or `new Date()` in tests — use `FIXED_DATE` pattern per project conventions

### MSW Handler Requirements

The streaming spec needs MSW handlers that mock the ABS streaming endpoint to return a response (can be a minimal valid audio response or just a 200 OK). The key is that `audio.src` is set correctly — browser audio behavior is tested via E2E assertions on state, not actual playback.

```typescript
// tests/support/msw/audiobookshelf-handlers.ts (already exists from E101-S01)
// Add handler for streaming endpoint if not already present:
http.get('http://192.168.1.50:13378/s/item/:itemId', ({ request }) => {
  const url = new URL(request.url)
  const token = url.searchParams.get('token')
  if (!token) return new HttpResponse(null, { status: 401 })
  // Return minimal valid response — Playwright will try to play this
  return new HttpResponse(new ArrayBuffer(0), { status: 200, headers: { 'Content-Type': 'audio/mpeg' } })
})
```
