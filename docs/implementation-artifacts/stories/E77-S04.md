# Story 77.4: Import Service with Schema Migration

Status: ready-for-dev

## Story

As a premium user,
I want Knowlune to safely import backup data into my local database,
so that I can restore my learning progress from any backup without data corruption.

## Acceptance Criteria

1. **Given** a `KnowluneExport` JSON object is provided for import **When** the schema version equals the current app version **Then** the import proceeds directly without migration.

2. **Given** a backup with schema version lower than the current app version **When** the migration chain runs **Then** each migration function is applied sequentially until the schema version matches current, and if no migration function exists for a version a `MigrationError` is thrown with "No migration path from schema {N}".

3. **Given** a backup with schema version higher than the current app version **When** the version check runs **Then** a `SchemaVersionError` is thrown with "This backup was made with a newer version of Knowlune. Please update the app."

4. **Given** the migrated/validated data is ready for import **When** post-migration validation runs **Then** record counts are verified for each table, critical field presence is checked (every course has `id`, every note has `courseId`), and validation failures throw a descriptive error before any IndexedDB writes (Adversarial #9).

5. **Given** the import begins **When** the Dexie transaction executes **Then** all existing tables are cleared within the transaction, all backup data is bulk-inserted table by table, and if any table write fails the entire transaction rolls back automatically with the user's existing data unchanged.

6. **Given** the import completes successfully **When** the transaction commits **Then** the function returns a summary of imported item counts: `{ courses: N, notes: N, sessions: N, ... }`.

## Dev Notes

### Architecture Constraints

- **This story has NO UI** -- it is a pure service layer consumed by Story 77.5
- **Can be developed in parallel with Stories 77.2 and 77.3** (no dependencies on Drive service)
- **Reuse `KnowluneExport` interface** from `src/lib/exportService.ts` -- do NOT create a new import schema
- **Reuse `CURRENT_SCHEMA_VERSION`** constant from `src/lib/exportService.ts`
- **Dexie transaction** provides atomic rollback (NFR-11) -- if any table write fails, all changes are reverted

### Migration Chain Pattern

```typescript
const migrations: Record<number, (data: KnowluneExport) => KnowluneExport> = {
  13: (data) => {
    // Example: add new fields with defaults for schema 13 -> 14
    data.data.studySessions = data.data.studySessions.map(s => ({
      ...s,
      focusScore: s.focusScore ?? null,
    }))
    data.schemaVersion = 14
    return data
  },
  // Future migrations added here as schema evolves
}
```

### Dexie Tables (from schema.ts)

All tables that must be cleared and repopulated during restore:
- `importedCourses`, `importedVideos`, `importedPdfs`
- `progress`, `bookmarks`, `notes`
- `studySessions`, `contentProgress`
- `challenges`, `reviewRecords`
- `learningPaths`, `learningPathEntries`
- `aiUsageEvents`

### Post-Migration Validation Rules (Adversarial #9)

- Every `importedCourse` must have `id` (string) and `name` (string)
- Every `note` must have `id` (string) and `courseId` (string)
- Every `studySession` must have `id` (string)
- Every `importedVideo` must have `id` (string) and `courseId` (string)
- All arrays must be actual arrays (not null/undefined)
- Record counts must be >= 0

### Performance Target

- NFR-2: Restore (download + import) must complete in <20 seconds for typical backup sizes
