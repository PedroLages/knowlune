# Story 107.1: Fix Cover Image Display

## Story

As a reader,
I want book cover images to display correctly throughout the app,
so that I can visually identify my books in the library, reader, and mini-player.

## Acceptance Criteria

- **AC-1**: Cover images from EPUBs display correctly in Library grid view (BookCard)
- **AC-2**: Cover images display correctly in Library list view (BookListItem)
- **AC-3**: Cover images display correctly in audiobook player (AudiobookRenderer, AudioMiniPlayer)
- **AC-4**: Cover URL resolution handles OPFS, http/https, and undefined cases gracefully
- **AC-5**: Blob URLs are properly cleaned up on component unmount to prevent memory leaks

## Implementation Notes

- **Hook pattern**: Created `useBookCoverUrl` hook to resolve `opfs-cover://` and `opfs://` URLs to displayable blob URLs
- **Lifecycle management**: Hook automatically revokes blob URLs on unmount or when coverUrl changes
- **External URL passthrough**: http/https URLs pass through unchanged (Open Library covers)
- **Graceful fallback**: Returns `null` for missing covers, allowing components to show placeholder icons

## Testing Notes

- Unit tests cover: null/undefined URLs, external URLs, OPFS URLs, blob URL lifecycle, URL changes
- E2E smoke tests pass (navigation, overview, courses)
- No story-specific E2E spec created (cover display is visual, tested via unit tests)

## Challenges and Lessons Learned

### Implementation Challenges

1. **Custom protocol resolution**: The `opfs-cover://` and `opfs://` identifiers are meaningful to OpfsStorageService but not to the browser. An `<img src="opfs://book123">` won't load — the browser doesn't understand the protocol. Required a React hook to resolve these to blob URLs.

2. **Blob URL memory management**: Converting stored files to blob URLs via `URL.createObjectURL()` creates a reference that must be revoked with `URL.revokeObjectURL()` to prevent memory leaks. This is often overlooked in React components.

3. **Async resolution in useEffect**: The cover URL resolution is async, requiring careful handling of cancellation (`isCancelled` flag) to prevent state updates after component unmount.

4. **LinkFormatsDialog inconsistency**: During implementation, discovered that LinkFormatsDialog was using a different pattern for URL resolution. Fixed to use the new hook pattern consistently.

### Patterns Worth Remembering

1. **Resource URL hooks pattern**: Documented in `docs/engineering-patterns.md` — a reusable pattern for any resource that needs blob URL resolution (audio files, document previews, etc.)

2. **useRef for previous value cleanup**: Using `useRef` to track the previous blob URL allows cleanup in the effect's return function, even when the component is unmounting.

3. **Silent-catch-ok pattern**: In `useBookCoverUrl.ts`, the catch block is intentionally silent because cover resolution failure is not an error condition — it just means "no cover to display". This is noted with `// silent-catch-ok` comment to satisfy ESLint.

### Decisions and Trade-offs

- **Hook vs inline resolution**: Chose to create a reusable hook rather than inline resolution in each component. This ensures consistent lifecycle management and makes the pattern discoverable for future use cases.
- **Null vs placeholder**: Hook returns `null` for missing covers, letting the component decide how to render the placeholder. This keeps the hook focused on URL resolution.
