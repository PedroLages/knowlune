## Implementation Notes

- FSRS due checking uses `due` field (ISO 8601 string) compared to current time

## Testing Notes

- All 85 tests across 4 modified files pass (22 retentionMetrics + 9 interleave + 33 exportService + 21 NotificationService)

- NotificationService test updated to reflect FSRS semantics: new cards get `due: now` at creation (always immediately due) rather than the old SM-2 pattern of missing `nextReviewAt`.

## Challenges and Lessons Learned

- FSRS forgetting curve values differ significantly from SM-2 linear decay. Test data that produced "low retention" with SM-2 (interval=3, 14 days elapsed) may produce "fading" or higher retention with FSRS (stability=3 → 69%). Must calibrate stability values against actual `forgetting_curve()` output.
- The NotificationService `checkSrsDueOnStartup` uses the real `isDue` function (not mocked), so mock data must include valid `due` fields. The old test relied on missing `nextReviewAt` being treated as "due" — FSRS requires explicit `due` dates since all cards get one at creation.
