# Story 87.1: Audiobook Import and Data Model (Dexie v38)

Status: done

## Story

As a learner,
I want to import an audiobook as a folder of MP3 files,
so that my personal audiobook collection is available in Knowlune alongside my EPUBs.

## Acceptance Criteria

**Given** the Dexie schema is updated (v31 bump adding `audioBookmarks` table with indexes on `bookId`, `chapterIndex`, `timestamp`, `createdAt`)
**When** the user clicks "Import Book" on the Library page
**Then** the import dialog now also accepts "Audiobook (MP3 folder)" as an import type
**And** the user can select multiple MP3 files (one per chapter)

**Given** the user selects a folder of MP3 files
**When** the import processes
**Then** filenames are parsed for chapter order (e.g., `01-introduction.mp3`, `02-chapter-one.mp3`)
**And** metadata (title, author) is extracted from the first MP3 file's ID3 tags if available
**And** all MP3 files are stored in OPFS at `/knowlune/books/{bookId}/chapter-{nn}.mp3`
**And** a Book record is created with `format: 'audiobook'`, chapters array with `startTime` offsets, and `totalDuration` calculated from all files
**And** the book appears in the Library with an audiobook badge/indicator

**Given** the music-metadata library is installed **When** `npm run build` is executed **Then** the production build succeeds without Node.js module externalization warnings related to music-metadata.

### ID3 Tag Extraction

For lightweight ID3 parsing without adding a heavy dependency (music-metadata is deferred to E88 for M4B):

- Option A: Use the browser `Audio` element to get duration, and a lightweight ID3v2 parser (< 5KB) for title/artist/cover
- Option B: Parse raw ID3v2 header bytes manually (ID3v2 spec is well-documented, headers are at file start)
- Do NOT import `music-metadata` here — that's a ~200KB dependency reserved for M4B in E88

### Chapter Duration Calculation

```typescript
// Get duration of each MP3 using HTML5 Audio
async function getAudioDuration(file: File): Promise<number> {
  return new Promise((resolve) => {
    const audio = new Audio()
    audio.addEventListener('loadedmetadata', () => resolve(audio.duration))
    audio.src = URL.createObjectURL(file)
  })
}
```

### music-metadata Vite Compatibility

The `music-metadata` npm package includes Node.js-specific APIs that may cause Vite 6+ bundling errors. Known issue: [music-metadata-browser#836](https://github.com/Borewit/music-metadata-browser/issues/836).

**Mitigation strategy (try in order):**

1. Install `music-metadata` and run `npm run build` — if it succeeds with Vite's automatic browser polyfills, no further action needed
2. If Node.js module errors appear, add Vite resolve aliases in `vite.config.ts`:

   ```ts
   resolve: {
     alias: {
       'node:stream': 'stream-browserify',
       'node:buffer': 'buffer/',
     }
   }
   ```

3. If aliasing is insufficient, move metadata parsing to a Web Worker using Vite's `?worker` import pattern to isolate Node.js dependencies from the main bundle

**Important:** The `includeChapters: true` option must be set when parsing M4B files to extract chapter markers.
