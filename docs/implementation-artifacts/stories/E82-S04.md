# Story 82.04: Plugin Ecosystem Hardening and E2E Validation

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a Knowlune user,
I want the plugin ecosystem, API, and webhook systems to be thoroughly validated end-to-end,
so that all Phase 3-4 features work reliably together before the next development wave.

## Acceptance Criteria

1. **Given** the full API & Plugin System (E78-E82) is implemented, **When** an E2E validation suite runs, **Then** the complete flow works: generate API token -> call REST endpoint with token -> receive data -> register webhook -> trigger event -> webhook delivered with retry.

2. **Given** the plugin browse page and registry are live, **When** the user browses, selects, and "installs" a plugin, **Then** the flow works end-to-end: browse -> search -> filter by category -> view detail -> copy install command.

3. **Given** xAPI is configured, **When** a course is completed, **Then** an xAPI statement is sent to the configured LRS and the delivery is verifiable.

4. **Given** all E80-E82 components are integrated, **When** the Settings page loads, **Then** the Developer section shows API tokens, webhooks, and xAPI settings without layout conflicts, and the page remains performant (FCP within 15% of baseline).

## Dev Notes

- **This is an integration/hardening story.** It validates that E80 (API tokens), E81 (webhooks), and E82-S01 through E82-S03 (plugin browse, registry, xAPI) work together correctly.
- **E2E test patterns**: Follow existing Playwright test patterns in `tests/`. Use deterministic time helpers if testing time-based features (token expiry, retry backoff).
- **No new features**: This story does not add new functionality. It tests, fixes, and polishes existing E80-E82 features.
- **Performance baseline**: Settings page FCP must remain within 15% of pre-E80 baseline (NFR1 extension).
- **Test data**: Use Supabase seed data for API tokens and plugin registry. Mock external webhook endpoints for delivery testing.
