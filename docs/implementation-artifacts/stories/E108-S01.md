# Story 108.1: Bulk EPUB Import

## Story

As a reader with a large EPUB collection,
I want to import multiple EPUB files at once via drag-and-drop or folder selection,
so that I can quickly populate my library without importing books one at a time.

## Acceptance Criteria

- **AC-1**: User can select or drag-and-drop multiple `.epub` files at once into the Library page or import dialog
- **AC-2**: A bulk import progress indicator shows: current file name, completed count / total count, and a progress bar
- **AC-3**: Each EPUB is processed sequentially (not in parallel) to avoid OPFS contention — metadata extraction, cover fetch, and storage happen one at a time
- **AC-4**: If an individual EPUB fails to import (corrupt file, read error), the error is logged per-file and the batch continues — remaining files are not blocked
- **AC-5**: After bulk import completes, a summary toast shows: "Imported X of Y books" with count of any failures
- **AC-6**: Existing single-file import flow (`BookImportDialog`) continues to work unchanged

  - [ ] 2.5 Handle abort via `AbortController` for cancellation support

## Design Guidance

- **Layout**: Extend existing `BookImportDialog` with a "bulk mode" panel that replaces the single-file metadata form
- **Progress UI**: Use `Progress` shadcn component + file name label inside the dialog
- **Cancellation**: Add a "Cancel" button that sets `AbortController.abort()` during bulk processing

## Implementation Notes

**Key files to modify:**
- `src/app/components/library/BookImportDialog.tsx` — add multi-file support
- `src/stores/useBookStore.ts` — `importBook` action already exists (single book)

**New files:**
- `src/app/hooks/useBulkImport.ts` — sequential import queue hook

**Existing patterns to follow:**
- `extractEpubMetadata` from `src/services/EpubMetadataService.ts`
- `fetchOpenLibraryMetadata` from `src/services/OpenLibraryService.ts`
- `opfsStorageService.storeBookFile` from `src/services/OpfsStorageService.ts`
- Single-file import flow in `BookImportDialog.tsx` (lines 80-130)

**OPFS constraint**: OPFS write operations are asynchronous and should not be parallelized to avoid lock contention. Process files sequentially.
