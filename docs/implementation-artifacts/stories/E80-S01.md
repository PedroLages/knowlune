# Story 80.01: API Token Generation and Management UI

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a Knowlune user,
I want to generate, view, and revoke Personal API Tokens from Settings,
so that I can securely access my learning data from external tools and scripts.

## Acceptance Criteria

1. **Given** the user navigates to Settings, **When** the page renders, **Then** a "Developer" card section appears (after Quiz Preferences, before Data Management) showing a description, "Generate New Token" button, and token list (or empty state).

2. **Given** the user clicks "Generate New Token", **When** the dialog opens, **Then** it shows: token name input (required), scope checkboxes (Courses, Sessions, Flashcards, Quizzes, Streaks, Notes -- Courses and Sessions checked by default), expiry select (30 days, 90 days, 1 year, Never).

3. **Given** the user fills the form and clicks "Generate Token", **When** generation completes, **Then** a 32-byte cryptographically random token is generated via `crypto.getRandomValues()`, the SHA-256 hash is stored in the Supabase `api_tokens` table (id, user_id, name, token_hash, scopes, created_at, expires_at), the plaintext token (prefixed `klp_`) is displayed ONCE with a copy button, a warning banner shows: "Copy this token now. You won't be able to see it again.", and a quick-start curl snippet is displayed.

4. **Given** the user clicks Copy, **When** the token is copied to clipboard, **Then** the button text changes to "Copied!" for 2 seconds, and `aria-live="polite"` announces "Copied" to screen readers.

5. **Given** the user tries to close the token dialog without copying, **When** they click X or outside the dialog, **Then** an AlertDialog warns: "You haven't copied your new token. It won't be shown again after you close this dialog."

6. **Given** the user views their token list, **When** tokens exist, **Then** each row shows: name, creation date, last used date, scopes as badges, expiry, and a Revoke button.

7. **Given** the user clicks Revoke on a token, **When** they confirm in the AlertDialog, **Then** the token's `revoked_at` is set in Supabase, the token is removed from the list, and a toast confirms: "Token '{name}' revoked".

8. **Given** the Developer section is rendered, **When** using keyboard only, **Then** all actions are accessible via Tab/Enter/Space (NFR13).

## Dev Notes

- **Architecture**: Token generation happens entirely client-side (`crypto.getRandomValues` + `crypto.subtle.digest`). Only the SHA-256 hash is sent to Supabase. The plaintext is never stored anywhere. [Source: architecture-api-plugin-system.md#Decision 10]
- **Token prefix**: All tokens use `klp_` prefix for identifiability (Knowlune Personal token).
- **Supabase dependency**: This story requires Supabase to be configured. The `api_tokens` table uses RLS with `auth.uid() = user_id`. Since Knowlune uses self-hosted Supabase on Unraid, ensure the migration runs against the correct instance.
- **Settings page position**: UX spec places Developer section at position 16 (after Quiz Preferences, before Data Management). The existing Settings page has 8+ card sections in a vertical stack. [Source: ux-design-api-plugin-system.md#Section 4]
- **Design tokens**: All UI must use design tokens -- no hardcoded colors. Use `bg-surface-sunken/30` for CardHeader, `text-muted-foreground` for metadata, `text-destructive` for Revoke button, `bg-warning/10` for warning banners. ESLint `design-tokens/no-hardcoded-colors` will block violations.
- **shadcn/ui components**: Use Dialog, AlertDialog, Input, Checkbox, Select, Button (variant="brand" for Generate Token), Badge (variant="secondary" for scopes), Tooltip, Sonner toast.
- **Previous epic context**: E78 (Plugin Foundation) and E79 (Plugin Ecosystem) established Settings page plugin sections. This story adds the Developer section as a separate Card below plugins.
- **Scopes available**: `read:courses`, `read:sessions`, `read:flashcards`, `read:quizzes`, `read:streaks`, `read:notes`. All read-only by default. [Source: architecture-api-plugin-system.md#Decision 10]
