# Story 79.03: Lazy-Load Plugins on Activation Events

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a Knowlune user,
I want plugins to load only when needed based on declared activation events,
so that the app stays fast even with many installed plugins.

## Acceptance Criteria

1. **Given** a plugin declares `activationEvents: ["route:/courses/*"]`, **When** the user is on the Overview page, **Then** the plugin is NOT loaded (no `import()` call).

2. **Given** a plugin declares `activationEvents: ["route:/courses/*"]`, **When** the user navigates to any `/courses/` route, **Then** the plugin is loaded and activated on demand.

3. **Given** a plugin declares no `activationEvents`, **When** it is in `active` state, **Then** it is loaded eagerly at app start (backward-compatible default).

4. **Given** 5 plugins are installed but only 2 have activation events matching the current route, **When** performance is measured, **Then** FCP/LCP is within 15% of baseline (NFR1).

## Dev Notes

### Architecture Decisions

- **Activation event format**: `activationEvents` is an optional string array in the manifest. Values are prefixed: `route:<glob>` for URL matching, `event:<eventType>` for EventBus triggers. Absence means eager loading (backward-compatible with E78 plugins).
- **No new dependencies for glob matching**: Use a simple regex conversion (`*` -> `[^/]+`, `**` -> `.*`) rather than adding `minimatch` dependency. Keep bundle lean.
- **Loading strategy**: Pending plugins are registered in PluginManager's state but their module is NOT imported. Only when activation condition matches does `import()` execute. This ensures zero JS execution cost for inactive plugins.
- **Performance budget**: NFR1 requires 5 active plugins within 15% of baseline FCP/LCP. Lazy loading is the primary mechanism to achieve this. Each plugin bundle must be < 100KB gzipped (NFR3).

### Dependencies (from E78)

- `PluginManager` singleton (E78-S02) -- extend with `pendingActivation` state
- Plugin manifest `activationEvents` field (already typed in architecture, E78-S01)
- EventBus (E78-S05) for `event:` activation triggers
- React Router `useLocation()` for route-based triggers

### Existing Patterns to Follow

- EventBus subscription: `eventBus.on('course:completed', handler)` pattern from `src/lib/eventBus.ts`
- Lazy loading: `React.lazy(() => import(...))` pattern used throughout `src/app/routes.tsx`
- Location tracking: `useLocation()` from `react-router` (already used in Layout.tsx for active nav state)

### Performance Constraints

- NFR1: 5 active plugins within 15% baseline FCP/LCP
- NFR2: Individual plugin render < 200ms
- NFR3: Plugin bundle < 100KB gzipped
- Lazy plugins must NOT appear in initial Vite bundle -- verify with `npm run build` + bundle analysis

### Project Structure Notes

- Activation matcher in `src/plugins/` (core plugin infrastructure)
- Hook in `src/plugins/hooks/` following existing hook pattern
- No new npm dependencies required
