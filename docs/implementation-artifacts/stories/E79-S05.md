# Story 79.05: Plugin Hot Reload in Dev Mode

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a plugin author,
I want to develop my plugin with hot reload against a running Knowlune instance,
so that I see changes instantly without restarting the app.

## Acceptance Criteria

1. **Given** a developer runs `npm run dev -- --plugin ./path-to-plugin`, **When** Vite dev server starts, **Then** the plugin at the specified path is loaded into Knowlune alongside normal plugins, and changes to the plugin source trigger HMR without full page reload.

2. **Given** a plugin in dev mode throws a render error, **When** the error boundary catches it, **Then** the developer sees the error overlay with source location, and fixing the error triggers HMR recovery.

## Dev Notes

### Architecture Decisions

- **Vite plugin approach**: Create a custom Vite plugin (`vite-plugin-knowlune-dev-plugin`) that handles: (1) reading CLI args, (2) adding FS allow paths, (3) watching external directories, (4) triggering HMR. This keeps `vite.config.ts` clean.
- **Module resolution**: Dev plugins are outside the project root. Vite needs `server.fs.allow` to permit imports from external paths. The Vite plugin adds the plugin directory dynamically.
- **HMR mechanism**: Vite watches files in `server.fs.allow` paths. The custom Vite plugin listens for changes in the dev plugin directory and calls `server.moduleGraph.invalidateModule()` + `server.ws.send({ type: 'full-reload' })` or more granular HMR updates.
- **No production impact**: `DevPluginLoader` and the Vite plugin only activate in dev mode (`import.meta.env.DEV`). They are tree-shaken from production builds.
- **React Fast Refresh**: Vite's `@vitejs/plugin-react` handles Fast Refresh for `.tsx` files. As long as the dev plugin's components follow React component conventions (named exports, function components), Fast Refresh should work automatically.

### Dependencies

- Vite plugin API (`configureServer`, `handleHotUpdate` hooks)
- PluginManager (E78-S02) for registering dev plugins
- PluginErrorBoundary (E78-S04) for error display
- No new npm dependencies -- uses Vite's built-in plugin API

### Key Files to Create/Modify

| File | Change |
|------|--------|
| `src/plugins/DevPluginLoader.ts` | **NEW** - Loads plugin from local path in dev mode |
| `vite-plugin-knowlune-dev-plugin.ts` | **NEW** - Custom Vite plugin for HMR and FS access |
| `vite.config.ts` | Import and register the custom Vite plugin (dev only) |

### Existing Patterns to Follow

- Vite config: `vite.config.ts` already uses `@vitejs/plugin-react` and `@tailwindcss/vite` -- add the new plugin conditionally (`process.env.NODE_ENV !== 'production'`)
- Plugin registration: Follow same `PluginManager.register()` pattern used by the normal plugin discovery flow (E78-S02)
- Error boundaries: `PluginErrorBoundary` from E78-S04 already wraps plugin renders

### Design Token Usage

- Dev mode badge: `bg-warning text-warning-foreground` for "DEV" badge
- No hardcoded colors

### Important Caveats

- **Tree-shaking**: Ensure `DevPluginLoader.ts` is only imported behind `import.meta.env.DEV` guard so it does not appear in production bundles
- **Security**: Dev mode only works on `localhost` -- no mechanism for loading arbitrary external plugins in production
- **Vite version**: Vite 6.4.1 -- check API compatibility for `server.moduleGraph` and `server.ws.send`
