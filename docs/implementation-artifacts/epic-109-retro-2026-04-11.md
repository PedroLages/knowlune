## Patterns and Themes

**Theme 1 — Positive: Shared helpers compound across epics**

The `dismissOnboarding` helper created in E109-S01 was used in every subsequent E2E spec in the epic. The `seedBookHighlights` helper from S02 seeded vocabulary tests in S05. When a recurring E2E problem is solved by extracting a shared helper — rather than fixing it inline per story — the fix compounds. Every subsequent story starts from a higher floor. This is the correct pattern and it worked in E109.

**Theme 2 — Recurring (x2 epics): Hover-only interactive elements**

Buttons visible only on `:hover` with no `focus` or `touch` trigger appeared in E108 (keyboard shortcuts panel) and again in E109-S01 (flashcard review buttons). Both required review fixes. Pattern: when implementing hover-revealed actions, always ask "is this reachable via keyboard Tab and focus, and via touch?" before submitting for review. This should be in the story pre-review checklist.

**Theme 3 — New: Dexie `.notEqual()` is unreliable for nullable/absent fields**

`Dexie.where(field).notEqual(value)` excludes documents where the field is absent entirely, not just where it equals the target value. The reliable pattern is a full table scan with in-memory filter: `table.toArray()` followed by `arr.filter()`. Document why with a comment (`// Intentional: .notEqual() unreliable for nullable fields — full scan with in-memory filter`). (E109-S02, `loadDailyHighlights`)

**Theme 4 — Systemic (recurring x6): Retro → epic transition has no inter-epic prep gap**

Six consecutive epics with sub-20% action item follow-through. All trace to the same structural cause: retro closes, first story starts in the same session. Until a structural gate exists between epic close and next S01, chore commits and carry-forward items will not be executed. The fix requires a change to `finish-story/SKILL.md` or `CLAUDE.md` — not a checklist item.

**Theme 5 — Positive: Later stories benefit from earlier infrastructure investment**

S01 introduced the Dexie vocabulary schema, optimistic update store pattern, and E2E helpers. S02-S05 spent their budgets on feature logic rather than infrastructure. S03-S05 each passed in one review round. The investment made at the start of an epic accrues to every subsequent story. This argues for intentional infrastructure-first sequencing when planning epics.

## Lessons Learned

### 1. Dexie `.notEqual()` is unreliable for nullable/absent fields — use full table scan

`Dexie.where(field).notEqual(value)` silently excludes documents where the field is absent (undefined or missing from the schema). For any field that may be null or undefined in existing records, use a full table scan with in-memory filter: `await table.toArray()` followed by `arr.filter(item => item.field !== value)`. Add a comment: `// Intentional: .notEqual() unreliable for nullable fields — full scan with in-memory filter`. This pattern is safe at personal library scale. (E109-S02, `loadDailyHighlights`)

### 2. Optimistic updates require ref-based snapshots to avoid stale closures

When rolling back Zustand state on async failure, reading the pre-update state via a closure captures the value at hook-call time, not at callback-execution time. By the time an async Dexie write fails and the rollback runs, the closure value may already be stale if other state updates have occurred. Fix: capture the snapshot immediately before the optimistic update using a `useRef` or a local variable read from `get()` inside the callback. Pattern already documented for `useBookStore` — apply universally to all optimistic-update stores. (E109-S02)

### 3. Shared test helpers compound across stories — extract at first reuse signal

When a test helper is written inline in a story spec, the next story that needs the same logic will rediscover the problem independently. When extracted to `tests/helpers/`, every subsequent story starts with the solution available. The `dismissOnboarding` helper (extracted in E109-S01) was referenced in every E2E spec through E109-S05. The extraction cost is small; the compounding value is large. Extract at the first reuse signal, not after the third rediscovery. (E109, across all stories)

### 4. Hover-only interactive elements fail accessibility — always verify focus and touch reachability

Buttons or controls that are only visible/triggerable via CSS `:hover` are inaccessible to keyboard users (Tab focus does not trigger `:hover`) and may be unreliable on touch devices (hover state is transient). Any interactive element that appears or activates on hover must also respond to `:focus-visible` (keyboard) and be touchable without hover. Add to pre-review checklist: "Is every interactive element reachable via keyboard Tab and touch, without hover?" (E109-S01 flashcard review buttons; recurrence from E108)

### 5. NFR gaps survive story-level review — run NFR assessment before declaring epic done

Two HIGH NFR findings — AnnotationSummary infinite skeleton and SearchAnnotations unbounded load — were not caught during S04 and S05 story reviews. They were identified by the post-epic NFR assessment. Reliability gaps (missing try/catch) and performance gaps (missing load cap) can be invisible in story-level review when tests pass and the feature works under light load. Run the NFR assessment as part of the epic close sequence, before the retrospective, and resolve HIGH items before the next epic starts.

### 6. Infrastructure-first sequencing within an epic accelerates subsequent stories

S01 introduced the Dexie schema, store pattern, and E2E helpers. S03-S05 had a single review round each because the foundation was stable. When planning an epic, identify which stories introduce shared infrastructure (schema changes, store patterns, shared helpers) and sequence them first. Stories that build on established infrastructure consistently pass review faster. (E109, infrastructure sequencing)
