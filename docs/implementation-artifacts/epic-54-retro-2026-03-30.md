## Challenges and Growth Areas

### 1. E54-S03 Targeted Dead Code (ImportedCourseDetail)

The most significant challenge: E54-S03's story specification said to add checkmarks to `ImportedCourseDetail.tsx`, but that component was already replaced by `UnifiedCourseDetail` in the route table. The initial implementation added 40 lines to a file that is never rendered.

Code review (Round 1) caught this as MEDIUM severity. The fix required pivoting mid-story to wire checkmarks through `LessonList.tsx` (which `UnifiedCourseDetail` actually uses) and reverting the dead code changes.

**Root cause:** Epic planning referenced the original component name without verifying it was still routed. The `UnifiedCourseDetail` refactor (from Epic 89) happened between planning and execution.

**Lesson:** Before implementing UI changes, verify the target component is actually referenced in `routes.tsx`. A 30-second route check would have prevented 4 fix commits and an extra review round.

### 2. E54-S01 E2E Tests Were Fundamentally Broken (6/9 Failing)

Round 2 code review discovered that 6 of 9 E2E tests for E54-S01 relied on `simulateVideoEnd()` which waits for a `<video>` element. But seeded ImportedVideo records lack `fileHandle` properties, so `LocalVideoContent` renders "Video not found" instead of a `<video>` element. The tests could never pass as written.

**Root cause:** E2E test design assumed a `<video>` element would exist from seeded data alone, without understanding that `LocalVideoContent` requires a valid `fileHandle` (File System Access API) to render video.

**Lesson:** When writing E2E tests for components that depend on browser APIs (File System Access, WebRTC, etc.), verify the component's rendering prerequisites first. Stub or mock at the right level — don't assume seeded database records are sufficient.

### 3. Dual Progress Systems in YouTube Courses

The story notes flagged a risk: `YouTubeCourseDetail` uses `db.progress` directly while `YouTubeLessonPlayer` uses `useContentProgressStore`. The celebration's course-completion check reads from `statusMap` (the store), but the course detail page reads from the `progress` table. This means a video completed in the lesson player might not immediately reflect as a checkmark in the course detail until both systems sync.

This wasn't resolved in E54 — it was documented as a known architectural divergence. It hasn't caused user-visible bugs yet because both systems eventually write to the same underlying Dexie tables, but the two read paths could diverge under race conditions.

**Lesson:** When multiple components track the same domain concept (progress) through different stores/tables, explicitly document the reconciliation strategy or unify them.

### 4. Session-Complete Dialog Interference in E54-S02 E2E

The E54-S02 auto-advance cancel test intermittently failed because a "session complete" dialog appeared over the cancel button. The fix (`fb76fff3`) added dialog dismissal handling. This is a recurring pattern — overlay dialogs blocking E2E test interactions.

**Lesson:** E2E test helpers should defensively dismiss known overlays (onboarding, session-complete, notification prompts) before asserting on page content.

## Key Insights and Patterns

### Pattern 1: Cross-Player Feature Parity via Shared Components

Wiring the same `AutoAdvanceCountdown` and `CompletionModal` across 3 player types (regular, imported, YouTube) without modifying the components validates the component API design. Future player types (podcast, PDF) can follow the same wiring pattern.

### Pattern 2: Story Planning Should Include Route Verification

E54-S03's dead-code detour could have been avoided with a single line in the story template: "Verify target component is in routes.tsx." Adding this to the pre-implementation checklist prevents wasted effort on unreachable code.

### Pattern 3: "Already Wired" Inventory in Story Notes

E54-S02's story notes explicitly listed what was already wired (`useContentProgressStore`, `handleAutoComplete`) vs what needed building. This reduced implementation time by ~50% compared to E54-S01 (which had to discover the existing state from scratch). Future stories should always inventory existing infrastructure.
