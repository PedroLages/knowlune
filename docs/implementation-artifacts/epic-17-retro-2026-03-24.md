**Pedro:** Three things really stood out to me this epic.

**1. Pure function architecture paid massive dividends for psychometric calculations.** Every analytics function in this epic — `calculateCompletionRate`, `calculateRetakeFrequency`, `calculateItemDifficulty`, `calculateDiscriminationIndices`, `detectLearningTrajectory` — is a pure function with no side effects. The fix (switching to `useMediaQuery` with `addEventListener('change')`) is a better pattern that we should standardize. This was exactly the kind of issue that manual code review might miss but the adversarial review agent caught.

## Key Insights and Learnings

1. **Psychometric functions are ideal candidates for the pure function pattern.** P-values, discrimination indices, and regression models have well-defined mathematical inputs and outputs with no side effects. The entire analytics.ts module (6.14 kB) contains zero I/O calls in its pure functions — only `calculateCompletionRate` and `calculateRetakeFrequency` are async (they query IndexedDB). This separation makes the module the most testable in the codebase.

2. **Ghost branch detection needs a systematic approach.** When an epic orchestrator pre-creates branches and stories, some may already be implemented on main (e.g., from a previous session that merged directly). Currently this is discovered manually by checking `git log`. An automated check — "does the branch have unique commits not on main?" — would save diagnosis time and prevent incomplete story artifacts.

3. **`useMediaQuery` over `useMemo` + `matchMedia` for reactive queries.** This pattern, caught by the E17-S05 code review, is the correct approach whenever a CSS media query result affects component behavior (not just initial render). The `useMemo` approach captures the value once; `useMediaQuery` subscribes to changes. This matters for reduced motion, color scheme, and breakpoint detection.

4. **Component-level test coverage scales with display complexity.** S01 and S02 (simple card displays) needed no component tests — E2E covered them. S03 (difficulty list with badges and suggestions), S04 (discrimination analysis with interpretation text), and S05 (recharts chart with responsive behavior) all warranted component tests because their rendering logic had meaningful branches. The rule of thumb: if the component has more than 2 conditional rendering paths, add component tests.

5. **Three technical debt items carried for 3 consecutive epics signals they need reclassification.** Items 4-6 from the E15/E16 retros (no-silent-catch investigation, shared formatCorrectAnswer extraction, h-5/w-5 cleanup) have been carried through E15, E16, and now E17 without action. Per the E16 team agreement, we should only commit to items with enforcement mechanisms. These three are voluntary and have proven they won't get done voluntarily. Accept them as permanent debt or create dedicated stories.
