## Challenges and Growth Areas

1. **Dev agent marking tasks complete without full implementation.** Story 1.2 had 3 critical review findings where tasks were marked `[x]` but hover effects, clickability, and keyboard focus were not actually implemented. Dev Agent Record contained false claims. Root cause: no self-verification step between "task done" and "story done."

2. **Store selector anti-patterns repeated.** `getAllTags()` called inside each `ImportedCourseCard` (Stories 1.3 and 1.4), causing N re-renders on any store change. The pattern was fixed each time, but the same mistake was repeated across stories because the lesson wasn't captured in documentation.

3. **High review feedback volume.** ~40+ issues across 4 stories. While reviews caught real problems, many issues were basic completion failures (missing hover states, missing ARIA labels) that the dev agent should have caught before review.

4. **Test coverage gaps consistently flagged.** Code reviews in Stories 1.1, 1.3, and 1.4 all flagged missing unit tests for new store actions, rollback/error paths, and component states. Tests for edge cases were consistently underwritten.

5. **Systemic theme token debt.** `bg-blue-600` hardcoded across the codebase instead of using `--primary` CSS variable. Flagged as a blocker in Story 1.3 design review. Pre-existing condition but growing with each new story.

## Key Insights and Learnings

1. **Lift computed values out of list items.** Store getters called inside mapped components cause N re-renders. Compute once in the parent, pass as prop. Watch for this pattern whenever a store getter appears inside a `.map()`.

2. **Pre-review self-verification catches basic issues.** A lightweight "verify implementation" step (visual check + keyboard tab + task-vs-code comparison) would let reviewers focus on architecture, not completion verification.

3. **Architectural patterns must be documented to prevent rediscovery.** Without explicit documentation in `project-context.md`, each story risks re-learning the same lessons (selector patterns, naming conventions, migration patterns).

4. **Dexie.js `upgrade()` is the right migration pattern.** Adding default values to existing records via the version upgrade callback ensures old data works with new features without a separate migration script.

5. **Workflow skills must be idempotent.** `/start-story` and `/review-story` failed on re-run after interruption. Making all steps check-before-act (branch exists? skip; file exists? skip) saves time when resuming.

6. **Name collisions surface early in brownfield projects.** Using prefixed types (`ImportedCourse`, `LearnerCourseStatus`) prevents confusing import errors. Check for name conflicts before defining new types.

7. **Tag normalization belongs at the store boundary.** Trim + lowercase at the point of storage, not in the UI component, prevents "React" and "react" from becoming separate tags.

8. **Combined filter tests need careful fixture design.** Test data must isolate each filter dimension to prove AND-semantics, not just happen to exclude by one filter.
