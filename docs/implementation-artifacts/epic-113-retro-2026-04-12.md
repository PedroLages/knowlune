## What Was Challenging

**Bob (Scrum Master):** "Three race conditions flagged HIGH in R1. Were they all the same class of bug, or distinct?"

**Pedro (Project Lead):** "Same root cause, three manifestations. All three store actions — `setRating`, `setReviewText`, `deleteReview` — captured `get().reviews` before calling `set()`. If two actions dispatched in rapid succession, the second action's captured state was already stale: it saw the pre-first-action reviews array, so the first action's result was overwritten. The fix is identical in all three: replace `const reviews = get().reviews; set({ reviews: updated })` with `set(state => { const reviews = state.reviews; ... return { reviews: updated }; })`. The callback receives the current state at the time `set` executes, not at capture time."

**Charlie (Senior Dev):** "The root cause isn't the pattern — it's that the pattern looks correct. Capturing state before a mutation is intuitive in non-reactive contexts. In Zustand, `get()` returns a snapshot; if the store updates between `get()` and `set()`, that snapshot is stale. The rule is: in any store action that reads-then-writes, always use the callback form so the read and write are atomic from Zustand's perspective."

**Quinn (QA Engineer):** "The `isLoaded: true` catch-block fix is a small but important correctness issue. If `loadReviews()` throws, the store stayed in `isLoaded: false`. Any hook subscribed to `isLoaded` would keep retrying the load indefinitely — or more precisely, would never consider the load complete. Setting `isLoaded: true` in the catch block means 'load attempted, failed, do not retry.' The component handles the error state separately. Without this fix, a one-time network or Dexie error would lock the reviews feature for the session."

**Alice (Product Owner):** "The markdown preview uses `innerHTML` with a direct conversion. The implementation notes call this out as 'intentional safety: user-owned local data, HTML entities escaped.' That's the right framing — this is a personal review field, not user-generated content from a remote source. Flagging it and deciding it's acceptable is better than silently doing it."

## Lessons Learned

### 1. Zustand read-then-write must use the callback form at every call site in the action

`const reviews = get().reviews; set({ reviews: modified })` is a race condition when two store actions can run concurrently. The callback form `set(state => ({ reviews: modified(state.reviews) }))` is atomic: the read and write happen at the same time from Zustand's perspective. This applies to every `get()` call inside a `set()` action — not just the first one. When a task specifies this pattern, the pre-review checklist must verify it appears at all applicable sites, not just that the pattern is used somewhere. (E113-S01, `setRating` / `setReviewText` / `deleteReview`)

### 2. Error paths in state machines must complete the state transition

`loadReviews()` failing left `isLoaded: false` — a permanent "loading" state for the session. Any state machine with a loading/error/success model must ensure error paths set the terminal state (`isLoaded: true`, `status: 'error'`, etc.). An unterminated error path means the UI can never exit the loading state. Retry logic is a separate concern and should be explicit, not implied by leaving the state incomplete. (E113-S01, `loadReviews` catch block)

### 3. False positives are identified by tracing the code path, not by deferring to the reviewer

When a review tool flags a potential issue (BLOCKER: silent data loss), the correct response is to trace the actual execution path: under what conditions does this code execute? What data is guaranteed to be present? What type guarantees apply? If the analysis shows the flag is a false positive, document it explicitly as a non-issue with the reasoning, and do not open a fix round for it. Deferring to the reviewer without analysis wastes a round. Dismissing without analysis risks missing real bugs. (E113-S01, GLM BLOCKER on `handleSaveText`)

### 4. Local state as immediate feedback layer, store as source of truth

The keyboard feedback MEDIUM finding — arrow key presses updating the visual display only after the store confirmed — was correct. `keyboardValue` local state solves this: the display uses `keyboardValue ?? storeValue`, so the UI responds instantly to keyboard input while the store update propagates asynchronously. This pattern applies to any interactive control where user input should feel immediate: maintain a local state layer for visual feedback, synchronize to the store after. The local state is temporary; the store is authoritative. (E113-S01, `StarRating` keyboard navigation)

### 5. Component responsibility boundaries make review findings local

When `StarRating` handles only display and input, and `BookReviewEditor` handles only editing state, and the store handles only persistence — each review finding is local to one of those three units. There are no findings that require simultaneous changes to multiple components. Narrow responsibility boundaries are a review-cost optimization: they convert "this finding requires coordinated changes across three files" into "this finding requires one change in one file." (E113-S01, all five R1 findings)
