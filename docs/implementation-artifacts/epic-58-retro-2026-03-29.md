## Challenges and Growth Areas

### 1. E2E Seeding Against TTL-Protected Tables
The E2E tests required three fix passes: (a) import wizard blocking setup, (b) TTL cleanup deleting seeded notifications with old timestamps, (c) ambiguous selectors. The TTL issue is specific to notifications (30-day expiry) but applies to any table with cleanup rules.

**Root cause:** No established pattern for seeding data in tables with TTL cleanup. Tests initially used arbitrary timestamps that fell outside the 30-day window.

**Lesson:** When seeding IndexedDB tables with TTL or cap cleanup, always use browser-relative timestamps (e.g., `new Date(Date.now() - 1000 * 60 * 60)` for 1 hour ago) rather than hardcoded past dates. Document this in test-patterns.md.

### 2. Story Notes Said "Reuse" When They Meant "Extract"
The story implementation notes said "Reuse icon + color mapping from NotificationCenter.tsx" and "Reuse relativeTime() helper." This was interpreted as "copy the code" rather than "extract to shared module." The duplication was caught by code review, but the initial implementation could have been correct if the story had said "Extract to `src/lib/notifications.ts` and import from both components."

**Root cause:** Ambiguous language in implementation notes. "Reuse" can mean copy or share; "extract" is unambiguous.

**Lesson:** In story implementation notes, use "extract to [path]" instead of "reuse from [source]" when the intent is to create shared code.

### 3. Design Review Skipped
Design review was skipped because Playwright MCP browser was not available. This means the responsive layout (AC7c) was not visually verified, and it's the largest gap in testarch-trace (MEDIUM).

**Root cause:** Playwright MCP browser availability is not guaranteed in all sessions.

**Lesson:** Accept as known limitation. Responsive gaps are covered when design review runs; when it doesn't, testarch-trace flags the gap explicitly.

## Key Insights and Patterns

### Pattern 1: Review Findings as Epic Seeds
The pipeline from adversarial review finding (C-03) to epic (E58) to implementation (S01) to merge (#148) validates a systematic approach to UX debt. Review findings should be triaged as potential story/epic candidates, not just defect fixes.

### Pattern 2: TTL-Aware Test Seeding
Any IndexedDB table with cleanup rules (TTL, cap, age-based pruning) requires test data seeded with browser-relative timestamps. The pattern: `new Date(Date.now() - offsetMs).toISOString()` where offsetMs is well within the TTL window.

### Pattern 3: Shared Helper Extraction as a Review Response
When code review finds duplication between a new component and an existing one, the correct response is extraction to a shared module (e.g., `src/lib/notifications.ts`), not inlining the duplicate or adding a "TODO: deduplicate" comment.

## Key Takeaways

1. **Adversarial review findings are a reliable source of targeted epics.** The C-03 finding became a concrete story, was implemented, reviewed, and merged in a single session. This validates the review-to-epic pipeline.

2. **TTL-aware test seeding is a new pattern to document.** Tables with cleanup rules (TTL, cap) require browser-relative timestamps in test data. Without this, tests fail non-deterministically as cleanup deletes seeded data.
