## Key Insights and Learnings

1. **Voluntary retro commitments are dead** — Three cycles of evidence: 56% → 14% → 0%. The only reliable mechanism is automation or immediate resolution. Future retros must either (a) automate the fix, (b) resolve it in-session, or (c) formally accept the risk and close the item.

2. **Inside-out rename is the refactoring safety pattern** — Start from types, let TypeScript errors cascade outward through data → lib → pages → components → routes → DB → tests. Every missed reference surfaces as a compile error. This should be documented in engineering-patterns.md.

3. **Smooth epics amplify the success paradox** — E23 was a clean, fast epic. As predicted by E16's insight, documentation discipline relaxed when things went well. S04 and S05 shipped fast with placeholder docs. The antidote must be structural (gate enforcement), not aspirational (team agreements).

4. **Consolidation requires full sweep with behavioral parity check** — When extracting utilities to shared locations, grep ALL consumers and verify behavioral parity. S06's `getInitials` consolidation missed `AuthorProfile.tsx` which had different behavior (no null guard). Partial consolidation is worse than duplication — it creates false confidence.

5. **Re-export removal is migration hygiene** — After migrating a utility from module A to module B, remove the re-export from A. Leaving it creates a ghost dependency that future consumers can import without error, defeating the purpose of consolidation.
