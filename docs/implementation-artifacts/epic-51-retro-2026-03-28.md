## Successes and Strengths

### 1. CSS-First Architecture Decisions
All three display features (motion, font, density) use CSS mechanisms (class toggles, custom properties, `@font-face` injection) rather than JavaScript-driven re-renders. This means settings apply instantly, persist across page loads via synchronous init scripts, and have zero runtime cost after initial application. This is a mature architectural pattern worth codifying.

### 2. Exceptional Test Coverage
100% acceptance criteria coverage (23/23 ACs) with 47 automated tests. This is one of the best-covered epics in the project. Every feature has both unit tests (hook behavior, utility functions) and E2E tests (user-facing flows). The traceability report confirms zero gaps.

### 3. Consistent Component Patterns
Following the established AgeRangeSection Card pattern made the new DisplayAccessibilitySection consistent with existing Settings UI. The icon badge, header background, content padding, and AlertDialog patterns are reusable for any future Settings section.

### 4. Hook Pattern Reuse
The `useReducedMotion`, `useAccessibilityFont`, and `useContentDensity` hooks all follow the established `useColorScheme.ts` pattern -- reading from `getSettings()`, listening to `settingsUpdated` and `storage` events, toggling classes on `documentElement`. This consistency makes the codebase predictable and reduces onboarding friction.

### 5. Defensive localStorage Handling
Zod validation for persisted enum values (`reduceMotion`, `contentDensity`) prevents crashes from corrupted or manually-edited localStorage. This defensive pattern is worth replicating for all enum-like persisted settings.

## Challenges and Growth Areas

### 1. Non-Reactive `shouldReduceMotion()` in 12 Components (Adversarial Finding)
The adversarial review identified that 12 components use the exported `shouldReduceMotion()` utility function (non-reactive) instead of the `useReducedMotion()` hook. These components read the value once and won't update when the setting changes at runtime. This is a correctness gap that should be addressed in a follow-up.

**Root cause:** The utility function was exported alongside the hook for convenience, but its non-reactive nature isn't obvious to consumers.

**Lesson:** When creating a reactive hook alongside a one-shot utility, either (a) don't export the utility, or (b) name it clearly (e.g., `getReducedMotionSnapshot()`) to signal its non-reactive nature.

### 2. S02 Required 2 Review Rounds (9 Issues Fixed)
Story S02 (Reduced Motion) was the most complex story -- removing 17 local MotionConfig overrides, updating 5 confetti components, adding flash prevention, and consolidating the existing "animations" toggle. The 9 review issues included:
- Missing E2E spec file (HIGH)
- Missing unit tests for hook (HIGH)
- Non-reactive function usage (HIGH)
- Flash prevention gap for `system` preference (MEDIUM)
- AlertDialogAction using raw `bg-brand` instead of `variant="brand"` (MEDIUM)

**Lesson:** Stories with broad codebase-wide refactoring (touching 17+ files) need extra attention to test coverage and pattern consistency. Consider splitting codebase-wide mechanical changes into a dedicated sub-task with its own test pass.

### 3. Flash Prevention Incomplete for Font and Density
The adversarial review noted that while motion has a flash prevention script in `index.html`, the font and density features lack equivalent scripts. Users with font or spacious mode enabled will see a brief flash of default styling on page load before React hydrates.

**Root cause:** The flash prevention pattern was established for motion but not systematically applied to all settings features.

**Lesson:** When adding settings that affect visual appearance, always consider flash prevention via synchronous `<head>` scripts. Create a checklist: "Does this setting change visual appearance on load? If yes, add init script."

### 4. Inconsistent `content-padding` Coverage
The adversarial review found that while `--content-gap` was applied to main page grids, `--content-padding` (content card padding) coverage was inconsistent across the app. Some card components were updated; others were not. This means spacious mode has partial effect on some pages.

**Root cause:** The task specified "10-15 component touch points" which was interpreted as highest-impact areas, leaving lower-priority components untouched.

**Lesson:** For cross-cutting CSS token retrofits, either (a) commit to comprehensive coverage with an auditable checklist, or (b) explicitly document which areas are deferred with a follow-up ticket.

### 5. Competing Motion Controls (UX Debt)
The existing "animations" toggle in Engagement Preferences and the new 3-state Motion RadioGroup in Display & Accessibility create potential user confusion. While S02 wired the old toggle to delegate to the new setting, having two controls for the same behavior is UX debt.

## Key Insights and Patterns

### Pattern 1: CSS Variable + Class Toggle Architecture
The pattern of defining CSS custom properties in `theme.css`, toggling a class on `<html>`, and having components reference `var(--token-name)` via Tailwind arbitrary values is highly effective. It provides:
- Instant visual updates without React re-renders
- Easy persistence (just toggle a class)
- Clean separation of concerns (CSS handles visuals, JS handles state)
- Zero runtime cost after initial application

This pattern should be the default for any visual preference setting.

### Pattern 2: Synchronous Init Scripts for Flash Prevention
The `reduce-motion-init.js` pattern of reading localStorage synchronously in `<head>` before stylesheets load is critical for any setting that affects initial visual state. Extend this pattern to font and density in a follow-up.

### Pattern 3: Settings Hook Lifecycle
The established hook lifecycle for settings-derived features is:
1. Read from `getSettings()` on mount
2. Listen for `settingsUpdated` custom events (same-tab)
3. Listen for `storage` events (cross-tab sync)
4. Apply effect (class toggle, CSS variable, etc.)
5. Clean up listeners on unmount

This is now validated across 4 hooks (`useColorScheme`, `useReducedMotion`, `useAccessibilityFont`, `useContentDensity`).

### Pattern 4: Lazy Font Loading via @fontsource
Dynamic `import('@fontsource/...')` with Vite code-splitting is the correct approach for optional fonts. The CSS injects `@font-face` rules, the actual font files download only when referenced by computed styles, and subsequent toggles just swap the CSS variable (no re-download). Zero cost for non-users.

### Pattern 5: `toEqual` Brittleness for Growing Objects
Extending `AppSettings` with 3 new required fields broke 3 unit tests using `toEqual` with exact object shapes. Prefer `toHaveProperty` or field-level assertions for objects that are likely to grow over time.
