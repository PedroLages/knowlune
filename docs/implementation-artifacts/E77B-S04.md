# Story E77B-S04: Drive Source Management UI and Sync Validation

## Story

As a user with Drive-imported courses,
I want to see which courses are Drive-sourced, reconnect folders when files change, and manage Drive configuration in Settings,
so that I have full visibility and control over my Drive integration.

## Acceptance Criteria

1. **Given** a Drive-imported course on the course card **When** displayed **Then** it shows a "Drive" badge with HardDrive icon next to file metadata.
2. **Given** a Drive-imported course detail page **When** opened **Then** it shows a Google Drive source banner with a "Reconnect" button.
3. **Given** the Reconnect button is clicked **When** a new Drive folder is selected **Then** `handleReconnectFolder` maps new file IDs to existing lessons by filename and updates `sourceDriveId` on the course.
4. **Given** a Drive folder reconnect **When** files are matched **Then** a toast shows the match count (e.g., "X of Y files matched").
5. **Given** the Settings page Integrations & Data section **When** opened **Then** it shows a Google Drive configuration card with connected account email, read scope status, and disconnect action.
6. **Given** the Drive configuration card **When** the read scope is not granted **Then** a "Grant Access" button is available to trigger `requestDriveReadScope()`.
7. **Given** the Drive configuration card **When** "Disconnect Google Drive" is clicked **Then** a confirmation dialog appears and on confirm signs out via `signOut()`.
8. **Given** the course import store's `updateCourseDetails` **When** called with `sourceDriveId` **Then** it updates the field on the database record.

## Challenges and Lessons Learned

- **Drive reconnect file matching**: The reconnect flow maps new Drive file IDs to existing lessons by comparing filenames case-insensitively (`f.name.toLowerCase() === video.filename.toLowerCase()`). This is a pragmatic approach that handles common rename/move scenarios without requiring a per-file sync protocol.
- **Scope authorization pattern**: The `requestDriveReadScope()` function triggers a browser redirect to Google OAuth, so the UI state after the redirect is handled separately (no `await` on the redirect). The `handleGrantReadScope` in DriveConfigurationSettings is fire-and-forget by design â€” the grant redirect takes the user out of the app.
- **Disconnect via sign-out**: Currently, disconnecting Drive from Settings signs out the entire session (`signOut()`) because the Supabase Google OAuth integration doesn't support per-scope token revocation. This is a known limitation documented in the disconnect dialog description.
- **Store extensibility pattern**: Adding `sourceDriveId` to `CourseDetailsUpdate` interface and the `updateCourseDetails` action in `useCourseImportStore` follows the existing optional-fields pattern (all `undefined`-on-omission via `?.`), keeping backward compatibility with existing callers that don't provide Drive fields.
- **No story-specific E2E spec**: This story does not include a dedicated E2E test spec â€” the UI changes (badge, banner, settings card) are covered by the unit test (`CourseCard.driveSource.test.tsx`) and would benefit from a dedicated E2E spec in a follow-up.
