## Successes and Strengths

### 1. Pattern Reuse: Stateless Service Functions

Charlie (Senior Dev): "The decision to model `quizGenerationService` on `courseTagger.ts` — stateless async functions, no class instances — was the right call. It made testing trivial: mock `callOllamaChat`, pass transcript text, assert return shape. No setup, no teardown."

**Pattern codified:** All AI pipeline orchestrators should be stateless modules, not class instances. Side effects (Dexie writes) happen at the end of the pipeline, never mid-stream.

### 2. Zero-Dependency Architecture

Dana (QA Engineer): "Four stories. Zero new npm packages. The Web Worker reuse for embeddings was particularly elegant — `all-MiniLM-L6-v2` was already loaded for note embeddings. Course embeddings just piggybacked on the existing coordinator."

Alice (Product Owner): "That's a genuine win for bundle size and long-term maintenance."

**Pattern codified:** Before adding a new npm dependency, audit whether the capability already exists in a loaded-but-narrow-use module (e.g., workers, vector stores, Zod schemas).

### 3. Never-Throw Service Boundary

Charlie (Senior Dev): "The `generateQuizForLesson()` never-throw contract is excellent. Every failure path returns `{ quiz: null, error: string }`. The hook layer handles the toast. The service layer is purely functional. No try/catch at the call site."

**Pattern codified:** Service functions that interact with external resources (LLM, Dexie) should return result objects, not throw. Reserve exceptions for programmer errors (wrong argument types), never for runtime failures (network timeout, empty response).

### 4. QC Pipeline: Deterministic Not Generative

Elena (Junior Dev): "I was surprised that the quality control step doesn't call the LLM at all. It's pure functions — cosine similarity, string matching, set membership. That's much faster to test and much cheaper to run."

Bob (Scrum Master): "That's a design principle worth highlighting. Not every AI feature needs more AI. The validation layer should be deterministic."

**Pattern codified:** Separate generation (LLM-dependent) from validation (deterministic). The QC pipeline is independently testable, independently deployable, and zero cost to run.

### 5. Non-Blocking Embedding Pattern

Charlie (Senior Dev): "The `deferInit(() => import(...).then(...).catch(...))` startup pattern for course embeddings is clean. Fire-and-forget, logged errors, import succeeds regardless. Pedro, that startup wiring pattern is now documented."

Pedro (Project Lead): "Good — I want that pattern used for any background startup task going forward. No lifecycle hooks, always `main.tsx`."

## Challenges and Growth Areas

### 1. Mock Data Semantic Consistency (QC Testing Gotcha)

Dana (QA Engineer): "S03 surfaced a real lesson: adding transcript grounding validation broke a pre-existing test because the test used `word0 word1 word2` as transcript content but the mock LLM response referenced domain-specific terms like `React` and `Virtual DOM`. The test passed because it didn't validate grounding — but once we did, the mismatch was exposed."

Bob (Scrum Master): "The lesson here is architectural: when you add a new validation step to an existing pipeline, you must audit all existing mocks for semantic consistency."

**Lesson:** When adding validation to a generation pipeline, audit all existing mocks. Transcript text and generated questions must share key terms if the pipeline validates grounding. Generic `wordN` fixtures break under semantic validation.

**Action item:** Add to engineering patterns: "Mock data for AI pipelines must be semantically consistent — transcript text and generated questions must share domain vocabulary."

### 2. Optimistic State vs. Confirmed Persistence

Charlie (Senior Dev): "The feedback thumbs UI had an anti-pattern flag in review: setting optimistic UI state before awaiting the Dexie write. The engineering checklist explicitly prohibits this for non-idempotent operations."

Pedro (Project Lead): "The rule is: await first, update UI on success, toast on failure. No reverting — toast IS the error feedback. That pattern is already in engineering-patterns.md?"

Charlie (Senior Dev): "It is now."

**Lesson:** For feedback/reaction UX, always await the DB write before updating optimistic UI state. The correction is: `await db.write()`, then `setState(newValue)`. On error: `toast.error(message)`, leave state unchanged.

### 3. Dexie Schema Synchronization (Checkpoint + Version Assertions)

Elena (Junior Dev): "S01 had a specific gotcha that tripped us: adding a new table and a new index on an existing table in the same migration means BOTH must appear in the checkpoint schema. Miss either one and the checkpoint consistency test fails."

Bob (Scrum Master): "And the schema version assertion in `schema.test.ts` must always be updated — `toBe(34)` → `toBe(35)`. That's a three-file touch for every migration: `schema.ts`, `checkpoint.ts`, and `schema.test.ts`."

**Lesson:** Every Dexie migration is a 3-file update: schema.ts (migration definition), checkpoint.ts (CHECKPOINT_SCHEMA + CHECKPOINT_VERSION), schema.test.ts (version assertion). Missing any one causes test failures that look unrelated to the change.

**Action item:** Add to CLAUDE.md or engineering-patterns.md: "Dexie migration checklist: (1) add to schema.ts, (2) update checkpoint.ts schema + version, (3) update schema.test.ts version assertion."

### 4. Silent Catch Discipline

Dana (QA Engineer): "The outer `.catch(() => {})` without a log message in `courseEmbeddingService` was flagged in review. The inner function already logs — but if the inner function itself throws unexpectedly, the outer catch swallows it silently."

Charlie (Senior Dev): "The rule is: always include `console.error(err)` in outer catches, even when you believe the inner function already handles it. Defense in depth."

**Lesson:** Outer `.catch` blocks must always include at minimum `console.error(err)` even when the inner function logs. This captures unexpected re-throws and avoids completely silent failures.

### 5. Store Embedding Refresh: Stale Optimistic State

Charlie (Senior Dev): "S04 had a subtle bug: the embedding service was called with pre-update course metadata because the Zustand store optimistic update hadn't propagated. Fix: fetch the updated course from store state _after_ the `set()` call."

**Lesson:** When triggering background work after a store update, read the _post-set_ store state, not the closure-captured pre-update value. Optimistic store writes are synchronous but the closure retains the old reference.

## Ollama Integration Patterns (Pedro's Reflection Focus)

Pedro (Project Lead): "Let me reflect on the Ollama integration specifically. What patterns did we establish that we should carry forward?"

Charlie (Senior Dev): "Three main ones:

1. **Proxy routing**: All Ollama calls go through `/api/ai/ollama/chat` — the Express proxy. Never direct `localhost:11434` calls from the frontend. This makes the origin configurable and testable.

2. **Health-gate UI**: The `GenerateQuizButton` is the reference implementation for 'show when available, disable with tooltip when offline'. That pattern should be applied to any future Ollama-backed feature button.

3. **aiConfiguration consent gate**: Every LLM call starts with `if (!aiConfiguration.consent) return early`. This is non-negotiable — documented in the service layer, not the hook layer."

Pedro (Project Lead): "Good. And the structured output pattern?"

Charlie (Senior Dev): "`callOllamaChat` with `format: QUIZ_RESPONSE_SCHEMA` + `QuizResponseSchema.safeParse()`. The schema enforces the shape; safeParse catches malformed output; retry handles transient failures. That's the complete contract for any structured-output Ollama feature."

## QC Pipeline Design Patterns (Pedro's Reflection Focus)

Dana (QA Engineer): "The QC pipeline design was elegant. Three checks, all deterministic:
- Cosine similarity for dedup (reuses existing `BruteForceVectorStore`)
- Set membership for answer uniqueness
- Term intersection for grounding

No LLM, no network, pure functions. Highly testable."

Pedro (Project Lead): "The `retryNeeded` flag design — where the service layer decides to retry based on QC output rather than QC deciding to retry — was the right call. Keeps QC pure and stateless."

Charlie (Senior Dev): "Exactly. QC reports, service decides. Single responsibility."

**Pattern codified:** QC pipelines should be pure reporting functions (`validQuestions, rejectedQuestions, retryNeeded`). The retry decision and loop belong in the orchestrating service, not the QC module.

## Course Embedding Service Patterns (Pedro's Reflection Focus)

Charlie (Senior Dev): "The embedding service established three reusable patterns:

1. **Non-blocking fire-and-forget**: `generateCourseEmbedding()` never throws, never returns a value the caller waits on. This is the contract for any background enrichment task.

2. **Content-hash invalidation**: SHA-256 of `title+description+tags` as a `sourceHash` — regenerate only when the hash changes. This is cheaper than comparing the full embedding and more reliable than timestamps.

3. **Backfill scan**: `generateMissingEmbeddings()` scans all courses without embeddings and backfills sequentially. This handles historical data and recovery from partial failures."

Pedro (Project Lead): "That backfill pattern should be the standard for any new embedding or enrichment pipeline we add."

## Web Worker Reuse Patterns (Pedro's Reflection Focus)

Elena (Junior Dev): "I was worried we'd need a new worker for quiz QC since it uses cosine similarity. But the existing `BruteForceVectorStore.cosineSimilarity()` is a pure synchronous function — no worker needed. Workers are only needed for the actual `generateEmbeddings` computation."

Charlie (Senior Dev): "Right. The coordinator pattern in `workers/coordinator.ts` handles the actual Worker lifecycle — pooling, message passing, transferables. Consumers just call `generateEmbeddings(text)`. The Worker complexity is fully encapsulated."

Pedro (Project Lead): "So the rule is: import from the coordinator, never instantiate a Worker directly. New embedding use cases just call `generateEmbeddings()`."

**Pattern codified:** Never instantiate `embedding.worker.ts` directly. Always use `generateEmbeddings` from `workers/coordinator.ts`. The coordinator handles pooling and lifecycle. New embedding consumers add zero overhead.

## Key Patterns Extracted

### Pattern 1: AI Service Stateless Functions
Stateless async functions (no class instances) + result objects (no throws) + Dexie writes at end of pipeline. Reference: `quizGenerationService.ts`.

### Pattern 2: Structured Ollama Output Contract
`callOllamaChat({ format: SCHEMA }) → safeParse(json) → retry up to 2x → result object`. Reference: `quizGenerationService.ts` + `quizPrompts.ts`.

### Pattern 3: Deterministic QC Pipeline
Pure reporting functions (no LLM, no side effects): `(questions, chunk) → { validQuestions, rejectedQuestions, retryNeeded }`. Retry decision in orchestrator, not in QC. Reference: `quizQualityControl.ts`.

### Pattern 4: Non-Blocking Background Enrichment
`deferInit(() => import('./service').then(s => s.backfill()).catch(console.error))` in `main.tsx`. Never in component lifecycle. Reference: `courseEmbeddingService.ts`.

### Pattern 5: Content-Hash Invalidation
SHA-256 of concatenated metadata fields as `sourceHash`. Regenerate only on hash change. Reference: `courseEmbeddingService.ts`.

### Pattern 6: Web Worker Encapsulation
Always call `generateEmbeddings()` from coordinator, never `new Worker(...)`. Reference: `workers/coordinator.ts`.

### Pattern 7: Dexie Migration Checklist (3-File Rule)
Every migration: (1) `schema.ts`, (2) `checkpoint.ts` (schema + version), (3) `schema.test.ts` (version assertion).

### Pattern 8: Semantic Mock Consistency
AI pipeline mocks must be semantically consistent — transcript text and generated questions must share domain vocabulary when grounding validation is active.

**Key Takeaways:**

1. **Stateless AI services with result objects** (never throw, never hold state) are the dominant pattern for LLM-backed features — they're testable, composable, and safe by default.
2. **Zero-dependency engineering** is achievable for AI features when the infrastructure (Web Workers, Dexie, Zod, vector store) is already in place. Audit existing capabilities before adding packages.
3. **Deterministic QC pipelines** (pure functions, no additional LLM calls) should always separate generation from validation. This keeps the QC layer fast, cheap, and independently testable.
4. **Mock semantic consistency** is a new discipline for AI-backed tests: mocks must reflect real-world data shape, not just structure. `word0..wordN` breaks grounding validation.
5. **Dexie migrations are a 3-file operation** — missing `checkpoint.ts` or the test assertion causes confusing failures that look unrelated to the migration.
