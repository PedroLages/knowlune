## Successes and Strengths

### Bob (Scrum Master)

**Bob:** Four stories in four days with only one requiring a second review round. Pedro, what drove that velocity?

### Pedro (Project Lead)

**Pedro:** Three things stood out to me about this epic.

**1. The scope reduction from 7 to 4 stories was the single best planning decision of the epic.** The original 7-story plan included prerequisite enforcement, learning path recommendations, and cross-path analytics. Cutting those let us focus on four self-contained features that each deliver standalone value. Career Paths, Flashcards, Heatmap, and Radar Chart are all useful independently -- none of them depend on each other at runtime. That isolation meant we could ship stories in any order, and a bug in one never blocked another. The lesson: when scoping an epic around a theme ("knowledge retention"), pick features that are thematically related but architecturally independent. If two features share a data model or a store, they should probably be in the same story rather than separate stories with hidden coupling.

**2. E20-S03's two-round review process actually worked as designed.** Round 1 found 13 issues (4 HIGH, 5 MEDIUM, 4 NITS). All 13 were fixed in a single commit, and Round 2 came back clean with only 2 cosmetic nits. The fact that Round 2 found zero regressions from the fixes means the developer understood the issues correctly and fixed them precisely. The 4 HIGH items were all genuine performance concerns -- memoizing 365-cell date formatting, debouncing event-driven refresh, scoping Dexie queries with `.where()`, and explicit sort for Map-based monthly summaries. These are the kind of findings that ESLint cannot catch because they require understanding the data flow across render cycles. The review agent earned its keep on this one.

**3. The SM-2 algorithm in E20-S02 was well-isolated, which made the review tractable.** `spacedRepetition.ts` is a pure-functional module with no side effects. The 23 unit tests cover boundary conditions, rating transitions, and interval calculations without any mocking. When the code review found that `FIXED_NOW = new Date()` at module load was stale, the fix was trivial because the algorithm itself was pure -- only the caller had the time dependency. Separating algorithm from state management is a pattern worth reinforcing.

### Charlie (Senior Dev)

**Charlie:** The roving tabindex implementation in the heatmap (E20-S03 Round 2) is one of the best accessibility improvements we have shipped. A 365-cell grid with individual `tabIndex={0}` creates an unusable keyboard experience. The fix -- arrow key navigation with a single tab stop on the focused cell -- is the correct ARIA grid pattern and was a nit that became a meaningful improvement. This pattern should be documented for any future dense-grid components.

### Dana (QA Engineer)

**Dana:** The E20-S03 test quality findings were the strongest evidence that the testing review agent adds value. The `if ((await cell.count()) > 0)` guard in the tooltip test was a silent-pass bug -- the test could never fail, which is worse than no test at all. Replacing it with `await expect(cell).toHaveCount(1)` turns it into a genuine regression detector. I also want to note that E20-S02 deliberately skipped a dedicated E2E spec, relying on 47 unit tests plus the navigation smoke test. For a store-heavy feature with minimal page interaction, that was the right trade-off. Not every story needs E2E coverage; some are better tested at the unit level.

## Challenges and Growth Areas

### Bob (Scrum Master)

**Bob:** What friction did you encounter?

### Pedro (Project Lead)

**Pedro:** Three friction points, ranked by time cost.

**1. The E20-S03 merge conflict with main was the most time-consuming non-development task.** After E20-S03's two review rounds were complete on the feature branch, merging to main produced conflicts in 4+ files because E20-S01 and E20-S04 had already been merged. The conflicts were in Layout.tsx (sidebar navigation entries), routes.tsx (route definitions), and sprint-status.yaml. This is the same sprint-status.yaml merge-conflict pattern we have flagged since Epic 21. The underlying cause is that every feature branch touches sprint-status.yaml to update story status, and every parallel branch diverges on it. For a 4-story epic where 3 stories merged before the 4th, this is structural. We need to either stop updating sprint-status.yaml in feature branches (deferred to main-branch-only updates) or accept the conflict tax.

**2. The E20-S02 BLOCKER -- `noteId` never passed to NoteEditor callsites -- was a missing implementation step, not a design flaw.** Task 4.4 in the story explicitly said "pass noteId from LessonPlayer/NoteCard to NoteEditor." All four NoteEditor callsites in LessonPlayer lacked the prop, which meant flashcards created from notes had `noteId: undefined`, breaking source-note traceability entirely. The code review caught it at confidence 95. This is exactly the kind of issue that an AC re-read before review would surface: AC1 says "select text and click Create Flashcard," which implies the note context (including noteId) flows through. The E27 retro already identified this pattern -- implementation-time AC drift -- and proposed adding an AC compliance check to the pre-review checklist. We still have not added it.

**3. The pre-existing non-deterministic Date in story-e11-s01.spec.ts keeps surfacing.** This was flagged in E27's review too. Every code review that runs E2E tests discovers this flaky test, spends time investigating, and confirms it is pre-existing. It is registered nowhere (known-issues.yaml does not list it). The investigation time is pure waste. We need to either fix it (likely a one-line `FIXED_DATE` replacement) or register it in known-issues.yaml so future reviews skip the investigation.

### Charlie (Senior Dev)

**Charlie:** The E20-S01 code review found a TOCTOU race in the `loadPaths` seed logic -- `count()` then `bulkAdd()` without a transaction. In StrictMode, two concurrent calls can both see `count === 0` and both attempt `bulkAdd()`, causing a `ConstraintError`. The fix was switching to `bulkPut()` (idempotent). This is the same class of bug we have seen in at least three other stories (E25-S02, E22-S04). We should add an ESLint rule or code review checklist item: "if you check-then-write to Dexie, wrap in a transaction or use an idempotent write."

### Dana (QA Engineer)

**Dana:** E20-S04 had an interesting E2E test stability challenge. The radar chart only renders when 2+ course categories exist, and the component relies on `requestIdleCallback` timing for data loading. The initial E2E test seeded courses before page load, but the reload timing meant `requestIdleCallback` could fire before IndexedDB writes completed. The fix was explicit IndexedDB seeding in `addInitScript` with a page reload after seeding. This is a pattern we have refined across several stories, and it keeps getting more reliable, but it is not yet documented in `engineering-patterns.md`.

## Key Insights and Learnings

1. **Architecturally independent stories within a themed epic maximize parallelism and minimize merge risk.** E20's four stories share a theme (learning pathways and knowledge retention) but share no runtime dependencies. Career Paths, Flashcards, Heatmap, and Radar Chart operate on different data models, different stores, and different pages. This meant any story could merge in any order without blocking others. The merge conflict that did occur (E20-S03) was purely in shared infrastructure files (Layout.tsx, routes.tsx, sprint-status.yaml), not in feature code.

2. **Two-round reviews are expensive but justified for performance-critical UI.** E20-S03's heatmap renders 365 cells with tooltips, date formatting, and event handlers. Round 1 found 4 HIGH performance issues that Round 2 verified were correctly fixed. For components with N>100 elements, the review investment pays back in user-facing performance. For simpler CRUD features like E20-S01 and E20-S04, one round sufficed.

3. **Pure-functional algorithm modules are the most reviewable and testable pattern.** E20-S02's `spacedRepetition.ts` (SM-2) and E20-S03's `activityHeatmap.ts` (grid builder) are both pure functions with explicit parameters and no side effects. Their unit tests need zero mocking. When the code review found issues, the fixes were localized and verifiable. Contrast with store-level bugs (stale closures, TOCTOU races) which require understanding async state flow.

4. **Silent-pass tests are worse than missing tests.** The E20-S03 tooltip test's `if` guard allowed it to pass without asserting anything. A missing test shows up as a gap in coverage reports; a silent-pass test shows up as "covered" while detecting nothing. Every `if`/conditional guard around an E2E assertion should be treated as a code smell during review.

5. **The AC compliance re-read action item from E27 is overdue and would have prevented E20-S02's BLOCKER.** Two consecutive epics have produced BLOCKERs that were AC drift (E27-S01: `replace: true`; E20-S02: missing `noteId`). Both were caught in review, but both could have been caught in 2 minutes by re-reading the ACs against the implementation. This is a discipline problem that keeps recurring because the process improvement has not been implemented.
