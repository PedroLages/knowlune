## Successes and Strengths

### 1. Clean Server Architecture with Good Separation of Concerns

The iCal backend separates route handling (`server/routes/calendar.ts`) from generation logic (`src/lib/icalFeedGenerator.ts`). The 404 information-hiding on invalid tokens (no distinction between "token doesn't exist" and "token expired") is a good security default. The route mounts before JWT middleware, enabling unauthenticated calendar access correctly.

### 2. Cryptographically Sound Token Generation

Using `crypto.getRandomValues(new Uint8Array(20))` (160-bit entropy, Web Crypto API) rather than `Math.random()` is the correct choice. The in-code comment explaining why Node's `crypto.randomBytes` must not be used in a browser context is valuable — this is exactly the kind of "decision archaeology" that prevents future regressions.

### 3. AlertDialog for Destructive Action (Regenerate)

The regenerate flow correctly gates behind a shadcn AlertDialog confirmation. The S03 design review pre-emptively recommended this before the UI story, and S04 delivered it correctly. The destructive action pattern (old calendar subscriptions stop working) is handled with appropriate user friction.

### 4. Correct `useId()` and `useCallback`/`useMemo` Usage in New Components

`StudyScheduleEditor` uses `useId()` for form field IDs (React 18+ pattern preventing collision in concurrent rendering), and all async operations have `useCallback` wrappers with appropriate deps. No inline styles detected across the new components.

### 5. S06 Widget Follows Established Patterns

`TodaysStudyPlan` widget on Overview correctly uses `Card`/`CardHeader`/`CardContent`, design tokens, proper ARIA labels, midnight crossover refresh, and scalar-dep memoization (`useFlashcardStore(s => s.flashcards.length)` rather than deep equality). S06 also had the best test coverage of the epic (4/6 ACs covered by E2E specs).

### 6. RLS Policies Complete on Both Tables

`study_schedules` and `calendar_tokens` both have correct RLS: SELECT/INSERT/UPDATE/DELETE scoped to `auth.uid() = user_id`. Server uses service role key only for the public feed endpoint — correct privilege separation.

## Challenges and Growth Areas

### 1. Persistent Testing Gap Across Multiple Stories (Critical Pattern)

Dana (QA Engineer): "Four out of six stories shipped with significant test coverage gaps. That's our biggest quality risk this epic."

| Story | AC Coverage | Notes |
|-------|------------|-------|
| E50-S01 | Unknown (no review found) | Data model story — Supabase migrations |
| E50-S02 | 0% (0/5 ACs) | `icalFeedGenerator.ts` has 0% unit test coverage; Express route has no tests |
| E50-S03 | 0% (0/4 ACs) | Zero unit or E2E tests despite story notes calling out 4 unit test cases |
| E50-S04 | 0% (0/6 ACs) | No E2E spec, no component unit tests |
| E50-S05 | 0% (0/5 ACs) | No E2E spec, no unit tests — complete gap |
| E50-S06 | 67% (4/6 ACs) | E2E spec exists; SRS feed events (AC1/AC2) untested |

**Root cause:** The iCal generation, token CRUD, and feed management functions require Supabase mocking or an Express test harness that was not set up during the epic. The testing infrastructure for server-side code is not yet established in the project.

**Lesson:** When an epic introduces a **server-side component** (Express route, Supabase migrations, server utilities), the test strategy must explicitly address how server code is tested (integration tests, mock harness, or contract tests) before implementation begins, not after review.

### 2. S05 Shipped with BLOCKER (Dead-Code Routing) — Required Extra Review Round

Bob (Scrum Master): "S05 had a BLOCKER on first review — the `StudyScheduleEditor` was integrated into `CourseHeader.tsx`, which is only used by `UnifiedCourseDetail.tsx` — a component no longer routed in the app. AC3 was unreachable."

The review caught this correctly and a fix commit (`fbfae0af`) moved the editor to `CourseOverview.tsx`. But this routing gap slipped through implementation.

**Root cause:** The developer was not aware that `UnifiedCourseDetail` had been superseded by `CourseOverview` in the routing layer. The routing architecture evolved during the mega epic run and some components became dead code.

**Lesson:** Before integrating new UI into an existing component, verify that component is actively routed. Quick check: `grep -r "UnifiedCourseDetail\|CourseOverview" src/app/routes.tsx`.

### 3. Double Toast on Regenerate (S04 HIGH Finding)

`handleRegenerate` in `CalendarSettingsSection.tsx` fired a `toast.warning()` after `regenerateFeedToken()`, which itself fires `toast.success()`. Two simultaneous toasts — one success, one warning — created contradictory feedback.

**Root cause:** The store-level toast and the component-level toast were written independently without coordination. This is a symptoms of store actions that have side effects (toasts) bleeding into UI components that add their own.

**Lesson:** Establish a clear convention: **either the store fires the toast, or the component fires the toast — never both.** Prefer component-level toasts (the store should only manage state, not UI side effects).

### 4. `createClient` Instantiated Per Request (S02 MEDIUM)

The Supabase client in the calendar route was instantiated on every request, rather than at module initialization. While not a correctness bug, this creates unnecessary overhead per iCal fetch — a calendar app polling every 15 minutes will hit this repeatedly.

**Lesson:** Supabase client initialization should always be module-level (singleton). Add to engineering patterns.

### 5. Timezone Bug in SRS Events (S06 MEDIUM — Correctness Risk)

`generateSRSSummaryEvents()` used `new Date(year, month - 1, day, hours, minutes)` which creates dates in the **JavaScript runtime's local timezone**, not the user's specified timezone. For users in non-local timezones, SRS events will appear at wrong times.

**Root cause:** JavaScript's `Date` constructor behavior with local vs. UTC is a classic footgun. The fix (use `Date.UTC()` or ical-generator's timezone-aware types) is straightforward.

**Lesson:** When working with calendar/scheduling features, all date construction must be explicit about timezone. Add a lint note or comment template for any new `new Date(year, month, ...)` calls in calendar code.

### 6. Stale State After Failed Token Regenerate (S03 HIGH — Security Invariant)

If `regenerateFeedToken()` deletes the old token but then fails on INSERT, the catch block leaves `feedToken` at the old value (now deleted from the DB). The user sees an active feed URL that returns 404.

**Root cause:** Error recovery in multi-step async operations (delete then insert) requires explicit state correction in the catch branch.

**Lesson:** Multi-step async operations that mutate external state (DB delete then insert) must handle partial failure. Pattern: if step 1 (delete) succeeds but step 2 (insert) fails, the catch block must set state to reflect what actually happened (no token), not what was true before the operation.

## Key Insights and Patterns

### Pattern 1: Server Code Needs Its Own Test Infrastructure

E50 introduced a meaningful Express server (`server/routes/calendar.ts`). The project has robust E2E patterns for browser interactions but no established pattern for testing Express routes or server utilities. This is a growing test debt as more server-side features land.

**Recommended approach for future server stories:** Either (a) use Supertest for route integration tests, or (b) test the generator functions (`icalFeedGenerator.ts`) as pure unit tests with no server dependency. The generator is a pure function — it should have been tested independently of the route.

### Pattern 2: Token-in-URL Architecture (Accepted Trade-off)

The calendar feed uses tokens in the URL path (`/api/calendar/:token.ics`). This is the industry standard for calendar subscriptions (Google Calendar, Outlook all use this pattern). The security review correctly marks this as ACCEPTABLE — tokens appear in server logs but this is unavoidable for standards-compliant iCal feeds. The 160-bit entropy and regeneration capability are the right mitigations.

### Pattern 3: `FREE_STUDY = '__free__'` Sentinel Pattern

Using a sentinel string constant for optional `courseId` (rather than `null` or `undefined`) in `StudyScheduleEditor` avoids nullable type complexity in a controlled Select component. This is a pragmatic pattern for UI forms where a Select cannot have an undefined value.

### Pattern 4: Store-Level vs. Component-Level Toast Convention

This epic made the toast ownership issue concrete. The agreed convention going forward:
- **Stores own state mutations** — they should NOT fire toasts
- **Components own UX feedback** — they fire toasts after awaiting store actions
- Exception: if a store action is called from multiple components, the store may fire a toast to avoid duplication — but this should be documented at the action definition

### Pattern 5: Scalar Deps for Store Subscriptions

`useFlashcardStore(s => s.flashcards.length)` (scalar) rather than `useFlashcardStore(s => s.flashcards)` (array reference) as a `useMemo` dependency prevents unnecessary recomputation when flashcard content changes but count doesn't. This is the correct pattern for performance-sensitive widget subscriptions.
