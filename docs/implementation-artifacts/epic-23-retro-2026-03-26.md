## Successes and Strengths

### Pedro (Project Lead)

**Pedro:** Five things stood out in this epic, and they tell a story about infrastructure maturity vs. process discipline.

**1. Zero hardcoded colors across all 6 stories -- the ESLint rule is invisible infrastructure.** The `design-tokens/no-hardcoded-colors` rule introduced in Epic 8 has been catching violations at save-time for so long that nobody thinks about it anymore. This is the gold standard for automation: when enforcement is so reliable that it becomes background noise. Contrast this with the retro follow-through rate (25%), which relies on voluntary discipline. The lesson is clear: automate enforcement, do not rely on commitments.

**2. Config-driven navigation architecture validated under stress.** E23-S04 restructured the sidebar from 3 groups to 3 different groups with different items. Only `navigation.ts` changed. `Layout.tsx` (the sidebar renderer) needed one small addition: a `data-testid` on group separators for E2E testing. `BottomNav.tsx` needed zero changes. The architecture from Epic 25 (`navigationGroups` array consumed generically by all navigation surfaces) proved its value -- group restructuring was a config change, not a refactoring.

**3. Inside-out rename (S03) is the definitive pattern for large-scale renames.** Types -> data -> lib -> pages -> components -> routes -> DB -> tests. Each layer change surfaces TypeScript errors in the next layer. Zero references missed across 53 files. This is now a validated, repeatable pattern.

**4. The ATDD test-first approach in S05 caught real integration issues.** Writing E2E tests first (before implementing the collapsible section) forced clarity on test IDs, expected behaviors, and seed data requirements. When the implementation landed, 3 of 6 test groups passed immediately. The ones that failed pointed directly to missing `data-testid` attributes and incorrect seeding -- quick fixes, not design problems.

**5. The Collapsible pattern from Challenges.tsx was directly reusable in S05.** Same Radix Collapsible component, same ChevronDown rotation, same localStorage persistence pattern. The implementation was pattern-matching, not invention. This is the compound return on establishing UI patterns early.

## Challenges and Growth Areas

### Pedro (Project Lead)

**Pedro:** Three recurring patterns emerged in the S04-S06 review rounds, and one of them is genuinely concerning.

**1. E2E import paths in `regression/` specs were wrong in ALL THREE stories (S04, S05, S06).** Every story that created E2E tests in `tests/e2e/regression/` used `../support/` instead of `../../support/`. This is the import path for tests in `tests/e2e/` (one level up). The regression directory is one level deeper. This was a BLOCKER in S04 (tests could not run at all) and had to be fixed in S05 and S06 as well. The same mistake, three times in a row, in a single session. This is not a knowledge gap -- it is a systematic failure to check the file's directory depth before writing imports.

**Why this keeps happening:** The plan templates and ATDD test templates show import paths for `tests/e2e/` (the original test location). When tests are placed in `tests/e2e/regression/` instead, the developer copies the template import path without adjusting for the extra directory level. This is a textual copying error that TypeScript cannot catch at save-time (Playwright does not type-check import paths in the same way). It only fails at runtime.

**Action required:** Either (a) add an ESLint rule that validates import paths for `tests/e2e/regression/` files, or (b) move ALL story E2E tests to `tests/e2e/regression/` by default so the template paths are correct for the common case.

**2. Missing progressive disclosure seeding in E2E tests.** S04 tests asserted that sidebar items were visible, but many items have `disclosureKey` which hides them until the user performs a triggering action. Without seeding `knowlune-sidebar-show-all-v1` in `beforeEach`, gated items are invisible and assertions fail. This was a BLOCKER in S04. It is the same class of problem as the welcome wizard blocking tests in E27 -- cross-cutting test state that new specs must opt into.

**3. Test ID mismatches between plan and implementation.** S06 had 6 issues, several related to test assertions using data-testid values that did not exist in the component. For example, the plan specified `data-testid="specialty-badges"` but the component rendered badges in a `<div>` without that testid. The fix was trivial (add the testid), but the mismatch should have been caught during implementation before reaching review.

## Key Insights and Learnings

1. **E2E import paths in nested directories are a systematic error pattern.** Three consecutive stories made the same `../support/` vs `../../support/` mistake for tests in `tests/e2e/regression/`. This is a template-copying error, not a reasoning error. The fix must be structural: either update templates to use the deeper path, add a lint rule, or consolidate test directories. Relying on developers to manually adjust paths has a 0% success rate in this session.

2. **Review round count correlates with test infrastructure gaps, not code complexity.** S01-S03 had 0 review issues each (1 round). S04-S06 had 5-6 issues each (2 rounds). The difference was not code complexity -- S04 was a single config file change. The issues were all test infrastructure: wrong import paths, missing localStorage seeds, missing test IDs. The code itself was clean in all three stories. This suggests that test infrastructure (fixtures, helpers, templates) is the primary bottleneck for review efficiency, not production code quality.

3. **The 2-round pattern is the "floor" for stories with new E2E tests.** Until the E2E test template and fixture problems are resolved, any story that creates new E2E test files will require at least 2 review rounds -- one for the code, one for the test infrastructure issues. This is predictable and should be factored into velocity estimates.
