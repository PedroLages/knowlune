## Successes and Strengths

### 1. Option B Pivot Was Fast and Produced No Schema Debt

S01 hit a blocker immediately: the original story design assumed `Course.modules[].lessons[].keyTopics[]` from the deprecated `Course` type. `useCourseStore` was dropped in E89-S01 (Dexie v30 schema). The pivot to Option B — using `ImportedCourse.tags[]`, `ImportedCourse.category`, and `Question.topic` — was decided before any code was written. The resulting design is actually cleaner: no schema migration, no new fields to populate, works with existing data.

The speed of the pivot was enabled by enough context in the story spec about the E89-S01 migration to reason about the problem immediately.

### 2. knowledgeTierUtils.ts Extraction Before Duplication

Tier logic was extracted into a shared util (`src/lib/knowledgeTierUtils.ts`) before it was copy-pasted across the widget (S03) and page (S04). This was proactive deduplication. Future epics E62 (FSRS-retention treemap) and E71 (Knowledge Map action suggestions) can import the same tier thresholds and badge styles without drift.

### 3. Pre-Review Edge Case Analysis in S02

Five HIGH-severity edge cases were identified and documented in the S02 story spec before implementation began: division by zero (all signals null), misleading 100% confidence (completion-only), stale data on quiz completion, NaN from null `reviewedAt`, and quiz topic canonicalization mismatch. This level of pre-implementation analysis is the target standard for scoring/math stories.

### 4. date-fns formatDistanceToNow Adoption

Consistent relative time formatting (`"about 3 days ago"`) across widget and page using `date-fns formatDistanceToNow`. Establishes the pattern for future time-display features (E57, E62).

### 5. Zero Design Token Violations

All four stories passed design review with zero hardcoded color findings.

## Challenges and Growth Areas

### 1. db.courses Schema Crash (S02 BLOCKER)

S02's store called `db.courses.toArray()` — a table dropped in E89-S01's Dexie v30 migration. This is the data-layer equivalent of E54-S03's dead-component problem. Both are 60-second verification failures caught only by code review (R1 for E56-S02, R1 for E54-S03).

The fix pattern is identical: before Task 1, verify that all referenced `db.*` tables exist in the current Dexie schema (`src/db/schema.ts`). This is not in the pre-implementation checklist. It needs to be.

### 2. S03 Required 3 Review Rounds (Touch Targets + Seeding)

Touch targets below 44x44px is a recurring review finding across multiple epics. It is not a complexity problem — it is a self-review failure. The same applies to E2E seeding helpers: the factory utilities exist in `tests/helpers/factories/`, the ESLint rule flags manual seeding, but developers continue writing one-off seeding code.

Neither issue required architectural reasoning to catch — they required reading the checklist.

### 3. Treemap Root Node Visual Defect (S04)

S04's treemap had a broken root node rendering that produced incorrect layout. It was a runtime visual defect visible within 3 seconds of loading the page with seed data. It was caught in review, not in self-testing.

The standard: before submitting for review, load the feature in the dev browser with seed data and visually confirm the primary UI renders without obvious defects. This is a 2-minute check that prevents one entire review round.

### 4. Zero First-Pass Stories

Every story had at least one review-preventing issue. Two of the four — the db.courses BLOCKER (S02) and the treemap visual defect (S04) — are self-testing failures, not complexity failures. The review pipeline is functioning correctly; it should not be the primary place these are caught.

## Key Takeaways

**1. Data layer assumptions need the same verification rigor as routing assumptions.**
E54 taught us to check `routes.tsx` before building into a component. E56 taught us to check `db.tables` before referencing them. Both are 60-second checks that prevent BLOCKER-class rework. The pre-implementation checklist must include both.

**2. Lessons don't stick unless they're embedded in the artifact developers read during work.**
Three E54 action items were documented in the retro narrative but never materialized into story template or checklist changes. This retro produces four concrete template updates — not narrative text. If the update isn't in the template, it won't be applied.

**3. Self-testing discipline is the highest-ROI improvement available right now.**
Zero first-pass stories with 8 total review rounds, at least 2 rounds of which were preventable by looking at the running UI for 3 minutes before submitting. The review pipeline is functioning — but it should not be the first place runtime visual defects and touch target failures are caught.
