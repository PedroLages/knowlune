## What Went Well

### 1. Simplification Over Abstraction (S01)
The review process correctly identified that the SM-2/FSRS branching in the initial implementation was based on a false assumption. The codebase is FSRS-only. Removing the branch simplified the aggregation layer and eliminated a permanent maintenance burden. This kind of early simplification is worth more than any feature work.

### 2. Shared Utility Extraction (S02)
Decay formatting logic that was duplicated in S02 was caught by the deduplication scan and extracted to a shared utility before merge. The utility is now available to future stories touching decay display. Pattern follows the "extract on second use" principle correctly.

### 3. Clean S01/S04 Execution
Both stories required only 2 review rounds and addressed real issues (redundant param pass-through in S01, dead code in dark mode test and Radix selector fragility in S04). These stories went through the review cycle efficiently.

### 4. Semantic State Testing Pattern for Dark Mode (S04)
The dark mode assertion failure forced a better test design — checking semantic CSS class/attribute state rather than computed RGB values. This is a more resilient test pattern that doesn't break when themes change. The fix is transferable to any component with theme-sensitive visual states.

## What Could Be Improved

### 1. Test Quality Pre-Submit Self-Review
S03 entered review with two class-one test anti-patterns: `Date.now()` indirection (non-deterministic time) and inline function replica (testing a copy). Both are documented in the test quality knowledge base. A deliberate self-review pass before submit would have caught these and eliminated at least one review round.

**Root cause:** Writing tests feels like the "final step before done," creating time pressure that skips the review step.

### 2. Duplication Visibility During Implementation (S02)
The duplicated decay formatting logic was caught post-implementation by the dedup scan, not during writing. The two instances were in the same PR, which means they were written close together in time. At the moment of writing the second instance, there was no signal that a shared utility should exist.

**Root cause:** No real-time "you've written this before" signal during implementation.

### 3. E2E Radix Selector Pattern Still Recurring
Fragile Radix internal selectors continue to appear in E2E tests despite documented guidance. S04 used a Radix selector that had to be replaced with a `data-testid` in review.

**Root cause:** Writing tests against the visible DOM is the path of least resistance. Adding `data-testid` requires touching the component file, which feels like extra scope.

### 4. MutationObserver HMR Leak — No Static Analysis Signal (S02)
The `MutationObserver` created in S02's UI component was not cleaned up, causing a memory leak under React HMR and potential issues in long-running sessions. There's no ESLint rule that would catch this specific pattern.

**Reflection:** This class of bug (imperative observer/listener not cleaned up in useEffect) is a recurring React pattern. The existing `react-hooks-async/async-cleanup` rule covers `async` cleanup but not synchronous observer teardown.

## Lessons Learned (Pattern Extraction)

### Pattern 1: Validate Algorithmic Assumptions Before Branching
When an implementation plan includes an algorithm fork (e.g., SM-2 vs. FSRS), validate which branches actually exist in the codebase before writing the code. Unnecessary branches are worse than missing features — they add surface area to maintain indefinitely.

**Operationalize:** Add "confirm algorithm branches in use" to story pre-flight checklist for any story involving scoring, scheduling, or algorithmic logic.

### Pattern 2: Test the Real Function, Never a Replica
Any test that re-implements the logic it's testing is validating the test, not the production code. Always import and call the actual function under test. If the function is hard to import in test context, that's a signal the function needs to be extracted to a testable module.

**Operationalize:** Add to the test quality self-review checklist: "Does this test import the real implementation, or does it re-implement the logic inline?"

### Pattern 3: Semantic State Over Computed Style for Theme-Sensitive Assertions
E2E assertions on computed colors (`fill`, `background-color`, `color` as RGB) break under dark mode with CSS custom properties. Instead, assert on the semantic state that drives the color: the CSS class applied, the `data-*` attribute, or the Tailwind variant. The visual correctness of that state is the design system's responsibility.

**Operationalize:** Add to E2E test patterns: "For assertions about themed visual states, assert on `data-*` attributes or CSS classes, not computed color values."
