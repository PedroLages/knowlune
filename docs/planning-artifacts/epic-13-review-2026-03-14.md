# Epic 13 Review: Navigate and Control Quiz Flow

**Date:** 2026-03-14
**Stories Reviewed:** 6
**Sub-Agent Reviews:** 6 parallel adversarial reviews

---

## Epic-Level Verdict: **REVISE**

No story is individually REJECT-level, but 3 systemic issues affect every story: zero QFR traceability (same governance gap as Epic 12), hardcoded colors in code samples violating the project's design token mandate, and Epic 12 duplication where Story 13.2 re-proposes infrastructure already built in Epic 12's revised stories. Timer persistence in Story 13.3 and crash-recovery order persistence in Story 13.5 are the highest-severity functional gaps.

---

## Systemic Findings

### Critical (affects multiple stories)

- **Zero QFR traceability across ALL 6 stories.** No `QFRs Covered` line at epic level. No `FRs Fulfilled` lines in any story. The quiz UX spec defines QFR1-QFR61 but not a single QFR is referenced in any Epic 13 story. Identical governance failure as Epic 12 (now fixed there). — *Affects: Stories 13.1-13.6*

- **Hardcoded colors in code samples.** Story 13.4 uses `text-gray-600` and `text-green-600` in its ScoreSummary code snippet. The project mandates design tokens (`text-muted-foreground`, `text-success`). ESLint rule `design-tokens/no-hardcoded-colors` will block these at save-time. — *Affects: Story 13.4 (explicitly), potential pattern in implementation*

- **Epic 12 duplication conflict.** Story 13.2 proposes to ADD `markedForReview: string[]` to QuizProgress and ADD a toggle action to useQuizStore. But the revised Story 12.1 already defines `markedForReview: string[]` and Story 12.3 already defines `toggleReviewMark`. Story 13.2 also names the action `toggleMarkForReview` (different from 12.3's `toggleReviewMark`). — *Affects: Stories 13.2, and any story consuming the mark-for-review feature*

### Cross-Reference Issues

- **Touch targets: 44px vs 48px.** Story 13.1 Design Review Focus says "≥44px touch targets" but the UX spec specifies `w-10 h-10` (40px) for grid buttons and `h-11` (44px) for action buttons — while the UX spec elsewhere mandates `min-h-12` (48px) for answer options. The grid buttons are actually 40px, which is below both the 44px and 48px targets. Inconsistency needs resolution.

- **Timer state schema undefined.** Story 13.3 says "timer state is preserved" and "restored correctly" but never defines what timer state IS (`remainingTimeMs`? `pausedAt`? `startTimestamp + elapsed`?). Story 13.4 says "the timer resets to the original time limit." Without a shared timer state type, Stories 13.3 and 13.4 will implement incompatible timer models.

- **`variant="primary"` vs UX spec styling.** Story 13.3's code snippet uses `variant="primary"` but the UX spec specifies explicit classes (`bg-brand text-brand-foreground hover:bg-brand-hover rounded-xl h-12 px-8`). If shadcn/ui's "primary" variant doesn't map to those exact tokens, the implementation drifts.

## Per-Story Reviews

### Story 13.1: Navigate Between Questions — REVISE

**Strengths:**
- Explicit dependency chain (12.1, 12.3, 12.4, 12.5)
- Question grid visual states well-specified (4 distinct states)
- E2E tests cover critical user journeys

**Key Issues:**
- Q1 "Previous" button behavior contradicts itself (disabled vs hidden vs "Start Over")
- `navigateToQuestion` has no bounds checking (negative/out-of-range index)
- Auto-save before navigation stated as AC but has zero technical specification
- Animation requirement from UX spec (fade + slide, 200ms) completely absent
- No responsive layout ACs for three breakpoints
- No QFR traceability
- "Marked for review" grid state missing despite being built in this component

### Story 13.2: Mark Questions for Review — REVISE

**Strengths:**
- Five Given/When/Then blocks cover full lifecycle
- Accessibility coverage matches UX spec
- Visual indicator design aligns with UX spec tokens

**Key Issues:**
- Duplicates infrastructure already in Epic 12 (markedForReview field and toggle action)
- Action naming mismatch: `toggleMarkForReview` vs Epic 12's `toggleReviewMark`
- `state.currentProgress!` non-null assertion will crash if called outside active quiz
- No QFR traceability
- Pause/resume persistence of marks not tested
- "Submit warning" E2E test has no corresponding AC

### Story 13.3: Pause and Resume Quiz — REVISE

**Strengths:**
- Clear resume UX ("Resume Quiz (5 of 12 answered)")
- Leverages existing Zustand persist from 12.3
- Dependencies explicit

**Key Issues:**
- "No data lost even if browser crashes" is physically impossible to guarantee with localStorage
- Timer state never defined (remainingSeconds? timestamp? countdown value?)
- Debounced auto-save contradicts crash recovery guarantee
- No AC for multi-quiz isolation (starting new quiz overwrites old progress)
- No AC for stale progress validation on resume (quiz content may have changed)
- `beforeunload` unreliable on mobile browsers (iOS Safari)
- No E2E test for timer restoration
- `variant="primary"` vs UX spec explicit styling

### Story 13.4: Unlimited Quiz Retakes — REVISE

**Strengths:**
- Clear mastery-based learning value
- Broad test coverage (unit, E2E, accessibility)
- Reasonable complexity estimate

**Key Issues:**
- `text-gray-600` and `text-green-600` hardcoded in code sample (violates design tokens)
- `retakeQuiz` action doesn't verify prior attempt is persisted to Dexie before clearing
- No AC for score regression display (lower score than previous best)
- "View All Attempts (Story 16.1)" referenced in AC but untestable
- Lesson page file missing from "Files to modify"
- Attempt numbering/ordering not defined

### Story 13.5: Randomize Question Order with Fisher-Yates Shuffle — REVISE

**Strengths:**
- Algorithm choice correct and well-specified
- Immutability and generic typing
- Distribution test included

**Key Issues:**
- Crash-recovery persistence completely missing (must store `questionOrder` in currentProgress)
- `quiz.questions = fisherYatesShuffle(...)` mutates quiz object, violating Zustand immutability
- No integration with UX spec's "Shuffle questions by default" user preference setting
- "Equal probability (1/n!)" AC not practically testable
- No seeded PRNG for deterministic testing
- E2E "different order on retake" will be flaky for small quizzes

### Story 13.6: Handle localStorage Quota Exceeded Gracefully — REVISE

**Strengths:**
- Layered storage strategy (localStorage → sessionStorage → IndexedDB)
- Non-blocking toast UX
- Small, well-bounded scope

**Key Issues:**
- `getItem` catches ALL exceptions (SecurityError in private browsing) but only documents QuotaExceeded
- No double-failure handling (both storages full → unhandled throw crashes Zustand)
- `removeItem` not wrapped in try/catch (crashes in private browsing)
- "Free space by clearing old data" AC has zero implementation design
- Stale data from split storage not reconciled (localStorage reads trump fresher sessionStorage)
- Toast suggests "clear browser data" which could delete IndexedDB quiz history
- No retry logic before fallback (architecture mandates exponential backoff)
