## Executive Summary

E09-S03 adds semantic search capabilities to the Notes page with a clean toggle interface, embedding count display, and tooltip guidance. The implementation demonstrates solid UX thinking with proper disabled states and helpful empty state messaging. However, **two blocker issues prevent merge**: hardcoded Tailwind colors violate design token standards, and the native checkbox lacks proper styling. The responsive design works well across all breakpoints with no horizontal scroll issues.

## Findings by Severity

### Blockers (Must fix before merge)

**1. Hardcoded Tailwind Colors on Tag Badges**
- **Location**: `src/app/pages/Notes.tsx:325, 489`
- **Issue**: Active tag badges use `bg-blue-600 text-white hover:bg-blue-700` instead of design tokens
- **Code**:
  ```tsx
  // Line 325 (within note cards)
  activeTag === tag && 'bg-blue-600 text-white hover:bg-blue-700'
  
  // Line 489 (filter bar)
  activeTag === tag ? 'bg-blue-600 text-white hover:bg-blue-700' : 'hover:bg-accent'
  ```
- **Impact**: Breaks theme consistency, prevents proper light/dark mode support, accumulates tech debt
- **Suggestion**: Replace with theme tokens:
  ```tsx
  activeTag === tag && 'bg-brand text-white hover:bg-brand-hover'
  // or use the existing Badge component's `variant="default"` which already uses brand tokens
  ```
- **Why it matters**: The design token system exists specifically to prevent hardcoded colors. This violates the project's established ESLint rule `design-tokens/no-hardcoded-colors` and undermines theme consistency across the app.

**2. Unstyled Native Checkbox**
- **Location**: `src/app/pages/Notes.tsx:444-450`
- **Issue**: Using native `<input type="checkbox" className="rounded">` instead of shadcn/ui Checkbox component
- **Evidence**: 
  - Computed `borderRadius: "0px"` (Tailwind `rounded` class not applying to native checkbox)
  - Minimal styling, inconsistent with rest of UI
- **Impact**: Visual inconsistency, poor hover/focus states, doesn't match design system
- **Suggestion**: Replace with shadcn/ui Checkbox component:
  ```tsx
  import { Checkbox } from '@/app/components/ui/checkbox'
  
  <Checkbox
    id="semantic-search"
    checked={useSemanticSearch}
    onCheckedChange={setUseSemanticSearch}
    disabled={vectorStoreInstance?.size === 0}
    data-testid="semantic-search-toggle"
  />
  <Label htmlFor="semantic-search">Semantic Search</Label>
  ```
- **Why it matters**: Checkboxes need proper focus indicators, hover states, and accessible click targets. The shadcn/ui Checkbox component provides all of this out-of-the-box and matches the existing design system.

### High Priority (Should fix before merge)

**3. Missing ARIA Label on Info Icon**
- **Location**: `src/app/pages/Notes.tsx:459-472`
- **Issue**: Info icon (`<Info className="h-4 w-4" />`) wrapped in tooltip but no `aria-label` on the TooltipTrigger
- **Impact**: Screen reader users won't know what the icon represents without activating it
- **Suggestion**: Add ARIA label to the trigger:
  ```tsx
  <TooltipTrigger asChild>
    <div data-testid="no-embeddings-info" aria-label="Semantic search requirements">
      <Info className="h-4 w-4 text-muted-foreground" />
    </div>
  </TooltipTrigger>
  ```

**4. Checkbox Lacks Explicit Label Association**
- **Location**: `src/app/pages/Notes.tsx:444-457`
- **Issue**: Checkbox is wrapped in `<label>` but has no `id`/`for` attribute pairing
- **Evidence**: Verified via browser evaluation:
  ```js
  checkbox: { id: "none", hasAssociatedLabel: true }
  label: { hasFor: false, wrapsCheckbox: true }
  ```
- **Impact**: While wrapping works, explicit association is more robust and better for assistive tech
- **Suggestion**: Add `id` to checkbox and `htmlFor` to label:
  ```tsx
  <label htmlFor="semantic-search-toggle" className="...">
    <input
      id="semantic-search-toggle"
      type="checkbox"
      ...
    />
  ```

### Medium Priority (Fix when possible)

**5. Similarity Badge Not Tested (Feature Incomplete)**
- **Location**: `src/app/pages/Notes.tsx:334-338`
- **Issue**: Similarity score badge (`{Math.round(item.similarity * 100)}% match`) only renders when `useSemanticSearch && item.similarity != null`, but no embeddings exist in test data (vectorStoreInstance.size = 0)
- **Impact**: Cannot visually verify badge styling, positioning, or text color contrast in this review
- **Suggestion**: In next story or testing phase:
  - Create test data with embeddings
  - Verify badge contrast meets WCAG AA (4.5:1)
  - Ensure badge doesn't overlap with timestamp on small screens
  - Test with various similarity percentages (0%, 50%, 100%)

**6. Embedding Count Display Could Be More Prominent**
- **Location**: `src/app/pages/Notes.tsx:453-456`
- **Issue**: Embedding count `(0 embeddings)` uses `text-muted-foreground` which is low contrast
- **Impact**: Users might miss this valuable context about feature availability
- **Suggestion**: Consider slightly darker color or add an indicator badge:
  ```tsx
  <Badge variant="secondary" className="ml-1">
    {vectorStoreInstance.size} embeddings
  </Badge>
  ```

### Nitpicks (Optional)

**7. Semantic Search Section Could Use More Spacing**
- **Location**: `src/app/pages/Notes.tsx:442-473`
- **Issue**: Semantic search section has same vertical spacing as tag filter (both use default `space-y-6` from parent)
- **Impact**: Very minor - section doesn't visually separate from search input above
- **Suggestion**: Consider adding `mt-1` or `pt-1` to create subtle hierarchy

## What Works Well

1. **Excellent Empty State UX**: The tooltip message "No embeddings available. Create some notes to enable semantic search." is clear, actionable, and friendly. This is exactly the kind of guidance learners need.

2. **Proper Disabled State Handling**: The checkbox correctly disables when `vectorStoreInstance?.size === 0`, preventing user confusion. The disabled cursor (`cursor: default`) correctly indicates non-interactivity.

3. **Responsive Design Excellence**: Tested at 1440px, 768px, and 375px:
   - No horizontal scroll detected (verified `scrollWidth: 364 < clientWidth: 375` at mobile)
   - Semantic search section adapts properly (`gap: 8px` maintained across breakpoints)
   - All text remains readable at mobile size

4. **Consistent Design Tokens (Mostly)**: Background color `rgb(250, 245, 238)` matches `#FAF5EE` specification perfectly. Note card border radius `24px` matches design standards.
