### Medium Priority (Fix when possible)

#### M1 — Switch element touch target is 32×18px, well below the 44×44px minimum

**Issue**: The Radix UI `<Switch>` component renders a 32×18px button (`role="switch"`). While the *row container* has `min-h-[44px]`, the clickable/tappable element itself is only 32×18px. On mobile, users must tap the narrow thumb element rather than an expanded hit area.

**Location**: `src/app/components/settings/NotificationPreferencesPanel.tsx` — all toggle rows, confirmed by computed size: `{ w: 32, h: 18 }` for `#notif-knowledge-decay` on both desktop and mobile.

**Evidence**: Mobile switch measurements (375px viewport):
```
notif-knowledge-decay: { w: 32, h: 18 }
notif-course-complete:  { w: 32, h: 18 }
// all 7 notification toggles identical
```

**Impact**: On touchscreens, a 18px tall target requires precision tapping. WCAG 2.5.5 (Target Size, AAA) recommends 44×44px. WCAG 2.5.8 (Target Size Minimum, AA in WCAG 2.2) requires 24×24px minimum — the 18px height falls short of even this threshold. For learners on phone browsers, this increases mis-tap rate and reduces usability of the notification preferences.

**Suggestion**: Wrap each `<Switch>` in a `<label>` that expands the tap region, or add padding/negative margin to increase the touch hit area without changing visual size. The existing label (`<Label htmlFor="notif-{type}">`) already handles the `for` association — adding `cursor-pointer` and extending its click area to cover the full row is one approach. Alternatively, use Tailwind's `[&>button]:min-h-[44px]` on the row or a transparent pseudo-element overlay via CSS.

Note: This is a platform-wide pattern affecting all notification toggles, not specific to the knowledge-decay addition. It is noted here because E60-S01 adds a new toggle following the same pattern.

#### M2 — Switches lack `aria-describedby` linking to description text

**Issue**: Each toggle has a description sub-text (e.g. "When topic retention drops below a safe threshold") that is visually paired with the switch, but the switch element has `aria-describedby: null`. Screen readers reading the switch aloud will announce the `aria-label` only and will not automatically read the description text.

**Location**: `src/app/components/settings/NotificationPreferencesPanel.tsx:120-125`

```tsx
<Switch
  id={`notif-${toggle.type}`}
  checked={enabled}
  onCheckedChange={checked => setTypeEnabled(toggle.type, checked)}
  aria-label={`${toggle.label} notifications`}
  // Missing: aria-describedby={`notif-${toggle.type}-desc`}
/>
```

And the description `<p>` at line 117:
```tsx
<p className="text-xs text-muted-foreground mt-0.5">{toggle.description}</p>
// Missing: id={`notif-${toggle.type}-desc`}
```

**Evidence**: Confirmed via `switchAriaDescribedby` audit — `ariaDescribedby: null` for all 7 notification-type switches including `notif-knowledge-decay`.

**Impact**: Screen reader users (VoiceOver, NVDA) will hear "Knowledge Decay Alerts notifications, switch, on" but will not hear the clarifying description "When topic retention drops below a safe threshold." For a feature that introduces a new and potentially unfamiliar concept (knowledge decay / spaced repetition), this context is particularly important for accessibility.

**Suggestion**: Add `id` props to each description paragraph and link them via `aria-describedby`:

```tsx
// On the <p> description element:
<p id={`notif-${toggle.type}-desc`} className="text-xs text-muted-foreground mt-0.5">
  {toggle.description}
</p>

// On the <Switch>:
<Switch
  id={`notif-${toggle.type}`}
  aria-label={`${toggle.label} notifications`}
  aria-describedby={`notif-${toggle.type}-desc`}
  // ...
/>
```

### Nitpicks (Optional)

#### N1 — `role="group"` on a `<div>` is redundant but not harmful

**Issue**: `<div role="group" aria-label="Notification type toggles">` uses `role="group"` on a `<div>`, which works correctly. A minor improvement would be using `<fieldset>` + `<legend>` which provides the same grouping semantics natively and does not need an explicit role.

**Location**: `NotificationPreferencesPanel.tsx:99`

**Impact**: Negligible — `role="group"` on `<div>` is valid ARIA and screen readers handle it well. A `<fieldset>` + `<legend>` would be semantically stronger but the current implementation passes WCAG.

**Suggestion**: Consider refactoring to `<fieldset>` / `<legend>` in a future pass. Not a priority for this PR.

---

#### N2 — `aria-label="Filter by all notifications"` is slightly awkward phrasing

**Issue**: The read-status filter button for "all" has `aria-label="Filter by all notifications"`. "Filter by all" is grammatically unusual — "Show all notifications" or "No read filter" would read more naturally to screen reader users.

**Location**: `src/app/pages/Notifications.tsx:133` — template string `Filter by ${status} notifications`

**Impact**: Minor — the meaning is clear enough. Screen reader users will understand the intent. This is purely a copy quality nitpick.

**Suggestion**: Differentiate the "all" case in the aria-label template:
```tsx
aria-label={status === 'all' ? 'Show all notifications' : `Filter by ${status} notifications`}
```

## Detailed Findings

### Finding 1: M1 — Switch touch target 32×18px

- **Issue**: Radix Switch renders 32×18px button
- **Location**: `NotificationPreferencesPanel.tsx:120` — `<Switch id={`notif-${toggle.type}`} ...>`
- **Evidence**: Computed `getBoundingClientRect()` on `#notif-knowledge-decay` at 375px: `{ w: 32, h: 18 }`
- **Impact**: Below WCAG 2.5.8 AA minimum (24px). On mobile learners who are toggling notification preferences will frequently mis-tap.
- **Suggestion**: Extend hit area via `className="min-h-[44px] p-[13px]"` on the Switch or use a transparent wrapper to expand the tap zone without changing visual appearance.

### Finding 2: M2 — Missing `aria-describedby` on switches

- **Issue**: `aria-describedby` is null on all 7 notification switches despite each having a visible description paragraph
- **Location**: `NotificationPreferencesPanel.tsx:111-125`
- **Evidence**: Audit of `switchAriaDescribedby` shows `ariaDescribedby: null` for all notification switches
- **Impact**: Screen reader users miss the description text "When topic retention drops below a safe threshold" for the knowledge-decay toggle — the most contextually important description on the panel since knowledge decay is a new concept for many learners.
- **Suggestion**: Add `id={`notif-${toggle.type}-desc`}` to the `<p>` and `aria-describedby={`notif-${toggle.type}-desc`}` to the `<Switch>`.

### Finding 3: N1 — `prefers-reduced-motion` not guarded on quiet-hours reveal

- **Issue**: The quiet hours time picker uses `animate-in fade-in-0 slide-in-from-top-1 duration-300` without a `motion-safe:` Tailwind modifier
- **Location**: `NotificationPreferencesPanel.tsx:154` — `className="grid grid-cols-2 gap-4 animate-in fade-in-0 slide-in-from-top-1 duration-300"`
- **Evidence**: Code review; animation classes present unconditionally
- **Impact**: Users with vestibular disorders who have set `prefers-reduced-motion: reduce` will still see the slide animation. Minor, as the animation is short (300ms) and subtle, but design principle compliance requires respecting this preference.
- **Suggestion**: Add the `motion-safe:` prefix: `motion-safe:animate-in motion-safe:fade-in-0 motion-safe:slide-in-from-top-1`. Alternatively, Tailwind v4 supports `@media (prefers-reduced-motion: no-preference)` in the theme configuration.

---

## Recommendations

1. **Address M2 first** (aria-describedby): This is a 5-minute fix that meaningfully improves screen reader UX for knowledge-decay specifically, since it's a novel concept that benefits from the description being announced.

2. **Address M1 in a settings-wide pass**: The switch touch target issue affects all notification toggles and other settings panels. Rather than patching the `NotificationPreferencesPanel` in isolation, consider a platform-wide fix to the `Switch` wrapper pattern to avoid repeated fixes across all settings panels.

3. **Add motion-safe guard to quiet hours animation**: Simple Tailwind prefix change, zero risk.

4. **File pre-existing axe violations as a separate chore**: The 4 third-party `button-name` and `label` violations in `.styles-module__` components should be tracked in `docs/known-issues.yaml` if they are not already, and scoped to the library responsible (likely Sonner or an embedded media player).
