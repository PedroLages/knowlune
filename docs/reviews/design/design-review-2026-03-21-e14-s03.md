## Executive Summary

E14-S03 adds a `FillInBlankQuestion` component that renders a text input with character counter inside a `fieldset/legend` semantic structure. The implementation is clean, consistent with existing question components, and passes all acceptance criteria visible in the live UI. Two medium-priority accessibility gaps were identified: the character counter lacks an `aria-live` region for screen reader announcement, and the input height (36px) falls below the 44px touch target minimum on mobile. Neither is a blocker, and both have clear, low-effort remediation paths.

---

## What Works Well

1. **Structural consistency with sibling components.** `FillInBlankQuestion` follows the exact same `fieldset/legend` pattern as `MultipleChoiceQuestion` and `TrueFalseQuestion` — same classes (`mt-6` fieldset, `text-lg lg:text-xl text-foreground leading-relaxed pb-4` legend), same `useId()` approach for ARIA linkage, and same `mode` prop for active/disabled states. The codebase reads as one cohesive whole.

2. **Excellent contrast ratios.** All computed contrast values substantially exceed WCAG AA requirements: legend text against card background scores 12.45:1, the muted character counter scores 7.42:1, and input text scores 13.5:1. Learners with visual impairments have strong readability.

3. **Correct ARIA association.** The `aria-labelledby` on the `<input>` correctly references the `<legend>` ID generated by `useId()`. The accessibility tree confirms this: the textbox is announced as `"What is the chemical symbol for water?"` — exactly what a screen reader user needs to understand the question before typing.

4. **Token-clean implementation.** Zero hardcoded hex colors, zero inline styles. All styling uses design tokens (`text-foreground`, `text-muted-foreground`, `w-full max-w-md`) or Tailwind utilities. The ESLint `design-tokens/no-hardcoded-colors` rule would have caught any violations — none are present.

5. **No console errors or regressions.** The only browser warning (`apple-mobile-web-app-capable`) is a pre-existing meta tag issue unrelated to this story.

6. **Correct responsive behavior.** No horizontal scroll at any breakpoint. The input width scales naturally from 324px on mobile to 448px on desktop via `w-full max-w-md`. Card padding correctly drops to `p-4` at mobile and `sm:p-8` at larger viewports (inherited from the Quiz page container).

7. **Robust keyboard navigation.** Tab order within the quiz card is logical: text input → Mark for Review checkbox → Submit button → Question nav buttons. The Submit button and Question 1 nav button both show visible 3px focus rings when focused via keyboard.

### Medium Priority (Fix when possible)

#### M1 — Character counter missing `aria-live` region

The `"0 / 500"` counter (`<span class="text-sm text-muted-foreground block">`) updates on every keystroke but has no `aria-live` attribute. Screen reader users typing an answer will hear their keystrokes but will not hear the count updating. For a learner approaching the 500-character limit, this creates a silent wall — they type, input stops accepting characters, and they have no auditory signal as to why.

- **Location**: `src/app/components/quiz/questions/FillInBlankQuestion.tsx:79`
- **Evidence**: `getComputedStyle` confirms `ariaLive: null` on the counter span.
- **Impact**: Screen reader users cannot monitor their character budget while composing an answer. This is particularly significant for longer free-text answers.
- **Suggestion**: Add `aria-live="polite"` and `aria-atomic="true"` to the counter span. Using `polite` (not `assertive`) ensures the announcement waits for the user to pause typing rather than interrupting each keystroke — a much better experience:
  ```tsx
  <span
    className="text-sm text-muted-foreground block"
    aria-live="polite"
    aria-atomic="true"
  >
    {inputValue.length} / {MAX_LENGTH}
  </span>
  ```

#### M2 — Input height 36px below 44px touch target minimum on mobile

The shared `Input` component uses `h-9` (36px) from the shadcn/ui defaults. At 375px mobile width, the computed input height is 35.998px — 8px short of the 44px WCAG 2.5.5 touch target minimum. This is a pre-existing platform-wide issue (not introduced by this story), but the fill-in-blank input is the primary interaction surface of this question type and is therefore the most important place to address it.

- **Location**: `src/app/components/ui/input.tsx:11` (`h-9` in base classes)
- **Evidence**: `inputHeight: 35.99826431274414`, `inputMeetsTouchTarget: false` at 375px viewport.
- **Impact**: Learners on mobile devices — especially those with motor impairments — may struggle to accurately tap the input, leading to frustration during timed quizzes.
- **Suggestion**: Override the height in the `FillInBlankQuestion` usage:
  ```tsx
  <Input
    className="w-full max-w-md min-h-[44px]"
    ...
  />
  ```
  Adding `min-h-[44px]` only at the quiz question level avoids disrupting all other `Input` usages while meeting the touch target requirement where it matters most. A follow-up platform-wide audit of `Input` usages in mobile-primary contexts is recommended.

### Nitpicks (Optional)

#### N1 — Counter announces every character in isolation; consider debounced announcement

With `aria-live="polite"` added (per M1), the counter will announce on every keystroke pause. For very fast typists or users dictating text, this may feel chatty. A potential enhancement is to only announce the counter when the user pauses for >1s, or only when approaching the limit (e.g., when fewer than 50 characters remain). This is a UX refinement, not a correctness issue.

#### N2 — No `aria-describedby` linking counter to input

While `aria-labelledby` correctly associates the legend with the input, an `aria-describedby` pointing to the counter element would let screen readers announce the current character count as supplementary context when the input receives focus. This is an enhancement beyond what the story requires.

## Recommendations

1. **Add `aria-live="polite" aria-atomic="true"` to the counter span** (M1 — 2-line change). This is the highest-impact accessibility improvement and should be addressed before or alongside this story's merge.

2. **Add `min-h-[44px]` to the `<Input>` in `FillInBlankQuestion`** (M2 — 1-word change). This resolves the touch target concern locally without modifying the shared `Input` component. Open a follow-up ticket for a platform-wide mobile touch target audit.

3. **Consider `aria-describedby` linking the counter to the input** (N2 — enhancement). Would give screen reader users the current count on input focus, improving awareness of the character budget before they start typing.

4. **No structural or visual regressions found.** The integration into `QuestionDisplay.tsx` is minimal and correct — the `fill-in-blank` case follows the identical dispatch pattern as `multiple-choice`, `true-false`, and `multiple-select`.
