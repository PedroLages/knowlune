## High Priority

**[High] `src/app/components/quiz/ScoreSummary.tsx:133` (confidence: 92)**
Trophy icon uses `h-4 w-4` instead of Tailwind v4 `size-4` shorthand. This pattern has recurred since E02-S05 across the codebase. Other icons in the same feature (e.g., `QuizResults.tsx:161` History icon) already use `size-4`.
Fix: `className="h-4 w-4 text-success"` → `className="size-4 text-success"`

**[High] `src/lib/analytics.ts:148-150` (confidence: 85)**
`new Date(a.completedAt).getTime()` is called on every comparison during sort. If `completedAt` is ever an invalid string (e.g. corrupted IndexedDB data), `getTime()` returns `NaN` and the sort comparator returns `NaN`, producing non-deterministic sort order with no error signal. This is a silent failure: corrupted attempt data would cause wrong improvement data to be shown with no indication of the problem.
Fix: Filter out or guard against attempts where `new Date(completedAt).getTime()` is NaN before sorting. Also consider caching parsed timestamps.

**[High] `src/lib/analytics.ts:174` (confidence: 82)**
The plan file (`docs/implementation-artifacts/plans/e16-s03-score-improvement.md`) states `isNewBest: true` on first attempt, but the implementation correctly sets it to `false` (per the story AC). The implementation is correct, but the stale plan could confuse future developers.
Fix: Update the plan file comment or add a code comment referencing the AC decision.

## Medium Priority

**[Medium] `src/app/components/quiz/ScoreSummary.tsx:124` (confidence: 78)**
`+0%` for zero improvement is semantically ambiguous (neither positive nor negative). The AC doesn't specify this case explicitly.
Fix: Consider showing `0%` (no sign) or `=0%` when `roundedImprovement === 0`.

**[Medium] `src/app/components/quiz/ScoreSummary.tsx:155-157` (confidence: 75)**
The improvement value renders as `{sign}{roundedImprovement}%` split across JSX lines. React generally collapses whitespace, but the pattern is fragile. E.g. certain renderers could insert a space between sign and number.
Fix: Combine into a single template literal: `` {`${sign}${roundedImprovement}%`} ``

**[Medium] `tests/e2e/e16-s03-score-improvement.spec.ts` (confidence: 75)**
The E2E test uses deprecated `elementHandle()` API followed by `page.evaluate()`. Playwright recommends using locator-based evaluation.
Fix: Replace `elementHandle()` + `page.evaluate()` with `panel.evaluate(el => ...)` or locator-based assertions.

**[Medium] `src/lib/analytics.ts:164-169` (confidence: 72)**
`bestAttemptNumber` uses the original array index, creating an implicit coupling: the function assumes callers pass attempts in chronological order (which the store does via `loadAttempts`). If any future caller passes unsorted attempts, `bestAttemptNumber` would be wrong with no error signal.
Fix: Document the assumption explicitly in the JSDoc, or compute `bestAttemptNumber` from the sorted array.
