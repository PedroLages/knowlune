#### Nits

- **Nit** [supabase/migrations/20260413000001_p0_sync_foundation.sql:29-30] (confidence: 55) `[Maintainability]`: `vector` and `supabase_vault` extensions are created without `WITH SCHEMA extensions`, while `moddatetime` and `pgcrypto` on lines 27-28 use it. In Supabase projects, the `extensions` schema is the conventional home for all extensions (keeps `public` clean for user tables). On titan this likely already worked because the extensions were pre-installed during preflight and `CREATE EXTENSION IF NOT EXISTS` is a no-op. On a fresh database, `vector` would land in `public` and `supabase_vault` in its own `vault` schema by default. Not a correctness issue тАФ `pgvector` resolves fine from `public` тАФ but inconsistent with lines 27-28. Fix: add `WITH SCHEMA extensions` to both (or remove it from 27-28 and rely on Supabase defaults) for consistency. ~1 minute edit, non-blocking.

- **Nit** [supabase/migrations/20260413000001_p0_sync_foundation.sql:149-246] (confidence: 40) `[Security]`: Functions don't `SET search_path` (e.g., `SET search_path = public, pg_temp`). Flagging as advisory only тАФ pinning `search_path` is a cheap belt-and-suspenders if future overloads ever flip to `SECURITY DEFINER`. No action required for this story.
