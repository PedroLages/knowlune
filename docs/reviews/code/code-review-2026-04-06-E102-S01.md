## Story-Related Issues

### MEDIUM — AbsProgress field name mismatch (potential runtime bug)

**File:** `src/data/types.ts:807`, `src/app/hooks/useAudiobookshelfProgressSync.ts:103`

The `AbsProgress` type defines `updatedAt: number` but the ABS API returns the field as `lastUpdate` (per story dev notes lines 109 and the type's own comment: "ABS field name: lastUpdate"). Since `absApiFetch` does a raw `JSON.parse(text) as T`, the actual JSON property `lastUpdate` is not mapped to `updatedAt`. At runtime, `absProgress.updatedAt` would be `undefined`, causing `resolveConflict` to always return `'use-local'` (since `undefined > number` is `false`).

**Impact:** Conflict resolution would never adopt the ABS position. The fetch-on-open sync would always push local to ABS rather than pulling ABS when it's ahead.

**Fix:** Either:
1. Rename `updatedAt` to `lastUpdate` in `AbsProgress` type (and all consumers), or
2. Add a mapping step in `fetchProgress` to transform `lastUpdate` -> `updatedAt`.

**Note:** Unit tests pass because mocks use `updatedAt` directly, masking the mismatch.

### MEDIUM — 404 detection via string matching is fragile

**File:** `src/services/AudiobookshelfService.ts:213`

`fetchProgress` detects 404 by checking `result.error.includes('(404)')`, which depends on the error message format in `absApiFetch` line 71: `Server error (${response.status})`. If that message changes, 404 handling breaks silently (would return error instead of null).

**Impact:** Low — both functions are in the same file, but a refactor of error messages could break this.

**Fix:** Consider returning the status code in the error result (e.g., `{ ok: false, error: string, status?: number }`) or handle 404 inside `absApiFetch` with a special case.

### LOW — `flushSyncQueue` catch swallowed silently in `updateServer`

**File:** `src/stores/useAudiobookshelfStore.ts:79`

`.catch(() => {})` on `flushSyncQueue()` is correct for fire-and-forget, but could hide persistent failures. The eslint-disable comment is missing (though the `// silent-catch-ok` pattern is used elsewhere).
