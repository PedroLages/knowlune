The test was already broken by v36 on main but v37 adds further drift.

2. **Navigation unit tests broken** — `src/app/config/__tests__/navigation.test.ts` expects Library group to have 4 items but story added "Books" making it 5. Tests at lines 10-18 (Library group count), 31-42 (Track group items -- off by 1 in overflow count), and 70 (overflow count expects 13 but now 14) all fail.

### MEDIUM

3. **`updateBookStatus` catch block silently rolls back without user notification** — `src/stores/useBookStore.ts:81` catches DB update failures and silently reloads state from DB. While the rollback is good defensive coding, the user gets no feedback that their status change failed. Should add `toast.error()` or similar after rollback. (ESLint does not flag this because the comment satisfies the `silent-catch-ok` heuristic, but the comment says "Rollback on failure" not "silent-catch-ok".)

4. **`importBook` mutates the `book` parameter** — `src/stores/useBookStore.ts:58` reassigns `book = { ...book, source: ... }` inside the function body. While this creates a new object (spread), the `let` reassignment of a parameter is a minor code smell. More importantly, the book that gets `db.books.put()` at line 62 has the updated source, but if `storeBookFile` returns `'indexeddb'`, the original source is kept. This means for IndexedDB fallback, the `source` on the stored book may not accurately reflect where the file is stored.

### LOW

5. **`readBookFile` requires `bookId` parameter for fallback mode but the OPFS mode extracts it from path** — `src/services/OpfsStorageService.ts:79` has `bookId` as optional, but in fallback mode (line 83) it returns `null` if `bookId` is missing. Callers must always remember to pass `bookId` for the fallback case. Consider extracting `bookId` from the path in all cases, or making `bookId` required.

6. **Prettier formatting auto-fixed** — `src/app/routes.tsx` had a minor formatting issue (lazy import wrapping). Auto-fixed and committed.
