## Code Review: E11-S05 — Interleaved Review Mode

### What Works Well

1. **Solid algorithm design.** The greedy interleaving algorithm in `src/lib/interleave.ts` is clean, well-documented, and appropriate for the scale (O(n^2) is fine for 10-50 notes). The 60/40 urgency/dissimilarity weighting is a reasonable heuristic, and Jaccard similarity on tags is a pragmatic choice over vector embeddings.

2. **Good state machine architecture.** The `SessionPhase` union type in `InterleavedReview.tsx` creates a clear finite state machine (`loading -> single-course-prompt | reviewing | empty -> summary`). The `phaseInitialised` ref prevents re-initialization on store updates, which is a thoughtful guard against a subtle reactivity bug.

3. **Test infrastructure investment.** The new `review-factory.ts` and `seedReviewRecords` / `seedNotes` helpers in the shared test infrastructure are well-structured, use `FIXED_DATE` from `test-time.ts`, and will pay dividends for future stories in this epic.

#### High Priority

- **[src/app/pages/InterleavedReview.tsx:126] (confidence: 85)**: **Missing dependencies in phase-initialization effect.** The effect at line 99 references `startSession` (defined at line 128) and `isInterleavedActive`, but `startSession` is not in the dependency array. Because `startSession` is a plain function (not wrapped in `useCallback`), it's re-created every render, so the missing dep doesn't cause a stale closure *today* -- but the `phaseInitialised` ref guards against repeated execution anyway. However, `startInterleavedSession` (called inside `startSession`) IS used but not listed. ESLint `react-hooks/exhaustive-deps` would flag this. Why: If `startInterleavedSession` reference ever changes (e.g., store re-creates it), the effect won't pick up the new reference, potentially calling a stale version. Fix: Either (a) wrap `startSession` in `useCallback` and add it to the deps array, or (b) inline the logic directly in the effect. The `phaseInitialised` ref makes this low-risk but it's a correctness smell that will confuse future maintainers.

- **[src/stores/useReviewStore.ts:178-202] (confidence: 82)**: **`rateInterleavedNote` does not handle `rateNote` failure.** When `rateNote` throws (IndexedDB write failure), the `await` on line 188 will throw, but `rateInterleavedNote` has no try/catch. The calling code in `InterleavedReview.tsx:139` (`handleRate`) also has no error handling. This means an IndexedDB failure during an interleaved session will cause an unhandled rejection -- the card won't advance, the UI won't show an error, and the learner is stuck. Why: `rateNote` already has its own error handling (toast + rollback), but it re-throws implicitly when `persistWithRetry` fails because `rateNote` does a rollback `set()` but does NOT re-throw -- actually, looking more carefully, `rateNote` catches the error and does NOT re-throw, so `rateInterleavedNote` won't throw. However, after a failure, `rateNote` rolls back `allReviews` to the pre-update state, but `rateInterleavedNote` still increments `interleavedIndex` on line 199 regardless of whether the rating was persisted. This means the session advances past the card even though the rating was lost. Fix: Check the store's `error` state after `await rateNote()` and only advance `interleavedIndex` if the rating succeeded, or return a success boolean from `rateNote`.

- **[src/app/pages/InterleavedReview.tsx:69] (confidence: 78)**: **Frozen `now` means stale due-check for long sessions.** `const [now] = useState(() => new Date())` creates a fixed timestamp at mount time. Unlike `ReviewQueue.tsx` (which refreshes `now` every 60s via `setInterval`), `InterleavedReview` uses the mount-time value for the entire session. For the initial queue build this is fine (consistent snapshot). But if a learner leaves the tab open for 30+ minutes, the retention predictions shown on cards will be stale. Why: Learners doing long interleaved sessions see increasingly inaccurate retention percentages. Fix: Use a stable `now` for the initial queue build (current approach) but pass `new Date()` to `predictRetention` in `InterleavedCard` for display-time retention. Alternatively, add a 60s refresh like `ReviewQueue` does, and only use the frozen `now` for `startInterleavedSession`.

#### Medium

- **[src/stores/useReviewStore.ts:219-223] (confidence: 75)**: **Hardcoded 95% "after" retention is misleading.** `const avgAfter = total > 0 ? (reviewed * 95 + (total - reviewed) * avgBefore) / total : 95` assumes every reviewed card jumps to 95% retention. In reality, a "hard" rating on a card with 10% retention does not produce 95% retention -- it produces maybe 40-60% depending on the new interval. Why: The summary shows learners an inflated "retention improvement" number that doesn't reflect actual algorithm output. A learner who rates everything "hard" will see a big improvement number, which is misleading. Fix: After `rateNote` completes, re-compute actual retention using `predictRetention` on the updated record from `allReviews`, and store it alongside each rating for accurate summary calculation.

- **[src/app/components/figma/InterleavedCard.tsx:32-61] (confidence: 72)**: **Duplicated markdown-stripping logic.** `getPromptExcerpt` and `getFullExcerpt` share identical regex chains for stripping markdown. This is a DRY violation and a maintenance risk -- if the stripping logic needs updating (e.g., to handle `~~strikethrough~~`), both functions need synchronized changes. Fix: Extract a shared `stripMarkdown(content: string): string` function and have both `getPromptExcerpt` and `getFullExcerpt` call it.

- **[src/app/components/figma/InterleavedCard.tsx:179] (confidence: 70)**: **`aria-live="polite"` region is always in the DOM.** The back face's `aria-live` region is present from initial render with content already inside it. Screen readers announce changes to `aria-live` regions, but since the content is there from the start (just visually hidden via `backfaceVisibility`), the announcement may not fire on flip because the DOM content didn't change -- only CSS visibility changed. Why: The AC specifies "Answer revealed. Rate your recall." should be announced. Fix: Either (a) conditionally render the back face content only when `isFlipped` is true, or (b) use a separate visually-hidden `aria-live` region that gets text injected on flip (e.g., `{isFlipped && <span className="sr-only" aria-live="polite">Answer revealed. Rate your recall.</span>}`).

- **[src/app/components/figma/InterleavedSummary.tsx:157] (confidence: 70)**: **RatingBar uses inline `style` for width.** `style={{ width: \`${pct}%\` }}` is an inline style for a dynamic value. This is acceptable for truly dynamic percentages, but the eslint-disable comment at the top of `InterleavedCard.tsx` doesn't cover this file. The `react-best-practices/no-inline-styles` rule may flag this. Fix: This is a legitimate use of inline styles (dynamic percentage), but add a targeted eslint-disable comment on that line to document the rationale.

#### Nits

- **Nit** [src/app/pages/InterleavedReview.tsx:278] (confidence: 65): Progress calculation `(interleavedIndex / total) * 100` means the progress bar shows 0% when viewing card 1 and ~66% when viewing card 3 of 3. Consider `((interleavedIndex) / total) * 100` for "completed" semantics vs `(current / total) * 100` for "position" semantics. The current behavior is correct for "completed" but may confuse learners who expect the bar to fill completely when they're on the last card.

- **Nit** [src/app/pages/InterleavedReview.tsx:88] (confidence: 60): `.catch(console.error)` on the `Promise.all` for data loading is adequate logging but doesn't surface the error to the user. The phase stays stuck on `'loading'` forever if all three promises fail. Consider setting `phase` to an error state in the catch handler.

- **Nit** [src/lib/__tests__/interleave.test.ts:8] (confidence: 55): `crypto.randomUUID()` in the test factory generates non-deterministic IDs. While this works (tests don't assert on IDs from the factory), deterministic IDs (e.g., `note-${counter}`) would make test output more readable during debugging.

### Recommendations

1. **First**: Fix the `rateInterleavedNote` advancing past failed ratings (High -- data integrity issue).
2. **Second**: Address the `aria-live` announcement not firing on card flip (Medium -- accessibility AC).
3. **Third**: Extract shared `stripMarkdown` utility from duplicated regex chains (Medium -- maintainability).
4. **Fourth**: Consider actual retention recalculation instead of hardcoded 95% for summary (Medium -- accuracy).
5. **Fifth**: Add `useCallback` for `startSession` or suppress the exhaustive-deps warning with a comment explaining the `phaseInitialised` guard (High -- code hygiene).
