## Code Review: E20-S03 -- 365 Day Activity Heatmap

### What Works Well

1. **Clean separation of concerns**: The data aggregation logic in `src/lib/activityHeatmap.ts` is pure-functional, taking explicit parameters (`sessions`, `today`, `days`) with no side effects. This makes the unit tests straightforward and deterministic.

2. **Proper async cleanup pattern**: The `useEffect` in `ActivityHeatmap.tsx:46-66` uses the `ignore` flag pattern correctly, preventing state updates after unmount. The `study-log-updated` event listener is also properly cleaned up.

3. **Thorough unit test coverage**: Boundary values for `getActivityLevel`, empty inputs, orphaned sessions, and date-windowing are all tested. The test file has excellent coverage of the pure utility functions.

### Findings

#### Blockers

None.

#### High Priority

- **`src/app/components/reports/ActivityHeatmap.tsx:43` (confidence: 85)**: `today` is memoized with `useMemo(() => toLocalDateString(), [])` using an empty dependency array. If the user keeps the app open past midnight, `today` never updates, so the heatmap silently shows yesterday as "today" with the wrong ring highlight until they refresh. Why: Learners using Knowlune across a study session spanning midnight will see an incorrect "today" indicator. Fix: This is acceptable for a session-scoped component (page navigation would remount), but consider adding a dependency on a coarse-grained timestamp or documenting the limitation with a comment explaining the trade-off.

- **`src/app/components/reports/ActivityHeatmap.tsx:175-178` (confidence: 82)**: Inline `style={{...}}` object on the grid container. The dynamic `gridTemplateColumns` and `gridTemplateRows` values depend on `totalWeeks` which changes only when data changes, so the inline style is functionally required (Tailwind cannot generate dynamic grid column counts). However, a new style object is created every render, causing React to re-apply the style even when `totalWeeks` hasn't changed. Why: With 365+ cells each containing a Tooltip, unnecessary re-renders have measurable cost. Fix: Memoize the style object with `useMemo` keyed on `totalWeeks`:
  ```ts
  const gridStyle = useMemo(() => ({
    gridTemplateColumns: `auto repeat(${totalWeeks}, minmax(8px, 1fr))`,
    gridTemplateRows: 'auto repeat(7, 1fr)',
  }), [totalWeeks])
  ```

- **`src/app/components/reports/ActivityHeatmap.tsx:213-218` (confidence: 80)**: `new Date(day.date + 'T12:00:00')` and `d.toLocaleDateString('en-US', {...})` are called **inside the inner render loop** for every visible cell (up to 365 cells). Date construction and `toLocaleDateString` with options are surprisingly expensive -- each call creates an `Intl.DateTimeFormat` instance internally. Why: On lower-end devices, rendering 365 Date objects + 365 locale format calls on every state change adds latency to the heatmap paint. Fix: Either pre-compute `formattedDate` and `ariaLabel` inside `buildHeatmapGrid` (adding them to `HeatmapDay`), or memoize a `DateTimeFormat` instance outside the loop:
  ```ts
  const dateFormatter = useMemo(
    () => new Intl.DateTimeFormat('en-US', { weekday: 'short', month: 'short', day: 'numeric' }),
    []
  )
  ```

- **`src/lib/activityHeatmap.ts:162` (confidence: 78)**: `getMonthlyHeatmapSummary` has a comment "months come in insertion order since dates are sorted" but the `dayMap` parameter is a `Map<string, number>` whose iteration order depends on insertion order -- which is only guaranteed to be sorted if the caller passes a sorted map. `aggregateSessionsByDay` does insert in order (line 55-59), but nothing in the type signature enforces this contract. If a future caller builds a `dayMap` differently, months could appear out of order. Why: The table view would show months in wrong order, confusing learners. Fix: Sort explicitly at the end of `getMonthlyHeatmapSummary` by parsing the date key, or document the precondition clearly in the JSDoc.

#### Medium

- **`src/app/components/reports/ActivityHeatmap.tsx:70-80` (confidence: 75)**: The `study-log-updated` event handler re-fetches **all** study sessions via `db.studySessions.toArray()` on every event, with no debounce. If multiple lessons are completed in rapid succession (e.g., batch import or quick quiz completions), this triggers multiple full-table scans and re-aggregations. Fix: Debounce the handler (300-500ms) or use a flag to skip redundant fetches while one is in-flight.

- **`src/app/components/reports/ActivityHeatmap.tsx:50` (confidence: 75)**: `db.studySessions.toArray()` loads the entire study sessions table into memory. For power users with years of data (thousands of sessions), this could be significant. The aggregation only needs sessions within the 365-day window. Fix: Use a Dexie `.where('startTime').above(cutoffDate)` query to filter at the database level instead of loading everything.

- **`tests/e2e/regression/story-e20-s03.spec.ts:104` (confidence: 72)**: The AC2 tooltip test wraps the assertion in `if ((await cellWithActivity.count()) > 0)`. This means if the locator fails to find the cell (e.g., due to a date formatting change or timezone issue), the test silently passes without verifying anything. Why: A green test that tests nothing is worse than a failing test. Fix: Replace the `if` guard with `await expect(cellWithActivity).toBeVisible()` so the test fails clearly if the cell is missing.

- **[Recurring] `src/app/components/reports/ActivityHeatmap.tsx:175` (confidence: 70)**: Inline style object instead of Tailwind utility. This is partially a recurring pattern, though here it is justified since Tailwind cannot handle dynamic grid column counts. The ESLint `no-inline-styles` rule should have a suppression comment. Fix: Add `{/* eslint-disable-next-line react-best-practices/no-inline-styles -- dynamic grid requires runtime column count */}` or use a `className` with `style` separated.

- **`src/app/components/reports/ActivityHeatmap.tsx:182-192` (confidence: 72)**: Month label lookup uses `.find()` inside `Array.from({ length: totalWeeks })`, making it O(weeks * monthLabels) per render. With ~53 weeks and ~12 month labels this is trivial, but a `Map<number, string>` pre-built from `monthLabels` would be cleaner. Fix: Convert to a Map keyed by `colStart` before the render loop.

- **`src/lib/activityHeatmap.ts:150-168` (confidence: 70)**: `getMonthlyHeatmapSummary` uses `toLocaleDateString('en-US', { month: 'short', year: 'numeric' })` as the Map key (e.g., `"Mar 2026"`). This couples the data structure to locale-formatted strings. If the app ever supports i18n, the same month could produce different keys. Fix: Use `YYYY-MM` as the internal key and format the label separately for display.

#### Nits

- **Nit** `src/app/components/reports/ActivityHeatmap.tsx:266` (confidence: 65): Legend swatch `<div>` elements have `cursor-default` but are wrapped in `TooltipTrigger`. The cursor style is fine semantically but inconsistent with the interactive tooltip behavior -- a `cursor-help` would better signal that hovering reveals information.

- **Nit** `src/app/components/reports/ActivityHeatmap.tsx:229` (confidence: 60): Each heatmap cell has `tabIndex={0}`, meaning 365 tab stops. This is technically accessible but may be tedious for keyboard users. Consider using roving tabindex or arrow-key navigation within the grid (`role="grid"` with `role="gridcell"` children) for a better keyboard UX.

- **Nit** `src/lib/activityHeatmap.ts:105` (confidence: 60): `Array<null>(firstDayOfWeek).fill(null)` -- the generic `Array<null>` is redundant when `.fill(null)` already sets values. `new Array(firstDayOfWeek).fill(null)` reads more cleanly.

- **Nit** `src/app/components/reports/ActivityHeatmap.tsx:88` (confidence: 55): `getMonthlyHeatmapSummary(dayMap)` is memoized on `[dayMap]`, but `dayMap` is a `Map` reference. This works because `setDayMap` creates a new Map on each update, but it's fragile -- if the Map were mutated in place, the memo would miss it. Consider documenting this assumption.

### Recommendations

1. **Fix the silent-pass E2E test** (Medium, `story-e20-s03.spec.ts:104`) -- this is the highest-risk item because it masks regressions. Remove the `if` guard.
2. **Memoize the grid style object and pre-compute date formatting** (High) -- straightforward performance improvement for 365-cell rendering.
3. **Add Dexie query filter** (Medium) -- use `.where()` instead of `.toArray()` to avoid loading unbounded data.
4. **Debounce the event handler** (Medium) -- prevents redundant full-table scans on rapid events.
5. **Sort months explicitly** in `getMonthlyHeatmapSummary` (High) -- defensive correctness that prevents future bugs.
