### Finding

#### Medium Priority

- **`src/lib/googleDriveToken.ts:12-14` (confidence: 85) [Correctness]**: JSDoc on `getDriveToken()` recommends the wrong recovery path for 401 errors.

  The JSDoc states:
  > On 401, the caller should re-invoke this function after handling the error: it refreshes the Supabase session transparently and returns the new token.

  This is misleading. `getDriveToken()` returns `session.provider_token` when it is truthy (line 24-25), regardless of whether the token has expired. After a Drive API 401 response, the `provider_token` is still present in the session object — it is simply expired. Re-invoking `getDriveToken()` will return the same expired token, not a fresh one.

  The caller must use `refreshDriveToken()` instead, which always calls `supabase.auth.refreshSession()` to obtain a new token from Supabase's server-side token exchange.

  **Impact on developers**: Following the documented guidance literally creates an infinite failure loop: call `getDriveToken()` -> get expired token -> retry Drive API -> 401 -> call `getDriveToken()` again -> same expired token -> repeat. This is a subtle bug that only manifests after the first hour of use (Google OAuth token lifetime).

  **Fix**: Update the JSDoc on `getDriveToken()` lines 12-14:
  ```typescript
  /**
   * On 401, the caller should call `refreshDriveToken()` to force a token
   * refresh, then retry the Drive API call with the new token.
   * (getDriveToken() only refreshes when provider_token is missing entirely.)
   */
  ```

  **Effort**: ~1 minute, one-line JSDoc edit.

### Recommendations

1. **Fix the JSDoc on `getDriveToken()`** — the one actionable finding. This is a 1-minute documentation fix that prevents a confusing failure mode during Drive API integration.
