## Code Review: E19-S02 — Stripe Subscription Integration

### What Works Well

1. **Solid webhook architecture.** The `stripe-webhook/index.ts` correctly verifies Stripe signatures via `constructEventAsync`, uses a service-role Supabase client for writes, and returns 200 for all handled events (preventing spurious retries). The metadata propagation pattern (copying `supabase_user_id` into subscription metadata during `checkout.session.completed`) is the right way to ensure subsequent lifecycle events can identify the user.

2. **Clean async cancellation pattern.** Both `useEffect` hooks in `SubscriptionCard.tsx` use `cancelled` flags to prevent state updates after unmount. This avoids the classic React async-state-on-unmounted-component problem.

3. **Well-designed Supabase migration.** The `001_entitlements.sql` migration includes RLS policies, a trigger for auto-creating free entitlement rows on sign-up, and proper `ON DELETE CASCADE`. The separation of write path (service-role webhook) from read path (user JWT with RLS) is correct.

### Findings

#### Blockers

- **[index.html:25-26] (confidence: 95) — CSP missing Stripe and Supabase domains.** The `connect-src` directive does not include `https://*.supabase.co` (or the self-hosted Supabase URL) or `https://api.stripe.com`. The `frame-src` directive is missing `https://checkout.stripe.com`. The story file mentions CSP was updated ("CSP configuration for Stripe" in lessons learned), but the actual `index.html` diff shows no Stripe or Supabase domains were added to any CSP directive. The `supabase.functions.invoke()` call in `checkout.ts:22` will be silently blocked by the browser's CSP in production. Why: Learners clicking "Upgrade to Premium" will see a generic error toast — no checkout possible. Fix: Add to `connect-src`: the Supabase functions URL (e.g., `https://*.supabase.co` or `http://localhost:*` already covers dev); add `https://checkout.stripe.com` to `frame-src` (Stripe Checkout uses an embedded redirect). Verify with the actual deploy URL. If using self-hosted Supabase on `titan.local`, the existing wildcard `http://titan.local:*` covers it for dev — but production needs `https://` entries.

- **[supabase/functions/create-checkout/index.ts:17] (confidence: 92) — CORS `Access-Control-Allow-Origin: *` wildcard on checkout endpoint.** The `create-checkout` edge function sets `'Access-Control-Allow-Origin': '*'`, allowing any website to create checkout sessions on behalf of authenticated users. An attacker with a stolen JWT (e.g., XSS on any page) could call this endpoint from their domain to initiate charges. Why: This is a financial endpoint — CORS should restrict to the known app origin. Fix: Replace `'*'` with the actual app origin:
  ```typescript
  const allowedOrigin = Deno.env.get('APP_URL') || 'http://localhost:5173'
  const corsHeaders = {
    'Access-Control-Allow-Origin': allowedOrigin,
    'Access-Control-Allow-Headers': 'authorization, x-client-info, apikey, content-type',
  }
  ```

#### High Priority

- **[supabase/functions/create-checkout/index.ts:52-66] (confidence: 88) — Customer lookup by email enables cross-account hijacking.** `stripe.customers.list({ email: user.email })` finds existing customers by email. If two Supabase users share the same email (possible in some auth configs), or if a user changes their email, the checkout session could be attached to a stale/wrong Stripe customer. Worse, the `metadata.supabase_user_id` on the existing customer is not checked — so subscription events may route entitlements to the wrong user. Why: A learner could end up paying for another account's premium. Fix: Prefer looking up/creating the Stripe customer by `metadata.supabase_user_id` rather than email. If a customer with that metadata exists, reuse it; otherwise create a new one. Email should be informational, not the primary key.

- **[supabase/functions/stripe-webhook/index.ts:68] (confidence: 85) — No HTTP method check on webhook endpoint.** The handler processes every HTTP method (GET, PUT, DELETE, etc.) identically. While Stripe only sends POSTs, the endpoint is publicly accessible and should reject non-POST requests explicitly. Why: Reduces attack surface. Automated scanners/crawlers sending GET requests will trigger the full handler, including `req.text()` parsing. Fix: Add at the top of the handler:
  ```typescript
  if (req.method !== 'POST') {
    return new Response('Method not allowed', { status: 405 })
  }
  ```

- **[supabase/functions/create-checkout/index.ts:21] (confidence: 82) — No HTTP method check on create-checkout.** Same issue: the handler only checks for OPTIONS (CORS preflight) but processes GET, DELETE, etc. as checkout creation attempts. Fix: Add `if (req.method !== 'POST' && req.method !== 'OPTIONS')` guard returning 405.

- **[src/app/components/settings/SubscriptionCard.tsx:96-108] (confidence: 85) — `isCheckoutLoading` never resets on redirect failure.** If `window.location.href = result.url` fails (e.g., popup blocker, CSP blocks the redirect, or the URL is malformed), the button stays in a permanent loading/disabled state. The user has no way to retry without refreshing. Why: A learner's upgrade button becomes permanently broken. Fix: Use a try/catch around the redirect, or use `window.location.assign()` which throws on invalid URLs. Also consider a timeout fallback:
  ```typescript
  window.location.href = result.url
  // If still here after 5s, redirect likely failed
  setTimeout(() => setIsCheckoutLoading(false), 5000)
  ```

- **[src/lib/checkout.ts:80-82] (confidence: 80) [Recurring] — Empty catch block silently swallows polling errors.** The `catch {}` block in `pollEntitlement` silently ignores all errors during polling, including auth failures (expired JWT), Supabase client errors, and network errors. The learner sees no feedback while the poll silently fails on every iteration. Why: After paying, a learner could stare at "Activating..." for 10 seconds with no explanation, then get a vague timeout message. Fix: At minimum, log the error:
  ```typescript
  } catch (err) {
    console.warn('Entitlement poll attempt failed:', err)
  }
  ```

#### Medium

- **[src/app/components/settings/SubscriptionCard.tsx:213] (confidence: 75) — Premium badge uses hardcoded `bg-brand` instead of design token variant.** Line 213 uses `className="bg-brand text-brand-foreground border-transparent"` on a `Badge`. The design guidance (story file line 137) specifies `bg-gold-muted text-gold` for the premium status indicator. Using brand-blue for the premium badge conflicts with the gold premium theme used everywhere else in the card. Fix: Use `className="bg-gold-muted text-gold border-transparent"` to match the card's gold premium accent.

- **[src/app/components/settings/SubscriptionCard.tsx:60] (confidence: 72) — `toastError.saveFailed` is semantically wrong for checkout cancellation.** "Save failed" is a CRUD operation toast. For checkout cancellation, the AC specifies: "I see a message that the upgrade was not completed." Using `saveFailed` may show a misleading prefix like "Save failed: Upgrade not completed." Fix: Use a generic toast error or a custom toast method if available:
  ```typescript
  toastError('Upgrade not completed')
  ```

- **[src/app/pages/Settings.tsx:204] (confidence: 78) — Unsafe `as` cast on URL parameter.** `checkoutParamRef.current as 'success' | 'cancel' | null` blindly trusts the URL query string. If a user navigates to `/settings?checkout=malicious`, the value passes through to `SubscriptionCard` as `checkoutStatus`. While the component only acts on `'success'` and `'cancel'`, the type assertion hides that a third value is possible. Fix: Validate explicitly:
  ```typescript
  const raw = checkoutParamRef.current
  const status = raw === 'success' || raw === 'cancel' ? raw : null
  ```

- **[supabase/functions/stripe-webhook/index.ts:10,15] (confidence: 75) — Non-null assertions on env vars crash silently.** `Deno.env.get('STRIPE_SECRET_KEY')!` and `STRIPE_WEBHOOK_SECRET` use `!` without validation. If either env var is missing, the Stripe constructor or webhook verification will throw an unhelpful runtime error deep in the Stripe SDK. Fix: Add startup validation:
  ```typescript
  const stripeKey = Deno.env.get('STRIPE_SECRET_KEY')
  if (!stripeKey) throw new Error('STRIPE_SECRET_KEY is not configured')
  ```

- **[src/lib/checkout.ts:95-101] (confidence: 70) [Recurring] — `cacheEntitlement` silently swallows IndexedDB write errors.** The function catches and logs but never surfaces IndexedDB write failures. If the local cache consistently fails (e.g., storage quota exceeded), the user will re-fetch from Supabase on every mount — a degraded experience with no visibility. Fix: Consider returning the error so callers can decide whether to surface it, or at minimum surface a non-blocking toast.

#### Nits

- **Nit** [src/app/components/settings/SubscriptionCard.tsx:189] (confidence: 60): `flex-shrink-0` is the old Tailwind class. v4 prefers `shrink-0`. Both work, but codebase should be consistent. Same at line 219.

- **Nit** [supabase/functions/create-checkout/index.ts:69] (confidence: 65): `req.json().catch(() => ({}))` silently ignores malformed JSON. This is fine for the `origin` field (has fallback), but if more required fields are added later, this pattern will mask bugs. Consider logging when JSON parsing fails.

- **Nit** [src/app/components/settings/SubscriptionCard.tsx:214] (confidence: 55): The "Monthly" text on line 214 is hardcoded. If Knowlune ever adds annual plans, this will be wrong. Consider deriving from `entitlement.planId` or at least adding a TODO comment.

### Recommendations

1. **Fix CSP first** (Blocker) — without Stripe/Supabase domains in CSP, checkout cannot work at all in production.
2. **Restrict CORS origin** (Blocker) — wildcard on a payment endpoint is a security issue regardless of JWT protection.
3. **Add method guards** to both Edge Functions (High) — trivial change, meaningful security hardening.
4. **Fix customer lookup** to use Supabase user ID metadata instead of email (High) — prevents cross-account issues.
5. **Add redirect failure fallback** for the upgrade button (High) — prevents permanent loading state.
6. **Add logging to empty catch** in polling (High) — aids debugging subscription activation issues.
7. **Fix Premium badge color** to use gold tokens per design guidance (Medium).
8. **Validate checkout URL parameter** instead of casting (Medium).
