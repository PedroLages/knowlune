## Code Review: E19-S02 -- Stripe Subscription Integration

**Review pass: 2 (post-fix verification)**

### What Works Well

1. **Webhook signature verification is correctly implemented.** The `stripe-webhook` handler uses `constructEventAsync` with a dedicated `WEBHOOK_SECRET`, and the signature check is the first operation after the method guard. This is the correct pattern for Stripe webhook security.

2. **Event replay protection is well-designed.** The `upsertEntitlement` function compares `event.created` timestamps against `updated_at` in the database, preventing out-of-order webhook delivery from corrupting entitlement state. The comparison uses ISO 8601 strings which sort lexicographically.

3. **Client-side architecture is clean.** The separation between `checkout.ts` (service layer), `SubscriptionCard.tsx` (UI), and `Settings.tsx` (routing integration) follows the project's established patterns. The polling loop with `AbortSignal` support and proper cleanup on unmount demonstrates careful async lifecycle management.

4. **Prior review findings addressed thoroughly.** CORS is restricted to `APP_URL`, method guards are on both Edge Functions, customer lookup uses metadata instead of email, env var validation is present for Stripe keys, and the `catch` blocks in polling now log warnings.

#### High Priority

- **[Recurring] [supabase/functions/create-checkout/index.ts:47-48] (confidence: 88)**: Same non-null assertion issue as the webhook handler. `SUPABASE_URL` and `SUPABASE_ANON_KEY` on lines 47-48 use `!` without validation, while `STRIPE_SECRET_KEY` and `STRIPE_PRICE_ID` on lines 26-29 are properly guarded. Pattern from: E19-S02 memory. Fix: Add the same `if (!var) throw new Error(...)` pattern for consistency.

- **[supabase/functions/stripe-webhook/index.ts:170-172, 197-199] (confidence: 85)**: When `customer.subscription.updated` or `customer.subscription.deleted` events cannot resolve a user ID, the handler `break`s and returns HTTP 200 to Stripe. This means Stripe considers the event delivered and will not retry. If the user ID resolution fails due to a transient database error (not just "user doesn't exist"), the entitlement update is permanently lost. Why: A learner who paid could remain on the free tier permanently if a DB blip causes the user lookup to fail during a `subscription.updated` event. Fix: Have `resolveUserId` propagate DB errors (throw) rather than returning `undefined` for both "not found" and "query failed." Then in the switch cases, catch the error and return 500 (triggering Stripe retry) for transient failures, while logging and returning 200 for genuinely missing users.

- **[src/lib/checkout.ts:87] (confidence: 82)**: The `setTimeout` in the polling loop does not respect the `AbortSignal`. If the signal fires during the sleep, the loop continues waiting for the full `intervalMs` before checking `signal.aborted` on the next iteration. This means unmount cleanup could be delayed by up to 2 seconds, and the subsequent poll request fires after the component has unmounted. Why: After paying and quickly navigating away, the polling continues running, wasting network requests and risking state updates on an unmounted component. Fix: Use a signal-aware sleep:
  ```typescript
  await new Promise<void>((resolve) => {
    const timer = setTimeout(resolve, intervalMs)
    signal?.addEventListener('abort', () => { clearTimeout(timer); resolve() }, { once: true })
  })
  ```

- **[src/app/components/settings/SubscriptionCard.tsx:116] (confidence: 80)**: `window.location.href = result.url` redirects the user to whatever URL the Edge Function returns. While the server validates the origin for success/cancel URLs, the `session.url` itself comes from Stripe and is not validated client-side. If the Edge Function were compromised or the response intercepted (MITM on a misconfigured dev setup), this becomes an open redirect. Why: For a payment flow, any redirect vulnerability is high-severity -- a phishing page could capture credentials. Fix: Validate that `result.url` starts with `https://checkout.stripe.com/` before redirecting:
  ```typescript
  if (!result.url.startsWith('https://checkout.stripe.com/')) {
    toastError.saveFailed('Invalid checkout URL received.')
    setIsCheckoutLoading(false)
    return
  }
  ```

#### Medium

- **[src/app/components/settings/SubscriptionCard.tsx:34-55 vs 58-99] (confidence: 75)**: Race condition between the two `useEffect` hooks. The mount-time effect (line 34) calls `getCachedEntitlement` which resolves and sets state to `'free'`. The checkout success effect (line 58) sets state to `'activating'`. Depending on microtask ordering, the mount effect could overwrite `'activating'` back to `'free'`. The test at line 200-202 documents this by making `getCachedEntitlement` never resolve -- proving the race exists. Fix: Add an early return in the mount effect when `checkoutStatus === 'success'` to avoid the race:
  ```typescript
  useEffect(() => {
    if (!user || checkoutStatus === 'success') return
    // ... rest of mount effect
  }, [user, checkoutStatus])
  ```

- **[src/app/components/settings/SubscriptionCard.tsx:118] (confidence: 72)**: The fallback `setTimeout(() => setIsCheckoutLoading(false), 5000)` is not cleared on unmount or in the `finally` block. If `handleUpgrade` completes with an error (early return on line 111), the timeout from a previous call may still be pending. The `finally` block on line 119 resets `checkoutInProgress` but does not clear this timer. Fix: Store the timer ID in a ref and clear it in `finally` and on unmount.

- **[supabase/functions/create-checkout/index.ts:69] (confidence: 72)**: The `ALLOWED_ORIGINS` array uses strict `includes()` comparison. If `APP_URL` is set to `https://knowlune.com` but the actual request origin is `https://www.knowlune.com` (or has a trailing slash), the check fails and checkout is blocked with "Invalid origin." Fix: Document in `.env.example` that `APP_URL` must exactly match the deployed domain, or normalize both values (strip trailing slashes, etc.) before comparing.

- **[src/app/components/settings/SubscriptionCard.tsx:239] (confidence: 68)**: "Monthly" text is hardcoded. If Knowlune adds annual plans, this label will be wrong for annual subscribers. Fix: Derive from `entitlement.planId` or add a `// TODO(E19-S03)` comment acknowledging this is placeholder.

### Recommendations

Ordered by impact:

1. **Fix the timestamp mismatch in replay protection** (Blocker) -- use `eventTime` for `updated_at` to ensure consistent comparison. Single line change, highest correctness impact.
2. **Add env var validation for Supabase URL/keys** (Blocker + High) -- four lines across two files, prevents cryptic deployment failures.
3. **Fix user ID resolution error handling** (High) -- distinguish "not found" from "DB error" to prevent permanent entitlement loss on transient failures.
4. **Add signal-aware sleep to polling loop** (High) -- prevents post-unmount network requests and state updates.
5. **Validate checkout URL before redirect** (High) -- defense-in-depth for the payment redirect.
6. **Address the effect race condition** (Medium) -- the test workaround proves the issue exists; fix it properly.
7. **Clear fallback setTimeout** (Medium) -- minor leak but easy fix.
