### Findings

#### High Priority

- **[Correctness] `src/app/hooks/useAudioPlayer.ts:264-271` + `src/app/hooks/useSleepTimer.ts:97-108` (confidence: 88)**: Single-file M4B: repeated `chapterend` dispatch during fade-out. When `evt.defaultPrevented` is true at line 271, the `break` exits the for-loop but `setCurrentChapterIndex(i)` is skipped. On the next 500ms polling tick, `audio.paused` is still false (audio is fading, not paused), and `i !== currentIdx` is still true, so `chapterend` fires again. This repeats ~10 times during the 5-second fade, starting 10 competing `fadeOutAndPause` rAF loops and calling `onPause()` 10 times.

  **Why it matters**: Multiple competing rAF loops manipulate `audio.volume` simultaneously, causing erratic volume behavior during the fade. Multiple `onPause()` calls record incorrect `pausedAtRef` timestamps, potentially triggering an unwanted 5s auto-rewind on next play.

  **Fix** (~5 min): Update `setCurrentChapterIndex(i)` to happen regardless of `defaultPrevented` in the single-file polling path. The chapter index should advance even when sleep timer intercepts â€” it only blocks auto-advance to the *next file*, not the index update. Alternatively, add a guard in `handleChapterEnd` (e.g., a `fadingRef` flag) to ignore duplicate calls:

  ```typescript
  // In useAudioPlayer.ts single-file chapter tracking (line 271):
  audio.dispatchEvent(evt)
  // Always update the chapter index â€” preventDefault only blocks auto-advance
  setCurrentChapterIndex(i)
  if (evt.defaultPrevented) break
  ```

  Or in `useSleepTimer.ts`:
  ```typescript

  const handleChapterEnd = (e: Event) => {
    if (fading) return
  ```

#### Medium

- **[Correctness] `src/app/hooks/useSleepTimer.ts:21-37` (confidence: 72)**: `fadeOutAndPause` has no cancellation mechanism. If the user cancels the timer during the 5-second fade (via "Off"), the rAF loop continues to completion and eventually calls `onPause()`, pausing audio the user intended to keep playing. The `eocCleanupRef` removes the event listener but cannot stop an in-progress fade.

  **Fix** (~10 min): Return a cancel function from `fadeOutAndPause` that uses `cancelAnimationFrame` and restores volume. Store it in a ref and call it from `cancelTimer()`. Low real-world risk (user is presumably asleep during EOC fade), but worth hardening:
