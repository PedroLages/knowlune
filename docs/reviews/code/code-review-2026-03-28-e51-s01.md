# Code Review: E51-S01 — Settings Infrastructure & Display Section Shell

## What Works Well

1. **Defensive localStorage validation.** The `VALID_CONTENT_DENSITY` / `VALID_REDUCE_MOTION` arrays with fallback-to-default logic in `getSettings()` (settings.ts:68-77) is a solid pattern that prevents corrupted enum values from propagating through the app. This should be the template for all future enum-like settings.

2. **Consistent section pattern.** `DisplayAccessibilitySection.tsx` follows the established AgeRangeSection Card/CardHeader/CardContent pattern precisely -- same icon badge styling, header background, content padding. This makes the Settings page feel cohesive and makes future sections predictable to build.

3. **Clean component API.** The `settings` + `onSettingsChange` prop contract is simple, testable, and keeps persistence logic in the parent. The component owns its own toast for the reset action (not the generic handler), which is the right separation.

## Findings

### Blockers

None.

### High Priority

- **[src/lib/__tests__/settings.test.ts] (confidence: 90)**: No unit tests for the new validation/sanitization logic. The `getSettings()` function now sanitizes corrupted localStorage values (lines 69-77 of settings.ts), but no test verifies this behavior. Missing test cases: `reduceMotion: 'invalid'` falls back to `'system'`; `contentDensity: 'garbage'` falls back to `'default'`; `accessibilityFont: 'yes'` (string) falls back to `false`. This is a HIGH finding because the sanitization logic was explicitly added as an edge case fix (story task 2.6) -- untested defensive code is false confidence. **Fix:** Add a `describe('getSettings validation')` block with tests for each corrupted field value.

- **[tests/e2e/story-e51-s01-settings-infrastructure.spec.ts:661-686] (confidence: 85)**: AC4 test ("fresh app returns correct defaults") does not verify any default values. The test navigates to settings, confirms the section is visible, then ends. AC4 states: "accessibilityFont is false, contentDensity is 'default', and reduceMotion is 'system'" -- none of these are asserted. The test comment on line 683 acknowledges the gap ("verify via UI state instead") but then only checks visibility. **Fix:** Either evaluate `getSettings()` in the browser context (via `page.evaluate`) or verify UI state -- e.g., the Switch for "Accessibility Font" is unchecked, the Motion Preference shows "Follow system".

- **[src/app/components/settings/DisplayAccessibilitySection.tsx:129] (confidence: 80)**: Reset `AlertDialogAction` uses default button variant instead of brand variant. Story design guidance (line 84) specifies "Cancel (outline) + Reset (brand variant)". The current code renders `<AlertDialogAction onClick={handleReset}>Reset</AlertDialogAction>` which gets `buttonVariants()` default (dark/primary style), not brand. While visually similar in light mode, this diverges from the design spec and from other destructive action patterns in the app. **Fix:** Add `className={buttonVariants({ variant: 'brand' })}` to `AlertDialogAction`, or restructure as `<AlertDialogAction asChild><Button variant="brand">Reset</Button></AlertDialogAction>`.

### Medium

- **[src/app/pages/Settings.tsx:919-923] (confidence: 75)**: `onSettingsChange` handler captures `settings` from render closure. Currently safe because only the reset button calls it, but when E51-S02/S03/S04 wire up the live toggles (Switch onChange), rapid toggling could read stale `settings`. **Fix:** Use functional state update pattern: `setSettings(prev => { const updated = { ...prev, ...updates }; saveSettings(updated); return updated })`. This prevents stale closure bugs in future stories and is a one-line change now.

- **[src/app/components/settings/DisplayAccessibilitySection.tsx:43] (confidence: 72)** `[Recurring]`: Icon uses `text-brand` on `bg-brand-soft` background. The styling rules specify `text-brand-soft-foreground` for content on `bg-brand-soft` backgrounds for WCAG AA compliance in dark mode. However, this matches the existing AgeRangeSection pattern at Settings.tsx:100, so this is a pre-existing inconsistency, not a regression. Confidence reduced because it's an icon (decorative), not body text. **Fix:** For consistency, address this project-wide when tackling the `text-brand` on `bg-brand-soft` pattern across all Settings sections.

- **[src/lib/__tests__/settings.test.ts:18-26, 30-40, 86-101] (confidence: 70)**: Three `toEqual` assertions still use exact object shapes (the lessons learned section even calls this out). When future stories add more fields to `AppSettings`, these tests will break again. The new fields were added to fix the breakage, but the pattern persists. **Fix:** Migrate to `expect(settings).toMatchObject({ displayName: 'Student' })` or field-level `toHaveProperty` assertions. This was identified in the story's own lessons learned but not acted on.

### Nits

- **Nit** [src/app/components/settings/DisplayAccessibilitySection.tsx:96] (confidence: 60): The `capitalize` class on the Motion Preference span is misleading -- the display text is already manually formatted (`'Follow system'`, `'Reduced'`, `'All motion'`), so `capitalize` has no visible effect on these strings but could cause issues if the logic changes. Remove it for clarity.

- **Nit** [src/app/components/settings/DisplayAccessibilitySection.tsx:57,75] (confidence: 55): Subsection labels use `<p className="text-sm font-medium">` instead of `<Label>` from `@/app/components/ui/label`. While functionally equivalent, using `<Label>` with proper `htmlFor` would associate labels with the Switch controls and improve screen reader experience when the toggles are enabled in S02-S04.

## Recommendations

1. **First**: Add unit tests for the validation/sanitization logic in `getSettings()` -- this is the highest-value gap since it's defensive code that currently has zero test coverage.
2. **Second**: Fix AC4 E2E test to actually verify default values, not just section visibility.
3. **Third**: Add brand variant to the Reset AlertDialogAction to match design spec.
4. **Fourth**: Switch `onSettingsChange` to functional state update pattern before S02-S04 wire up live toggles.
5. **Optional**: Address `toEqual` brittleness and label semantics as part of ongoing cleanup.
