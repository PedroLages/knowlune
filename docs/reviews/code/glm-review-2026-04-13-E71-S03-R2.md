### Findings

#### Blockers

- **`src/stores/useKnowledgeMapStore.ts:378-385` (confidence: 90)**: **Fabricated lesson data could pollute `SuggestedActionsPanel` with misleading actions.** The store creates fake lesson objects keyed by `courseId` instead of real lesson IDs. This means a topic linked to one course could get a `lesson-rewatch` suggestion pointing to a nonsensical URL (`/courses/courseId/lessons/courseId`), or a `flashcard-review` / `quiz-refresh` might be suppressed because a fabricated lesson with `completionPct = scoreResult.score` is treated as a valid activity. This is a silent correctness bug — the UI surfaces actions derived from synthetic data rather than real user state.
  Fix: Filter out fabricated lessons entirely. Only include real lesson data (which is unavailable until E56). Replace the mapping with `lessons: []` and add a comment referencing the TODO already present.

#### High Priority

- **`src/stores/useKnowledgeMapStore.ts:388` (confidence: 85)**: **`fsrsStability` map is never passed to `generateActionSuggestions`.** The function accepts an options bag with `fsrsStability: Map<string, number>`, but the store call provides no options (`generateActionSuggestions(topicsWithScores)`). FSRS stability data (from `predictRetention`) is available in the scored topics' `scoreResult` but is never forwarded, so decay urgency always falls back to the less accurate recency heuristic even when FSRS data exists. This silently produces suboptimal ranking. The unit test confirms the function works correctly when `fsrsStability` is supplied, but the integration never supplies it.
  Fix: Build the `fsrsStability` map from scored topics that have FSRS data and pass it as `{ fsrsStability }` to `generateActionSuggestions`.

- **`src/stores/useKnowledgeMapStore.ts:381` (confidence: 75)**: **Stale FSRS data not included in suggestions — `scoreResult` may contain FSRS stability but it's not extracted.** If `scoreResult` contains stability information (as computed in earlier steps), the store should extract and forward it. Without this, the entire FSRS integration tested in `actionSuggestions.test.ts` (AC 8) is dead code in production.
  Fix: Extract FSRS stability from scored topics where available:
  ```ts
  const fsrsStability = new Map<string, number>()
  for (const t of scoredTopics) {
    if (t.scoreResult.stability != null) {
      fsrsStability.set(t.canonicalName, t.scoreResult.stability)
    }
  }
  const suggestions = generateActionSuggestions(topicsWithScores, { fsrsStability })
  ```

#### Medium

- **`tests/e2e/story-e71-s03.spec.ts:142-156` (confidence: 70)**: **Responsive layout test doesn't verify dual-render guard.** The test visits `/knowledge-map` twice (desktop then mobile) but never asserts that exactly **one** panel instance is rendered on each viewport. Because the component is conditionally rendered (`{isMobile && <SuggestedActionsPanel>}` and `{!isMobile && <SuggestedActionsPanel>}`), a bug causing both or neither to render on a given viewport would pass this test. The test checks `toBeVisible()` but not count.
  Fix: Add `await expect(page.getByTestId('suggested-actions-panel')).toHaveCount(1)` after each viewport assertion to ensure exactly one instance renders.

#### Nits

- **`src/stores/useKnowledgeMapStore.ts:426-428` (confidence: 60)**: `getSuggestedActions()` is documented as "backward compat" and simply returns `get().suggestions`, but no call site in the diff uses it. If no external consumer exists, this is dead code that adds maintenance surface. Minor — not a correctness issue.
