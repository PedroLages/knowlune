#### Medium

- **[src/lib/actionSuggestions.ts:195](https://github.com/knowlune/knowlune/blob/HEAD/src/lib/actionSuggestions.ts#L195) (confidence: 80)**: **Deduplication ignores equal-priority actions from different topics that happen to share a `canonicalName`.** If two distinct topic objects (e.g., from different courses) share the same `canonicalName`, the current logic silently drops one of them based on iteration order — even if they represent genuinely different learning content (different lessons, different flashcard decks). Since `canonicalName` is the only dedup key, any collision causes silent data loss.
  **Fix**: If topic uniqueness is guaranteed by `canonicalName` (i.e., the input is pre-aggregated), document that assumption explicitly on the `topics` parameter. If not, dedup on a compound key (e.g., `canonicalName` + `actionType`) or add a comment explaining why `canonicalName` alone is sufficient.

- **[src/lib/actionSuggestions.ts:120](https://github.com/knowlune/knowlune/blob/HEAD/src/lib/actionSuggestions.ts#L120) (confidence: 70)**: **`generateLessonSuggestion` assumes a non-empty `lessons` array for `reduce`, but if an element has `completionPct` as `NaN` or `undefined`, the comparison silently breaks.** While the empty-array guard exists, the `reduce` initializer reuses the first element without type-checking `completionPct`. If a lesson has `completionPct: undefined`, the `<` comparison evaluates to `false`, and that lesson is treated as having 100% completion — meaning the *wrong* lesson could be selected.
  **Fix**: Add a defensive fallback in the reduce comparator, e.g., `(lesson.completionPct ?? 100) < (lowest.completionPct ?? 100)`, or validate `completionPct` is a number in `[0, 100]`.

- **[src/lib/actionSuggestions.ts:99](https://github.com/knowlune/knowlune/blob/HEAD/src/lib/actionSuggestions.ts#L99) (confidence: 60)**: **`recencyDecayFactor` clamps output to `[0, 100]`, but `fsrsDecayFactor` only clamps the lower bound to `0`.** For negative `stability` inputs (which shouldn't happen in FSRS but aren't guarded against), `100 - stability * 2` would produce values > 100, propagating into the urgency formula and potentially inflating scores above intended bounds (though `calculateUrgencyScore` does clamp `[0, 100]`). The asymmetry is a latent correctness risk.
  **Fix**: Add `Math.min(100, ...)` to `fsrsDecayFactor` for symmetry, or validate inputs are non-negative.

#### Nits

- **[src/lib/actionSuggestions.ts:226](https://github.com/knowlune/knowlune/blob/HEAD/src/lib/actionSuggestions.ts#L226) (confidence: 50)**: `.slice(0, maxSuggestions)` truncates after sorting, which means ties in `urgencyScore` have non-deterministic ordering between runs (sort stability depends on engine). Not a bug today, but could cause flaky UI tests if screenshot-based.
  **Fix**: Add a secondary sort key (e.g., `canonicalName` alphabetically) to ensure deterministic ordering: `.sort((a, b) => b.urgencyScore - a.urgencyScore || a.canonicalName.localeCompare(b.canonicalName))`.

- **[src/lib/actionSuggestions.ts:110](https://github.com/knowlune/knowlune/blob/HEAD/src/lib/actionSuggestions.ts#L110) (confidence: 40)**: `actionRoute` for flashcard/quiz suggestions uses `encodeURIComponent(topic.canonicalName)`, but the lesson route interpolates `courseId` and `lessonId` raw. If these IDs ever contain special characters (unlikely with UUIDs, but not validated), the URL could be malformed.
  **Fix**: Apply `encodeURIComponent` to `courseId` and `lessonId` in the lesson route for consistency.
