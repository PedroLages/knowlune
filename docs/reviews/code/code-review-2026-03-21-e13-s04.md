#### High Priority

- **`src/app/pages/QuizResults.tsx:71-77` (confidence: 90)**: `handleRetake` silently swallows errors. The catch block logs to console but never surfaces feedback to the learner. If `retakeQuiz` fails (e.g., Dexie read error fetching the quiz), the user clicks "Retake Quiz", nothing happens, and there is no indication why. Why: A learner who clicks "Retake Quiz" and gets no response will think the app is broken. Fix: Add `toast.error('Could not start retake. Please try again.')` in the catch block, matching the existing error toast pattern used in `loadAttempts` on line 36.

  ```ts
  } catch (err: unknown) {
    console.error('[QuizResults] Failed to retake quiz:', err)
    toast.error('Could not start retake. Please try again.')
  }
  ```

- **`src/app/pages/Quiz.tsx:110-126` (confidence: 82)**: The attempt count query runs sequentially inside the `.then()` callback of the quiz fetch, creating a waterfall. The `async found =>` conversion means the outer `.catch()` on line 128 no longer catches errors from the attempt count query in all code paths -- the inner `try/catch` handles it, but `setFetchState('found')` is called on line 118 *before* the attempt count query resolves. This means `hasCompletedBefore` updates *after* the component renders with `fetchState === 'found'`, causing the start screen to briefly flash "Start Quiz" then change to "Retake Quiz". Why: Learners see a label flicker on the start screen, which feels unpolished. Fix: Run both queries in parallel using `Promise.all`, or at minimum move `setFetchState('found')` after the attempt count resolves:

  ```ts
  .then(async found => {
    if (ignore) return
    if (!found) { setFetchState('error'); return }
    setQuiz(found)
    setSavedProgress(loadSavedProgress(found.id))

    // Check attempt history before marking as found
    let completed = false
    try {
      const count = await db.quizAttempts.where('quizId').equals(found.id).count()
      completed = count > 0
    } catch (err) {
      console.warn('[Quiz] Failed to check attempt history:', err)
    }
    if (!ignore) {
      setHasCompletedBefore(completed)
      setFetchState('found')
    }
  })
  ```

#### Medium

- **`src/app/components/quiz/ScoreSummary.tsx:121-123` (confidence: 75)**: `Math.round(percentage) - Math.round(previousBestPercentage)` can produce a delta that differs from what the visual display implies. For example, if `percentage = 84.6` and `previousBestPercentage = 84.4`, `roundedPct` = 85, `Math.round(previousBestPercentage)` = 84, so `delta = 1`. The display says "Previous best: 84%" and "(+1%)" which is correct. But if both are 84.5, `delta = 0` and it shows "Same as best" even though the raw values differ by a tiny amount. This is edge-case-correct but could surprise. Fix: Consider rounding both to `roundedPct` style (`Math.round(Math.min(100, Math.max(0, x)))`) for consistency, though current behavior is acceptable.

- **`src/app/components/quiz/ScoreSummary.tsx:128` (confidence: 72)**: Triple non-null assertion (`previousBestPercentage!`) in the `improvementSrText` ternary chain. The `delta != null` guard proves `previousBestPercentage` is defined, but TypeScript's control flow analysis does not narrow optional parameters through intermediate variables. The assertions are technically safe here but reduce type safety. Fix: Extract `const prevBest = previousBestPercentage!` once after the null check, or restructure as an early return/helper.

- **`src/app/pages/QuizResults.tsx:146-152` (confidence: 78)**: The "View All Attempts (Coming Soon)" `<span>` has `aria-disabled="true"` but is not a focusable element. `aria-disabled` on a non-interactive `<span>` has no semantic meaning for assistive technology -- it is not in the tab order, and screen readers will not announce it as a disabled interactive element. Why: The AC says "I can click 'View All Attempts' to see full history" -- the current implementation is intentionally a placeholder, but `aria-disabled` on a `<span>` is misleading for screen readers. Fix: Either remove `aria-disabled` (it is already styled as muted text, which is sufficient), or use a `<button disabled>` for proper semantics:

  ```tsx
  <button
    disabled
    className="text-muted-foreground text-sm inline-flex items-center gap-1 min-h-[44px] cursor-default"
  >
    <History className="size-4" aria-hidden="true" />
    View All Attempts (Coming Soon)
  </button>
  ```

- **`src/app/pages/Quiz.tsx:110` (confidence: 70)**: Converting `.then(found => {` to `.then(async found => {` means the callback now returns a Promise. If the inner `await` throws outside the inner `try/catch` (e.g., a future code edit adds a line after the try/catch), the outer `.catch()` on line 128 would catch it, but the `ignore` flag check on line 111 would have already passed. This is fragile. Fix: Keep the inner async block self-contained or refactor to `async/await` for the entire effect body (standard pattern in this codebase).
