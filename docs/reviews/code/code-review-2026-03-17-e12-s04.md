## Code Review: E12-S04 — Create Quiz Route and QuizPage Component

### What Works Well

1. **Solid async cleanup pattern in Quiz.tsx** — The `useEffect` with `let ignore = false` and cleanup return properly prevents stale state updates on unmount (lines 57-79). This is exactly the right pattern for Dexie lookups in effects, and it's documented in the lessons learned section.

2. **Clean component decomposition** — QuizStartScreen and QuizHeader are well-separated, props-driven components with proper TypeScript interfaces. The page component (Quiz.tsx) acts as the orchestrator, and the child components are presentation-only.

3. **E2E test quality** — Tests use the shared `seedQuizzes` helper, the `makeQuiz`/`makeProgress` factories, proper sidebar seeding, and specific role+name locators (especially the AC4 fix for sidebar nav collision).

### Findings

#### Blockers

- **[src/types/quiz.ts:129 vs src/app/components/quiz/QuizStartScreen.tsx:28 vs src/app/components/quiz/QuizHeader.tsx:24] (confidence: 92)**: Unit mismatch between Zod schema documentation and actual usage. The `QuizSchema.timeLimit` JSDoc says "Time limit in milliseconds" and `QuizProgressSchema.timeRemaining` says "Remaining time in milliseconds" — but the entire codebase treats both as **minutes**: `QuizStartScreen` renders `${quiz.timeLimit} min` (line 28), `useQuizStore.startQuiz` copies raw `quiz.timeLimit` to `timeRemaining` (store line 68), and `QuizHeader` converts with `progress.timeRemaining * 60` to get seconds (line 24). If any future story trusts the Zod JSDoc and passes milliseconds, the timer will display 16.6 hours instead of 10 minutes. **Why**: A learner seeding quiz data via an import/API in a future epic will produce wildly wrong timer values. **Fix**: Update the Zod schema comments to say "Time limit in **minutes**" (`quiz.ts:129`) and "Remaining time in **minutes**" (`quiz.ts:213`). This aligns documentation with all existing runtime behavior. Alternatively, change the unit to milliseconds everywhere, but that's a larger refactor touching E12-S03.

#### High Priority

- **[Recurring] [src/app/pages/Quiz.tsx:73] (confidence: 85)**: Silent error swallowing in Dexie `.catch()`. The catch block on the quiz lookup (line 73-74) sets `fetchState` to `'error'` but does not log the actual error. If the IndexedDB query fails (e.g., missing index, corrupt database), there is no console output to debug. **Why**: A learner seeing "No quiz found for this lesson" won't know if it's a missing quiz or a database corruption issue, and no diagnostic trace exists for developers. **Fix**: `.catch((err) => { console.error('[Quiz] Failed to load quiz:', err); if (!ignore) setFetchState('error') })`. This is a recurring silent failure pattern (see agent memory: "Empty catch blocks on IndexedDB/localStorage operations").

- **[src/app/components/quiz/QuizHeader.tsx:28-34] (confidence: 82)**: Timer `useEffect` has empty dependency array but reads `remainingSeconds` from state initializer. If `QuizHeader` re-mounts with a different `progress` prop (e.g., navigating between quizzes without full unmount, or React Strict Mode double-mount in dev), the timer will reinitialize from the *new* initial state but the interval closure captures `setRemainingSeconds` correctly since it uses a callback updater. However, the real issue is: the timer **never syncs back to the store**. The countdown runs locally in `useState` but `progress.timeRemaining` in the Zustand store remains at the original value. If the learner navigates away and the progress is persisted, they get the full time back on resume. **Why**: A learner who leaves with 2 minutes remaining and returns will see the full 10 minutes again — the timer resets on every mount. **Fix**: Add a `useEffect` that writes `remainingSeconds` back to the store periodically (e.g., every 10 seconds) via a `updateTimeRemaining` action, or at minimum on unmount cleanup.

- **[src/app/pages/Quiz.tsx:86-95] (confidence: 80)**: `handleResume` bypasses all validation when restoring progress from localStorage. The raw `savedProgress` from `loadSavedProgress` is cast as `QuizProgress` via `as` (line 23) without Zod validation, then injected directly into the store via `setState`. If the localStorage data is corrupt, stale (from a different quiz version), or maliciously crafted, the store will contain invalid state. **Why**: A learner with stale localStorage from a quiz that was later updated (questions added/removed) will see mismatched `questionOrder` IDs that don't correspond to actual questions, producing blank question displays. **Fix**: Validate with `QuizProgressSchema.safeParse(parsed)` in `loadSavedProgress`, and verify `savedProgress.quizId === quiz.id` before allowing resume (already partially done via `answeredCount > 0` check, but no structural validation).

- **[Recurring] Uncommitted changes in working tree (confidence: 95)**: `git diff HEAD` shows 6 files with uncommitted modifications including `src/stores/useQuizStore.ts` (formatting changes) and `src/app/components/onboarding/OnboardingOverlay.tsx` (logic changes). These changes are not part of the committed diff and will not be included in a PR. **Why**: This is a recurring pattern (10 consecutive stories E03-S02 through E05-S06). The store formatting fixes in particular suggest a review finding was applied but never committed. **Fix**: Stage and commit all pending changes, or stash unrelated changes before the review cycle.

#### Medium

- **[src/app/pages/Quiz.tsx:83] (confidence: 75)**: `handleStart` calls `startQuiz(lessonId).catch(console.error)` — a fire-and-forget pattern. If `startQuiz` fails (e.g., Dexie is unavailable), the UI remains on the start screen with no feedback to the learner. The error is logged to console but the user sees nothing. **Fix**: Set a local error state in the catch handler and render an inline error message, e.g., "Failed to start quiz. Please try again."

- **[src/app/components/quiz/QuizHeader.tsx:44] (confidence: 72)**: `aria-live="polite"` on the timer span updates every second as `remainingSeconds` changes. Per the story's own accessibility guidance (line 133 of the story file): "aria-live updating every minute (not every second -- too noisy)". The current implementation announces the time every second to screen readers, which is disruptive. **Fix**: Use a separate visually-hidden `aria-live` element that updates only when the minute value changes, or debounce the `aria-label` update to once per minute.

- **[src/app/components/quiz/QuizStartScreen.tsx:12] (confidence: 70)**: `savedProgress.answers` is accessed without null guard. `loadSavedProgress` in Quiz.tsx checks `Object.keys(parsed.answers ?? {}).length`, but `QuizStartScreen` accesses `savedProgress.answers` directly (line 12). If `savedProgress` somehow has `answers: undefined` (e.g., from a corrupted localStorage entry that passed the `loadSavedProgress` check due to a race), `Object.keys(undefined)` will throw. **Fix**: Use `Object.keys(savedProgress.answers ?? {}).length` for defensive consistency, matching the pattern already used in `loadSavedProgress`.

#### Nits

- **Nit** [src/app/pages/Quiz.tsx:3] (confidence: 65): Direct import `from '@/db'` couples the page component to the database layer. The story notes document this as intentional ("avoids coupling start-screen render to store state"), which is a reasonable tradeoff. However, it means two different code paths query the same Dexie table (`Quiz.tsx` and `useQuizStore.startQuiz`). Consider extracting a shared `getQuizByLessonId` query function in a future story to keep the Dexie query surface in one place.

- **Nit** [src/app/components/quiz/QuizHeader.tsx:17] (confidence: 60): `progress.questionOrder.length || quiz.questions.length` uses `||` which treats `0` as falsy. If `questionOrder` is an empty array (valid per schema), this falls through to `quiz.questions.length`. This is likely correct behavior but `??` would be semantically clearer since `questionOrder.length` is always a number.

- **Nit** [src/app/components/quiz/QuizStartScreen.tsx:24-32] (confidence: 60): Badge text content is not wrapped in semantic elements. Consider using `<Badge>` from shadcn/ui for consistency with the rest of the design system, rather than raw `<span>` elements with manually applied badge styles.

### Recommendations

1. **First**: Fix the Zod schema JSDoc comments for `timeLimit` and `timeRemaining` — the "milliseconds" documentation is factually wrong and will cause bugs in future epics. This is a one-line fix per field.
2. **Second**: Add `console.error` to the Dexie `.catch()` in Quiz.tsx — silent failures make debugging impossible.
3. **Third**: Add Zod validation to `loadSavedProgress` — raw `JSON.parse` + `as` cast is a corruption vector.
4. **Fourth**: Address the timer not syncing back to the store — this will cause resume bugs when the timer feature is fully wired in E15.
5. **Fifth**: Fix the `aria-live` frequency — the AC spec explicitly calls out per-minute updates, not per-second.
6. **Sixth**: Commit all uncommitted changes before merging.
