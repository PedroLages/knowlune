## BLOCKERs

### B1 ‚Äî `loadLearnerModel` is never called ‚Äî the entire E72 personalization feature is dead in production

**Epics:** E57, E72  
**Files:** `src/ai/hooks/useTutor.ts`, `src/app/components/tutor/TutorChat.tsx`, `src/stores/useTutorStore.ts`

`useTutorStore` exposes `loadLearnerModel(courseId)` which loads the persistent learner model from IndexedDB. Neither `useTutor`, `TutorChat`, nor any parent component ever calls it. As a result:

- `useTutorStore.getState().learnerModel` is always `null`
- `serializeLearnerModelForPrompt` is never executed
- The learner model slot in the system prompt is permanently empty
- `triggerSessionBoundaryUpdate` short-circuits silently (`if (!state.learnerModel) return`)
- Every session boundary update is skipped ‚Äî no learning data is ever persisted

The tutor never personalizes. Everything built in E72 is dead code in production.

### B2 ‚Äî Edit dialog deletions are silent no-ops ‚Äî the edit feature cannot remove anything

**Epic:** E72  
**Files:** `src/app/components/tutor/TutorMemoryEditDialog.tsx:35-44`, `src/ai/tutor/learnerModelService.ts:84-86`

`handleRemoveStrength` filters out the target concept and passes the reduced array to `updateLearnerModel`. But `updateLearnerModel` uses **additive merge semantics**: `deduplicateConcepts([...existing.strengths, ...updates.strengths])`. The "deleted" concept still exists in `existing.strengths`, survives deduplication, and is never removed.

The user clicks X, the spinner runs, the call succeeds, and the concept reappears. The edit dialog cannot delete anything. A `replaceLearnerModel` function (full overwrite, not merge) is needed for UI-initiated edits.

### B3 ‚Äî Greedy JSON regex in `sessionAnalyzer` silently discards LLM session analysis on virtually every call

**Epics:** E57, E72  
**File:** `src/ai/tutor/sessionAnalyzer.ts:333`

```typescript
responseText.match(/\{[\s\S]*\}/)
```

This is greedy. LLM responses commonly include explanatory prose with curly braces (markdown code blocks, example objects, template literals). The regex will match from the first `{` in the response to the very last `}`, capturing garbage. `JSON.parse` throws, the catch fires, and the system silently falls back to local insights.

LLM-enriched session analysis almost certainly fails on the majority of real responses. The fallback mask means this has gone undetected.

### B4 ‚Äî Tier threshold logic duplicated across two files with no single source of truth

**Epic:** E56  
**Files:** `src/lib/knowledgeScore.ts:77-81`, `src/lib/knowledgeTierUtils.ts:35-39`

`getKnowledgeTier` and `getTierFromScore` are textually identical functions ‚Äî same magic numbers (`>=70`, `>=40`), completely decoupled. The store uses `knowledgeScore.ts`; the widget and popover use `knowledgeTierUtils.ts`. If one is updated without the other, topic tiers silently diverge between the knowledge map page and the dashboard widget. This will happen.

### B5 ‚Äî SVG `<g>` elements cannot receive focus ‚Äî keyboard navigation is broken across the entire treemap

**Epic:** E56  
**File:** `src/app/components/knowledge/TopicTreemap.tsx:78-91`

`CustomCell` applies `role="button"`, `tabIndex={0}`, and an `onKeyDown` handler to SVG `<g>` elements. In all major browsers, `<g>` is not a focusable element regardless of `tabIndex`. Keyboard users cannot navigate or activate any cell. The focus ring `rect` with `group-focus-visible:opacity-100` also never renders.

This is a WCAG 2.1 AA failure given the project's stated accessibility requirements. The mobile accordion fallback is accessible, but the desktop treemap ‚Äî the primary UI ‚Äî is not.

## HIGH

### H1 ‚Äî RAG 10s timeout blocks the entire message pipeline with no fallback

**Epic:** E57  
**Files:** `src/ai/tutor/tutorRAG.ts:26`, `src/ai/hooks/useTutor.ts:242`

`retrieveTutorContext` has a 10-second timeout. Under Ollama load, this causes the user to stare at nothing for 10 seconds before the timeout fires, the catch returns empty chunks, and only then does the prompt builder run. There is no "start generating without RAG while RAG resolves" strategy. For self-hosted Ollama (the primary deployment target), this is an unacceptable UX freeze on every message send.

### H2 ‚Äî Fire-and-forget `lazyEmbedTranscript` ‚Äî RAG is never available on the first user message

**Epic:** E57  
**File:** `src/ai/hooks/useTutor.ts:232`

On the first message for a lesson, `hasEmbeddings` is `false`, so `lazyEmbedTranscript` fires and is forgotten. Immediately after, `if (hasEmbeddings)` ‚Äî still `false` ‚Äî skips RAG retrieval entirely. The first question (typically the most important) always gets zero transcript grounding. There is no mechanism to await a partial result or retry. RAG only activates on the second message and beyond.

### H3 ‚Äî Fire-and-forget `.catch(() => {})` on session boundary updates silently drops all learning data

**Epic:** E72  
**File:** `src/ai/hooks/useTutor.ts:153`

```typescript
void updateFromSession(...).catch(() => {})
```

Every failure ‚Äî Dexie errors, JSON parse, LLM timeouts ‚Äî is discarded. If session boundary updates fail consistently (e.g., corrupted IndexedDB record), the user gets zero indication their progress is not being saved. The entire value proposition of the learner model silently degrades.

### H4 ‚Äî `visibilitychange` does not fire on mobile crash/force-quit/OOM kill

**Epic:** E72  
**File:** `src/ai/hooks/useTutor.ts:121-127, 156-166`

Session boundary detection relies on component unmount and `visibilitychange`. Neither fires when iOS Safari is killed by the OS, Android receives a force-stop, or the app crashes. In-memory session data (Zustand) is lost. `pagehide` is the recommended replacement on mobile and fires more reliably. No periodic checkpoint exists as a safety net.

### H5 ‚Äî Naive 6-word concept extraction produces garbage stored in learner model and injected into prompts

**Epics:** E57, E72

## Cross-Epic Themes

### Theme 1: Silent degradation by design

Every epic chose fire-and-forget for background work (embedding, session analysis, profile building). This is architecturally correct ‚Äî these tasks must not block the user. But combined with `.catch(() => {})`, failures are invisible. A systematic logging/telemetry layer is needed so degradation is observable without user reports.

**Instances:** E57 H2, E57 H3/E72 H3, E72 B3

### Theme 2: The learner model system is entirely non-functional

B1 renders all of E72 dead in production. B2 makes the edit dialog non-functional. B3 means LLM-enriched analysis silently falls back on virtually every call. B1+H6 together mean session boundary updates never run. These are not independent bugs ‚Äî they form a coherent failure of the entire personalization loop. Fix B1 first; it unblocks meaningful testing of everything else.

### Theme 3: Hardcoded thresholds with no empirical basis

Knowledge tier thresholds, FSRS weakness heuristics, recency decay coefficients, urgency formula weights, RAG position boost constants, minimum similarity scores ‚Äî all are magic numbers. None are documented with rationale. None are configurable. None have been validated against user behavior data. This is acceptable for an initial implementation but should be flagged as a calibration debt item.

**Instances:** E56 L13, E57 implicit, E63 M10

### Theme 4: Missing integration test coverage for multi-module chains

Each epic tested its modules in isolation with mocked dependencies. The integration paths ‚Äî `useTutor ‚Üí learnerProfileBuilder ‚Üí tutorPromptBuilder`, `sessionAnalyzer ‚Üí updateFromSession ‚Üí learnerModelService ‚Üí Dexie`, `computeScores ‚Üí topicResolver ‚Üí knowledgeScore` ‚Äî have no end-to-end tests. Bugs at integration seams (like B1) go undetected.
