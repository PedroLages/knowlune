Timer syncs back to store every 60 seconds, on visibility change, and on unmount (QuizHeader.tsx:69-106). `loadSavedProgress` validates with `QuizProgressSchema.safeParse` (Quiz.tsx:26). The AC3 E2E test clicks Resume and verifies position restoration (spec line 166-167). The setInterval self-clears at zero (QuizHeader.tsx:50-53). 3. **Zod validation on localStorage input is properly implemented.** `safeParse` with a descriptive warning log and graceful fallback to `null`, plus the additional guard checking `Object.keys(result.data.answers).length === 0` to prevent showing a resume button when there is no meaningful state.

### Findings

#### High Priority

- **[src/stores/useQuizStore.ts:193 + src/app/pages/Quiz.tsx:22-37] (confidence: 90)**: Persist middleware only saves `currentProgress`, not `currentQuiz`, making crash recovery silently incomplete. On browser refresh during an active quiz: (a) Zustand rehydrates `currentProgress` from `levelup-quiz-store`, but `currentQuiz` remains `null`; (b) the active-quiz branch (Quiz.tsx:158) requires both `currentProgress !== null` AND `currentQuiz !== null`, so it falls through; (c) `loadSavedProgress` reads from `quiz-progress-${quizId}` which is never written by production code; (d) the learner sees a bare "Start Quiz" button with progress silently discarded. The E2E test for AC3 works only because it seeds the per-quiz localStorage key directly -- this key has no production writer. Why: Learners who experience a browser refresh during a timed quiz lose all progress and timer state. Fix: Add `currentQuiz` to the partialize function: `partialize: state => ({ currentProgress: state.currentProgress, currentQuiz: state.currentQuiz })`. Alternatively, add a production code path that writes progress to the per-quiz localStorage key.

- **[src/app/components/quiz/QuizHeader.tsx:106] (confidence: 82)**: `useEffect` dependency `[remainingSeconds === null]` is a boolean expression in the dependency array. While this works correctly (React compares the computed `true`/`false` value and only re-runs when timed/untimed state changes), it is an unusual pattern that a future developer may "fix" to `[remainingSeconds]`, causing the sync interval to tear down and re-create on every tick (every second). This would break the 60-second sync cadence and cause excessive store writes. Why: Maintenance risk on a time-critical feature that affects quiz fairness. Fix: Extract to a named variable at the top of the component: `const isTimed = remainingSeconds !== null`, then use `[isTimed]` as the dependency. This makes the intent explicit.

- **[src/app/pages/Quiz.tsx:93-95] (confidence: 78)**: `handleStart` calls `startQuiz(lessonId).catch(console.error)`. The `.catch(console.error)` is unreachable dead code -- `startQuiz` wraps its entire body in try/catch (useQuizStore.ts:76-79) and never rejects. The page does subscribe to `selectError` (line 55) and renders error state (line 135), so the error flow actually works. But the `.catch(console.error)` misleads readers into thinking errors are only console-logged without user feedback. Why: Future developers may add rejection logic in the store, expecting the caller to handle it, not realizing the catch is unreachable. Fix: Remove `.catch(console.error)` since `startQuiz` handles errors internally, or replace with a comment: `// startQuiz handles errors internally via store state`.

#### Medium

- **[src/app/components/quiz/QuizHeader.tsx:24] (confidence: 75)**: `progress.questionOrder.length ?? quiz.questions.length` -- the `??` operator triggers only on `null`/`undefined`, not on `0`. Since `questionOrder` is a `z.array(z.string())` (always an array after Zod validation), `.length` is always a number and `??` never falls through. If `questionOrder` is an empty array (corrupt state), `totalQuestions` becomes 0, and the progress text shows "Question 1 of 0". Fix: Use `progress.questionOrder.length || quiz.questions.length` (logical OR) if the intent is to fall back when the array is empty, or remove the fallback entirely since Zod guarantees `questionOrder` is populated.

- **[src/app/pages/Quiz.tsx:100-105] (confidence: 70)**: `handleResume` uses `useQuizStore.setState()` to inject progress directly, setting `currentQuiz: quiz` where `quiz` is the locally-fetched Dexie result. If the quiz definition in Dexie has changed since progress was saved (questions edited, reordered, or removed), the restored `savedProgress.questionOrder` references question IDs that may no longer exist. This would cause undefined behavior when `QuestionDisplay` renders (Story 12.5). Fix: Add a sanity check that all `savedProgress.questionOrder` entries exist in `quiz.questions` before restoring. If they do not match, discard saved progress and show "Start Quiz" instead.

- **[src/app/components/quiz/QuizHeader.tsx:30-33] (confidence: 70)**: Timer state is initialized from `progress.timeRemaining` on mount but does not react to prop changes. If a parent re-renders QuizHeader with a different `progress` object (e.g., after Zustand rehydration race), the timer continues counting from the original mount value. Currently safe because QuizHeader only mounts once per active quiz session, but it violates the expectation that component state should respond to prop changes. Fix: No code change required now, but add a comment documenting that QuizHeader must unmount/remount (not just re-render) for a timer reset.

#### Nits

- **Nit** [src/app/components/quiz/QuizHeader.tsx:67] (confidence: 65): The `// intentionally empty` comment on the timer effect's dependency array is helpful. Consider extracting the countdown logic into a `useCountdownTimer` hook to make the empty-deps intent architectural rather than documented-by-comment.

- **Nit** [src/app/components/quiz/QuizStartScreen.tsx:24] (confidence: 60): `savedProgress.answers ?? {}` -- the `??` fallback is unnecessary because `answers` is required in `QuizProgressSchema`. After `safeParse` succeeds upstream, `answers` is always an object. The fallback suggests distrust of the validation and may confuse readers about whether `safeParse` is actually being called.

- **Nit** [src/app/pages/Quiz.tsx:34] (confidence: 55): The bare `catch` block in `loadSavedProgress` silently swallows JSON.parse errors. The `safeParse` above handles Zod validation failures with a warning log, but if `JSON.parse` throws (malformed JSON in localStorage), nothing is logged. Consider `catch (e) { console.warn('[Quiz] Failed to parse saved progress:', e); return null }`.

### Recommendations

Fix in this order:

1. **Persist `currentQuiz`** (useQuizStore.ts:193) -- one-line change to partialize that prevents silent data loss on browser refresh. This is the highest-impact finding.
2. **Extract `isTimed` variable** (QuizHeader.tsx:106) -- prevents maintenance-induced timer bugs, two-line change.
3. **Clean up dead `.catch(console.error)`** (Quiz.tsx:94) -- remove or comment to prevent confusion about error handling flow.
4. **Fix `??` to `||`** on QuizHeader.tsx:24 -- or remove fallback entirely since Zod guarantees the array is populated.
