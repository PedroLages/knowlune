## Summary

New `src/lib/ankiExport.ts` module that generates Anki-compatible `.apkg` files from IndexedDB flashcards using sql.js (ASM.js build) + JSZip. Includes unit tests, vite config chunk splitting, and a type declaration for sql.js ASM.

## Findings

### MEDIUM — No error handling if sqlDb operations fail (ankiExport.ts:322-400)

The `sqlDb.run(createAnkiTemplate())` call and subsequent `sqlDb.exec()` / `sqlDb.run()` calls are not wrapped in try-catch. If the SQL template has an issue or a card insert fails, the SQLite database object will never be closed (memory leak). Consider wrapping the sqlDb usage in a try-finally block:

```ts
const sqlDb = new SQL.Database()
try {
  // ... all sqlDb operations
} finally {
  sqlDb.close()
}
```

Currently `sqlDb.close()` is only called on the happy path (line 406). An exception during card insertion would leak the database.

### MEDIUM — `Date.now()` used for ID generation is not deterministic (ankiExport.ts:324)

`const now = Date.now()` is used to generate deck IDs, model IDs, card IDs, and note IDs. This is fine for production but makes the export non-reproducible — exporting the same flashcards twice produces different `.apkg` files. Not a bug, but worth noting that this intentionally uses wall-clock time (not a test concern since tests mock at a higher level).

### LOW — `mockDbExec` returns same value for both `SELECT decks` and `SELECT models` queries (test:38-43)

The mock returns the same JSON structure for both the decks and models queries. The production code calls `sqlDb.exec('SELECT decks FROM col')` and `sqlDb.exec('SELECT models FROM col')` — both hit the same mock. This works because the test only verifies that `mockDbRun` was called (not the specific SQL), but it means the test doesn't validate the deck/model update logic accurately.

### LOW — `yieldToUI()` duplicated from flashcardExport.ts (ankiExport.ts:20-22)

The `yieldToUI()` function is identical to the one in `flashcardExport.ts`. Consider extracting to a shared utility (e.g., `exportService.ts` already exists and would be a natural home). Minor DRY concern.

### INFO — SQL injection safety via parameterized queries

Good: All card/note INSERT statements use parameterized queries (`?` placeholders). The only string interpolation in SQL is in `createAnkiTemplate()` which uses `JSON.stringify()` on hardcoded objects — safe since no user input flows into the template.

### INFO — Good architectural decisions

- ASM.js build avoids WASM binary file issues in production (EC-HIGH addressed)
- Dynamic import keeps sql.js off main bundle (AC5)
- Manual chunk splitting in vite.config.ts ensures sql.js is isolated
- Reuses `deriveFlashcardTags()` from S01 (AC3)
- HTML stripping addresses EC-HIGH for Tiptap content

## Verdict

**PASS with advisories.** Two MEDIUM issues (try-finally for DB cleanup, non-deterministic IDs) and two LOW issues. No blockers. The implementation is solid, correctly addresses the edge cases identified in the story, and the test suite covers all acceptance criteria.
