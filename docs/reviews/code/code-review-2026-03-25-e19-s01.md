## Code Review: E19-S01 — Authentication Setup

### What Works Well
1. **Error mapping is thoughtful.** The `mapSupabaseError()` function in `useAuthStore.ts` translates cryptic Supabase errors into user-friendly messages, and the exported `NETWORK_ERROR_MESSAGE` constant avoids string duplication across three form components. This is a solid pattern for auth UX.

2. **Proper cleanup on auth listener.** The `onAuthStateChange` subscription in `App.tsx:42-49` correctly returns an unsubscribe function, preventing memory leaks. The `setSession` approach keeps the Zustand store reactive without manual persistence.

3. **Accessibility baseline is strong.** Form inputs use `htmlFor`/`id` associations, error alerts use `role="alert"`, loading states have `aria-busy`, and touch targets are `min-h-[44px]` throughout. The magic link success state uses `aria-live="polite"`.

### Findings

#### Blockers

- **[src/stores/useAuthStore.ts:50-130] (confidence: 92)**: **Shared `loading` and `error` state across all three auth methods creates race conditions.** All tabs (EmailPasswordForm, MagicLinkForm, GoogleAuthButton) subscribe to the same global `loading` and `error` from the store. If a user starts a magic link request, then switches to the Google tab, the Google tab inherits the loading spinner from the magic link request. Worse, `clearError()` in AuthDialog's mode/open effect (`AuthDialog.tsx:57-59`) fires on every tab switch, silently discarding errors the user hasn't seen yet. Why: Learners could see phantom loading states or miss error messages when switching between auth methods. Fix: Move `loading` and `error` to local component state in each form (they're per-form concerns, not global state). The store actions already return `{ error?: string }` -- use that return value directly instead of subscribing to shared store error state.

- **[src/lib/auth/supabase.ts:12] (confidence: 95)**: **Supabase client is created with empty strings when env vars are missing, causing silent runtime failures.** `createClient('' , '')` produces a client that will make HTTP requests to `''` (relative URL), which will hit the Vite dev server and return HTML -- silently failing with confusing errors rather than crashing early. Why: Developers (or CI) without `.env` configured will get cryptic network errors instead of a clear "auth not configured" message. Fix: Either throw an error when env vars are missing (fail fast), or create a null/mock client that surfaces a clear "Supabase not configured" error on every method call. At minimum, guard the `createClient` call:
  ```typescript
  export const supabase = supabaseUrl && supabaseAnonKey
    ? createClient(supabaseUrl, supabaseAnonKey)
    : null
  ```
  Then handle the null case in the store with a clear error message.

#### High Priority

- **[src/stores/useAuthStore.ts:57-66, 69-78, 81-90, 93-106] (confidence: 88)**: **No try/catch around Supabase SDK calls -- unhandled promise rejections on network failures.** The `signUp`, `signIn`, `signInWithMagicLink`, and `signInWithGoogle` actions `await` Supabase methods directly. While Supabase typically returns `{ error }` objects, true network failures (DNS resolution failure, CORS errors, aborted requests) can throw exceptions that bypass the error-return pattern. Why: A learner on flaky WiFi could trigger an unhandled rejection that leaves `loading: true` permanently, freezing the form. Fix: Wrap each `await supabase.auth.*` call in try/catch:
  ```typescript
  signUp: async (email, password) => {
    set({ loading: true, error: null })
    try {
      const { error } = await supabase.auth.signUp({ email, password })
      if (error) { /* ... */ }
      set({ loading: false })
      return {}
    } catch (e) {
      const mapped = NETWORK_ERROR_MESSAGE
      set({ loading: false, error: mapped })
      return { error: mapped }
    }
  }
  ```

- **[src/app/components/auth/EmailPasswordForm.tsx:20] (confidence: 82)**: **Destructuring all store state causes unnecessary re-renders.** `const { loading, error, signUp, signIn, clearError } = useAuthStore()` subscribes to the entire store. Any state change (e.g., `setSession` from another tab) triggers a re-render of every form component. Why: On session restore at app launch, all three tab forms re-render even though only `initialized` changed. Fix: Use individual selectors:
  ```typescript
  const loading = useAuthStore(s => s.loading)
  const error = useAuthStore(s => s.error)
  const signUp = useAuthStore(s => s.signUp)
  ```
  This is the standard Zustand pattern used elsewhere in the codebase (e.g., `AuthDialog.tsx:49` correctly uses `useAuthStore(s => s.clearError)`).

- **[src/app/components/auth/AuthDialog.tsx:88] (confidence: 85)**: **`Tabs defaultValue="email"` is uncontrolled -- tab state is not reset when dialog reopens.** Radix Tabs with `defaultValue` only sets the initial value. If the user selects the "Magic Link" tab, closes the dialog, and reopens it, the tab remains on "Magic Link" (Radix preserves internal state because the component stays mounted behind the Dialog). Why: Learners expect a fresh dialog state on reopen. Fix: Use controlled `value`/`onValueChange` with local state, and reset it in the `open` effect:
  ```typescript
  const [activeTab, setActiveTab] = useState('email')
  useEffect(() => {
    if (open) setActiveTab('email')
  }, [open])
  // ...
  <Tabs value={activeTab} onValueChange={setActiveTab}>
  ```

- **[src/app/pages/Settings.tsx:464] (confidence: 78)**: **Account card uses raw `<h2>` instead of `<CardTitle>`.** Every other card in Settings uses the `CardTitle` component (which renders semantically correct heading levels and consistent styling). The Account card manually renders `<h2 className="text-lg font-display leading-none">Account</h2>`. Why: Inconsistency in heading hierarchy and styling; if `CardTitle` styling changes, Account card won't pick it up. Fix: Use `<CardTitle className="text-lg font-display leading-none">Account</CardTitle>` to match the pattern.

#### Medium

- **[src/app/components/auth/AuthDialog.tsx:119-127] (confidence: 72)**: **Mode toggle button has no focus ring styling.** The `<button type="button" onClick={toggleMode} className="font-medium text-brand hover:underline">` lacks `focus-visible:outline` or `focus-visible:ring-*` styles. Other interactive elements in the codebase use the shadcn Button component which includes focus ring styles. Why: Keyboard users cannot see which element is focused when tabbing through the dialog. Fix: Use `<Button variant="link">` instead of a raw `<button>`, or add `focus-visible:underline focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2`.

- **[src/app/pages/Settings.tsx:66-67] (confidence: 70)**: **Duplicate `LogOut` import.** Line 20 imports `RotateCcw` and other icons from `lucide-react`, then line 23 has a separate `import { LogOut } from 'lucide-react'`. The `LogOut` icon should be consolidated into the existing import statement on lines 7-22. Fix: Add `LogOut` to the existing multi-line import and remove line 23.

- **[src/stores/useAuthStore.ts:44] (confidence: 72)**: **Network error detection is too broad.** `message.includes('Network')` will match any error containing the word "Network" (e.g., hypothetical "Network policy violation" or "Social Network provider error"). Why: Could incorrectly show the retry button for non-network errors. Fix: Match more specifically against known Supabase network error patterns, e.g., `message.includes('Failed to fetch') || message.includes('NetworkError') || message.includes('network request failed')`.

- **[src/app/components/auth/GoogleAuthButton.tsx:28-67, src/app/components/auth/AuthDialog.tsx:16-37] (confidence: 68)**: **Google SVG icon is duplicated.** The Google "G" logo SVG is defined in both `GoogleTabIcon` (AuthDialog.tsx:16-37) and `GoogleIcon` (GoogleAuthButton.tsx:5-26) with slightly different sizes (`size-4` vs `size-5`). Fix: Extract to a shared component `GoogleLogo` accepting a `className` prop, export from a shared location.

#### Nits

- **Nit** [src/app/components/auth/EmailPasswordForm.tsx:25] (confidence: 65): Email validation `!email.includes('@') || !email.includes('.')` is extremely permissive -- accepts `@.`, `.@`, etc. The `type="email"` HTML validation is stricter. Consider using a simple regex or rely solely on the browser's built-in email validation (the `required` and `type="email"` attributes already provide this). The client-side check is redundant with the native validation.

- **Nit** [src/app/components/auth/AuthDialog.tsx:131-148] (confidence: 60): Legal links (`/privacy`, `/terms`) point to routes that likely don't exist in Knowlune. Using `target="_blank"` on these SPA routes will open a blank page in a new tab. Either remove the links until pages exist, or use `#` placeholders with `aria-disabled`.

- **Nit** [docs/implementation-artifacts/plans/e19-s01-authentication-setup.md:287] (confidence: 90): The implementation plan contains what appears to be a real anon key: `sb_publishable_ACJWlzQHlZjBrEguHvfOxg_3BJgxAaH`. While anon keys are designed to be public, having them in committed documentation is unnecessary exposure. Consider replacing with a placeholder.

### Recommendations

1. **First**: Wrap all Supabase SDK calls in try/catch (High -- prevents permanent loading freeze on network errors)
2. **Second**: Move `loading`/`error` to local component state or use individual Zustand selectors (Blocker -- race condition between tabs + re-render fix)
3. **Third**: Guard Supabase client creation against missing env vars (Blocker -- silent failures in CI/dev)
4. **Fourth**: Control Tabs state and reset on dialog open (High -- stale tab state on reopen)
5. **Fifth**: Add focus ring to mode toggle button (Medium -- keyboard accessibility)
6. **Sixth**: Consolidate duplicate imports and Google SVG (Medium -- maintainability)
