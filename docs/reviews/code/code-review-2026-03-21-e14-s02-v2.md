### Findings

#### High Priority

- **[MultipleSelectQuestion.tsx:40-46, MultipleChoiceQuestion.tsx:33-39, TrueFalseQuestion.tsx:28-34] (confidence: 90)**: Number key shortcuts via `document.addEventListener('keydown')` do not guard against focus being inside text inputs. The quiz page renders within the Layout component which includes a search bar in the header. If a learner clicks the search bar and types a number, `e.preventDefault()` fires, swallowing the keystroke and triggering an option selection instead. This also affects any future fill-in-blank question type that uses an `<input>`. Fix: Add an early return when `document.activeElement` is an `<input>`, `<textarea>`, or `[contenteditable]`:
  ```typescript
  const handleKeyDown = useCallback((e: KeyboardEvent) => {
    if (!isActive) return
    const tag = (e.target as HTMLElement)?.tagName
    if (tag === 'INPUT' || tag === 'TEXTAREA' || (e.target as HTMLElement)?.isContentEditable) return
    // ... rest of handler
  }, [...])
  ```

- **[QuizActions.tsx:1, QuizNavigation.tsx:1] (confidence: 75)**: `forwardRef` is unnecessary on React 19. The project is on React `^19.2.4` (per `package.json`). React 19 passes `ref` as a regular prop, making `forwardRef` a deprecated pattern. While functionally harmless today, React will emit deprecation warnings in future minor releases, and it adds indirection (wrapper component) that complicates the component signature. Fix: Convert to regular function components that accept `ref` as a prop:
  ```typescript
  export function QuizActions({ ref, ...props }: QuizActionsProps & { ref?: React.Ref<HTMLButtonElement> }) { ... }
  ```

#### Medium

- **[Recurring] [MultipleSelectQuestion.tsx:92, MultipleChoiceQuestion.tsx:87, TrueFalseQuestion.tsx:83] (confidence: 80)**: `w-5 h-5` on `<kbd>` elements should use the Tailwind v4 `size-5` shorthand. This is a recurring pattern (tracked since E02-S05). Fix: Replace `w-5 h-5` with `size-5` in all three components.

- **[MultipleSelectQuestion.tsx:33-36] (confidence: 72)**: `handleToggle` is defined as a plain function but referenced inside `handleKeyDown`'s `useCallback`. ESLint `react-hooks/exhaustive-deps` would flag `handleToggle` as a missing dependency. While functionally correct (because `value` and `onChange` are in the deps), this creates a maintenance trap -- a future developer adding state to `handleToggle` without adding it to `handleKeyDown`'s deps would introduce a stale closure. Fix: Either wrap `handleToggle` in `useCallback` and add it to `handleKeyDown`'s deps, or inline the toggle logic directly in `handleKeyDown`.

#### Nits

- **Nit** [QuizActions.tsx, QuizNavigation.tsx] (confidence: 65): The `forwardRef` approach attaches `ref` to either the Next button or the Submit button depending on `isLast`. When transitioning from the second-to-last question to the last, the ref target changes from Next to Submit. This works because React handles ref switching across renders, but consider whether the ref semantics are clear to future maintainers -- a comment explaining "ref always targets the primary action button" would help.

- **Nit** [MultipleSelectQuestion.tsx:79] (confidence: 60): `key={`${index}-${option}`}` uses index as part of the key. Since options come from static quiz data and don't reorder, this is safe. But `option` alone would suffice (options should be unique within a question). Using index in the key can mask bugs if options were ever dynamically reordered.

### Recommendations

1. **Fix the number key shortcut input conflict (High).** This is a real user-facing bug -- typing numbers in the search bar will silently select quiz options. Apply the `activeElement` guard to all three question components.
2. **Remove `forwardRef` (High).** Align with React 19 patterns. This is a straightforward refactor across two files.
3. **Apply `size-5` shorthand (Medium).** Three-line fix across three files.
4. **Stabilize `handleToggle` reference (Medium).** Wrap in `useCallback` or inline.
