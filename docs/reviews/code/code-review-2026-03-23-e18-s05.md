## Code Review: E18-S05 — Integrate Quiz Completion with Study Streaks

### What Works Well

1. **Clean architecture decision.** The developer correctly identified that the epic's suggestion to create a `useStreakStore` was misaligned with the actual codebase architecture. Extending the existing `logStudyAction` / `studyDaysFromLog` pattern instead of creating a new store is the right call -- less code, less surface area, and consistent with how `progress.ts` already logs `lesson_complete`.

2. **Solid fire-and-forget isolation.** The `logStudyAction` call in `submitQuiz` (lines 170-180 of `useQuizStore.ts`) is wrapped in its own try/catch, placed after the successful Dexie write, and mirrors the existing `setItemStatus` pattern. This correctly implements AC4's requirement that streak failure must never block quiz submission.

3. **Working tree is clean.** All changes are committed. This breaks the 10-story recurring pattern of uncommitted fixes (E03-S02 through E05-S06). The lessons learned section is thorough and captures real issues encountered during implementation.

### Findings

#### Blockers

#### High Priority

- **[`src/lib/__tests__/studyLog.test.ts:424`] (confidence: 90)**: Stale test name "only counts lesson_complete actions" is now factually incorrect. After the E18-S05 changes, `activityFromLog` counts both `lesson_complete` AND `quiz_complete`. The test still passes because it only checks that `video_progress` does NOT count -- which is still true -- but the test name misleads future developers into thinking quiz completions are excluded from activity. **Why:** A developer reading this test name will assume `getStudyActivity` ignores quiz completions, contradicting the actual behavior. This is a latent documentation bug that will cause confusion when someone extends the activity system again. **Fix:** Rename to `'only counts lesson_complete and quiz_complete actions (not video_progress, note_saved, etc.)'` and add a companion test that explicitly verifies `quiz_complete` is counted by `getStudyActivity` (not just `getStreakSnapshot`). The new `quiz_complete` test section covers `getStreakSnapshot` but not `getStudyActivity` directly.

- **[`tests/e2e/story-e18-s05.spec.ts`] (confidence: 85)**: AC4 (streak logging failure does not block quiz submission) has **no E2E test**. The file header comment lists AC4 in scope, but no test implements it. The unit test (`useQuizStore.streakIntegration.test.ts:151`) covers this well, so E2E coverage is not strictly required -- but the comment creates a false impression of coverage. **Why:** If someone checks "which ACs are E2E-tested" by reading the file header, they'll incorrectly conclude AC4 is covered. This matters for regression confidence when refactoring the quiz submission flow. **Fix:** Either (a) add an E2E test that mocks `localStorage.setItem` to throw before quiz submission and verifies the quiz still completes, or (b) remove AC4 from the file header comment and add a note: `// AC4 covered by unit test: useQuizStore.streakIntegration.test.ts`.

- **[`src/stores/useQuizStore.ts:175`] (confidence: 80)**: `new Date().toISOString()` in the `logStudyAction` call uses real wall-clock time, which diverges from the `Date.now()` used on line 98 (`startTime: Date.now()`) and line 140 (`timeSpent: Date.now() - currentProgress.startTime`). In E2E tests, the Date mock set via `addInitScript` makes this deterministic. But in the unit tests (`useQuizStore.streakIntegration.test.ts`), there's **no `vi.useFakeTimers()`** -- the timestamp assertion uses `expect.any(String)` to paper over non-determinism. **Why:** The unit test cannot verify that the logged timestamp matches the expected date, which matters for idempotency logic (`studyDaysFromLog` deduplicates by date). If a timezone edge case caused the logged date to differ from "today", the unit test would not catch it. **Fix:** Add `vi.useFakeTimers()` + `vi.setSystemTime()` to `useQuizStore.streakIntegration.test.ts` beforeEach, and assert the exact timestamp string in the `logStudyAction` call expectation.

#### Medium

- **[`src/lib/studyLog.ts:31-33`] (confidence: 75)**: `saveLog()` has no try/catch around `localStorage.setItem()`. If localStorage is full, `logStudyAction` will throw. The caller in `useQuizStore.ts` (line 170-180) catches this, which is correct -- but `logStudyAction` is also called from `progress.ts` (lines 170, 195, 214, 289, 349) WITHOUT any try/catch wrapper. This means a full localStorage could cause `markLessonComplete`, `markVideoProgress`, or `saveNote` to throw unhandled. This is a pre-existing issue not introduced by this story, but the story expands the blast radius by adding another `logStudyAction` caller. **Fix:** Wrap `saveLog` in try/catch inside `logStudyAction` itself (catch, log, and re-throw or silently fail depending on policy). Alternatively, add try/catch around each `logStudyAction` call in `progress.ts`.

- **[`tests/e2e/story-e18-s05.spec.ts:44`] (confidence: 78)**: `TODAY_STR` computation uses `new Intl.DateTimeFormat('sv-SE').format(new Date(FIXED_DATE)).substring(0, 10)` at module level. This runs in the Node.js Playwright runner (not the browser), so it uses the CI/local machine's timezone. Meanwhile, the app's `toLocalDateString` uses `date.toLocaleDateString('sv')` in the browser context. If the CI runner's timezone differs from the Date-mocked browser timezone, `TODAY_STR` will not match the app's recorded date for `FIXED_DATE`. The FIXED_DATE is noon UTC (`T12:00:00.000Z`), which is safe for most UTC-negative timezones but could produce a different date in UTC+13 (e.g., Samoa). **Fix:** Derive `TODAY_STR` from `FIXED_DATE` using the same logic as `toLocalDateString` (or hardcode `'2025-01-15'` since `FIXED_DATE` is `2025-01-15T12:00:00.000Z` -- noon UTC is the same calendar date in all timezones within +/-12 hours of UTC).

- **[`src/stores/useQuizStore.ts:176`] (confidence: 72)**: The `metadata` object in the `logStudyAction` call includes `{ timeSpent, passed }` but omits `quizId` and `score` (which the implementation plan specified). The `StudyAction.metadata` field is typed as `Record<string, unknown>`, so there's no compile-time enforcement of which keys are included. **Why:** Future analytics or debugging that filters by quizId in the study log won't find it. The plan explicitly listed `quizId` and `score` as metadata fields. **Fix:** Add `quizId: currentQuiz.id` and `score: result.percentage` to the metadata object to match the plan and improve debuggability.

#### Nits

- **Nit** [`tests/e2e/story-e18-s05.spec.ts:102-108`] (confidence: 65): `afterEach` cleanup removes `study-log`, `study-longest-streak`, and `study-streak-pause` but not `study-streak-freeze-days` or the per-quiz `quiz-progress-*` keys. This is unlikely to cause cross-test pollution since no tests set freeze days, but it's inconsistent with the localStorage keys used by `studyLog.ts`. Consider adding `localStorage.removeItem('study-streak-freeze-days')` for completeness.

- **Nit** [`src/lib/__tests__/studyLog.test.ts:336-339`] (confidence: 60): Pre-existing test "ignores non-lesson_complete actions" at line 336 only tests `video_progress`. With `quiz_complete` now also counting toward streaks, this test name is potentially confusing. It's technically correct (video_progress is a non-streak action) but the name implies ALL non-lesson_complete types are ignored, which is no longer true. Consider renaming to "ignores video_progress actions for streak calculation".

### Recommendations

1. **First:** Rename the stale test at `studyLog.test.ts:424` and add explicit `getStudyActivity` coverage for `quiz_complete`. This prevents the next developer from being misled. (5 minutes)

2. **Second:** Add `vi.useFakeTimers()` to the streak integration unit test to make timestamp assertions deterministic. (10 minutes)

3. **Third:** Clarify the AC4 E2E gap -- either add the test or fix the comment. (5-15 minutes depending on approach)

4. **Fourth:** Add missing metadata fields (`quizId`, `score`) to the `logStudyAction` call to match the plan. (2 minutes)

5. **Optional:** Address the `saveLog` try/catch gap in `studyLog.ts` -- this is pre-existing but worth a follow-up story.
