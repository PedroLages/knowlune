## Summary

This story refactors AI consumers (aiSummary, noteQA, thumbnailService) and the Anthropic LLM client to use the centralized `getLLMClient` factory with per-feature model resolution, replacing duplicated provider/model/apiKey plumbing. The server `providers.ts` is updated to import `PROVIDER_DEFAULTS` from the shared `modelDefaults` module. Tests are rewritten to mock the LLM factory layer instead of raw fetch/SDK calls.

## Findings

### MEDIUM — Duplicated fallback logic (DRY violation)

**Files:** `src/lib/aiSummary.ts:177-204`, `src/lib/noteQA.ts:157-185`

The AC8 fallback pattern (catch LLMError, resolve feature model, check if non-default, get API key, create fallback client, re-stream) is copy-pasted across both files (~25 lines each). This should be extracted into a shared helper, e.g. `streamWithFallback(featureId, messages)` in the LLM factory or a new utility.

**Classification:** STORY-RELATED

### MEDIUM — Fallback silently yields nothing when API key missing

**Files:** `src/lib/aiSummary.ts:193-194`, `src/lib/noteQA.ts:171-172`

In the fallback path, if `getDecryptedApiKeyForProvider()` returns null/undefined, the code falls through without retrying and then `throw firstError` fires. This is technically correct (the original error is thrown), but the user sees a cryptic AUTH_ERROR instead of a clear "API key not found" message. Consider adding a specific error message for this case.

**Classification:** STORY-RELATED

### LOW — `resolved.model` could equal `defaultModel` on AUTH_ERROR

**Files:** `src/lib/aiSummary.ts:187`, `src/lib/noteQA.ts:166`

If the user hasn't overridden the model (so `resolved.model === defaultModel`), an AUTH_ERROR means the API key itself is bad, not the model. The fallback correctly skips retry in this case, but the error message to the user will be the raw LLMError without context about "your API key may be invalid." This is a minor UX gap.

**Classification:** STORY-RELATED

### LOW — `thumbnailService.ts` uses `resolved.model || hardcoded-string` fallback

**File:** `src/lib/thumbnailService.ts:153`

`resolveFeatureModel('thumbnailGeneration')` should always return a model (from the three-tier cascade). The `|| 'gemini-2.0-flash-preview-image-generation'` fallback suggests the cascade might return empty string. If so, that's a bug in `resolveFeatureModel`. If not, the fallback is dead code.

**Classification:** STORY-RELATED

### NIT — AnthropicClient constructor now requires model (breaking change)

**File:** `src/ai/llm/anthropic.ts:25`

Removing the default parameter is correct for this story (forces callers to be explicit), but any external caller that relied on the default will break at compile time. The test was updated. Verify no other callers exist.

**Classification:** STORY-RELATED

### NIT — Dynamic import in catch block

**Files:** `src/lib/aiSummary.ts:195`, `src/lib/noteQA.ts:173`

`await import('@/ai/llm/factory')` inside the catch block to get `getLLMClientForProvider` is unusual since `getLLMClient` is already imported at the top of both files. `getLLMClientForProvider` could be imported statically alongside `getLLMClient`.

**Classification:** STORY-RELATED
