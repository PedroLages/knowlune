## What Works Well

1. **Clean VTT parser implementation.** The inline `parseVTT()` in `TranscriptPanel.tsx` is well-structured, handles both `HH:MM:SS.mmm` and `MM:SS.mmm` formats, uses SRT comma-to-period normalization, and correctly skips blocks without timestamp lines. No third-party dependency needed.

2. **Proper abort pattern on fetch.** The `cancelled` boolean in the `useEffect` cleanup (TranscriptPanel lines 67-85) correctly prevents state updates on unmounted components. This is a textbook React async cleanup pattern.

3. **Good component extraction.** Pulling the progress bar into `ChapterProgressBar` with a hidden `<input type="range">` for keyboard accessibility is a solid architectural decision. The z-index layering (z-10 for range, z-20 for markers) allows tooltips to work while the range input handles seeking.

#### B2. `captions` prop NOT passed from LessonPlayer to VideoPlayer -- subtitle rendering broken

**[src/app/pages/LessonPlayer.tsx:234-250]**: The `VideoPlayer` component accepts a `captions?: CaptionTrack[]` prop and renders `<track>` elements for subtitle display (VideoPlayer lines 554-563). However, `LessonPlayer` never passes the `captions` prop:

```tsx
<VideoPlayer
  src={getResourceUrl(videoResource)}
  title={lesson.title}
  // ... other props
  chapters={videoChapters}     // <-- chapters ARE passed
  // captions={???}            // <-- captions NOT passed
  onTimeUpdate={handleTimeUpdate}
  // ...
/>
```

The `captionSrc` is extracted (line 113) and used for TranscriptPanel, but the full `captions` array is never threaded to VideoPlayer. This means:

- The captions toggle button in VideoPlayer never renders (guarded by `captions && captions.length > 0` at line 738).
- No `<track>` elements are rendered on the `<video>` element.
- Subtitle functionality is silently broken for any lesson with captions data.

**Why**: Learners adding caption data to their courses see a transcript panel but can't toggle actual video subtitles. The two features should work together.

**Fix**: Add `captions={videoResource.metadata?.captions}` to the VideoPlayer invocation in LessonPlayer.

### High Priority

#### H1. Inline style on TranscriptPanel container violates "Tailwind utilities only" convention

**[src/app/pages/LessonPlayer.tsx:414]**:
```tsx
<div className="bg-card rounded-2xl shadow-sm overflow-hidden" style={{ height: '400px' }}>
```

This is the only `style={}` attribute in any new component in this PR. The project convention (per CLAUDE.md) is "All styling uses Tailwind utility classes." The height should use a Tailwind class.

**Why**: Inline styles bypass the design system, aren't responsive, and can't be overridden via Tailwind's responsive modifiers. A fixed 400px height also doesn't adapt to viewport -- on mobile it may be too tall, on desktop too short.

**Fix**: Replace with `h-[400px]` or better, a responsive height like `h-80 lg:h-[400px]`. If the transcript panel needs to fill available space, consider `max-h-[400px]` with a flex layout.

#### H2. `onTimeUpdate` fires ~4x/second, causing `setVideoCurrentTime` to trigger re-renders on the entire LessonPlayer tree

**[src/app/pages/LessonPlayer.tsx:135-144]**: The `handleTimeUpdate` callback calls `setVideoCurrentTime(time)` on every `timeupdate` event from the HTML video element (typically fires every 250ms during playback). This state lives in the LessonPlayer component, which owns the entire page tree.

```tsx
const handleTimeUpdate = useCallback(
  (time: number) => {
    setVideoCurrentTime(time)  // ~4 re-renders per second of entire LessonPlayer
    if (courseId && lessonId && time - lastSaveTimeRef.current >= 5) {
      lastSaveTimeRef.current = time
      saveVideoPosition(courseId, lessonId, time)
    }
  },
  [courseId, lessonId]
)
```

Every 250ms, the entire LessonPlayer re-renders -- including ModuleAccordion, NoteEditor, BookmarksList, PdfViewer, navigation buttons, etc. Only TranscriptPanel actually needs `currentTime`.

**Why**: On lower-end devices, this causes noticeable jank during video playback. The LessonPlayer component tree is large.

**Fix**: Use a ref for `videoCurrentTime` instead of state, and only trigger a re-render when the active cue actually changes. Alternatively, store `videoCurrentTime` in a ref, and pass it to TranscriptPanel via a separate mechanism (e.g., a small context provider that only TranscriptPanel subscribes to, or move the `activeCue` computation into LessonPlayer and only pass `activeCueIndex` as a prop).

A simpler partial fix: throttle `setVideoCurrentTime` to once per second (the granularity at which cues typically change):

```tsx
const lastSetTimeRef = useRef(0)
const handleTimeUpdate = useCallback(
  (time: number) => {
    if (time - lastSetTimeRef.current >= 1) {
      lastSetTimeRef.current = time
      setVideoCurrentTime(time)
    }
    // position saving stays at 5s interval
    if (courseId && lessonId && time - lastSaveTimeRef.current >= 5) {
      lastSaveTimeRef.current = time
      saveVideoPosition(courseId, lessonId, time)
    }
  },
  [courseId, lessonId]
)
```

#### H3. TranscriptPanel auto-scroll uses `scrollIntoView({ behavior: 'smooth' })` without respecting `prefers-reduced-motion`

**[src/app/components/figma/TranscriptPanel.tsx:94-98]**:
```tsx
useEffect(() => {
  if (activeCueRef.current) {
    activeCueRef.current.scrollIntoView({ behavior: 'smooth', block: 'nearest' })
  }
}, [activeCue])
```

This smooth scroll animation fires every time the active cue changes (every few seconds during playback). Users with `prefers-reduced-motion` should get `behavior: 'auto'` instead.

**Why**: WCAG 2.1 guideline 2.3.3 (Animation from Interactions). This is a recurring pattern in this codebase (noted in S05/S06 reviews). Users with vestibular disorders may find constant smooth scrolling disorienting.

**Fix**:
```tsx
const prefersReducedMotion = window.matchMedia('(prefers-reduced-motion: reduce)').matches
activeCueRef.current.scrollIntoView({
  behavior: prefersReducedMotion ? 'auto' : 'smooth',
  block: 'nearest',
})
```

#### H4. Zero unit tests for ChapterProgressBar and TranscriptPanel

No unit test files exist for either new component:
- `tests/unit/**/ChapterProgressBar*` -- not found
- `tests/unit/**/TranscriptPanel*` -- not found

The E2E tests cover integration but don't test:
- ChapterProgressBar: `formatTime()` edge cases (0 seconds, hours-long videos), chapter markers at exactly 0% or 100% (filtered out -- is this correct?), bookmark click handler, keyboard seeking via range input
- TranscriptPanel: `parseVTT()` with malformed input, missing WEBVTT header, empty cue text, overlapping cue times, `parseTime()` with edge formats, error state rendering, loading state rendering

**Why**: VTT parsing is the most likely place for bugs in production (malformed user-provided VTT files). Without unit tests, edge cases in the parser will silently fail.

**Fix**: Add unit tests for at minimum: `parseVTT()`, `parseTime()`, `formatTime()`, and `formatCueTime()`. Export these functions from the module for testing, or test via component render tests.

#### M3. VTT parser does not strip VTT styling tags from cue text

**[src/app/components/figma/TranscriptPanel.tsx:36]**:
```tsx
cues.push({ startTime, endTime, text: textLines.join(' ') })
```

WebVTT cue text can contain inline tags like `<b>`, `<i>`, `<u>`, `<c.classname>`, and `<ruby>`. These are rendered as literal text in the transcript panel because React escapes them (no XSS risk), but they look ugly: a cue like `<b>Important:</b> This is key` renders as the literal string including angle brackets.

**Fix**: Strip VTT tags before storing cue text:
```tsx
const cleanText = textLines.join(' ').replace(/<[^>]+>/g, '')
cues.push({ startTime, endTime, text: cleanText })
```

#### M4. `activeCue` detection uses reference equality (`cue === activeCue`), fragile if cues array is ever recreated

**[src/app/components/figma/TranscriptPanel.tsx:112]**:
```tsx
const isActive = cue === activeCue
```

This works correctly because `cues` state is stable (set once on fetch) and `activeCue` is derived via `useMemo` which returns the same object reference from the same array. However, if cues were ever re-parsed (e.g., if `src` changes), the reference equality would briefly be incorrect during the transition. Using index-based comparison would be more robust.

**Fix**: Use `idx === activeCueIndex` where `activeCueIndex` is the memoized index:
```tsx
const activeCueIndex = useMemo(
  () => cues.findIndex(c => currentTime >= c.startTime && currentTime < c.endTime),
  [cues, currentTime],
)
```

#### M5. `TranscriptPanel` shows "Loading transcript..." indefinitely if VTT file is empty or contains only the WEBVTT header

**[src/app/components/figma/TranscriptPanel.tsx:104-106]**:
```tsx
if (!cues.length) {
  return <div className="p-4 text-sm text-muted-foreground">Loading transcript...</div>
}
```

If the fetch succeeds but `parseVTT()` returns an empty array (valid VTT with no cues, or just the WEBVTT header), the component permanently shows "Loading transcript..." because `error` is false and `cues.length` is 0. There's no way to distinguish "still loading" from "loaded but empty."

**Fix**: Add a `loaded` state:
```tsx
const [loaded, setLoaded] = useState(false)
// In .then: setLoaded(true); setCues(parseVTT(text))
// Then:
if (!loaded) return <div>Loading transcript...</div>
if (!cues.length) return <div>No transcript cues found</div>
```
