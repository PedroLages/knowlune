## Code Review: E11-S06 — Per-Course Study Reminders

### What Works Well

1. **Clean separation of concerns.** The scheduling engine (`courseReminders.ts`), scheduling hook (`useCourseReminders.ts`), and UI components are well-isolated. The hook lives in Layout.tsx and operates via a ref + interval pattern that avoids stale closures -- a mature approach.

2. **Proper notification dedup and cleanup.** The `useCourseReminders` hook correctly clears the interval on unmount and re-initializes on `course-reminders-updated` events. The localStorage-based per-course dedup keys prevent duplicate notifications across tabs. The `sendCourseReminder` function gates on permission before constructing the Notification.

3. **Solid accessibility throughout.** DaySelector uses `role="checkbox"` with `aria-checked`, pills have `aria-label` for full day names, Switch has course-name-aware aria-labels, permission prompts use `role="alert"` with `aria-live="polite"`, and all interactive elements meet the 44px touch target minimum.

### Findings

#### Blockers

- **[src/lib/courseReminders.ts:109] (confidence: 85)**: AC2 requires the notification to include "a direct link to resume studying that course." The `data: { url: ... }` property on the Notification is set, but there is **no `click` event handler** registered on the Notification object to actually navigate the user. `new Notification(...)` creates a notification that does nothing when clicked -- the `data` property is inert without a handler. Why: Learners see a notification, tap it, and nothing happens. The deep-link AC is unmet. Fix: Either register a `notificationclick` handler via a Service Worker, or attach an `onclick` handler to the Notification instance:
  ```ts
  const notification = new Notification(...)
  notification.onclick = () => {
    window.focus()
    window.location.href = `/courses/${reminder.courseId}`
  }
  ```
  Note: `notification.onclick` works for foreground notifications; a Service Worker `notificationclick` listener is needed for background notifications.

- **[src/app/components/figma/CourseReminderSettings.tsx:120-139] (confidence: 80)**: `handleSaveNew` is an async function called from an `onClick` handler with **no try/catch**. If `saveCourseReminder` (a Dexie write) throws, the promise rejection is unhandled. Why: A learner who configures a reminder and gets a silent IndexedDB error will think it saved (the form closes because `setShowAddForm(false)` is never reached, but the error is not surfaced). Fix: Wrap in try/catch, surface error via toast or inline error state:
  ```ts
  async function handleSaveNew() {
    if (!selectedCourseId || selectedDays.length === 0) return
    const course = courses.find(c => c.id === selectedCourseId)
    if (!course) return
    try {
      await saveCourseReminder(reminder)
      setShowAddForm(false)
    } catch (error) {
      console.error('[CourseReminderSettings] Failed to save:', error)
      // surface to user via toast or state
    }
  }
  ```

#### High Priority

- **[Recurring] [src/app/components/figma/CourseReminderSettings.tsx:142-148] (confidence: 85)**: `handleToggle` and `handleSaveEdit` are async handlers without try/catch. Both call Dexie operations that can throw. If `toggleCourseReminder` fails, the Switch UI will have already toggled (optimistic via `onCheckedChange`) but the DB won't reflect it. Pattern from: E03-S03 (fire-and-forget async). Fix: Add try/catch, revert UI state on failure, and surface error to user.

- **[src/app/hooks/useCourseReminders.ts:30] (confidence: 80)**: The scheduler **never starts** if notifications aren't granted at mount time, and there's no mechanism to re-check permission later. If a learner configures reminders (AC4 allows saving without permission), then later grants permission via browser settings, the scheduler won't activate until the page is fully reloaded. Why: AC4 says "it activates once permissions are granted" -- but there's no visibility change listener, permission change listener, or periodic permission re-check. Fix: Add a `setInterval` or `navigator.permissions.query('notifications').addEventListener('change', ...)` to detect permission grants, or at minimum re-check permission on each interval tick rather than gating the entire scheduler startup.

- **[src/lib/courseReminders.ts:64-72] (confidence: 78)**: `shouldFireReminder` compares against a 2-minute window (`diff >= 0 && diff <= 1`), but the interval runs every 60 seconds. If the interval ticks at minute 0 and minute 1 relative to target, this works. However, if the JS event loop delays the interval (common under heavy tab load), a tick could land at minute 2, missing the window entirely. The comment says "2-minute window" but the code checks `diff <= 1` (actually a 1-minute window: minute 0 and minute 1). Why: Learners could miss notifications if the browser throttles background tab timers. Fix: Widen the window to `diff <= 2` (true 2-minute tolerance) to match the comment and account for timer drift.

- **[src/app/components/figma/CourseReminderSettings.tsx:70-85] (confidence: 75)**: `handleAddClick` checks `currentPerm === 'default'` and returns early (line 77), showing the permission prompt but never opening the add form. If the user dismisses the browser's native permission dialog without choosing (e.g., clicking away), the component is stuck showing the permission prompt with no way to proceed. The "Continue without notifications" button only appears when `showDeniedGuidance` is true, which requires the user to click "Enable Notifications" first. Why: A learner who cancels the permission prompt must reload the page. Fix: Add a "Skip for now" or "Continue without" option to the permission prompt state, not just the denied state.

#### Medium

- **[src/lib/courseReminders.ts:45] (confidence: 72)**: `toggleCourseReminder` uses `new Date().toISOString()` directly. While this is production code (not a test), it makes unit testing the function non-deterministic. Fix: Accept an optional `now` parameter for testability:
  ```ts
  export async function toggleCourseReminder(id: string, enabled: boolean, now = new Date()): Promise<void>
  ```

- **[src/app/components/figma/CourseReminderSettings.tsx:126] (confidence: 70)**: Same pattern -- `new Date().toISOString()` in `handleSaveNew`. The `CourseReminderRow.tsx:62` has the same issue. This makes snapshot testing and time-dependent logic harder to verify. Fix: Inject time or use a utility function.

- **[src/app/components/figma/CourseReminderSettings.tsx:153-154] (confidence: 72)**: The Card does not use `rounded-[24px]` as specified in the project design conventions for cards. It inherits the shadcn Card default `rounded-lg`. Fix: Add `className="rounded-[24px]"` to the Card, consistent with other Settings cards if they follow this convention.

- **[src/app/hooks/useCourseReminders.ts:33-42] (confidence: 70)**: The 60-second interval runs continuously even when no reminders are enabled. All reminders could be disabled, but the interval keeps ticking every minute, checking `reminder.enabled` in a loop. This is minor but wasteful. Fix: Filter to enabled reminders before starting the scheduler. If none are enabled, skip starting the interval entirely.

- **[tests/e2e/story-e11-s06.spec.ts:19-32] (confidence: 68)**: The `createImportedCourse` helper duplicates logic from the existing test factory pattern. The project has `tests/support/fixtures/factories/` for shared factories. Fix: Consider extracting to `tests/support/fixtures/factories/course-factory.ts` if one doesn't exist, or use the existing seeding helpers more consistently.

#### Nits

- **Nit** [src/app/components/figma/CourseReminderSettings.tsx:304] (confidence: 65): The time `<input>` is styled with raw Tailwind border/focus classes rather than using shadcn's `<Input>` component. The same pattern appears in CourseReminderRow.tsx:92-98. Using `<Input type="time" />` would be more consistent with the component library.

- **Nit** [src/lib/courseReminders.ts:14-22] (confidence: 60): `DAY_NAMES` starts with Sunday (JS `getDay()` convention), but the DaySelector and CourseReminderRow `DAY_ORDER` start with Monday. While both are correct for their purposes, the inconsistency could confuse future maintainers. Consider adding a comment explaining the Sunday-first order matches `Date.getDay()`.

- **Nit** [src/data/types.ts:322] (confidence: 55): The comment says "Spaced Review System (Story 11.1)" for `DayOfWeek`, but this type was introduced/reused for course reminders (E11-S06). The type itself is fine, but the section comment is potentially misleading if `DayOfWeek` was added as part of 11.1 and reused here.

### Recommendations

1. **First**: Fix the notification click handler (Blocker) -- without it, AC2's deep-link requirement is unmet.
2. **Second**: Add try/catch to all async event handlers (`handleSaveNew`, `handleToggle`, `handleSaveEdit`) -- these are silent failure vectors.
3. **Third**: Address the permission re-activation gap (High) -- the scheduler should eventually start when permissions are later granted, per AC4.
4. **Fourth**: Fix the `shouldFireReminder` window to actually be 2 minutes as the comment states.
5. **Fifth**: Add an escape hatch from the permission prompt state (High) -- users who dismiss the native dialog shouldn't be stuck.
