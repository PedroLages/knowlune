## Code Review: E12-S01 — Create Quiz Type Definitions

### What Works Well

1. **Schema-first design with inferred types.** Using `z.infer<typeof Schema>` instead of duplicating interface definitions eliminates the category of bugs where types and runtime validation drift apart. This is exactly the right pattern for a quiz subsystem spanning Epics 12-18.

2. **Thorough refinement logic.** The `QuestionSchema.refine()` block enforces type-specific option constraints (MC 2-6, TF exactly 2, MS 2-6, FIB absent) with a single refinement pass. The boundary tests (2 min, 6 max, 7 rejection) are well-chosen.

3. **Comprehensive test coverage.** 48 tests covering all four question types, boundary values for passingScore (0 and 100), timer accommodations, nullable fields, and type inference. The test factory pattern (`makeMCQuestion`, etc.) keeps tests readable and maintainable.

### Findings

#### High Priority

- **[src/types/quiz.ts:101] (confidence: 92)**: `Question` type is inferred from `BaseQuestionSchema` instead of `QuestionSchema`, which means the exported `Question` type does not reflect the refinement constraints. While Zod refinements cannot narrow the TypeScript type (a `.refine()` returns `ZodEffects` with the same inferred type), the real issue is **semantic inconsistency**: consumers import `Question` believing it represents a validated question, but `z.infer<typeof BaseQuestionSchema>` is identical to `z.infer<typeof QuestionSchema>` at the type level anyway. The problem surfaces when a consumer does `QuestionSchema.parse(data)` — the return type is the BaseQuestionSchema shape, not a branded/narrowed type. This is a non-issue today but becomes one if someone tries to use the `Question` type as a discriminated union in Epic 14+ scoring logic. **More critically**, if `BaseQuestionSchema` is ever modified independently from `QuestionSchema`, the exported type would diverge from what passes validation. Why: Future quiz stories building on these types may assume `Question` = "passed QuestionSchema validation", leading to subtle correctness bugs. Fix: Either (a) export `Question` as `z.infer<typeof QuestionSchema>` for clarity (functionally identical today but self-documenting), or (b) add a JSDoc on the `Question` type noting it is the *structural* type and that runtime validation requires `QuestionSchema.safeParse()`.

- **[src/types/quiz.ts:130-132] (confidence: 85)**: `createdAt` and `updatedAt` use `z.string()` with no format validation. Any string passes — including empty strings, "not-a-date", or "hello". Why: When quiz data is loaded from IndexedDB or localStorage (crash recovery), malformed timestamps will silently propagate to the UI, potentially crashing date formatting code downstream (e.g., `new Date("not-a-date")` produces `Invalid Date`). Fix: Use `z.string().min(1)` at minimum for non-empty enforcement, or ideally `z.string().datetime()` (Zod v3.22+) which validates ISO 8601 format. This matches the JSDoc's "ISO 8601" promise.

- **[src/types/quiz.ts:86-99] (confidence: 82)**: The single `.refine()` produces a generic error message `"Question options do not match the expected constraints for this question type"` for all four failure modes. When a learner or developer encounters a validation error, they cannot tell whether the issue is "MC needs 2-6 options" vs "FIB must not have options" vs "TF needs exactly 2". Why: Downstream quiz creation/editing flows (Epic 13+) will use these schemas for form validation. Generic messages force developers to reverse-engineer which constraint failed. Fix: Use `.superRefine()` with `ctx.addIssue()` per case to produce type-specific error messages, e.g., `"Multiple-choice questions require 2-6 options, got ${q.options?.length ?? 0}"`.

- **[src/types/quiz.ts:76] (confidence: 78)**: `correctAnswer: z.union([z.string(), z.array(z.string())])` accepts empty strings and empty arrays as valid answers. A question with `correctAnswer: ""` or `correctAnswer: []` would pass validation. Why: This is a data integrity risk — a quiz author could accidentally save a question with no correct answer, making it impossible to score. Fix: Use `z.union([z.string().min(1), z.array(z.string().min(1)).min(1)])` to ensure at least one non-empty correct answer exists.

#### Medium

- **[src/types/quiz.ts:118] (confidence: 75)**: `QuizSchema.questions` uses `z.array(QuestionSchema).min(1)`, but `QuestionSchema` is a `ZodEffects` (due to `.refine()`), not a `ZodObject`. This works correctly in Zod, but it means `QuizSchema` itself becomes opaque to schema introspection tools (`.shape` access, `.pick()`, `.omit()`, `.extend()`) that later stories may need for partial quiz editing forms. Fix: Consider using `.superRefine()` on the full `QuizSchema` instead, keeping `BaseQuestionSchema` (a ZodObject) in the questions array and applying question-type refinements at the quiz level. Alternatively, document this limitation for future story implementers.

- **[src/types/__tests__/quiz.test.ts:78] (confidence: 74)**: `makeQuiz()` casts questions with `as unknown as Quiz['questions']`. This double-cast (`Partial<Question>[]` -> `unknown` -> refined type) bypasses TypeScript's type safety in tests. The cast is needed because `QuestionSchema` produces `ZodEffects` whose inferred output type differs from `Question`. Why: If the schema shape changes, tests will still compile and pass even if the factory data is structurally wrong — the cast hides the mismatch. Fix: Build quiz test data by passing raw objects directly through `QuizSchema.safeParse()` and asserting success, rather than constructing typed objects and casting. Alternatively, type the factory return as `z.input<typeof QuestionSchema>` (the input type).

- **[src/types/quiz.ts:195] (confidence: 72)**: `QuizProgressSchema.answers` uses `z.record(z.string(), z.union([z.string(), z.array(z.string())]))` which allows empty string keys. A crash-recovery restore with `{ "": "answer" }` in the answers record would be accepted. Fix: Use `z.record(z.string().min(1), ...)` to ensure answer keys are valid question IDs.

- **[src/types/quiz.ts:160-181] (confidence: 70)**: `QuizAttemptSchema` has no cross-field validation between `score`, `percentage`, `passed`, and `answers`. A QuizAttempt with `score: 0, percentage: 100, passed: true, answers: []` passes validation. Why: While the schema intentionally validates structure not business logic, this means corrupted attempt data (e.g., from a bug in scoring code) will be silently persisted. Fix: Add a comment documenting that business-rule consistency (score matches sum of pointsEarned, percentage = score/total * 100, passed = percentage >= passingScore) is enforced at the service layer, not the schema layer. This prevents future developers from assuming the schema guarantees consistency.

#### Nits

- **Nit** [src/types/quiz.ts:78] (confidence: 65): `explanation: z.string()` allows empty strings. Consider `z.string().min(1)` or `.optional()` — an empty explanation provides no learning value but passes validation silently.

- **Nit** [src/types/quiz.ts:45] (confidence: 60): `QuestionMediaSchema.url` validates `z.string().min(1)` but does not validate URL format. A value like "not-a-url" would pass. Consider `z.string().url()` for stricter validation, especially since media URLs will be used in `<img src>` / `<video src>` attributes.

- **Nit** [src/types/__tests__/quiz.test.ts:17-70] (confidence: 55): Test factories are defined inline rather than in `tests/support/fixtures/factories/`. For a pure types story this is acceptable, but if these factories are needed in E2E tests for later quiz stories, they should be extracted to the shared factory location to avoid duplication.

### Recommendations

1. **First**: Add `z.string().min(1)` or `z.string().datetime()` validation to `createdAt`/`updatedAt` — this is the highest-risk gap since malformed dates will crash downstream UI code.

2. **Second**: Tighten `correctAnswer` to reject empty strings/arrays — a quiz with no correct answer is a data integrity bug that is hard to diagnose later.

3. **Third**: Replace the single `.refine()` with `.superRefine()` and type-specific error messages — this directly impacts developer experience for Epic 13+ quiz creation forms.

4. **Fourth**: Add a JSDoc comment on the `Question` type export clarifying its relationship to `BaseQuestionSchema` vs `QuestionSchema`.

5. **Fifth**: Document the intentional gap between structural validation (schema) and business-rule validation (service layer) in `QuizAttemptSchema`.
