## Findings

### Blockers

**[B1] `src/lib/studySchedule.ts:44` — locale-dependent `toLocaleDateString()` (confidence: 95)**

`toLocaleDateString()` is called without a locale argument, while the rest of the codebase uses `toLocaleDateString('sv')` (via `toLocalDateString()` from `@/lib/dateUtils`). Without a locale, the date string format is system-dependent, meaning the same timestamp could be bucketed into different day strings on different machines. This directly affects the `getDistinctStudyDays` count, which gates whether the widget shows "insufficient data" (AC2).

Fix: Import and use `toLocalDateString` from `@/lib/dateUtils` instead of bare `new Date(entry.timestamp).toLocaleDateString()`.

---

**[B2] `src/lib/studySchedule.ts:159-160` — non-null assertions on nullable values (confidence: 92)**

`goal!` is used after checking `weeklyMinutes === null`, but `weeklyMinutes` being non-null does not guarantee `goal` is non-null. More critically, `dailyMinutes!` on line 160 is asserted non-null, but `calculateDailyStudyDuration` can return `null` for session-count goals — while the current flow should prevent this, a future refactor could break this assumption silently.

A null `dailyMinutes` passed to `allocateTimeAcrossCourses` would cause `NaN` allocations, showing broken data to learners.

Fix: Add explicit null checks with early returns or use the already-computed `weeklyMinutes` to derive `dailyMinutes` inline.

### High Priority

**[H1] `tests/e2e/story-e07-s05.spec.ts:17-19` — incorrect timestamp formula in `makeStudyLog` (confidence: 90)**

The formula `Date.now() - ((i % daySpread) * 86400000) + hour * 3600000` adds `hour * 3600000` to `Date.now()`, which already contains the current time-of-day. Entries for `i=0` are ~9 hours in the *future* relative to the current wall clock — the computed "hour of day" won't be `9`, it will be `currentHour + 9`.

The E2E tests pass because they only assert visibility, not the specific hour value shown. This makes AC1 tests weaker than they appear.

Fix: Construct timestamps using `new Date(year, month, day, hour, 0, 0)` with explicit date components, subtracting days from a known base date.

---

**[H2] `src/lib/studySchedule.ts:110-126` — rounding doesn't preserve total duration (confidence: 85)**

`allocateTimeAcrossCourses` applies `Math.round` independently to each course's proportional share, so the sum of allocated minutes can differ from `dailyMinutes`. Example: 3 courses with equal scores and `dailyMinutes=100` gives `100/3=33.33`, rounded to 33 each = 99 total (1 minute lost).

Learners see per-course allocations that don't add up to the displayed daily total.

Fix: Use largest-remainder allocation — compute floor values, then distribute remaining minutes one-at-a-time to courses with the largest fractional remainders.

---

**[H3] `src/app/components/StudyScheduleWidget.tsx:28-47` — stale closure on session store reads (confidence: 78)**

`buildActiveCoursesWithMomentum()` calls `useSessionStore.getState()` imperatively inside a `useCallback` with empty deps. If `useSessionStore` state updates after mount (e.g. a lesson is completed), the callback won't reflect the new sessions because `refresh` is memoized once.

Fix: Either subscribe to `useSessionStore` reactively (hook form), remove the `useCallback` memoization, or add relevant store values to the dependency array.

---

**[H4] `src/lib/studySchedule.ts:79-84` — implicit unit assumption (confidence: 82)**

`getWeeklyGoalMinutes` returns `goal.target` for weekly time goals and `goal.target * 7` for daily time goals. The implementation is self-consistent (minutes throughout) but the AC3 says "weekly goal hours" — there is potential for a future developer to misread the interface and introduce a units bug silently.

Fix: Add a JSDoc comment clarifying all values are in minutes throughout the pipeline.
