### Findings

#### Blockers

- **Blocker** `vite.config.ts` ŌĆö `registerType` and `swSrc` mismatch in spec code block (confidence: 95): The spec's `VitePWA()` code block at lines 146-181 shows `registerType: 'prompt'` and `swSrc: 'src/sw.ts'`. These are stale values from E61-S01. The current codebase has `registerType: 'autoUpdate'` and `swSrc: 'sw.ts'` (with `srcDir: 'src'`). This creates two failure scenarios for the implementer:

  - **Merge failure**: Developer keeps the existing `srcDir: 'src'` but changes `swSrc` to `'src/sw.ts'` as shown in the spec, resolving to `src/src/sw.ts`. Build fails ŌĆö this file does not exist.

  - **Replace failure**: Developer copies the entire `VitePWA()` block verbatim, losing `registerType: 'autoUpdate'` and changing it to `'prompt'`. Every SW update would require manual user approval via `PWAUpdatePrompt` ŌĆö a UX regression where users must click approve on each deployment.

  - **Root cause**: The spec code block was drafted from E61-S01 and not reconciled with the current config, which was changed to `autoUpdate` and `swSrc: 'sw.ts'` / `srcDir: 'src'` at some point after E61-S01.

  - **Fix**: Change `registerType` to `'autoUpdate'` and `swSrc` to `'sw.ts'` (preserving the existing `srcDir: 'src'`). The spec comment should flag this explicitly: "Note: keep `registerType: 'autoUpdate'` ŌĆö do not change from the current config."

  - **Effort**: ~2 minutes
  - **Category**: [Correctness] [AI Smell]
  - **Autofix class**: `gated_auto`

#### High Priority

- **High** `ChunkErrorBoundary` ŌĆö non-chunk error re-throw requires `throw in render()` pattern (confidence: 85): The spec (line 281) states non-chunk errors "must be re-thrown to the parent RouteErrorBoundary so they are reported via `reportError()`". However, React error boundary semantics mean that throwing in `componentDidCatch` does NOT propagate to a parent error boundary. The correct mechanism is `throw this.state.error` in the `render()` method after the error type check:

  ```tsx
  render() {
    if (this.state.hasError) {
      if (this.isChunkLoadError(this.state.error)) {
        // Chunk error: return the appropriate fallback
        if (!navigator.onLine) return <OfflineRouteFallback />;
        // Online chunk failure ŌĆö call reportError and show generic error
        return <GenericChunkError onRetry={this.handleRetry} />;
      }
      // Non-chunk error: propagate to RouteErrorBoundary
      throw this.state.error;
    }
    return this.props.children;
  }
  ```

  If the implementer instead `throw`s in `componentDidCatch` or tries `this.setState({ hasError: false })` to re-render children, the error is silently swallowed ŌĆö no fallback from any boundary, no `reportError()` call, just a blank content area.

  - **Why**: Non-chunk errors in lazy routes (e.g., `TypeError: Cannot read properties of undefined`) would be silently lost, with no feedback to the user or to error monitoring.

  - **Fix**: Document the `throw in render()` pattern explicitly in Task 5.4, and add a unit test that non-chunk TypeError errors propagate to RouteErrorBoundary (verify `RouteErrorFallback` renders, not `OfflineRouteFallback`).

  - **Effort**: ~15 minutes (documentation + unit test)
  - **Category**: [Correctness]
  - **Autofix class**: `manual`

- **High** `ChunkErrorBoundary` ŌĆö missing `reportError()` for chunk errors when online (confidence: 80): The spec defines two states for chunk errors:
  - **Offline** + chunk error: show `OfflineRouteFallback`. No `reportError()` call ŌĆö acceptable because offline is expected.
  - **Online** + chunk error: "Show a generic error with retry button when online" (line 283).

  For the online case, the `ChunkErrorBoundary` catches the error before `RouteErrorBoundary`, so `RouteErrorBoundary`'s `componentDidCatch` with `reportError()` never fires. The spec does not instruct `ChunkErrorBoundary` to call `reportError()` for the online case. If a CDN issue, corrupted deployment asset, or server error causes chunks to return 404s while the user IS online, this error would be invisible to monitoring ŌĆö the user sees a generic error message but the development team never knows.

  - **Fix**: Add `reportError(error, 'ChunkErrorBoundary')` call when the error IS a chunk load failure AND `navigator.onLine` is true. Import `reportError` from `@/lib/errorTracking`. This mirrors how `RouteErrorBoundary` reports errors.

  - **Effort**: ~5 minutes
  - **Category**: [Silent Failure]
  - **Autofix class**: `manual`

- **High** `src/sw.ts` ŌĆö font runtime caching rule lacks `CacheableResponsePlugin` (confidence: 75): The proposed font `registerRoute` (lines 196-201) uses `CacheFirst` without `CacheableResponsePlugin`. While Workbox's default only caches status 200 responses, the absence of an explicit `CacheableResponsePlugin` means:
  - **Partial cache slot consumption**: Failed font requests (transient 500, auth redirect, 404) are attempted but not cached. However, the `ExpirationPlugin` tracks the entry against its `maxEntries: 50` limit, so the failed entry occupies a slot. After 50 unique font URLs with at least one failure each, subsequent successful fonts are evicted.
  - **Inconsistency**: Every other `CacheFirst`/`StaleWhileRevalidate` rule in `sw.ts` (local-images, unsplash-images, hf-models) uses `CacheableResponsePlugin({ statuses: [0, 200] })`. The font rule is the sole exception.

  - **Why**: If a user visits a page online while font CDN is transiently degraded, the font cache fills with failed entries. Later, when the user goes offline, fonts fail to load (because no successful cache entry was ever created) ŌĆö causing FOUT (flash of unstyled text).

  - **Fix**: Add `new CacheableResponsePlugin({ statuses: [0, 200] })` to the font caching rule's `plugins` array, consistent with all other runtime caching rules.

  - **Effort**: ~2 minutes
  - **Category**: [Correctness]
  - **Autofix class**: `gated_auto`

#### Medium

- **Medium** `ChunkErrorBoundary` ŌĆö online/offline oscillation could cause re-render loop (confidence: 70): The spec (line 284) says "Reset error state when going back online". On an unreliable connection that flaps between online/offline:
  1. User navigates offline -> chunk error caught -> shows `OfflineRouteFallback`
  2. Goes online -> boundary resets, React re-renders children, lazy import re-triggers
  3. Connection drops mid-fetch -> import fails again -> shows `OfflineRouteFallback` again
  4. Goes online again -> resets and retries again ŌĆö ad infinitum while connection flaps

  Each cycle causes re-render of `Suspense` + `PageLoader` + component mount/unmount. On slow devices (phones, low-end laptops), this wastes battery and causes UI flickering. With rapid oscillation (~5-10 seconds), the user sees a flashing UI.

  - **Fix**: Track retry count in `ChunkErrorBoundary` state. After N consecutive failures (suggest N=3), switch to a persistent error state that requires manual "Retry" button click, with exponential backoff before re-enabling auto-retry on the `online` event.

  - **Effort**: ~30 minutes
  - **Category**: [Correctness]
  - **Autofix class**: `manual`

- **Medium** `vite.config.ts` ŌĆö `globPatterns` excludes `assets/*.svg` (confidence: 70): The proposed `globPatterns` includes `*.svg` (matches root SVGs like `favicon.svg`) but NOT `assets/*.svg`. Vite emits imported SVGs (e.g., `import logoUrl from './logo.svg'`) to `dist/assets/logo-[hash].svg` with a hashed URL. These per-asset SVGs are NOT matched by `*.svg`. If a lazy-loaded component references an imported SVG and that component's chunk is runtime-cached via the route-chunks rule, the SVG URL within the chunk points to `/assets/logo-[hash].svg` which has no cache entry. Offline, the browser fetches it and fails ŌĆö the SVG renders as a broken image.

  - **Impact**: Low ŌĆö most UI icons use `lucide-react` (inline SVGs, not file imports). Asset SVGs are typically small logos or decorative elements. Failures manifest as missing images on lazy routes offline, not data loss. Impact depends on how many imported SVGs exist in lazy-loaded components.

  - **Fix**: Add `'assets/*.svg'` to `globPatterns`. SVGs are typically <20KB each, so the precache size impact is negligible.

  - **Effort**: ~1 minute
  - **Category**: [Correctness]
  - **Autofix class**: `safe_auto`

#### Nits

- **Nit** `src/app/routes.tsx` ŌĆö import style inconsistency (confidence: 80): The spec proposes `import { ChunkErrorBoundary } from '@/app/components/ChunkErrorBoundary'` but the existing file uses `'./components/RouteErrorBoundary'` (relative imports for co-located components in the same directory). Using `@/` for the new imports creates an inconsistent pattern within the same file.

  - **Fix**: Use `'./components/ChunkErrorBoundary'` and `'./components/OfflineRouteFallback'` to match the existing convention.

  - **Effort**: ~1 minute
  - **Category**: [Maintainability]
  - **Autofix class**: `safe_auto`

### Recommendations

1. **Fix Blocker first** ŌĆö correct `registerType` to `'autoUpdate'` and `swSrc` to `'sw.ts'` in the spec's injectManifest code block (lines 147, 172). Without this fix, the build either fails entirely (`src/src/sw.ts`) or silently changes SW update behavior (prompt on every deploy). This is a 2-minute edit that must happen before implementation starts.

2. **Document the `throw in render()` pattern** in Task 5.4. The implementer needs to know that re-throwing non-chunk errors must happen in `render()`, not `componentDidCatch`. Add a brief comment in the spec and a unit test verifying non-chunk errors propagate to `RouteErrorBoundary`.

3. **Add `reportError()` call** in `ChunkErrorBoundary` for online chunk failures. Without this, legitimate deployment/CDN errors that occur while the user is online go unmonitored.

4. **Add `CacheableResponsePlugin`** to the font caching rule to prevent transient failures from occupying cache slots and to align with the pattern used by every other caching rule.

5. **Address oscillation guard (Medium)** and **SVG pattern (Medium)** as time allows ŌĆö these are edge cases, not blocking.

6. **Use relative imports in `routes.tsx`** (Nit) ŌĆö matches existing convention and avoids an unnecessary diff.
