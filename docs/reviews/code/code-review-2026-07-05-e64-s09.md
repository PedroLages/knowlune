## Code Review: E64-S09 — Service Worker Precache Optimization

### Review Context

This is a **pre-implementation specification review** (0 code changes on the branch). The story is in `ready-for-dev` status. Findings focus on spec gaps, implementation risks, and acceptance criteria ambiguities that will surface during development if not addressed now.

### What Works Well

1. **Thorough dependency scoping**: The story clearly documents its dependency on E61-S01 (injectManifest migration), confirms it's already in place, and correctly identifies that it can run in parallel with E61-S02 (different sections of `src/sw.ts`). The updated note at the top shows the story was already refined once to target the right configuration location. Good hygiene.

2. **Explicit chunk name caveat**: The spec repeatedly warns that chunk name patterns are build-dependent and must be validated against actual build output in Task 1. The Implementation Notes section carries the same caveat. This prevents a common failure mode where hardcoded patterns don't match Vite's output.

3. **Well-structured subtasks**: The breakdown into 7 tasks with clear prerequisites (Task 1 must complete before Tasks 2-3) and logical ordering (build -> inspect -> verify) creates a clear implementation flow. The parallel between Task 6 (build verification) and Task 7 (offline behavior) is clean.

### Findings

#### Blockers

- **Font caching regression** (`vite.config.ts` line 549, `src/sw.ts` lines 66-81) (confidence: 90): The `globIgnores` in the proposed `injectManifest` config excludes `**/*.woff2` from precache, but **no runtime caching rule is added for fonts** in `src/sw.ts`. Currently, Knowlune loads variable fonts (Inter, Literata, DM Sans, Space Grotesk, Atkinson Hyperlegible) from `@fontsource-variable` packages — these are separate `.woff2` files in `dist/assets/` (not base64-inlined, as each is 20-50KB). With the change, fonts will hit `setDefaultHandler(new NetworkOnly())` on line 81, which means **fonts will not load offline at all**. The story's note says "woff2 files are excluded via globIgnores and cached at runtime instead" but no runtime rule is specified.
  - **Why**: This is a regression from current behavior where fonts load offline via precache. After the change, users on metered connections or offline will see system font fallback or a flash of unstyled text (FOUT) for all routes.
  - **Fix**: Add a `CacheFirst` runtime caching rule for font files before the chunk caching rule, OR add `woff2` back to `globPatterns` (fonts change rarely and benefit from precache), OR keep the approach but add the missing caching rule. Example:
    ```ts
    // f. Font files: CacheFirst, 50 entries, 30 days
    registerRoute(
      /\/assets\/.+\.woff2$/i,
      new CacheFirst({
        cacheName: 'fonts',
        plugins: [new ExpirationPlugin({ maxEntries: 50, maxAgeSeconds: 30 * 24 * 60 * 60 })],
      })
    );
    ```
  - **Effort**: ~5 minutes (add 7-line `registerRoute` call in `src/sw.ts` before the route-chunk rule).

#### High Priority

- **ChunkErrorBoundary integration underspecified** (`src/app/routes.tsx` lines 200-206, story line 253) (confidence: 85): The existing `SuspensePage` component wraps every lazy route in `RouteErrorBoundary > Suspense`. The proposed `ChunkErrorBoundary` needs to live **between** `RouteErrorBoundary` and `Suspense` to catch `TypeError` from failed dynamic `import()` calls. If placed outside `RouteErrorBoundary`, it's redundant. If placed inside `Suspense`, it will never catch chunk load errors (React throws these at the nearest error boundary above `Suspense`). The story says "Apply the boundary in routes.tsx or App.tsx" but doesn't specify the component hierarchy.
  - **Why**: Without this clarity, the implementer may place the boundary in the wrong position, rendering it ineffective. The `RouteErrorBoundary` currently shows a generic "Something went wrong" with a retry button that won't fix offline chunk failures — the whole purpose of `ChunkErrorBoundary` is to show a different message offline.
  - **Fix**: In the story's Implementation Notes (line 208-226) or in the `SuspensePage` definition, specify the exact wrapping order: `RouteErrorBoundary > ChunkErrorBoundary > Suspense`. Update the story's example from `{ element: (<ChunkErrorBoundary><Suspense>...</Suspense></ChunkErrorBoundary>) }` to show integration with `SuspensePage`.
  - **Effort**: ~10 minutes (spec clarification + one nested component adjustment in routes.tsx).

- **AC 3 overpromises offline route behavior** (story line 40-42) (confidence: 85): AC 3 says routes "display correctly" offline when only the JS chunk is runtime-cached. Most routes (Reports, Notes, Flashcards, Quiz, etc.) also require data from IndexedDB or API calls. Only having the JS chunk means the component renders, but the data-dependent UI (tables, charts, statistics) will show empty states or loading spinners that never resolve. This sets unrealistically high expectations for what "display correctly" means.
  - **Why**: The developer implementing this story will pass the specified Tasks but may not anticipate that AC 3 cannot pass for most routes as written. When QA tests offline Reports, the chunk will load but the page will be empty — a failing AC that requires scope clarification rather than code fixes.
  - **Fix**: Rephrase AC 3 from "displays correctly" to "the route chunk loads from cache and the page structure renders, even if data-dependent sections depend on runtime availability." Alternatively, accept that this AC only tests chunk caching and add a note that data persistence is out of scope.
  - **Effort**: 5 minutes (spec wording change) — saves potential hours of debugging "why does the route show empty offline."

#### Medium

- **Route-chunk JS pattern is overly broad** (story line 164-175) (confidence: 75): The `registerRoute` callback uses `/^\/assets\/.*\.js$/i` to match route chunks. This pattern catches ALL non-precached JS in `/assets/`, including any future build artifacts, debug scripts, or dynamically loaded non-route utilities. Workbox prioritizes precached responses for matching URLs, which mitigates the risk in practice. However, if a chunk is later added to precache, there would be a brief window where it's served via StaleWhileRevalidate instead of CacheFirst.
  - **Why**: While functional, the pattern is a design smell — it conflates "all uncached JS" with "route chunks." A more specific pattern (e.g., matching known route chunk name patterns from `manualChunks`) would better communicate intent and avoid future confusion.
  - **Fix**: Either (a) tighten the pattern to match known route chunk patterns (e.g., check the URL against a list of known route chunk prefixes), or (b) add a clarifying comment: "Intentionally broad — catches any JS chunk not in the precache manifest. Workbox serves precached assets first, so this is a safe fallback for dynamically-loaded route chunks."
  - **Effort**: ~2 minutes (add comment) or ~15 minutes (implement explicit pattern matching).

#### Nits

- **Nit** `h-12 w-12` should be `size-12` (story line 192) (confidence: 90): In the `OfflineRouteFallback` example code, `className="mx-auto h-12 w-12 text-muted-foreground mb-4"` uses the Tailwind v3 pattern for equal width/height. Tailwind v4 provides the `size-*` shorthand. Change to `className="mx-auto size-12 text-muted-foreground mb-4"`.

### Recommendations

1. **Fix the font caching regression (BLOCKER)** before starting implementation — this is the most critical issue and will cause offline font rendering to break.
2. **Clarify AC 3 wording (HIGH)** — determine what "display correctly" actually means for this story. Without this clarification, the implementer and reviewer will disagree on whether the AC passes.
3. **Specify the ChunkErrorBoundary integration point (HIGH)** — update the story's Implementation Notes to show exactly where it fits in the existing `SuspensePage` hierarchy.
4. **Address the route-chunk JS pattern (MEDIUM)** — at minimum add a clarifying comment to prevent future confusion.
5. Apply the Tailwind v4 nit as part of Task 5.
