### Residual Issues (from previous findings not fully resolved)

#### Medium

- **Story line 221-227 — Font caching code block lacks `CacheableResponsePlugin` (confidence: 75)**: Task 4.0 at line 86 correctly specifies `CacheableResponsePlugin({ statuses: [0, 200] })` for the font runtime caching rule. However, the canonical code example in the "Route Runtime Caching" section (lines 221-227) still omits it:

  ```ts
  registerRoute(
    /\/assets\/.+\.woff2$/i,
    new CacheFirst({
      cacheName: 'fonts',
      plugins: [new ExpirationPlugin({ maxEntries: 50, maxAgeSeconds: 30 * 24 * 60 * 60 })],
    })
  );
  ```

  A developer who copies the code example verbatim would miss the plugin. The task description says to include it, but code examples are what implementers typically copy. Every other `CacheFirst` rule in `sw.ts` includes `CacheableResponsePlugin({ statuses: [0, 200] })` (lines 30-36, 55-63).

  - **Why**: Without the plugin, transient CDN 5xx/404 responses for font requests occupy `ExpirationPlugin`'s `maxEntries: 50` slots despite not being cached. After 50 unique failures, subsequent successful fonts are evicted on fetch — causing FOUT when offline.
  - **Fix**: Add `new CacheableResponsePlugin({ statuses: [0, 200] })` to the `plugins` array in the code example block (line 225), matching the task description and the pattern in all other rules.
  - **Effort**: ~2 minutes
  - **Category**: [Correctness]
  - **Autofix class**: `safe_auto`
