## Code Review: E13-S06 — Handle localStorage Quota Exceeded Gracefully

### What Works Well

1. **Clean adapter architecture.** The `quotaResilientStorage` module implements Zustand's `StateStorage` interface cleanly, keeping fallback logic decoupled from the quiz store. Any future Zustand persist store can reuse it with zero changes.

2. **Cross-browser error detection.** The `isQuotaExceeded()` helper handles both the standard `QuotaExceededError` and Firefox's non-standard `NS_ERROR_DOM_QUOTA_REACHED` variant, preventing silent failures on Firefox.

3. **Thorough test coverage for the adapter.** 14 unit tests cover normal paths, fallback paths, throttling, Firefox variants, and double-failure scenarios. The `Storage.prototype` override strategy is well-documented and correctly handles jsdom's shared-Map limitation.

### Findings

#### High Priority

- **[src/stores/useQuizStore.ts:332-335] (confidence: 88)**: Subscriber calls `showThrottledWarning()` even when both localStorage AND sessionStorage fail. The inner `catch` at line 332 silently swallows sessionStorage errors, then execution falls through to `showThrottledWarning()` on line 335 unconditionally. This tells the user "Quiz progress will be saved for this session only" when in reality nothing was saved. Why: Learners believe their progress is safe when it is not. Fix: Move `showThrottledWarning()` inside the `try` block after `sessionStorage.setItem` succeeds, matching the adapter's `sessionSucceeded` pattern:
  ```typescript
  try {
    if (progress && quiz) {
      sessionStorage.setItem(`quiz-progress-${quiz.id}`, JSON.stringify(progress))
      showThrottledWarning()
    }
  } catch {
    // Both storages failed - no misleading toast
    console.error('[useQuizStore] Both storages failed for quiz backup')
  }
  ```

- **[src/lib/quotaResilientStorage.ts:124] (confidence: 85)**: `toastError.saveFailed(...)` shows the message "Save failed. Check your connection and try again." when both storages are full. The `details` parameter ("Both localStorage and sessionStorage are full") only logs to `console.error` -- the user sees an irrelevant network error message. Why: Learners see confusing "check your connection" guidance when the real problem is storage quota. Fix: Use `toastError.generic()` (which was added in this story) or create a dedicated `toastError.storageFull()` variant with an accurate user-facing message like "Storage is completely full. Your quiz progress cannot be saved. Clear browser data to fix this."

- **[src/app/pages/Quiz.tsx:209] (confidence: 82)**: The inner `catch` at the `beforeunload` handler catches ALL errors from `localStorage.setItem`, not just quota errors, and unconditionally falls back to `sessionStorage.setItem`. A `SecurityError` (private browsing, iframe sandbox) would be caught here and the code would attempt `sessionStorage.setItem` -- which will also throw `SecurityError` in the same security context, causing the outer catch to fire. While this ultimately logs via the outer catch, the error message is misleading. Why: Debugging becomes harder when the real error is obscured by an unnecessary fallback attempt. Fix: Check `isQuotaExceeded` in the inner catch:
  ```typescript
  try {
    localStorage.setItem(key, value)
  } catch (innerErr) {
    if (isQuotaExceeded(innerErr)) {
      sessionStorage.setItem(key, value)
    }
    // Non-quota errors fall through to outer catch
  }
  ```

- **[src/lib/toastHelpers.ts:166-168] (confidence: 80)**: `toastError.generic()` is added but never called anywhere in the codebase. Dead code that increases the API surface without justification. Why: Dead exports confuse future developers about intended usage patterns. Fix: Either wire it into `quotaResilientStorage.ts:124` (replacing the misleading `saveFailed` call) or remove it. If keeping it, add at least one call site.

#### Medium

- **[src/stores/useQuizStore.ts:317-324] (confidence: 75)**: Subscriber does not attempt `clearStaleQuizKeys()` before falling back to sessionStorage, unlike the adapter which runs cleanup as Phase 1. This is a missed recovery opportunity -- clearing stale keys might free enough space to avoid the sessionStorage fallback entirely. Why: Learners get downgraded to session-only storage when a cleanup could have preserved localStorage persistence. Fix: Export `clearStaleQuizKeys` from `quotaResilientStorage.ts` and call it in the subscriber's catch block before attempting sessionStorage.

- **[src/lib/quotaResilientStorage.ts:52-69] (confidence: 72)**: `clearStaleQuizKeys` only removes `quiz-progress-*` keys but the largest storage consumer is typically the Zustand persist key `levelup-quiz-store` itself (which contains the full store state). The cleanup is effective for orphaned per-quiz keys but cannot help if the main persist key is the one bloating storage. Why: If a single quiz with many questions produces a large `levelup-quiz-store` entry, clearing small orphaned backup keys may not free enough space. Fix: Document this limitation or consider also pruning stale fields within `levelup-quiz-store`.

- **[src/lib/toastHelpers.ts:92] (confidence: 70)**: AC4 specifies the toast should suggest "clearing browser data **or using a different browser**". The current message only says "Try clearing browser data to fix this." -- missing the alternative browser suggestion. Why: Spec compliance gap; learners on a shared/kiosk machine may not be able to clear browser data. Fix: Update message to "Try clearing browser data or using a different browser."

#### Nits

- **Nit** [src/lib/toastHelpers.ts:84-96] (confidence: 60): `toastWarning` is placed between `toastSuccess` and `toastError`, breaking the severity-ascending order (success < warning < error). Minor DX inconsistency.

- **Nit** [src/stores/useQuizStore.ts:337] (confidence: 55): `console.error` message says "localStorage sync failed" but this path now also involves sessionStorage fallback logic. Consider updating to "storage sync failed" for accuracy.

- **Nit** [src/lib/quotaResilientStorage.ts:4] (confidence: 50): `THROTTLE_MS = 30_000` is a magic number with no inline explanation. The story doc covers it well, but a brief comment like `// Zustand persist fires on every set(); 30s prevents toast flood` would help future maintainers.

### Recommendations

1. **First**: Fix the subscriber's misleading toast on double-failure (High #1) -- this is the most impactful correctness bug, causing learners to believe progress is saved when it is not.
2. **Second**: Fix the misleading `saveFailed` toast in the adapter (High #2) -- "check your connection" is wrong guidance for a quota problem.
3. **Third**: Add `isQuotaExceeded` guard to the `beforeunload` handler (High #3) -- prevents unnecessary fallback attempts on security errors.
4. **Fourth**: Wire `toastError.generic()` into a call site or remove it (High #4).
5. **Fifth**: Add `clearStaleQuizKeys` call to the subscriber (Medium #1) and update the toast message for AC4 compliance (Medium #3).
