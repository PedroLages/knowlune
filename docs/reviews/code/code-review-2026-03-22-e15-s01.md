# Code Review: E15-S01 — Display Countdown Timer with Accuracy

## What Works Well

1. **Date.now() anchor pattern is correct** -- the core drift-prevention approach (computing `endTime` once, then deriving remaining from wall clock) is the industry-standard solution for accurate browser timers. Good engineering choice.

2. **Dual-element accessibility pattern** -- separating the visual `role="timer"` from the `aria-live="polite"` screen reader region avoids per-second announcements while still providing periodic updates. Threshold announcements at 25% and 10% are a thoughtful addition.

3. **Clean component extraction** -- QuizHeader went from 131 lines of mixed concerns to 41 lines of pure presentation. QuizTimer and useQuizTimer each have a single responsibility. The prop interface (`timeRemaining | null`) elegantly handles untimed quizzes.

## Findings

### Blockers

- **[src/app/pages/Quiz.tsx:211-218 + src/hooks/useQuizTimer.ts:71-83,86-90,112] (confidence: 92)**: **Circular dependency between store sync and hook re-initialization causes timer reset every 60 seconds.** The hook writes `timeRemaining` back to `useQuizStore.currentProgress.timeRemaining` every 60s (line 86-90). The Quiz page subscribes to `currentProgress` via Zustand selector (line 88 of Quiz.tsx), causing a re-render. On re-render, `timerInitialSeconds` (line 211-214) recomputes from the updated store value, producing a *different* number. Because `useQuizTimer`'s effect depends on `[initialSeconds]` (line 112), the effect tears down and re-runs, computing a fresh `endTime`. This creates a 60-second reset cycle that can cause visible timer jumps and unnecessarily tears down/recreates two intervals and two event listeners every minute. **Why:** Learners may see the timer stutter or jump backward by a second at sync boundaries, undermining the "accuracy" promise of this story. **Fix:** Decouple the sync from the hook's initialization. Either (a) memoize `timerInitialSeconds` with `useRef` so it only captures the value once when the quiz starts and does not change on store updates, or (b) move the store sync out of the hook entirely and into a separate effect in Quiz.tsx that reads from the hook's return value, breaking the circular write-read chain.

### High Priority

- **[src/hooks/useQuizTimer.ts:71-72] (confidence: 88)**: **Store sync skips the final `0` value -- crash recovery after expiry restores stale time.** `syncToStore` guards with `if (remainingSeconds > 0)`, so when the timer reaches exactly 0 the store never learns the quiz expired. If the browser crashes between expiry and the `submitQuiz` completing, crash recovery would restore the last synced value (e.g., 58 seconds) and the learner would see a phantom timer instead of the results page. **Why:** Data integrity for crash recovery -- learners who lose connectivity at the exact moment of expiry would see incorrect state. **Fix:** Change guard to `if (remainingSeconds >= 0)` or add a separate `syncToStore(0)` call in the expiry branch alongside `onExpireRef.current()`.

- **[src/hooks/useQuizTimer.ts:53] (confidence: 85)**: **`onExpireRef.current()` does not await the async callback, creating an unhandled promise rejection risk.** `handleTimerExpiry` in Quiz.tsx (line 200) is `async` and calls `submitQuiz` which hits IndexedDB. When called via `onExpireRef.current()` (a synchronous call site), the returned Promise is discarded. If `submitQuiz` throws and the internal catch in `handleTimerExpiry` somehow doesn't catch it (e.g., if `toast.error` on line 201 throws first -- unlikely but possible), the rejection is unhandled. More practically, the caller has no way to know if submission actually succeeded. **Why:** Silent failure path for the auto-submit feature -- the learner's answers could be lost without any error feedback. **Fix:** Either `void onExpireRef.current()` (explicit fire-and-forget acknowledgment) or better, wrap the call: `Promise.resolve(onExpireRef.current()).catch(console.error)`.

- **[src/hooks/useQuizTimer.ts:8-12] (confidence: 82)**: **`formatTime` produces malformed output for negative inputs.** `Math.floor(-1 / 60)` = `-1` and `(-1) % 60` = `-1`, producing `"-1:-1"`. The hook internally guards with `Math.max(0, ...)` but `formatTime` is exported and available to any caller. A negative value from a race condition or test setup would render garbled text. **Why:** Defensive programming for a public utility -- the function's contract should handle any number gracefully. **Fix:** Add `const clamped = Math.max(0, totalSeconds)` as the first line of `formatTime`, or guard with `if (totalSeconds < 0) return '00:00'`.

### Medium

- **[src/app/pages/Quiz.tsx:200-207] (confidence: 75)**: **`toast.error` fires before `submitQuiz` -- learner sees "Time's up!" even if submission fails.** The toast on line 201 fires unconditionally before the `try/await submitQuiz()`. If `submitQuiz` throws, the learner has already been told their quiz was submitted when it was not. The catch block comment says "Store already shows error toast" but the damage is done -- two conflicting toasts. **Why:** Misleading feedback erodes trust. A learner who sees "submitted" but then gets an error is confused. **Fix:** Move the toast inside the try block after `submitQuiz` succeeds, or change to a neutral message first ("Submitting your quiz...") and show the success/failure toast after.

- **[src/app/components/quiz/QuizTimer.tsx:19-20] (confidence: 78)**: **Timer shows warning/urgent colors at `timeRemaining === 0`.** When `timeRemaining` is 0 and `totalTime` is 900: `isWarning = (0 <= 225 && 0 > 90)` = false, `isUrgent = (0 <= 90)` = true. So the timer briefly shows red "00:00" before auto-submit navigates away. This is cosmetically fine, but if auto-submit fails (network error), the learner is stuck looking at a red "00:00" timer with no obvious action to take. **Why:** Edge case UX -- the component should display an expired state (e.g., "00:00" with a different visual treatment or hidden entirely) rather than "urgent" red at exactly zero. **Fix:** Add an `isExpired = timeRemaining === 0` check and either hide the timer or show a distinct style.

- **[src/app/components/quiz/QuizTimer.tsx:33] (confidence: 72)**: **Minute-boundary announcement fires at `timeRemaining === 0`.** When `timeRemaining` reaches 0 and `0 % 60 === 0`, the screen reader announces "Time remaining: 0 minutes" -- redundant since the expiry toast will also fire. **Why:** Screen reader noise at a critical moment. **Fix:** Add `&& timeRemaining > 0` to the minute-boundary condition.

### Nits

- **Nit** [src/hooks/useQuizTimer.ts:67,100] (confidence: 70): Two separate `visibilitychange` listeners are registered on the same `document` -- one for recalculating the timer (line 67) and one for syncing to the store (line 100). These could be combined into a single listener that does both operations based on `visibilityState`, reducing event listener overhead.

- **Nit** [src/app/components/quiz/QuizTimer.tsx:1] (confidence: 65): `useState`, `useEffect`, and `useRef` are all used, but consider whether the `liveAnnouncement` state could be derived directly during render (as a local variable) instead of stored in state. The `useEffect` that updates it is only needed because it's state -- if it were a ref with a `key` prop pattern or computed inline, the component would be simpler.

- **Nit** [tests/e2e/story-15-1.spec.ts:1017-1019] (confidence: 60): `saveRealDateNow` stores the reference via `page.evaluate` but this runs after page load. If any app code calls `Date.now()` during hydration before `saveRealDateNow` is called, the reference is fine (it's the real one). But the pattern is fragile -- consider using `page.addInitScript` instead so the reference is saved before any app code runs.

## Recommendations

1. **Fix the circular store-sync / hook re-initialization loop first** (Blocker) -- this affects timer accuracy, the story's core promise. Use a ref to capture `timerInitialSeconds` once at quiz start.
2. **Move `toast.error` after `submitQuiz` succeeds** (Medium) -- quick fix, prevents misleading UX.
3. **Guard `syncToStore` to allow `remaining === 0`** (High) -- one character change (`> 0` to `>= 0`), fixes crash recovery edge case.
4. **Add negative-input guard to `formatTime`** (High) -- defensive, prevents garbled display from any future caller.
5. **Handle the async `onExpire` Promise** (High) -- add `.catch()` to prevent unhandled rejections.
