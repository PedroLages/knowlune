## Code Review: E04-S03 -- Automatic Study Session Logging

### What Works Well
1. **Solid store architecture.** The Zustand store follows the established `persistWithRetry` + optimistic update pattern, with proper rollback on failure and meaningful error logging. This is consistent with existing stores (`useNoteStore`, etc.) and easy to reason about.
2. **Sensible orphan recovery strategy.** Using `endTime === undefined` as the orphan signal and closing with `lastActivity` as the end time is a robust approach. The `Math.max(session.duration, totalSeconds)` guard in `recoverOrphanedSessions` correctly handles edge cases where accumulated duration might exceed the naive start-to-lastActivity calculation.
3. **Clean idle detection hook.** `useIdleDetection` is well-designed with passive event listeners, proper cleanup, and clear separation of idle/active/activity callbacks. The ref-based idle state avoids unnecessary re-renders.

### Findings

#### Blockers

- **[Recurring] Working tree contains uncommitted ImportedLessonPlayer integration, E2E test fixes, and schema test updates** (confidence: 98). The committed branch (commit `c221799`) has NO session tracking in `ImportedLessonPlayer.tsx`, stale E2E tests with `startTimestamp`/`endTimestamp` property names (vs the actual `startTime`/`endTime` field names), and a schema test expecting version 5 with no `studySessions` table. The working tree has all fixes applied but uncommitted. This is the **7th consecutive story** with this pattern.
  - Committed LessonPlayer has `endSession()` in useEffect cleanup return (removed in working tree) plus `endSession` in dependency array (causes effect to re-fire when `endSession` identity changes)
  - E2E tests in committed version check for `session.startTimestamp` and `session.endTimestamp` properties that don't exist on the `StudySession` type (actual fields: `startTime`, `endTime`)
  - Schema test in committed version expects `db.verno === 5` and doesn't include `studySessions` in the table list
  Why: All 5 E2E tests will fail on the committed branch. The schema unit test will also fail. Merging this branch ships broken tests.
  Fix: `git add src/app/pages/ImportedLessonPlayer.tsx src/app/pages/LessonPlayer.tsx src/db/__tests__/schema.test.ts tests/e2e/story-e04-s03.spec.ts && git commit`.

- **`endSession()` is async but called from `beforeunload` and `pagehide` handlers** (`/Volumes/SSD/Dev/Apps/Elearningplatformwireframes-worktrees/e04-s03/src/app/pages/LessonPlayer.tsx:282-284`, `/Volumes/SSD/Dev/Apps/Elearningplatformwireframes-worktrees/e04-s03/src/stores/useSessionStore.ts:113-140`) (confidence: 95). `endSession()` returns a `Promise` and calls `persistWithRetry()` which itself does exponential backoff with `setTimeout`. Browsers do **not** wait for async operations during `beforeunload`/`pagehide`. The IndexedDB `put()` will almost certainly be killed mid-flight, leaving an orphaned session. The Zustand state update (`set({ activeSession: null })`) happens synchronously, but the Dexie write will not complete.
  Why: Every tab close or page unload creates an orphaned session. The orphan recovery on next load will fix it, but `duration` will be based on `lastActivity` (which may be stale if `updateLastActivity` hasn't been called recently) rather than the actual close time. Users lose study time data.
  Fix: Use synchronous `navigator.sendBeacon()` with a small JSON payload for the `beforeunload` path, or switch to `IndexedDB` direct (non-Dexie) synchronous-ish write. Alternatively, maintain a periodic heartbeat that persists `lastActivity` to Dexie every 30-60 seconds, so orphan recovery has recent data.

#### High Priority

- **`useIdleDetection` callbacks are inline arrows -- effect tears down and re-registers all 5 event listeners on every render** (`/Volumes/SSD/Dev/Apps/Elearningplatformwireframes-worktrees/e04-s03/src/app/pages/LessonPlayer.tsx:108-112`, `/Volumes/SSD/Dev/Apps/Elearningplatformwireframes-worktrees/e04-s03/src/app/hooks/useIdleDetection.ts:54`) (confidence: 92). The hook has `[onIdle, onActive, onActivity]` as useEffect dependencies. LessonPlayer passes `() => pauseSession()`, `() => resumeSession()`, `() => updateLastActivity()` -- all inline arrow functions that get new references every render. Since `updateLastActivity` calls `set()` on every user interaction (mousedown, keydown, scroll, etc.), this creates a cascade: interaction -> `set()` -> re-render -> new callbacks -> useEffect cleanup -> remove 5 listeners -> add 5 listeners -> repeat.
  Why: Hundreds of unnecessary listener teardown/setup cycles per session. On slower devices, this can cause visible jank during typing or scrolling.
  Fix: Either wrap the callbacks in `useCallback` in LessonPlayer, or (better) use refs inside `useIdleDetection` to store the callbacks, removing them from the dependency array:
  ```typescript
  const onIdleRef = useRef(onIdle)
  const onActiveRef = useRef(onActive)
  const onActivityRef = useRef(onActivity)
  useEffect(() => { onIdleRef.current = onIdle }, [onIdle])
  // ... then call onIdleRef.current() in the timeout
  // useEffect deps become []
  ```

- **`updateLastActivity()` fires on every mousedown/keydown/scroll/wheel/touchstart, calling `set()` each time -- triggers Zustand re-render of entire LessonPlayer tree** (`/Volumes/SSD/Dev/Apps/Elearningplatformwireframes-worktrees/e04-s03/src/stores/useSessionStore.ts:64-72`, `/Volumes/SSD/Dev/Apps/Elearningplatformwireframes-worktrees/e04-s03/src/app/hooks/useIdleDetection.ts:27`) (confidence: 90). `onActivity` is called from `resetTimer` which fires on every user interaction event. `updateLastActivity` does `set({ activeSession: { ...activeSession, lastActivity: now } })` -- a full Zustand state update. LessonPlayer destructures `useSessionStore()` without a selector, so it subscribes to ALL state changes. Every keystroke, scroll, or click triggers a full re-render of LessonPlayer (a very large component tree with VideoPlayer, NoteEditor, ModuleAccordion, etc.).
  Why: Learners typing notes or scrolling through content will experience unnecessary re-renders ~10-30 times per second. Combined with the listener teardown issue above, this is a significant performance bottleneck.
  Fix: Either (a) throttle/debounce `updateLastActivity` to once per 30 seconds (the timestamp only matters for orphan recovery, so it doesn't need sub-second precision), or (b) use a ref instead of Zustand state for `lastActivity` (persist it in the `endSession`/`pauseSession` calculations only), or (c) at minimum, use a selector in LessonPlayer: `const startSession = useSessionStore(s => s.startSession)` etc.

- **`pauseSession` duration calculation is inverted -- computes time since last activity, not time of active work** (`/Volumes/SSD/Dev/Apps/Elearningplatformwireframes-worktrees/e04-s03/src/stores/useSessionStore.ts:78-81`) (confidence: 85). When idle detection triggers `pauseSession`, the code calculates `activeSeconds = currentTime - lastActivityTime`. But `lastActivity` was just updated moments ago by `onActivity`/`updateLastActivity`. So `activeSeconds` will be near-zero (the gap between the last `updateLastActivity` and the idle timeout firing). The *actual* active time between the previous pause/start and now is not being tracked -- it's the time from when the session started (or last resumed) to the last activity. The current logic accumulates near-zero durations at each pause.
  Why: Study session durations will be systematically underreported. A learner who studies for 30 minutes then goes idle will have `duration` of approximately 0 (or a few seconds) added, because `lastActivity` was set to "now" by the most recent `updateLastActivity` call just before the idle timeout.
  Fix: Track `lastResumedAt` or `activeStartTime` as a separate field. When pausing, compute `activeSeconds = pauseTime - lastResumedAt`. When resuming, set `lastResumedAt = now`. `lastActivity` should remain for orphan recovery purposes only.

- **No unit tests for `useSessionStore` or `useIdleDetection`** (confidence: 95). There is no `useSessionStore.test.ts` or `useIdleDetection.test.ts` anywhere in the repo. The store has complex logic (duration calculation, orphan recovery, optimistic updates with rollback) that is entirely untested at the unit level. E2E tests cover happy paths but not edge cases like concurrent `startSession` calls, `endSession` when no active session, or `pauseSession` with negative time deltas.
  Why: Duration calculation bugs (like the one above) would be caught immediately with unit tests. The orphan recovery `Math.max` logic, the idle time exclusion math, and the optimistic update/rollback patterns all need targeted unit testing.
  Fix: Create `src/stores/__tests__/useSessionStore.test.ts` with tests for: startSession (creates session, rolls back on failure), endSession (calculates duration, handles no active session), pauseSession (accumulates duration correctly), resumeSession (resets activity timestamp), recoverOrphanedSessions (closes orphans, handles empty case), getTotalStudyTime (filters by course, excludes incomplete sessions). Create `src/app/hooks/__tests__/useIdleDetection.test.ts` with tests for timeout behavior using `vi.useFakeTimers()`.

#### Medium

- **`recoverOrphanedSessions` uses `.filter()` (full table scan) instead of using the `endTime` index** (`/Volumes/SSD/Dev/Apps/Elearningplatformwireframes-worktrees/e04-s03/src/stores/useSessionStore.ts:161-163`) (confidence: 78). The Dexie schema declares an `endTime` index on `studySessions`, but `recoverOrphanedSessions` uses `db.studySessions.filter(session => session.endTime === undefined)` which loads every record into memory and checks the condition in JavaScript. With Dexie, `undefined` values are not indexed (they're "holes" in the index), so a direct index query isn't straightforward, but a more efficient approach exists.
  Fix: Use `db.studySessions.where('endTime').equals('').toArray()` with a sentinel value like empty string for "no end time", or accept the full scan but add a comment explaining why (orphan recovery runs once per app load, table is small).

- **`getTotalStudyTime` is a synchronous function on the store that reads `sessions` state -- but it's called during render in Overview without `useMemo`** (`/Volumes/SSD/Dev/Apps/Elearningplatformwireframes-worktrees/e04-s03/src/app/pages/Overview.tsx:62`) (confidence: 75). `getTotalStudyTime()` is called on every render of Overview. It iterates over all sessions with `.reduce()`. While not expensive now, it creates unnecessary work on every re-render. More importantly, `getTotalStudyTime` is extracted via `useSessionStore()` without a selector, so Overview subscribes to ALL session store changes.
  Fix: Use a selector: `const getTotalStudyTime = useSessionStore(s => s.getTotalStudyTime)` and/or memoize the result. Better yet, compute `totalStudyTime` as a derived value inside `loadSessionStats` and store it directly.

- **`idleTime` field on `StudySession` is declared in the type but never updated** (`/Volumes/SSD/Dev/Apps/Elearningplatformwireframes-worktrees/e04-s03/src/data/types.ts:180`) (confidence: 90). The `StudySession` interface declares `idleTime: number` and it's initialized to `0` in `startSession`, but no code ever increments it. `pauseSession` and `endSession` calculate active time but never update `idleTime`.
  Fix: Either remove `idleTime` from the interface (YAGNI), or compute and accumulate it in `pauseSession` (e.g., `idleTime: activeSession.idleTime + IDLE_TIMEOUT_MS / 1000`).

- **`videosWatched` array is declared but never populated** (`/Volumes/SSD/Dev/Apps/Elearningplatformwireframes-worktrees/e04-s03/src/stores/useSessionStore.ts:45`) (confidence: 92). AC2 requires "the session record is updated with: total duration, list of videos watched, and pages viewed during the session." `videosWatched` is initialized as `[]` but no store action ever adds video IDs to it. There's no mechanism to track which videos were watched during a session.
  Why: AC2 is partially unmet. The session records duration but not content coverage.
  Fix: Add a `trackVideoWatched(videoId: string)` action that pushes to `activeSession.videosWatched` (with dedup). Call it from LessonPlayer when a video starts playing.

- **Committed LessonPlayer has `endSession` in useEffect cleanup AND in dependency array** (`/Volumes/SSD/Dev/Apps/Elearningplatformwireframes-worktrees/e04-s03/src/app/pages/LessonPlayer.tsx:271-280`, committed version) (confidence: 88). The committed version has `endSession` in the dependency array of the AC1 effect. Since `endSession` reads from `get()` (Zustand), its identity is stable, but if `set()` is called (e.g., by `updateLastActivity`), the component re-renders, the destructured `endSession` gets a new ref, the effect re-fires, and `startSession` is called again -- creating a duplicate session. The working tree fixes this by removing the cleanup return and `endSession` from deps.
  Fix: Commit the working tree version which removes `endSession` from the dependency array and the cleanup return.

### Recommendations

1. **First**: Commit the working tree changes (ImportedLessonPlayer integration, LessonPlayer fix, E2E test updates, schema test). This is the blocker that prevents the branch from even passing tests.
2. **Second**: Fix the `beforeunload` async issue. Either use `navigator.sendBeacon` or add a periodic heartbeat that persists `lastActivity` to Dexie every 30-60 seconds so orphan recovery has accurate data.
3. **Third**: Fix the duration calculation logic in `pauseSession`/`endSession`. Track `activeStartTime` (set on session start and resume) separately from `lastActivity` (set on every interaction for orphan recovery).
4. **Fourth**: Stabilize `useIdleDetection` callbacks with refs to prevent listener teardown churn, and throttle `updateLastActivity` to avoid per-interaction re-renders.
5. **Fifth**: Add unit tests for `useSessionStore` (especially duration math and orphan recovery) and `useIdleDetection`.
6. **Sixth**: Implement `videosWatched` tracking to fully satisfy AC2.
