## Findings

### 🔴 Blockers

**[B1] Stale progress bug — AC5 only partially satisfied**
`src/app/components/RecommendedNext.tsx:63-77` | confidence: 92

`useMemo` depends only on `[sessions]` but calls `getAllProgress()` inside the memo body. If a learner completes a lesson (which updates localStorage progress), the recommendations will not recalculate until the session store changes. A learner who marks lessons complete from the course page and returns to the dashboard sees stale recommendations.

Fix: Add a progress dependency signal — either call `getAllProgress()` outside the memo and include the result as a dependency, or add a lightweight `progressVersion` counter to a store/context that increments on `saveAllProgress`.

---

**[B2] Mutable cache reference passed to recommendation algorithm**
`src/app/components/RecommendedNext.tsx:63-64` | confidence: 88

`getAllProgress()` returns the raw `_progressCache` reference. This mutable reference is passed directly into `getRecommendedCourses`. If any other code mutates the cached object between renders (e.g., `ensureProgress` at `src/lib/progress.ts:140` mutates the returned object in-place), recommendation data becomes corrupted.

Fix: Clone the returned value in `getAllProgress()` (`return { ..._progressCache }`) or freeze it.

### 🟡 High Priority

**[H1] AC3 E2E test does not verify navigation**
`tests/e2e/story-e07-s02.spec.ts:92-103` | confidence: 85

The AC3 test only reads the `href` attribute and asserts it contains `/courses/`. It never clicks the link or verifies the URL changed. If a click handler calls `preventDefault`, or the `CourseCard` `guardNavigation` logic blocks navigation, this test still passes.

Fix: Replace `getAttribute('href')` assertion with `await firstLink.click()` followed by `await expect(page).toHaveURL(/\/courses\//)`.

---

**[H2] AC5 has zero E2E test coverage**
`tests/e2e/story-e07-s02.spec.ts` | confidence: 82

"Rankings refresh after study session" has no E2E test. The `useMemo` reactivity depends on session store changes — this is exactly the kind of async behaviour that benefits from E2E validation.

Fix: Add a test that seeds progress, verifies initial order, triggers a session change, reloads, and verifies the order or content changes.

---

**[H3] Dead code in scoring algorithm**
`src/lib/recommendations.ts:58-59` | confidence: 78

`thirtyDaysAgo` is computed but immediately voided with `void thirtyDaysAgo`. The comment says "available for future use when E07-S01 provides momentum" — this is speculative YAGNI code that adds cognitive load in a scoring algorithm.

Fix: Remove the two lines entirely. When E07-S01 is implemented, add it then.

### 🟠 Medium

**[M1] Hardcoded colour instead of theme token (recurring)**
`src/app/components/RecommendedNext.tsx:39` | confidence: 75

`bg-blue-100 dark:bg-blue-900/30` is hardcoded in the EmptyState icon circle instead of using a semantic theme token. This pattern has recurred since E03-S05 across multiple stories.

Fix: Use `bg-brand-soft` or define a semantic token for decorative icon backgrounds in `theme.css`.

---

**[M2] Missing intermediate responsive breakpoint**
`src/app/components/RecommendedNext.tsx:101` | confidence: 72

`grid-cols-1 sm:grid-cols-3` with no `md:grid-cols-2` step means at 640–767px, three cards render at ~180px each — too narrow for the `CourseCard` overview variant content (badge, title, instructor, progress bar).

Fix: Use `grid-cols-1 sm:grid-cols-2 lg:grid-cols-3`.

---

**[M3] Skeleton duplicated between Overview.tsx and RecommendedNext.tsx**
`src/app/pages/Overview.tsx:122-135` | confidence: 70

The loading skeleton layout is defined twice — inline in Overview.tsx and as `RecommendedNextSkeleton` inside `RecommendedNext.tsx`. Changes to the skeleton must be made in two places.

Fix: Export `RecommendedNextSkeleton` from `RecommendedNext.tsx` and import it in Overview.tsx, or remove the Overview inline skeleton entirely.
