## High Priority

### [HIGH-1] No NaN/Infinity guard in `calculateNormalizedGain`
**File:** `src/lib/analytics.ts:119-120` | **Confidence:** 90

`calculateNormalizedGain` does not guard against `NaN`, `Infinity`, or negative `initialScore` inputs. Corrupted IndexedDB records could produce `NaN` which propagates through `Math.round(normalizedGain! * 100)` in `ScoreSummary.tsx:191`, rendering "NaN%" in the UI.

**Fix:** Add early return: `if (!Number.isFinite(initialScore) || !Number.isFinite(finalScore)) return null`

---

### [HIGH-2] Unit tests in wrong file location
**File:** `src/lib/analytics.test.ts` | **Confidence:** 92

A pre-existing `src/lib/__tests__/analytics.test.ts` already exists for `analyzeTopicPerformance`. The new tests for `calculateNormalizedGain` and `interpretNormalizedGain` were placed in a co-located `src/lib/analytics.test.ts`. This creates two separate test files for the same module â€” the normalized gain tests are orphaned from the existing suite, reducing discoverability during future refactoring.

**Fix:** Move `calculateNormalizedGain` and `interpretNormalizedGain` test suites into `src/lib/__tests__/analytics.test.ts`.

---

### [HIGH-3] useMemo dependency could be simplified
**File:** `src/app/pages/QuizResults.tsx:64-65` | **Confidence:** 85

The `normalizedGain` useMemo lists `[attempts, lastAttempt?.percentage]` as dependencies. Since `lastAttempt` is derived from `attempts` (not memoized), `lastAttempt?.percentage` is always a fresh evaluation â€” both values are 100% derived from `attempts`. This creates a misleading impression that `lastAttempt` is an independent data source; a future developer could desynchronize the two.

**Fix:** Simplify dependency array to `[attempts]`.
