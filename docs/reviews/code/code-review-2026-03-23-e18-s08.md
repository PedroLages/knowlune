## Code Review: E18-S08 -- Display Quiz Availability Badges on Courses Page

### What Works Well

1. **Clean three-state semantic on `QuizScoreMap`.** The `Map<string, number | null>` design (key present = quiz exists, null = never attempted, absent = no quiz) is expressive and avoids boolean flags. The `has()` / `get()` distinction in ModuleAccordion reads naturally.

2. **Proper async cleanup pattern.** The `ignore` flag in `useQuizScoresForCourse` correctly prevents stale state updates after unmount or dependency change. This is the textbook pattern for React async effects.

3. **Accessible quiz badge.** The `aria-label` on `QuizBadge` correctly differentiates between "Take quiz" and "Quiz score: X%" states, and includes the lesson title for screen reader context. `aria-hidden="true"` on the icon is correct.

#### High Priority

- **[Recurring] [tests/e2e/story-e18-s08.spec.ts:76-116] (confidence: 90)**: Manual `indexedDB.open()` seeding duplicates the shared helper `seedIndexedDBStore` from `tests/support/helpers/indexeddb-seed.ts`. The project already has `seedQuizzes()` and `seedQuizAttempts()` convenience wrappers (lines 253-266 of `indexeddb-seed.ts`), plus `seedIndexedDBStore()` for the courses table. The duplicated `seedToStore` function in the test file omits the frame-accurate `requestAnimationFrame` retry logic that the shared helper uses, making it potentially less reliable in headless Chromium. Why: Divergent seeding logic means a fix to the shared helper (e.g., retry timing improvements) won't propagate here, and the ESLint `test-patterns/use-seeding-helpers` rule exists precisely to prevent this. Fix: Replace the local `seedToStore` with imports from `tests/support/helpers/indexeddb-seed.ts`:
  ```typescript
  import { seedIndexedDBStore, seedQuizzes, seedQuizAttempts } from '../../support/helpers/indexeddb-seed'
  // then:
  await seedIndexedDBStore(page, 'ElearningDB', 'courses', [testCourse])
  await seedQuizzes(page, [quiz])
  await seedQuizAttempts(page, [completedAttempt])
  ```

- **[src/hooks/useQuizScoresForCourse.ts:72] (confidence: 85)**: `courseId` is in the `useEffect` dependency array but is never referenced inside the effect body. The effect uses `lessonIdKey` (derived from `modules`) to query quizzes. This means if `courseId` changes but the lesson IDs happen to be identical (theoretically possible with cross-course lesson sharing), the effect would re-fire but produce the same result. More importantly, including an unused dependency is a code smell that suggests the developer intended to use `courseId` in the query (e.g., to scope quizzes by course) but forgot. Why: The current query fetches ALL quizzes matching those lesson IDs regardless of course, which could show badges from a different course's quizzes if lesson IDs collide. Fix: Either remove `courseId` from the dependency array (and document why it's not needed), or use it in the query to scope results.

- **[Recurring] [src/app/components/courses/QuizBadge.tsx:32] (confidence: 85)**: Touch target is `h-7` (28px), well below the 44px WCAG 2.5.5 / project minimum. The badge is an interactive `<Button>` that learners will tap on mobile to start quizzes. Why: Learners on touch devices will have difficulty accurately tapping a 28px target, especially within a dense lesson list. This is a recurring pattern (sub-44px touch targets flagged since E02-S07). Fix: Add `min-h-[44px]` or wrap in a container with sufficient padding. Alternatively, since the badge sits inline in a lesson row, ensure the tappable area extends via padding: `className="... h-7 px-2 text-xs shrink-0 min-h-[44px] py-0"` (or use `py-2` to increase visual + touch height).

#### Medium

- **[tests/e2e/story-e18-s08.spec.ts] (confidence: 75)**: No `afterEach` or `test.afterEach` cleanup to clear seeded quiz/course/attempt data. While Playwright creates fresh browser contexts per test by default (assuming the test fixture does this), the story file's own pre-review checklist (line 100) explicitly flags "E2E afterEach cleanup uses `await`" as unchecked. If context isolation isn't guaranteed by the fixture, seeded data from one test could leak into another. Fix: Add `test.afterEach` that clears the `quizzes`, `quizAttempts`, and `courses` stores, or verify and document that the test fixture handles context isolation.

- **[src/hooks/useQuizScoresForCourse.ts:23-29] (confidence: 72)**: The `useMemo` for `lessonIdKey` depends on `[modules]`, but `modules` is a prop that may be a new array reference on every parent render (e.g., if `ModuleAccordion`'s parent creates the modules array inline or derives it from a selector without memoization). If `modules` is referentially unstable, `lessonIdKey` recomputes every render. While the string result would be the same (preventing the useEffect from re-firing due to `lessonIdKey` being a primitive), the flatMap/map/join computation runs unnecessarily. Fix: This is minor since the computation is cheap, but for correctness the pattern should either (a) document that `lessonIdKey` is intentionally a stable primitive even if `useMemo` recomputes, or (b) use a deeper comparison. No action required unless profiling shows it's hot.

- **[tests/e2e/story-e18-s08.spec.ts:167-173] (confidence: 70)**: AC3 test ("lesson with completed quiz shows score badge") verifies text content (`Quiz: 85%`) but does not verify the success color. The AC specifically says "badge with my best score in a success color." The test should assert that the score text has the `text-success` class or equivalent computed color. Fix: Add an assertion like:
  ```typescript
  const scoreSpan = badge.locator('span.text-success')
  await expect(scoreSpan).toBeVisible()
  ```

#### Nits

- **Nit** [src/hooks/useQuizScoresForCourse.ts:54] (confidence: 60): `Math.max(...quizAttempts.map(a => a.percentage))` will throw `RangeError` if `quizAttempts` exceeds the call stack argument limit (~100K+). Not a realistic concern for quiz attempts per user, but `quizAttempts.reduce((max, a) => Math.max(max, a.percentage), 0)` is a safer idiom.

- **Nit** [src/app/components/courses/QuizBadge.tsx:32] (confidence: 55): `h-7` alongside `size="sm"` on the Button may conflict -- `size="sm"` from the button variants likely sets its own height (commonly `h-9`). The explicit `h-7` in className should win via `cn()` / tailwind-merge, but it's worth confirming this doesn't cause unexpected layout. The explicit override is intentional per the design guidance, so this is fine if verified visually.

- **Nit** [docs/implementation-artifacts/e18-s08-display-quiz-availability-badges-courses-page.md:6] (confidence: 50): Story status is `in-progress` but all tasks are checked off. Should be updated to `completed` (or `in-review`).

### Recommendations

1. **First**: Replace manual `seedToStore` in the test file with the shared `seedIndexedDBStore` / `seedQuizzes` / `seedQuizAttempts` helpers. This is the highest-value fix for test maintainability.
2. **Second**: Address the `courseId` dependency -- either remove it from the array with a comment, or use it to scope the query.
3. **Third**: Add the success color assertion to the AC3 test.
4. **Fourth**: Evaluate the 28px touch target on the quiz badge; consider increasing to meet the 44px mobile minimum.
