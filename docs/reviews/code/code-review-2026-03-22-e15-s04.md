## Code Review: E15-S04 -- Provide Immediate Explanatory Feedback per Question

### What Works Well

1. **Derived state pattern in AnswerFeedback is exemplary.** No useState, no useEffect, no stale closures. The component is a pure function of `(question, userAnswer, isTimerExpired)`, making it trivially testable and impossible to desync. This is the right architecture for synchronous feedback.

2. **Non-judgmental design tokens are well-chosen.** Using `warning` (orange) instead of `destructive` (red) for incorrect answers follows established educational UX research. The `stateConfig` lookup table keeps visual concerns declarative and easy to audit.

3. **Accessibility fundamentals are solid.** `role="status"` + `aria-live="polite"` on the feedback container, decorative icons with `aria-hidden="true"`, and text labels alongside color indicators. QuestionBreakdown uses `role="img"` with descriptive `aria-label` on status icons.

### Findings

#### Blockers

_None._

#### High Priority

- **[src/app/components/quiz/AnswerFeedback.tsx:20] (confidence: 92)**: `deriveFeedbackState` misses empty-array check for timer-expired multiple-select questions. The guard `userAnswer === undefined || userAnswer === ''` does not catch `userAnswer = []` (empty array). When a timed quiz with multiple-select questions expires and the learner selected zero options, the code falls through to `calculatePointsForQuestion` instead of returning `'time-expired'` state. The learner sees "Not quite" (0 points) instead of "Not answered in time", which is both incorrect and misleading.
  Why: Misrepresents unanswered state to the learner on timed multiple-select quizzes.
  Fix: Extend the guard to include empty arrays:
  ```typescript
  if (isTimerExpired && (userAnswer === undefined || userAnswer === '' || (Array.isArray(userAnswer) && userAnswer.length === 0))) {
    return { state: 'time-expired', pointsEarned: 0, isCorrect: false }
  }
  ```

- **[src/app/components/quiz/AnswerFeedback.tsx:62, src/app/components/quiz/QuestionBreakdown.tsx:33] (confidence: 85) [Recurring]**: `formatCorrectAnswer` is duplicated identically in two files. This is a small utility, but duplication in quiz components has been a recurring pattern (see `formatTimestamp` duplication flagged since E02-S08). When the display format changes (e.g., wrapping array items in quotes, adding "and" before the last item), both locations must be updated in lockstep.
  Why: Duplication leads to inconsistent display when one copy is updated and the other is not.
  Fix: Extract to a shared location such as `src/lib/quiz-helpers.ts` or `src/lib/scoring.ts` (which already exports quiz-related utilities).

#### Medium

- **[src/app/components/quiz/AnswerFeedback.tsx:117] (confidence: 80) [Recurring]**: Uses `h-5 w-5` and `h-6 w-6` instead of Tailwind v4 `size-5` / `size-6` shorthand. Same pattern on lines 144, 150, 156. This has been recurring since E02-S05.
  Fix: Replace `h-5 w-5` with `size-5`, `h-6 w-6` with `size-6`, `h-4 w-4` with `size-4`. Note the responsive `sm:h-6 sm:w-6` on line 117 becomes `sm:size-6`.

- **[src/app/components/quiz/QuestionBreakdown.tsx:144] (confidence: 78)**: The `role="status"` with `aria-live="polite"` on the expanded detail panel (line 144) fires a screen reader announcement every time the user expands a question row. Since this is a user-initiated toggle (click to expand), `role="region"` with an `aria-label` is more appropriate than a live region. Live regions are designed for content that updates without user action (like the inline feedback in AnswerFeedback). Using it here creates unnecessary verbosity for screen reader users navigating the results breakdown.
  Fix: Replace `role="status" aria-live="polite"` with `role="region" aria-label="Details for question ${row.question.order}"`.

- **[src/app/components/quiz/__tests__/QuestionBreakdown.test.tsx] (confidence: 82)**: No unit tests for the new expandable details feature added to QuestionBreakdown. The test file tests the collapsible header, correct/incorrect icons, and points display, but zero tests cover:
  - Clicking a question row to expand details
  - Explanation rendering in expanded state
  - Correct answer display for incorrect questions
  - "Not answered in time" text for unanswered questions
  - Disabled button state when no details are available (correct answer, no explanation)
  Fix: Add tests that click a question row button, verify expanded details render, and verify the button is disabled when `hasDetails` is false.

- **[src/app/components/quiz/QuestionBreakdown.tsx:96-106] (confidence: 75)**: The per-question `<button>` is always rendered even when `hasDetails` is false, with `disabled={!hasDetails}`. A disabled button for a correctly-answered question with no explanation is semantically odd -- it suggests an action exists but is unavailable, rather than communicating that there is simply nothing to expand. This may confuse keyboard users who Tab through every row.
  Fix: Conditionally render a `<button>` when `hasDetails` is true, otherwise render a `<div>` with the same layout. This reduces the tab-stop count and makes the interaction model clearer.

- **[src/app/pages/Quiz.tsx:452-454] (confidence: 72)**: The visibility guard for AnswerFeedback uses a triple-condition check: `currentAnswer !== undefined && currentAnswer !== '' && !(Array.isArray(currentAnswer) && currentAnswer.length === 0)`. This duplicates the "is unanswered" concept already defined in `countUnanswered` (line 78-86) and `isUnanswered` in QuestionBreakdown (line 28-31). Three independent implementations of the same concept.
  Fix: Extract a shared `isAnswered(answer)` or `isUnanswered(answer)` utility, import it in both Quiz.tsx and QuestionBreakdown.tsx.

#### Nits

- **Nit** [src/app/components/quiz/AnswerFeedback.tsx:80-85] (confidence: 65): The variables `partialTitle`, `correctCount`, `totalCorrect`, `selectedCorrectly`, `selectedIncorrectly`, and `missedCorrect` are all declared with `let` at the top of the component body and only assigned inside the `if (state === 'partial' ...)` block. This mutable-then-conditionally-assign pattern is valid but could be cleaner as an extracted `computePartialBreakdown()` function that returns a readonly object, keeping the render function focused on JSX.

- **Nit** [src/app/components/quiz/AnswerFeedback.tsx:110] (confidence: 60): The `animate-in slide-in-from-bottom-2 fade-in duration-300` animation classes are applied universally, but the story design guidance says "No animation on timer-expired (all feedback appears at once in results view)". The AnswerFeedback component does not distinguish between inline quiz usage and results-page usage. Currently this is not a visible issue because the results page uses QuestionBreakdown (not AnswerFeedback) for timer-expired display, but if AnswerFeedback is ever used in the results context, the animation would play against spec.

### Recommendations

1. **Fix the empty-array timer-expired bug** (High #1) -- this is a correctness issue for timed multiple-select quizzes
2. **Add QuestionBreakdown expandable detail unit tests** (Medium #3) -- the new feature has zero unit test coverage
3. **Fix the aria role on QuestionBreakdown expanded details** (Medium #2) -- improves screen reader experience
4. **Extract `formatCorrectAnswer` and `isUnanswered` to shared utilities** (High #2, Medium #4) -- reduces duplication
5. **Use `size-*` shorthand** (Medium #1) -- easy find-and-replace cleanup
