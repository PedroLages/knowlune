## What Works Well

1. **Clean SM-2 algorithm architecture.** The `SpacedRepetitionState` interface extraction is a textbook example of widening types for reuse. Both `ReviewRecord` and `Flashcard` satisfy it via structural typing.

2. **Proper optimistic update + rollback pattern.** The store follows the established Knowlune convention exactly: optimistic mutation, `persistWithRetry`, full rollback on failure with user-facing toast. The `rateFlashcard` rollback even restores `reviewIndex` and `sessionRatings`.

3. **Excellent accessibility on the review card.** `aria-hidden` toggled on front/back faces, `role="button"` with `tabIndex` management, `aria-live="polite"` announcement on flip, keyboard shortcuts, and `MotionConfig reducedMotion="user"`.

## Blockers

**B1 — `noteId` never passed to NoteEditor callsites** (confidence: 95)

Location: `src/app/pages/LessonPlayer.tsx:757,879,925,988`, `NoteEditor.tsx:1082`

All four NoteEditor callsites in LessonPlayer (and NoteCard) lack the `noteId` prop. The `CreateFlashcardDialog` passes `noteId={noteId}`, but since `noteId` is always `undefined` at runtime, all flashcards created from notes will have `noteId: undefined` — breaking source-note traceability and making the `noteId` index in Dexie useless.

Fix: Pass the note's ID from LessonPlayer/NoteCard to NoteEditor in all callsites. This is a missing implementation step from Task 4.4.

## High Priority

**H1 — Stale `FIXED_NOW` at module load** (confidence: 90)

Location: `src/app/pages/Flashcards.tsx:27,32`

`FIXED_NOW = new Date()` is evaluated once at module load time. If the user keeps the page open past midnight, no new cards become due. Also, `formatNextReviewDate` at line 32 creates `new Date()` on every call — inconsistency between frozen and live time.

Fix: Use live `new Date()` consistently. Consider a `useRef` or `useMemo` that refreshes on `visibilitychange`.

**H2 — `handleRate` has no try/catch — UI freezes on DB failure** (confidence: 88)

Location: `src/app/pages/Flashcards.tsx:134-155`

If `rateFlashcard` throws after all `persistWithRetry` retries, `setIsRating(false)` is never reached. The `isRating` flag stays `true` permanently, disabling all rating buttons and keyboard shortcuts. The user is stuck with no recovery path except a page refresh.

Fix: Wrap in try/catch/finally; move `setIsRating(false)` and `setIsFlipped(false)` into a `finally` block.

**H3 — Entire store destructured without selectors** (confidence: 85)

Location: `src/app/pages/Flashcards.tsx:79-91`

Every store state change triggers a full component re-render. `getStats()` (line 219) runs `getDueFlashcards()` which calls `predictRetention()` for every card on every render. With 100+ cards this becomes noticeable.

Fix: Use individual selectors: `const flashcards = useFlashcardStore(s => s.flashcards)`. Extract stats into `useMemo`.

**H4 — Load error state not shown** (confidence: 82)

Location: `src/app/pages/Flashcards.tsx:104-108`

`loadFlashcards()` and `loadImportedCourses()` are fire-and-forget. The `error` state is never rendered — load failure is indistinguishable from "no flashcards created yet."

Fix: Add an error state UI between the loading skeleton and dashboard, or show a toast on error.

## Medium Priority

**M1 — `formatNextReviewDate` logic error** (confidence: 78)

Location: `src/app/pages/Flashcards.tsx:29-40`

Any date before tomorrow (including today and overdue cards) returns "Tomorrow". A card due today should say "Today"; overdue cards should say "Overdue" or "Today".

Fix: Add today/overdue comparison before the "Tomorrow" branch.

**M2 — Phase `'reviewing'` with undefined `currentCard`** (confidence: 75)

Location: `src/app/pages/Flashcards.tsx:481-482`

`return null` renders a blank screen if `currentCard` is undefined in reviewing phase (race condition or multi-tab reset). No error boundary catches this.

Fix: Redirect to dashboard phase or show "No cards available" with a back button.

**M3 — `CreateFlashcardDialog.handleCreate` not wrapped in try/catch** (confidence: 74)

Location: `src/app/components/notes/CreateFlashcardDialog.tsx:44-52`

`createFlashcard` can throw. Dialog closes unconditionally on `finally`, so on failure the dialog dismisses without feedback that the card was not saved.

Fix: Add try/catch; close dialog only on success.

**M4 — Dexie schema comment mismatch** (confidence: 72)

Location: `src/db/schema.ts:482-506`

The lessons learned doc says `++id` but actual schema uses `id` (explicit UUID). Maintenance hazard for future developers.

Fix: Update lessons learned to say `id` (UUID, not auto-increment).
