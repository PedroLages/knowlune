## Code Review: E11-S01 — Spaced Review System

### What Works Well

1. **Clean algorithm separation.** The SM-2 variant in `src/lib/spacedRepetition.ts` is pure functions with no side effects, making it trivially testable. The 19 unit tests with a fixed date are textbook deterministic testing. Well done.

2. **Thoughtful optimistic update with rollback (AC5).** The `rateNote` flow in `useReviewStore.ts` applies the update to state first, persists with retry, and rolls back on failure while preserving the pending rating for the toast retry button. This matches the acceptance criteria precisely and provides good UX resilience.

3. **Correct reactive pattern in ReviewQueue.** The decision to derive `dueReviews` and `nextReviewDate` via `useMemo` on `allReviews` rather than calling store methods directly (lines 49-55, 99-105 of `ReviewQueue.tsx`) shows understanding of Zustand's subscription model. The inline comment explaining why is excellent for maintainability.

### Findings

#### Blockers

None.

#### High Priority

- **[Recurring] `src/stores/useReviewStore.ts:42-44` (confidence: 90)**: `loadReviews()` and `loadNotes()` are called fire-and-forget inside useEffect in `ReviewQueue.tsx:41-44`. Neither call has `.catch()` error handling. If `loadReviews` throws (e.g., IDB not available), the promise rejection is unhandled. While the store internally catches errors and sets state, the `async` function returns a rejected promise if something unexpected happens before the try/catch (e.g., `db` is undefined). Why: Unhandled promise rejections can crash the app in strict mode or cause silent data loss for learners. Fix: Add `.catch(console.error)` to both calls, or wrap in an async IIFE with try/catch:
  ```typescript
  useEffect(() => {
    loadReviews().catch(e => console.error('[ReviewQueue] load failed:', e))
    loadNotes().catch(e => console.error('[ReviewQueue] notes load failed:', e))
  }, [loadReviews, loadNotes])
  ```
  Pattern from: E03-S03, E07-S02, E9B-S06 (recurring fire-and-forget).

- **`src/stores/useReviewStore.ts:47` (confidence: 85)**: `new Date()` called inside `rateNote` makes the store non-deterministic at test time. The unit tests in `useReviewStore.test.ts` do not control the `now` value passed to `rateNote` -- the test on line 100 asserts `updatedReview!.nextReviewAt` is greater than `FIXED_DATE`, but this works only because the actual `new Date()` at test time is well past the fixed date. This is fragile: if someone changes the test dates or the machine clock drifts, assertions could fail non-deterministically. Why: Flaky unit tests erode developer confidence and slow CI. Fix: Accept an optional `now?: Date` parameter in `rateNote` (consistent with `getDueReviews` which already accepts `now`), default to `new Date()` in production, and pass `FIXED_DATE` in tests.

- **`src/app/pages/ReviewQueue.tsx:36` (confidence: 80)**: `const [now] = useState(() => new Date())` creates a stable time reference for the page session, but it means **retention percentages and due status never update** while the page remains mounted. A learner who keeps the review queue open for 30+ minutes will see stale retention values and notes that become due during the session won't appear. Why: Learners in active study sessions may keep the page open for extended periods; stale data reduces trust in the system. Fix: Consider a `useEffect` interval that updates `now` every 60 seconds, or re-derive on each rating action. A simpler approach is to compute `now` fresh in the `useMemo` dependency by adding a `refreshKey` counter that increments after each rating.

- **`src/stores/useReviewStore.ts:86-92` (confidence: 82)**: The toast retry button calls `get().retryPendingRating()` without awaiting or catching. Since `retryPendingRating` calls `rateNote` which can throw (if `persistWithRetry` exhausts retries again), the click handler produces an unhandled promise rejection. Why: A learner retrying a failed save could trigger a console error with no visible feedback if the retry also fails. Fix: Wrap in a `.catch()` to ensure the second failure is also surfaced:
  ```typescript
  onClick: () => {
    get().retryPendingRating().catch(() => {
      // rateNote already shows a new toast on failure
    })
  }
  ```

#### Medium

- **`src/app/pages/ReviewQueue.tsx:159` (confidence: 75)**: Non-null assertion `noteMap.get(record.noteId)!` on line 159 is guarded by the `validReviews` filter above (line 70), but a race condition exists: if notes are loaded asynchronously and `validReviews` is computed before `noteMap` fully populates, the assertion could throw. The `useMemo` dependencies (`dueReviews` and `noteMap`) are derived from different stores (`allReviews` from review store, `notes` from note store), and their loading is not coordinated. Fix: Add a defensive check inline: `const note = noteMap.get(record.noteId); if (!note) return null;` inside the map callback, or add a combined loading guard.

- **`src/app/pages/ReviewQueue.tsx:41-44` (confidence: 72)**: `loadReviews` and `loadNotes` are store functions created via `create()` -- their references are stable across renders (Zustand guarantees this). However, listing them as useEffect dependencies is misleading because they'll never change. This is harmless but could confuse future maintainers into thinking the effect re-runs when the functions change. Fix: Use `// eslint-disable-next-line react-hooks/exhaustive-deps` with a comment explaining Zustand stability, or reference them outside the dependency array.

- **`tests/e2e/story-e11-s01.spec.ts:239` (confidence: 78)**: AC5 E2E test is `test.skip`'d with a comment explaining Dexie doesn't expose a hookable `put()` path. While the justification is reasonable and unit tests cover the logic, a skipped AC5 E2E means the toast-retry flow is never validated end-to-end (toast rendering, keyboard reachability of retry button, visual rollback). Why: The acceptance criteria specifically requires toast with retry; E2E is the only way to verify the UI contract. Fix: Consider an alternative simulation approach -- e.g., monkey-patching `Dexie.prototype` at the page level, or injecting a `__TEST_FAIL_NEXT_WRITE__` flag that the store checks in dev mode.

- **`src/lib/spacedRepetition.ts:42` (confidence: 70)**: `calculateNextReview` has a default parameter `now: Date = new Date()`. While this is fine for production use, it means any caller that forgets to pass `now` gets a non-deterministic result. The store's `rateNote` does call it with `now` (line 48-52 of the store), so this is currently safe. But `calculateNextReview` is exported -- future callers could omit `now`. Fix: Consider removing the default and requiring explicit `now` parameter to enforce determinism at the API boundary.

#### Nits

- **Nit** `src/app/components/figma/ReviewCard.tsx:20-24` (confidence: 65): `getRetentionColor` is defined but its output is only used inside the Badge's `<span>` (line 93), while `getRetentionBadgeClasses` already applies both background and text color to the Badge itself. The inner `<span className={getRetentionColor(retention)}>` redundantly applies `text-destructive`/`text-warning`/`text-success` when the parent Badge already has these same text color classes from `getRetentionBadgeClasses`. Fix: Remove the inner `<span>` wrapper and `getRetentionColor`, or ensure the two functions serve distinct purposes.

- **Nit** `tests/e2e/story-e11-s01.spec.ts:29` (confidence: 60): `crypto.randomUUID()` in test factory `createDueReview` introduces non-deterministic IDs. While this doesn't affect test correctness (IDs are not asserted), deterministic IDs make debugging failed test runs easier since you can correlate IDB state with expected data. Fix: Use sequential IDs like `review-${noteId}` or import from a test factory.

- **Nit** `src/stores/useReviewStore.ts:55` (confidence: 60): `crypto.randomUUID()` generates the review record ID. This is fine for production but means the store test on line 98 (`useReviewStore.test.ts`) must use `.find()` to locate the updated record rather than asserting by ID. Minor, but a factory pattern with injectable ID generation would improve test precision.

### Recommendations

1. **First**: Add `.catch()` to the fire-and-forget `loadReviews()`/`loadNotes()` calls in `ReviewQueue.tsx` useEffect (H1). Quick fix, breaks the recurring pattern.
2. **Second**: Add `now` parameter to `rateNote` and pass it from tests (H2). Prevents future flakiness.
3. **Third**: Add `.catch()` to the toast retry `onClick` handler (H4). Ensures retry failures are visible.
4. **Fourth**: Consider a staleness refresh strategy for the `now` state in ReviewQueue (H3). Important for real usage but lower urgency since the page is new.
5. **Fifth**: Address the AC5 E2E skip if feasible (M3). If not, document the gap prominently in the story file.
