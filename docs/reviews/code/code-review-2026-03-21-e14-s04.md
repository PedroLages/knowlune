#### High Priority

- **[src/app/components/quiz/MarkdownRenderer.tsx:21-27] (confidence: 85)**: Inline vs block code detection is fragile. The `code` component uses `className` (which react-markdown only sets for fenced blocks WITH a language tag, e.g., `` ```javascript ``) to distinguish inline from block code. Fenced blocks WITHOUT a language specifier (`` ``` ``) get `className === undefined`, meaning they receive inline code styling (`bg-muted`, padding, rounded). The `<pre>` override with `[&>code]:bg-transparent [&>code]:p-0` compensates via CSS specificity, but this is a fragile coupling â€” any refactor to the `pre` override will silently break unstyled fenced blocks. **Why**: A learner writing a plain code block without a language tag sees subtly different DOM classes than one with a language tag, and the only safety net is a CSS specificity hack. **Fix**: Check the `node` parameter to detect if the parent is `<pre>`, which is the canonical way to distinguish inline vs block code in react-markdown:
  ```tsx
  code: ({ children, className, node }) => {
    const isBlock = node?.position && node?.properties?.className
    // Or simpler: rely on the pre wrapper entirely and only style inline code here
    const isInline = !className && !(node as any)?.properties?.className
    // ...
  }
  ```
  Or restructure so the `code` component only handles inline styling and the `pre` component fully handles block code styling (including its child `<code>`).

- **[tests/e2e/story-14-4.spec.ts:973-984] (confidence: 92)**: AC2 contrast test does not verify contrast. The test is named "code block background contrasts with code text (>=4.5:1)" and the story file claims "Contrast test computes actual luminance from computed styles and verifies WCAG AA ratio." In reality, the test only asserts the background color is not empty and not transparent â€” it never computes luminance or a contrast ratio. This is a false-positive test that will always pass regardless of actual contrast. **Why**: If a theme change breaks contrast below 4.5:1, this test will not catch it, and the AC will appear satisfied when it is not. **Fix**: Compute the actual WCAG relative luminance from both the background color and the text color, then assert the ratio:
  ```typescript
  const { bg, fg } = await pre.evaluate(el => {
    const cs = getComputedStyle(el)
    const code = el.querySelector('code')
    return { bg: cs.backgroundColor, fg: code ? getComputedStyle(code).color : cs.color }
  })
  // Parse RGB, compute luminance, assert ratio >= 4.5
  ```

#### Medium

- **[Recurring] [src/app/components/quiz/MarkdownRenderer.tsx:25] (confidence: 80)**: String template literal for className instead of `cn()`. Line 25 uses `` `text-foreground font-mono text-sm ${className}` `` which bypasses Tailwind Merge conflict resolution. If `className` contains `text-base` or another conflicting utility, both classes will be emitted and the last one in CSS source order wins (unpredictable). **Fix**: Use `cn('text-foreground font-mono text-sm', className)` and import `cn` from `@/app/components/ui/utils`.

- **[src/app/components/quiz/questions/markdown-config.tsx] (confidence: 90)**: Dead code. This file (`REMARK_PLUGINS`, `MARKDOWN_COMPONENTS`) is no longer imported anywhere after the refactor. All four question components now import from `MarkdownRenderer` instead. **Fix**: Delete `src/app/components/quiz/questions/markdown-config.tsx`.

- **[tests/e2e/story-14-4.spec.ts] (confidence: 75)**: AC2 specifies "code blocks and inline code render correctly in both light and dark themes using design tokens" but no E2E test exercises dark mode. All tests run in the default (light) theme. While the design tokens guarantee dark mode works in theory, the AC explicitly calls it out. **Fix**: Add a test (or extend the contrast test) that toggles dark mode (`document.documentElement.classList.add('dark')`) and verifies code block and inline code styling still applies.

- **[src/app/components/quiz/MarkdownRenderer.tsx:42-49] (confidence: 72)**: XSS is safe but undocumented. `react-markdown` strips raw HTML by default (no `rehype-raw`), which is the correct choice for rendering user/author-provided Markdown in quiz questions. However, this security-critical assumption is not documented. If someone adds `rehypeRaw` to enable HTML rendering later, it would open an XSS vector. **Fix**: Add a comment near the `Markdown` usage: `// Intentionally NOT using rehype-raw â€” raw HTML is stripped for safety`.

#### Nits

- **Nit** [src/app/components/quiz/MarkdownRenderer.tsx:15] (confidence: 65): The `<p>` component override applies `my-2` spacing unconditionally. When a question starts with a paragraph (the common case), this adds unnecessary top margin above the question text. Consider `first:mt-0` or using `space-y-2` on the wrapper instead.

- **Nit** [tests/e2e/story-14-4.spec.ts:877-879] (confidence: 60): `seedQuizData` seeds after initial navigation (`page.goto('/')` then `seedQuizData` then `page.goto(.../quiz)`). This double-navigation pattern works but is slower than seeding before the first navigation. The sidebar localStorage seed is already in `addInitScript`, so the IndexedDB seed could follow the same pre-navigation pattern if the helper supports it.
