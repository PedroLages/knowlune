## Code Review: E14-S04 — Support Rich Text Formatting in Questions (Re-Validation)

### What Works Well

1. **Clean `aria-labelledby` pattern across all 4 question components.** The `<fieldset aria-labelledby={labelId}>` + `<div id={labelId}>` pattern is consistently applied, maintaining HTML validity while preserving screen reader accessibility. The `useId()` hook ensures unique IDs even under concurrent rendering.

2. **Design token discipline.** Every color in `MarkdownRenderer` uses semantic tokens (`bg-surface-sunken`, `bg-muted`, `text-foreground`, `border-border`). No hardcoded colors escaped the ESLint gate. The `[&>code]` reset technique on `<pre>` to override inline code styles is clever and avoids duplication.

3. **Test coverage breadth.** 10 E2E tests covering all 4 ACs, plus unit tests for each Markdown element type. The factory-based test data setup (`makeQuiz`, `makeQuestion`) follows established project patterns, and the sidebar seed (`eduvi-sidebar-v1`) is correctly included. The contrast ratio computation using canvas pixel readback for oklch resolution is a strong testing technique.

### Findings

#### Blockers

- **[tests/e2e/story-14-4.spec.ts:221] (confidence: 95)**: Dark mode E2E test uses `page.emulateMedia({ colorScheme: 'dark' })` but the app uses **class-based** dark mode. The theme system in `src/styles/theme.css:1` defines `@custom-variant dark (&:is(.dark *))` and dark overrides live under `.dark` at line 111. There is no `prefers-color-scheme` media query anywhere in the stylesheet. `emulateMedia({ colorScheme: 'dark' })` sets the CSS media feature, which the app completely ignores -- the test runs in light mode and measures light-mode contrast, making it a false positive. **Why**: AC2 explicitly requires "code blocks and inline code render correctly in both light and dark themes using design tokens." This acceptance criterion is unverified. **Fix**: Replace `emulateMedia` with class-based dark mode activation:
  ```typescript
  // Before navigateToQuiz, add .dark to <html>
  await page.addInitScript(() => {
    document.documentElement.classList.add('dark')
  })
  ```
  Or use `page.evaluate` after navigation: `await page.evaluate(() => document.documentElement.classList.add('dark'))`.

#### High Priority

- **[src/app/components/quiz/MarkdownRenderer.tsx:31] (confidence: 82)** `[Recurring]`: String template `` `text-sm ${className}` `` inside `cn()` call. The `className` prop from react-markdown contains language identifiers like `language-javascript`. While this works today because language classes don't conflict with Tailwind utilities, string interpolation inside `cn()` defeats Tailwind Merge's class conflict resolution. If `className` ever contains a conflicting `text-*` class, `cn()` cannot deduplicate it. This pattern has been flagged since E01-S03 and was supposedly fixed in this story. **Why**: Inconsistency with project convention where `cn()` is used specifically to handle class merging. **Fix**: Use `cn('text-sm', className)`:
  ```typescript
  className ? cn('text-sm', className) : 'bg-muted text-[0.875em] px-1.5 py-0.5 rounded'
  ```

#### Medium

- **[tests/e2e/story-14-4.spec.ts:188-215,231-257] (confidence: 78)**: The `colorToRgb` + `luminance` contrast ratio computation is duplicated verbatim across the light mode and dark mode tests (~30 lines each). When the dark mode test is fixed per the blocker above, this duplication will still exist. **Fix**: Extract to a shared helper function at the top of the test file:
  ```typescript
  async function getContrastRatio(locator: Locator): Promise<number> {
    return locator.evaluate(el => { /* shared colorToRgb + luminance logic */ })
  }
  ```

- **[src/app/components/quiz/MarkdownRenderer.tsx:45-47] (confidence: 72)**: The `<img>` component renders user-provided `src` attributes from Markdown without URL validation. While `react-markdown` (without `rehype-raw`) limits this to Markdown image syntax (`![alt](src)`), the `src` value comes from quiz question text which could be compromised or author-injected. A `javascript:` URL in an `<img src>` is not exploitable in modern browsers, but tracking pixels and SSRF-style URLs (internal network probing) remain risks. **Fix**: Add a URL scheme guard:
  ```typescript
  img: ({ src, alt }) => {
    if (!src || !/^https?:\/\/|^\//.test(src)) return null
    return <img src={src} alt={alt ?? ''} className="max-w-full h-auto rounded my-2" />
  }
  ```

#### Nits

- **Nit** [src/app/components/quiz/MarkdownRenderer.tsx:21] (confidence: 60): The `<p>` override applies `my-2` unconditionally. When a question begins with a paragraph (the common case), this creates unnecessary top margin above the question text inside the `MarkdownRenderer` wrapper `<div>`. Consider `first:mt-0` on the `<p>` or `[&>*:first-child]:mt-0` on the wrapper to tighten the top edge.

- **Nit** [tests/e2e/story-14-4.spec.ts:80-82] (confidence: 55): `seedQuizData` is a single-line wrapper around `seedIndexedDBStore`. Consider inlining it into `navigateToQuiz` to reduce indirection -- the abstraction adds no reuse value since it's only called from one place.

### Recommendations

1. **First (Blocker)**: Fix the dark mode E2E test to use class-based activation (`document.documentElement.classList.add('dark')`) instead of `emulateMedia`. This is the only issue that leaves an acceptance criterion unverified.
2. **Second (High)**: Replace `\`text-sm ${className}\`` with `cn('text-sm', className)` in MarkdownRenderer. One-liner fix.
3. **Third (Medium)**: Extract the duplicated contrast ratio computation into a shared helper, ideally when fixing the dark mode test.
4. **Fourth (Medium)**: Add URL validation to the `<img>` override for defense-in-depth.
