## Code Review: E02-S10 — Caption and Subtitle Support

### What Works Well

1. **Blob URL lifecycle management is thorough.** Both LessonPlayer and ImportedLessonPlayer correctly track blob URLs in refs, revoke previous URLs before creating new ones, and clean up on unmount. This prevents memory leaks that compound as learners navigate between lessons.

2. **Clean separation of concerns.** The VideoPlayer stays presentational (receives `onLoadCaptions` callback), while page components own parsing, validation, persistence, and error feedback. This preserves the existing architecture boundary.

3. **Cancellation pattern on async loads.** The `let cancelled = false` guard in the caption-loading effect prevents stale state updates when a learner navigates away before the Dexie read resolves. Well-applied React pattern.

### Findings

#### High Priority

- **[Recurring] `src/app/pages/LessonPlayer.tsx:379`, `src/app/pages/ImportedLessonPlayer.tsx:130` (confidence: 92)**: `handleLoadCaptions` is an async callback with no try/catch. If `saveCaptionForVideo` throws (IndexedDB write failure, quota exceeded, file.text() rejection on corrupted file), the promise rejects unhandled. The learner sees nothing — no toast, no feedback. `saveCaptionForVideo` itself does `await file.text()` and `await db.videoCaptions.put(record)`, either of which can throw independently of the validation-level error return. **Why:** A learner with a full IndexedDB (common on mobile) will get a silent failure and think captions are broken. **Fix:** Wrap the body of `handleLoadCaptions` in try/catch:
  ```typescript
  const handleLoadCaptions = useCallback(async (file: File) => {
    if (!courseId || !lessonId) return
    try {
      const result = await saveCaptionForVideo(courseId, lessonId, file)
      if (!result.captionTrack) { toast.error(result.error); return }
      // ... existing success logic
    } catch (err) {
      toast.error('Failed to load captions. Please try again.')
      console.error('[captions] Load failed:', err)
    }
  }, [courseId, lessonId])
  ```

- **[Recurring] `src/app/pages/LessonPlayer.tsx:352`, `src/app/pages/ImportedLessonPlayer.tsx:105` (confidence: 90)**: `getCaptionForVideo().then(track => ...)` has no `.catch()`. If the IndexedDB read fails (database upgrade in progress, corrupted DB, permission denied), the promise rejects silently. The learner sees no captions and no error. **Why:** On subsequent visits, learners who previously loaded captions will wonder why they disappeared. **Fix:** Add `.catch()` with console.error at minimum:
  ```typescript
  getCaptionForVideo(courseId, lessonId)
    .then(track => { if (cancelled) return; /* ... */ })
    .catch(err => console.error('[captions] Failed to restore:', err))
  ```

- **`src/lib/captions.ts:75-81` (confidence: 85)**: The `srtToWebVTT` regex for removing sequence numbers (`/^\d+\s*\n(?=\d{2}:\d{2})/gm`) can silently corrupt SRT content if cue text contains lines starting with digits followed by a newline before a timestamp-like pattern. More critically, the regex only handles digit-only sequence number lines immediately before timestamps — but real SRT files can have blank lines of varying count or BOM characters. While edge-case, it means some valid SRT files will produce malformed WebVTT (duplicate sequence numbers left in as cue text), causing subtitles to display incorrectly. **Why:** Learners downloading SRT files from subtitle sites (OpenSubtitles, Subscene) encounter real-world format variations. **Fix:** Parse SRT structurally (split into blocks like `parseSRT` already does) rather than regex-replace:
  ```typescript
  export function srtToWebVTT(srtText: string): string {
    const cues = parseSRT(srtText)
    const vttCues = cues.map(c => {
      const start = formatVTTTime(c.startTime)
      const end = formatVTTTime(c.endTime)
      return `${start} --> ${end}\n${c.text}`
    }).join('\n\n')
    return `WEBVTT\n\n${vttCues}\n`
  }
  ```

#### Medium

- **`src/app/pages/LessonPlayer.tsx:345-399` and `src/app/pages/ImportedLessonPlayer.tsx:96-142` (confidence: 82)**: ~50 lines of identical caption logic duplicated between the two page components: state declarations, load effect, cleanup effect, handleLoadCaptions callback, and blob URL ref management. This violates DRY and means any bug fix must be applied in two places. **Fix:** Extract a custom hook `useCaptionLoader(courseId, lessonId)` that returns `{ userCaptions, handleLoadCaptions, mergedCaptions }`. Both pages consume it identically.

- **`src/lib/captions.ts:18-24` (confidence: 78)**: `parseTime` does no validation — `parseFloat` on unexpected input returns `NaN`, which propagates silently into `startTime`/`endTime` as `NaN`. The `validateCaptionFile` function only checks `cues.length === 0` but doesn't check for NaN timestamps within parsed cues. A partially malformed file (valid structure, garbled timestamps) would pass validation but produce broken captions where no cue ever matches the current playback time. **Fix:** Add a NaN guard in `parseTime` (return -1 or throw) and filter out cues with invalid timestamps in the parsers.

- **`src/app/components/figma/VideoPlayer.tsx:737` (confidence: 75)**: Track key changed from `caption.language` to `{caption.language}-${index}`. This is an improvement for uniqueness, but using array index as part of a React key means React won't properly reconcile when the user replaces their caption file (same index 0, different content). The blob URL changes but React may reuse the DOM `<track>` element without updating `src`. **Fix:** Use `caption.src` or `caption.label` as the key, which changes when the underlying data changes:
  ```typescript
  key={caption.src}
  ```

- **`src/lib/captions.ts:151` (confidence: 72)**: `createdAt: new Date().toISOString()` — while this is production code (not test code), the timestamp is never exposed to the user and is only used for storage. No issue per se, but note that this field has no consumer in the current codebase. If it's intended for future "last loaded" display, consider whether it should update on re-load or only on first load (current `put()` overwrites silently).

#### Nits

- **Nit** `src/lib/captions.ts:30` (confidence: 65): `TIMESTAMP_RE` is a module-level constant but `parseTime` is not guarded against empty-string input. While the regex upstream prevents empty strings from reaching `parseTime` in the normal flow, the function is exported and could be called directly with bad input.

- **Nit** `src/lib/captions.ts:88-93` (confidence: 60): `detectCaptionFormat` splits on `.` and takes `.pop()`, which works but returns `undefined` for files with no extension (no `.` in name). The downstream `if (!format)` check catches this via null comparison, but TypeScript types it as `string | undefined` flowing into a `'srt' | 'vtt' | null` return — works at runtime but is subtly type-imprecise.

- **Nit** `tests/e2e/story-e02-s10.spec.ts:124-148` (confidence: 60): AC3 test loads captions, presses C, then only asserts the button is visible — it doesn't verify the actual toggle state changed (e.g., `aria-pressed` flipped, or `aria-label` changed from "Disable" to "Enable"). The test proves the button exists but not that pressing C actually toggled anything.

### Recommendations

1. **First**: Add try/catch to both `handleLoadCaptions` callbacks (High — silent failure on IndexedDB write errors).
2. **Second**: Add `.catch()` to both `getCaptionForVideo` effect calls (High — silent failure on IndexedDB read errors).
3. **Third**: Consider structural SRT-to-WebVTT conversion instead of regex (High — real-world SRT variation).
4. **Fourth**: Extract duplicated caption logic into `useCaptionLoader` hook (Medium — maintainability).
5. **Fifth**: Add NaN guard to `parseTime` / validate timestamps in parsed cues (Medium — edge case robustness).
