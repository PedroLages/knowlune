#### Medium

- **`src/app/components/library/AboutBookDialog.tsx:107-111,151-152` (confidence: 70)** `[Correctness]`: Format display logic does not handle the `pdf` variant of `BookFormat`. The `BookFormat` type is `'epub' | 'pdf' | 'audiobook'` (types.ts:636), but the badge and metadata grid use `book.format === 'audiobook' ? 'Audiobook' : book.format.toUpperCase()`. For PDF books, `toUpperCase()` returns "PDF" which is visually correct. However, the badge styling uses `bg-brand-soft text-brand-soft-foreground` for all formats -- there is no visual differentiation between EPUB and PDF formats. This is not a bug today (the library may not support PDF yet), but the ternary logic assumes a binary audiobook/non-audiobook distinction that will become a three-way distinction when PDF support ships. Fix: Consider using a format-to-label map or a switch statement for clarity when PDF support arrives. Effort: ~5 minutes when the time comes. This is informational for now.

#### Nits

- **Nit** `tests/e2e/story-e107-s04.spec.ts:45,60,77` (confidence: 65): Test seed data uses `source: { type: 'local' }` without the required `opfsPath: string` field from the `ContentSource` type. Since seeding bypasses TypeScript via `Record<string, unknown>[]`, no compile error occurs, and the dialog never reads `source`. However, if library page components ever access `source.opfsPath` (e.g., to render a file path), these tests could fail. Fix: Add `opfsPath: '/test/book.epub'` to test data for completeness.
