## Findings

### HIGH — Unhandled exception if schedule has malformed `startTime`

**File:** `src/lib/icalFeedGenerator.ts:79–101` / `server/routes/calendar.ts:106`

**What:** `generateICalFeed()` calls `new Date(NaN)` when `schedule.startTime` is not in `"HH:MM"` format (e.g., empty string, null coerced to string, or corrupted DB row). `ical-generator` throws `Error: start has to be a valid date!` — verified in local test. The caller at `calendar.ts:106` has no try-catch around this call, so Express receives an unhandled rejection → 500 response (leaks stack trace in dev, crashes worker in prod).

**Why it matters:** Data from Supabase `study_schedules` is not validated before being passed to the generator. A single corrupted row makes the feed permanently unavailable (500) rather than gracefully degrading (skip bad event, return valid feed with remaining events).

**Fix (2 options):**

Option A — Validate `startTime` in `icalFeedGenerator.ts` and skip invalid schedules:
```typescript
// In generateICalFeed, before const [hours, minutes] = ...
const timeParts = schedule.startTime?.match(/^(\d{1,2}):(\d{2})$/)
if (!timeParts) {
  console.warn(`[icalFeedGenerator] Skipping schedule ${schedule.id}: invalid startTime "${schedule.startTime}"`)
  continue
}
const hours = parseInt(timeParts[1], 10)
const minutes = parseInt(timeParts[2], 10)
```

Option B — Wrap `generateICalFeed` call in try-catch in `calendar.ts`:
```typescript
let ical: string
try {
  ical = generateICalFeed(schedules, timezone)
} catch (err) {
  console.error('[calendar] iCal generation error:', (err as Error).message)
  res.status(500).send('Internal Server Error')
  return
}
```

Option A is preferred — fail-skip-continue (graceful degradation) is better UX than a 500.

**Effort:** ~10 min

### MEDIUM — `createClient` instantiated per request

**File:** `server/routes/calendar.ts:39`

**What:** `createClient(supabaseUrl, supabaseKey)` is called on every request. The Supabase JS client is not a lightweight object — it initializes internal auth state, realtime connection setup, and HTTP client. For a potentially frequently-polled calendar feed (calendar apps poll every 15–60 min per subscriber), this creates unnecessary allocation per request.

**Why it matters:** Low impact for few users, but the correct pattern in this codebase (per `server/middleware/entitlement.ts`) is to use a module-level singleton or create the client once at route initialization.

**Fix:**
```typescript
// Move outside the router.get handler, at module level:
const getSupabaseClient = (() => {
  let client: ReturnType<typeof createClient> | null = null
  return () => {
    if (!client) {
      const url = process.env.VITE_SUPABASE_URL!
      const key = process.env.SUPABASE_SERVICE_ROLE_KEY!
      client = createClient(url, key)
    }
    return client
  }
})()
// Then: const supabase = getSupabaseClient()
```

Note: The env var check should remain inside the handler (fail-fast if missing), but the client creation can be lazy-singleton.

**Effort:** ~15 min
