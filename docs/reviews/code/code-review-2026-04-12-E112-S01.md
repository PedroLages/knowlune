## Code Review: E112-S01 — Reading Speed, ETA & Time-of-Day Patterns

### What Works Well

1. **Clean service architecture.** `ReadingStatsService.ts` is well-structured with pure computation functions (`computeAverageReadingSpeed`, `computeETA`, `getTimeOfDayPattern`) separated from data fetching. The JSDoc comments with formula documentation make the code self-documenting.

2. **Robust edge case handling in usePagesReadToday.** The `estimatePagesFromSessions` helper correctly filters sub-30s sessions, caps estimates at `currentPage`, and handles both EPUB (CFI) and PDF (page) position types. The fallback-to-heuristic pattern is clean.

3. **Good test coverage for the new speed computation.** Tests cover null cases (no finished books, zero totalPages), the core formula, and the 90-day window filter. The `usePagesReadToday` tests cover speed integration, fallback, cap, and multi-book aggregation.

### Findings

#### High Priority

- **[Correctness] src/app/components/reports/ReadingStatsSection.tsx:71-76 (confidence: 92)**: Error fallback object is missing the required `avgReadingSpeedPagesPerHour` property from the `ReadingStats` interface. The catch block sets stats to `{ timeReadTodayMinutes: 0, booksInProgress: 0, totalBooksFinished: 0, readingTrend: [] }` but the interface requires `avgReadingSpeedPagesPerHour: number | null`. This is a TypeScript error that should fail type-checking. Why: If TypeScript strict mode catches this during build, it blocks deployment; if not, the missing field causes `stats?.avgReadingSpeedPagesPerHour` to be `undefined` rather than `null`, which still works at runtime but violates the type contract. Fix: Add `avgReadingSpeedPagesPerHour: null` to the fallback object. ~1 min.

- **[Correctness] src/services/ReadingStatsService.ts:136 (confidence: 85)**: The ETA formula `avgPagesPerDay = (avgSpeedPagesPerHour * (totalSeconds / 3600)) / (sessionCount / 7)` computes "total pages read in the last 30 days / (sessions per week)". This doesn't yield "pages per day" — it yields "pages per session-week". The intent appears to be: total pages read / number of days with activity. The current formula divides total pages by `sessionCount/7`, which assumes 7 sessions = 1 week, but `sessionCount` is sessions in 30 days, not per week. If a user had 30 sessions in 30 days, `sessionCount/7 = 4.28`, so `avgPagesPerDay = totalPagesRead / 4.28` — that's ~7x actual daily rate. Why: ETA will be wildly inaccurate for users with frequent sessions. Fix: Compute actual daily reading rate as `totalPagesRead / 30` (days in window) or count distinct reading days. ~15 min.

- **[Correctness] src/services/ReadingStatsService.ts:106 (confidence: 88)**: `computeETA` is exported but never called anywhere in the codebase. AC2 requires "each in-progress book shows an ETA" in the ReadingStatsSection, but the function is dead code — no component invokes it. Why: AC2 is not implemented in the UI. Fix: Either wire `computeETA` into `ReadingStatsSection` (or a new per-book ETA display), or document AC2 as deferred. ~30 min if wiring up UI.

#### Medium

- **[Testing] src/services/__tests__/ReadingStatsService.test.ts:300-310 (confidence: 78)**: `getTimeOfDayPattern` has zero behavioral tests — only an existence check (`typeof getTimeOfDayPattern === 'function'`). AC4's bucketing logic (Morning/Afternoon/Evening/Night boundaries, dominant calculation, <7 session threshold) is untested. The comment says "E2E tests will validate" but no E2E spec exists for this story. Why: The bucketing boundary logic (especially the Night wrap-around) is the most error-prone part and has no test coverage. Fix: Add unit tests that mock sessions with known hours and verify bucket counts and dominant period. ~20 min.

- **[Testing] src/services/__tests__/ReadingStatsService.test.ts:180-188 (confidence: 75)**: `getReadingStats` test only checks the function exists (`typeof getReadingStats === 'function'`). Since this is the main aggregation entry point that combines all sub-functions, it should have at least one integration-level test verifying the returned shape. Fix: Add a test that mocks the DB to return minimal data and asserts all fields are present with correct types. ~10 min.

- **[AI Smell] src/services/ReadingStatsService.ts:56-96 (confidence: 72)**: `computeAverageReadingSpeed` sums `totalPages` across ALL finished books but only sums session duration for sessions in the last 90 days. If a book was finished 2 years ago but has no recent sessions, its pages still inflate the numerator while its reading time is excluded from the denominator. This produces an inflated speed. Why: Speed metric will be unrealistically high for users with old finished books. Fix: Only include `totalPages` for books that have at least one session in the 90-day window. ~10 min.

#### Nits

- **Nit [Maintainability]** src/services/ReadingStatsService.ts:64 (confidence: 65): `new Date()` used without deterministic time in a service function. While this is production code (not test code), it makes the 90-day window untestable without fake timers. The tests do use `vi.useFakeTimers()` which covers this, so this is not blocking — just noting the implicit dependency.

### Recommendations

1. **Fix the ETA formula (High).** The `avgPagesPerDay` calculation is mathematically incorrect. This is the highest-impact fix since it affects user-facing data accuracy.
2. **Wire up `computeETA` or defer AC2.** Dead code that represents an unimplemented AC needs resolution.
3. **Add `avgReadingSpeedPagesPerHour: null` to the error fallback.** One-line fix for type safety.
4. **Fix speed inflation from old books.** Filter `totalPages` to only books with recent sessions.
5. **Add behavioral tests for `getTimeOfDayPattern`.** The bucketing logic is the riskiest untested code.
