# Code Review: E13-S03 -- Pause and Resume Quiz

## What Works Well

1. **Dual localStorage strategy is well-architected.** The Zustand persist middleware (`levelup-quiz-store`) handles same-session rehydration while the per-quiz `quiz-progress-{quizId}` key provides cross-session crash recovery. The `subscribe` listener at `useQuizStore.ts:277-283` is the right pattern -- it fires synchronously on every state change, avoiding the need to duplicate writes in every action.

2. **Zod validation on localStorage reads is solid.** `loadSavedProgress()` in `Quiz.tsx:37-53` uses `QuizProgressSchema.safeParse()` with proper error logging and graceful fallback to null. This prevents corrupted or tampered localStorage data from crashing the app.

3. **beforeunload safety net is correctly scoped.** The effect at `Quiz.tsx:186-197` only registers the listener when a quiz is active (`isQuizActive`), reads fresh state from `useQuizStore.getState()` (avoiding stale closures), and properly cleans up on unmount.

## Findings

### Blockers

None.

### High Priority

- **[Recurring] `src/stores/useQuizStore.ts:277-283` (confidence: 82)**: Subscribe listener writes to localStorage on every state change but has no error handling. `localStorage.setItem` can throw when storage is full (`QuotaExceededError`). Since this is the crash recovery mechanism, a silent failure here means the learner loses progress on browser crash. Why: If a learner's device has limited storage, the one safety net they rely on silently fails. Fix: Wrap in try/catch with `console.error` logging. Consider a toast or subtle indicator if the per-quiz backup fails.

  ```ts
  useQuizStore.subscribe(state => {
    const progress = state.currentProgress
    const quiz = state.currentQuiz
    if (progress && quiz) {
      try {
        localStorage.setItem(`quiz-progress-${quiz.id}`, JSON.stringify(progress))
      } catch (err) {
        console.error('[useQuizStore] Per-quiz localStorage sync failed:', err)
      }
    }
  })
  ```

- **`src/stores/useQuizStore.ts:277-283` (confidence: 78)**: Subscribe listener only handles the `progress && quiz` branch -- when `currentProgress` transitions to `null` (via `submitQuiz` or `clearQuiz`), the per-quiz key is NOT cleaned up by the subscriber. The explicit `localStorage.removeItem` calls in `submitQuiz` (line 145) and `clearQuiz` (line 186) handle this, but `startQuiz` does NOT clean up the old quiz's per-quiz key before starting a new one. Scenario: Learner starts quiz A, answers some questions (key `quiz-progress-A` written), then navigates away and starts quiz B without resuming A. Key `quiz-progress-A` remains orphaned forever. Why: Stale localStorage keys accumulate over time, wasting storage and potentially causing confusing "Resume" prompts for abandoned quizzes. Fix: In `startQuiz`, before setting new state, clear the old per-quiz key if one exists:

  ```ts
  // Clear any existing per-quiz backup before starting fresh
  const oldQuizId = get().currentProgress?.quizId
  if (oldQuizId) localStorage.removeItem(`quiz-progress-${oldQuizId}`)
  ```

- **`src/app/pages/Quiz.tsx:148-155` (confidence: 85)**: `handleResume` calls `localStorage.removeItem` at line 155, but `useQuizStore.setState` at line 148 triggers the subscribe listener which immediately writes the key back. The `removeItem` is dead code -- it executes but its effect is immediately undone. Why: Not a bug per se (the subscribe listener keeps the key in sync, which is correct during active quiz), but dead code creates confusion about the intended sync protocol. Future developers may add logic depending on the key being absent after resume. Fix: Remove the `localStorage.removeItem` at line 155 and add a comment explaining the subscribe listener owns the key lifecycle during active quiz.

### Medium

- **`src/stores/useQuizStore.ts:277-283` (confidence: 72)**: The subscribe listener fires on *every* state change (including `isLoading`, `error`, `attempts` changes), not just `currentProgress` changes. This means `JSON.stringify(progress)` + `localStorage.setItem` runs for every `loadAttempts` call, every `clearError`, etc. -- even when progress hasn't changed. Why: Minor performance concern -- `JSON.stringify` on every state change is wasteful when progress is unchanged. More importantly, it makes the sync behavior harder to reason about. Fix: Add a shallow equality check on the previous progress reference:

  ```ts
  let prevProgress: QuizProgress | null = null
  useQuizStore.subscribe(state => {
    const progress = state.currentProgress
    const quiz = state.currentQuiz
    if (progress && quiz && progress !== prevProgress) {
      try {
        localStorage.setItem(`quiz-progress-${quiz.id}`, JSON.stringify(progress))
      } catch (err) {
        console.error('[useQuizStore] Per-quiz localStorage sync failed:', err)
      }
    }
    prevProgress = progress
  })
  ```

- **`src/app/pages/Quiz.tsx:186-197` (confidence: 70)**: The `beforeunload` handler duplicates the exact same logic as the subscribe listener (`localStorage.setItem` with `JSON.stringify`). Since the subscribe listener fires synchronously on every state change, the per-quiz key is *always* up to date before `beforeunload` fires. The handler is redundant for all cases except one: if Zustand's internal state update is somehow incomplete at the moment the tab closes. Why: Redundancy is low-risk, but it adds maintenance surface. Fix: Keep the handler (defense in depth is reasonable for crash recovery), but add a comment explaining when it would actually trigger that the subscribe listener misses. If the answer is "never," consider removing it.

- **`src/app/components/quiz/QuizStartScreen.tsx:58` (confidence: 68)**: `autoFocus` is used on the Resume button. While this satisfies the AC, `autoFocus` can be jarring for screen reader users and disrupts the natural tab order for keyboard users who navigate from the top of the page. The WCAG 2.1 guideline 3.2.1 (On Focus) doesn't prohibit `autoFocus`, but the [Web Content Accessibility Guidelines - Understanding SC 3.2.1](https://www.w3.org/WAI/WCAG21/Understanding/on-focus.html) recommends caution. Fix: This is an acknowledged AC requirement, so keep it, but consider wrapping in a `useEffect` with `buttonRef.current?.focus()` so focus only moves after initial render, giving assistive technology time to announce the page first.

### Nits

- **Nit** `src/app/components/quiz/QuizActions.tsx:29,36` (confidence: 75): Icon sizing uses `size-4` (correct Tailwind v4 shorthand), but spacing uses `mr-1` and `ml-1` instead of `gap` on the button's flex container. The parent `<Button>` already uses `inline-flex items-center` via CVA, so `gap-1` on the button would be cleaner than margin on each icon. Low priority.

- **Nit** `src/app/pages/Quiz.tsx:184-185` (confidence: 65): `isQuizActive` is computed inline on every render. Since it depends on `currentProgress` and `quiz` which are already in state/selectors, this is fine, but extracting it as a `useMemo` would make the intent clearer and prevent the `useEffect` from re-running when unrelated state changes cause a re-render (though the effect itself is cheap).

## Recommendations

1. **First**: Add try/catch to the subscribe listener (High -- silent failure on the crash recovery mechanism).
2. **Second**: Clean up stale per-quiz localStorage keys in `startQuiz` (High -- storage leak over time).
3. **Third**: Remove the dead `localStorage.removeItem` in `handleResume` (High -- misleading dead code).
4. **Fourth**: Add shallow equality check to subscribe listener (Medium -- unnecessary work on every state change).
