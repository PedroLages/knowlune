## Code Review: E12-S03 — Create useQuizStore with Zustand

### What Works Well

1. **Snapshot rollback pattern is well-designed.** The `submitQuiz` action captures `{ currentQuiz, currentProgress }` before the async Dexie write and atomically restores on failure. This is a textbook optimistic-update-with-rollback and prevents the UI from showing "submitted" when persistence actually failed.

2. **`partialize` scope is correct and well-documented.** Only `currentProgress` is serialized to localStorage. The implementation notes explain why `currentQuiz` and `attempts` are excluded (stale definitions, recoverability from DB). The unit test verifies this via the persist API.

3. **Scoring logic is clean and well-separated.** `src/lib/scoring.ts` is a pure function with no side effects, handles all four question types, and correctly handles unanswered questions. The `maxScore > 0` guard prevents division by zero.

### Findings

#### Blockers

None.

#### High Priority

- **[Recurring] `src/stores/useQuizStore.ts:48-54` (confidence: 90)**: `startQuiz` has no try/catch around the Dexie query `db.quizzes.where('lessonId').equals(lessonId).first()`. If IndexedDB is unavailable (private browsing, quota exceeded, corrupted DB), this throws an unhandled rejection. The `set({ isLoading: true })` at line 49 will leave the store stuck in loading state permanently. Why: Learners in private browsing mode or with corrupted IndexedDB will see an infinite spinner with no error message. Fix: Wrap the Dexie query in try/catch, set `isLoading: false` and `error` in the catch block, similar to the pattern in `submitQuiz`. Pattern from: fire-and-forget async operations without error handling (recurring since E03-S03).

- **`src/stores/useQuizStore.ts:138-141` (confidence: 85)**: `loadAttempts` has no try/catch around its Dexie query. If `db.quizAttempts.where('quizId').equals(quizId).sortBy('completedAt')` fails, the promise rejects unhandled, and the caller gets no feedback. Why: A learner viewing their quiz history will see no error indication if the DB read fails. Fix: Wrap in try/catch, set error state, and consider a toast notification.

- **`src/lib/scoring.ts:63` / `src/types/quiz.ts:155` (confidence: 80)**: Type mismatch between scoring output and Zod schema. When a question is unanswered, `scoring.ts:63` sets `userAnswer: userAnswer ?? ''` (empty string). But `AnswerSchema.userAnswer` at `quiz.ts:155` is `z.union([z.string().min(1), z.array(z.string().min(1)).min(1)])` -- requiring `.min(1)`, so empty string fails validation. This won't cause a runtime error today since validation isn't called during scoring, but any future code that validates a `QuizAttempt` with `QuizAttemptSchema.safeParse()` will reject unanswered questions. Why: Zod schemas are declared as "the single source of truth" (quiz.ts line 8). The scoring output violates this contract. Fix: Either change `AnswerSchema.userAnswer` to allow empty string (`z.string()` instead of `z.string().min(1)`), or use a sentinel value like `'__unanswered__'` in scoring.ts. The schema and scoring logic must agree on how to represent "no answer."

- **`src/stores/useQuizStore.ts:117-121` (confidence: 78)**: Cross-store call `useContentProgressStore.getState().setItemStatus(...)` is awaited but has no error handling. If `setItemStatus` throws (e.g., its own Dexie write fails), the entire `submitQuiz` catch block fires, which reverts `currentProgress` and shows "Quiz attempt could not be saved" -- even though the quiz attempt WAS saved successfully to Dexie. Why: The learner sees a false failure message, their progress mark is lost, and `currentProgress` is restored (suggesting they need to resubmit), but resubmitting will create a duplicate `quizAttempts` record. Fix: Wrap the cross-store call in its own try/catch. Log the error but do NOT revert the quiz submission state -- the attempt is already persisted.

#### Medium

- **`src/stores/useQuizStore.ts:65-66` (confidence: 75)**: AC states `timeRemaining` should be set based on `quiz.timeLimit * timerAccommodation multiplier`, but the code sets `timeRemaining: quiz.timeLimit ?? null` with no multiplier applied and `timerAccommodation` is hardcoded to `'standard'`. The story file acknowledges this is "gated behind a future FR" but the AC text at line 39 explicitly states "sets timeRemaining based on quiz.timeLimit x timerAccommodation multiplier." This is a partial implementation of the stated AC. Fix: Either update the AC to defer the multiplier to a later story, or apply the multiplier now. If deferring, add a `// TODO(E##): Apply timerAccommodation multiplier` comment to make the gap explicit.

- **`src/stores/__tests__/useQuizStore.test.ts:526-530` (confidence: 72)**: The persist partialize test has a dead code branch at line 526-530 (`else { expect(true).toBe(true) }`) that silently passes if `persistApi` is falsy. This can never happen with Zustand v4+ (the test comment even says so), but a test that auto-passes on failure defeats its purpose. Fix: Remove the else branch and let the test fail if `persistApi` is unexpectedly missing.

- **`src/stores/useQuizStore.ts:96-107` (confidence: 70)**: The `QuizAttempt` object is constructed manually with `crypto.randomUUID()`, `Date.now()`, and `new Date().toISOString()`. This is correct but introduces a subtle inconsistency: `attempt.timeSpent` uses `Date.now() - currentProgress.startTime` (line 103), but `attempt.startedAt` reconstructs the start time from `new Date(currentProgress.startTime).toISOString()` (line 105). If `startTime` was stored as a non-integer or non-millisecond value, these would diverge. Consider extracting `startedAt` directly from `currentProgress.startTime` once (DRY) and documenting the unit expectation.

#### Nits

- **Nit** `src/stores/useQuizStore.ts:128` (confidence: 65): The bare `catch` (no parameter) at line 128 discards the original error object. While the error message is set in Zustand state and a toast is shown, `console.error` is never called with the original exception. In `toastHelpers.ts:96-98`, `toastError.saveFailed(details)` only `console.error`s if `details` is provided, but the store passes a hardcoded string. The actual Dexie error (e.g., "QuotaExceededError") is lost. Fix: Capture the error (`catch (err)`) and pass `err instanceof Error ? err.message : String(err)` as the details argument to `toastError.saveFailed`.

- **Nit** `src/stores/__tests__/useQuizStore.test.ts` (confidence: 60): Quiz data objects are repeated verbatim across 5+ tests. Consider extracting a `makeQuiz()` factory function (consistent with project convention of test factories at `tests/support/fixtures/factories/`) to reduce boilerplate and make tests easier to maintain.

### Recommendations

1. **First**: Add try/catch to `startQuiz` and `loadAttempts` (High #1, #2) -- these are the most likely to hit real users.
2. **Second**: Isolate the cross-store call in `submitQuiz` with its own error boundary (High #4) -- prevents false failure on successful submission.
3. **Third**: Reconcile the `AnswerSchema.userAnswer` min(1) constraint with scoring's empty-string default (High #3) -- schema/implementation divergence will bite in Epic 14 validation.
4. **Fourth**: Clarify the timerAccommodation AC gap (Medium #1) and capture the original error in the catch block (Nit #1).
