## Code Review: E68-S01 — Model Download Progress UI

### What Works Well

1. **Robust worker-worker communication protocol** — The `onPipelineProgress` callback in `embedding.worker.ts` correctly maps Transformers.js progress events to the typed `WorkerProgressUpdate` protocol, including careful handling of the indeterminate (total=0) edge case by clamping to -1. The `coordinator.ts` routing change (`isDownloadProgress` checked before the `requestId` guard) is correctly ordered and well-documented.

2. **Comprehensive defensive timeout strategy** — The `EmbeddingModelProgressToast` has two distinct timeout layers (15s first-progress, 120s stall) with proper cleanup paths. The code correctly cancels timers in the completion path, preventing stale timeout callbacks from firing after download completes.

3. **Thorough test coverage of progress event flow** — The coordinator test (`coordinator.test.ts`) has a dedicated `ProgressReportingWorker` mock that deliberately omits `requestId` to verify the defensive routing path, and all five ACs have corresponding tests.

### Findings

#### High Priority

- **`src/app/components/embeddings/EmbeddingModelProgressToast.tsx:133-145` (confidence: 85)** — False error toast fires on every visit when no model download is expected. The 15s first-progress timeout is set unconditionally on component mount, but the warm-up that triggers the download is gated on `supportsWorkers()` and `deviceMemory >= 4GB` (in `App.tsx:101-129`). On a device without worker support or with <4GB RAM, no download is initiated, yet after 15s the user sees "The AI model download did not start. Check your connection and reload the page." — a confusing and incorrect error.
  - **Why it matters**: Every visitor who doesn't meet the warm-up criteria will see a false error toast after 15s of idle browsing (reading Overview, changing Settings, etc.). This is a UX regression.
  - **Fix**: Either (a) gate the 15s timeout on receiving the first progress event (change timeout purpose to "download stalled" not "download never started"), or (b) gate component mount on the same conditions as the warm-up (`supportsWorkers()` && `deviceMemory >= 4`).
  - **Effort**: ~10 min
  - **Autofix**: `manual`

#### Medium

- **`src/ai/hooks/useModelDownloadProgress.ts` (confidence: 85)** — Unused hook with zero production callers and misleading documentation. The hook exists (115 lines, 4 exported types + 1 function) but the only consumer, `EmbeddingModelProgressToast.tsx`, imports only the `DownloadProgressDetail` type and implements its own progress-tracking logic using refs. The component's header comment says "Uses the useModelDownloadProgress hook" but no such call exists. This is both dead code (the hook function goes untested and uncalled) and misleading documentation.
  - **Why it matters**: The 115-line hook has 0 test coverage, 0 production callers, and creates maintenance surface area for no benefit. A developer reading the code will waste time understanding a hook that isn't used, then get confused by the contradicting documentation.
  - **Fix**: Either (a) refactor `EmbeddingModelProgressToast` to actually consume the hook and remove the duplicated ref-based logic, or (b) remove the hook file (keeping `DownloadProgressDetail` in types.ts if still needed). Option (b) is ~5min; option (a) is ~30min.
  - **Effort**: ~5-30 min
  - **Category**: `[AI Smell]` — AI-generated code with dead abstraction
  - **Autofix**: `manual`

- **`src/app/components/embeddings/EmbeddingModelProgressToast.tsx:82-83` (confidence: 90)** — Unreachable guard condition. The `if (hasCompletedRef.current) return` on line 82-83 is dead code: the `handleProgress` function checks the same ref at line 54 and returns early. The completion handler at lines 57-79 always sets `hasCompletedRef.current = true` and returns, so line 82-83 can never evaluate to `true` when reached. This will confuse maintainers.
  - **Why it matters**: Low severity, but dead code reduces clarity.
  - **Fix**: Remove lines 82-83 (the `if (hasCompletedRef.current) return` block).
  - **Effort**: 1 min
  - **Autofix**: `safe_auto`
  - **Category**: `[Maintainability]`

### Recommendations

1. Fix the false error toast (High): This is the most impactful issue — it affects every visit where warm-up is skipped. Gate the 15s timeout on the first progress event or gate the component mount.
2. Decide on the unused hook (Medium): Either delete it or integrate it. If it's kept for future use, mark it explicitly as `@internal` / `@experimental` so it's clear it's not production code.
3. Remove the two dead-code lines (Nit): Follows from #2 if the component's own logic is kept.
