## Findings

### MEDIUM: Race condition in concurrent tag merging — `src/lib/ollamaTagging.ts:56-60`

Both `triggerAutoAnalysis` (cloud) and `triggerOllamaTagging` (Ollama) run concurrently after import. Both read `course.tags` from the snapshot passed at call time (always `[]` for new imports), compute `merged` independently, and write to IndexedDB. If both produce tags, the second writer silently overwrites the first writer's tags.

**Impact:** Whichever finishes last wins. If cloud analysis returns `['python', 'ai']` and then Ollama returns `['machine-learning']`, the final state will be `['machine-learning']` — cloud tags are lost.

**Fix:** Read current tags from DB before merging:
```typescript
const freshCourse = await db.importedCourses.get(course.id)
const existingTags = freshCourse?.tags || []
const merged = [...new Set([...existingTags, ...result.tags])]
```

The same pattern exists in `autoAnalysis.ts:98-100` (pre-existing, same root cause).

### LOW: No `aria-live` region for tag completion state — `src/app/components/figma/ImportedCourseCard.tsx:401-409`

The "AI tagging..." indicator has `aria-live="polite"` which is good. However, when tagging completes and the indicator disappears, screen readers won't announce the newly added tags. Consider adding an `aria-live` region that announces "X tags added" on completion.

### LOW: `autoAnalysisStatus` memory not cleaned up — `src/lib/ollamaTagging.ts:72`

`autoAnalysisStatus` entries accumulate in the Zustand store forever. For users who import many courses over time, this map grows unboundedly. Consider cleaning up entries after a timeout or on page navigation.

### NIT: `triggerOllamaTagging` catch logs to `console.warn` while `runOllamaTagging` catch logs to `console.error` — `src/lib/ollamaTagging.ts:34,74`

Inconsistent log levels for essentially the same failure path. The outer `.catch()` in `triggerOllamaTagging` should never fire (the inner try-catch handles everything), but if it does, it should use `console.error` for consistency.
