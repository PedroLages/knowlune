#### High Priority

- **[Correctness] src/app/components/audiobook/AudiobookRenderer.tsx:266-279 (confidence: 90)**: Keyboard shortcuts `[` and `]` for speed adjustment only call `useAudioPlayerStore.getState().setPlaybackRate()` but do NOT call `useBookStore.getState().updateBookPlaybackSpeed(bookId, rate)`. This means speed changes made via keyboard are not persisted per-book to IndexedDB, violating AC-5 ("playback speed persists per-book"). The SpeedControl popover correctly calls both (`handleSelect` at SpeedControl.tsx:32-35). Why: Keyboard users lose their speed preference on navigation or page reload. Fix: Extract the persist logic into a shared function or add `useBookStore.getState().updateBookPlaybackSpeed(book.id, newRate)` after each `setPlaybackRate` call in both keyboard shortcut actions. Effort: ~5 minutes.

#### Medium

- **[Correctness] src/app/components/audiobook/AudiobookRenderer.tsx:266-279 (confidence: 75)**: The keyboard shortcuts use `0.25` as the minimum speed (`Math.max(0.25, current - 0.25)`) while `updateBookPlaybackSpeed` validates the range as `0.5-3.0`. If the keyboard persist is added per the HIGH finding above, speeds below 0.5 would be silently rejected by the store. Fix: Use `Math.max(0.5, ...)` to match the store validation boundary.
