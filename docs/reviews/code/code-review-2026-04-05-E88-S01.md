### Findings

#### HIGH -- No URL validation before fetch (confidence: 90)
**File:** `src/services/OpdsService.ts:52`
The `validateCatalog` function accepts any string as URL and passes it directly to `fetch()`. While the URL input has `type="url"`, server-side/service-level validation is missing. A malformed URL like `javascript:alert(1)` would be passed to fetch (which would fail safely, but it's defensive programming to validate). More importantly, credentials sent over `http://` are transmitted in cleartext. A warning when using HTTP with auth credentials would improve security posture.

#### MEDIUM -- Component exceeds 300-line threshold (confidence: 95)
**File:** `src/app/components/library/OpdsCatalogSettings.tsx:12`
At 402 non-blank lines, this component triggers the ESLint component-size warning. The list view, add/edit form, and delete confirmation are all in one file. Extractable candidates: `OpdsCatalogForm` (form fields + test/save logic) and `OpdsCatalogListItem` (individual catalog row with edit/delete).

#### MEDIUM -- ESLint no-silent-catch warnings are false positives (confidence: 85)
**File:** `src/services/OpdsService.ts:54,98`
The `error-handling/no-silent-catch` ESLint rule flags both catch blocks, but both return `{ ok: false, error: "..." }` to the caller -- this IS user-visible feedback via the Result pattern. The ESLint rule does not recognize the discriminated union return as user feedback. Consider adding `// silent-catch-ok: returns error result to caller` comments to suppress the warnings.

#### LOW -- Password stored in plaintext in IndexedDB (confidence: 80)
**File:** `src/data/types.ts:729-730`
The `OpdsCatalog.auth` field stores username and password as plain strings in IndexedDB. For a local-first app this is acceptable (IndexedDB is sandboxed per origin), but if Supabase sync (Epic 19) includes this table, credentials would be transmitted and stored server-side in plaintext. Track as tech debt for pre-sync encryption.

#### LOW -- clearTimeout may not fire on successful fetch (confidence: 70)
**File:** `src/services/OpdsService.ts:53`
The `clearTimeout(timeoutId)` on line 53 correctly clears the timeout after a successful fetch. However, if the `response.text()` call on line 86 is slow (large response), the abort could still fire after the fetch succeeds but before text parsing completes. The signal is already passed to fetch, so the abort would not affect `response.text()` -- but this is worth noting for awareness.

### Verdict: PASS (with 1 HIGH advisory)
The HIGH finding about URL validation is real but not a blocker for this story -- the feature targets technical users managing their own self-hosted servers, and HTTP is common on local networks. Recommend addressing in a follow-up.
