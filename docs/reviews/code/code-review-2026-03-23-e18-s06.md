## Code Review: E18-S06 -- Display Quiz Performance in Overview Dashboard

### What Works Well

1. **Clean module separation.** Extracting `quizMetrics.ts` instead of bloating `analytics.ts` is the right call. The two files serve distinct domains (study analytics vs. quiz metrics), and keeping them separate prevents the "god module" anti-pattern.

2. **Well-documented hardcoded completionRate.** The 100% completionRate is documented in both the JSDoc and inline comments with a forward reference to Story 17.1. The story's Lessons Learned section also explains the rationale. This is exemplary for future maintainers.

3. **Proper ignore-flag pattern in useEffect.** The async cleanup using `let ignore = false` with `ignore = true` on unmount is the React team's recommended pattern for Dexie queries that don't support AbortController. Consistent with the existing `getTotalStudyNotes()` effect in Overview.tsx (line 68-77).

### Findings

#### Blockers

- **[QuizPerformanceCard.tsx:112-113 nested inside :98-134] (confidence: 95)**: Nested interactive elements -- `<button>` card wraps `QuizEmptyState`, which contains `<Button asChild><Link to="/courses">Find Quizzes</Link></Button>` (line 67-69). This creates a `<button>` > `<a>` nesting that violates the HTML spec and WCAG 4.1.2 (Name, Role, Value). Screen readers announce inconsistent roles, keyboard focus trapping is unpredictable, and some browsers may ignore the inner interactive entirely. The `e.stopPropagation()` on the "View Detailed Analytics" link (line 126) prevents the *populated* state from double-navigating, but the empty state has no such guard -- the empty state `<Link>` click will *also* trigger the outer `<button>` click, navigating to `/reports?tab=quizzes` instead of `/courses`.
  Why: A learner with no quiz history clicks "Find Quizzes" expecting to land on `/courses` but gets redirected to `/reports?tab=quizzes` instead, or assistive technology users can't activate the link at all.
  Fix: Two options: (a) When `metrics.totalQuizzes === 0`, don't render the outer `<button>` -- use a plain `<div>` wrapper for the empty state (the card shouldn't be clickable when there's nothing to drill into). (b) Add `onClick={e => e.stopPropagation()}` to the `QuizEmptyState` Link, matching the pattern used for the analytics link. Option (a) is architecturally cleaner.

#### High Priority

- **[Recurring] [QuizPerformanceCard.tsx:82-91] (confidence: 90)**: Fire-and-forget async without error handling. `calculateQuizMetrics().then(result => ...)` has no `.catch()` handler. If IndexedDB is unavailable (private browsing quota exceeded, corrupt DB, permission revoked), the Promise rejects silently, `metrics` stays `null` forever, and the user sees an infinite skeleton. The component never transitions to an error or empty state.
  Why: Learners in restrictive browser environments see a perpetual loading skeleton with no way to recover. This is a recurring pattern (agent memory: "Fire-and-forget async operations without `.catch()` -- recurring since E03-S03").
  Fix: Add `.catch(() => setMetrics({ totalQuizzes: 0, averageScore: 0, completionRate: 0 }))` to gracefully degrade to the empty state, or introduce an `error` state with a retry button. At minimum, log the error with `console.error` so debugging is possible.

- **[QuizPerformanceCard.tsx:82-91, quizMetrics.ts:25-39] (confidence: 85)**: No error boundary or graceful degradation for Dexie failure at the `calculateQuizMetrics` level. The function itself does not wrap the `db.quizAttempts.toArray()` call in try/catch. While the component *could* catch via `.catch()`, neither layer handles failures. This violates the "defense in depth" principle for IndexedDB operations.
  Why: A corrupt IndexedDB or version mismatch during a Dexie schema migration causes an unhandled rejection that surfaces as a React error boundary crash (if one exists) or a white screen (if not).
  Fix: Wrap the Dexie call in `calculateQuizMetrics` with try/catch, returning the zero-state fallback on failure: `catch (error) { console.error('Failed to load quiz metrics:', error); return { totalQuizzes: 0, averageScore: 0, completionRate: 0 }; }`.

- **[tests/e2e/story-e18-s06.spec.ts] (confidence: 85)**: AC2 (skeleton loading state) is documented in the test file header comments but has no corresponding test case. The test suite has AC1, AC3, AC3b, and AC4 -- AC2 is missing entirely. The skeleton is visible only during the brief async gap before `calculateQuizMetrics` resolves, which is admittedly hard to test in E2E, but the acceptance criterion exists and is untested.
  Why: A regression that breaks the skeleton (e.g., changing the null check order) would go undetected.
  Fix: Add a test that asserts `quiz-performance-skeleton` is visible before data loads, then transitions. This can be done by intercepting or delaying the Dexie response, or by simply asserting the skeleton testid appears before the card testid on initial load (since E2E page load gives enough time to observe the skeleton).

#### Medium

- **[QuizPerformanceCard.tsx:99-103] (confidence: 75)**: The outer `<button>` has `className="w-full text-left ..."` with manual style resets to make it look like a card. This is a semantic concern -- the button's `cursor` defaults to `pointer` from browser UA styles (good), but `text-left` is fighting the button's default `text-align: center`. More importantly, the button has no explicit `type="button"` attribute. While there's no parent `<form>`, if this card is ever composed inside one, it will submit the form on click.
  Why: Defensive coding. Future composition could trigger unintended form submission.
  Fix: Add `type="button"` to the `<button>` element.

- **[QuizPerformanceCard.tsx:108] (confidence: 72)**: `opacity-60` on the ClipboardList icon uses Tailwind's built-in opacity utility, which is fine, but it creates a mismatch with how opacity is handled elsewhere in the codebase (many components use `text-muted-foreground` for subdued icons rather than opacity). The `opacity-60` approach means the icon color's contrast ratio depends on the background, making WCAG AA compliance harder to verify.
  Why: Inconsistent styling pattern. Other dashboard cards use `text-muted-foreground` for subdued icon styling.
  Fix: Replace `text-brand opacity-60` with `text-muted-foreground` for consistency with other dashboard cards, or verify the resulting contrast ratio meets 3:1 for non-text elements (WCAG 1.4.11).

- **[Overview.tsx:244-251] (confidence: 70)**: The Quiz Performance section uses inline `initial`/`whileInView`/`viewport`/`transition` props instead of the `variants={fadeUp}` pattern used by other sections (e.g., Hero Zone at line 213, Metrics Strip at line 229). This inconsistency means the Quiz Performance card animates independently from the stagger container, breaking the coordinated entrance animation rhythm.
  Why: Visual inconsistency -- the quiz card pops in on scroll while earlier sections stagger together on initial load.
  Fix: Use `<motion.section variants={fadeUp}>` to match the Hero Zone and Metrics Strip pattern. The `whileInView` approach is used by lower sections (Engagement Zone, Study History), so this is at least consistent with those -- but it means the card enters differently than the sections immediately above it. Consider which group it should belong to and match accordingly.

#### Nits

- **Nit** [QuizPerformanceCard.tsx:44] (confidence: 60): `MetricRow` uses `py-1` (4px vertical padding), which combined with `text-sm` (20px line height) gives a touch target height of about 28px. While these rows aren't interactive (they're `<span>` elements inside a `<div>`), the outer `<button>` covers them, so the overall card touch target is fine. No action needed, but noted for awareness if rows ever become individually interactive.

- **Nit** [quizMetrics.ts:33] (confidence: 55): `allAttempts.reduce((sum, a) => sum + a.percentage, 0) / totalQuizzes` assumes all `percentage` values are valid numbers. The Zod schema enforces `z.number().min(0).max(100)` at write time, but IndexedDB data could be manually corrupted. Extremely defensive, but worth noting that a `NaN` or `undefined` percentage would cascade into `NaN` for `averageScore`.

- **Nit** [story-e18-s06.spec.ts:50] (confidence: 65): `await expect(card.getByText('3')).toBeVisible()` is brittle -- the text "3" could match any element on the page containing "3" (e.g., "3 courses", "March 3"). Using `card.getByText('3')` scopes to the card, which helps, but a more resilient approach would be `card.locator(':text-is("3")')` for exact match or asserting within a more specific container.

### Recommendations

1. **Fix the nested interactive blocker first** (lines 112-113 + 55-72). Either conditionally render the outer `<button>` only when `totalQuizzes > 0`, or add `e.stopPropagation()` to the empty state Link.
2. **Add `.catch()` to the useEffect** (lines 82-91) to prevent infinite skeleton on IndexedDB failure.
3. **Add try/catch in `calculateQuizMetrics`** for defense in depth on the data layer.
4. **Add an AC2 skeleton test** or document why it's intentionally excluded.
5. **Add `type="button"`** to the outer button element.
6. **Align the motion animation pattern** with surrounding sections.
