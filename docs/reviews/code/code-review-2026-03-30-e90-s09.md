## Findings

### MEDIUM — Error message leaks upstream error detail to client

**File:** `server/routes/models.ts:122`, `server/routes/models.ts:135`

The error responses forward raw upstream error text and internal error messages to the client. While this follows the pattern used by existing OpenAI/Groq routes (pre-existing pattern), it can leak internal details. Consider sanitizing for production.

**Classification:** PRE-EXISTING (same pattern in OpenAI/Groq routes)

### LOW — `testConnection` catch swallows all errors silently

**File:** `src/lib/aiConfiguration.ts:213`

The `catch {}` block in `testConnection` returns `false` without logging. Acceptable for a boolean connectivity test, but a `console.warn` would help debugging.

**Classification:** STORY-RELATED (nit)

### LOW — `.slice(0, 50)` applied after filter but before sort

**File:** `src/lib/modelDiscovery.ts:333-338`

The 50-model limit is applied before sorting, which means the final sorted list may not contain the "top 50" by any meaningful ranking. OpenRouter returns models in an unspecified order. Consider sorting first, then slicing, or documenting the intent.

**Classification:** STORY-RELATED (nit)

### NON-ISSUE — Hardcoded `https://knowlune.app` referer

Static referer strings in `server/providers.ts:48` and `server/routes/models.ts:114` are appropriate for OpenRouter attribution headers. Not a secret.

### NON-ISSUE — API key regex validation

The `^sk-or-v1-[A-Za-z0-9]{48,}$` pattern in `aiConfiguration.ts` is a reasonable client-side format check. Server-side validation happens via the actual API call.

## Verdict

**PASS** — Clean, well-structured implementation following existing provider patterns. No blockers or high-severity issues.
