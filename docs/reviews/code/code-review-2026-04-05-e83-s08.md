## Findings

### MEDIUM

1. **Silent empty return in `fetchOpenLibraryMetadata` when offline** — `src/services/OpenLibraryService.ts:30`
   The function returns `{}` silently when offline. Callers may not know the fetch was skipped vs. genuinely found no metadata. Consider returning a discriminated result (e.g., `{ offline: true }`) or logging/toasting. The `BookMetadataEditor` has its own guard with a toast, but other callers of `fetchOpenLibraryMetadata` won't get user feedback.

### LOW

2. **`navigator.onLine` is unreliable** — Multiple files
   `navigator.onLine` only detects whether the device has a network interface, not actual internet connectivity. A user behind a captive portal or with a broken DNS will see `onLine: true` but requests will fail. The existing try/catch in the service handles this gracefully, so this is informational — the offline guards are a UX optimization, not a replacement for error handling.

3. **No E2E test for the offline badge** — `src/app/pages/Library.tsx:80`
   The `data-testid="library-offline-badge"` is added but no test exercises it. The existing `offline-awareness.spec.ts` tests a different page. Consider adding a Library-specific offline test in a follow-up.

## Verdict

**PASS** — No blockers. The changes are minimal, well-scoped, and follow existing patterns. The offline badge has proper accessibility attributes (role="status", aria-label). Design tokens are used correctly (bg-warning/10, text-warning-foreground).
