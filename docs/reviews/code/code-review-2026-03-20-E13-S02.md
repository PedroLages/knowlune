## Code Review: E13-S02 — Mark Questions for Review

### What Works Well

1. **Clean component decomposition with single responsibilities.** MarkForReview, ReviewSummary, QuestionGrid, QuizActions, and QuizNavigation each own one concern. Props are narrow and well-typed. The composition hierarchy in Quiz.tsx is easy to follow.

2. **Strong accessibility implementation.** Dynamic `aria-label` on grid buttons includes "marked for review" when flagged, `aria-current="step"` (correct token, not the vague `"true"`), `aria-hidden` on all decorative icons, `role="list"` on ReviewSummary, and proper `htmlFor`/`id` checkbox-label wiring. This is above average for the codebase.

3. **Defensive edge case handling.** ReviewSummary filters orphaned question IDs via `indexOf !== -1`, the store's `toggleReviewMark` no-ops when progress is null, `navigateToQuestion` validates `Number.isInteger` and bounds, and `QuestionGrid.isAnswered` correctly handles both string and array answer types.

### Findings

#### Blockers

None.

#### High Priority

- **`Quiz.tsx:168-178` (confidence: 88)**: `handleSubmitClick` only opens the submit confirmation dialog when `countUnanswered > 0`. When all questions are answered but some are marked for review, the quiz submits immediately without showing ReviewSummary. AC5 states "When I view the 'Questions Marked for Review' section...I see a list of all marked question numbers." This section is never shown when all questions are answered. Why: A learner who answered all questions but flagged some for uncertainty never gets the chance to revisit flagged questions before submission -- defeating the purpose of the "mark for review" feature. Fix: Open the dialog when `countUnanswered > 0 || currentProgress.markedForReview.length > 0`:
  ```tsx
  if (countUnanswered(q.questions, progress.answers) > 0 || progress.markedForReview.length > 0) {
    setShowSubmitDialog(true)
  } else {
    handleSubmitConfirm()
  }
  ```

- **`ReviewSummary.tsx:22` (confidence: 90)**: Count text uses `markedForReview.length` but buttons render from `markedIndices` (which filters out IDs not found in `questionOrder`). If a marked question ID is orphaned (e.g., quiz was updated while progress persists), the text says "2 questions marked" but only 1 button renders. Why: Learners see a count that does not match the visible jump links, creating confusion about which questions they flagged. Fix: Use `markedIndices.length` for the count:
  ```tsx
  {markedIndices.length} {markedIndices.length === 1 ? 'question' : 'questions'} marked for review:
  ```

- **[Recurring] `MarkForReview.tsx:14` (confidence: 85)**: The outer `<div>` has no minimum height constraint. The combined checkbox + label row measures only ~20px tall -- well below the 44px minimum touch target required by the design system and WCAG 2.1 SC 2.5.5. Why: On mobile, learners wanting to flag a question for review will frequently mis-tap this control, especially under exam pressure. This recurring pattern has been flagged since E02-S07. Fix: Add `min-h-[44px]` to the container:
  ```tsx
  <div className="flex items-center gap-2 mt-4 min-h-[44px]">
  ```

- **`QuestionGrid.tsx:51` (confidence: 82)**: The Lucide `<Bookmark>` SVG has `fill="none"` as a presentational attribute on the root element. The CSS class `fill-warning` targets the SVG root, but the SVG path inherits the presentational `fill="none"` rather than the CSS value. The bookmark renders as a faint outline at 12px -- nearly invisible as a review indicator. Why: The entire visual purpose of the bookmark indicator is to be a filled amber badge that signals "this question is flagged." An invisible indicator means learners cannot use the grid to identify flagged questions. Fix: Use the `fill` JSX prop to set `currentColor` directly:
  ```tsx
  <Bookmark className="size-3 text-warning" fill="currentColor" />
  ```

#### Medium

- **`MarkForReview.tsx:18` (confidence: 75)**: `onCheckedChange` receives `boolean | 'indeterminate'` from Radix but `onToggle` signature is `() => void`. TypeScript allows this assignment (a zero-arity function is assignable to a parameter-accepting callback), so there is no compile error. However, if the Checkbox ever emits `'indeterminate'`, the toggle fires without the component being aware of the state transition. Fix: Wrap explicitly for clarity and safety: `onCheckedChange={() => onToggle()}`.

- **[Recurring] `tests/e2e/story-e13-s02.spec.ts:63-99` (confidence: 80)**: Manual IndexedDB seeding via `indexedDB.open('ElearningDB')` retry loop duplicates the shared helper pattern. The story's own Testing Notes say "No manual IndexedDB seeding -- use the seeding helper." This pattern has been recurring since E03-S03. Why: When the DB name, retry logic, or schema changes, this spec breaks silently while specs using the shared helper keep working. Fix: Import and use `seedIndexedDBStore` from `tests/support/helpers/indexeddb-seed.ts`.

- **`ReviewSummary.tsx:20` (confidence: 75)**: `aria-label="Questions marked for review"` is placed on a plain `<div>`, which has no implicit ARIA role. Screen readers will not announce this label because `aria-label` is only effective on elements with a semantic role. Fix: Add `role="region"` alongside `aria-label`, or use a `<section>` element:
  ```tsx
  <section className="mt-3" aria-label="Questions marked for review">
  ```

- **`ReviewSummary.tsx:30` (confidence: 72)**: Jump buttons have `min-h-[44px]` but no `min-w-[44px]`. Single-character labels like "Q1" render at ~33px wide, below the 44px minimum touch target width. Fix: Add `min-w-[44px]` to the button className.

#### Nits

- **Nit** `QuizActions.tsx:42` (confidence: 60): Submit Quiz button uses `bg-brand text-brand-foreground hover:bg-brand-hover` as raw classes instead of `variant="brand"`, which was added specifically for this pattern (per project conventions). Using the variant ensures hover, focus, and disabled states are centrally managed.

- **Nit** `QuestionGrid.tsx:30` (confidence: 55): `markedForReview.includes(questionId)` is O(n) per bubble, giving O(n*m) total. Negligible for typical quiz sizes (< 50), but converting to a `Set` via `useMemo` would be more idiomatic. Fix: `const markedSet = useMemo(() => new Set(markedForReview), [markedForReview])` then `markedSet.has(questionId)`.

- **Nit** `Quiz.tsx:310-313` (confidence: 55): The inline arrow function `(idx) => { navigateToQuestion(idx); setShowSubmitDialog(false) }` creates a new function on every render inside the dialog. Consider extracting to a `useCallback` for stability.

### Recommendations

1. **First**: Fix the submit dialog gate (`Quiz.tsx:174`) to also open when `markedForReview.length > 0` -- this is an AC gap where learners who answered everything but flagged questions never see the ReviewSummary.
2. **Second**: Fix the Bookmark fill rendering (`QuestionGrid.tsx:51`) -- the core visual indicator of this feature is currently invisible.
3. **Third**: Fix the ReviewSummary count mismatch (`ReviewSummary.tsx:22`) -- use `markedIndices.length` instead of `markedForReview.length`.
4. **Fourth**: Add `min-h-[44px]` to MarkForReview container -- recurring touch target deficiency.
5. **Fifth**: Replace manual IDB seeding in E2E spec with shared helper.
