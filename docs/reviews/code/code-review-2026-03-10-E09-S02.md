## What Works Well

1. **Clean coordinator architecture.** `WorkerCoordinator` demonstrates solid separation of concerns: lazy spawning, idle timeouts, and per-request tracking via `requestId` are well-structured. The singleton pattern with `beforeunload` cleanup is appropriate for a global worker pool.

2. **Type-safe message protocol.** `types.ts` provides a comprehensive discriminated union for `WorkerResponse` with proper type guards (`isSuccessResponse`, `isErrorResponse`). This prevents ad-hoc type casting in message handlers.

3. **Search worker is clean and defensive.** `search.worker.ts` validates index state before executing queries and properly wraps all work in try/catch with structured error responses.

## Findings

### Blockers

**[B1] `terminate()` orphans pending promises — coordinator.ts:356-366** (confidence: 95)

`terminate()` calls `this.pendingRequests.clear()` without rejecting any pending promises. Every in-flight `executeTask()` call becomes a permanently hanging promise — it will never resolve or reject. This directly breaks AC3 (visibility change terminates workers) and AC4 (crash recovery rejects pending requests), because `terminate()` is called on `visibilitychange` and the promises are silently orphaned.

**Impact:** Any component awaiting an AI result when the user switches tabs will hang indefinitely, causing memory leaks and frozen UI states.

**Fix:** Iterate `pendingRequests`, call `pending.reject(new Error('Worker terminated'))` and `clearTimeout(pending.timeout)` for each, then clear the map.

---

**[B2] Global error handler sends response with no `requestId` — search.worker.ts:74-81** (confidence: 92)

The global `self.addEventListener('error')` handler posts a response with `type: 'error'` but **no `requestId`**. The coordinator's `handleMessage` filters on `response.requestId !== requestId` (line 116), so this crash response will never match any pending request. The coordinator will never reject the pending promise — it hangs until the 30s timeout.

**Impact:** AC4 requires "pending requests are rejected with descriptive error messages" on crash; instead, learners get a generic timeout 30 seconds later.

**Fix:** Include the last known `requestId` in the error response, or have the coordinator's global `worker.onerror` handler (line 198-201) directly reject all pending requests for that worker.

### High Priority

**[H1] Per-request `addEventListener` accumulates O(N²) listeners — coordinator.ts:113-127** (confidence: 88)

Each `sendMessage()` call adds a new `addEventListener('message', handleMessage)` to the worker. The listener is only removed when its specific `requestId` matches. If the worker crashes or `terminate()` is called, these listeners are never removed. For concurrent requests, N listeners accumulate, each inspecting every message.

**Fix:** Use a single persistent `onmessage` handler per worker (set in `spawnWorker`), route messages through `pendingRequests` map by `requestId`, eliminating per-request listener registration.

---

**[H2] `bulkSaveEmbeddings` has no try/catch — vector-store.ts:45-66** (confidence: 82)

If a batch fails mid-way (e.g., `QuotaExceededError` on IndexedDB), earlier batches are committed but later ones are lost — partial write with no rollback and no error surfaced to the caller. A learner's embedding index could become inconsistent, producing degraded search results.

**Fix:** Wrap the entire loop in a Dexie transaction (`db.transaction('rw', db.embeddings, async () => { ... })`) or at minimum propagate errors so callers can retry.

---

**[H3] Only 3/9 ACs covered by E2E tests — story-e09-s02-worker-infrastructure.spec.ts** (confidence: 90)

Missing: AC2 (idle termination), AC3 (visibility change), AC4 (crash recovery), AC5 (graceful degradation), AC8 (ES module workers), AC9 (`useWorkerCoordinator` cleanup). The story's core value — robust lifecycle management — is almost entirely untested.

**Fix:** At minimum add tests for AC4 (inject a worker that posts an error) and AC5 (override `Worker` to `undefined` and verify graceful failure).

---

### Medium

**[M1] `supportsModuleWorkers()` is misleading — workerCapabilities.ts:12-16** (confidence: 75)

Simply delegates to `supportsWorkers()`. Firefox versions 4-113 support Workers but not module workers. Since the coordinator uses `{ type: 'module' }` (line 194), calling `supportsModuleWorkers()` would incorrectly return `true` on those browsers, causing a runtime crash when spawning.

**Fix:** Implement actual detection or rename to `supportsWorkers()` and document the limitation.

---

**[M2] `scheduleIdleTermination` called on every request send — coordinator.ts:213-222** (confidence: 78)

`updateWorkerActivity` calls `scheduleIdleTermination` which clears and resets the 60s timer on every request. If requests are sent frequently (e.g., typing search), the timer keeps resetting and the worker never terminates. The timer should only be scheduled when `activeRequests` drops to 0 (which `decrementActiveRequests` already handles).

**Fix:** Remove the `scheduleIdleTermination` call from `updateWorkerActivity`.

---

**[M3] `worker.onerror` passes potentially-null `event.error` — coordinator.ts:198-201** (confidence: 74)

`ErrorEvent.error` can be `null` in many browsers (cross-origin/security errors). `handleWorkerError` calls `error?.message` which works but `console.error` on line 312 would log `null`.

**Fix:** Pass `event.error ?? new Error(event.message ?? 'Unknown worker error')` to ensure a proper Error object.

## Recommendations (Priority Order)

1. Fix `terminate()` to reject all pending requests before clearing
2. Fix search worker's global error handler to include `requestId`
3. Replace per-request `addEventListener` with single persistent message router
4. Add E2E tests for AC4 (crash recovery) and AC5 (no-worker fallback)
5. Wrap `bulkSaveEmbeddings` in a Dexie transaction
6. Remove redundant `scheduleIdleTermination` call in `updateWorkerActivity`
