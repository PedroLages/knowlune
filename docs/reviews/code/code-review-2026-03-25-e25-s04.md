# Code Review: E25-S04 — Author Auto-Detection During Import

**Date:** 2026-03-25
**Reviewer:** Claude Opus 4.6 (automated)
**Scope:** 11 files changed, 1264 insertions, 3 deletions

## Architecture Assessment

The implementation cleanly separates concerns into a pure detection function and an async DB layer. Integration into `persistScannedCourse()` is minimal and non-disruptive. The error handling follows the "best-effort enrichment" pattern correctly — author detection failure never blocks course import.

## Findings

### HIGH

**1. [code-review] Race condition in `matchOrCreateAuthor()` — `src/lib/authorDetection.ts:70-90`**
Confidence: 75/100

The function does a read-then-write without a Dexie transaction. If two imports with the same author name happen concurrently (e.g., user imports two folders rapidly), both could see no existing author and both create duplicates.

Mitigation: This is LOW risk in practice — the app imports one course at a time via a blocking directory picker, making concurrent calls to `matchOrCreateAuthor()` with the same name nearly impossible. However, wrapping lines 70-90 in `db.transaction('rw', db.authors, async () => { ... })` would make this bulletproof with minimal code change.

**2. [code-review] `matchOrCreateAuthor()` loads all authors into memory — `src/lib/authorDetection.ts:70`**
Confidence: 60/100

`db.authors.toArray()` fetches every author record for a linear scan. For a personal learning app this is fine (likely <100 authors), but if the table grows, this becomes an O(n) scan per import.

Mitigation: Could use a Dexie index on a normalized name field, or `db.authors.filter()`. Low priority given expected dataset size.

### MEDIUM

**3. [code-review] `new Date().toISOString()` in `matchOrCreateAuthor()` — `src/lib/authorDetection.ts:79`**
Confidence: 55/100

Using `new Date()` is fine in production code (the ESLint rule only applies to test files). However, the timestamp created here is independent from the `now` variable in `persistScannedCourse()`, meaning the author's `createdAt` and the course's `importedAt` may differ by milliseconds. This is cosmetic, not functional.

### NITS

**4. [code-review] Comment says "slug-based ID" but uses UUID — `src/lib/authorDetection.ts:77`**
Confidence: 95/100

The comment `// Create new author with slug-based ID` is misleading — the code uses `crypto.randomUUID()`, which produces a UUID, not a slug. Minor documentation inaccuracy.

**5. [code-review] `seedAuthors` parameter type is `Record<string, unknown>[]` — `tests/support/helpers/seed-helpers.ts`**
Confidence: 70/100

Other seed helpers in this file also use `Record<string, unknown>[]`, so this is consistent. But it means no type safety on the author shape at the call site. Consistent with existing patterns, so not blocking.
