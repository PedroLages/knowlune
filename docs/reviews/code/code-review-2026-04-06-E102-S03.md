## Findings

### MEDIUM

1. **CollectionCard.tsx:33-39** — O(n*m) book resolution in `useMemo`. The `allBooks.find()` inside a loop creates quadratic complexity. For large libraries (1000+ books) with large collections, this could cause noticeable lag. Consider building a `Map<absItemId, Book>` from `allBooks` first:
   ```ts
   const absIdToBook = useMemo(() => {
     const map = new Map<string, Book>()
     for (const b of allBooks) if (b.absItemId) map.set(b.absItemId, b)
     return map
   }, [allBooks])
   ```
   **Confidence: 75** — Only matters at scale; current usage is likely fine.

### LOW

2. **CollectionsView.tsx:21-26** — `useEffect` calling async `loadCollections()` without an ignore flag. The store's `collectionsLoaded` guard prevents double-fetch, but if the component unmounts mid-fetch, the `set()` call in the store will update unmounted state. Zustand handles this safely (no React warning), so this is informational only.

3. **CollectionCard.tsx:45** — `servers.find(s => s.status === 'connected')` is computed on every render outside of `useMemo`. Should be memoized or moved to the store as a selector.

### NIT

4. **CollectionCard.tsx:50-55** — `handleKeyDown` for Enter/Space is redundant on a native `<button>` element. Can be removed.

5. **CollectionCard.tsx:137** — `localBook.progress >= 100` — progress should never exceed 100; the `>=` is defensive but `=== 100` would be more precise.

## Security

- No secrets, no user input handling, no XSS vectors. `getCoverUrl()` constructs URLs from server config. Safe.

## Verdict

**PASS** — 1 MEDIUM (performance at scale), 2 LOW, 2 NIT. No blockers.
