### Findings

#### High Priority

- **[Correctness] `src/services/ReadingStatsService.ts:362` (confidence: 85)**: `new Date().getFullYear()` makes `getReadingSummary` non-deterministic. The "books finished this year" count depends on the system clock, and the test at `ReadingStatsService.test.ts:787` hardcodes `2026` dates without mocking `Date`, so the test will break on January 1, 2027. The story notes say to use `FIXED_DATE` for year-boundary tests but this was not done. Why: Tests silently rot and give false passes/fails depending on when they run. Fix: Either accept an optional `referenceDate` parameter (defaulting to `new Date()`) or use `vi.useFakeTimers({ now: FIXED_DATE })` in the test `beforeEach`. Effort: ~15 min.

- **[Correctness] `src/services/ReadingStatsService.ts:381-397` (confidence: 78)**: `avgPagesPerSession` divides total pages of *finished books only* by the session count of *all book sessions* (including sessions for in-progress and want-to-read books). This produces a misleadingly low average when a user has many sessions on books they haven't finished. AC3 says "Average pages per session (all-time)" which is ambiguous, but the denominator/numerator mismatch is a logic smell. Why: A user with 5 finished books (1000 pages total) and 100 sessions across all books would see "10 pages/session" when their finished-book sessions might actually average much higher. Fix: Either filter sessions to only those belonging to finished books (consistent numerator/denominator), or include all books' pages in the numerator. Effort: ~10 min.

#### Medium

- **[Correctness] `src/services/ReadingStatsService.ts:333-334` (confidence: 72)**: The genre cap and threshold interact in a way that can produce unexpected results. A genre with exactly 5% (count >= threshold) gets included in named, but the loop condition `named.length < 8 && count >= threshold` means a genre above 5% still gets pushed to "Other" once 8 named genres are already selected. This is likely intentional per AC2 ("legend caps at 8 genres"), but the combined condition means a genre at 6% could be grouped into "Other" if 8 larger genres exist. Consider documenting this edge in a code comment so future maintainers understand the intentional behavior. Fix: Add a comment: `// Intentional: 8-genre cap takes priority over 5% threshold`. Effort: ~2 min.

- **[Testing] `src/services/__tests__/ReadingStatsService.test.ts:624,639,656,679,705` (confidence: 75)**: All `getGenreDistribution` and `getReadingSummary` tests use dynamic `await import()` rather than the top-level import. This means each test gets a fresh module instance, which works but is fragile â€” it bypasses the mock setup flow and makes tests harder to read. The existing tests in this file (lines 1-607) use top-level imports with `vi.mocked()` for the same service. Why: Inconsistent test patterns reduce maintainability. Fix: Use the same `vi.mocked(db.books.where)` pattern with a top-level import of `getGenreDistribution` / `getReadingSummary`. Effort: ~15 min.

#### Nits

- **Nit [AI Smell]** `src/app/components/reports/GenreDistributionCard.tsx:107` (confidence: 60): The legend color swatch uses `style={{ backgroundColor: ... }}` which is an inline style. The ESLint rule `react-best-practices/no-inline-styles` would flag this, though CSS variable references via `style` are one of the few legitimate inline style use cases (Tailwind cannot use dynamic `var()` at build time). Acceptable as-is.
