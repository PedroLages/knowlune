## Findings

### HIGH — Double Toast on Regenerate (CalendarSettingsSection.tsx:78-81)

**What**: `handleRegenerate` calls `regenerateFeedToken()` and then immediately fires `toast.warning(...)`. But `regenerateFeedToken()` in the store (useStudyScheduleStore.ts:243) already fires `toast.success('Feed URL regenerated. Update your calendar subscription.')` on success. This results in two simultaneous toasts on a successful regeneration — one success and one warning — which is confusing.

**Why it matters**: Users see contradictory feedback. The success toast says "updated", the warning toast says subscriptions will stop. Both are technically correct but firing together creates noise and anxiety.

**Fix** (~5 min): Remove the duplicate `toast.warning()` from `handleRegenerate`. The store's success message should be updated to include the subscription warning: `toast.success('Feed URL regenerated. Re-subscribe your calendar apps with the new URL.')`. Or alternatively, suppress the store's toast and keep the component's warning-level toast.

```typescript
// Before (two toasts):
const handleRegenerate = useCallback(async () => {
  await regenerateFeedToken()
  toast.warning('Old calendar subscriptions will stop updating.')
}, [regenerateFeedToken])

// After (one toast — update the store message instead):
const handleRegenerate = useCallback(async () => {
  await regenerateFeedToken()
  // Store shows toast.success; don't duplicate
}, [regenerateFeedToken])
```

### MEDIUM — useEffect Re-runs loadFeedToken When isLoaded Changes (CalendarSettingsSection.tsx:38-52)

**What**: `isLoaded` is in the dependency array. When `loadSchedules()` completes, `isLoaded` becomes `true`, triggering the effect to re-run. On the second run, `loadFeedToken()` is called again unnecessarily (there's no guard on it, only on `loadSchedules`).

**Why it matters**: In the common path (first load), this causes two redundant `loadFeedToken()` calls — one on mount and one when `isLoaded` flips to `true`. If `loadFeedToken()` makes a Supabase request, this doubles the network traffic on first view.

**Fix** (~5 min): Remove `isLoaded` from deps and call `loadSchedules()` unconditionally in the effect (the store should deduplicate internally):

```typescript
useEffect(() => {
  let ignore = false
  async function init() {
    if (ignore) return
    await loadFeedToken()
    await loadSchedules()
  }
  init()
  return () => { ignore = true }
}, [loadFeedToken, loadSchedules]) // remove isLoaded
```

Or restructure with two separate effects — one for `loadFeedToken`, one for `loadSchedules`.

---

### LOW — Array Index Key in FeedPreview (FeedPreview.tsx:100)

**What**: `key={i}` uses array index as React key. The event list is sorted and can change when schedules update.

**Why it matters**: React may reuse DOM nodes incorrectly on re-sort, causing visual glitches or stale content in list items.

**Fix** (~2 min): Use a stable composite key:
```typescript
key={`${event.title}-${event.dayLabel}-${event.time}`}
```
