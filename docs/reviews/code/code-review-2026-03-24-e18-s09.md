### What Works Well

1. **Zod-validated localStorage with graceful degradation.** `quizPreferences.ts` uses `safeParse` and returns a spread copy of defaults on any failure -- corrupted data never crashes the app. The defensive pattern is exactly right for browser storage that can be tampered with.

2. **Frozen feedback preference prevents mid-quiz UX surprise.** `Quiz.tsx:110` uses `useState(() => getQuizPreferences().showImmediateFeedback)` to snapshot the preference at mount time. This is a thoughtful design choice that prevents a confusing experience if the learner changes settings in another tab while a quiz is active.

3. **Cross-tab sync via storage event.** `QuizPreferencesForm.tsx:31-42` listens for both the custom `quiz-preferences-updated` event (same tab) and the native `storage` event (other tabs), with proper cleanup in the effect return. Solid pattern.

### Findings

#### Blockers

None.

#### High Priority

- **[src/lib/quizPreferences.ts:42] (confidence: 90)**: `saveQuizPreferences` does not handle `localStorage.setItem` throwing. If localStorage is full (QuotaExceededError) or inaccessible (SecurityError in some privacy modes), the function throws an uncaught exception. The caller in `QuizPreferencesForm.update()` has no try/catch, so the state update on line 45 succeeds but the toast on line 48 fires even though nothing was actually persisted -- the learner sees "Quiz preferences saved" but their preferences are lost on reload. Why: Silent data loss with a misleading success message. Fix: Wrap the `localStorage.setItem` call in try/catch. On error, either (a) return `current` unchanged and show a `toastError.storageFull()`, or (b) throw and let the caller handle it. The rest of the codebase already has this pattern (see `quotaResilientStorage.ts`).

- **[src/app/pages/Quiz.tsx:76] (confidence: 85)**: `loadSavedAccommodation` calls `getQuizPreferences()` twice on the validation-failure path (lines 73 and 76). Each call re-reads and re-parses localStorage. More importantly, the `catch` block on line 79 falls back to `'standard'` rather than to `getQuizPreferences().timerAccommodation`, creating an inconsistency: a JSON parse error in the per-lesson key ignores the global preference entirely, while a Zod validation failure respects it. Why: A learner who set 2x accommodation globally would silently get standard timing if their per-lesson key is corrupted in a way that causes a parse exception (e.g., the string is not valid JSON at all). Fix: Change line 79's catch return to `return getQuizPreferences().timerAccommodation` for consistency with the other fallback paths. Also extract the `getQuizPreferences()` call to a local variable to avoid double-parsing:
  ```ts
  const globalDefault = getQuizPreferences().timerAccommodation
  ```

- **[Recurring] [src/app/components/settings/QuizPreferencesForm.tsx:81] (confidence: 80)**: Unsafe type assertion `value as QuizPreferences['timerAccommodation']` on the `onValueChange` callback. Radix RadioGroup's `onValueChange` emits `string`, and this cast bypasses the Zod schema entirely. If a future developer adds a radio option with a typo (e.g., `'150'` instead of `'150%'`), the invalid value silently passes through to `saveQuizPreferences` which merges it without validation. Why: `saveQuizPreferences` does not validate the patch -- it trusts the caller. Combined with the type assertion, there is no runtime validation of the value between the UI and storage. Fix: Either (a) validate the patch in `saveQuizPreferences` using the Zod schema before merging, or (b) use a type guard / parse at the callback site instead of a bare cast. Option (a) is more defensive and protects all callers.

#### Medium

- **[src/stores/useQuizStore.ts:85] (confidence: 75)**: The shuffle preference check `getQuizPreferences().shuffleQuestions || quiz.shuffleQuestions` reads localStorage synchronously inside an async function that has already awaited a Dexie query. This works, but the OR logic means the learner cannot disable shuffle for a quiz that has `quiz.shuffleQuestions: true` in its definition. The AC says "I can still override preferences per-quiz if the quiz UI allows it" -- but this is a one-way OR: user preference can only add shuffle, never remove it. Fix: Consider documenting this as intentional (quiz author forces shuffle), or change to: if the quiz definition has `shuffleQuestions: true`, always shuffle regardless of user preference (quiz author intent takes priority). Either way, document the precedence rule.

- **[src/app/components/settings/QuizPreferencesForm.tsx:97] (confidence: 72)**: The `RadioGroupItem` is `sr-only` inside a `<label>` that wraps the entire card. This works for screen readers because the `<label>` associates with the radio input. However, the `<label>` itself has no `role` attribute, and the visual selected state relies solely on border color (`border-brand`). For users with color vision deficiency, there is no secondary indicator (no checkmark, no icon change) distinguishing the selected timer option from unselected ones. The toggle switches for feedback and shuffle have built-in visual affordance via the Switch component's thumb position. Why: WCAG 1.4.1 (Use of Color) -- information conveyed solely through color. Fix: Add a visual indicator for the selected state, such as a small checkmark icon or a filled dot (similar to the theme radio cards in Settings.tsx line 504 which use `w-2 h-2 bg-brand rounded-full`).

- **[src/lib/__tests__/quizPreferences.test.ts] (confidence: 70)**: No test for `saveQuizPreferences` when localStorage throws. The unit tests verify happy-path persistence and merging, but do not mock `localStorage.setItem` to throw `QuotaExceededError`. This is the exact scenario identified in the High finding above. Fix: Add a test that mocks `localStorage.setItem` to throw and verifies the function's behavior (currently it throws -- after the fix it should handle gracefully).

#### Nits

- **Nit** [src/lib/quizPreferences.ts:16] (confidence: 60): The storage key is `'levelup-quiz-preferences'` but the app has been rebranded to "Knowlune." Other localStorage keys in the codebase also use `levelup-` prefix (e.g., `levelup-quiz-store`), so this is consistent internally, but worth noting as a future cleanup task for the full rebrand.

- **Nit** [src/app/components/settings/QuizPreferencesForm.tsx:56] (confidence: 55): The icon uses `text-brand` on `bg-brand-soft` background. Per the styling rules, `text-brand-soft-foreground` should be used instead of `text-brand` on `bg-brand-soft` backgrounds for proper contrast in dark mode. However, this is an icon (not text), so the WCAG contrast requirements are relaxed. Flagging for awareness.

### Recommendations

1. **First**: Fix the missing error handling in `saveQuizPreferences` (High #1) -- this is the only path that can mislead learners about data persistence.
2. **Second**: Make `loadSavedAccommodation`'s catch block consistent with its other fallback paths (High #2) -- quick one-line fix.
3. **Third**: Add runtime validation to `saveQuizPreferences` to protect against bad input from any caller (High #3).
4. **Fourth**: Add the selected-state visual indicator to timer radio cards (Medium #2) for color-independence.
5. **Fifth**: Add the `localStorage.setItem` failure unit test (Medium #3).
