## Code Review: E18-S03 — Ensure Semantic HTML and Proper ARIA Attributes

### What Works Well

1. **Separation of visual and semantic progressbars.** The dual-progressbar approach in `QuizHeader.tsx` (visual 0-100% Progress + sr-only 1-based question-count div) is a well-reasoned solution to the conflicting requirements between visual representation and screen reader semantics. The implementation notes thoroughly document the rationale.

2. **`aria-live="off"` on the timer with separate live-region announcements.** The design correctly avoids flooding screen readers with per-second timer updates while still providing minute-boundary announcements via a separate sr-only `aria-live="polite"` span. This is exactly how timer accessibility should work.

3. **Thorough acceptance criteria coverage in tests.** Each AC has at least one dedicated test, and the test structure is clean with well-named describe blocks. The helper functions (`navigateToQuiz`, `startQuiz`) keep test code DRY. The sidebar seed pattern is correctly applied.

### Findings

#### Blockers

- **`src/app/components/quiz/AnswerFeedback.tsx:114` (confidence: 92)**: Heading hierarchy skips from `h2` to `h4`. The active quiz has `h1` (quiz title in QuizHeader) and `h2` (sr-only "Question N of M" in Quiz.tsx). AnswerFeedback uses `<h4>` for the feedback title ("Correct!", "Not quite"), skipping `h3` entirely. This violates WCAG 1.3.1 (Info and Relationships) and directly contradicts AC2 which requires "headings follow a logical hierarchy (h1 for quiz title, h2 for question, h3 for subsections)". Screen reader users navigating by heading level will miss the `h4` content when stepping through `h3`. Why: Learners using screen readers will not discover feedback headings when navigating the heading hierarchy. Fix: Change `<h4>` to `<h3>` at line 114 of `AnswerFeedback.tsx`. This makes the hierarchy h1 (title) -> h2 (question) -> h3 (feedback subsection), matching the AC specification exactly.

- **`src/app/components/quiz/QuizHeader.tsx:39,47` (confidence: 90)**: Two `role="progressbar"` elements share identical `aria-label="Quiz progress"`. When a screen reader user navigates by role/landmark, both progressbars are announced with the same name — one reporting "50% complete" (the visual bar) and the other reporting "Question 1 of 2" (the sr-only bar). The user cannot distinguish which is which without inspecting values. This is confusing and violates the ARIA authoring practices guidance that accessible names should be unique within a context. Why: Screen reader users encountering two identically-named progressbars will be confused about which represents what. Fix: Differentiate the labels. The sr-only progressbar should use `aria-label="Question progress"` (or "Question 1 of 2") while the visual Progress keeps `aria-label="Quiz progress"`. Alternatively, since the visual Progress already has a computed label via `labelFormat` (e.g., "50% complete"), remove the explicit `aria-label="Quiz progress"` prop from the `<Progress>` at line 39 to let the default format apply.

#### High Priority

- **`src/app/components/quiz/questions/*.tsx` — empty `<legend>` (confidence: 82)**: All four question components add `<legend className="sr-only" />` — a self-closing empty legend. Per the HTML spec, an empty `<legend>` provides no accessible name to the `<fieldset>`. While the `aria-labelledby` attribute on the fieldset provides the name (and overrides `<legend>` per ARIA name computation), this pattern is fragile: if `aria-labelledby` is ever removed, the fieldset becomes completely unlabeled. More importantly, some screen readers (JAWS, older NVDA) may still announce "group" with no label when encountering an empty legend, because they check `<legend>` first before falling back to `aria-labelledby`. Why: Some assistive technology combinations may announce an unlabeled group, confusing learners. Fix: Instead of an empty self-closing `<legend />`, include the question text: `<legend className="sr-only">{question.text}</legend>`. This makes the legend a genuine fallback. The `aria-labelledby` still takes precedence for modern ATs, but older ones get a proper label. The implementation plan actually recommended this approach ("add `<legend className="sr-only">{question.text}</legend>`") but the implementation went with empty legends instead.

- **`tests/e2e/story-e18-s03.spec.ts` (confidence: 85)**: AC3 requires both `role="status"` AND `role="alert"` for dynamic content, but the test suite only verifies `role="status"` on AnswerFeedback. The `role="alert"` exists on `TimerWarnings.tsx:51` but has no corresponding test in this story's spec. Why: Without a test, the `role="alert"` on timer warnings could regress silently. The AC explicitly lists both roles. Fix: Add a test to AC3's describe block that triggers a timer warning (e.g., use a quiz with very short timeLimit or mock time to cross the 25% threshold) and assert `role="alert"` is present with `aria-live="assertive"`.

- **`src/app/components/quiz/QuizHeader.tsx:36-42` (confidence: 78)**: The `<Progress>` component is passed `aria-valuenow`, `aria-valuemin`, `aria-valuemax` as props, which override the internally-computed values via `{...props}` spread in `progress.tsx:36`. While the values happen to be identical (both use `progressValue` and 0/100), the externally-passed `aria-label="Quiz progress"` DOES override the internal `aria-label={labelFormat(normalizedValue)}` which would produce a more informative "45% complete". This makes the visual progressbar less informative for screen readers than it was before this change. Why: Before E18-S03, the visual Progress announced "45% complete" (dynamically computed). Now it announces "Quiz progress" (static string), losing the percentage context for screen reader users. Fix: Remove the explicit `aria-label`, `aria-valuenow`, `aria-valuemin`, and `aria-valuemax` props from the `<Progress>` call at lines 36-42. The Progress component already computes all of these internally and provides a better `aria-label` via `labelFormat`.

#### Medium

- **[Recurring] `src/app/components/quiz/questions/FillInBlankQuestion.tsx:119` (confidence: 80)**: String interpolation for className instead of `cn()`. Line 119 uses `` className={`mt-2 rounded-lg p-3 text-sm ${isCorrect ? '...' : '...'}`} `` instead of the project's `cn()` utility. While not introduced by this story (pre-existing), this file was touched by this PR (the legend was added at line 72). Pattern from: E01-S03 recurring. Fix: Replace with `cn('mt-2 rounded-lg p-3 text-sm', isCorrect ? 'bg-success-soft text-foreground' : 'bg-warning/10 text-foreground')`.

- **`tests/e2e/story-e18-s03.spec.ts` — AC2 heading hierarchy test (confidence: 75)**: The test at line 128 asserts `h2` is attached and contains "Question 1", but it does not verify the heading hierarchy ordering (h1 before h2, no skipped levels). A DOM presence check does not validate hierarchy correctness. The AC says "headings follow a logical hierarchy". Fix: Add an assertion that verifies ordering, e.g., check that the h1 appears before h2 in the DOM, and optionally use an axe-core rule scan for heading-order.

- **`tests/e2e/story-e18-s03.spec.ts` — No test for untimed quiz edge case (confidence: 72)**: All AC5 tests use a quiz with `timeLimit: 300`. The timer and sr-only progressbar sections assert their presence, but there is no negative test verifying that for an untimed quiz, `role="timer"` is absent. If the timer renders for untimed quizzes (due to a bug), users would see a meaningless timer element. Fix: Add a test with an untimed quiz (no `timeLimit`) that asserts `[role="timer"]` is not attached.

#### Nits

- **Nit** `src/app/pages/Quiz.tsx:435` (confidence: 65): The `totalQuestions` fallback uses `||` (logical OR): `currentProgress.questionOrder.length || currentQuiz.questions.length`. If `questionOrder` is an empty array (length 0, which is falsy), this falls back to `questions.length`. This is likely intentional, but `??` would not work the same way since 0 is not nullish. Worth a comment explaining the fallback for future readers.

- **Nit** `src/app/components/quiz/QuizActions.tsx:31,43` (confidence: 60): The `aria-label` on Previous/Next buttons overrides the visible text content. Per ARIA naming spec, `aria-label` replaces (not supplements) the button's accessible name. So "Previous" visible text is ignored by screen readers — they only hear "Previous question". This is fine for the AC, but it means the visible label and the announced label diverge, which can confuse users of voice control software who say "click Previous" but the button's accessible name is "Previous question". Consider using `aria-label` only on icon-only buttons, and removing it from buttons that already have visible text (which inherently provides an accessible name).
