**Review Date**: 2026-02-21

## What Works Well

1. **Anchor + inner-div pattern for layout stability.** The two-div approach (`video-anchor` as the outer flow placeholder, `mini-player` as the inner that becomes `position: fixed`) is architecturally sound. Outer div keeps `aspect-video` dimensions in flow at all times, eliminating a conditional spacer and avoiding layout shift. Better than the spacer approach originally described in implementation notes.

2. **`prefers-reduced-motion` check in `handleMiniPlayerClick`.** The scroll-back correctly checks the media query at call time (not at render time) and downgrades from `smooth` to `instant`.

3. **Theater button visibility gating with `hidden xl:flex`.** Clean CSS-only responsive hiding without JavaScript breakpoint detection. The conditional rendering (`onTheaterModeToggle &&`) also guards the DOM.

## Findings

### Blockers

**[LessonPlayer.tsx:264 + VideoPlayer.tsx:564] — Mini-player click bubbles into VideoPlayer `<video onClick={togglePlayPause}>`, pausing the video**

When `isMiniPlayer` is true and the user clicks the mini-player wrapper, the click bubbles through the `<video>` element (VideoPlayer.tsx line 564), which calls `togglePlayPause()` and pauses the video. This sets `isVideoPlaying = false`, making `isMiniPlayer` false and hiding the mini-player. The E2E "clicking mini-player scrolls back" test passes accidentally — it only checks `position !== fixed`, which becomes true because the **pause** hides the mini-player, not because the scroll-back worked correctly. No test verifies the video remains playing after the click.

**User impact**: Learners clicking the mini-player to scroll back will have their video unexpectedly paused every time.

**Fix**:
```tsx
onClick={isMiniPlayer ? (e: React.MouseEvent) => {
  e.stopPropagation()
  handleMiniPlayerClick()
} : undefined}
```
Also consider adding `pointer-events-none` to child elements when in mini-player mode to prevent interaction passthrough entirely.

### High Priority

**[LessonPlayer.tsx:265-270 + VideoPlayer.tsx:442-444] — T key double-toggle when focus is inside VideoPlayer container**

The wrapper div has `onKeyDown` for `'t'` (LessonPlayer line 265-269) AND VideoPlayer registers a `window`-level keydown handler for `'t'` (VideoPlayer line 442-444). When focus is on a control inside VideoPlayer's `containerRef`:
1. React's synthetic `onKeyDown` on the wrapper fires first (bubbling up from focused element), calling `setIsTheaterMode(prev => !prev)`.
2. The `window` listener fires next; since `containerRef.current.contains(document.activeElement)` is true, it calls `onTheaterModeToggle?.()` → `setIsTheaterMode(prev => !prev)` again.

React batches two functional updater calls → net zero change. Theater mode is impossible to activate via T key when focus is inside VideoPlayer controls. E2E tests pass only because they explicitly focus the wrapper div itself (outside VideoPlayer's `containerRef`).

**Fix**: Remove the `onKeyDown` handler from the wrapper div entirely. VideoPlayer's window-level T handler is sufficient.

---

**[hooks/useIntersectionObserver.ts:19 + LessonPlayer.tsx:74] — Options object recreates IntersectionObserver on every render**

Call site passes `{ threshold: 0.3 }` as an inline object literal (LessonPlayer line 74). New reference each render + `options` in the `useEffect` dependency array (hook line 19) = observer disconnected and recreated on every render. During video playback, time-update events fire at ~4Hz, so the observer is churn-cycled 4×/second.

**Fix** — either option:

(a) Memoize at call site:
```tsx
const observerOptions = useMemo(() => ({ threshold: 0.3 }), [])
const isVideoIntersecting = useIntersectionObserver(videoWrapperRef, observerOptions)
```

(b) Destructure primitives inside the hook (preferred — callers can't forget):
```tsx
const { threshold, root, rootMargin } = options ?? {}
useEffect(() => {
  // ...
}, [ref, threshold, root, rootMargin])
```

---

**[LessonPlayer.tsx:289] — `onTheaterModeToggle` inline arrow function causes VideoPlayer keyboard effect to re-subscribe every render**

`onTheaterModeToggle={() => setIsTheaterMode((prev) => !prev)}` creates a new function reference each render. Inside VideoPlayer this prop is in the keyboard effect's dependency array (line 473), causing the `window` keydown listener to be torn down and re-attached on every LessonPlayer render.

**Fix**:
```tsx
const handleTheaterModeToggle = useCallback(() => {
  setIsTheaterMode((prev) => !prev)
}, [])
```

### Medium Priority

**No unit tests for `useIntersectionObserver` hook**

New reusable hook at `src/app/hooks/useIntersectionObserver.ts` with non-trivial behavior (initial state = `true` to avoid flash, cleanup on unmount, ref-null guard). No test file exists.

**Fix**: Add Vitest tests that mock `IntersectionObserver` and verify: (a) initial state is `true`, (b) state updates on `isIntersecting: false` callback, (c) observer disconnects on unmount, (d) handles `ref.current === null` gracefully.

---

**[LessonPlayer.tsx:258] — `tabIndex={0}` always makes wrapper a focus trap even when not in mini-player mode**

The wrapper div always appears in the tab order, adding an extra tab stop that does nothing when mini-player is inactive (no role, no label, `onClick` is undefined).

**Fix**: `tabIndex={isMiniPlayer ? 0 : -1}`. Also add `aria-label="Mini player - click to scroll back to main video"` when `isMiniPlayer`.

---

**[LessonPlayer.tsx:9] — Import uses relative path instead of `@/` alias**

`from '../hooks/useIntersectionObserver'` — other imports in this file use `@/` for project modules.

**Fix**: `import { useIntersectionObserver } from '@/app/hooks/useIntersectionObserver'`
