## Code Review: E15-S05 — Display Performance Summary After Quiz

### What Works Well

1. **Excellent separation of concerns.** The analytics logic is a pure function in `src/lib/analytics.ts` with zero side effects, making it trivially testable. The component is purely presentational. This is textbook architecture.

2. **Thorough unit test coverage.** The 14 unit tests for `analyzeTopicPerformance` cover boundary conditions (70% threshold exact hit), empty inputs, max growth area limits, and sort order verification. The boundary test at line 227-240 of `analytics.test.ts` is especially well-designed.

3. **Proper accessibility patterns.** `useId()` for `aria-labelledby`, `aria-hidden="true"` on decorative icons, semantic `<section>`, `<ul>`, `<li>`, and `<h3>` elements. Color is never the sole indicator of meaning (icons + text + percentages).

### Findings

#### Blockers

None.

#### High Priority

- **[src/app/components/quiz/PerformanceInsights.tsx:28] (confidence: 82)**: "Incorrect" count uses `text-warning` (amber) instead of `text-destructive` (red). The design guidance at story line 110 specifies the correctness summary uses `text-muted-foreground`, while `text-warning` is reserved for Growth Opportunities elements (lines 105-107). Using `text-warning` for "incorrect" creates semantic confusion: amber communicates "attention needed" while red communicates "wrong answer." Across the quiz subsystem, `text-destructive` is the established convention for incorrect/failure states (see `ScoreSummary.tsx:46-47` where `<50%` score uses `text-destructive`). Why: Learners receive mixed visual signals about what "incorrect" means vs. what "growth opportunity" means when both share the same color. Fix: Change `text-warning` to `text-destructive` for the incorrect count span:
  ```tsx
  <span className="text-destructive font-medium">{incorrectCount} incorrect</span>
  ```

- **[src/lib/analytics.ts:64-66] (confidence: 78)**: Skipped questions are pushed to `incorrectQuestionNumbers`, which means "Review questions 3, 4" in growth areas conflates unanswered questions with wrong answers. A learner who skipped Q3 but got Q4 wrong sees both listed identically under "Review questions." The AC (line 52) says "Review questions 3, 7, 11 on Functions" — implying questions the learner *answered incorrectly*, not ones they never attempted. Why: Telling a learner to "review" a question they intentionally skipped is misleading — they may know the material but ran out of time. Fix: Track skipped and incorrect question numbers separately, or rename the suggestion to "Review questions 3, 4 (includes skipped)" to set expectations. Minimally, add a comment documenting the intentional conflation:
  ```ts
  // Skipped questions are treated as incorrect for topic analysis purposes
  entry.incorrectQuestionNumbers.push(question.order)
  ```

- **[src/lib/__tests__/analytics.test.ts, src/app/components/quiz/__tests__/PerformanceInsights.test.tsx] (confidence: 75)**: Helper functions `q()`, `correct()`, `wrong()` are duplicated between the two test files (analytics.test.ts:9-33, PerformanceInsights.test.tsx:10-30). Both define identical `q()` and `correct()`/`wrong()` factory functions. Why: When the `Question` type changes (e.g., Epic 13 adds required fields), both files break independently and must be updated separately. Fix: Extract shared helpers into `tests/support/fixtures/factories/question-helpers.ts` or co-locate with the existing `quiz-factory.ts`. The E2E test already uses the centralized `makeQuestion()` factory — the unit tests should follow suit.

#### Medium

- **[src/app/components/quiz/PerformanceInsights.tsx:39] (confidence: 72)**: The responsive grid `sm:grid sm:grid-cols-2 sm:gap-4 space-y-4 sm:space-y-0` creates a 2-column layout even when only one section (strengths or growth) is present. When a learner aces all topics (100% across the board), only the strengths section renders. It occupies the left column of a 2-column grid with the right column empty. Why: Visual imbalance for high-performing learners. Fix: Conditionally apply grid only when both sections are present:
  ```tsx
  <div className={cn(
    'space-y-4',
    strengths.length > 0 && growthAreas.length > 0 && 'sm:grid sm:grid-cols-2 sm:gap-4 sm:space-y-0'
  )}>
  ```
  Note: This would require importing `cn()` from `@/app/components/ui/utils`.

- **[src/lib/analytics.ts:85-86] (confidence: 70)**: `hasMultipleTopics` is derived from `uniqueTopics.size > 1`, but `uniqueTopics` is constructed by creating a `new Set(topicMap.keys())` — which is redundant since `topicMap` is already a `Map` with unique keys. `topicMap.size > 1` achieves the same result without the intermediate `Set` allocation. Fix:
  ```ts
  const hasMultipleTopics = topicMap.size > 1
  ```

- **[tests/e2e/story-15-5.spec.ts:248-249] (confidence: 72)**: The test asserts `not.toBeVisible()` for "Your Strengths" and "Growth Opportunities" using `getByText` with case-insensitive regex. This will fail if the text appears anywhere on the page (e.g., in the encouraging message). The story notes (line 168) mention this exact ambiguity was discovered during development for `getByText(/growth/i)`. While the current encouraging messages don't contain "your strengths" literally, this is fragile. Fix: Use `getByRole('heading', { name: /your strengths/i })` consistently, matching the pattern used in `story-15-5.spec.ts:215` and the lesson learned in the story file.

#### Nits

- **Nit** [src/app/components/quiz/PerformanceInsights.tsx:1] (confidence: 65): `useMemo` dependency is `[questions, answers]` which compares by reference. If the parent re-renders with new array references containing the same data (e.g., Zustand selector returning new arrays), `analyzeTopicPerformance` recalculates unnecessarily. For a pure function operating on small datasets (quiz-sized), this is negligible — but worth noting if performance becomes a concern.

- **Nit** [tests/e2e/story-15-5.spec.ts] (confidence: 60): Test file is 352 lines (target 300). The significant duplication in quiz-taking steps (answer 5 questions + submit appears 6 times) could be extracted into a helper like `async function completeQuizWithAnswers(page, answers: string[])` to reduce line count by ~80 lines.

- **Nit** [src/lib/analytics.ts:81] (confidence: 55): `.sort((a, b) => a - b)` mutates the `incorrectQuestionNumbers` array in-place. Since the array was just constructed on line 59, this is safe. But using `.toSorted()` would be more idiomatic for immutable patterns and signals intent more clearly (available in all modern browsers and Node 20+).

### Recommendations

1. **(High)** Fix the `text-warning` to `text-destructive` color token on the incorrect count — quick 1-line fix, improves semantic consistency across the quiz subsystem.
2. **(High)** Add a code comment on line 66 of `analytics.ts` documenting the intentional treatment of skipped questions as "incorrect" for topic analysis, or separate them. This prevents future developers from "fixing" what appears to be a bug.
3. **(High)** Extract duplicated `q()`, `correct()`, `wrong()` test helpers to a shared location. The E2E tests already use centralized factories.
4. **(Medium)** Fix the 2-column grid when only one section renders — conditional `cn()` application.
5. **(Medium)** Simplify `new Set(topicMap.keys())` to `topicMap.size > 1`.
6. **(Medium)** Use `getByRole('heading', ...)` in the AC3 `not.toBeVisible()` assertions for robustness.
