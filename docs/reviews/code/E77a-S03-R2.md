## Findings

### HIGH

#### 1. Progress never reaches visible 100% -- closure staleness in finally block

**File**: `src/app/components/settings/DataAndBackupPanel.tsx` (lines 117-118 conditional, 120-121 reset)

**What**: The `finally` block checks `uploadPhase !== 'Complete!'` to decide whether to reset progress. But `uploadPhase` at that point is the stale closure value from the last render ('Uploading to Google Drive...'), not the value queued by `setUploadPhase('Complete!')` in the success branch. The condition evaluates to `true`, so progress/phase get reset to 0/'' via the React 19 automatic batched render. The user never sees the progress bar reach 100% before it disappears.

**Why**: AC2 explicitly requires "progress reaches 100%". The current implementation makes progress jump from ~70% to not rendered, skipping the 100% milestone entirely. For an upload flow that takes seconds, this is a noticeable visual gap -- the user can't confirm the export+upload completed.

**Fix**: Use a local variable tracked in the function scope instead of reading React state from the closure:

```typescript
let succeeded = false

try {
  // ... upload logic ...
  setUploadProgress(100)
  setUploadPhase('Complete!')
  succeeded = true
  toast.success(...)
} catch (error) {
  // ... error handling ...
} finally {
  setIsUploading(false)
  if (!succeeded) {
    setUploadProgress(0)
    setUploadPhase('')
  }
}
```

**Effort**: ~2 minutes

**Confidence**: 95 -- certain. The closure captures `uploadPhase` at render time, and React 19's automatic batching means the finally block's `setUploadProgress(0)` always overrides the success path's `setUploadProgress(100)`.

#### 2. Error toasts do not match Acceptance Criteria messages

**File**: `src/app/components/settings/DataAndBackupPanel.tsx` (lines 77-92)

**What**: Every error path in `handleSendToDrive` routes through `toastError.saveFailed()`, which always displays "Save failed. Check your connection and try again." -- regardless of the actual error type. The ACs demand distinct messages:

| AC | Required toast | Actual toast |
|----|---------------|-------------|
| AC3 | "Export failed" | "Save failed..." (via `toastError.saveFailed('Export failed. Try again?')`) |
| AC4 | Quota toast (e.g. "Your Google Drive is full") | "Save failed..." (via `toastError.saveFailed(error.message)`) |
| AC7 | "Upload failed. Try again?" | "Save failed..." (via `toastError.saveFailed(error.message)`) |
| AC9 | Generic toast + console.error | ✅ "Save failed..." matches generic |

The `details` parameter to `saveFailed()` is only logged to `console.error`, never shown to the user -- so the distinct error context is invisible to users.

**Why**: Three ACs (3, 4, 7) specify distinct error messages per failure mode, but the implementation uses a single generic helper that hides error specificity. This makes debugging harder for users -- they see the same text for export failures, quota issues, and network errors, so they can't determine the right action to take.

**Fix**: Replace `toastError.saveFailed(...)` calls with direct `toast.error()` calls using the AC-specified messages. For quota errors, use `toast.error('Your Google Drive is full. Free up space and try again.', ...)`. For network errors, use `toast.error('Upload failed. Try again?', ...)`. For export-phase errors, use `toast.error('Export failed. Try again?', ...)`. Keep `console.error` for debugging in all cases.

**Effort**: ~10 minutes

**Confidence**: 90 -- concrete AC text vs implementation mismatch, traceable through the `toastError.saveFailed` definition.

### MEDIUM

#### 3. Cached `knownTokenState` skips token freshness check on subsequent clicks

**File**: `src/app/components/settings/DataAndBackupPanel.tsx` (lines 26-37)

**What**: The component caches token presence in `knownTokenState` after the first click. If `knownTokenState` is `'present'`, subsequent clicks skip the actual token check entirely and proceed straight to the upload:

```typescript
let tokenAvailable: boolean
if (knownTokenState === 'untested') {
  const token = await getDriveToken()
  tokenAvailable = token !== null
  setKnownTokenState(tokenAvailable ? 'present' : 'absent')
} else {
  tokenAvailable = knownTokenState === 'present'  // skips fresh token check
}
```

If the token expires or is revoked between clicks (e.g., user revokes app in Google Security settings, or token naturally expires), the component will attempt the full export+upload flow before getting a 401/403 error, and only then show the reconnect dialog.

**Why**: This causes an unnecessary failed upload attempt (with data serialized, memory allocated, upload attempted) before the user is prompted to reconnect. On slow connections or large datasets, this wasted round-trip can take 10-30 seconds before surfacing the real problem. The progress indicator briefly shows activity, then errors out.

**Fix**: The simplest approach is to always call `getDriveToken()` on each click and update `knownTokenState` accordingly. Since the token check is a lightweight Supabase session read (no network request), there is no performance concern. Alternatively, set `knownTokenState` to `'untested'` in the `finally` block so the next click always re-checks.

**Effort**: ~5 minutes

**Confidence**: 80 -- the behavior is suboptimal but the error recovery path eventually handles it. Not a correctness bug, but a UX polish issue that wastes user time on failed uploads.

## Recommendations

1. **Fix the progress reset bug first** (Finding 1) -- this visibly breaks AC2 and is a 2-line fix (local `succeeded` flag).
2. **Fix the toast messages** (Finding 2) -- three ACs are directly unmet. Replace `toastError.saveFailed()` with explicit `toast.error()` calls per error type.
3. **Fix the token caching** (Finding 3) -- add `setKnownTokenState('untested')` in the `finally` block to ensure every click does a fresh token check.
