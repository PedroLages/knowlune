# Code Review — E89-S04 Build Unified CourseDetail Page (2026-03-29)

## Summary

Reviewed 1,186 lines changed across 6 source files. The unified CourseDetail page introduces 4 new components (CourseHeader, LessonList, CourseProgress, AISummaryPanel) with adapter-driven architecture. Overall solid implementation with good error handling, accessibility, and design token compliance.

## Findings

### MEDIUM

**1. [CourseHeader.tsx:57] Adapter bypass — direct `course.source` check**

CourseHeader's JSDoc (line 6) states "Never checks `course.source` directly — uses adapter capabilities" but line 57 does exactly that: `const isYouTube = course.source === 'youtube'`. This violates the adapter pattern that UnifiedCourseDetail correctly follows via `adapter?.getSource()`.

**Fix:** Either pass `isYouTube` as a prop from the parent (which already computes it via adapter), or use a capability from `ContentCapabilities` to drive source-specific rendering.

**2. [UnifiedCourseDetail.tsx:138] Stale closure in thumbnail cleanup**

The `useEffect` cleanup at line 136-139 captures `thumbnailUrl` from the render cycle when the effect was created. If `adapter` changes and triggers a new effect, the cleanup revokes the stale (possibly null) URL instead of the current blob URL, causing a memory leak.

**Fix:** Use a ref to track the current thumbnail URL for cleanup:
```tsx
const thumbnailRef = useRef<string | null>(null)
// In effect: thumbnailRef.current = url
// In cleanup: if (thumbnailRef.current?.startsWith('blob:')) revokeObjectUrl(thumbnailRef.current)
```

### LOW

**3. [LessonList.tsx:420] React Fragment without key in mapped output**

When `!hasMultipleGroups`, `renderLocalGroups` returns `<>{videoItems}</>` from inside `groups.map()`. The Fragment lacks a key. Currently safe because there's typically only one group, but fragile if the grouping logic changes.

**4. [CourseHeader.tsx:30] Unused `capabilities` prop**

`ContentCapabilities` is accepted as a prop but immediately destructured to `_capabilities` and never used. The comment says "reserved for future capability-driven rendering" which is fine for now, but could be removed until actually needed.

**5. [LessonList.tsx:459] Array index used as React key**

`renderYouTubeGroups` uses `key={`group-${groupIndex}`}` which is an index-based key. Acceptable since groups aren't reordered, but could be improved with stable keys.

### Positive Observations

- Excellent error handling: loading skeleton, error state, load error with retry
- Proper cancel pattern with `let ignore = false` in effects
- All icons have `aria-hidden="true"`, interactive elements have `aria-label`
- No hardcoded colors — design tokens used consistently
- `silent-catch-ok` comments document intentional empty catches
- Search highlight escapes regex special characters (XSS-safe)
- Lazy loading on video thumbnails (`loading="lazy"`)

## Verdict

No blockers. 2 MEDIUM findings (adapter bypass, stale closure) should be addressed but don't block shipping.
