#### Blockers
- **[src/app/components/library/LocalSeriesView.tsx:39](confidence: 92)**: **Stale closure / broken memoization** — `getBooksBySeries` is omitted from the `useMemo` dependency array with a comment claiming it's a "stable store selector reference," but in Zustand, `useBookStore(s => s.getBooksBySeries)` returns the same *closure* identity only until the store re-creates it (e.g., during HMR or if the store is ever restructured). More critically, `filteredBookIds` is included as the sole dependency, but it's derived from the *same* filtered books that `getBooksBySeries()` internally re-computes via `get().getFilteredBooks()`. This means `filteredBookIds` and the internal computation can drift — if filters change in a way that doesn't alter `filteredBookIds` but does alter sort/grouping behavior, the memo will return stale results. Fix: Add `getBooksBySeries` to the dependency array. If referential instability is a concern, wrap `getBooksBySeries` in `useCallback` or use `useBookStore.getState().getBooksBySeries()` directly inside the memo (reading from `getState()` avoids the closure identity issue entirely).

#### High Priority
- **[src/app/components/library/BookMetadataEditor.tsx:295](confidence: 88)**: **`seriesSequence` persisted as string allows non-numeric junk** — `seriesSequence.trim() || undefined` will happily save any arbitrary string (e.g., "abc", "!@#") to the database. While the sort function handles NaN gracefully, the field is intended to represent a position within a series. The `Input` has no `type="number"` or validation, and the placeholder says "e.g. 1, 2.5" suggesting numeric input. Invalid data will silently persist and could cause confusing UX. Fix: Either validate that the value is parseable as a number before saving (with a user-facing error), or set `type="number"` on the Input (and handle the empty-string case). At minimum, add `inputMode="decimal"` to the input element.
- **[src/app/hooks/useAudiobookshelfSync.ts:116-117](confidence: 85)**: **`series` field maps from `absItem.media.metadata.series` but ABS API returns `series` as an array of `{name, sequence}` objects, not a string** — In the ABS API, `media.metadata.series` is typically an array like `[{name: "Harry Potter", sequence: "1"}]`. This code treats it as a plain string. If the ABS client types define `series` as a string already (pre-processed), this is fine, but if it's the raw ABS response shape, this will serialize `[object Object]` into the DB. Fix: Verify the ABS client types — if `series` is `Array<{name: string, sequence: string}>`, extract the first entry's `name` and `sequence` fields explicitly.
- **[tests/support/fixtures/library-page-fixture.ts:31](confidence: 90)**: **Breaking existing E2E tests** — `switchToListView` now clicks `[data-testid="local-view-list"]`, but that button only renders when `filters.source !== 'audiobookshelf'` (per the Library.tsx conditional). Any existing E2E test using this fixture helper while in the ABS source context will fail with a locator-not-found error. The old `[data-testid="view-toggle-list"]` is now only rendered for ABS source. Fix: Make `switchToListView` context-aware — check which toggle is visible, or restore the ABS toggle's data-testid as a separate element, or conditionally select the correct testid based on the current source filter.

#### Medium
- **[src/app/components/library/LocalSeriesCard.tsx:133](confidence: 75)**: **`book.progress` may be `undefined`** — The destructured `const progress = book.progress ?? 0` handles the nullish case, but the `isFinished` check at line 131 uses `book.progress >= 100` before the null coalescing. If `book.progress` is `undefined`, `undefined >= 100` evaluates to `false` (safe), but `book.progress` of `0` with `status === 'finished'` creates a subtle inconsistency where `isFinished` is `true` but the progress bar shows `0%`. The `isFinished` logic is defensively correct but the visual display will be misleading. Fix: Use `progress` (the coalesced value) in the `isFinished` check, or derive `isFinished` solely from `book.status`.
- **[src/app/components/library/LocalSeriesCard.tsx:39](confidence: 70)**: **`group.name` used as React key** — `group.name` is the human-entered series name. If two users create series with identical names (or a book's series field has trailing whitespace variation like `"Dune "` vs `"Dune"`), the keys will collide and React will merge/wrongly reconcile the cards. The `seriesMap` grouping already trims nothing — `"Dune"` and `"Dune "` would be separate groups but would produce the same test slug. Fix: Consider normalizing series names (trim + collapse whitespace) during grouping, or use a more unique key.
