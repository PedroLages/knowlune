## Code Review: E14-S03 — Display Fill-in-Blank Questions

### What Works Well

1. **Clean component structure.** `FillInBlankQuestion` follows the established TrueFalseQuestion pattern closely — same fieldset/legend semantics, same markdown rendering, same import structure. New developers can read one question component and understand them all.

2. **Solid scoring reuse.** Rather than writing new scoring logic, the implementation correctly relies on the existing `fill-in-blank` case in `src/lib/scoring.ts` (lines 20-26). Zero duplication.

3. **Good E2E coverage with deterministic patterns.** All 7 tests use `input.blur()` to trigger immediate saves instead of `waitForTimeout` — this is the right approach. The tests cover all four acceptance criteria including case-insensitive scoring and answer persistence through navigation.

### Findings

#### Blockers

None.

#### High Priority

- **[`FillInBlankQuestion.tsx:31-47`] (confidence: 85)**: **Debounce effect fires unnecessarily on external value sync.** The sync effect (line 31) updates `inputValue` whenever the `value` prop changes (e.g., navigating back to this question). This triggers the debounce effect (line 36) which calls `onChange(inputValue)` after 300ms — writing the same value back to the store that just provided it. The `isInitialMount` guard only protects the very first mount; subsequent prop-driven value changes still trigger the write-back. Why: Learners navigating between questions trigger redundant store writes on every visit. In quizzes with many questions, this creates unnecessary re-renders up the component tree. Fix: Guard the debounce effect against prop-driven changes by tracking whether the change was user-initiated:
  ```tsx
  const userEdited = useRef(false)

  // Sync external value
  useEffect(() => {
    userEdited.current = false
    setInputValue(value ?? '')
  }, [value])

  // Debounced save — only for user edits
  useEffect(() => {
    if (!userEdited.current) return
    const timer = setTimeout(() => onChange(inputValue), DEBOUNCE_MS)
    return () => clearTimeout(timer)
  }, [inputValue, onChange])

  // In the Input onChange handler:
  onChange={e => { userEdited.current = true; setInputValue(e.target.value) }}
  ```

- **[`FillInBlankQuestion.tsx:49-52`] (confidence: 80)**: **`handleBlur` and debounce timer double-fire `onChange`.** When a learner types then tabs/clicks away, `handleBlur` calls `onChange(inputValue)` immediately, and the pending debounce timer fires the same call ~300ms later. This is two identical writes to the store per blur event. Why: Redundant store writes cause unnecessary re-renders. More importantly, if `onChange` has side effects (e.g., Dexie persistence in future epics), this could cause duplicate writes. Fix: Clear the debounce timer on blur. Store the timer ref:
  ```tsx
  const timerRef = useRef<ReturnType<typeof setTimeout>>()

  useEffect(() => {
    // ... existing guard ...
    timerRef.current = setTimeout(() => onChange(inputValue), DEBOUNCE_MS)
    return () => clearTimeout(timerRef.current)
  }, [inputValue, onChange])

  function handleBlur() {
    clearTimeout(timerRef.current)
    onChange(inputValue)
  }
  ```

#### Medium

- **[`QuestionDisplay.test.tsx`] (confidence: 75)**: **No unit test for fill-in-blank dispatch in QuestionDisplay.** The file has dispatch tests for `multiple-choice`, `true-false`, and `multiple-select`, but no corresponding test for `fill-in-blank`. The E2E tests cover the full flow, but the unit test suite is now asymmetric. Fix: Add a test case:
  ```tsx
  it('dispatches fill-in-blank questions to FillInBlankQuestion', () => {
    render(
      <QuestionDisplay
        question={makeQuestion({ type: 'fill-in-blank', correctAnswer: 'React' })}
        value={undefined}
        onChange={vi.fn()}
        mode="active"
      />
    )
    expect(screen.getByRole('textbox')).toBeInTheDocument()
    expect(screen.getByPlaceholderText(/type your answer/i)).toBeInTheDocument()
  })
  ```

- **[`FillInBlankQuestion.tsx:66`] (confidence: 72)**: **Disabled state uses `opacity-60` on wrapper AND `disabled` on Input.** The `opacity-60` on the wrapper div compounds with whatever opacity the disabled Input already applies (shadcn/ui Input applies `disabled:opacity-50`). Combined, the input text will appear at roughly 30% opacity, which may fail WCAG contrast requirements for disabled controls. Fix: Remove `opacity-60` from the wrapper div, or apply it only to the character counter, not the input:
  ```tsx
  <div className="space-y-2">
    <Input ... disabled={!isActive} className={cn('w-full max-w-md', !isActive && 'opacity-60')} />
    <span className={cn('text-sm text-muted-foreground block', !isActive && 'opacity-60')}>
  ```

- **[`FillInBlankQuestion.tsx:21`] (confidence: 70)**: **`legendId` generated but only used on `id` attribute — no `htmlFor` association.** The `inputId` is generated (line 22) and set on the Input (line 68), but there is no `<label htmlFor={inputId}>`. The input relies on `aria-labelledby={legendId}` which is correct for accessibility, but the `inputId` is unused beyond the `id` attribute itself. Fix: Either remove `inputId` (dead code), or add a visible `<label>` for the input field to improve click target area.

#### Nits

- **Nit** [`FillInBlankQuestion.tsx:4`] (confidence: 65): `cn` is imported but only used once (line 66) for a simple conditional. Not wrong, but the other question components (TrueFalseQuestion) use `cn` more heavily. If the component stays this simple, the import is marginal overhead.

- **Nit** [`QuizNavigation.tsx`] (confidence: 60): The formatting change (lines 42-44 collapsing a ternary to one line) is unrelated to this story. Not harmful, but adds noise to the diff. Consider keeping non-functional reformats in separate commits.

### Recommendations

1. **First**: Fix the debounce guard to distinguish user edits from prop syncs (High #1). This prevents redundant store writes on every question navigation.
2. **Second**: Clear the debounce timer on blur (High #2). Simple one-line fix that prevents the double-fire.
3. **Third**: Add the missing unit test for fill-in-blank dispatch (Medium #1). Keeps the test suite symmetric.
4. **Fourth**: Review the compounding opacity on disabled state (Medium #2). Quick visual check in review mode.
