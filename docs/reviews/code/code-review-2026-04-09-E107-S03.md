## Code Review: E107-S03 -- Fix TOC Loading and Fallback

### What Works Well

- **Clean timeout pattern with proper cleanup.** The `useEffect` at `BookReader.tsx:131-139` creates a timeout that auto-clears on unmount or when `isTocLoading` changes, with the correct `[isTocLoading]` dependency array. No stale closure risk.

- **Well-structured test-mode infrastructure.** The mock EPUB generator (`minimalEpub.ts`) creates a proper EPUB 3.0 ZIP with all required files (mimetype, container.xml, nav.xhtml, toc.ncx, content.opf, chapters). The test-mode flag is gated behind both a window property check and an explicit `enableTestMode()` call, preventing accidental activation.

- **Comprehensive unit tests.** Both `ReaderHeader.test.tsx` and `TableOfContents.test.tsx` cover the key state transitions (loading, empty, populated, fallback), boundary values (0%, 100%, rounding), and prop precedence (chapter name preferred over progress percentage). Tests use proper mock isolation and shared patterns.

### Findings

#### High Priority

- **[src/main.tsx:8-16] (confidence: 85)** `[AI Smell]` Test-mode code runs unconditionally in production. The `if (typeof window !== 'undefined')` block dynamically imports `BookContentService` and exposes `enableTestMode` on every page load, even in production. While the module-level `TEST_MODE` flag defaults to `false` and won't activate without explicit calls, two concerns remain: (a) the dynamic import of `BookContentService` and its transitive dependency `./__tests__/minimalEpub` (which imports JSZip) adds unnecessary async work on every cold start; (b) the `__enableBookContentTestMode__` function is exposed on `window` in production, meaning any third-party script could activate mock EPUB loading and prevent real book content from displaying.

  Why: A malicious or buggy browser extension could call `window.__enableBookContentTestMode__()`, causing all EPUB content to be replaced with the test mock. The JSZip import also adds unnecessary async module loading overhead.

  Fix: Gate the entire block behind a `import.meta.env.DEV` or `import.meta.env.MODE === 'test'` check. This prevents both the security surface and the unnecessary production import:
  ```typescript
  if (typeof window !== 'undefined' && import.meta.env.DEV) {
    import('./services/BookContentService').then(({ enableTestMode }) => {
      (window as any).__enableBookContentTestMode__ = enableTestMode
      if ((window as any).__BOOK_CONTENT_TEST_MODE__) enableTestMode()
    })
  }
  ```
  For E2E tests, `page.addInitScript` can still set the `__BOOK_CONTENT_TEST_MODE__` flag, and `BookContentService.isTestMode()` already checks this window flag directly -- the `main.tsx` bridge is redundant for E2E since tests also call `page.evaluate()` to invoke the function. The `__BOOK_CONTENT_TEST_MODE__` window flag check in `isTestMode()` is the real safety net; the `enableTestMode()` export in `main.tsx` is belt-and-suspenders.

- **[src/services/BookContentService.ts:17] (confidence: 82)** `[AI Smell]` Production service imports from `__tests__` directory. `BookContentService.ts` imports `createMinimalEpub` from `./__tests__/minimalEpub`, which pulls in JSZip (a 100KB+ library) into the production bundle even though it is only used in test mode. Moving `jszip` to `devDependencies` (already done in this branch) does NOT remove it from the production bundle -- Vite still bundles imports from source files regardless of `package.json` dependency type. The import is unconditional at the top of the file.

  Why: This adds JSZip to every user's production bundle even though test-mode EPUB generation is only for E2E tests. Bundle size impact is significant.

  Fix: Use a dynamic import inside the `isTestMode()` branch instead:
  ```typescript
  if (this.isTestMode()) {
    const { createMinimalEpub } = await import('./__tests__/minimalEpub')
    return createMinimalEpub()
  }
  ```
  This ensures JSZip is only loaded when test mode is actually active (never in production). The `getEpubContent` method is already async so the dynamic import has no API impact.

#### Medium

- **[tests/e2e/story-107-03.spec.ts:145-153] (confidence: 80)** `[Testing]` AC-1 test declares `loadingIndicator` locator but never asserts on it. The variable is constructed on lines 145-148 but unused -- the only assertion is that `toc-panel` is visible (line 152). This means AC-1 (loading state displayed) is effectively untested at the E2E level. The test passes regardless of whether a loading indicator ever appears. The comment on line 150 acknowledges this ("loading state may resolve very quickly") but the test should still assert the loading state existed at some point, or at minimum use `expect().toHaveCount()` with `maxRetries`.

  Fix: Either add `await expect(loadingIndicator).toBeVisible({ timeout: 1000 }).catch(() => {})` to attempt verification, or restructure to check loading state before the panel fully loads by opening the panel earlier in the render cycle. Alternatively, accept this as a known limitation and add a comment with `// Intentional: loading state races with TOC resolution in test mode, verified by unit tests instead`.

- **[tests/e2e/story-107-03.spec.ts:98,102,182,281,286] (confidence: 75)** `[Testing]` Multiple `waitForTimeout()` calls without justification comments. `openBookReader` uses `waitForTimeout(500)` and `waitForTimeout(200)` for dialog dismissal. AC-3 uses `waitForTimeout(6000)` for the timeout test. Edge case tests use `waitForTimeout(100)` for rapid open/close. Per the test-patterns ESLint rule, hard waits should have `// Intentional hard wait: [reason]` comments.

  Fix: Add justification comments. For the 6000ms wait specifically, this is actually testing the timeout behavior so it's justified -- but the shorter waits in `openBookReader` and edge cases could potentially be replaced with `expect().toBeVisible()` or `expect().not.toBeVisible()` assertions.

#### Nits

- **Nit** [src/services/BookContentService.ts:56-63] (confidence: 70): `BookContentService.setTestMode()` static method is defined but never called anywhere. The module-level `enableTestMode()` function and the `isTestMode()` instance method handle all test-mode activation paths. Dead code.

- **Nit** [tests/e2e/story-107-03.spec.ts:171-173] (confidence: 65): `__TEST_TOC_TIMEOUT__` window flag is set in `addInitScript` but never read by any application code. It appears to be a leftover from a planned but unimplemented mock for simulating TOC timeout.

### Recommendations

1. **Gate `main.tsx` test-mode code behind `import.meta.env.DEV`** and convert the `createMinimalEpub` import to dynamic. This is the highest-impact fix -- removes JSZip from production bundles and eliminates the window attack surface. Effort: ~15 minutes.

2. **Fix AC-1 E2E test** to either assert on the loading indicator or document why it can't be reliably tested. The unit tests cover this well, so the E2E gap is low-risk. Effort: ~10 minutes.

3. **Add justification comments** to `waitForTimeout` calls per the test-patterns ESLint rule. Effort: ~5 minutes.
