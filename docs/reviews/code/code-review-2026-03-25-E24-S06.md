## Findings

### MEDIUM

**M1 — Missing `role="listitem"` on sortable rows** — `VideoReorderList.tsx:192`
The container div uses `role="list"` but `SortableVideoRow` renders a plain `<div>` without `role="listitem"`. Screen readers expect matching list semantics. Add `role="listitem"` to the outer div in `SortableVideoRow`.

**M2 — Test mock returns wrong value for success path** — `EditCourseDialog.test.tsx:7`
`mockUpdateCourseDetails` is mocked as `.mockResolvedValue(undefined)`. The component checks `if (!success)` (line 111 of EditCourseDialog.tsx), so `undefined` is falsy and triggers `toast.error`. The "calls updateCourseDetails on Save" test only verifies the function was called, not that the success path (toast.success, dialog close) works. Should be `.mockResolvedValue(true)` with assertions on dialog close.

### LOW

**L1 — No abort/cleanup for video loading promise** — `EditCourseDialog.tsx:54-62`
The `useEffect` that loads videos via `db.importedVideos.where(...).sortBy('order').then(setVideos)` has no cleanup. If the dialog closes and reopens quickly with a different course, a stale promise could call `setVideos` with videos from the previous course. Consider an `isCancelled` flag pattern:
```ts
let cancelled = false
db.importedVideos.where(...).then(v => { if (!cancelled) setVideos(v) })
return () => { cancelled = true }
```

**L2 — Non-deterministic IDs in test factory** — `VideoReorderList.test.tsx:22`
`makeVideo()` uses `Math.random()` for default IDs. While all tests override the `id` field explicitly, this violates the deterministic test data convention. Use a counter or fixed default instead.

**L3 — Inline style warning (acceptable)** — `VideoReorderList.tsx:61`
ESLint warns about inline `style` prop for dnd-kit transform. This is explicitly excepted by the rule ("dynamic transforms"). No action needed.
