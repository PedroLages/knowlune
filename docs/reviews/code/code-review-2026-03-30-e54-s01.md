## Findings

### MEDIUM — Duplicate `getLessons()` calls (story-related)

**File:** `src/app/pages/UnifiedLessonPlayer.tsx:182`

The `showCelebration` callback calls `adapter.getLessons()` each time a video ends or a manual toggle fires. This is *in addition* to the call in `useLessonNavigation` and the lesson metadata `useEffect`. That's 3 calls to `getLessons()` per lesson load, and an additional call on every video end event.

The code already has a comment acknowledging this (line 120): "NOTE: getLessons() is also called inside useLessonNavigation hook — known duplication." The story adds a third call path.

**Impact:** Minor performance — `getLessons()` likely hits IndexedDB. Not a blocker since it's read-only and fast, but worth consolidating in a future story.

**Recommendation:** Consider using the `lessons` array already returned by `useLessonNavigation` hook (it exposes `lessons` in its return type) instead of re-fetching inside `showCelebration`.

### LOW — Race condition in `handleVideoEnded` if `setItemStatus` fails (story-related)

**File:** `src/app/pages/UnifiedLessonPlayer.tsx:197-214`

When `setItemStatus` throws, the error is caught and toasted, but execution continues to `showCelebration()` and `setShowAutoAdvance(true)`. This means the user sees a celebration modal even though the lesson was NOT marked complete. The celebration is misleading in this edge case.

**Recommendation:** Return early after the catch to avoid showing celebration on failed persistence:
```typescript
try {
  await setItemStatus(courseId, lessonId, 'completed', [])
} catch {
  toast.error('Failed to mark lesson as complete')
  return // Don't celebrate if persistence failed
}
```

### LOW — `checkCourseCompletion` assumes current lesson is complete (story-related)

**File:** `src/app/pages/UnifiedLessonPlayer.tsx:160-161`

The function returns `true` for the current lesson (`if (l.id === lessonId) return true`) assuming it was just marked complete. But this function is also called from `showCelebration` which is called from `handleManualStatusChange`. If the manual toggle marks a lesson as "not-started" and then back to "completed", the `checkCourseCompletion` check happens before the store updates propagate. In practice this is fine because `handleManualStatusChange` only calls `showCelebration` when `status === 'completed'`, so the assumption holds.

No fix needed — documenting for completeness.
