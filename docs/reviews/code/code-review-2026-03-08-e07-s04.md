## What Works Well

1. **Clean separation of concerns.** Business logic lives in dedicated `src/lib/atRisk.ts` and `src/lib/completionEstimate.ts` modules as pure functions, following the established `momentum.ts` pattern. Props flow down from `Courses.tsx` through `CourseCard` -- no business logic leaks into UI components.

2. **Well-structured batch calculation.** The `loadCourseMetrics` effect groups all sessions once, then iterates courses a single time to compute momentum, at-risk, and estimates together. This avoids N+1 IndexedDB queries and keeps the async boundary clean with proper `ignore` flag cleanup.

3. **Accessible badge design.** `AtRiskBadge` includes `aria-label`, `aria-hidden` on decorative icons, a tooltip for context, and a `data-testid` for E2E targeting. This mirrors the `MomentumBadge` pattern exactly -- good consistency.

### High Priority

**H1. [Recurring] `/src/app/components/figma/CompletionEstimate.tsx:29`] (confidence: 92) -- Hardcoded `text-gray-500` instead of theme token.**

The `CompletionEstimate` component uses `text-gray-500`:

```tsx
className={cn('inline-flex items-center gap-1.5 text-sm text-gray-500', className)}
```

The project has `--color-muted-foreground` mapped to `text-muted-foreground` in `theme.css`, which is what every other metadata line in `CourseCard.tsx` uses (e.g., line 504: `text-xs text-muted-foreground`). Hardcoded Tailwind color classes bypass the theme system and will not adapt to dark mode correctly -- `text-gray-500` stays the same gray in dark mode, while `text-muted-foreground` shifts to the appropriate contrast color.

Why: In dark mode, learners will see low-contrast gray text on a dark background, making the completion estimate unreadable.

Fix: Replace `text-gray-500` with `text-muted-foreground`.

**H2. [`/src/lib/completionEstimate.ts:40,44`] (confidence: 90) -- Division by zero when all recent sessions have `duration: 0`.**

If all sessions within the 30-day window have `duration: 0` (e.g., orphaned sessions, sessions that never ended properly), the average calculation produces:

```typescript
const totalMinutes = recentSessions.reduce((sum, s) => sum + s.duration / 60, 0)  // 0
averageSessionMinutes = totalMinutes / recentSessions.length  // 0
// ...
const sessionsNeeded = Math.ceil(remainingContentMinutes / averageSessionMinutes) // Infinity
```

`Math.ceil(X / 0)` produces `Infinity`, which then renders as "Est. ~Infinity sessions" in the UI. The `StudySession` type allows `duration: 0` for orphaned sessions (line 213 of `types.ts`: `duration: number // Active seconds`).

Why: A learner who opened a session tracker but never studied would see "Infinity sessions" on every course card.

Fix: Guard against zero average by falling back to the default:

```typescript
if (recentSessions.length === 0 || averageSessionMinutes <= 0) {
  averageSessionMinutes = DEFAULT_SESSION_MINUTES
}
```

---

**H3. [`/src/lib/completionEstimate.ts:44`] (confidence: 85) -- `Math.ceil(0 / 30)` returns `0`, but `remainingMinutes` is also `0` -- inconsistent display.**

When `remainingContentMinutes` is `0` (course fully completed), `Math.ceil(0 / 30) = 0`, producing `sessionsNeeded: 0`. The UI guard in `CourseCard.tsx:518` checks `completionEstimate.remainingMinutes > 0`, which correctly hides the estimate. However, if `remainingContentMinutes` is negative (more lessons completed than total, which can happen with stale progress data), `Math.ceil(-15 / 30) = 0`, `remainingMinutes` would be `-15`, and `-15 > 0` is false, so it hides -- but the data itself is semantically wrong.

While the UI guard saves the display, the calculation function should clamp `remainingContentMinutes` to `Math.max(0, ...)` for data integrity downstream.

Why: Other consumers of the `CompletionEstimate` interface (e.g., future reports page, tooltips) won't have the same UI guard and could display negative remaining time.

Fix: Add `remainingContentMinutes = Math.max(0, remainingContentMinutes)` at the top of `calculateCompletionEstimate`.

**H4. [`/src/app/components/figma/AtRiskBadge.tsx:26`] (confidence: 82) -- Hardcoded Tailwind orange colors instead of a theme token, inconsistent with `MomentumBadge` pattern.**

`AtRiskBadge` uses raw Tailwind classes:
```tsx
'bg-orange-50 text-orange-700 dark:bg-orange-900/30 dark:text-orange-300'
```

Meanwhile, `MomentumBadge` uses theme tokens (`text-momentum-hot bg-momentum-hot-bg`) that are defined in `theme.css` with proper light/dark mode values. The project has `--warning: #d97706` and `--warning-foreground: #ffffff` in the theme system (theme.css lines 63-64) which exist specifically for this use case.

Why: Hardcoded colors create a maintenance burden and diverge from the design token system. If the brand's warning palette changes, these won't update.

Fix: Either use the existing `text-warning bg-warning/10` pattern, or add dedicated at-risk tokens to `theme.css` (e.g., `--at-risk`, `--at-risk-bg`) and reference them as `text-at-risk bg-at-risk-bg`, mirroring the momentum token pattern.

**H5. [`/tests/e2e/story-e07-s04.spec.ts:12-15`] (confidence: 80) -- Missing sidebar state seed in `beforeEach` -- tablet viewport will fail.**

The `beforeEach` hook navigates directly to `/courses` without seeding the sidebar localStorage state:

```typescript
test.beforeEach(async ({ page }) => {
  await page.goto('/courses')
})
```

Per CLAUDE.md and established patterns (e.g., `tests/e2e/offline-smoke.spec.ts:17-22`), tests must seed `localStorage.setItem('knowlune-sidebar-v1', 'false')` before navigation to prevent the Sheet overlay from blocking interactions at 640-1023px viewports. The CI matrix includes "Mobile Chrome", "Mobile Safari", and "Tablet" projects that will hit this viewport range.

Why: AC6 test (line 191) clicks on `[data-testid="sort-select"]` which will be blocked by the sidebar overlay on tablet, causing flaky test failures in CI.

Fix: Add sidebar seeding before navigation:

```typescript
test.beforeEach(async ({ page }) => {
  await page.evaluate(() => {
    localStorage.setItem('knowlune-sidebar-v1', 'false')
  })
  await page.goto('/courses')
})
```

**M1. [`/tests/e2e/story-e07-s04.spec.ts:195`] (confidence: 78) -- Hard wait `page.waitForTimeout(500)` in AC6 sort test.**

```typescript
// Give it a moment to sort
```

CLAUDE.md explicitly prohibits `waitForTimeout()` without justification. The sort is synchronous (a React state update + re-render), so this wait is unnecessary. A Playwright assertion with auto-retry would be more deterministic.

**M2. [`/src/lib/atRisk.ts:25,38`] and [`/src/lib/completionEstimate.ts:25,26`] (confidence: 75) -- `Date.now()` calls make pure functions impure and hard to unit-test.**

Both calculation functions call `Date.now()` internally. The `momentum.ts` module has the same pattern, but as the traceability matrix noted, there are zero unit tests for these functions. The `Date.now()` dependency means unit tests would need to mock the global `Date`, adding unnecessary complexity.

Fix: Accept an optional `now` parameter (defaulting to `Date.now()`) to enable deterministic unit tests:

```typescript
export function calculateAtRiskStatus(
  sessions: StudySession[],
  momentumScore: MomentumScore,
  now: number = Date.now()
): AtRiskStatus { ... }
```

**M3. [`/src/app/components/figma/CompletionEstimate.tsx:32-33`] (confidence: 72) -- Whitespace between unit and plural suffix renders as "session s".**

```tsx
Est. ~{displayValue} {displayUnit}
{pluralSuffix}
```

JSX preserves the newline between `{displayUnit}` and `{pluralSuffix}` as a space, so the output renders as `Est. ~3 session s` instead of `Est. ~3 sessions`. This is a rendering bug.

Fix: Concatenate the unit and suffix on the same expression:

```tsx
Est. ~{displayValue} {displayUnit}{pluralSuffix}
```

Or use template literal: `{`${displayUnit}${pluralSuffix}`}`.
