### What Works Well

1. **Defense-in-depth architecture.** Validation at three layers (momentum.ts, Courses.tsx, StudyScheduleWidget.tsx) ensures corrupted data cannot propagate even if one guard is bypassed. This is the correct pattern for a client-side app with no schema enforcement at the storage layer.

2. **Appropriate use of `isFinite()`.** The single `isFinite(s.duration)` check catches `NaN`, `Infinity`, and `-Infinity` in one call -- cleaner and more correct than chaining multiple guards. The implementation notes document this decision, which helps future maintainers.

3. **Test data design.** The corrupted session test fixtures cover a good variety of corruption vectors: missing fields, wrong types, malformed timestamps, boundary numeric values, and mixed valid+invalid. The manual IDB seeding is correctly justified since factory helpers enforce valid schemas by design.

#### Blockers

#### High Priority

- **[tests/e2e/regression/story-e07-s07.spec.ts:219-238] (confidence: 92)**: `NaN` and `Infinity` do not survive Playwright's `page.evaluate()` serialization. Playwright uses structured clone / JSON semantics to pass arguments from Node to browser context. `NaN` becomes `null` and `Infinity` becomes `null` in JSON. The test at line 222 seeds `duration: NaN` and line 233 seeds `duration: Infinity`, but by the time these reach the browser's IndexedDB, they are stored as `null`, not as `NaN`/`Infinity`. This means the test validates the "wrong type" path (`typeof null !== 'number'`) rather than the intended `isFinite()` guard path. The test passes but for the wrong reason -- it does not exercise the `isFinite()` branch at all. Why: A future refactor that removes the `isFinite()` check would not be caught by this test, leaving a real corruption vector unguarded. Fix: Seed these values from within the browser context using `page.evaluate()` with the values constructed inside the callback:
  ```typescript
  // Inside page.evaluate callback:
  store.add({ ...baseSession, id: 'corrupt-nan', duration: Number('not-a-number') })
  store.add({ ...baseSession, id: 'corrupt-inf', duration: 1/0 })
  ```

- **[src/lib/momentum.ts:24-28] (confidence: 85)**: `isValidSession` accepts `StudySession` type but at runtime, corrupted data from IndexedDB may not conform to the `StudySession` interface at all -- the object could be `null`, `undefined`, or a primitive. Dexie's `toArray()` can return whatever was stored. If a corrupted record is stored as `null` or a non-object, accessing `s.courseId` throws a TypeError. Why: A single `null` record in IndexedDB would crash the entire Courses page despite all the defensive code. Fix: Add a null/object guard as the first check:
  ```typescript
  function isValidSession(s: StudySession): boolean {
    if (!s || typeof s !== 'object') return false
    // ... existing checks
  }
  ```

- **[src/app/components/StudyTimeAnalytics.tsx:54, src/app/pages/SessionHistory.tsx:88, src/stores/useSessionStore.ts:230-232, src/lib/xapiStatements.ts:189, src/lib/exportService.ts:174] (confidence: 80)**: Five other `studySessions.toArray()` call sites have no corrupted-data guards. The story's AC states "corrupted session data exists ... navigation works correctly without errors", which implies app-wide resilience. These unguarded consumers will still crash or produce NaN-poisoned output if they encounter corrupted data. Why: If a learner has corrupted sessions, navigating to Session History, Reports, or triggering xAPI export would still crash. Fix: Either export `isValidSession` from `momentum.ts` (or a shared `lib/sessionValidation.ts`) and apply it at each call site, or add a Dexie middleware/hook that validates on read. This could be a follow-up story if scope is intentionally limited to Courses page.

#### Medium

- **[tests/e2e/regression/story-e07-s07.spec.ts:51] (confidence: 78)**: `indexedDB.open(DB_NAME)` without a version number opens the DB at its current version. If the Dexie schema has been updated between test runs and the DB doesn't exist yet, this opens at version 1 with no object stores, and the `db.transaction(STORE_NAME, 'readwrite')` call will throw `NotFoundError`. The test works because `goToCourses()` navigates first (which triggers Dexie to create the DB), but this ordering dependency is fragile. Fix: Add a comment documenting this dependency, or use `indexedDB.open(DB_NAME, currentVersion)` explicitly.

- **[tests/e2e/regression/story-e07-s07.spec.ts:116,156,196,...] (confidence: 75)**: Every test manually calls `await indexedDB.clearStore(STORE_NAME)` at the end. If a test fails before reaching the cleanup line, corrupted data leaks into subsequent tests. Fix: Move cleanup into `test.afterEach()`:
  ```typescript
  test.afterEach(async ({ indexedDB }) => {
    await indexedDB.clearStore(STORE_NAME)
  })
  ```

- **[src/lib/momentum.ts:37-41] (confidence: 72)**: `console.warn` on every render cycle with corrupted data creates log noise. If the Courses page re-renders (state change, tab switch), the warning fires repeatedly for the same corrupted sessions. This is minor in production but clutters test output and developer consoles. Fix: Consider memoizing the warning (e.g., a `Set` of already-warned courseIds) or logging only once per mount cycle via the caller.

- **[tests/e2e/regression/story-e07-s07.spec.ts:289-331] (confidence: 70)**: The "momentum calculation skips corrupted sessions and uses valid ones" test (AC2) asserts only that the badge text matches `/hot|warm|cold/i`. This is a tautology -- the badge always shows one of these three values, even with zero sessions (which yields score 0 = "cold"). The test does not verify that the *valid* session actually contributed to the momentum score. Fix: Assert a specific tier based on the seeded data. With `FIXED_DATE` as both `Date.now()` and the session's `startTime`, recency is 0 days = score 100, so the badge should be "Hot" (assuming non-zero completion). Assert `expect(badgeText).toMatch(/hot/i)` to confirm the valid session was actually used.

#### Nits

- **Nit** [tests/e2e/regression/story-e07-s07.spec.ts:47-48] (confidence: 65): `DB_NAME = 'ElearningDB'` is hardcoded in the test helper. If the Dexie DB name changes in `src/db/schema.ts`, this test will silently seed the wrong database. Consider importing the DB name from a shared constant.

- **Nit** [src/app/pages/Courses.tsx:72] (confidence: 60): The filter `s => typeof s.courseId === 'string' && s.courseId` treats empty string as invalid. This is correct behavior but is an implicit contract -- worth a brief comment noting the intent (empty courseId = corrupted, not just missing).

### Recommendations

1. **First**: Fix `NaN`/`Infinity` test serialization (High #1) -- tests are passing for the wrong reason.
2. **Second**: Add null/object guard to `isValidSession` (High #2) -- prevents TypeError on truly corrupted records.
3. **Third**: Move test cleanup to `afterEach` (Medium #2) -- prevents test pollution on failure.
4. **Fourth**: Strengthen the mixed valid+corrupted assertion (Medium #4) -- currently a tautology.
5. **Consider for follow-up story**: Guard the other five `toArray()` call sites (High #3) -- true app-wide resilience.
