### Findings

#### High Priority

- **[VideoPlayer.tsx:347] (confidence: 90)**: `stepPlaybackSpeed` does not handle the case where `playbackSpeed` is not in the `PLAYBACK_SPEEDS` array. `PLAYBACK_SPEEDS.indexOf(playbackSpeed)` returns `-1` when localStorage contains a non-standard value (e.g., `"1.3"`, `"abc"`, or `"NaN"` from corruption). When `currentIndex === -1` and `direction === 'up'`, `PLAYBACK_SPEEDS[-1 + 1]` resolves to `PLAYBACK_SPEEDS[0]` (0.5x) -- silently jumping the user to the slowest speed instead of the next speed up. When `direction === 'down'`, `currentIndex <= 0` is true for `-1`, so the user gets "Already at minimum speed" with no way to recover. Why: A learner who manually set an unsupported speed (or whose localStorage was corrupted) would be stuck or confused. Fix: Add a guard at the top of `stepPlaybackSpeed`:
  ```typescript
  if (currentIndex === -1) {
    // Snap to nearest valid speed
    const nearest = PLAYBACK_SPEEDS.reduce((prev, curr) =>
      Math.abs(curr - playbackSpeed) < Math.abs(prev - playbackSpeed) ? curr : prev
    )
    changePlaybackSpeed(nearest)
    return
  }
  ```

- **[LessonPlayer.tsx:504-506, 514-516] (confidence: 78)**: `handleFocusNotes` uses `document.querySelector('[contenteditable="true"]')` -- a global DOM query that grabs the first contenteditable element on the page. This is fragile and could focus the wrong element if other contenteditable elements exist (e.g., a future inline edit field, or a third-party widget). More importantly, when the notes panel first mounts, a single `requestAnimationFrame` may fire before TipTap has initialized its contenteditable element, causing `editor` to be `null` and the focus call to silently fail. Why: The N key would appear to do nothing -- the panel opens but focus stays on the player, forcing the user to click. Fix: Use a ref on the NoteEditor component with an imperative `focus()` method (already mentioned in the story tasks as "2.4 Add NoteEditor ref/imperative handle for focus" but not implemented). Alternatively, use a more targeted selector like `[data-testid="note-editor"] [contenteditable="true"]` and retry with a short timeout/MutationObserver.

#### Medium

- **[VideoPlayer.tsx:131-134] (confidence: 75)**: `parseFloat` on localStorage produces any float, not just values in `PLAYBACK_SPEEDS`. Combined with the finding above, this is the entry point for invalid state. While this is pre-existing code (not introduced in this PR), the new `stepPlaybackSpeed` function makes it exploitable. Fix: Validate the parsed value against `PLAYBACK_SPEEDS` during initialization:
  ```typescript
  const [playbackSpeed, setPlaybackSpeed] = useState(() => {
    const saved = localStorage.getItem(STORAGE_KEY_PLAYBACK_SPEED)
    const parsed = saved ? parseFloat(saved) : 1
    return PLAYBACK_SPEEDS.includes(parsed) ? parsed : 1
  })
  ```

- **[LessonPlayer.tsx:498-509] (confidence: 72)**: `handleFocusNotes` is wrapped in `useCallback` with `[notesOpen]` as the dependency, but it closes over `setNotesOpen` and `setPendingNoteFocus` (stable) and calls `requestAnimationFrame` with `document.querySelector`. The `useCallback` dependency is correct for the branching logic, but the function is recreated every time `notesOpen` changes, which causes `VideoPlayer`'s `useEffect` (keyboard handler) to re-register. This is harmless (the existing pattern already re-registers on every render since no functions are `useCallback`-wrapped), but wrapping only `handleFocusNotes` in `useCallback` creates a false sense of optimization. Fix: Either remove the `useCallback` (matching other handlers) or memoize all keyboard handler dependencies consistently. Low priority -- cosmetic.

- **[VideoShortcutsOverlay.tsx:23] (confidence: 70)**: The `<` and `>` shortcut entry uses a single row with `keys: ['<', '>']` and `description: 'Speed down/up'`. The `ShortcutRow` component renders a `+` separator between multiple keys (line 55: `{i > 0 && <span>+</span>}`), so this displays as `< + >` which implies the user should press both keys simultaneously. It should either be two separate entries or use a `/` separator instead of `+`. Fix: Split into two entries or add a conditional separator character:
  ```typescript
  { keys: ['<'], description: 'Speed down' },
  { keys: ['>'], description: 'Speed up' },
  ```

#### Nits

- **Nit** [LessonPlayer.tsx:497] (confidence: 60): Comment reads `/ N keyboard shortcut:` -- appears to be missing the leading `/` for a `//` comment. This is likely a rendering artifact from the diff but worth verifying in the actual file.

- **Nit** [story-e21-s02-keyboard-shortcuts.spec.ts:19] (confidence: 55): `page.waitForLoadState('networkidle')` in `focusPlayer()` is a Playwright-discouraged pattern as it can hang indefinitely if long-polling or WebSocket connections exist. Current usage works because the lesson player page settles quickly, but consider using `page.waitForLoadState('domcontentloaded')` for resilience.

### Recommendations

1. **(High)** Fix `stepPlaybackSpeed` to handle `indexOf === -1` -- this is a correctness bug reachable from corrupted localStorage.
2. **(High)** Replace `document.querySelector('[contenteditable="true"]')` with a targeted ref or scoped selector to prevent silent focus failures and future fragility.
3. **(Medium)** Validate localStorage-parsed speed against `PLAYBACK_SPEEDS` array during initialization.
4. **(Medium)** Fix the `< + >` display in the shortcuts overlay to avoid implying a simultaneous keypress.
