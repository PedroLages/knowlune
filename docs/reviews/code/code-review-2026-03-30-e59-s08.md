## Story-Related Issues

### HIGH

1. **[HIGH] E2E test AC2 fails — `review-empty-state` testid not found** — `tests/e2e/story-e59-s08.spec.ts:98`
   The test seeds a future-due review record and expects `getByTestId('review-empty-state')` to appear, but the element is never rendered. Either the app does not render an empty state with that testid when all cards are future-due, or the seeded data is not being filtered correctly by the FSRS `due` field. This is a story-related bug — the test assertion does not match the actual app behavior.

2. **[HIGH] E2E test AC4 fails — flashcard due count shows 2 instead of 1** — `tests/e2e/story-e59-s08.spec.ts:203`
   The test seeds 1 due flashcard and 1 future flashcard, then asserts `flashcard-stats-due` contains "1". The actual value is "2", meaning the app counts both cards as due. Either the flashcard dashboard does not filter by the FSRS `due` field, or `createFutureFlashcard()` produces a card that the app still considers due. The test expectation may be correct but the due-date filtering logic in the flashcard store/component is not FSRS-aware yet.

### MEDIUM

The inline review record helpers were updated to FSRS fields, but the app's review/retention UI may still expect SM-2 fields (or the FSRS fields are not being interpreted correctly by the review store). These failures need investigation — they may indicate the production review store has not been fully migrated to FSRS yet, or the test data shape is incorrect.
