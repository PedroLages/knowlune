## Code Review: E16-S05 — Display Score Improvement Trajectory Chart

### What Works Well

1. **Clean component API design.** `ScoreTrajectoryChart` receives pre-mapped data and a single `passingScore` prop. The parent (`QuizResults`) owns the data transformation via `useMemo`, keeping the chart component pure and testable. This is a textbook separation of concerns.

2. **AC-aligned guard clause.** The `attempts.length < 2` early return at line 40 cleanly implements AC3 and prevents recharts from rendering a meaningless single-point line chart. The guard is tested in both unit and E2E tests.

3. **Good recharts configuration.** Animation is disabled (`isAnimationActive={false}`) preventing E2E flakiness, the Y-axis domain is locked to `[0, 100]`, and the `ChartContainer` wrapper from shadcn/ui is used instead of raw `ResponsiveContainer` for theme consistency.

### Findings

#### Blockers

None.

#### High Priority

- **[Recurring] src/app/components/quiz/ScoreTrajectoryChart.tsx:32 (confidence: 92)**: Hardcoded hex color `stroke="#fff"` in the custom dot SVG circle. This bypasses the theme system entirely -- in dark mode, white stroke on a dark background may still work visually, but it violates the design token contract and will not adapt if the card background token changes. Why: Learners using dark mode may see incorrect contrast on dot outlines if card backgrounds change. Fix: Replace `stroke="#fff"` with `stroke="var(--color-card)"` to use the card background token, which correctly flips between light and dark modes.

- **src/app/components/quiz/ScoreTrajectoryChart.tsx:88 (confidence: 82)**: `makeCustomDot(passingScore)` is called inline during render, creating a **new function reference on every render**. Recharts compares the `dot` prop by reference; a new function forces recharts to re-mount all dot elements on every parent re-render. This is a performance anti-pattern for chart components. Why: In QuizResults, any state change (e.g., tooltip hover, resize) triggers re-render of ScoreTrajectoryChart, causing all dots to unmount/remount unnecessarily. Fix: Memoize the custom dot factory with `useMemo`:
  ```tsx
  const customDot = useMemo(() => makeCustomDot(passingScore), [passingScore])
  // ...
  <Line dot={customDot} ... />
  ```

- **tests/e2e/story-e16-s05.spec.ts:63-97 (confidence: 85)**: Manual `indexedDB.open()` seeding logic duplicates the shared helper at `tests/support/helpers/indexeddb-seed.ts` which already exports `seedQuizzes()` and `seedQuizAttempts()`. This is a known anti-pattern caught by the `test-patterns/use-seeding-helpers` ESLint rule. Why: Duplicated seeding logic drifts from the shared implementation; if the DB schema changes (e.g., store name rename), this test breaks while tests using the shared helper get updated in one place. Fix: Replace the inline `seedData` function with:
  ```ts
  import { seedQuizzes, seedQuizAttempts } from '../../support/helpers/indexeddb-seed'
  // ...
  await seedQuizzes(page, [quiz])
  await seedQuizAttempts(page, [attempt1, attempt2])
  ```

#### Medium

- **src/app/components/quiz/ScoreTrajectoryChart.tsx:36-40 (confidence: 75)**: `useIsMobile()` hook is called unconditionally before the early-return guard. React hooks must be called in the same order every render (which is satisfied), but calling `useIsMobile` triggers a `useEffect` + `addEventListener('change', ...)` even when the component returns `null` (fewer than 2 attempts). This is wasted work for the common case of a learner's first quiz attempt. Fix: This is technically correct per React rules of hooks, but consider restructuring: move the guard to the parent (`QuizResults.tsx`) so the component is not even mounted when there are fewer than 2 attempts. The `trajectoryData` memo already exists in QuizResults -- add a conditional render:
  ```tsx
  {trajectoryData.length >= 2 && (
    <ScoreTrajectoryChart attempts={trajectoryData} passingScore={currentQuiz.passingScore} />
  )}
  ```
  Then the internal guard becomes a defensive backup rather than the primary path.

- **src/app/components/quiz/ScoreTrajectoryChart.tsx:47 (confidence: 72)**: Inline `style={{ height: chartHeight }}` is used for responsive height. The project convention (from `.claude/rules/styling.md`) prefers Tailwind utilities over inline styles. The `useIsMobile` approach adds a JS-level media query when pure CSS would suffice. Fix: Use Tailwind responsive classes on the ChartContainer: `className="w-full h-[200px] sm:h-[300px]"` and remove `style` prop and `useIsMobile` import entirely. This eliminates the hook overhead and is more aligned with the project's CSS-first philosophy.

- **src/app/components/quiz/__tests__/ScoreTrajectoryChart.test.tsx (confidence: 70)**: Unit tests do not cover the custom dot color differentiation logic (above/below passing score). The `makeCustomDot` factory is a key part of AC2 ("data points above the line are visually distinguished from those below"), but tests mock `Line` as `() => null`, making dot rendering untestable. Why: If the dot color logic regresses (e.g., comparison operator flipped from `>=` to `>`), no test catches it. Fix: Add a dedicated test that renders `makeCustomDot(70)` directly and asserts the fill color based on payload percentage. Since it returns JSX, this can be tested without mocking recharts:
  ```tsx
  it('colors dots green when at/above passing score', () => {
    const Dot = makeCustomDot(70)
    const { container } = render(<svg><Dot cx={10} cy={10} payload={{ percentage: 70 }} /></svg>)
    expect(container.querySelector('circle')).toHaveAttribute('fill', 'var(--color-success)')
  })
  ```

#### Nits

- **Nit** src/app/components/quiz/ScoreTrajectoryChart.tsx:46 (confidence: 70): The heading uses `<h4>` but this component's position in the page hierarchy is not guaranteed. If rendered after an `<h1>` (the quiz title), the heading level jump from h1 to h4 skips h2 and h3, which is a minor WCAG heading hierarchy concern. Consider using a prop for heading level or a generic `<p>` with `role="heading" aria-level="2"`.

- **Nit** tests/e2e/story-e16-s05.spec.ts:70 (confidence: 65): The loop variable `attempt` in the retry logic shadows the outer test concept of "quiz attempt," making the code harder to read. Consider renaming to `retryCount` or `i`.
