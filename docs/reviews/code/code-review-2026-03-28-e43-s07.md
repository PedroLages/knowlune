## Summary

The implementation introduces a typed event bus, NotificationService, and wires 5 stores to emit domain events. NotificationCenter.tsx was refactored from mock data to real store data. Architecture is clean and well-organized.

## Findings

### HIGH Priority

**1. [H1] Streak milestone emitted on every session end — not just milestones**
- **File:** `src/stores/useSessionStore.ts:221-222`
- **Confidence:** 95
- **Description:** The store emits `streak:milestone` for ANY positive streak value (`if (currentStreak > 0)`), not just at milestone thresholds. While `NotificationService.handleEvent()` filters for milestone values, this creates unnecessary event bus traffic on every session end. More critically, if another future subscriber listens to `streak:milestone`, it will receive non-milestone events.
- **Fix:** Move the milestone threshold check into the store before emitting:
  ```typescript
  const STREAK_MILESTONES = [7, 14, 30, 60, 100, 365]
  if (STREAK_MILESTONES.includes(currentStreak)) {
    appEventBus.emit({ type: 'streak:milestone', days: currentStreak })
  }
  ```

**2. [H2] `hasReviewDueToday()` uses `new Date()` for deduplication — not deterministic**
- **File:** `src/services/NotificationService.ts:28-29`
- **Confidence:** 85
- **Description:** The deduplication logic creates `todayStart` via `new Date()` and converts to ISO string for comparison. If the stored `createdAt` uses a slightly different timezone or ISO format, the filter `n.createdAt >= todayIso` may produce incorrect results. Also, comparing ISO strings lexicographically works for UTC but is fragile if dates include timezone offsets.
- **Fix:** Use `toLocaleDateString('sv-SE')` pattern (per engineering-patterns.md) for date comparison, or compare Date objects directly.

**3. [H3] Missing `await` in `handleNotificationClick` — `storeMarkRead` is async**
- **File:** `src/app/components/figma/NotificationCenter.tsx:83`
- **Confidence:** 90
- **Description:** `storeMarkRead(id)` is an async function that persists to Dexie, but the click handler does not await it. Navigation via `navigate(actionUrl)` happens immediately after. If the Dexie write fails, the notification appears marked as read in the store (optimistic update issue depends on store implementation — checking store: it updates after persist, so a failed persist would leave the notification unread, which is OK). This is actually fine since markRead persists first then updates store. But fire-and-forget async calls can swallow errors.
- **Severity adjusted:** MEDIUM — the error is logged in the store's catch block, but the UI will not reflect failure feedback. Consider adding `await` or at minimum ensuring the store handles the error gracefully (which it does).

### MEDIUM Priority

**4. [M1] `getCurrentStreak()` is synchronous (reads localStorage) but called in async context**
- **File:** `src/stores/useSessionStore.ts:220`
- **Confidence:** 75
- **Description:** `getCurrentStreak()` reads from localStorage synchronously. This is called inside the `.then()` chain of `endSession`. If localStorage is slow or corrupted, it could throw synchronously inside the promise chain. The error would be caught by the existing `.catch()` handler, so it's not critical. However, the streak value computed here might be stale if multiple sessions end rapidly.
- **Impact:** Low — localStorage reads are effectively instant and the catch handler exists.

**5. [M2] NotificationService uses module-level mutable state (`unsubscribers` array)**
- **File:** `src/services/NotificationService.ts:21`
- **Confidence:** 80
- **Description:** The `unsubscribers` array is module-level mutable state. While `initNotificationService()` calls `destroyNotificationService()` first (idempotent), in React Strict Mode the effect runs twice, which would: init → destroy → init → destroy → init. This should be fine due to the idempotent design, but the module-level state pattern is fragile for testing.

**6. [M3] `console.log` in production code**

- **Confidence:** 90
- **Description:** `console.log('[NotificationService] Initialized...')` should use `console.debug` or be removed for production. This will appear in every app startup in the browser console.

**7. [M4] `notificationIcons` Record type uses `Record<NotificationType, ...>` but has fallback**
- **File:** `src/app/components/figma/NotificationCenter.tsx:23,31`
- **Confidence:** 70
- **Description:** The icon and color maps are typed as `Record<NotificationType, ...>` which guarantees all keys are present at compile time. Yet the code uses `?? DEFAULT_ICON` fallback at runtime (line 159-160). The fallback is defensive but redundant given the Record type. If NotificationType is extended in the future, TypeScript will require updating these Records — which is the correct behavior. The fallback is harmless but misleading.
