### What Works Well

- `generateHexToken()` correctly uses `crypto.getRandomValues(new Uint8Array(20))` — 160-bit entropy, browser-safe, no Node dependency. The comment explaining why Node's `crypto.randomBytes` must not be used is valuable and precise.
- `feedLoading` boolean guards all async token operations, preventing race conditions from rapid UI toggling. This directly addresses EC-38 from the edge case review.
- RLS policies are complete and correct — SELECT, INSERT, UPDATE, DELETE all scoped to `auth.uid() = user_id`. Server uses service role key to bypass RLS for legitimate server-side reads (confirmed in `server/routes/calendar.ts`).
- `upsert` with `onConflict: 'user_id'` in `generateFeedToken` correctly handles idempotent token creation — toggling the feed off and on without a regenerate preserves the URL.

#### High Priority

- **[src/stores/useStudyScheduleStore.ts:220] (confidence: 90)** `[Correctness]`: `regenerateFeedToken` delete error is not checked. `await supabase.from('calendar_tokens').delete().eq('user_id', user.id)` returns `{ error }` which is silently discarded. If the delete fails (network error, RLS policy hiccup), the subsequent INSERT will succeed, leaving the user with two tokens — but the DB's `UNIQUE (user_id)` constraint will reject it with a violation, surfacing a confusing error. Worse, if the delete silently fails and the insert is skipped due to a logic error, the old token remains active. Why: Security invariant — regeneration must guarantee the old URL stops working. Fix: `const { error: deleteError } = await supabase.from('calendar_tokens').delete().eq('user_id', user.id); if (deleteError) throw deleteError;` Effort: ~2 min.

- **[src/stores/useStudyScheduleStore.ts:226-241] (confidence: 85)** `[Correctness]`: After a successful delete but failed INSERT, state is inconsistent. The `catch` block leaves `feedToken` at whatever it was before the call (the old token value), but the DB has no row. `getFeedUrl()` will return a URL with a token that no longer exists. Why: User sees an active feed URL that silently returns 404. Fix: In the catch block after the insert fails, explicitly clear state: `set({ feedToken: null, feedEnabled: false })` — the old token was already deleted, so the correct state is "no token". Effort: ~5 min.

#### Medium

- **[src/lib/icalFeedGenerator.ts:145-165] (confidence: 75)** `[Silent Failure]`: `generateIcsDownload` has no error handling. If `generateICalFeed` throws (malformed schedule, ical-generator error), the exception bubbles uncaught to the caller. Since this is likely called from a button handler, the error will be swallowed silently unless the UI wraps it. Fix: Wrap in try/catch with `toast.error('Failed to generate calendar file')`. Effort: ~5 min.

- **[src/stores/useStudyScheduleStore.ts:139-160] (confidence: 72)** `[Silent Failure]`: `loadFeedToken` catches errors silently. If this fails on app start (Supabase unavailable, session expired), the feed state remains at initial values (`feedToken: null, feedEnabled: false`) — which is actually the correct safe default. However, if Zustand persist was previously storing `feedEnabled: true`, the in-memory state won't reflect that and the UI will show the feed as disabled while the DB has a valid token. Fix: Either explicitly reset to safe defaults in catch (`set({ feedToken: null, feedEnabled: false })`), or surface a non-blocking toast. Effort: ~3 min.

#### Nits

- **Nit** [src/stores/useStudyScheduleStore.ts:163-201] (confidence: 60): `generateFeedToken` and `regenerateFeedToken` both call `supabase.auth.getUser()` independently. Could be a shared `getCurrentUser()` helper — minor DRY violation, not worth a refactor now.

- **Nit** [src/stores/useStudyScheduleStore.ts:276] (confidence: 55): `VITE_API_BASE_URL` is not documented in `.env.example`. Developers setting up the project won't know this env var exists. Fix: Add `# VITE_API_BASE_URL=https://api.yourdomain.com  # Optional: override for calendar feed URL base` to `.env.example`.
