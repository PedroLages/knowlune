## Code Review: E14-S02 — Display Multiple Select Questions with Partial Credit

### What Works Well

1. **Clean PCM scoring implementation.** The `calculatePointsForQuestion()` helper in `/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/lib/scoring.ts` is well-isolated, uses Set for efficient lookups, and preserves backward compatibility for all other question types via the fallback path.

2. **Consistent component pattern.** `MultipleSelectQuestion` mirrors the structural pattern of `TrueFalseQuestion` and `MultipleChoiceQuestion` — fieldset/legend, design tokens, shared markdown config. This makes the quiz component family predictable and maintainable.

3. **Thoughtful E2E test design.** The test file at `/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/tests/e2e/story-e14-s02.spec.ts` covers all seven ACs, uses factory-based seeding, scopes selectors to fieldset to avoid MarkForReview collisions, and tests keyboard navigation with Tab (not Arrow) correctly for checkboxes.

### Findings

#### Blockers

- **[MultipleSelectQuestion.tsx:26-31] (confidence: 92): `useMemo` abused as a side-effect hook.** The `useMemo` on line 26 is used solely for its side-effect (`console.warn`). `useMemo` is a pure computation hook — React may skip or re-run it at any time (React docs explicitly state "your code should work without it"). This is a side-effect masquerading as a computation. **Note:** This is an existing pattern copied from `TrueFalseQuestion.tsx:21` and `MultipleChoiceQuestion.tsx:26`, so it is pre-existing tech debt — but it should not be propagated further. Why: React 19's compiler may aggressively drop or re-run memos, causing warnings to fire unpredictably or not at all. Fix: Replace with `useEffect` for development-only validation, or a top-of-render `if` block with `console.warn` (which is idempotent and acceptable in render).

- **[MultipleSelectQuestion.tsx — entire file] (confidence: 95): AC6 (feedback display) is completely missing from the component.** The story requires: "Multiple Select questions show 'X of Y correct' with per-option indicators (correct selected, correct missed, incorrect selected)." The component has zero logic for `mode !== 'active'` — no review-mode rendering, no per-option feedback icons (check, alert, x), no "X of Y correct" summary. The E2E test for AC6 passes only because it asserts against the `ScoreSummary` component (quiz-level "X of Y correct"), NOT the per-question indicators the AC specifies. Why: Learners reviewing submitted quizzes will see no per-option feedback on multiple-select questions — they cannot tell which options they got right or wrong. Fix: Implement review-mode rendering with per-option indicators as specified in the story's Design Guidance section (correct selected = `text-success` + check icon, correct missed = `text-warning` + alert icon, incorrect selected = `text-destructive` + x icon).

#### High Priority

- **[MultipleSelectQuestion.tsx:51] (confidence: 85): "Select all that apply" hint is not programmatically associated with the legend for screen readers.** The story's accessibility requirements (AC7) state: "'Select all that apply' must be part of or associated with the legend for screen readers." Currently the `<span>` is a sibling of `<legend>`, placed outside the legend element, with no `aria-describedby` link. Screen readers will read the legend text but not the hint. Why: Screen reader users will not know this is a "select all that apply" question — they will only hear the question text and must infer the interaction model. Fix: Either move the hint `<span>` inside `<legend>` (after the Markdown), or add `aria-describedby` on the fieldset pointing to an `id` on the hint span.

- **[scoring.ts:50-51] (confidence: 80): Unsafe `as string[]` cast on `question.correctAnswer` without runtime guard.** If `correctAnswer` is somehow a string (e.g., data corruption, future refactoring error), `new Set(question.correctAnswer as string[])` will create a Set of individual characters, silently producing wrong scores. The `isCorrectAnswer` function on line 30 properly checks `Array.isArray(correct)`, but `calculatePointsForQuestion` does not. Why: A corrupted or misconfigured question could silently award incorrect points without any error signal. Fix: Add a runtime guard: `if (!Array.isArray(question.correctAnswer)) return { pointsEarned: 0, isCorrect: false }`.

- **[scoring.ts:84] (confidence: 78): `userAnswer ?? ''` coerces unanswered multiple-select to empty string instead of empty array.** When a multiple-select question is unanswered (`userAnswer === undefined`), the fallback `?? ''` produces `userAnswer: ''` (a string) in the `Answer` record. The `AnswerSchema` at `/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/types/quiz.ts:157` expects `string | string[]` — an empty string is valid but semantically wrong for multiple-select (should be `[]`). Why: Downstream consumers checking `Array.isArray(answer.userAnswer)` to detect multiple-select answers will misclassify unanswered multiple-select questions. Fix: `userAnswer: userAnswer ?? (question.type === 'multiple-select' ? [] : '')`.

#### Medium

- **[MultipleSelectQuestion.tsx:53] (confidence: 72): Redundant `role="group"` on a div inside a `<fieldset>`.** The `<fieldset>` already provides implicit group semantics. Adding `role="group"` with `aria-labelledby` on a nested div creates a redundant grouping that some screen readers may announce twice, adding verbosity without benefit. Fix: Remove `role="group"` and `aria-labelledby` from the inner div — the `<fieldset>/<legend>` pair already provides the grouping.

- **[MultipleSelectQuestion.tsx:59] (confidence: 75): Key uses index prefix `${index}-${option}` — fragile if options contain duplicates.** If a question has duplicate option text (unlikely but not validated), two options would share the same key. More importantly, the index prefix makes keys unstable across re-renders if the options array is ever reordered (e.g., shuffle). Fix: Use a stable unique identifier if available from the Question type, or at minimum use just `option` if duplicates are validated against at the schema level (which they currently are not).

- **[scoring.test.ts] (confidence: 70): No unit test for the PCM clamping edge case specified in AC5.** AC5 explicitly states: "When I select 1 correct and 2 incorrect, my raw score is (1 - 2) / 3 = -0.33, clamped to 0." The test quiz uses 2 correct answers (A, C) with 2 points. The "completely wrong selection" test (`['B', 'D']`) tests 0 correct / 2 incorrect, which clamps naturally. But there is no test for the mixed case where `correct - incorrect < 0` but `correct > 0` — the specific scenario the AC calls out. Fix: Add a test case like `{ q1: ['A', 'B', 'D'] }` (1 correct, 2 incorrect) and assert score is 0 (PCM: (1-2)/2 = -0.5, clamped to 0).

#### Nits

- **Nit** [MultipleSelectQuestion.tsx:73] (confidence: 65): `aria-label={option}` on the Checkbox is redundant when the checkbox is already wrapped in a `<label>` with visible text. The label association via wrapping provides the accessible name. The explicit `aria-label` creates a duplicate announcement in some screen readers. Consider removing it.

- **Nit** [LessonPlayer.tsx, CourseCard.tsx, story-e07-s07.spec.ts, story-e13-s03.spec.ts] (confidence: 60): The diff includes formatting-only changes to files unrelated to this story (collapsed Button JSX, trailing comma removal, type assertion reformatting). These are harmless but add noise to the PR diff. Consider separating formatting commits from feature work.

### Recommendations

1. **First: Implement AC6 feedback display** (Blocker) — this is the largest gap. The component needs review-mode rendering with per-option indicators.
2. **Second: Fix the accessibility hint association** (High) — move "Select all that apply" inside the legend or add aria-describedby.
3. **Third: Add runtime guard on correctAnswer type** (High) — defensive coding against data corruption.
4. **Fourth: Fix userAnswer coercion for empty multiple-select** (High) — type consistency for downstream consumers.
5. **Fifth: Add the PCM clamping unit test** (Medium) — covers the explicit AC5 scenario.
6. **Sixth: Address useMemo misuse** (Blocker as pattern, but low runtime risk) — replace with useEffect or render-time check. Consider fixing across all three question components.
