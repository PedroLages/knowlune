### MEDIUM — Conflicting ARIA roles on TutorModeChips
**File:** `src/app/components/tutor/TutorModeChips.tsx:31-34`

The component uses both `role="radio"` with `aria-checked` AND `aria-pressed` on the same element. These are semantically conflicting — `aria-pressed` is for toggle buttons, `aria-checked` is for radio buttons. Pick one pattern:
- If using `role="radio"`, use `aria-checked` only, and wrap in `role="radiogroup"` (currently using `role="group"`)
- If using toggle buttons, drop `role="radio"` and `aria-checked`, keep `aria-pressed`

**Recommendation:** Use `role="radiogroup"` on the container and `role="radio"` + `aria-checked` on buttons. Remove `aria-pressed`.

### LOW — VALID_SHORT regex overly broad
**File:** `src/ai/tutor/hintLadder.ts:32`

`[a-z]{1,5}` matches any 1-5 letter word. Not currently a bug because IMPLICIT_FRUSTRATION check runs first (e.g., "help" is caught before reaching VALID_SHORT). But it's fragile — if check order changes or new frustration keywords < 6 chars are added, they'd be silently suppressed. Consider replacing with explicit allowlist: `/^(yes|no|ok|okay|true|false|yep|nope|yeah|nah|\d+|[A-Z]{1,5})$/i` where uppercase-only branch handles acronym answers like "TCP".

### LOW — No de-escalation path in hint ladder
**File:** `src/ai/tutor/hintLadder.ts:71-101`

The hint ladder only escalates, never de-escalates. If a student gives a correct answer after frustration, the hint level stays high. Consider resetting or decrementing on long, confident messages. This is acceptable for MVP but worth noting for future iteration.
