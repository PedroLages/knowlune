## Findings

### [High] Double-counting of in-progress quizzes that are already completed
**File:** `src/lib/analytics.ts:136-137` | **Confidence:** 92

`inProgressQuizIds` (or `currentProgress.quizId`) from localStorage is added to `startedCount` without checking overlap with `completedQuizIds`. If a learner completes quiz `q1` and the Zustand store still has `currentProgress` referencing `q1` (until `clearQuiz()` is called), the quiz appears in both `completedQuizIds` and `inProgressCount`. This inflates `startedCount` and deflates the completion rate.

**Fix:** Filter in-progress quiz IDs against `completedQuizIds` before counting:
```typescript
const trulyInProgress = inProgressIds.filter(id => !completedQuizIds.has(id))
inProgressCount = trulyInProgress.length
```

### [High] `inProgressQuizIds` does not exist in the Zustand store
**File:** `src/lib/analytics.ts:136` | **Confidence:** 85

`useQuizStore` only persists `currentProgress` and `currentQuiz` to localStorage. There is no `inProgressQuizIds` field in the `QuizState` interface. The `parsed?.state?.inProgressQuizIds` branch will always be `undefined`, and the unit test for it (`'uses inProgressQuizIds array if present in localStorage'`) passes only because it manually sets localStorage — it does not reflect real application state.

**Fix:** Remove the `inProgressQuizIds` branch entirely, or add an explicit code comment explaining this is forward-compatible with a future multi-quiz store shape.

---

### [High] `beforeEach` is at module scope, not inside the `describe` block
**File:** `src/lib/__tests__/analytics.test.ts:279-282` | **Confidence:** 90

The `beforeEach` clearing `db.quizAttempts` and `localStorage` is defined between the two `describe` blocks, causing it to run before every test in the entire file — including the `analyzeTopicPerformance` tests. This is a test isolation concern that will confuse future maintainers.

**Fix:** Move the `beforeEach` inside the `describe('calculateCompletionRate')` block.

---

### [Medium] `addInitScript` called after initial navigation may not apply
**File:** `tests/e2e/story-e17-s01.spec.ts:38-48` | **Confidence:** 78

`page.addInitScript()` is called after `page.goto('/')` has already navigated. If `goToReports` ever uses client-side routing rather than a fresh page load, localStorage will not be seeded and the test would fail silently (or pass for wrong reasons).

**Fix:** Move `addInitScript` before the first `page.goto('/')` call.

---

### [Medium] Full table scan in `calculateCompletionRate`
**File:** `src/lib/analytics.ts:126` | **Confidence:** 75

`db.quizAttempts.toArray()` loads all quiz attempt records into memory just to extract unique `quizId` values. Performance degrades linearly with quiz history size.

**Fix:** Use `db.quizAttempts.orderBy('quizId').uniqueKeys()` to get distinct IDs directly from the index.

---

### [Medium] Empty-state test can silently pass without asserting anything
**File:** `tests/e2e/story-e17-s01.spec.ts:15-26` | **Confidence:** 75

All assertions are inside an `if (!isPageEmpty)` guard. When the Reports page is in its overall empty state (no activity), the inner `expect` calls are never reached. The "No quizzes started yet" text could be absent and the test would still pass.

**Fix:** Seed minimal non-quiz activity (e.g., via `seedStudySessions`) to prevent the page-level empty state, then unconditionally assert the quiz card empty text.

---

### [Medium] Coupling analytics to raw localStorage key
**File:** `src/lib/analytics.ts:132` | **Confidence:** 72

The magic string `'levelup-quiz-store'` in analytics code tightly couples it to the Zustand persist middleware's storage key. If the key changes, `calculateCompletionRate` silently returns 0 in-progress quizzes.

**Fix:** Extract to a shared constant, or expose an `isQuizInProgress()` selector from `useQuizStore`.

---

### [Nit] Long single-line test data objects
**File:** `tests/e2e/story-e17-s01.spec.ts:33-35` | **Confidence:** 65

E2E attempt objects are ~170-char single lines. Consider the `makeAttempt` factory or multi-line formatting.

---

### [Nit] Two `useEffect` hooks could be merged with `Promise.all`
**File:** `src/app/pages/Reports.tsx:85-95` | **Confidence:** 60

Minor maintainability improvement: merge both data-fetching effects into one using `Promise.all`.

## Recommendations

1. Fix double-counting (High #1) — correctness bug that produces wrong percentages in real usage
2. Remove or document phantom `inProgressQuizIds` (High #2) — misleading test
3. Move `beforeEach` into `describe` block (High #3) — test hygiene
4. Verify `addInitScript` timing in E2E (Medium) — fragile test ordering
5. Address empty-state test gap (Medium) — AC5 is not reliably asserted
