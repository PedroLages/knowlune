## Findings

### MEDIUM: `generateSRSSummaryEvents` — SRS events use local time, not user timezone

**File:** `src/lib/icalFeedGenerator.ts:205`

```typescript
const dtstart = new Date(year, month - 1, day, safeHours, safeMinutes, 0)
```

`new Date(year, month-1, day, hours, minutes)` creates a date in the **JavaScript runtime's local timezone**, not in the user's specified `timezone`. The `ical-generator` library's `timezone` option on the calendar affects how dates are *formatted* in the output, but if the JS Date object is created in local time, the DTSTART will be offset incorrectly for users in non-local timezones.

**Fix:** Use UTC construction and let `ical-generator` handle timezone conversion, or pass the timezone-aware time string:

```typescript
// Option A: Use UTC and let ical-generator handle timezone
const dtstart = new Date(Date.UTC(year, month - 1, day, safeHours, safeMinutes, 0))

// Option B: Use ical-generator's ICalDateTimeValue with explicit timezone
```

**Impact:** Calendar events will appear at wrong times for users in non-local timezones. Since the calendar feed is a key feature, this affects timezone-aware users.

**Effort:** ~15 minutes
