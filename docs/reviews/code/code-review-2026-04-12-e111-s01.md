## Code Review: E111-S01 — Audio Clips

### What Works Well

1. **Optimistic updates with rollback** — The Zustand store consistently applies optimistic state changes and rolls back on DB failure across all CRUD operations. This is well-structured and user-friendly.

2. **Solid accessibility** — 44px touch targets, ARIA labels on all interactive elements, semantic `<ul role="list">` with sortable items, keyboard-navigable inline editing with Enter/Escape.

3. **Dexie transaction for reorder** — `reorderClips` wraps multiple updates in `db.transaction('rw', ...)`, ensuring atomicity. The fallback re-reads from DB on failure rather than guessing state.

### Findings

#### High Priority

- **[Correctness] `src/stores/useAudioClipStore.ts:39-46` (confidence: 85)**: `loadClips` has no error handling. If the Dexie query fails (e.g., schema not yet migrated, corrupted DB), the promise rejects with an unhandled error and `isLoaded` stays `false` forever — no user feedback, no retry path. Why: Users see a permanent "Loading clips..." state with no way to recover. Fix: Wrap in try/catch, show `toast.error`, and optionally set an error state. ~5 min.

- **[Correctness] `src/app/components/audiobook/AudiobookRenderer.tsx:318-321` (confidence: 80)**: `handlePlayClip` uses `setTimeout(() => { seekTo(startTime); play() }, 100)` after `loadChapter`. This is a race condition — if chapter loading takes longer than 100ms, seekTo runs before audio is ready. Also, the timeout is not cleaned up on unmount. Why: Clip playback may start at wrong position or fail silently. Fix: Use the audio element's `canplay` event or the existing loading state to gate the seek, and store the timeout ID for cleanup. ~15 min.

- **[Silent Failure] `src/app/components/audiobook/AudiobookRenderer.tsx:328-337` (confidence: 78)**: The clip-scoped auto-pause effect uses `currentTime >= activeClipEnd.endTime - 0.1` tolerance, but if playback is paused/resumed manually or the user seeks past the end, `activeClipEnd` is never cleared. This means a stale clip boundary persists and could auto-pause the user at an unexpected time during normal playback. Why: Confusing UX — user resumes playback and gets immediately paused. Fix: Clear `activeClipEnd` when the user manually interacts with play/pause/seek controls. ~10 min.

#### Medium

- **[Correctness] `src/app/components/audiobook/ClipButton.tsx:88` (confidence: 72)**: `[AI Smell]` String interpolation for conditional className: `` `size-5 ${isRecording ? 'text-destructive' : ''}` `` — should use `cn()` for class merging consistency. Affects: ClipButton only. Fix: `cn('size-5', isRecording && 'text-destructive')`. ~1 min.

- **[Correctness] `src/app/components/audiobook/ClipListPanel.tsx:273-283` (confidence: 70)**: Two separate `useEffect` hooks handle clip loading — one for `open` changes, one for `bookId` changes while open. The second effect is redundant because `loadClips` already has a guard (`isLoaded && loadedBookId === bookId`), and the first effect already depends on `[open, bookId, loadClips]`. The second effect fires unnecessarily. Fix: Remove the second `useEffect` (lines 279-283) — the first already covers the bookId change case. ~2 min.

#### Nits

- **Nit** `src/app/components/audiobook/AudiobookRenderer.tsx:498` (confidence: 65): The `Scissors` icon import on line 13 is imported but only used for the clips panel button — `ClipButton` imports it separately. Minor, no action needed.

### Recommendations

1. **First**: Add error handling to `loadClips` — simplest fix, highest user impact.
2. **Second**: Fix the `handlePlayClip` setTimeout race condition — use event-driven approach.
3. **Third**: Clear `activeClipEnd` on manual user interactions to prevent stale auto-pause.
