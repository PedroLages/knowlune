## Code Review: E111-S01 — Audio Clips

### What Works Well

1. **Optimistic updates with rollback** -- The Zustand store consistently applies optimistic state changes and rolls back on DB failure across all CRUD operations. This is well-structured and user-friendly.

2. **Solid accessibility** -- 44px touch targets, ARIA labels on all interactive elements, semantic `<ul role="list">` with sortable items, keyboard-navigable inline editing with Enter/Escape.

3. **Dexie transaction for reorder** -- `reorderClips` wraps multiple updates in `db.transaction('rw', ...)`, ensuring atomicity. The fallback re-reads from DB on failure rather than guessing state.

### Findings

#### High Priority

- **[Correctness] `src/stores/useAudioClipStore.ts:39-46` (confidence: 90)**: `loadClips` has no error handling. If the Dexie query fails (e.g., schema not yet migrated, corrupted DB), the promise rejects with an unhandled error and `isLoaded` stays `false` forever -- no user feedback, no retry path. Why: Users see a permanent "Loading clips..." state with no way to recover. Fix: Wrap in try/catch, show `toast.error`, and set `isLoaded: true` so the empty state message appears. ~5 min.

- **[Correctness] `src/app/components/audiobook/AudiobookRenderer.tsx:316-320` (confidence: 82)**: `handlePlayClip` uses `setTimeout(() => { seekTo(startTime); play() }, 100)` without cleanup. If the component unmounts during the 100ms window (e.g., user navigates away), `seekTo` and `play` execute on stale references. Why: Potential console errors or unexpected audio playback after navigation. Fix: Store the timeout ID in a ref and clear it in a cleanup effect. ~5 min.

- **[Silent Failure] `src/app/components/audiobook/AudiobookRenderer.tsx:328-337` (confidence: 78)**: The clip-scoped auto-pause effect uses `currentTime >= activeClipEnd.endTime - 0.1` tolerance, but `activeClipEnd` is never cleared when the user manually interacts with play/pause/seek. If a user pauses during clip playback, seeks elsewhere, then resumes, the stale boundary auto-pauses them unexpectedly. Why: Confusing UX -- user resumes playback and gets immediately paused at an old clip boundary. Fix: Clear `activeClipEnd` when the user manually presses play/pause or seeks. ~10 min.

#### Medium

- **[Accessibility] `src/app/components/audiobook/ClipListPanel.tsx:103-116` (confidence: 75)**: `SortableClipItem` renders `<div role="listitem">` but it is already wrapped in `<li>` (line 353). This creates duplicate semantics -- assistive technology may announce "list item, list item". Fix: Remove `role="listitem"` from the inner div since the parent `<li>` already provides the semantics. ~1 min.

- **[Correctness] `src/app/components/audiobook/ClipListPanel.tsx:279-283` (confidence: 72)**: Second `useEffect` for bookId change detection is redundant -- the first effect (line 273) already depends on `[open, bookId, loadClips]` and `loadClips` has a same-book guard. The second effect fires unnecessarily on every `loadedBookId` change. Fix: Remove the second `useEffect`. ~2 min.

#### Nits

- **Nit** `src/app/components/audiobook/ClipButton.tsx:88` (confidence: 65): String interpolation for className `` `size-5 ${isRecording ? 'text-destructive' : ''}` `` should use `cn()` per project convention. Fix: `cn('size-5', isRecording && 'text-destructive')`. ~1 min.

### Recommendations

1. **First**: Add error handling to `loadClips` -- simplest fix, highest user impact
2. **Second**: Clean up the `setTimeout` in `handlePlayClip` on unmount
3. **Third**: Clear `activeClipEnd` on manual user interactions to prevent stale auto-pause
