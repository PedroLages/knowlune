## Story-Related Issues

### MEDIUM

1. **Orphaned JSDoc comment for `sanitizeAIRequestPayload`** — `src/lib/aiConfiguration.ts:333-345`
   The JSDoc block for `sanitizeAIRequestPayload` (lines 333-345) is now separated from its function (line 367) by the new `getOllamaServerUrl` and `isOllamaDirectConnection` functions inserted between them. The JSDoc attaches to `getOllamaServerUrl` instead of `sanitizeAIRequestPayload`.

2. **SSRF potential in Ollama proxy endpoint** — `server/index.ts:69-114`
   The `/api/ai/ollama` endpoint accepts a user-provided `ollamaServerUrl` and forwards requests to it. While this is by design for Ollama integration, there is no allowlist or validation beyond URL format. An attacker with access to the local proxy could target internal services. Consider adding a private IP range allowlist or at minimum documenting the trust boundary.

3. **Missing `// silent-catch-ok` on server-side catch block** — `server/index.ts:103`
   The Ollama endpoint catch block properly logs and returns errors to the client, but triggers the ESLint `error-handling/no-silent-catch` warning. Add `// silent-catch-ok — server-side: error logged + returned as HTTP response` to suppress the warning, matching the pattern that should also be applied to the pre-existing handlers.

### LOW

4. **Semantic mismatch: `validateApiKey` validates URL for Ollama** — `src/lib/aiConfiguration.ts:167`
   The `AIProvider.validateApiKey` method is reused for Ollama URL validation. The parameter name `key` in the signature is misleading when it's actually a URL. This is a minor naming concern since the `usesServerUrl` flag disambiguates at runtime, but could confuse future developers.

5. **OllamaLLMClient exists but story notes say "No custom client needed"** — `src/ai/llm/ollama-client.ts`
   The implementation notes in the story file (line 65) state "No custom OllamaLLMClient needed" and suggest using `createOpenAI` from `@ai-sdk/openai`. However, a full custom `OllamaLLMClient` was implemented. The server-side proxy (`server/providers.ts:65-70`) does use the recommended `createOpenAI` approach. The custom client is used for direct-connection mode, which is a reasonable architectural choice, but the story notes should be updated to reflect the actual implementation.

6. **Duplicate finish-reason yield in streaming** — `src/ai/llm/ollama-client.ts:115-132`
   When a chunk has `delta.content` AND `finish_reason === 'stop'`, the code yields the content chunk (line 116-120), then also yields an empty `{content: '', finishReason: 'stop'}` chunk (line 130-131). The first yield already includes the finishReason from line 118, so the consumer sees the stop signal twice. This is harmless but slightly wasteful.

### NIT

7. **`stream: true` in request body is redundant for proxy mode** — `src/ai/llm/ollama-client.ts:65`
   In proxy mode, the Express handler always uses `streamText()` regardless of the `stream` field. The field is only meaningful in direct mode (OpenAI-compat API). Not a bug, just noise in the proxy request body.
