## Code Review: E13-S05 — Randomize Question Order with Fisher-Yates Shuffle

**Re-review** after fixes applied. Previous blockers and high-priority items (readonly param, empty questions guard, deterministic Math.random mock, distribution test tolerance, .sort() mutation, seedQuizzes helper, data-testid) have been addressed.

### What Works Well

1. **Clean extraction**: Moving the inline `shuffleArray` to a reusable `src/lib/shuffle.ts` with generics and `readonly` parameter is exactly the right refactor. The function is pure, tested, and reusable.

2. **Empty quiz guard**: The new `quiz.questions.length === 0` guard in `startQuiz` (line 56-64) prevents downstream crashes from empty `questionOrder` arrays. Good defensive programming.

3. **Deterministic E2E approach**: Using `addInitScript` to mock `Math.random` with known sequences is the correct pattern for testing randomization in Playwright -- no flakiness from actual randomness.

#### High Priority

- **[tests/e2e/story-e13-s05.spec.ts:64-65] (confidence: 95)**: Incorrect shuffle result comments. The comments document expected shuffle outputs as `[5,2,4,3,1]` and `[4,3,1,5,2]` but the actual Fisher-Yates results are `[3,2,5,4,1]` and `[4,1,5,2,3]` respectively.

  Manual trace for Sequence A `[0.1, 0.9, 0.3, 0.7]` on `[q1,q2,q3,q4,q5]`:
  - i=4: j=floor(0.1*5)=0, swap(4,0) -> `[q5,q2,q3,q4,q1]`
  - i=3: j=floor(0.9*4)=3, swap(3,3) -> no-op
  - i=2: j=floor(0.3*3)=0, swap(2,0) -> `[q3,q2,q5,q4,q1]`
  - i=1: j=floor(0.7*2)=1, swap(1,1) -> no-op
  - Result: `[3,2,5,4,1]` (comment says `[5,2,4,3,1]`)

  Sequence B `[0.5, 0.4, 0.8, 0.2]`:
  - i=4: j=floor(0.5*5)=2, swap(4,2) -> `[q1,q2,q5,q4,q3]`
  - i=3: j=floor(0.4*4)=1, swap(3,1) -> `[q1,q4,q5,q2,q3]`
  - i=2: j=floor(0.8*3)=2, swap(2,2) -> no-op
  - i=1: j=floor(0.2*2)=0, swap(1,0) -> `[q4,q1,q5,q2,q3]`
  - Result: `[4,1,5,2,3]` (comment says `[4,3,1,5,2]`)

  Why: Tests still pass because they only assert `not.toEqual(originalOrder)` and `not.toEqual(firstOrder)`, which hold for the real results. But incorrect comments are worse than no comments -- a future developer debugging a shuffle change will trust these derivations and waste time. Fix: Update both comment lines to the correct results, or remove the specific expected-output comments and keep only the swap trace.

#### Medium

- **[tests/e2e/story-e13-s05.spec.ts:75-98] (confidence: 78)**: `mockMathRandom` has a confusing and unnecessary closure structure with parameter shadowing. The outer function accepts `sequences: number[][]` (line 75) and returns an inner function that also accepts `sequences: number[][]` (line 76). The outer `sequences` parameter is completely dead code -- only the inner parameter is used (populated by `addInitScript`'s second argument). This works because Playwright serializes the returned function and passes the second arg as the parameter, but the wrapper adds no value and the shadowing is misleading. Fix: Remove the outer wrapper entirely and pass the inner function directly:

  ```typescript
  function randomMockScript(sequences: number[][]) {
    let seqIndex = 0
    let callIndex = 0
    const originalRandom = Math.random.bind(Math)
    Math.random = () => { /* ... */ }
  }

  // Usage:
  await page.addInitScript(randomMockScript, [RANDOM_SEQUENCE_A])
  ```

- **[tests/e2e/story-e13-s05.spec.ts:143,164] (confidence: 72)**: The `addInitScript` call pattern obscures what arguments reach the browser. On line 143, `mockMathRandom([RANDOM_SEQUENCE_A])` is evaluated in Node.js (returning the inner function), and `[RANDOM_SEQUENCE_A]` is passed separately as the serialized argument. A reader cannot tell at a glance which `sequences` value is actually used in the browser. This is a direct consequence of the wrapper structure above -- fixing the wrapper fixes this too.

- **[tests/e2e/story-e13-s05.spec.ts:159,201] (confidence: 75)**: `not.toEqual` assertions verify the mock works but don't verify the *exact* expected order. Since the mock is deterministic, the test could assert `toEqual(['Question 3', 'Question 2', 'Question 5', 'Question 4', 'Question 1'])` for AC1 and the specific expected orders for AC2. This would catch regressions where the shuffle algorithm changes (e.g., iteration direction) but the mock still produces a different-from-original order. Why: A broken shuffle that still produces non-original orders would pass the current test. Fix: Assert exact expected orders (after correcting the derivations above).

#### Nits

- **Nit** [tests/e2e/story-e13-s05.spec.ts:69-73] (confidence: 60): The JSDoc for `mockMathRandom` says it falls back "to 0.5" when sequences are exhausted, but the implementation falls back to `originalRandom()` (line 95). The "0.5" was likely from a previous iteration. Fix: Update the JSDoc to say "falls back to the original Math.random".

- **Nit** [src/stores/useQuizStore.ts:240-264] (confidence: 55): `onRehydrateStorage` validates `currentQuestionIndex` and `markedForReview` but not `questionOrder`. If a user has persisted state from before `questionOrder` was added, it could be `undefined`, leading to runtime errors when the quiz renders. This is a pre-existing issue (not introduced by this story) but worth noting for a future hardening pass. Fix: Add a guard like `if (!Array.isArray(state.currentProgress.questionOrder)) { state.currentProgress.questionOrder = state.currentQuiz.questions.map(q => q.id) }`.

- **Nit** [docs/implementation-artifacts/13-5-randomize-question-order-with-fisher-yates-shuffle.md:59] (confidence: 65): Task 3.3 says "distribution test over 10,000 runs" but the implementation uses 50,000 runs (`shuffle.test.ts:46`). The story doc is stale. Fix: Update to "50,000 runs with +/-8% tolerance".

3. **Assert exact shuffled orders** (Medium) -- deterministic mocks enable exact assertions; use them.
