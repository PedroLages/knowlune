## Code Review: E13-S05 — Randomize Question Order with Fisher-Yates Shuffle

### What Works Well

1. **Clean extraction pattern.** The inline `shuffleArray` was extracted to `src/lib/shuffle.ts` as a generic, immutable utility with zero behavioral change in the store. Import wiring is correct and the store tests continue to pass against the extracted function.

2. **Thorough unit test suite.** The distribution test (10K runs, position-frequency matrix) is a statistically rigorous way to verify uniformity. Edge cases (empty array, single element, generic types) are all covered.

3. **Correct Fisher-Yates implementation.** The algorithm iterates backward with `Math.floor(Math.random() * (i + 1))`, which is the canonical unbiased variant. Immutability via spread-copy is handled correctly.

### Findings

#### Blockers

None.

#### High Priority

- **[tests/e2e/story-e13-s05.spec.ts:118,154] (confidence: 92)**: E2E tests AC1 and AC2 assert `expect(order).not.toEqual(originalOrder)` and `expect(secondOrder).not.toEqual(firstOrder)`. With 5 questions, each assertion has a 1/120 (0.83%) probability of false failure. Across both tests that compounds to ~1.7%. In CI running across multiple projects/retries, this is a meaningful flakiness source. Why: Flaky tests erode trust in the test suite and cause spurious CI failures, wasting developer time. Fix: Mock `Math.random` via `page.addInitScript` to produce a known deterministic permutation for AC1 (verifying it differs from original), and a different known permutation for AC2 (verifying it differs from AC1's). This eliminates randomness from the assertion while still proving the shuffle code path executes. Alternatively, run the shuffle multiple times and assert that *at least one* ordering differs (reduces flakiness to `(1/120)^N`).

#### Medium

- **[src/lib/__tests__/shuffle.test.ts:9] (confidence: 78)**: `expect(result.sort()).toEqual([1, 2, 3, 4, 5])` calls `.sort()` on `result`, which mutates `result` in place. While the test works correctly because `result` is not used after this line, `.sort()` mutating the array under test is a code smell -- if someone adds an assertion after this line that references `result`, it will see the sorted version, not the shuffled version. Fix: Use `expect([...result].sort()).toEqual([1, 2, 3, 4, 5])` or `expect(result).toEqual(expect.arrayContaining([1, 2, 3, 4, 5]))` combined with a length check.

- **[src/lib/shuffle.ts:2] (confidence: 72)**: The parameter type `array: T[]` allows callers to pass mutable arrays. Adding `readonly` (`array: readonly T[]`) would make the function signature more permissive -- callers with `readonly` arrays could use it without casting. This is a minor type-safety improvement. Fix: `export function fisherYatesShuffle<T>(array: readonly T[]): T[]`.

- **[src/lib/__tests__/shuffle.test.ts:34-57] (confidence: 70)**: The distribution test uses a 5% tolerance on 10,000 runs. While this works in practice, the expected count per cell is 2000 and the standard deviation is approximately `sqrt(10000 * 0.2 * 0.8) = 40`, so a 3-sigma bound is 120/2000 = 6%. The 5% tolerance (100/2000) is approximately a 2.5-sigma bound, giving a ~1.2% false failure probability per cell, and with 25 cells, the overall false failure probability is non-trivial (~26% if independent, though they are not). Fix: Either increase runs to 50,000 (reduces variance by sqrt(5)) or widen tolerance to 8%.

#### Nits

- **Nit** [src/lib/shuffle.ts:6] (confidence: 60): The leading semicolon on `;[shuffled[i], shuffled[j]]` is necessary for ASI safety but unusual. This is inherited from the original code and is fine -- just noting it is a deliberate ASI guard, not an error.

- **Nit** [tests/e2e/story-e13-s05.spec.ts:105,123,159] (confidence: 55): The `as unknown as Record<string, unknown>` casts on quiz data suggest a type mismatch between the factory output and `seedIndexedDBStore`'s expected input. Consider updating the seed helper to accept `Quiz` (or a generic) to avoid double-casting.

### Recommendations

1. **Fix E2E flakiness first** (High) -- deterministic `Math.random` mock eliminates the only real risk in this PR.
2. **Widen distribution test tolerance or increase runs** (Medium) -- prevents spurious unit test failures in CI.
3. **Add `readonly` to parameter type** (Medium) -- low effort, improves API ergonomics.
4. **Fix `.sort()` mutation in test** (Medium) -- defensive coding in test prevents future bugs.
