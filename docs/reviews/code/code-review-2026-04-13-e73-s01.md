## Findings

### MEDIUM

1. **No guard for totalTokens < FIXED_TOTAL in budgetAllocator** — `allocateTokenBudget(400, 'socratic')` would produce `variableBudget = -150`, making `scale` negative and returning nonsensical negative slot values. Add a minimum budget guard.
   - File: `src/ai/prompts/budgetAllocator.ts:57`

2. **consumeTransitionContext is defined but never called** — `switchMode` writes a transition context string, but no consumer exists in the codebase. The LLM never receives the mode switch signal. Acceptable if consumption is planned for a later story, but should be documented with a TODO.
   - File: `src/stores/useTutorStore.ts:212-216`

3. **modeHistory grows unboundedly** — `switchMode` pushes to `modeHistory` array with no cap. Long sessions could accumulate hundreds of entries. Consider capping at ~50.
   - File: `src/stores/useTutorStore.ts:207`

### LOW

4. **prunePairs groups by array index, not role pattern** — If messages don't follow strict user/assistant alternation, pairs would be misaligned. The standard conversation pattern makes this unlikely but not impossible.
   - File: `src/ai/prompts/conversationPruner.ts:121`

5. **Duplicate prune-summary IDs** — `makePruneSummary` always uses `id: 'prune-summary'`. Multiple prune operations in the same session would create messages with identical IDs.
   - File: `src/ai/prompts/conversationPruner.ts:148`

### NITS

6. **Date.now() in makePruneSummary** — Non-deterministic timestamp for synthetic messages. Not a test issue (no test checks it), but a `0` sentinel would be more explicit.
   - File: `src/ai/prompts/conversationPruner.ts:149`

## Architecture Assessment

- **Registry pattern**: Clean, well-typed, frozen. Good use of `Object.freeze()` for immutability.
- **Budget allocator**: Pure function, deterministic (modulo the negative-budget edge case). Proportional scaling with remainder-to-response is a solid approach.
- **Conversation pruner**: Mode-aware strategy pattern is clean. Three strategies cover the mode families well.
- **Store changes**: `switchMode` properly separates from `setMode` (backward compatible). Transition context is a good bridge for future stories.
- **UI components**: TutorModeChips has proper radiogroup semantics, keyboard navigation, tooltip support, and transcript-gated disabling. TutorEmptyState is well-structured with mode-specific content.
