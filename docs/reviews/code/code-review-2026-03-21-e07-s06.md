#### High Priority

- **[src/stores/useCourseStore.ts:15-19] (confidence: 82)**: `loadCourses()` lost its idempotency guard. The original `if (get().isLoaded) return` prevented redundant IndexedDB queries. The new logic always calls `db.courses.toArray()` on every invocation, even when courses are already loaded. This means every component mount that triggers `loadCourses()` (Layout.tsx:185, main.tsx:41) now performs an unnecessary IndexedDB read in production. Why: Learners on lower-end devices will see avoidable async I/O on every navigation. The fix for the E2E race condition (IndexedDB populated after initial load) should preserve the production fast-path. Fix: Add the race-condition handling without removing the guard entirely:
  ```typescript
  loadCourses: async () => {
    if (get().isLoaded && get().courses.length > 0) return
    const courses = await db.courses.toArray()
    if (courses.length > 0 || !get().isLoaded) {
      set({ courses, isLoaded: true })
    }
  },
  ```
  This retains the early bailout when courses are already loaded AND non-empty, while still allowing a retry when the first load returned an empty array (the race condition scenario).

- **[Recurring] [src/stores/useCourseStore.ts:16] (confidence: 78)**: `db.courses.toArray()` has no error handling. If IndexedDB is unavailable or the query fails, the unhandled promise rejection propagates silently. This is a pre-existing issue amplified by the guard removal (now executes more frequently). Why: A corrupted IndexedDB or quota error would crash the app on every navigation. Fix: Wrap in try/catch with fallback:
  ```typescript
  loadCourses: async () => {
    if (get().isLoaded && get().courses.length > 0) return
    try {
      const courses = await db.courses.toArray()
      if (courses.length > 0 || !get().isLoaded) {
        set({ courses, isLoaded: true })
      }
    } catch (error) {
      console.error('[useCourseStore] Failed to load courses:', error)
      if (!get().isLoaded) set({ isLoaded: true })
    }
  },
  ```
