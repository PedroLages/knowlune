### High Priority

- **[src/stores/useCareerPathStore.ts:46-51] (confidence: 90)**: **TOCTOU race on seed logic.** `loadPaths` checks `count()`, then `bulkAdd()` in two separate operations without a transaction. When `CareerPaths.tsx` and `CareerPathDetail.tsx` both mount simultaneously (e.g., navigating from list to detail quickly, or React StrictMode double-invocation), two concurrent calls can both observe `count === 0` and both attempt `bulkAdd()`. This will throw a `ConstraintError` on duplicate primary keys (Dexie's `bulkAdd` rejects duplicates), causing the second call to fall into the catch block and show a "Failed to load career paths" toast to the learner on first app load. Why: Users see a spurious error toast the first time they visit Career Paths. Fix: Wrap the check+seed in a Dexie transaction, or use `bulkPut()` instead of `bulkAdd()` (idempotent), or guard `loadPaths` with an in-flight promise to prevent concurrent execution.

- **[src/stores/useCareerPathStore.ts:119,129,151] (confidence: 85)**: **Stale closure on `enrollments` during `set()`.** Both `enrollInPath` and `dropPath` destructure `const { enrollments } = get()` at the start, then use that snapshot in the `set()` call after an `await`. If another operation modifies `enrollments` during the async gap (e.g., rapid enroll-then-drop, or `loadPaths` running concurrently), the `set()` overwrites the intermediate state with the stale snapshot plus the local change, silently dropping the other mutation. Why: Rapid button clicks can cause enrollment state corruption. Fix: Use `set()` with a callback (`set(state => ({ enrollments: state.enrollments.map(...) }))`) to read the latest state at write time, or re-read `get().enrollments` after the `await`.

- **[src/stores/useCareerPathStore.ts:66-70] (confidence: 82)**: **[Recurring] Fire-and-forget `refreshCourseCompletion` without user-visible error handling.** The `.catch()` logs to console but does not surface the error. If course completion data fails to load, progress bars show 0% indefinitely with no indication that data is stale. The user sees "0/3 courses complete" when they may have completed some. Why: Silent data staleness misleads learners about their progress. Fix: Either retry with exponential backoff (like `persistWithRetry`), or surface a warning via `toast.warning('Progress data may be outdated')`.

- **[src/stores/useCareerPathStore.ts:113-115] (confidence: 80)**: **[Recurring] Silent catch on `refreshCourseCompletion` internal error.** The catch block at line 113 logs but does not call `toast.error()` or set an error state. Combined with the fire-and-forget call at line 66-70, this creates a double-silent-failure: the caller catches silently, and the function itself catches silently. Fix: At minimum set `error` state so the UI can render a warning, or propagate the error to the caller.

### Medium

- **[src/stores/useCareerPathStore.ts:46] (confidence: 75)**: **No `loadPaths` deduplication guard.** Both page components call `loadPaths()` in their `useEffect`. Because Zustand stores are singletons, navigating between the list and detail page triggers two sequential loads. While functionally correct (the second load just re-reads the DB), it's wasteful -- each call does a `db.careerPaths.count()` + `db.careerPaths.toArray()` + `db.pathEnrollments.where(...).toArray()` + N `contentProgress` queries. Fix: Add an early return if `isLoaded` is already `true`, or cache a loading promise to deduplicate concurrent calls.

- **[src/app/pages/CareerPaths.tsx:129] (confidence: 78)**: **`sortedPaths` is not sorted.** `useMemo(() => [...paths], [paths])` creates a shallow copy but applies no sort order. The variable name `sortedPaths` is misleading -- paths render in whatever order Dexie returns them. Why: The learner sees paths in insertion order rather than any intentional order (alphabetical, enrollment status, etc.). Fix: Either apply an actual sort (e.g., by title, or enrolled-first) or rename to `pathsList` to match the behavior.

- **[src/app/pages/CareerPathDetail.tsx:143-144] (confidence: 72)**: **Course display name derived from ID with naive regex.** `courseId.replace(/-/g, ' ').replace(/\b\w/g, c => c.toUpperCase())` converts `6mx` to `6mx` (no hyphens to replace) and `nci-access` to `Nci Access`. These are not user-friendly names. The actual course name exists in the `courses` table in IndexedDB, but is never fetched. Why: Learners see cryptic identifiers instead of real course names. Fix: Look up the course name from the `courses` or `importedCourses` table, or add a `courseName` field to the seed data stages.

- **[tests/e2e/career-paths.spec.ts:20-21] (confidence: 70)**: **`networkidle` is a deprecated/unreliable Playwright wait strategy.** Used 20 times across the spec. Playwright's `networkidle` waits for 500ms of no network activity, which is fragile (IndexedDB operations are not network requests, so `networkidle` resolves immediately before data is rendered). It works here by coincidence because the app renders quickly, but it's not semantically correct. Fix: Replace with `await expect(locator).toBeVisible()` for the element you actually need, which uses Playwright's auto-retry and is deterministic.

- **[src/app/pages/CareerPathDetail.tsx:191] (confidence: 72)**: **Completed courses cannot be navigated to.** `if (isLocked || isCompleted) return content` renders a non-interactive div for completed courses. A learner who completed a course may want to revisit it for reference, but cannot click through. This blocks re-access to completed material. Fix: Wrap completed courses in a `<Link>` as well (with a visual distinction from in-progress courses).

### Nits

- **Nit** [src/app/pages/CareerPathDetail.tsx:184] (confidence: 65): Badge uses `text-[10px]` arbitrary value. Consider using `text-2xs` or defining a token if this size is reused elsewhere.

- **Nit** [src/app/pages/CareerPathDetail.tsx:27-32, src/app/pages/CareerPaths.tsx:25-30] (confidence: 60): `ICON_MAP` is duplicated identically between both page files. Extract to a shared `src/data/careerPaths/icons.ts` to DRY up the mapping.

- **Nit** [tests/e2e/career-paths.spec.ts:184] (confidence: 55): `void href // used above` is a dead-code workaround for an unused variable. The `href` was only needed for the `getAttribute` call, which was apparently refactored. Remove the variable entirely.

- **Nit** [src/stores/useCareerPathStore.ts:124,165] (confidence: 60): `'active' as PathEnrollmentStatus` is an unnecessary type assertion. The string literal `'active'` already matches the union type. The cast obscures a genuine type error if the value were changed to something invalid.

## Recommendations

Fix in this order:

1. **[HIGH] Fix TOCTOU seed race** -- switch `bulkAdd` to `bulkPut` in `loadPaths` (one-line fix, prevents first-load error toasts).
2. **[HIGH] Fix stale closure in `enrollInPath`/`dropPath`** -- use `set(state => ...)` callback pattern instead of captured `enrollments`.
3. **[HIGH] Surface `refreshCourseCompletion` failures** -- add `toast.warning()` or error state so learners know when progress data may be stale.
4. **[MEDIUM] Add `loadPaths` deduplication** -- early return if `isLoaded`, or cache the in-flight promise.
5. **[MEDIUM] Look up real course names** -- fetch from `courses` table instead of deriving from ID.
6. **[MEDIUM] Rename or sort `sortedPaths`** -- apply intentional ordering or fix the misleading name.
