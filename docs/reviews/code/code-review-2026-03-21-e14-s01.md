## Code Review: E14-S01 — Display True/False Questions

### What Works Well

1. **Clean component architecture.** TrueFalseQuestion mirrors MultipleChoiceQuestion's pattern precisely — same prop interface, same fieldset/legend/RadioGroup structure, same design token usage. This makes the quiz subsystem predictable and easy to onboard.

2. **Data-driven option rendering.** Options come from `question.options` rather than being hardcoded as `['True', 'False']`, enabling future i18n and respecting the Zod schema validation. The `console.warn` for unexpected option counts is a sensible defensive measure.

3. **Solid E2E coverage.** All five ACs have dedicated tests. The scoring test (AC3) exercises the full submit flow end-to-end, and the responsive test (AC4) validates actual bounding boxes rather than relying on class assertions alone.

#### High Priority

- **[tests/e2e/story-e14-s01.spec.ts:211] (confidence: 78)**: CSS selector `#${ariaLabelledBy}` is fragile with React `useId()`-generated IDs. React's `useId()` produces IDs containing colons (e.g., `:r0:`), which are special characters in CSS selectors. While Playwright's `page.locator()` appears to handle this today, raw `#:r0:` is invalid CSS per the W3C spec. A Playwright or React version bump could break this.
  Why: Flaky E2E test failure blocks CI/CD and wastes developer time debugging framework internals.
  Fix: Use attribute selector instead: `page.locator(`[id="${CSS.escape(ariaLabelledBy!)}"]`)` or simply `page.locator(`[id="${ariaLabelledBy}"]`)` — attribute selectors treat values as strings, no escaping needed.

#### Medium

- **[src/app/components/quiz/questions/TrueFalseQuestion.tsx:61] (confidence: 82)**: `hover:bg-accent` applied unconditionally on unselected options, even in review mode. When `mode !== 'active'`, the component applies `cursor-default opacity-60` but does not suppress the hover background change. Users in review mode see visual feedback that suggests interactivity, contradicting the disabled state.
  Fix: Gate the hover class on `isActive`:
  ```tsx
  isSelected
    ? 'border-brand bg-brand-soft'
    : cn('border-border bg-card', isActive && 'hover:bg-accent')
  ```
  Note: Same issue exists in `MultipleChoiceQuestion.tsx:65` — fix both for consistency.

- **[src/app/components/quiz/questions/TrueFalseQuestion.tsx:9-13 and MultipleChoiceQuestion.tsx:9-13] (confidence: 75)**: `REMARK_PLUGINS` and `MARKDOWN_COMPONENTS` are duplicated identically across both question components. Epic 14 adds at least 3 more question types (multiple-select, fill-in-blank, rich-text), so this duplication will multiply.
  Fix: Extract to a shared `src/app/components/quiz/questions/markdown-config.ts` module:
  ```tsx
  export const REMARK_PLUGINS = [remarkGfm]
  export const MARKDOWN_COMPONENTS = {
    p: ({ children }) => <span>{children}</span>,
  }
  ```

- **[tests/e2e/story-e14-s01.spec.ts:197-228] (confidence: 72)**: AC5 specifies "supports keyboard arrow navigation within the group" but the test only verifies Tab+focus — it does not test Arrow key navigation between True and False options. While Radix RadioGroup provides this natively, the AC explicitly calls it out, meaning the test does not fully verify the acceptance criterion.
  Fix: Add arrow key navigation assertion after the focus check:
  ```ts
  await firstRadio.focus()
  await page.keyboard.press('ArrowRight')  // or ArrowDown
  await expect(page.getByRole('radio').nth(1)).toBeFocused()
  ```

#### Nits

- **Nit** [src/app/components/quiz/questions/TrueFalseQuestion.tsx:28-31] (confidence: 60): `console.warn` is a side effect in the render body — it fires on every re-render when `options.length !== 2`. Per React's render purity guidelines, diagnostic side effects should live in `useEffect`. Same pattern exists in MultipleChoiceQuestion, so this is a pre-existing convention — but worth noting for future cleanup.

- **Nit** [src/app/components/quiz/QuestionDisplay.tsx:48] (confidence: 55): `const tfOnChange = (answer: string) => onChange(answer)` creates a new function reference each render. This is intentional for type narrowing (`string` to `string | string[]`), and matches the `mcOnChange` pattern — but `useCallback` would stabilize the reference if child components ever rely on referential equality.

- **Nit** [tests/e2e/story-e14-s01.spec.ts:116] (confidence: 50): `page.getByRole('radio').first()` assumes True is the first radio. This works because `options` is `['True', 'False']`, but a label-based selector like `page.getByRole('radio', { name: 'True' })` would be more self-documenting and resilient to option reordering. Note: Radix RadioGroupItem may not expose accessible names this way — verify feasibility.

### Recommendations

1. **Fix the hover state in review mode** (Medium) — Quick `cn()` conditional, fixes both TF and MC components.
2. **Use attribute selector in E2E AC5 test** (High) — One-line change, prevents future CSS selector breakage.
3. **Add arrow key navigation assertion to AC5** (Medium) — Completes AC coverage.
4. **Extract shared markdown config** (Medium) — Do this before E14-S02 to prevent 5-way duplication.
