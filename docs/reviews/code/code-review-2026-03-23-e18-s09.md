# Code Review: E18-S09 -- Configure Quiz Preferences in Settings

**Reviewer:** Adversarial Code Review Agent (Opus)
**Date:** 2026-03-23
**Branch:** `feature/e18-s09-configure-quiz-preferences-in-settings`

## Executive Summary

Clean, well-structured story. The localStorage abstraction in `quizPreferences.ts` is solid -- Zod validation on read, spread-copy defaults to avoid reference sharing, and a sensible merge-on-save pattern. The cross-tab sync via custom events is a nice touch. The main issues cluster around: (1) silent failure on save, (2) a type-safety gap between the preferences schema and the quiz system's `TimerAccommodation` type, (3) the `saveQuizPreferences` function accepting arbitrary `Partial<QuizPreferences>` without validation, and (4) a duplicated IndexedDB seeding helper in the E2E tests.

---

## What Works Well

1. **Zod validation on localStorage read** (`quizPreferences.ts:31`). Corrupted or tampered storage is handled gracefully with a fallback to defaults. The `safeParse` + spread-copy pattern is exactly right.

2. **Frozen feedback preference at mount** (`Quiz.tsx:110`). Using `useState(() => getQuizPreferences().showImmediateFeedback)` prevents mid-quiz UX surprises if the user changes settings in another tab. Thoughtful design.

3. **Comprehensive E2E coverage**. All four ACs are covered with settings-page and quiz-integration tests. The reload-and-verify pattern for persistence is solid.

## Findings

### High Priority

- **[Recurring] `src/lib/quizPreferences.ts:39-43` (confidence: 85)**: `saveQuizPreferences` has no error handling. If `localStorage.setItem` throws (quota exceeded, private browsing, SecurityError), the function silently fails -- the user sees the toast "Quiz preferences saved" (dispatched by the caller in `QuizPreferencesForm.tsx:48`) but data was not actually persisted. This is a **silent data loss** pattern. The learner changes their timer accommodation to 2x, sees the success toast, then starts a quiz and gets standard timing.

  Why: Learners with accessibility needs (timer accommodation) will believe their accommodation is active when it is not.

  Fix: Wrap `localStorage.setItem` in a try/catch. On failure, either (a) throw so the caller can show an error toast instead of success, or (b) return a `{ success: boolean }` result. At minimum:
  ```typescript
  export function saveQuizPreferences(patch: Partial<QuizPreferences>): QuizPreferences {
    const current = getQuizPreferences()
    const updated = { ...current, ...patch }
    try {
      localStorage.setItem(STORAGE_KEY, JSON.stringify(updated))
    } catch (err) {
      console.error('[quizPreferences] Failed to save:', err)
      throw err // Let caller handle (show error toast)
    }
    return updated
  }
  ```
  And in `QuizPreferencesForm.tsx`, wrap the `update()` body in try/catch to show an error toast on failure.

- **`src/lib/quizPreferences.ts:6-7` vs `src/types/quiz.ts:33` (confidence: 82)**: Type divergence between `QuizPreferencesSchema.timerAccommodation` (enum: `'standard' | '150%' | '200%'`) and `TimerAccommodationEnum` (enum: `'standard' | '150%' | '200%' | 'untimed'`). While the story deliberately excludes `'untimed'` from preferences, the `loadSavedAccommodation` function in `Quiz.tsx:76` returns `getQuizPreferences().timerAccommodation` as a `TimerAccommodation` -- this works today because the preferences type is a subset, but there is no compile-time guarantee. If someone adds a new option to one schema but not the other, it will silently break.

  Why: Future maintainers may add a new timer option to one location and miss the other, causing runtime mismatches.

  Fix: Derive the preferences timer type from the canonical `TimerAccommodationEnum` using `.exclude()`:
  ```typescript
  const PreferencesTimerEnum = TimerAccommodationEnum.exclude(['untimed'])
  ```
  This creates a compile-time link between the two schemas.

- **`src/app/components/settings/QuizPreferencesForm.tsx:81` (confidence: 80)**: Unsafe type assertion `value as QuizPreferences['timerAccommodation']`. The `onValueChange` callback from `RadioGroup` delivers a `string`, and this blindly casts it. While the UI only renders valid options, a programmatic or accessibility tool interaction could theoretically deliver an unexpected value. The Zod validation only happens on *read* from localStorage, not on *write*.

  Why: If an invalid value slips through, it gets persisted to localStorage and will be rejected on next read, reverting to defaults -- confusing for the user.

  Fix: Validate the incoming value before saving:
  ```typescript
  onValueChange={value => {
    const parsed = QuizPreferencesSchema.shape.timerAccommodation.safeParse(value)
    if (parsed.success) update({ timerAccommodation: parsed.data })
  }}
  ```

### Medium

- **[Recurring] `tests/e2e/quiz-preferences-settings.spec.ts:60-96` (confidence: 80)**: Manual IndexedDB seeding (`seedQuizData`) duplicates the shared helper at `tests/support/helpers/indexeddb-seed.ts:seedIndexedDBStore`. The logic is nearly identical (retry loop, `indexedDB.open`, store check). This is a known recurring anti-pattern.

  Fix: Replace with the shared helper:
  ```typescript
  import { seedIndexedDBStore } from '../../support/helpers/indexeddb-seed'
  // ...
  await seedIndexedDBStore(page, 'ElearningDB', 'quizzes', [quiz])
  ```

- **`src/stores/useQuizStore.ts:85` (confidence: 75)**: `getQuizPreferences()` is called synchronously inside the Zustand `startQuiz` action. This reads from localStorage on every quiz start. While not a performance issue today, it means the shuffle preference is read at `startQuiz` time but is NOT frozen like the feedback preference is in `Quiz.tsx:110`. If a user has two tabs open and changes shuffle preference in tab B while tab A is on the start screen, tab A will pick up the new preference when they click "Start Quiz". This is inconsistent with the feedback preference behavior (which is frozen at mount). Document whether this inconsistency is intentional or align the behaviors.

  Fix: Either freeze the shuffle preference at mount like feedback is, or document why it is acceptable for shuffle to be dynamic.

- **`src/lib/quizPreferences.ts:39-41` (confidence: 72)**: `saveQuizPreferences` accepts `Partial<QuizPreferences>` but does not validate the patch against the schema. A caller could pass `{ timerAccommodation: 'invalid' as any }` and it would be persisted. While `getQuizPreferences` validates on read, the corrupted value would sit in storage until then, and any direct `localStorage.getItem` read elsewhere would see garbage.

  Fix: Validate the merged result with `QuizPreferencesSchema.parse(updated)` before writing.

- **`src/app/components/settings/QuizPreferencesForm.tsx:71` (confidence: 70)**: The "Timer accommodation default" label uses a plain `<Label>` without `htmlFor`, so it is not programmatically associated with the `RadioGroup`. Screen readers will not announce the label when the radio group receives focus.

  Fix: Add an `id` to the `RadioGroup` and `htmlFor` on the `Label`, or use `aria-labelledby` on the `RadioGroup` pointing to the label element.

### Nits

- **Nit** `src/app/components/settings/QuizPreferencesForm.tsx:56` (confidence: 65): The icon uses `text-brand` on a `bg-brand-soft` background. Per the styling rules, the correct token for text on `bg-brand-soft` is `text-brand-soft-foreground` to ensure WCAG AA contrast in dark mode.

- **Nit** `src/lib/__tests__/quizPreferences.test.ts` (confidence: 60): No test for what happens when `localStorage.setItem` throws in `saveQuizPreferences`. Given the High Priority finding above, adding a test that mocks `localStorage.setItem` to throw would verify the error handling once it is implemented.

- **Nit** `tests/e2e/quiz-preferences-settings.spec.ts` (confidence: 55): No E2E test for the shuffle preference being applied during quiz start. AC3 mentions "quiz reads saved preferences and applies them as defaults" but the E2E only tests timer and feedback, not shuffle. While shuffle is tested at the unit level via `useQuizStore.test.ts`, an integration gap exists.

## Passed Checks

- **Security**: No XSS risk -- localStorage values are parsed through Zod, not injected into DOM. No `dangerouslySetInnerHTML`. No secrets in client code.
- **Architecture**: Clean separation -- `quizPreferences.ts` (data layer), `QuizPreferencesForm.tsx` (UI), `Quiz.tsx` (consumer). Follows existing project patterns.
- **Error handling on read**: `getQuizPreferences` handles invalid JSON, failed Zod validation, and localStorage access errors with graceful fallback to defaults.
- **Cross-tab sync**: Custom event + native `storage` event listener pattern is correct and clean.
- **Design tokens**: Uses `bg-brand-soft`, `border-brand`, `text-muted-foreground`, `bg-background`, `border-border` throughout. No hardcoded hex colors detected.
- **Touch targets**: `min-h-[44px]` on toggle rows ensures WCAG touch target compliance.
- **Import paths**: Consistent `@/` alias usage throughout.

Fix in this order:

1. **[High]** Add error handling to `saveQuizPreferences` and surface save failures to the user in `QuizPreferencesForm`
2. **[High]** Derive preferences timer enum from canonical `TimerAccommodationEnum` to prevent future drift
3. **[High]** Validate radio group value before casting in `QuizPreferencesForm`
4. **[Medium]** Replace duplicated `seedQuizData` with shared `seedIndexedDBStore` helper
5. **[Medium]** Add `aria-labelledby` to the timer accommodation RadioGroup
6. **[Medium]** Validate merged preferences against schema before writing to localStorage
