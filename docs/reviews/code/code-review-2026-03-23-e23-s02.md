#### Medium

- **`tests/e2e/story-e23-s02.spec.ts:22` (confidence: 75)**: Sidebar selector `nav[data-testid="sidebar"], aside` is fragile. There is no `data-testid="sidebar"` on any `<nav>` element in the codebase, so the first half of the CSS selector never matches. The fallback `aside` works only because at `/` the Layout aside is the only one rendered, but this selector would silently break if another `<aside>` were added to the Overview page (e.g., a notification panel). Fix: Use `aside[aria-label="Sidebar"]` which matches the actual markup at `Layout.tsx:313` and is semantically precise.

- **`tests/e2e/story-e23-s02.spec.ts:91` (confidence: 72)**: Page heading negative assertion uses `.filter({ hasText: 'My Classes' })` but the old heading was actually "My Progress" (visible in the diff at `MyClass.tsx:116,136`). The test passes (no element contains "My Classes" either) but it is asserting against the wrong old text. If the story's intent is to verify the old label is gone, the assertion should check for "My Progress" since that was the actual previous heading text. Fix: Change to `.filter({ hasText: 'My Progress' })` or remove the negative assertion since it validates nothing meaningful.

- **`tests/support/helpers/navigation.ts:44-46` (confidence: 70)**: The `goToMyClass` helper silently swallows `waitForSelector` failures with `.catch(() => {})`. If the page heading never renders (e.g., due to a crash or wrong heading text), the helper reports success and downstream assertions may give confusing error messages. This is a pre-existing pattern (not introduced by this story) but was touched in this diff when the heading text was updated. Fix: Remove `.catch(() => {})` so that navigation failures surface immediately, or at minimum log a warning: `.catch((e) => console.warn('goToMyClass: heading not found', e))`.

#### Nits

- **Nit** `tests/e2e/story-e23-s02.spec.ts:52` (confidence: 65): Command palette test uses `Meta+k` which is macOS-only. On Linux/Windows CI the shortcut is `Control+k`. This test will fail on non-macOS Playwright runners. Consider: `await page.keyboard.press(process.platform === 'darwin' ? 'Meta+k' : 'Control+k')` or use both shortcuts. Currently low confidence because CI likely runs macOS, but worth noting for portability.

- **Nit** `src/app/components/figma/SearchCommandPalette.tsx:49` (confidence: 60): The search item `id` was changed from `page-my-progress` to `page-my-courses`, which is correct for the rename. However, users who had bookmarked or stored command palette state referencing the old ID would lose that association. Not a real concern for a client-only app with no persisted command palette state, but documents the ID change for awareness.

- **Nit** `docs/implementation-artifacts/e23-s02-rename-my-classes-to-my-courses.md:31-35` (confidence: 90): All tasks are marked `[ ]` (unchecked) despite implementation being complete and all review gates passing.
