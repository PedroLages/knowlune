## Remaining Issues

### Story-Related (LOW)

1. **[LOW] `isAllowedOllamaUrl` does not block `0.0.0.0`** — `server/index.ts:32-56`
   - `0.0.0.0` resolves to the local machine on many OSes. A crafted URL like `http://0.0.0.0:3001` could hit the proxy itself.
   - Mitigated by: (a) this is a local dev tool, not deployed to production; (b) user explicitly configures the URL; (c) Ollama typically runs on LAN IPs.
   - Recommend adding `hostname === '0.0.0.0'` to the blocklist in a future hardening pass.

2. **[LOW] No unit tests for `isAllowedOllamaUrl`** — `server/index.ts:32-56`
   - Security-critical function lacks test coverage. Server-side code is outside the current Vitest/Playwright test infrastructure, so this is an infrastructure gap rather than a story gap.
   - Recommend adding a server test suite in a future epic.

## Architecture Assessment

The implementation is well-structured:

- **Clean separation**: `OllamaLLMClient` handles client-side concerns (dual connection modes, Ollama-specific timeouts/errors). Server-side proxy uses `createOpenAI` with custom `baseURL` for SDK compatibility.
- **Security**: SSRF protection via `isAllowedOllamaUrl`, Zod schema validation, loopback blocking.
- **Error UX**: Helpful messages for common Ollama issues (server unreachable, timeout).
- **Factory pattern**: Clean integration into `getLLMClient()` with proper Ollama-specific branching.
