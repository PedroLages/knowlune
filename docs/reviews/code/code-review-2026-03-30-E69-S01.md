## Code Review: E69-S01 -- Storage Estimation Service and Overview Card

### What Works Well

1. **Clean architecture separation.** `storageEstimate.ts` is a pure service layer with no UI concerns, and `StorageManagement.tsx` delegates all estimation to it. The `StorageCardShell` extraction eliminates four duplicate `CardHeader` blocks -- good DRY refactor.

2. **Defensive async patterns.** `Promise.allSettled` at both the category and table levels means one corrupted table cannot tank the entire dashboard. The `cancelled` flag in `useEffect` prevents state updates after unmount. The `handleRefresh` guard (`if (refreshing) return`) prevents double-clicks.

3. **Previous review findings substantially addressed.** All five prior findings (error state, summary line, `courses` table removal, `useCallback` removal, Blob `.size`) are fixed in commit `53602d0b`. No uncommitted changes detected in story files -- the recurring 10-story uncommitted-fix pattern is broken (positive).

### Findings

#### Blockers

None.

#### High Priority

- **`src/lib/storageEstimate.ts:71-76` (confidence: 82)**: Blob estimation returns only the FIRST Blob's `.size` and ignores non-Blob metadata in the same row. A `courseThumbnails` row with `{ courseId, blob, mimeType, createdAt }` counts only `blob.size`, losing ~50-100 bytes of metadata per row. Conversely, `screenshots` rows where the Blob is the second field scanned would correctly hit the Blob path, but if field iteration order places `courseId` first, `JSON.stringify` runs instead -- serializing the Blob as `{}` (2 bytes) and massively underestimating. Why: Learners see wildly inaccurate size for blob-heavy categories, undermining trust in the dashboard. Fix: Sum all Blob `.size` values in the row PLUS `JSON.stringify` the non-Blob fields, or at minimum sum ALL Blobs in the row rather than returning on the first:
  ```typescript
  let blobBytes = 0
  let hasBlobField = false
  for (const val of Object.values(row as Record<string, unknown>)) {
    if (val instanceof Blob) { blobBytes += val.size; hasBlobField = true }
  }
  return sum + (hasBlobField ? blobBytes : new Blob([JSON.stringify(row)]).size)
  ```

- **`src/app/components/settings/StorageManagement.tsx:167` (confidence: 78)**: `ChartContainer` has no `aria-label`. AC1 (Task 6.6) explicitly requires `aria-label="Storage usage breakdown chart"` on the chart. The `sr-only` table at line 195 provides a text alternative, but screen readers encountering the chart area itself get no announced label. Why: WCAG 1.1.1 (Non-text Content) requires programmatic alternatives for visual content. Fix: Add `aria-label="Storage usage breakdown chart"` to the `ChartContainer` div.

- **`src/app/components/settings/StorageManagement.tsx:350` (confidence: 75)**: Error state condition `error && !overview` means a refresh failure AFTER a successful initial load shows no error feedback. If `handleRefresh` fails, `setError(true)` fires but `overview` is still truthy from the first load -- the normal state renders unchanged with no indication the refresh failed. Why: Learner clicks Refresh, sees spinner, spinner stops, nothing changes -- no toast, no error indicator. Fix: Either show a toast on refresh failure (`import { toastError }`) or check `error` independently of `overview` in the render path.

#### Medium

- **`src/lib/storageEstimate.ts:117` (confidence: 72)**: `getStorageOverview` never throws in practice -- `getStorageEstimate` catches internally (returns null), and `Promise.allSettled` catches per-table. The component's `try/catch` in `load()` and `handleRefresh` are unreachable dead branches. This is not a bug (defensive coding is good), but the `setError(true)` path is effectively dead code. Fix: Consider whether `getStorageOverview` should intentionally re-throw on catastrophic failures (e.g., `db` is undefined) so the error state is actually exercisable, or add a unit test that forces a throw to prove the path works.

- **`src/app/components/settings/StorageManagement.tsx:103` (confidence: 70)**: Critical banner's "Free Up Space" button scrolls to `#data-management` but AC4 specifies scrolling to `#cleanup-actions`. No element with `id="cleanup-actions"` exists in Settings.tsx. The adaptation to `#data-management` is reasonable but the AC mismatch should be documented. Fix: Either add `id="cleanup-actions"` to the relevant section in Settings.tsx, or update the story AC to reflect the actual target.

- **`src/app/components/settings/StorageManagement.tsx:224-227` (confidence: 72)**: `CategoryBreakdownLegend` percentage is calculated relative to `categorizedTotal` (sampled estimate), not `totalUsage` (browser-reported). This means category percentages sum to 100% of sampled data, not 100% of actual usage. If sampled categories account for 40% of real usage, each category's percentage is inflated ~2.5x. The summary line above correctly shows browser-reported total, creating a visual inconsistency. Fix: Consider showing percentage of `totalUsage` instead, or adding a footnote explaining these are relative proportions of categorized data.

- **`src/lib/storageEstimate.ts:62` (confidence: 68)**: `estimateTableSize` calls `db.table(tableName)` with a string. If a table name in `CATEGORY_MAP` doesn't match a Dexie table (e.g., typo or schema change), `db.table()` throws synchronously. This is caught by the outer `try/catch`, so it degrades to 0 bytes -- acceptable. But there's no type safety between `CATEGORY_MAP` table names and the actual Dexie schema. Fix: Consider typing table names against `keyof typeof db` or adding a comment noting the maintenance coupling with `schema.ts`.

#### Nits

- **Nit** `src/app/components/settings/StorageManagement.tsx:56` (confidence: 65): `font-display` class on `CardTitle` -- this is a custom utility that depends on a `@font-face` declaration with `font-display` descriptor. Verify this renders the intended heading font and not an unintended `font-display: auto` CSS property.

- **Nit** `src/app/components/settings/StorageManagement.tsx:4` (confidence: 60): `Link` imported from `react-router` but only used once (empty state CTA). Minor -- no action needed, but if the empty state CTA changes to a button action, this import becomes dead.

- **Nit** `src/lib/__tests__/storageEstimate.test.ts` (confidence: 60): No test covers the case where `getStorageEstimate` itself rejects (as opposed to returning null). Since `getStorageOverview` does not wrap `getStorageEstimate()` in try/catch, this would be the one path that exercises the component's error state. Adding this test would close the dead-code concern from the Medium finding above.

### Recommendations

1. **Fix refresh-failure silent swallow (H3)** -- add `toastError('Unable to refresh storage data')` in the `handleRefresh` catch block. Quick fix, high UX impact.
2. **Add `aria-label` to ChartContainer (H2)** -- one-line fix for AC compliance.
3. **Improve Blob estimation accuracy (H1)** -- sum all Blobs per row instead of returning on first. Moderate effort, improves data quality.
4. **Reconcile category percentage base (M3)** -- decide whether legend percentages should be relative to sampled total or browser-reported total.
5. **Document `#cleanup-actions` AC mismatch (M2)** -- either fix the scroll target or update the AC.
