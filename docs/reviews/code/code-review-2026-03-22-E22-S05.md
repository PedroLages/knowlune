## Findings

### [High] Stale closure over `importedCourses` in useEffect — `Courses.tsx:131`
**Confidence: 92**

The effect uses `importedCourses` inside `loadCourseMetrics()` but the dependency array only includes `[allCourses]`. When a new course is imported, the Zustand store updates `importedCourses`, but the effect does not re-run. Momentum scores for newly imported courses will not be calculated until an unrelated event fires.

**Fix:** Add `importedCourses` to the dependency array: `[allCourses, importedCourses]`

---

### [High] `filtered` computed via IIFE on every render, defeats downstream `useMemo` — `Courses.tsx:138-160`
**Confidence: 82**

The `filtered` constant is computed via an IIFE that runs on every render, then passed to `useMemo` for `sortedCourses` at line 162. Since `filtered` is a new array reference each render, the `useMemo` at line 162 that depends on `[filtered, sortMode, momentumMap]` recomputes every render, defeating its purpose. Same pattern applies to `filteredImportedCourses` at line 174.

**Fix:** Wrap both in `useMemo` with proper dependencies `[allCourses, selectedFilter, searchQuery]` and `[importedCourses, searchQuery, selectedFilter, selectedStatuses]`.

---

### [High] Duplicated 300-character chip className strings — `Courses.tsx:340, 348`
**Confidence: 85**

Lines 340 and 348 each contain ~300-character className strings that are nearly identical (differing only by `mr-1`). The implementation plan specified a `chipClassName` helper but it was not implemented.

**Fix:** Extract a shared constant:
```typescript
const chipBase = "h-auto rounded-full! border px-4 py-3 sm:py-1.5 ..."
```

### [Medium] O(n×m) counting — quadratic complexity in `buildUnifiedFilterChips` — `filterChips.ts:38-44`
**Confidence: 75**

For each chip, the code iterates over ALL preseeded + imported courses with nested `.some()` on tags. For small datasets fine, but with 50+ imported courses and 10+ tags each, becomes expensive on every render.

**Fix:** Pre-compute a `Map<string, number>` in a single pass over all courses, then look up counts during chip construction.

---

### [Medium] Incomplete `db` mock causes silent metric failure in tests — `Courses.test.tsx:97-98`
**Confidence: 72**

The mock provides `db.importedVideos` but `Courses.tsx` calls `db.studySessions.toArray()`. This throws and is silently caught at line 118, meaning no momentum/atRisk/estimate data is set. Tests pass but exercise a degraded state.

**Fix:** Add `studySessions: { toArray: () => Promise.resolve([]) }` to the db mock.

### [Medium] `allTags` variable name is misleading — `Courses.tsx:169-172`
**Confidence: 78**

`allTags` only contains imported course tags (not all tags). It's passed to `ImportedCourseCard` for the `TagEditor` component but the name implies broader scope.

**Fix:** Rename to `importedTags` or `availableImportedTags`.

---

### [Nit] `titleCase` regex capitalizes after hyphens — `filterChips.ts:7-8`
**Confidence: 65**

`\b\w` capitalizes after hyphens (e.g., "behavioral-analysis" → "Behavioral-Analysis"). Acceptable since hyphenated slugs get `categoryLabels` lookup first, but edge case for user-added hyphenated tags.

**Fix:** Use space-split approach: `s.split(' ').map(w => w.charAt(0).toUpperCase() + w.slice(1)).join(' ')`
