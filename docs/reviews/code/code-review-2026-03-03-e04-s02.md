- **[progress.tsx:14-21](src/app/components/ui/progress.tsx#L14-L21) (confidence: 92)**: `value` is destructured from props but `normalizedValue` is never passed back to `<ProgressPrimitive.Root>` as the `value` prop. Radix internally consumes `value` (line 55-56 of the Radix source: `const { value: valueProp = null, ... } = props`). Since `value` is destructured out by the wrapper, it's absent from `...props`, so Radix receives `valueProp = null`. This causes Radix to set `data-state="indeterminate"` on the DOM element instead of `"loading"` (partial) or `"complete"` (100%). Any CSS or JS that relies on `data-state` (e.g., `data-[state=complete]:bg-green-500`) will break. Why: Learners seeing 100% progress but `data-state="indeterminate"` creates an inconsistent DOM model and breaks any selector-based styling. Fix: Pass `normalizedValue` as `value` to the Radix root:
  ```tsx
  <ProgressPrimitive.Root
    value={normalizedValue}
    // ...rest
  >
  ```

### High Priority

- **[story-e04-s02.spec.ts:50-66](tests/e2e/story-e04-s02.spec.ts#L50-L66) (confidence: 90)**: AC2 E2E test is entirely conditional -- the test body only executes `if ((await contentItem.count()) > 0)` where `contentItem` looks for `[data-testid*="content-status"]`. Fix: Either seed a completion state via localStorage before the test, or use a `test.fixme()` / `test.skip()` annotation with a clear explanation, so CI reports it as skipped rather than falsely passing.

- **[story-e04-s02.spec.ts:69-88,91-122](tests/e2e/story-e04-s02.spec.ts#L69-L88) (confidence: 88)**: AC3 and AC4 E2E tests are also vacuously passing. AC4 loops through cards looking for `ariaValue === '100'` -- if no course is at 100%, the loop completes with zero assertions. For AC4, use `page.evaluate()` to set a course's completedLessons to all lessons via localStorage before testing the 100% badge.

- **[progress.tsx:27-30](src/app/components/ui/progress.tsx#L27-L30) (confidence: 78)**: The explicit `role="progressbar"`, `aria-valuemin={0}`, `aria-valuemax={100}`, `aria-valuenow={normalizedValue}` props are redundant with Radix's built-in attributes (confirmed in Radix source at `node_modules/radix-ui/node_modules/@radix-ui/react-progress/dist/index.js:72-76`). Radix already renders these attributes. The explicit props end up in `...progressProps` and override Radix's values, so there's no functional harm -- but if Radix's behavior ever changes (e.g., the component adds `aria-busy` or adjusts `aria-valuetext`), the override will mask it. Why: Defensive ARIA is good intent but coupling to Radix internals creates a maintenance burden. Fix: Remove the explicit ARIA attributes and rely on Radix. If custom `aria-label` is needed, pass only that. Alternatively, add a comment explaining the intentional override and why.

- **[progress.tsx:37](src/app/components/ui/progress.tsx#L37) (confidence: 82)**: `duration-500 ease-out` transition on the progress bar indicator does not respect `prefers-reduced-motion`. The story notes claim "500ms ease-out transition respects motion preferences" but there is no `motion-reduce:` modifier applied. The card shell at line 603 of CourseCard.tsx correctly uses `motion-reduce:hover:scale-100`, showing the pattern is known. Why: Learners with vestibular disorders or motion sensitivity will see the 500ms animation regardless of OS preference. Fix: Add `motion-reduce:duration-0` or `motion-reduce:transition-none` to the indicator class list.
