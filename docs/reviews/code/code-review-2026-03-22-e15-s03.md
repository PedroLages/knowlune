# Code Review: E15-S03 — Display Timer Warnings at Key Thresholds

## What Works Well

1. **Callback-over-state architecture**: Using `onWarning` callback with `onWarningRef` pattern (mirroring the existing `onExpireRef`) avoids re-render cascades through all timer consumers. This is the right design choice for a frequently-ticking hook.

2. **Dual ARIA live region strategy**: The separation of `aria-live="polite"` for 25% and `aria-live="assertive"` for 10%/1min is spec-correct and demonstrates thoughtful accessibility engineering. The `aria-atomic="true"` ensures screen readers re-announce the full message.

3. **`warningsFiredRef` single-fire guard**: Prevents threshold callbacks from re-firing on every tick after crossing a threshold. Clean pattern that follows the existing `hasFiredRef` convention.

## Findings

### Blockers

- **[src/hooks/useQuizTimer.ts:82-96] (confidence: 92)**: All three warning thresholds fire synchronously in the same `recalculate()` tick for short quizzes. For a 2-minute quiz, when `remaining = 30`: `30/120 = 0.25` triggers 25%, `0.25 <= 0.10` is false so 10% skips, but `30 <= 60` triggers 1min. For a 4-minute quiz at `remaining = 24`: `24/240 = 0.10` triggers both 25% AND 10%, and `24 <= 60` triggers 1min. All three `onWarning` calls execute in the same synchronous block, React 19 batches the resulting `setWarningState` calls in Quiz.tsx, and only the last value (`'1min'`) survives. `TimerWarnings` never sees `'25%'` or `'10%'` — those toasts and ARIA announcements are silently lost. Why: Learners with short quizzes (1-5 minutes) miss the graduated urgency escalation that the AC mandates. Fix: Either (a) fire toasts directly in the hook callback (avoiding the intermediate state), (b) accumulate warnings in an array and process them sequentially in TimerWarnings via a queue, or (c) add `setTimeout(0)` spacing between callbacks so React processes each state update independently. Option (a) is simplest — move `toast.info/warning` calls into `handleTimerWarning` in Quiz.tsx and keep TimerWarnings purely for ARIA.

### High Priority

- **[src/app/components/quiz/TimerWarnings.tsx:40] (confidence: 90)**: The persistent 1-minute toast (`duration: Infinity`) is never dismissed when the timer expires or the quiz is submitted. `handleTimerExpiry` in Quiz.tsx navigates to the results page, but Sonner toasts persist across route changes by default (they are rendered in the root layout via `<Toaster />`). The learner sees "00:XX remaining" lingering on the results screen alongside the "Time's up!" error toast. Why: Confusing UX — a "remaining" toast on a completed quiz. Fix: Store the toast ID from `toast.warning()` (it returns a string/number ID) and call `toast.dismiss(id)` in `handleTimerExpiry`, or use `toast.dismiss()` (no args) to clear all toasts before showing the expiry message.

- **[src/app/components/quiz/TimerWarnings.tsx:24-25] (confidence: 85)**: The `prevLevelRef` guard (`warningLevel === prevLevelRef.current`) is redundant with `warningsFiredRef` in the hook. However, it introduces a subtle bug: if the component unmounts and remounts (e.g., React Suspense boundary, error boundary recovery), `prevLevelRef` resets to `null` but the hook's `warningsFiredRef` still shows the warning as fired. The callback will not re-fire, so the ARIA announcement is lost on remount. More critically, if `remainingSeconds` changes but `warningLevel` stays the same (which cannot happen with current hook logic but could with future changes), the effect would also be skipped due to the ref guard — making it fragile. Why: Defense-in-depth is good, but this particular guard masks a category of bugs. Fix: Remove the `prevLevelRef` guard entirely — the hook already guarantees each level fires exactly once, so the component can trust that any `warningLevel` change is a new warning.

- **[src/hooks/useQuizTimer.ts:82] (confidence: 80)**: Warnings skip when `remaining === 0`. The guard `if (remaining > 0 && onWarningRef.current)` means a quiz that jumps from `remaining > 60` to `remaining === 0` in one tick (e.g., tab hidden for a long time, then `visibilitychange` fires) will never fire any warnings. Why: A learner who switches away from a quiz tab for the entire duration gets zero warning toasts — the timer just expires silently. Fix: Consider firing at least the most critical warning (`'1min'`) when `remaining === 0` and no warnings have fired yet, before the expiry callback executes. This gives the learner context that the timer expired due to low time.

### Medium

- **[src/app/components/quiz/TimerWarnings.tsx:44] (confidence: 75)**: The `useEffect` dependency array includes `remainingSeconds`, but the effect early-returns when `warningLevel === prevLevelRef.current`. This means changing `remainingSeconds` while `warningLevel` stays the same triggers the effect but does nothing. While harmless, it causes unnecessary effect invocations on every timer tick if `warningState` were to be updated more frequently. Fix: Remove `remainingSeconds` from the dependency array — the effect only cares about `warningLevel` changes. Access `remainingSeconds` via a ref if the stale closure concern exists.

- **[tests/e2e/story-15-3.spec.ts:731] (confidence: 72)**: The persistence test uses `waitForTimeout(6000)` which adds 6 real seconds to test runtime. While justified for verifying Infinity-duration toast persistence, a shorter wait (2-3s) would still prove the toast did not auto-dismiss (Sonner's default is 4s, and the 25% toast auto-dismisses at 3s). Fix: Reduce to `waitForTimeout(3500)` — still exceeds the longest auto-dismiss duration (3s for 25% toast) while cutting test time nearly in half.

- **[tests/e2e/story-15-3.spec.ts:347] (confidence: 70)**: Test file is 347 lines, approaching the project's 300-line target. Not a blocker now, but the accommodation test (AC6, lines 801-843) could be extracted into a separate describe block in a dedicated file if more tests are added. Fix: Consider splitting into `story-15-3-warnings.spec.ts` and `story-15-3-accommodations.spec.ts` if the file grows further.

### Nits

- **Nit** [src/hooks/useQuizTimer.ts:45] (confidence: 65): `warningsFiredRef` uses verbose property names (`twentyFivePercent`, `tenPercent`, `oneMinute`). A `Set<WarningLevel>` would be cleaner: `warningsFiredRef.current.has('25%')` / `warningsFiredRef.current.add('25%')`. Matches the `WarningLevel` type directly without a mapping layer.

- **Nit** [src/app/components/quiz/TimerWarnings.tsx:32-42] (confidence: 60): The `switch` statement could use the `closeButton` option on the persistent toast for explicit dismissal affordance (AC says "persistent warning remains visible until time expires" — the global `closeButton: true` on Toaster handles this, but being explicit in the call site documents intent). `toast.warning(\`...\`, { duration: Infinity, closeButton: true })`.

## Recommendations

Fix in this order:

1. **[Blocker]** Fix the batched-state-update problem for short quizzes — move toast firing into the callback or use a warning queue. This is a correctness bug that silently drops warnings.
2. **[High]** Dismiss the persistent toast on timer expiry — store the toast ID and call `toast.dismiss(id)` in `handleTimerExpiry`.
3. **[High]** Address the `remaining === 0` skip — fire at least the 1-minute warning before expiry for tab-hidden scenarios.
4. **[High]** Remove `prevLevelRef` guard in TimerWarnings — it is redundant and fragile.
5. **[Medium]** Clean up the `remainingSeconds` dependency in the useEffect.
