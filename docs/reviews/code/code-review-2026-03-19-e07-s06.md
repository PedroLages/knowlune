## Code Review: E07-S06 — E2E Test for Course Suggestion Tiebreaker

### What Works Well

1. **Thorough mathematical proof in comments.** The test includes step-by-step score calculations in the inline comments, making it easy for future maintainers to understand WHY a particular course should win. This is the gold standard for algorithm-validation tests.

2. **Excluded course seeding pattern.** Marking all non-candidate courses as 100% complete to remove them from the candidate pool is a clean isolation strategy. The `Array.from({ length: 1000 })` trick to exceed any course's lesson count is pragmatic.

3. **`closeCompletionModal` scoping fix.** Changing from a page-wide `button` locator to `page.getByRole('dialog').locator(...)` is a genuinely valuable fix that improves reliability for ALL tests using this helper, not just the new one.

### Findings

#### Blockers

- **[tests/e2e/regression/story-e07-s03.spec.ts:526-527] (confidence: 95)**: Wrong course ID `'behavior-skills'` used for seeding progress data. The actual course ID in `src/data/courses/behavior-skills.ts:6` is `'behavior-skills-breakthrough'`. The algorithm looks up `allProgress[course.id]` at `src/lib/suggestions.ts:40`, so the seeded data at key `'behavior-skills'` is never found. The `behavior-skills-breakthrough` course sees 0 progress and 0 recency, giving it `finalScore = 0.172` vs confidence-reboot's `0.457`. The test passes, but the primary sort (`b.score !== a.score` at `suggestions.ts:69`) resolves the winner before the tiebreaker path at line 72 is ever reached. **This test does not validate tiebreaker behavior at all** -- it validates that a course with progress beats one without, which is trivially true. Why: The entire story exists to close a tiebreaker coverage gap; if the tiebreaker code path is never exercised, the gap remains open. Fix: Change all `'behavior-skills'` references to `'behavior-skills-breakthrough'` in the new test (lines 461, 482-excluded list check, 526-527, 530). Also update `ALL_COURSE_IDS` at line 35, which has the same bug (affects AC5 test).

#### High Priority

- **[tests/e2e/regression/story-e07-s03.spec.ts:556] (confidence: 80)**: `textContent()` call without Playwright auto-retry. The pattern `const suggestedTitle = await suggestionCard.locator('h2').textContent()` followed by `expect(suggestedTitle?.toLowerCase()).toMatch(...)` fetches text content at a single point in time. If the h2 text renders asynchronously or updates after initial mount, this could return `null` or stale content. The identical pattern exists at line 336 (pre-existing), but duplicating a fragile pattern is not ideal. Why: A race between the suggestion card becoming visible (line 552) and its h2 content being populated could cause intermittent failures. Fix: Use Playwright's auto-retrying assertion: `await expect(suggestionCard.locator('h2')).toHaveText(/confidence reboot/i)`. This retries until the text matches or times out, eliminating the race window.

- **[tests/e2e/regression/story-e07-s03.spec.ts:35] (confidence: 92)**: Pre-existing bug: `ALL_COURSE_IDS` includes `'behavior-skills'` but the real course ID is `'behavior-skills-breakthrough'`. This affects the AC5 test (line 189-242) which tries to mark ALL courses complete. Because `behavior-skills-breakthrough` is never seeded as complete, the algorithm finds it as a candidate rather than showing the "all courses complete" congratulatory message. This is likely a contributing factor to the "pre-existing AC5 failure" documented in the story. Why: Fixing the ID here would unblock the AC5 test, which is currently masked as "pre-existing." Fix: Change `'behavior-skills'` to `'behavior-skills-breakthrough'` in `ALL_COURSE_IDS` at line 35.

#### Medium

- **[tests/e2e/regression/story-e07-s03.spec.ts:467] (confidence: 75)**: Comment says `progress = 10/20 = 50%` but should verify the denominator. The test relies on `confidence-reboot` having exactly 20 lessons, confirmed by current course data. However, if anyone adds a lesson to confidence-reboot, the progress percentage changes, potentially narrowing the margin enough to flip the result. Fix: Add a comment anchoring the assumption: `// 20 total lessons as of 2026-03 (see src/data/courses/confidence-reboot.ts)` or, better, add a static assertion test that verifies the expected lesson count.

- **[tests/e2e/regression/story-e07-s03.spec.ts:445] (confidence: 70)**: Test name says "AC2" but is actually a supplementary tiebreaker test for AC2. The existing test at line 244 is already named "AC2". Having two tests with "AC2" prefix creates ambiguity in test reports. Fix: Rename to something like `'tiebreaker selects highest momentum when tag overlap counts match (E07-S06 gap coverage)'` to distinguish it from the original AC2 test.

#### Nits

- **Nit** [tests/e2e/regression/story-e07-s03.spec.ts:557] (confidence: 65): The regex `/(confidence|reboot)/` matches either word independently. If a course named "Reboot Your Skills" existed, it would false-positive. Consider `/(confidence.*reboot|reboot.*confidence)/i` or simply the exact string `/confidence reboot/i`.

- **Nit** [docs/implementation-artifacts/7-6-e2e-test-for-course-suggestion-tiebreaker.md:58] (confidence: 60): Pre-review checklist items are all unchecked. While this is a test-only story and most items don't apply, marking N/A items or checking applicable ones signals the checklist was actually reviewed.

### Recommendations

1. **Fix the course ID mismatch (Blocker).** Change `'behavior-skills'` to `'behavior-skills-breakthrough'` in the new test AND in `ALL_COURSE_IDS`. Re-run the test to confirm it still passes -- if it does not, the tiebreaker algorithm or your score math needs investigation.

2. **Switch to auto-retrying assertion (High).** Replace `textContent()` + manual `expect` with `toHaveText()` at line 556 (and optionally backport to line 336).

3. **Disambiguate the test name (Medium).** Rename from "AC2: tiebreaker selects..." to avoid collision with the existing AC2 test at line 244.

4. **After fixing the ID, verify the margin.** With correct seeding (`behavior-skills-breakthrough` at 3/13 progress, 10 days ago), recalculate both finalScores. Both should now be 0.286*0.6 + momentum*0.4, with identical tagScore, forcing the sort to fall through to the tiebreaker at `suggestions.ts:72`. If the finalScores are not *exactly* equal due to floating point, the primary sort still resolves before the tiebreaker -- you may need to ensure the scores are equal to machine precision, or the test still does not exercise the tiebreaker path.
