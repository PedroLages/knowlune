#### Blockers
- **[useReadingGoalStore.test.ts:277] (confidence: 92)**: `checkYearlyGoalReached` asserts that the function returns `false` when `count > target` (`count=13, target=12`). This means reaching 13 books in a year would *not* trigger the yearly goal reached notification. The comment even says "Past threshold". The method likely uses strict equality (`===`) instead of `>=`, which is almost certainly a bug in the implementation — any count beyond the target should still count as goal reached. Fix: Change the assertion at line 277 from `toBe(false)` to `toBe(true)`, and fix the underlying `checkYearlyGoalReached` implementation to use `>=` instead of `===`.

#### High Priority
- **[useReadingGoalStore.test.ts:13] (confidence: 85)**: `vi.useFakeTimers()` + `vi.setSystemTime()` in `beforeEach` globally replaces `Date`, but `useNotificationPrefsStore.test.ts` also uses `new Date()` for quiet-hours logic. If these test files ever run in the same worker (or if a shared module is imported), the fake timer leaks. More critically for this file: `checkDailyGoalMet` constructs `todayStr` (likely using `new Date()`), and while fake timers mock `Date.now()`, some Dexie/internal code may depend on real timers. This isn't a production bug but a test fragility issue — the `afterEach` `vi.useRealTimers()` will mask any leaked state. **Correction**: Actually scoped correctly with afterEach. Downgrading concern — but the real issue is that the `loadGoal` function reads from `localStorage` and calls `new Date()` internally; if the store was imported at module level, the fake timer wouldn't apply to the store's module-level `Date` capture. Since `vi.resetModules()` is used, this is handled. No action needed.

#### Medium
- **[useAudiobookshelfStore.test.ts:288-337] (confidence: 80)**: The `loadSeries` pagination test builds results of length 50 and 25 to match `total: 75`. If the store's pagination logic uses `total` to decide whether to fetch another page, the test is correct. However, if the store uses `results.length < page_size` as the termination condition, this test would pass even with a broken `total` field. The page size is implicitly 50, but never explicitly asserted — meaning a change to the page size could silently break pagination. Fix: Add an assertion verifying `fetchSeriesForLibrary` was called with the expected `page` parameter values (e.g., page 0 then page 1).

- **[useStudyScheduleStore.test.ts:385-392] (confidence: 75)**: The `regenerateFeedToken` test for "handles insert error after delete" asserts `feedToken` becomes `null`. However, the mock only returns `delete` and `insert` methods from `mockSupabase.from()` — since `from()` is called twice (once for delete, once for insert), and `mockReturnValue` returns the same object both times, the mock structure actually works because the single object has both `delete` and `insert`. But this is fragile: if the real implementation calls `from('feed_tokens')` separately for each operation, the mock doesn't differentiate between the two calls. Fix: Use `mockImplementation` keyed on the table name, or use `mockReturnValueOnce` for chained calls.

- **[useStudyScheduleStore.test.ts:259-263] (confidence: 70)**: The `updateSchedule` test for "prevents id override" passes `{ id: 'new-id', title: 'Changed' } as Partial<StudySchedule>`. If the store implementation doesn't explicitly strip `id` from update data, the test would still pass because Dexie's `put()` uses the key from the object — but `db.studySchedules.put()` would either create a new record or update based on the key path. If the store uses `db.studySchedules.update(id, data)` instead of `put()`, the `id` field would be harmlessly written but ignored. The test asserts only `schedules[0].id === 'keep-id'` but doesn't verify the Dexie record's id. Fix: Verify the Dexie record id wasn't changed: `expect((await db.studySchedules.get('keep-id'))?.id).toBe('keep-id')`.

- **[useNotificationPrefsStore.test.ts:18] (confidence: 65)**: `FIXED_DATE` is declared but never used in this test file. The `isInQuietHours` tests rely on `new Date()` being the real current time (since no fake timers are used), making the tests non-deterministic. The `00:00-23:59` and `23:00-22:59` test cases are designed to be time-insensitive, but edge cases (e.g., `23:59` boundary at exactly 23:59 real time) could cause flaky failures. Fix: Use `vi.useFakeTimers()` and `vi.setSystemTime(FIXED_DATE)` to make `isInQuietHours` tests deterministic.

#### Nits
- **[useReadingGoalStore.test.ts:12] (confidence: 55)**: `FIXED_DATE` is defined as `'2026-03-23T10:00:00.000Z'` which is a Monday. The streak tests at lines 236–255 create consecutive days on Saturday/Sunday (March 22 is Sunday, March 23 is Monday). The streak logic likely compares date strings (YYYY-MM-DD), so this works, but choosing weekday dates for "consecutive days" testing would be more intuitive.
