## Round 2 Findings

### MEDIUM — Dark vibrant brand color fails WCAG AA by 0.02 (FIXED IN-REVIEW)

**File:** `src/styles/theme.css:287`
**Classification:** STORY-RELATED

The dark vibrant `--brand` token `oklch(0.58 0.18 275)` paired with `--brand-foreground: #ffffff` yields a contrast ratio of 4.48:1, just below the WCAG AA threshold of 4.5:1. The canvas pixel readback in E2E tests caught this precisely.

**Fix applied:** Reduced lightness from 0.58 to 0.57, yielding `oklch(0.57 0.18 275)`. Also adjusted `--brand-hover` from 0.52 to 0.51 to maintain the 0.06 lightness gap between brand and hover states.

### MEDIUM — WCAG contrast E2E tests used broken color resolution method (FIXED IN-REVIEW)

**File:** `tests/e2e/regression/story-e21-s04.spec.ts:246-258, 294-317, 346-362`
**Classification:** STORY-RELATED

The WCAG contrast tests (added per Round 1 recommendation) used a `probe.style.color` approach expecting `getComputedStyle().color` to return `rgb()`. Modern Chromium (v130+) returns OKLCH values directly from computed styles, causing `parseColor()` to throw `Cannot parse color: oklch(...)`.

**Fix applied:** Replaced probe element approach with canvas 2D pixel readback (`fillRect` + `getImageData`), which always resolves to RGB regardless of input color space. Also extended `parseColor()` to handle hex `#rrggbb` format returned by canvas `fillStyle`.
