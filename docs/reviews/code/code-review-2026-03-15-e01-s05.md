## Code Review: E01-S05 — Detect Missing or Relocated Files

### What Works Well

1. **Clean separation of concerns.** `fileVerification.ts` is a pure async utility with no side effects, `useFileStatusVerification.ts` orchestrates the batch verification with toast aggregation, and `ImportedCourseDetail.tsx` handles presentation. Each layer is independently testable.

2. **Proper async cleanup pattern.** The `ignore` flag in `useFileStatusVerification` prevents state updates after unmount. The `Promise.allSettled` usage gracefully handles mixed success/failure without short-circuiting.

3. **Thoughtful test strategy.** Recognizing that `FileSystemHandle` is browser-only and leveraging null handles to naturally exercise the missing-file path is pragmatic. The E2E tests verify the user-facing behavior without complex mocking.

### Findings

#### High Priority

- **[Recurring] [src/app/pages/ImportedCourseDetail.tsx:115,121,162,167] (confidence: 92)**: String interpolation for `className` instead of `cn()`. Four instances use template literals like `` className={`size-5 shrink-0 ${isUnavailable ? 'text-muted-foreground' : 'text-brand'}`} `` instead of the project's `cn()` utility. This is a recurring pattern since E01-S03 across 15+ stories. Why: `cn()` (via `tailwind-merge`) correctly resolves conflicting Tailwind classes; raw string concatenation can produce invalid class combinations if extended later. Fix: Import `cn` from `@/app/components/ui/utils` and use `cn('size-5 shrink-0', isUnavailable ? 'text-muted-foreground' : 'text-brand')` for all four instances.

- **[Recurring] [src/app/pages/ImportedCourseDetail.tsx:54-67] (confidence: 88)**: Dexie queries are fire-and-forget with no `.catch()`. Both `db.importedVideos.where(...).sortBy('order').then(...)` and `db.importedPdfs.where(...).toArray().then(...)` lack error handling. If IndexedDB is corrupted or the store is missing, the promise rejects silently and the component shows an empty course with no feedback to the learner. Why: A learner who imported a course and then sees no content has no way to diagnose the problem. Fix: Add `.catch(err => { console.error('Failed to load course content:', err) })` to both chains, and consider setting an error state to show a user-facing message.

- **[src/hooks/useFileStatusVerification.ts:64-67] (confidence: 85)**: `Promise.allSettled` rejection handler maps to `'unknown'` key. When a settlement fails, the code does `verified.set('unknown', 'missing')`. If multiple items fail, they all overwrite the same `'unknown'` key, losing all but the last failure. More critically, the actual item's ID is never added to the map, so `fileStatuses.get(item.id)` returns `undefined` and the component falls back to `'checking'` state forever. Why: Learners would see items stuck in a loading/checking state with no badge and no toast, a silent failure. Fix: Since `Promise.allSettled` always fulfills, the `rejected` branch only fires if the inner `async` throws (which `verifyFileHandle` catches internally). This branch is effectively dead code, but if you want safety, you need access to the item ID. Use `.map((item, index) => ...)` and correlate by index in the results loop.

- **[src/hooks/useFileStatusVerification.ts:72-80] (confidence: 80)**: Toast only reports missing files, not permission-denied files. The aggregated toast counts `missingFiles` but ignores items with `'permission-denied'` status. A learner with 3 files where 1 is missing and 2 need re-authorization only sees "1 file not found" with no mention of the permission issues. Why: The story AC2 says "a toast notification identifies the affected file" -- permission-denied is equally an affected state. Fix: Either include permission-denied files in the toast or fire a separate toast for them: `toast.warning('2 files need permission', { description: ... })`.

#### Medium

- **[src/hooks/useFileStatusVerification.ts:88] (confidence: 75)**: `useEffect` dependency on `videos` and `pdfs` arrays triggers re-verification on every render. Both `videos` and `pdfs` are `useState` arrays from `ImportedCourseDetail`. Since `setVideos` creates a new array reference each time the Dexie query resolves, the effect fires correctly on data load. However, if the parent re-renders for any other reason (e.g., store update), `videos`/`pdfs` maintain their reference so this is stable. Still, consider documenting this assumption or using a `useMemo` wrapper on the items array construction inside the effect to avoid accidental re-verification if array references change in a future refactor. Fix: Acceptable as-is, but add a comment noting the dependency behavior.

- **[src/app/pages/ImportedCourseDetail.tsx:155-157] (confidence: 78)**: PDF items only check for `'missing'` status, not `'permission-denied'`. The `isMissing` variable is `status === 'missing'` only, so a PDF with `'permission-denied'` status renders at `opacity-75` (normal PDF opacity) without a badge from `FileStatusBadge`. This is because `FileStatusBadge` IS called with the status and would render the permission badge, but the opacity logic doesn't treat permission-denied as degraded. Fix: Use `const isUnavailable = status === 'missing' || status === 'permission-denied'` (same pattern as video items) for consistent visual treatment.

- **[tests/e2e/story-e01-s05.spec.ts] (confidence: 72)**: AC3 test does not verify available files are "fully functional" alongside missing files. AC3 states "available files are fully functional AND missing files show the badge." The test only checks that missing items are disabled and have no links, but never seeds a course with a mix of available and missing files (impossible without real handles in test context). The test strategy comment acknowledges this limitation, which is good. Fix: Consider adding a unit test in `ImportedCourseDetail.test.tsx` that mocks the status map with a mix of `'available'` and `'missing'` items and verifies the available item is a clickable `<a>` while the missing one is a `<div>`.

#### Nits

- **Nit** [src/app/pages/ImportedCourseDetail.tsx:9] (confidence: 65): `FileStatus` type is imported but only used in the inline `FileStatusBadge` component. Since `FileStatusBadge` is defined in the same file, the import could be co-located with the component if it were extracted. Not a problem currently, but if `FileStatusBadge` is extracted to its own file later, remember to move the import.

- **Nit** [src/lib/fileVerification.ts:26] (confidence: 70): Bare `catch` block without logging. The `catch` in `verifyFileHandle` maps all errors to `'missing'`, which is correct behavior, but logging the error would help diagnose unexpected failures (e.g., browser API changes, SecurityError). Fix: `catch (error) { console.warn('File handle verification failed:', error); return 'missing' }`.

- **Nit** [tests/e2e/story-e01-s05.spec.ts:88] (confidence: 60): E2E test seeds videos but not PDFs in AC1 test, then AC2 seeds both. Consider consistency -- AC1 could also seed PDFs to verify PDF status indicators alongside video ones.

### Recommendations

1. **First**: Add `.catch()` to both Dexie queries in `ImportedCourseDetail.tsx` (lines 54-67) to prevent silent failures.
2. **Second**: Replace all four string interpolation `className` instances with `cn()` calls.
3. **Third**: Fix the `verified.set('unknown', 'missing')` dead-code/data-loss bug in the `rejected` branch of `useFileStatusVerification.ts`.
4. **Fourth**: Include `'permission-denied'` files in the toast notification and align PDF opacity logic with video logic.
5. **Fifth**: Add `console.warn` to the bare `catch` in `fileVerification.ts`.
