# Code Review: E06-S02 — Track Challenge Progress

## What Works Well

1. **Clean progress calculation architecture.** The `challengeProgress.ts` module cleanly separates each calculation type into its own function with a dispatcher, making it easy to test and extend. The switch statement in `calculateProgress` is exhaustive over `ChallengeType`.

2. **Proper expired/active separation with Collapsible.** The `useMemo` partitioning and Radix Collapsible component give users a clean UX for expired challenges without cluttering the active view. The `parseLocalDate` usage avoids UTC midnight shift bugs.

3. **Progress capping and completedAt tracking.** The store correctly caps `currentProgress` at `targetValue` and sets `completedAt` only once (preserving the original timestamp). The UI percentage calculation with `Math.min(100, ...)` provides a second safety net.

### High Priority

- **[Recurring] [src/lib/challengeProgress.ts:13] (confidence: 90)**: **String comparison for date filtering is fragile and may produce wrong results with mixed formats.** The filter `p.updatedAt >= challenge.createdAt` performs lexicographic string comparison. This works when both values are full ISO 8601 timestamps (`"2026-03-05T10:00:00.000Z"`), but `updatedAt` in `ContentProgress` is typed as `string // ISO 8601` with no runtime validation. If any record stores `updatedAt` as a date-only string (e.g., `"2026-03-05"`) or without milliseconds, the comparison silently produces incorrect results. Why: Learners could see wrong completion counts -- either overcounting (crediting pre-challenge completions) or undercounting (missing valid completions). Fix: Parse both dates before comparing:
  ```typescript
  .filter(p => new Date(p.updatedAt).getTime() >= new Date(challenge.createdAt).getTime())
  ```

- **[src/app/pages/Challenges.tsx:103-107] (confidence: 85)**: **useEffect with `.then()` chain lacks cleanup -- stale updates on unmount or StrictMode double-mount.** `loadChallenges().then(() => refreshAllProgress())` has no cancellation mechanism. If the component unmounts between the two calls (e.g., user navigates away quickly), `refreshAllProgress` still executes, calling `set()` and `db.challenges.bulkPut()` against potentially stale state. In React 18 StrictMode, this fires twice, causing duplicate IndexedDB writes. Why: Could cause flash of stale data or duplicate DB writes during navigation. Fix: Use an `ignore` flag pattern:
  ```typescript
  useEffect(() => {
    let ignore = false
    loadChallenges().then(() => {
      if (!ignore) refreshAllProgress()
    })
    return () => { ignore = true }
  }, [loadChallenges, refreshAllProgress])
  ```

- **[src/stores/useChallengeStore.ts:59-60] (confidence: 82)**: **Optimistic UI update followed by fire-and-forget DB write in `refreshAllProgress`.** The store calls `set({ challenges: updated })` then `await db.challenges.bulkPut(updated)`. If the `bulkPut` fails, the UI shows updated progress but the DB has stale data. On next page load, the user sees old values -- a confusing regression. The error is only logged to console. Why: Learners could see their progress "reset" on page reload after a silent DB write failure. Fix: Either (a) write to DB first and only update state on success, or (b) add a toast notification on failure so the user knows data may not have persisted:
  ```typescript
  try {
    await db.challenges.bulkPut(updated)
    set({ challenges: updated })
  } catch (error) {
    console.error('[ChallengeStore] Failed to refresh progress:', error)
    toast.error('Progress update may not have saved')
  }
  ```

### Medium

- **[src/lib/challengeProgress.ts:36-38] (confidence: 75)**: **Streak progress is not scoped to challenge creation date, contradicting AC4.** AC4 states "the system reads the user's current or longest streak count **since the challenge creation date**." The implementation calls `getCurrentStreak()` which returns the global current streak regardless of when the challenge was created. A learner who started a streak before creating the challenge would get credit for pre-challenge streak days. The code comment acknowledges this ("Streak is global") suggesting it was deliberate, but it diverges from the written AC. Fix: If intentional, update the AC to match. If not, implement a streak calculation that only counts consecutive days from `challenge.createdAt` onward.

- **[src/lib/challengeProgress.ts:10-14] (confidence: 72)**: **Full table scan for completion progress -- no `updatedAt` index.** The query uses `.where('status').equals('completed')` (indexed) but then `.filter(p => p.updatedAt >= challenge.createdAt)` (unindexed scan). For learners with hundreds of completed items, this scans every completed record in memory. The `contentProgress` table schema at `src/db/schema.ts:175` indexes `[courseId+itemId], courseId, itemId, status` but not `updatedAt`. Fix: Add `updatedAt` to the contentProgress index in a new DB version, then use `.where('[status+updatedAt]')` for an efficient compound query. Or accept the scan cost and add a comment documenting the tradeoff.

- **[src/app/pages/Challenges.tsx:179] (confidence: 70)**: **CollapsibleTrigger missing accessible label for screen readers.** The trigger text is `Expired (N)` which is adequate, but the `ChevronDown` icon has no `aria-hidden` attribute, potentially being announced as a meaningless image. Additionally, the trigger should indicate its expanded/collapsed state via `aria-expanded`. Radix Collapsible may handle this internally, but verify. Fix: Add `aria-hidden="true"` to the ChevronDown icon to be safe:
  ```tsx
  <ChevronDown aria-hidden="true" className={cn('size-4 transition-transform', expiredOpen && 'rotate-180')} />
  ```

- **[tests/e2e/story-e06-s02.spec.ts:62-68] (confidence: 70)**: **[Recurring] afterEach IDB cleanup is fire-and-forget.** The `try/catch` around `indexedDB.clearStore` calls swallows errors with an empty catch block. If cleanup fails silently, subsequent tests may see leftover data, causing flaky test results. This was flagged in E06-S01 as well. Fix: At minimum, log the error:
  ```typescript
  } catch (e) {
    console.warn('Test cleanup failed:', e)
  }
  ```

### Nits

- **Nit** [src/app/pages/Challenges.tsx:60] (confidence: 65): Icon container uses `bg-blue-600/10 text-blue-600` which are hardcoded Tailwind colors rather than theme tokens. Consider using theme variables for consistency with the design system.

- **Nit** [src/lib/__tests__/challengeProgress.test.ts:136-138] (confidence: 60): Streak progress test is trivially shallow -- it only tests that `calculateStreakProgress()` returns 0 in an empty localStorage. There is no test that verifies streak data is actually read and returned correctly, since setting up localStorage in Vitest with `fake-indexeddb` requires manual seeding. Consider adding a test that seeds `localStorage` with study-log data and verifies a non-zero return.

- **Nit** [src/stores/useChallengeStore.ts:50] (confidence: 55): `Math.min(raw, challenge.targetValue)` caps the raw value at `targetValue`, but for time-based challenges `raw` is a float (hours). If `targetValue` is 10 and `raw` is 10.5, `currentProgress` becomes 10. The UI then shows `10 / 10 hours` and `100%`, which is correct but loses precision. This is fine for display but worth noting if `currentProgress` is ever used for exact comparisons downstream.

## Recommendations

1. **Fix the date string comparison** in `calculateCompletionProgress` (High #1) -- this is a correctness risk that could silently produce wrong progress values.
2. **Add useEffect cleanup** (High #2) -- prevents stale state updates on unmount and is a React best practice.
3. **Write DB before updating UI state** in `refreshAllProgress` (High #3) -- prevents the "progress reset on reload" scenario.
4. **Clarify AC4 streak scoping** (Medium #1) -- resolve the gap between the AC wording and the implementation.
5. **Add `updatedAt` index** (Medium #2) -- proactive performance improvement before the contentProgress table grows.
