# Code Review R2: E92-S01 — Supabase P0 Migrations & Extensions

**Scope:** R1 fixup commit `b48e6904`. Focus: new fixup migration, rollback, verification script.

### What Works Well

- **Follow-up migration over edit-in-place** is the right call — the original is already applied to titan; linear history preserved with clear rationale documented in-file.
- **Clamp logic (`LEAST(p_updated_at, now() + interval '5 minutes')`)** is correct and cheap; combined with `GREATEST(existing, v_clamped_updated_at)` it defends against both future-dated incoming writes and against regression if a prior unclamped write already poisoned the row.
- **Verification script** is re-runnable, uses clear failure messages, and the added fixup-specific tests (unknown status raise, CHECK violation, clamp, LWW progression, idempotent client_request_id) exercise exactly the behaviors the R1 fixes changed.
- **STRICT + IMMUTABLE + plpgsql with RAISE** on `_status_rank` is the correct choice — STRICT handles NULL propagation automatically, RAISE handles unknown non-NULL values, IMMUTABLE still valid with raises (Postgres permits this).

#### High Priority

- **supabase/tests/e92-s01-verify.sql:179-220 (confidence: 92)** `[Testing]` — **AC3 cross-user RLS test is a false pass.** The first DO block (lines 78-170) only creates rows for `v_userA`; no `content_progress`, `video_progress`, or `study_sessions` rows are ever inserted for `v_userB`. Then the AC3 block runs `SELECT count(*) WHERE user_id = v_userB` and asserts it equals 0 — which would be true whether RLS works correctly or is entirely disabled. Same flaw in the cross-user UPDATE/DELETE checks: `ROW_COUNT = 0` because there are no target rows, not because RLS filtered them. **Fix:** Before the RLS block, run a handful of `upsert_*` calls with `p_user_id = v_userB` (as superuser, outside the auth-scoped txn) to seed rows that RLS must actively hide. Then the assertions become meaningful. Effort: ~10 lines added to the seed DO block.

#### Medium

- **supabase/migrations/20260417000002_p0_sync_foundation_fixups.sql:34-44 (confidence: 85)** `[Correctness]` — **Comment/code mismatch on CHECK constraint.** The comment (line 37) claims `NOT VALID so existing rows don't block`, but the `ADD CONSTRAINT` statement does not include `NOT VALID`. On a database where any pre-existing row has `progress_pct = 100 AND status = 'in_progress'`, this migration will fail. Titan was cleaned, so this didn't bite you — but the defensive posture the comment promises isn't actually in effect. **Fix:** either append `NOT VALID` (matches comment; then a future `VALIDATE CONSTRAINT` when safe), or delete the `NOT VALID` half of the comment. I'd pick `NOT VALID` — production rollouts eventually hit a non-clean DB. Effort: 1 token.

#### Nits

- **Nit** supabase/migrations/20260417000002_p0_sync_foundation_fixups.sql:51-86 (confidence: 95) `[Maintainability]` — Two back-to-back `CREATE OR REPLACE FUNCTION public._status_rank` definitions. The first (lines 51-68) is overwritten immediately by the second (71-86) and is dead on arrival; its `EXCEPTION WHEN OTHERS` handler also can't fire because the `CASE ... ELSE NULL` body doesn't raise. Remove the first block; keep the explicit IF/ELSIF version. Effort: delete ~18 lines.
- **Nit** supabase/migrations/rollback/20260413000001*p0_sync_foundation_down.sql (confidence: 70) `[Maintainability]` — Rollback filename ties it to the original migration timestamp but the comment correctly notes it also undoes the fixups (functions are re-defined then dropped; table CASCADE clears fixup constraints/columns). Consider either renaming to a neutral name (e.g., `p0_sync_foundation_down.sql`) or adding a matching `20260417000002*\*\_down.sql` for symmetry. Low priority — current behavior is correct.
- **Nit** supabase/migrations/20260417000002_p0_sync_foundation_fixups.sql:15 (confidence: 65) `[Performance]` — `ADD COLUMN ... NOT NULL DEFAULT gen_random_uuid()` with a VOLATILE default triggers a full table rewrite in PG (non-volatile defaults get the fast metadata path). Titan's tables are near-empty so this is fine, but worth noting before this pattern is reused on a populated production table. If ever needed on a large table, the two-step pattern (add nullable → backfill → set NOT NULL) is safer.

### Recommendations

1. **Fix the AC3 RLS test (High).** Seed at least one row per progress/session table for `v_userB` before the auth-scoped blocks so the cross-user assertions actually exercise RLS. Without this, verification silently passes even on a broken RLS config.
2. **Reconcile the `NOT VALID` comment with the CHECK statement (Medium).** Cheap, makes the migration safe to re-run against non-clean environments.
3. **Delete the dead first `_status_rank` definition (Nit).** Reduces file by ~18 lines and removes a red-herring EXCEPTION handler that can't fire.
