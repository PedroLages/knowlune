## Code Review: E18-S07 — Surface Quiz Analytics in Reports Section

### What Works Well

1. **Proper async cleanup pattern.** The `useEffect` in `QuizAnalyticsDashboard.tsx` uses the `ignore` flag pattern correctly (lines 31-46), preventing stale state updates on unmount. The `.catch()` handler logs the error and exits the loading state gracefully. This is a notable improvement over earlier stories.

2. **Design token compliance.** The `scoreColor()` helper at `QuizAnalyticsDashboard.tsx:20-24` correctly maps percentage ranges to `text-success`, `text-warning`, and `text-destructive` design tokens. No hardcoded Tailwind colors detected in new code.

3. **No uncommitted changes.** Working tree is clean, meaning all findings are against committed code. This breaks the recurring uncommitted-fixes pattern from E03-E05 era.

### Findings

#### High Priority

- **`src/lib/analytics.ts:105-107` (confidence: 92)**: `topPerforming` and `needsImprovement` overlap when there are 5 or fewer quizzes. With 3 unique quizzes, `sortedByScore.slice(0, 5)` returns all 3, and `[...sortedByScore].reverse().slice(0, 5)` also returns all 3 in reversed order. The learner sees the same quiz in both the "Top Performing" and "Quizzes Needing Practice" lists simultaneously, which is confusing and misleading. Why: A learner with 2-3 quizzes sees contradictory categorization of the same quiz. Fix: Filter `needsImprovement` to exclude IDs already in `topPerforming`, or only populate `needsImprovement` when there are more than 5 quizzes: `const needsImprovement = sortedByScore.length > 5 ? [...sortedByScore].reverse().slice(0, 5) : []`.

- **[Recurring] `src/app/components/reports/QuizAnalyticsDashboard.tsx:109,126,224` (confidence: 90)**: String interpolation for className instead of `cn()`. Lines 109, 126, and 224 use `` `text-3xl font-bold ${scoreColor(...)}` `` pattern. This is the recurring anti-pattern tracked since E01-S03 (now 17+ occurrences across the codebase). Why: String interpolation doesn't handle class conflicts or conditional merging. If `scoreColor` ever returns a class that conflicts with existing utilities, Tailwind merge won't resolve it. Fix: Import `cn` from `@/app/components/ui/utils` and replace all 3 instances: `cn('text-3xl font-bold', scoreColor(summary.averageScore))`.

- **`src/app/components/reports/QuizAnalyticsDashboard.tsx:26-46` (confidence: 85)**: No error state rendered to the user. When `calculateQuizAnalytics()` rejects, the catch block logs the error and sets `loading = false`, but `summary` remains `null`. The component then falls through to the empty state at line 65 (`!summary || summary.totalQuizzesCompleted === 0`), which displays "No quiz data yet" with a CTA to browse courses. Why: A learner whose IndexedDB read fails sees "No quiz data yet" even though they have quiz data -- the message is actively misleading and hides a real error. There is no way to retry. Fix: Add an `error` state variable. In the catch block, set it. Render a distinct error state (e.g., "Failed to load quiz analytics. Please try again.") with a retry button that re-triggers the fetch.

- **`src/lib/__tests__/analytics.test.ts` (confidence: 88)**: `calculateQuizAnalytics()` has zero unit tests. The test file covers `analyzeTopicPerformance`, `calculateImprovement`, `calculateNormalizedGain`, `calculateRetakeFrequency`, `interpretRetakeFrequency`, `calculateItemDifficulty`, and `calculateDiscriminationIndices` -- but not the new `calculateQuizAnalytics` function. The story's own lessons-learned section at line 144 explicitly calls this out. The function has non-trivial logic: grouping by quizId, computing averages, sorting top/bottom performers, handling the overlap issue above. Why: The most important new function in this story has no isolated test coverage. Regressions in aggregation logic (e.g., the overlap bug above) would go undetected. Fix: Add test cases for `calculateQuizAnalytics` covering: empty attempts, single quiz with multiple attempts, multiple quizzes, overlap scenario with <= 5 quizzes. The existing db mock needs extending to include `db.quizzes.toArray`.

#### Medium

- **`src/app/components/reports/QuizAnalyticsDashboard.tsx:181` (confidence: 78)**: `new Date(attempt.completedAt).toLocaleDateString()` uses the no-argument form of `toLocaleDateString()`, producing locale-dependent output. Other components in the codebase use explicit locale strings (e.g., `'en-US'` in `SessionHistory.tsx:37`, `'sv-SE'` in `Settings.tsx:100`). The pre-review checklist specifies `toLocaleDateString('sv-SE')` for date handling. Why: Inconsistent date formatting across the Reports page; output varies by browser/OS locale. Fix: Use `toLocaleDateString('en-US', { month: 'short', day: 'numeric', year: 'numeric' })` to match the pattern in `SessionHistory.tsx`, or extract to a shared `formatDate` utility.

- **`src/app/components/reports/QuizAnalyticsDashboard.tsx:65` (confidence: 82)**: AC2 specifies the empty state message must be "No quiz data yet. Complete a quiz to see your analytics." (single string). The implementation splits this into `title="No quiz data yet"` and `description="Complete a quiz to see your analytics."` which is fine for the EmptyState component pattern -- but AC2 also specifies "a CTA linking to available quizzes". The CTA links to `/courses` with label "Browse Courses". This is acceptable but the story says "available quizzes" which implies linking to a quiz-specific listing. Why: Minor AC fidelity gap -- linking to courses is reasonable since quizzes are embedded in courses, but worth confirming intent. Fix: Verify with product that `/courses` is the correct destination, or link to a filtered view if a quiz listing exists.

- **`src/app/pages/Reports.tsx:193` (confidence: 75)**: `setSearchParams({ tab })` replaces ALL existing search params. If the URL had other query params (e.g., from future features or external links), they would be silently dropped when switching tabs. Why: Future-proofing concern -- any additional URL state on the Reports page would be lost on tab switch. Fix: Use the functional form to preserve existing params: `setSearchParams(prev => { prev.set('tab', tab); return prev })`.

- **`src/lib/analytics.ts:110` (confidence: 80)**: `totalQuizzesCompleted` is set to `uniqueQuizzes` (unique quizzes attempted), but the field name and AC1 say "Total quizzes completed". These are semantically different: a learner who attempted 3 unique quizzes expects to see "3" for "quizzes completed", but the naming is ambiguous about whether "completed" means "unique quizzes attempted at least once" or "total quiz submissions". The JSDoc at line 42 clarifies the intent, but the displayed label in the UI card (line 90: "Total Quizzes Completed") could confuse learners who've taken the same quiz 5 times. Why: Minor but the metric card displays a value that could be misinterpreted. Fix: Consider renaming the UI label to "Unique Quizzes Attempted" or "Quizzes Taken" for clarity, or keep as-is if product confirms the intent.

#### Nits

- **Nit** `src/app/components/reports/QuizAnalyticsDashboard.tsx:217` (confidence: 72): Quiz title links in Top Performing and Needs Improvement have `max-w-[160px] truncate`, which hard-caps the title width regardless of available space. On wider viewports, this wastes horizontal space. Fix: Consider `truncate max-w-[60%]` or `min-w-0 truncate` with flex layout to let titles fill available space.

- **Nit** `src/app/components/reports/QuizAnalyticsDashboard.tsx:184-189` (confidence: 65): The "Details" link navigates to `/reports/quiz/${attempt.quizId}` but AC3 (Task 4: QuizDetailAnalytics page + route) is not implemented in this branch. Clicking "Details" will produce a 404 or blank page. The story doc acknowledges this at line 148. Not a blocker since the story explicitly defers Task 4, but the links are live and clickable today.

- **Nit** `src/lib/analytics.ts:77` (confidence: 60): `Math.max(...attempts.map(a => a.percentage))` can cause a stack overflow with very large arrays (>~65k) due to spread into function arguments. Unlikely at quiz scale but worth noting for robustness. Fix: Use `attempts.reduce((max, a) => Math.max(max, a.percentage), -Infinity)`.

### Recommendations

1. **First**: Fix the `topPerforming`/`needsImprovement` overlap logic (H1) -- this is a visible data bug for most learners who have fewer than 6 quizzes.
2. **Second**: Replace string interpolation with `cn()` in 3 locations (H2) -- mechanical fix, addresses recurring codebase pattern.
3. **Third**: Add an error state to `QuizAnalyticsDashboard` (H3) -- prevents misleading "no data" message on IndexedDB failures.
4. **Fourth**: Add unit tests for `calculateQuizAnalytics()` (H4) -- the function's aggregation logic needs coverage, especially the overlap scenario.
5. **Fifth**: Fix `toLocaleDateString()` inconsistency (M1) and `setSearchParams` param replacement (M3).
