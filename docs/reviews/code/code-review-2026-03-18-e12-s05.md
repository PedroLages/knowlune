# Code Review: E12-S05 — Display Multiple Choice Questions

## What Works Well

1. **Clean polymorphic dispatcher pattern**: `QuestionDisplay.tsx` is a thin switch with clear extensibility for Epic 14 question types. The `mode` prop surface for future review modes (Epic 16) is a smart forward-compatibility choice.

2. **Semantic HTML and accessibility**: The `<fieldset>` / `<legend>` / `RadioGroup` structure is textbook accessible form design. Keyboard navigation comes free from Radix's RadioGroup primitive.

3. **Solid test coverage**: 15 unit tests covering rendering, selection, Markdown, edge cases (empty options, >6 options), semantic structure, and mode forward-compat. The E2E tests validate real user flows including mobile touch targets with bounding box assertions.

## Findings

### High Priority

- **[MultipleChoiceQuestion.tsx:50-51] (confidence: 92)**: **Duplicate option text causes React key collisions and broken radio behavior.** Using `key={option}` (the option string itself) means if a quiz author writes two options with identical text (e.g., "None of the above" appearing twice due to a data bug), React will silently render only one and the RadioGroup will malfunction. Why: Corrupted quiz data from IndexedDB should degrade gracefully, not break the UI. Fix: Use the option index as key (`key={index}`) or a composite key (`key={\`${index}-${option}\``). The index is stable here since options don't reorder within a render.

- **[MultipleChoiceQuestion.tsx:53-56] (confidence: 90)**: **Border width mismatch (1px vs 2px) causes layout shift on selection.** Unselected options use `border` (1px) while selected uses `border-2` (2px). The extra 1px on each side causes a visible 2px total content shift when an option is selected or deselected. Why: Learners will see the option labels "jump" on every click, which feels broken. Fix: Give unselected options `border-2 border-transparent` (or `border-2 border-border`) so the border width is constant and only the color changes. The AC specifies `border border-border` for unselected, but this is a UX defect in the spec — constant geometry is essential for interactive elements.

- **[MultipleChoiceQuestion.tsx:37] (confidence: 78)**: **`remarkPlugins` array recreated every render triggers unnecessary Markdown re-parsing.** `remarkPlugins={[remarkGfm]}` creates a new array reference on each render, causing react-markdown to reinitialize its plugin pipeline. Why: During rapid option selection (each click triggers a re-render via store update), the question text Markdown gets re-parsed unnecessarily. Fix: Hoist the plugins array to module scope: `const REMARK_PLUGINS = [remarkGfm]` and reference it as `remarkPlugins={REMARK_PLUGINS}`.

- **[Quiz.tsx:191] (confidence: 85)**: **`submitAnswer` closure captures `currentQuestion` from render scope — safe now but fragile.** The `onChange` callback `answer => submitAnswer(currentQuestion.id, answer)` closes over `currentQuestion` which is derived from store state. Currently safe because the callback fires synchronously on user click, but if `submitAnswer` becomes async (likely for E12-S06 score calculation), a stale `currentQuestion.id` could submit to the wrong question. Why: When question navigation ships, rapid clicks could race. Fix: Consider extracting `currentQuestion.id` into a `const` and passing it explicitly, or better yet, derive the current question ID inside the store action itself.

### Medium

- **[MultipleChoiceQuestion.tsx:26-31] (confidence: 75)**: **`useEffect` for console warning is unnecessary side-effect machinery.** The warning for unusual option counts runs on mount and when `options.length` changes. This could just be a plain `if` statement at the top of the render body (before the JSX return), since `console.warn` is a synchronous, side-effect-free-enough operation that doesn't need effect scheduling. Fix: Replace the `useEffect` with a direct conditional: `if (options.length < 2 || options.length > 6) console.warn(...)`. This is simpler, fires on every render that has bad data (not just when length changes), and avoids the effect overhead.

- **[QuestionDisplay.tsx:37-38] (confidence: 80)**: **Type assertions at dispatch boundary bypass TypeScript safety.** `value as string | undefined` and `onChange as (answer: string) => void` are type assertions that the compiler cannot verify. If a future developer passes `string[]` for a multiple-choice question's value (e.g., from a store bug), the assertion will silently pass the wrong type through. Fix: Add a runtime guard: `if (typeof value !== 'string' && value !== undefined) { console.warn(...); return null }` — or use discriminated union types on the Question type itself so the compiler can narrow automatically.

- **[story-e12-s05.spec.ts:919] (confidence: 72)**: **`waitForLoadState('networkidle')` is a soft anti-pattern in Playwright.** The `networkidle` strategy waits for no network activity for 500ms, which is fragile (a stray analytics ping or service worker could delay it indefinitely or pass prematurely). Why: Flaky E2E tests erode developer trust. Fix: Replace with `await page.waitForLoadState('domcontentloaded')` followed by waiting for a specific visible element that confirms the page is ready (e.g., `await expect(page.getByRole('button', { name: /start quiz/i })).toBeVisible()`).

- **[MultipleChoiceQuestion.tsx:36] (confidence: 70)**: **`<legend>` containing block-level Markdown output is technically invalid HTML.** The `<legend>` element expects phrasing content, but `react-markdown` renders `<p>` tags (block elements) inside it. While browsers handle this gracefully (as noted in the lessons learned), it produces invalid HTML that could confuse assistive technologies or future browser versions. Fix: Pass a custom `components={{ p: ({children}) => <span>{children}</span> }}` to react-markdown to force inline rendering inside the legend.

### Nits

- **Nit** [MultipleChoiceQuestion.tsx:22] (confidence: 65): `question.options ?? []` — the `options` field is typed as `string[] | undefined` from the Zod schema. This fallback is correct but consider adding an explicit type guard or assertion at the `QuestionDisplay` level to fail fast on malformed data rather than silently rendering zero options.

- **Nit** [story-e12-s05.spec.ts] (confidence: 60): No E2E test for AC2's persistence requirement: "the selection persists via Zustand store if I navigate away and return." The test selects an option and verifies visual state, but never navigates away and back to confirm the answer is still selected. This is a gap relative to the AC, though it may be covered by store-level unit tests.

- **Nit** [story-e12-s05.spec.ts:925] (confidence: 60): `seedQuizzes(page, [quiz as unknown as Record<string, unknown>])` — the double cast through `unknown` suggests a type mismatch between the factory output and the seed helper's expected input. This should be fixed at the type level rather than cast around.

## Recommendations

Fix in this order:
1. **Layout shift** (border-2 mismatch) — most visible UX bug, quick fix
2. **Duplicate key** risk — defensive fix, prevents silent corruption
3. **Hoist remarkPlugins** — trivial performance win
4. **Replace useEffect warning** with direct conditional — simplification
5. **Legend block content** — HTML validity, low risk but good practice
6. Address remaining medium/nit items as time allows
