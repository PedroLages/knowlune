### Findings

#### Blocker

- **`src/app/components/embeddings/EmbeddingModelProgressToast.tsx:200,224-261` and `src/app/components/embeddings/__tests__/EmbeddingModelProgressToast.test.tsx:345-358` (confidence: 95, category: `[Correctness]`)**: Flaky test with underlying bug — worker-crash handler fires a false error toast when warm-up was never attempted.

  **What**: The component sets `warmupAttemptedRef.current = true` on mount (line 200) whenever the first-progress timeout gate passes (i.e., `supportsWorkers()` is true AND deviceMemory >= 4GB). This happens BEFORE the warm-up actually fires (which is delayed 3s + requestIdleCallback in App.tsx). The worker-crash handler (line 224) checks `warmupAttemptedRef.current` and shows a toast: "The AI model download failed: OOM."

  The test "ignores worker-crash when no download was in progress" asserts no error toast is shown. It passes when run as part of the full test suite due to test-ordering side effects (shared timer state between tests), but **fails when run in isolation**:

  ```
  npx vitest run ... -t "ignores worker-crash" → FAIL
  Expected "vi.fn()" to not be called at all, but actually been called 1 times
  ```

  **Why it matters**: If a worker crashes before the warm-up timer fires (or if the warm-up is skipped due to its own internal gate), the toast would show "The AI model download failed: OOM" even though no download was initiated. This is a UX regression. Additionally, a flaky test will cause false failures in CI and erode trust in the test suite.

  **Fix**: Add a `!toastIdRef.current` guard to the worker-crash handler, so error toasts are only shown if a progress toast was actually displayed (confirming download started):

  ```ts
  const handleWorkerCrash = (event: Event) => {
    // Only react if we were actually waiting for a download
    if (!warmupAttemptedRef.current || hasCompletedRef.current) return
    // Surface crash only if we've actually shown download progress
    if (!toastIdRef.current) return
    // ... rest of handler ...
  }
  ```

  This also fixes the flaky test: the test dispatches a crash before any progress event, so `toastIdRef.current` is null, and the handler returns early.

  **Effort**: ~5 min
  **Autofix**: `manual`

#### Medium

- **`src/app/App.tsx:102-112` and `src/app/components/embeddings/EmbeddingModelProgressToast.tsx:193-198` (confidence: 75, category: `[Maintainability]`)**: Gate logic for warm-up eligibility duplicated in two files. Both independently check `supportsWorkers()` and `deviceMemory >= 4GB` to determine whether warm-up should be attempted. If one threshold or condition changes (e.g., lowering to 2GB, adding a battery check), the two will diverge, causing either false "download did not start" errors (component gate too permissive) or silent masked errors (component gate too strict, timeout never fires when it should).

  **Fix**: Extract the gate into a shared utility function in `src/ai/lib/workerCapabilities.ts`:

  ```ts
  export function shouldAttemptModelWarmup(): boolean {
    if (!supportsWorkers()) return false
    const deviceMemory = (navigator as { deviceMemory?: number }).deviceMemory
    return deviceMemory === undefined || deviceMemory >= 4
  }
  ```

  Then import from the same source in both App.tsx and the toast component.

  **Effort**: ~15 min
  **Autofix**: `gated_auto`

#### Nits

- **`src/app/components/embeddings/EmbeddingModelProgressToast.tsx` (confidence: 65, category: `[Architecture]`)**: First-progress timeout (15s) starts from component mount, but warm-up starts 3s (setTimeout) + `requestIdleCallback` delay later. On a busy main thread where `requestIdleCallback` is deferred >12s, the effective timeout for download start would be < 3s — potentially firing before the warm-up's HTTP request even leaves the browser. Unlikely in practice (page load typically has ample idle time after 3s), but the timer anchoring is technically wrong.

  **Fix**: Either document the intentional 3s fudge factor in a comment, or restructure so the timeout starts from a "warm-up started" event. Not worth a refactor unless evidence shows actual false errors on slow devices.

  **Effort**: ~1 min for comment, ~20 min for full restructure
  **Autofix**: `advisory`

### Recommendations

1. **Fix the blocker first** (5 min): Add the `!toastIdRef.current` guard to the worker-crash handler. This resolves both the false error toast and the flaky test.

2. **Extract shared gate** (15 min): Move the warm-up eligibility check to a shared utility. Prevents future divergence when the gate inevitably changes (different threshold, additional conditions).

3. **Document the timing gap** (1 min): Add a comment noting the 15s first-progress timeout counts from mount, not from warm-up start, and that `requestIdleCallback` delay could erode the effective window. Low priority.
