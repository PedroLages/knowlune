#### Medium

1. **[Correctness] src/stores/useReadingQueueStore.ts:38-43 (confidence: 75)**: `loadQueue` uses `isLoaded` as a guard but never resets it. If the queue is loaded, a book is added externally (e.g., from another tab via Dexie's multi-tab support), and the user navigates away and back, the queue will not refresh from DB because `isLoaded` is already `true`. This is acceptable for a single-tab app but could cause stale state if multi-tab usage is ever introduced. Fix: Consider resetting `isLoaded` on a visibility change listener, or document the single-tab assumption. ~15 min.

2. **[Correctness] src/stores/useReadingQueueStore.ts:89-110 (confidence: 72)**: `reorderQueue` writes each entry individually in a Dexie transaction loop. For large queues this is correct (transaction ensures atomicity) but the loop uses individual `update()` calls. A `bulkPut()` with the full reordered entries would be more efficient and simpler. Fix: Replace the for-loop with `await db.readingQueue.bulkPut(reordered)`. ~5 min.

3. **[Maintainability] src/app/components/library/ReadingQueue.tsx:163 (confidence: 70)**: `bookMap` is rebuilt on every render (`new Map(books.map(...))`). With the current store design this runs on every parent re-render. For a small number of books this is fine, but it could be wrapped in `useMemo` keyed on `books` for clarity of intent. Fix: `const bookMap = useMemo(() => new Map(books.map(b => [b.id, b])), [books])`. ~2 min.
