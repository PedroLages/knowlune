## Code Review — E101-S02: Server Connection & Authentication UI (2026-04-05)

### Summary

Reviewed 8 new/modified files (1273 insertions). Architecture follows established OpdsCatalogSettings pattern closely. Store, components, and tests are well-structured.

### Findings

#### Blockers
*(none)*

#### High Priority
- **[AudiobookshelfSettings.tsx:179 (confidence: 90)]** `handleSave` has no error handling around `await addServer(...)` / `await updateServer(...)`. The store methods catch errors internally and show toasts but don't rethrow or return a success indicator. `handleSave` unconditionally proceeds to `resetForm()` and `setMode('list')` even on failure, making it appear the save succeeded. Fix: Have store methods return a boolean, or wrap in try/catch and only transition on success.
  - **[Consensus: 2 — Claude + GLM]** Both reviewers flagged this independently.

#### Medium
- **[useAudiobookshelfStore.ts:45 (confidence: 75)]** `addServer` uses `db.audiobookshelfServers.put(server)` which silently overwrites on duplicate ID. Semantically `add()` would be more correct (fails fast on duplicates). UUID collision is astronomically unlikely, but `add()` is safer and matches the method name's intent.
- **[AudiobookshelfSettings.tsx:156 (confidence: 70)]** `selectedLibraryIds.length` in `handleTestConnection`'s `useCallback` dependency array captures stale closure. If the user toggles checkboxes while the test request is in-flight, the auto-select logic may behave unexpectedly. Fix: Use functional state update: `setSelectedLibraryIds(prev => prev.length === 0 ? libResult.data.map(lib => lib.id) : prev)`.

#### Nits
- **[AudiobookshelfServerCard.tsx:84,93]** Edit and Remove buttons use `size-9` (36px) which is below the 44px WCAG minimum touch target. (Pre-existing pattern from KI-035.)

### Architecture Assessment
- Store pattern mirrors `useOpdsCatalogStore` correctly (Zustand + Dexie, isLoaded guard)
- Component decomposition is clean: Settings (orchestrator) -> ListView, Form, DeleteDialog
- All catch blocks have `toast.error()` (no silent failures)
- API key never logged to console
- Design tokens used throughout (no hardcoded colors)
- `isInsecureUrl()` reused from existing service (no reimplementation)

### Verdict
1 HIGH finding (handleSave error handling). Should be fixed before shipping.
