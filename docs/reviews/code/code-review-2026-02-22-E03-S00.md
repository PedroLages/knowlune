## Findings

### Blockers

#### 1. `&videoId` unique index prevents multiple notes across courses with shared lesson IDs

**[/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/db/schema.ts:51]** and **[/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/lib/progress.ts:275-301]**

The Dexie schema declares `&videoId` (unique index) on the notes table:
```
notes: 'id, courseId, &videoId, *tags, createdAt, updatedAt'
```

This means only ONE note can exist for any given `videoId` **globally across all courses**. If two different courses share a lesson ID (e.g., both have a lesson `"lesson-1"`), the second note insert throws a `ConstraintError`.

Additionally, `addNote()` in `progress.ts` is documented as "Add a new note to a lesson (for multiple notes per lesson)" but calls `db.notes.add(newNote)` with a fresh UUID as `id` and the same `videoId` -- the unique constraint blocks this entirely.

**Why**: A learner with two courses that happen to share any lesson ID silently loses their note. The `addNote()` API is a public function that promises multi-note support but will always fail on the second call for the same lesson. Data loss is a blocker.

**Fix**: The schema and the API must agree on cardinality. If the intent is one-note-per-video (as `saveNote`'s upsert logic suggests), remove `addNote()` entirely or deprecate it. If multi-note is intended, change `&videoId` to a non-unique index. If it's one-note-per-course-per-video, use a compound unique index: `'id, courseId, [courseId+videoId], *tags, createdAt, updatedAt'`.

#### 2. MiniSearch index is never updated after initialization -- search goes stale immediately

**[/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/stores/useNoteStore.ts]** (all CRUD actions), **[/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/lib/progress.ts:214-311]**, and **[/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/lib/noteSearch.ts:44-80]**

`addToIndex()`, `updateInIndex()`, and `removeFromIndex()` are implemented and tested in `noteSearch.ts`, but **none of these functions are called anywhere** in production code. A grep across the entire `src/` directory (excluding tests) shows zero call sites. The search index is populated once at startup in `main.tsx` (line 15-16) and never mutated again.

Every note created, updated, or deleted after app load will be invisible to search until a full page reload.

**Why**: MiniSearch integration is Task 6 of this story. If the index goes stale on every mutation, the search feature is fundamentally broken for learners.

**Fix**: Call the appropriate index function in whichever write path is canonical (store or `progress.ts`):
```typescript
// After note creation:
addToIndex(note)
// After note update:
updateInIndex(note)
// After note deletion:
removeFromIndex(noteId)
```

### High Priority

#### 3. Migration error is silently swallowed -- toast notification never fires (AC5 violation)

**[/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/db/schema.ts:104-106]**

The `try/catch` inside the v4 `.upgrade()` callback catches all errors and only calls `console.error('[Migration] Failed:', error)`. It does **not** re-throw. This means if the migration fails (e.g., `bulkAdd` hits a duplicate key, or localStorage data is malformed), the Dexie version upgrade still completes successfully to v4. The migration will never retry on subsequent loads because it is version-guarded.

The AC explicitly requires: "the error is logged: `[Migration] Failed: {error}`" **AND** "a toast notification warns: 'Data migration incomplete. Some features may be limited.'" The toast is only triggered in the `db.open().catch()` handler in `main.tsx` (line 18-21), but the swallowed error means `db.open()` resolves successfully -- the toast never fires.

```typescript
// Line 104-106 -- current:
} catch (error) {
  console.error('[Migration] Failed:', error)
}
// Should be:
} catch (error) {
  console.error('[Migration] Failed:', error)
  throw error  // Let db.open().catch() handle the toast
}
```

**Why**: A learner whose migration silently fails will never see the warning toast, will never get a retry, and their notes are permanently lost.

**Fix**: Re-throw the error after logging so it propagates to the `db.open().catch()` handler.

#### 4. `saveNote` in progress.ts queries by `videoId` alone, ignoring `courseId`

**[/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/lib/progress.ts:222]**

```typescript
const existing = await db.notes.where({ videoId: lessonId }).first()
```

This does NOT filter by `courseId`. If two courses have a lesson with the same ID (e.g., `"lesson-1"`), saving a note in course B finds and overwrites the note belonging to course A.

Compare with `getNotes()` at line 256 which correctly queries `{ courseId, videoId: lessonId }`.

**Why**: Cross-course note corruption. Notes from one course silently overwrite notes from another.

**Fix**: Change line 222 to `db.notes.where({ courseId, videoId: lessonId }).first()`.

#### 5. `LessonPlayer` loads notes twice on mount with a lint suppression

**[/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/app/pages/LessonPlayer.tsx:58-62, 110-116]**

Two separate useEffects call `getNote(courseId, lessonId).then(setNoteText)`:

```typescript
// Effect 1 (line 58-62):
useEffect(() => {
  if (courseId && lessonId) {
    getNote(courseId, lessonId).then(setNoteText)
  }
}, []) // eslint-disable-line react-hooks/exhaustive-deps

// Effect 2 (line 110-116):
useEffect(() => {
  setShowAutoAdvance(false)
  if (courseId && lessonId) {
    setCompleted(isLessonComplete(courseId, lessonId))
    getNote(courseId, lessonId).then(setNoteText)
  }
}, [courseId, lessonId])
```

On initial mount, both effects fire, causing two IndexedDB reads for the same data. The first effect has an empty dependency array with a suppressed lint rule, hiding the stale closure risk.

**Why**: Double I/O on every mount, suppressed lint warning hiding real dependency issues.

**Fix**: Remove the first useEffect entirely. The second one already handles both mount and lesson-change scenarios.

### Medium

#### 6. Dual write paths for notes -- `progress.ts` and `useNoteStore` both write to `db.notes`

**[/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/lib/progress.ts:214-311]** and **[/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/stores/useNoteStore.ts]**

Both `progress.ts` (`saveNote`, `addNote`, `deleteNote`) and `useNoteStore` (`saveNote`, `addNote`, `deleteNote`) write directly to `db.notes`. The `LessonPlayer` still imports from `progress.ts` (line 27-28), not the store. No component anywhere imports `useNoteStore` or `useBookmarkStore` (confirmed via grep across `src/app/`).

This means:
- The store's in-memory `notes` array is never populated in production
- The store's optimistic update + rollback logic is dead code
- `progress.ts` functions lack `persistWithRetry`, unlike the store equivalents
- Future components reading from the store will see empty/stale state

**Why**: Two write paths to the same table will cause divergent state when Epic 3 stories start rendering note lists from the store.

**Fix**: Either (a) wire `LessonPlayer` and other consumers to use the stores now, or (b) mark the `progress.ts` note functions with `@deprecated` and add a clear TODO comment documenting the planned migration path. At minimum, add `persistWithRetry` to `progress.ts` functions for parity.

#### 7. No test for migration error handling path (AC5)

**[/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/db/__tests__/schema.test.ts]**

The migration test covers only the happy path (notes migrated, localStorage retained). There is no test for:
- Malformed localStorage JSON (corrupted data)
- Duplicate note IDs in localStorage
- Missing required fields (no `id`, no `content`)
- The error logging message format (`[Migration] Failed: {error}`)
- The toast notification on failure

AC5 explicitly requires: "the error is logged" AND "a toast notification warns." Neither is tested.

**Fix**: Add at least one test that seeds localStorage with malformed data and verifies the error is re-thrown (once finding 3 is fixed) or at minimum that `console.error` is called with the correct format.

#### 8. `ProgressCourseCard` uses string interpolation instead of `cn()` utility

**[/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/app/components/ProgressCourseCard.tsx:83-89, 136-142]**

```typescript
className={`group card-hover-lift h-full cursor-pointer ${

This is a recurring pattern flagged in every review since S03. The project has `cn()` from `@/app/components/ui/utils` specifically for this purpose.

  status === 'not-started' && 'opacity-80 hover:opacity-100'

```

#### 10. `persistWithRetry` delays array is coupled to `maxRetries` default but not validated

**Nit** [/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/lib/persistWithRetry.ts:6]:
```typescript
const delays = [1000, 2000, 4000]
```
If a caller passes `maxRetries = 5`, `delays[3]` and `delays[4]` are `undefined`, causing `setTimeout(resolve, undefined)` which resolves to `setTimeout(resolve, 0)` -- no backoff on retries 4 and 5.

**Fix**: Generate delays dynamically: `const delay = Math.min(1000 * Math.pow(2, attempt), 8000)`
