## Findings

### MEDIUM: `Date.now()` usage in useDashboardOrder hook (Story-Related)

**File:** `src/hooks/useDashboardOrder.ts:86, 91, 109`

The IntersectionObserver callback uses `Date.now()` for timing section visibility. While the ESLint `deterministic-time` rule only applies to test files, this makes the component harder to test in unit tests. The relevance scoring in `dashboardOrder.ts:156` also uses `Date.now()` directly.

**Impact:** Low risk in production, but makes unit testing the scoring algorithm dependent on real time. Consider accepting a `now` parameter for testability.

### MEDIUM: Silent catch blocks in dashboardOrder.ts (Story-Related)

**File:** `src/lib/dashboardOrder.ts:72-74, 90-92, 132-134, 146-148`

Four catch blocks silently swallow errors. While the comments explain the intent ("Corrupted data", "Storage quota exceeded"), the `error-handling/no-silent-catch` ESLint rule would flag these. They should either add `// silent-catch-ok` comments or surface the error to the user (at least for quota exceeded).

**Recommendation:** Add `// silent-catch-ok: corrupted data fallback` comments to the data-read catches, and consider a `toast.error()` for storage quota exceeded scenarios.

### LOW: `eslint-disable` empty deps in useEffect (Story-Related)

**File:** `src/hooks/useDashboardOrder.ts:70, 115`

Two useEffect hooks use `// eslint-disable` via comment "Only run on mount - intentionally empty deps". This is an acceptable pattern but the comment on line 70 is present while line 115 has no justification comment.

### LOW: Inline style in SortableSectionRow (Story-Related)

**File:** `src/app/components/DashboardCustomizer.tsx:61-64, 69`

The ESLint `react-best-practices/no-inline-styles` rule flags line 69. This is a legitimate exception since dnd-kit requires dynamic `transform` and `transition` CSS properties. The ESLint rule documentation lists "dynamic transforms" as an explicit exception.

**Action:** No fix needed - this is a valid exception.

### LOW: Prettier formatting issue (Story-Related, Auto-Fixed)

**File:** `src/app/components/DashboardCustomizer.tsx:98-101`

Long aria-label line exceeded Prettier's print width. Auto-fixed during this review.

### NIT: `new Set()` created on every render (Story-Related)

**File:** `src/hooks/useDashboardOrder.ts:156`

`pinnedSections: new Set(config.pinnedSections)` creates a new Set instance on every render. This causes referential inequality and may trigger unnecessary re-renders in child components using the Set. Consider using `useMemo` for this conversion.

## Architecture Assessment

**Positive:**
- Clean separation of concerns: pure logic (`dashboardOrder.ts`), React hook (`useDashboardOrder.ts`), UI component (`DashboardCustomizer.tsx`)
- localStorage-based persistence is appropriate for this feature scope
- IntersectionObserver for automatic usage tracking is well-implemented
- dnd-kit integration follows their recommended patterns
- Forward-compatible: `getOrderConfig()` handles missing/removed sections gracefully

**Concerns:**
- No debouncing on `recordSectionView` / `recordSectionTime` -- each intersection event writes to localStorage immediately. High-frequency scroll could cause many writes.
- The relevance scoring algorithm (recency 40%, views 30%, time 30%) is hardcoded with no way to tune without code changes.

## Verdict

**PASS with minor recommendations.** No blockers. The architecture is clean, the feature is well-tested, and the code follows project conventions. The MEDIUM items are worth addressing but none are blocking.
