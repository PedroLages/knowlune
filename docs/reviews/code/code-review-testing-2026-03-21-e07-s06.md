#### High Priority

- **`tests/e2e/regression/story-e07-s03.spec.ts:441` (confidence: 95)**: The test is named "tiebreaker selects highest momentum when tag overlap counts match" and the AC states "the suggestion selects the course with the highest momentum score (tiebreaker)." Both imply that the line-72 `getMomentumProxy` tiebreaker branch in `/Volumes/SSD/Dev/Apps/Elearningplatformwireframes/src/lib/suggestions.ts:72` is what decides the winner. It is not. With identical `tagScore = 2/7` for both candidates but different momentum, `finalScore` differs (0.457 vs 0.275 — a gap of 0.182). The comparison at `suggestions.ts:69` (`if (b.score !== a.score) return b.score - a.score`) resolves the sort before the tiebreaker on line 72 is ever reached. The test correctly asserts the right course wins, but it does not exercise the tiebreaker code path it claims to validate. The unit test at `src/lib/suggestions.test.ts:122` has the same structural issue: because `finalScore = tagScore * 0.6 + momentum * 0.4`, any difference in momentum produces a different `finalScore`, meaning the primary sort always fires. The line-72 tiebreaker is only reachable when both `finalScore` and `tagOverlapCount` are numerically equal — which requires artificially identical momentum values, a degenerate scenario. Fix: Either (a) accept that the AC phrase "tiebreaker" means "momentum is a factor in ranking" (in which case rename the test to reflect what is actually tested: "finalScore correctly incorporates momentum, selecting higher-momentum course"), or (b) add a separate test seeding two courses with identical `finalScore` (achieved by making momentum contributions exactly cancel tag differences) to exercise the literal line-72 path. Option (a) is the pragmatic choice; the algorithm design means true line-72 cases only arise from floating-point coincidences.

### Edge Cases to Consider

- **True line-72 tiebreaker (no test at any level)**: The `getMomentumProxy` comparison at `suggestions.ts:72` fires only when two candidates have numerically identical `finalScore` and `tagOverlapCount`. Because momentum is factored into `finalScore`, this is only possible if two courses have equal tags, equal progress ratio, and equal recency — a near-impossible real-world scenario. Neither the unit tests nor the E2E test exercises this path. This is acceptable given the degenerate nature of the scenario, but a unit test with hard-coded identical scores (mocking `Date.now` to make recency identical, and setting equal progress) would provide formal coverage of the branch.

- **Never-started candidate with zero progress**: When a course has no progress entry, `allProgress[course.id]` returns `undefined`, giving `completedLessons = 0` and `recencyScore = 0`. The unit test at `src/lib/suggestions.test.ts:175` covers this. The E2E seeding for excluded courses uses 1000 fake completed lessons, so excluded courses are never in this state. Candidate courses do have progress entries seeded, so the zero-progress edge case is not exercised by the E2E test. This is covered adequately at the unit level.

- **Floating-point score equality**: `finalScore` values for two candidates could theoretically be equal due to floating-point arithmetic rather than identical inputs. The sort at line 69 uses `b.score !== a.score`, which is an exact floating-point comparison. If two scores happen to be equal due to rounding, the sort falls through to line 70 and line 72. No test exercises this path. Acceptable as an implementation detail rather than a user-facing behavior.

- **`closeCompletionModal` called when no dialog exists**: The updated helper at line 62–69 now correctly waits for the dialog with `waitFor({ state: 'visible' })` before interacting. A descriptive timeout failure will surface if the dialog never appears. This edge case is now handled adequately.
