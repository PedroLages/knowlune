### What Works Well

1. **WAI-ARIA toolbar pattern in QuestionGrid is textbook.** Roving tabindex with `focusedIndex` state, `onFocus` synchronization, `role="toolbar"`, `aria-label`, Home/End support, and wrapping arrow navigation. This is a correct, complete implementation of the WAI-ARIA toolbar pattern and significantly improves keyboard ergonomics by reducing the number of Tab stops.

3. **Test structure uses focus barriers before keyboard assertions.** Every test that sends keyboard events first waits for `question-focus-target` to be focused (e.g., line 100, 233, 258 of the spec file), preventing the race condition between React's useEffect and Playwright's keyboard press. #### High Priority

- **`src/app/components/quiz/QuestionGrid.tsx:47-49` (confidence: 92)**: Missing `e.preventDefault()` on Enter key handler causes double-invocation of `onQuestionClick`. The `handleKeyDown` handler on the toolbar `<div>` intercepts Enter and calls `onQuestionClick(focusedIndex)`, then returns without calling `e.preventDefault()`. The Enter event continues propagating and triggers the browser's native button activation behavior, which fires the button's `onClick={() => onQuestionClick(i)}` handler. Result: `navigateToQuestion` is called twice per Enter press. While idempotent for the same index (the store just sets the same value), this causes an unnecessary re-render and could cause visible flicker if the programmatic focus `useEffect` fires twice in quick succession. More critically, it violates the WAI-ARIA toolbar keyboard interaction model where handled keys should always be prevented from reaching default behavior. Why: Double state updates on every Enter press means the `useEffect` that focuses `questionTextRef` fires twice, which could cause screen readers to announce the question text twice. Fix: Add `e.preventDefault()` before the `return` in the Enter case:
  ```tsx
  case 'Enter':
    e.preventDefault()
    onQuestionClick(focusedIndex)
    return
  ```

- **`tests/e2e/regression/story-e18-s01.spec.ts` (confidence: 90)**: AC4 (Checkboxes -- Tab to each independently, Space to toggle) has zero test coverage. The spec file header comment lists AC4 at line 10 but no test exercises it. The test data includes a `multiple-select` question (q3), and the quiz navigates through it in other tests, but no test verifies: (a) Tab moves between individual checkboxes, (b) Space toggles each checkbox independently. This is a complete gap in the acceptance criteria test matrix. Why: If checkbox Tab-stop behavior regresses (e.g., a Radix update changes focus management), this AC has no regression safety net. Fix: Add a test that navigates to Q3 (the multiple-select question), Tabs through individual checkboxes, and verifies Space toggles them:
  ```ts
  test('AC4: Checkboxes â€” Tab between and Space to toggle', async ({ page }) => {
    // Navigate to Q3 (multiple-select)
    // Tab into first checkbox, Space to toggle, Tab to second, Space to toggle
    // Verify both are checked
  })
  ```

- **`src/app/pages/Quiz.tsx:324-326` (confidence: 78)**: The `useEffect` that programmatically focuses `questionTextRef` fires on initial mount when `currentProgress` transitions from `null` to `{currentQuestionIndex: 0}`, but it also fires on every `currentProgress` state change that includes `currentQuestionIndex` -- even when the index hasn't changed. Because the dependency is `currentProgress?.currentQuestionIndex`, a new `currentProgress` object with the same `currentQuestionIndex` value will NOT retrigger the effect (React compares primitive values). However, when the quiz first starts and `currentProgress` goes from `null` (dependency = `undefined`) to `{currentQuestionIndex: 0}` (dependency = `0`), the effect fires. This is correct. But when `currentProgress` becomes `null` again (quiz submitted via `handleSubmitConfirm`), the dependency changes back to `undefined`, and the effect fires -- calling `.focus()` on a potentially unmounting DOM element. While `?.` prevents a crash, this is a wasted focus call during navigation transition. Why: Minor -- no user-visible impact. But if a future code change adds cleanup logic or side effects to this element, the spurious focus during unmount could cause subtle bugs. Fix: Guard the focus call:
  ```tsx
  useEffect(() => {
    if (currentProgress?.currentQuestionIndex != null) {
      questionTextRef.current?.focus()
    }
  }, [currentProgress?.currentQuestionIndex])
  ```

#### Medium

- **`src/app/pages/Quiz.tsx:451-454` (confidence: 75)**: The `question-focus-target` div lacks a semantic `role` attribute. When a screen reader receives programmatic focus on this div, it will announce the inner text content but may prefix it with "group" or give no role announcement, depending on the screen reader. Adding `role="region"` and `aria-label="Question content"` (or `aria-roledescription="Question"`) would give screen reader users a consistent landmark when focus lands here. Why: Screen reader users navigating by keyboard may be confused when focus lands on an unlabeled div -- they hear text content but have no context about what UI region they're in. Fix: Add `role="region"` and `aria-label` to the wrapper div:
  ```tsx
  <div
    ref={questionTextRef}
    tabIndex={-1}
    className="outline-none"
    data-testid="question-focus-target"
    role="region"
    aria-label={`Question ${(currentProgress?.currentQuestionIndex ?? 0) + 1}`}
    ...
  ```

- **`src/app/components/quiz/__tests__/QuestionGrid.test.tsx` (confidence: 82)**: The existing unit tests for QuestionGrid were not updated to cover the new roving tabindex behavior. The test file still only verifies rendering, click handling, and review indicators (pre-existing tests from before E18-S01). No unit test verifies: (a) `role="toolbar"` is rendered, (b) tabIndex assignment (focusedIndex=0, others=-1), (c) arrow key navigation via `handleKeyDown`, (d) Home/End key behavior. While E2E tests cover these scenarios, unit tests provide faster feedback and better isolation. Why: If the QuestionGrid keyboard behavior regresses, the unit test suite will pass but the component is broken -- developers won't catch it until slow E2E tests run. Fix: Add unit tests for the new keyboard interaction model using `userEvent.keyboard` or `fireEvent.keyDown`.

- **`src/app/components/quiz/QuestionGrid.tsx:23` (confidence: 72)**: `useState(currentIndex)` only captures the initial prop value. If `currentIndex` starts at a non-zero value (e.g., resuming a quiz at question 5), the `useEffect` at line 28-30 handles the sync. However, there's a single render where `focusedIndex` could differ from `currentIndex` if the component mounts with a non-zero `currentIndex` -- the initial state is set to the correct value by `useState(currentIndex)`, so this is actually fine. But the `buttonRefs` array at line 24 is initialized as empty (`[]`). On the first render, `buttonRefs.current[nextIndex]?.focus()` would be called on an element that hasn't been populated yet by the ref callbacks. The `?.` guards against null, but if `handleKeyDown` fires before React populates the refs (impossible in practice since keyboard events require focus, which requires render), it would silently fail. This is a theoretical concern only.

  More practically: `buttonRefs.current` is never trimmed when `total` decreases. If a quiz has 10 questions and the user somehow gets to a state where `total` shrinks (not possible in current implementation but defensive coding principle), stale refs in indices >= new `total` could cause `.focus()` on unmounted elements. Fix: Reset refs when total changes:
  ```tsx
  useEffect(() => {
    buttonRefs.current = buttonRefs.current.slice(0, total)
  }, [total])
  ```
