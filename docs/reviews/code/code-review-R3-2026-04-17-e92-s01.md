### Nits

- **Nit** [supabase/migrations/20260417000002_p0_sync_foundation_fixups.sql:54-71] (confidence: 90) `[Maintainability]`: Dead code — the first `CREATE OR REPLACE FUNCTION public._status_rank(s TEXT)` definition (lines 54-71) is immediately superseded by the second at lines 74-89. The first version returns NULL from CASE in a STRICT IMMUTABLE function, which would propagate NULL into `_status_rank(EXCLUDED.status) > ...` comparisons (silently false, status never advances) — but the second `CREATE OR REPLACE` overwrites it before anything calls it, so there's no runtime hazard. However, if someone ever comments out the second definition (e.g., mid-debug), the broken first version becomes live. Fix: delete lines 54-71 entirely; keep only the explicit IF/ELSIF version. ~2 min.

- **Nit** [supabase/migrations/20260417000002_p0_sync_foundation_fixups.sql:187-192] (confidence: 80) `[Architecture]`: The `vector` / `supabase_vault` schema inconsistency (created without `WITH SCHEMA extensions` in the original migration) is documented but not tracked. - **Nit** [supabase/tests/e92-s01-verify.sql:192-194] (confidence: 70) `[Testing]`: The RLS test uses `SET LOCAL ROLE authenticated` then `set_config('request.jwt.claims', ...)`. This works on self-hosted Supabase where `auth.uid()` reads from `request.jwt.claims->>sub`, but the script doesn't assert that this wiring is in place. If `auth.uid()` is ever redefined to read from a different source, the test silently becomes a no-op (userA would be NULL, RLS policies `auth.uid() = user_id` would filter everything for userA — counts still 0, assertions still pass). Fix: add a one-line sanity check `ASSERT (SELECT auth.uid()) = '00000000-0000-0000-0000-e92501000001'::uuid` after the `set_config` call. ~1 min.
