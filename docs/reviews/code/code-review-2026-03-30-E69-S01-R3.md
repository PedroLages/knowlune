# Code Review: E69-S01 -- Storage Estimation Service & Overview Card (R3)

## What Works Well

1. **Resilient async architecture.** `Promise.allSettled()` in both `estimateCategory` and `getStorageOverview` means a single corrupt Dexie table never crashes the whole dashboard. This is exactly the right pattern for a read-only estimation feature.

2. **Thorough state machine.** The component covers all five states (loading, error, API unavailable, empty, normal) with proper early returns. Each state has a reasonable UI and the cancelled-flag pattern prevents set-state-after-unmount.

3. **Accessibility done right.** `role="alert"` with `aria-live` on warning banners, `aria-busy` on loading, a visually-hidden `<table>` as screen reader alternative to the chart, and `role="list"`/`role="listitem"` on the legend grid. The 44px minimum touch targets on all buttons are correct.

### Findings

#### High Priority

- **[src/lib/storageEstimate.ts:44-50] (confidence: 90)**: Category mapping is missing tables from the current schema (v29). The `courses` table mapping lists `importedCourses`, `importedVideos`, `importedPdfs` but the story spec (Dev Notes line 106) says it should also include `courses`. While `courses` was dropped in v30, that drop is specifically for the *regular* course data table -- and the mapping already references other tables that may not have data. More importantly, several other tables with potentially significant storage are completely absent from any category: `studySessions`, `contentProgress`, `bookmarks`, `progress`, `quizzes`, `quizAttempts`, `youtubeVideoCache`, `youtubeChapters`, `aiUsageEvents`, `courseReminders`, `challenges`, `authors`, `careerPaths`, `pathEnrollments`, `entitlements`, `learningPaths`, `learningPathEntries`, `notifications`, `notificationPreferences`. While the plan says these go into `uncategorizedBytes`, for a learner with thousands of quiz attempts or study sessions, the chart could show a tiny categorized total vs a huge "Total Usage" number reported by the browser, with no explanation of where 80% of their data lives. Why: Users may think their "Courses" data is small and safe to keep, but the real storage hog (study sessions, quiz attempts) is invisible. Fix: Either add an "Other" category covering the remaining mapped tables, or add a visible "Uncategorized: ~{size}" line beneath the legend so learners understand the full picture.

- **[src/app/components/settings/StorageManagement.tsx:145] (confidence: 85)**: Unused variable `percent` in `StorageOverviewBar`. Line 145 computes `const percent = Math.round(overview.usagePercent * 100)` but this value is never used -- the summary line on line 163 uses `{percent}%` which refers to a *different* `percent` in the JSX template literal... wait, actually it does use it in the template on line 165. However, trace more carefully: line 162-165 reads `({percent}%)` where `percent` is the local const from line 145. This is actually fine -- false alarm on the unused variable. **Withdrawn.**

- **[src/lib/storageEstimate.ts:62-93] (confidence: 82)**: `estimateTableSize` creates `new Blob([JSON.stringify(row)])` for every sampled row. For tables with large text content (notes with long markdown, quiz attempts with answer data), this is fine for estimation. But for tables storing `ArrayBuffer` or `Uint8Array` data (not `Blob` instances), `JSON.stringify` will serialize them as `{"0":0,"1":0,...}` -- massively inflating the estimated size. The code correctly handles `Blob` fields via `instanceof Blob`, but `ArrayBuffer` and typed arrays are not handled. Why: If any Dexie table stores binary data as `ArrayBuffer` (not `Blob`), the estimate will be wildly inaccurate -- potentially 5-10x too high. Fix: Add `instanceof ArrayBuffer` and `ArrayBuffer.isView()` checks alongside the Blob check:
  ```typescript
  if (val instanceof Blob) {
    blobBytes += val.size
  } else if (val instanceof ArrayBuffer) {
    blobBytes += val.byteLength
  } else if (ArrayBuffer.isView(val)) {
    blobBytes += val.byteLength
  } else {
    nonBlobFields[key] = val
  }
  ```

- **[src/app/components/settings/StorageManagement.tsx:293-305] (confidence: 80)**: `handleRefresh` is an async function used as a click handler but is not wrapped in any guard against concurrent invocations beyond the `if (refreshing) return` check. The issue is that `refreshing` is React state -- between the click and the next render, a rapid double-click could invoke `handleRefresh` twice before `setRefreshing(true)` takes effect. Why: Double-clicking Refresh could fire two parallel `getStorageOverview()` calls, with the second one potentially overwriting the first with stale data (race condition). Fix: Use a ref-based guard alongside the state check:
  ```typescript
  const refreshingRef = useRef(false)
  async function handleRefresh() {
    if (refreshingRef.current) return
    refreshingRef.current = true
    // ...
    finally { refreshingRef.current = false; setRefreshing(false) }
  }
  ```

#### Medium

- **[src/app/components/settings/StorageManagement.tsx:103-104] (confidence: 75)**: "Free Up Space" button calls `document.getElementById('data-management')?.scrollIntoView()` but the `#data-management` element lives *above* the Storage Management card in the DOM (it is a preceding sibling in Settings.tsx). This means clicking "Free Up Space" scrolls *up* to the data management card, which currently has no cleanup actions (those are planned for E69-S03). The user sees a card about data management but has no actionable cleanup options. Why: Learners at 95%+ storage are told to "Free Up Space" but arrive at a section with no cleanup buttons -- a dead end. Fix: Consider disabling or relabeling the button until S03 lands, or at minimum add a tooltip explaining that cleanup actions are coming soon.

- **[src/lib/storageEstimate.ts:62] (confidence: 72)**: Default `sampleSize = 5` is hardcoded. The story spec (line 39) says `sampleSize=10`. While 5 is a reasonable default for performance, it reduces estimation accuracy for tables with high variance in row sizes (e.g., `notes` where some have screenshots and some are short text). Fix: Consider increasing to 10 as specified, or document the deviation.

- **[src/app/components/settings/StorageManagement.tsx:37] (confidence: 70)**: `DISMISS_KEY = 'storage-warning-dismissed'` uses `sessionStorage`. The AC (line 22) says "so it reappears next session" which is correct for sessionStorage. However, the key name has no namespace prefix. Other parts of the app use `knowlune-` prefixed keys (e.g., `knowlune-sidebar-v1`). Why: Key collision risk with other apps on the same origin. Fix: Rename to `knowlune-storage-warning-dismissed`.

- **[src/app/components/settings/__tests__/StorageManagement.test.tsx] (confidence: 70)**: Tests do not cover the scenario where `handleRefresh` fails -- specifically, they don't verify that `toast.error()` is called on refresh failure. The `sonner` toast mock is missing entirely. Fix: Add `vi.mock('sonner')` and test that refresh failure shows a toast.

#### Nits

- **Nit** [src/app/components/settings/StorageManagement.tsx:145] (confidence: 65): `const percent` in `StorageOverviewBar` duplicates the same computation done in `QuotaWarningBanner` (line 82). Both compute `Math.round(usagePercent * 100)`. Consider extracting a tiny helper or passing the pre-computed value as a prop if both components need it.

- **Nit** [src/lib/storageEstimate.ts:83] (confidence: 60): `new Blob([JSON.stringify(nonBlobFields)]).size` is called inside a `.reduce()` loop for every row in the sample. For very large metadata objects this allocates a Blob per row. For a sample of 5 this is fine, but if `sampleSize` is ever increased significantly, consider a lighter estimation approach (e.g., `new TextEncoder().encode(JSON.stringify(nonBlobFields)).length`).

- **Nit** [src/app/components/settings/StorageManagement.tsx:34] (confidence: 55): `transcripts` chart color uses `var(--color-muted)` while the other 5 categories use `var(--chart-N)`. This is intentional per the spec's chart config, but `--color-muted` typically resolves to a gray that may not be visually distinguishable from the card background in some themes. Consider defining a `--chart-6` token for consistency.

---

### Recommendations

1. **(High)** Add ArrayBuffer/typed-array handling to `estimateTableSize` to prevent wildly inflated estimates for binary data.
2. **(High)** Add a ref-based guard to `handleRefresh` to prevent double-invocation race conditions.
3. **(High)** Address the large gap between `categorizedTotal` and `totalUsage` by surfacing uncategorized bytes to the user, or expanding the category mapping.
4. **(Medium)** Namespace the sessionStorage key with `knowlune-` prefix.
5. **(Medium)** Increase sample size to 10 per the story spec, or document the deviation.
6. **(Medium)** Add toast mock and refresh-failure test case.
