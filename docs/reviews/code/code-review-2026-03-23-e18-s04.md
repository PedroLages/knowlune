# Code Review: E18-S04 — Verify Contrast Ratios and Touch Targets

## What Works Well

1. **Methodical contrast audit with real math.** The implementation plan includes actual hex-to-contrast-ratio calculations, and the dark mode `--brand-soft-foreground` fix was adjusted from `#8b92da` (4.44:1, fails) to `#a0a8eb` (5.99:1, passes). This is exactly the rigor WCAG compliance demands.

2. **Transition-disable pattern for dark mode E2E tests.** Using `page.addStyleTag({ content: '* { transition: none !important; }' })` before applying the `.dark` class is the correct solution for eliminating CSS timing uncertainty. This avoids the `waitForTimeout` anti-pattern while achieving deterministic dark-mode testing.

3. **axe-core tag coverage is comprehensive.** Using `['wcag2a', 'wcag2aa', 'wcag21a', 'wcag21aa']` catches WCAG 2.1-specific rules (orientation, reflow) that `wcag2aa` alone misses. The `.exclude()` patterns target only dev-tooling overlays, not application content.

## Findings

### Blockers

- **`src/app/components/quiz/ReviewQuestionGrid.tsx:33` (confidence: 95)**: `ring-ring/50` focus indicator remains unfixed. Calculated contrast of `ring-ring` (oklch(0.708 0 0) = ~#b3b3b3) at 50% opacity against `bg-card` white is approximately 1.41:1 -- far below the 3:1 minimum required by AC4. The component also uses `focus-visible:outline-none` which suppresses the global high-contrast `*:focus-visible` outline, making this low-contrast ring the ONLY keyboard focus indicator. The implementation plan (Step 2) explicitly listed this file but it was not modified.
  Why: Keyboard-only learners navigating the quiz review page literally cannot see which question button is focused.
  Fix: Change `focus-visible:ring-[3px] focus-visible:ring-ring/50` to `focus-visible:ring-2 focus-visible:ring-brand focus-visible:ring-offset-2 focus-visible:ring-offset-background` (matching the fixed `QuestionGrid.tsx` pattern).

- **`src/app/components/quiz/QuestionBreakdown.tsx:60` (confidence: 95)**: `ring-ring` focus indicator unfixed. Full-opacity `ring-ring` (#b3b3b3) on white card background is ~2.10:1 contrast -- fails the 3:1 minimum for non-text elements (AC2/AC4). This component also suppresses the global outline with `focus-visible:outline-none`.
  Why: The "Question Breakdown" collapsible trigger on the QuizResults page is invisible to keyboard users when focused.
  Fix: Replace `focus-visible:ring-ring` with `focus-visible:ring-brand` on line 60.

- **`src/app/components/quiz/QuizReviewContent.tsx:109` (confidence: 95)**: "Back to Results" link uses `ring-ring` (2.10:1 contrast on card background). Same issue -- `focus-visible:outline-none` suppresses the global outline, leaving only the low-contrast ring.
  Why: The primary navigation affordance to exit quiz review mode is invisible when focused via keyboard.
  Fix: Replace `focus-visible:ring-ring` with `focus-visible:ring-brand`.

### High Priority

- **`src/app/pages/QuizResults.tsx:207` (confidence: 90)**: "Back to Lesson" link on the QuizResults page also uses `ring-ring` with `focus-visible:outline-none`. Same WCAG AC4 failure (2.10:1 contrast). This is a quiz page link that falls within the story's scope but was not audited or fixed.
  Why: Keyboard-only learners cannot see focus on the primary exit link from quiz results.
  Fix: Replace `focus-visible:ring-ring` with `focus-visible:ring-brand`.

- **`src/app/pages/Quiz.tsx:387` (confidence: 90)**: "Back to course" link in the quiz error/empty state uses `ring-ring`. Same 2.10:1 contrast failure on card background. This is the error recovery path for keyboard users.
  Why: If a quiz fails to load, the only escape link has an invisible focus indicator.
  Fix: Replace `focus-visible:ring-ring` with `focus-visible:ring-brand`.

- **`tests/e2e/story-e18-s04.spec.ts` — Missing test coverage for QuizResults and QuizReview pages (confidence: 88)**: The E2E spec tests the quiz start screen and active quiz taking (both light and dark mode), but does NOT test the QuizResults page (where QuestionBreakdown lives with unfixed `ring-ring`) or the QuizReview page (where ReviewQuestionGrid and QuizReviewContent live with unfixed `ring-ring`). Since the three BLOCKER findings above affect those exact pages, the test gap masks the violations.
  Why: axe-core scans would likely catch the contrast failures on those pages if they were tested. The missing coverage means regressions on quiz results/review pages go undetected.
  Fix: Add axe-core scans for the QuizResults page (after completing a quiz) and the QuizReview page (navigating from results to review), in both light and dark mode.

- **`tests/e2e/story-e18-s04.spec.ts:359-395` — AC4 focus indicator tests only verify `isFocused`, not contrast or visibility (confidence: 82)**: The three focus indicator tests (lines 360, 372, 383) call `.focus()` then assert `toBeFocused()`. This proves the element CAN receive focus but says nothing about whether the focus indicator is visible or has sufficient contrast. AC4 requires >=3:1 contrast AND >=2px thickness for focus indicators.
  Why: These tests would pass even if the focus ring were transparent. They don't verify the actual visual property that the AC specifies.
  Fix: After asserting focus, use `page.evaluate()` to read the computed `outline` or `box-shadow` style and verify the color contrast. Alternatively, rely solely on the axe-core scans (which DO check focus indicator visibility) and document that AC4 contrast verification is delegated to axe-core tests.

### Medium

- **`src/app/components/quiz/MarkForReview.tsx:24` — Label has `min-h-[44px]` but no width constraint (confidence: 72)**: The Label element has `min-h-[44px]` for vertical touch target, but the "Mark for Review" text plus the small bookmark icon could render narrower than 44px on some locales/font sizes. AC3 requires >=44px in BOTH dimensions ("or full width on mobile"). The `flex items-center` on the label means the width depends on content length.
  Why: "Mark for Review" is long enough in English, but if the text were ever shortened or the component reused, the width guarantee breaks. The parent `div` has `gap-2` adding some spacing, but the checkbox (16px) is a separate tap target.
  Fix: Add `min-w-[44px]` to the Label, or verify through the E2E test that width is also >= 44px (the current test on line 341 only checks `box!.height`).

- **`docs/implementation-artifacts/` — Duplicate story files (confidence: 95)**: Two story files exist: `18-4-verify-contrast-ratios-and-touch-targets.md` (placeholder template, 4.1kB) and `e18-s04-verify-contrast-ratios-and-touch-targets.md` (actual implementation tracking, 7.7kB). The `18-4-*` file has unchecked task boxes and empty sections.
  Why: Future automation parsing story status will find conflicting metadata (both claim `story_id: E18-S04`).
  Fix: Delete `docs/implementation-artifacts/18-4-verify-contrast-ratios-and-touch-targets.md`.

- **`tests/e2e/story-e18-s04.spec.ts:333-341` — Mark for Review touch target test only checks height (confidence: 78)**: The test verifies `box!.height >= 44` but does not check `box!.width >= 44`. Other touch target tests (e.g., the start button test on line 258-259) check both dimensions. AC3 states ">=44px tall AND >=44px wide."
  Why: Inconsistent with the start button test which checks both. A narrow touch target would pass this test but fail the AC.
  Fix: Add `expect(box!.width).toBeGreaterThanOrEqual(44)` on line 341.

### Nits

- **Nit** `tests/e2e/story-e18-s04.spec.ts:84` (confidence: 65): The `as unknown as Record<string, unknown>` cast on the quiz object passed to `seedQuizzes` is a code smell suggesting a type mismatch between the factory output and the seed helper's expected input. While functional, it masks potential data shape issues at the boundary. Consider updating `seedQuizzes` to accept the Quiz type directly or adding a typed adapter.

- **Nit** `src/styles/theme.css:303` (confidence: 60): The global `@apply border-border outline-ring/50` sets a default outline using `ring/50` (the low-contrast ring token at 50% opacity). This does not cause a WCAG issue because `*:focus-visible` on line 324-327 overrides it with the high-contrast `var(--brand)` outline. However, the `outline-ring/50` in the base layer is dead code that could confuse future developers examining the focus system.

## Recommendations

Fix in this order:

1. **Fix the three BLOCKER components** (ReviewQuestionGrid, QuestionBreakdown, QuizReviewContent) -- identical one-line token swap each, matching the already-fixed QuestionGrid pattern.
2. **Fix the two HIGH quiz page links** (QuizResults.tsx:207, Quiz.tsx:387) -- same `ring-ring` to `ring-brand` swap.
3. **Add E2E axe-core scans** for QuizResults and QuizReview pages (light + dark mode) to the test spec.
4. **Add width check** to the Mark for Review touch target test.
5. **Delete the duplicate story file** (`18-4-*.md`).
6. Consider strengthening focus indicator tests or documenting that AC4 contrast verification is delegated to axe-core.
