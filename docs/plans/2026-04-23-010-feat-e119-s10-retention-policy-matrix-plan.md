# feat: E119-S10 Retention Policy Matrix

## Overview

Replace the TODO placeholder in `docs/compliance/retention.md` with a complete retention matrix
covering every data artefact Knowlune holds. Mirror every row as a typed TypeScript export in
`src/lib/compliance/retentionPolicy.ts`. A parity test guards against future drift between the
matrix, the sync table registry, and the consent purposes enum. Update the Privacy Policy
retention section to reference the matrix and reflect actual periods.

## Problem Frame

GDPR Art 5(1)(e) requires documentation of storage limitation. Knowlune currently has no
authoritative record of what data is retained, for how long, or how it is deleted. The
retention-tick job (S11) needs a typed contract to enforce. The privacy notice retention section
is a single generic paragraph that does not reflect actual periods. (see origin:
docs/brainstorms/2026-04-23-e119-s10-retention-policy-matrix-requirements.md)

## Requirements Trace

- R1. `docs/compliance/retention.md` — complete matrix (AC-1, AC-2)
- R2. `src/lib/compliance/retentionPolicy.ts` — typed export mirroring every matrix row (AC-3)
- R3. Parity test — tableRegistry ↔ retentionPolicy ↔ retention.md alignment + null-period flag (AC-4, AC-6)
- R4. `PrivacyPolicy.tsx` retention section — links to retention.md, reflects actual periods (AC-5)

## Scope Boundaries

- Retention enforcement job (S11) is out of scope — this plan defines the contract; S11 enforces it.
- Annual retention review workflow (S13) is out of scope.
- Invoice retention confirmation with accountant — noted as TBC in retention.md but not blocked.

## Context & Research

### Relevant Code and Patterns

- `src/lib/sync/tableRegistry.ts` — exports `tableRegistry` array and `ERASURE_TABLE_NAMES`. The registry currently has 37 entries (one TypeScript type artefact `string` is not a real table — skip it). All real `supabaseTable` values must appear in the retention matrix.
- `src/lib/compliance/consentService.ts` — `CONSENT_PURPOSES` enum with 5 values: `ai_tutor`, `ai_embeddings`, `voice_transcription`, `analytics_telemetry`, `marketing_email`.
- `src/lib/compliance/__tests__/consentParity.test.ts` — exact pattern to mirror for the retention parity test: reads markdown, parses artefact column, compares against TypeScript exports using Vitest `describe/it/expect`.
- `src/app/pages/legal/PrivacyPolicy.tsx` — section 7 "Data Retention" at line ~283 is a single paragraph to replace.
- `docs/compliance/consent-inventory.md` — reference for markdown table style.
- `docs/compliance/retention.md` — currently a TODO placeholder to be replaced entirely.

### Institutional Learnings

- Parity tests in this codebase read markdown with `readFileSync`, parse via regex, and compare against runtime exports. No external markdown parser is used.
- The `retentionPolicy.ts` module must be a pure module (no Dexie, React, or Zustand imports) to match the pattern of other compliance modules.

## Key Technical Decisions

- **Typed `RetentionEntry` interface**: Each entry carries `artefact`, `dataCategories`, `lawfulBasis`, `period` (string | null), `deletionMechanism`, `owner`, and `notes`. `period: null` signals indefinite retention and triggers the parity test flag.
- **`retentionPolicy.ts` as the TypeScript source of truth**: The markdown matrix is the human-readable view; the TypeScript export is the machine-readable contract S11 consumes. Both must stay in sync via the parity test.
- **Parity test parses artefact column**: The test extracts the first pipe-delimited column from each table row in retention.md and compares against `RETENTION_POLICY` artefact values and `tableRegistry` supabaseTable names. This mirrors how `consentParity.test.ts` parses the consent inventory.
- **Storage buckets and auxiliary artefacts included**: The matrix covers 4 Storage buckets (audio, covers, exports, attachments), auth/session logs, syncQueue dead-letter, breach register, invoices, and exports bucket. These appear in the TypeScript export with descriptive artefact keys (not supabaseTable names) and are excluded from the tableRegistry cross-check (which only applies to sync tables).
- **Consent purposes cross-check**: Each consent purpose must map to at least one retention entry (the data generated by that purpose). The parity test verifies this mapping via a `purposeArtefacts` map in `retentionPolicy.ts`.

## Open Questions

### Resolved During Planning

- **Are there exactly 26 sync tables?** No — the registry has grown. There are 37 real entries (38 grep hits, one is a TypeScript artefact). The plan covers all of them; the story's "26" is outdated. The parity test enforces the actual count.
- **How to link retention.md from PrivacyPolicy.tsx?** Use a plain anchor link: `<a href="/legal/privacy#data-retention">detailed retention matrix</a>` or an external link to the docs URL. Since retention.md is a docs file (not served by the app), the Privacy Policy will reference it by its GitHub/docs URL or note it as available on request. A prose reference without a hyperlink is also acceptable for now.

### Deferred to Implementation

- **Invoice retention exact number** — 10y placeholder is used; retention.md notes TBC.

## System-Wide Impact

- **Interaction graph:** `retentionPolicy.ts` will be imported by S11's `retention-tick/index.ts`. No other runtime code reads it yet.
- **Error propagation:** Parity test failures block CI — any new table added to `tableRegistry` without a matching `retentionPolicy.ts` entry fails the test.
- **State lifecycle risks:** None — this is documentation + read-only TypeScript data.
- **Unchanged invariants:** `hardDeleteUser.ts` erasure logic, `consentService.ts`, and `tableRegistry.ts` are not modified by this plan.
- **Integration coverage:** The parity test exercises the full cross-reference chain at CI time without needing a running app.
