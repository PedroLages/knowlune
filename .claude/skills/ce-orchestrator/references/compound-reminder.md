# Compound reminder — Phase 3 gate + templates

When the orchestrator reaches Phase 3 (PR created), it asks the user whether to run `/ce:compound` now or defer to post-merge. The goal is to close the every.to-style compound loop with user control over timing.

## The gate (Phase 3.1)

Three options via AskUserQuestion:

1. **Defer to post-merge (Recommended)** — lessons from shipped code are most durable. PR body gets an unchecked checkbox task. User runs `/ce:compound` manually after merge.
2. **Run /ce:compound now** — dispatches the skill immediately before exit. Captures implementation lessons before PR review may change them. Best when the PR itself is the lesson (proving an approach, documenting a tricky refactor).
3. **Skip** — this PR has no lessons worth compounding (straightforward additions, docs-only changes, config tweaks).

## Why deferred is the default recommendation

Terminal state is **PR created, not merged** (per user decision 2026-04-18). `/ce:compound` documents solved problems with lessons from *shipped* behavior, so running it before merge risks documenting theoretical lessons from code that may still change in review.

Deferring to post-merge also makes it opt-in: not every PR warrants a compound doc. Bug fixes with non-obvious root causes, new patterns, and surprising failures benefit. Straightforward feature additions often don't.

## When "Run now" is the right call

- The PR is about **proving an approach**, not shipping a feature — the implementation journey itself is the lesson.
- You hit a **surprising failure** during the run (e.g., R3 review escalation, an unexpected pre-check failure that exposed a test anti-pattern).
- The problem is **unlikely to change in review** — e.g., you debugged a subtle race condition, fix is minimal, lesson is about the invariant you had to uncover.
- You want to **capture the "what didn't work" section** while the failed approaches are fresh. This is the section git history can't reconstruct and the one future-you will thank you for.

## When "Skip" is the right call

- Docs-only or config-only PRs.
- Straightforward feature additions with no surprises (a new button, a new card).
- Experiments you expect to revert.
- Work on a well-understood pattern already documented in `docs/solutions/`.

## Terminal-output template

Printed to the coordinator's user-facing stdout right before exit:

```text
✓ PR created: <prUrl>
Demo:         <demoUrl or "—">
Review:       R<rounds> (<blockers|high|medium> → 0)
Residual:     <N> LOW/NIT (see <location>)
Branch:       <branch>
Last-green:   <lastGreenSha>

After merge, run:
    /ce:compound "<slug>"

This closes the compound loop — documents lessons in docs/solutions/ so
the system learns. Skip if this PR taught no surprising lessons.
```

### Field semantics

- `<slug>`: derived from plan filename (e.g., `2026-04-18-001-feat-keyboard-help-plan.md` → `keyboard-help`). The coordinator computes this once in Phase 0 and passes it through.
- `<demoUrl>`: empty dash if `shouldCapture` was false or demo-reel errored.
- `<location>`: `docs/known-issues.yaml` if it exists and was appended to; otherwise `(see above)` pointing to inline LOW/NIT list earlier in the output.
- `R<rounds>`: actual rounds taken (1 if green on R1, 3 if escalated).

## PR body template (compound reminder section)

Appended to the PR body generated by `/ce:git-commit-push-pr`. The dispatcher sub-agent (§11 in sub-agent-prompts.md) passes this block as context:

```markdown
---

## Compound loop

- [ ] After merge, run `/ce:compound "<slug>"` to document lessons in `docs/solutions/`.
- Skip if this PR taught no surprising lessons (straightforward feature additions usually don't need a compound doc).
- Reference the plan that produced this PR: `<planPath>`.
```

### Why a checkbox, not a comment

GitHub renders `- [ ]` as an interactive checkbox. When the user runs `/ce:compound` post-merge, they can tick it to record the loop closed. Plain prose would get lost; a checkbox stays visible until ticked or manually removed.

## Escalation variant (R3 escalation path)

When the orchestrator exits via "Escalate to PR with banner", the compound reminder still goes in — but is prefixed with the escalation warning. Full PR body preamble:

```markdown
⚠️ **REVIEW-ESCALATED** — R3 review loop exited with residual findings.

Residual:
- BLOCKER: <N>
- HIGH: <N>
- MEDIUM: <N>

Human review required before merge. See `.context/compound-engineering/ce-review/<runId>/` for full findings.

---

## Compound loop

- [ ] After merge (**and after addressing residual findings**), run `/ce:compound "<slug>"`.
- Document what made the fix loop fail to converge — likely a compound-worthy lesson.
```

The escalation variant explicitly hints that a compound doc is valuable here: R3 escalations are rare and represent exactly the kind of surprising failure that `/ce:compound` is designed to capture.

## What the reminder does NOT do

- Does not write a placeholder `docs/solutions/...-pending.md` stub. Tried in plan v0, rejected as "pre-creating empty docs pollutes the solutions tree; compound should be user-triggered post-merge."
- Does not open a follow-up issue in GitHub. Too heavy for v1; the PR checkbox is the lightest signal.
- Does not block merge in any way. Terminal state stays PR-created; user merges when ready and may or may not run `/ce:compound`.

## v2+ considerations

- **Optional auto-compound flag:** `--auto-compound` could trigger `/ce:compound` immediately after merge detection (via webhook or polling). Deferred — too magical, too many ways to break. Keep it user-triggered.
- **Compound batching:** run `/ce:compound` on the last N merged PRs at once via a separate skill. Useful if compound debt builds up. Different scope; not this skill's problem.
